# HTML Story File Parsing Rules

## Required Source Format: HTML (not CSV / not pasted text)

Sprint exports under `/knowledge/sprints/Sprint N/` **must be JIRA HTML exports**.

**Why HTML, not CSV:**
- HTML preserves rich-text markup — strikethrough, bold, lists, tables, links.
- CSV / plain-text exports flatten everything to one stream — **struck-through deprecated AC or Solution content becomes indistinguishable from live authoritative content.**
- Stories in this project frequently have AC bullets or Solution paragraphs that have been **struck through** to indicate "no longer in scope" or "superseded by a later AC". Without HTML markup, those signals are lost.

**Export from JIRA via**: Issues → JQL search → **Export → HTML (Current fields)** or **HTML (All fields)** → save under `/knowledge/sprints/Sprint N/`.

If you receive a CSV or pasted-text export instead of HTML, **stop and ask for an HTML re-export** before analyzing. Do not infer or guess what was struck through.

## Strikethrough = Non-Authoritative

When extracting `Acceptance Criteria`, `Solution`, `Description`, or any other field, treat the following as **non-authoritative and excluded from analysis**:

### JIRA Wiki Markup (Primary Format in Exports)

- `-text-` (JIRA wiki strikethrough) — text surrounded by single dashes
- `{color:#hex}text{color}` (JIRA color markup) — often used to highlight deprecated or changed content

**IMPORTANT**: The HTML exports from JIRA contain JIRA wiki markup, **not** standard HTML strikethrough tags. The per-story markdown files generated by `scripts/split-sprint-stories.py` filter this markup during processing.

### HTML Markup (Rare in Current Exports)

If reading raw HTML directly (not recommended), also filter:
- `<s>...</s>` (HTML5 strikethrough)
- `<strike>...</strike>` (legacy strikethrough)
- `<del>...</del>` (deletion marker)
- Any inline style with `text-decoration: line-through`

**Note**: Analysis in April 2026 found ZERO instances of standard HTML strikethrough tags across all 20 sprint exports. All strikethrough uses JIRA wiki markup `-text-`.

**Treat struck-through content as if it were not in the field.** Specifically, do not:
- Quote it as part of the AC or Solution
- Include it in a count of AC criteria
- Use it to infer the design or current state
- Surface it as "what was originally planned" — that's intent-archaeology and is out of scope

If, after stripping struck-through content, a field becomes empty or trivial, treat the field as **missing** and call that out (per the role skill workflows in `.cursor/skills/`).

### Other rich-text signals

| Markup | Meaning | How to handle |
|---|---|---|
| `-text-` (JIRA wiki) | Withdrawn / superseded | **Exclude** |
| `{color:...}` (JIRA wiki) | Highlight/emphasis (often deprecated content) | **Exclude** |
| `*text*` (JIRA wiki bold) | Emphasis | Preserve when quoting; treat as authoritative |
| `_text_` (JIRA wiki italic) | Emphasis | Preserve when quoting; treat as authoritative |
| `[link\|url]` (JIRA wiki link) | Cross-references | Preserve link target; cite when relevant |
| `<strong>`, `<b>`, `<em>`, `<i>` | HTML emphasis (if present) | Preserve when quoting; treat as authoritative |
| `<ul><li>`, `<ol><li>` | List structure (typically the AC enumeration) | Each `<li>` is usually one AC |
| `<a href="...">` | HTML cross-references | Preserve link target; cite when relevant |
| `<table>` (inside Solution) | Structured data (mapping tables, data models) | Preserve table structure when quoting |

### Edge cases

- **Whole AC bullet struck through**: skip the bullet; renumber the live ones and note the count change in one line.
- **Mid-sentence strikethrough**: skip the struck span; surrounding live text remains authoritative.
- **Strikethrough inside `<pre>` / `<table>`**: same rule — non-authoritative.
- **Whole field struck through**: treat the field as missing; trigger the "Missing Solution / Missing AC" branch of the relevant skill.

This rule does **not** override `metadata-is-source-of-truth.mdc` — deployed metadata still wins for current-state facts. Strikethrough handling only governs how JIRA *intent* is read.

## File Format

Sprint HTML files are JIRA exports with this structure:

```html
<table id="issuetable" class="aui" border="1">
    <thead>
        <tr class="rowHeader">
            <th>Issue Key</th>
            <th>Summary</th>
            <th>Acceptance Criteria</th>
            <th>Solution</th>
            <th>Status</th>
            <th>Components</th>
            <!-- many more columns -->
        </tr>
    </thead>
    <tbody>
        <tr>
            <td class="issuekey">PROJ-1234</td>
            <td>Story title</td>
            <td>AC content...</td>
            <td>Solution details...</td>
            <td>Done</td>
            <td>ComponentName</td>
            <!-- more cells -->
        </tr>
        <!-- more story rows -->
    </tbody>
</table>
```

## Critical Fields to Extract

### Priority 1: Must Extract Every Time
1. **Issue Key** - JIRA story number (e.g., PROJ-1234)
2. **Acceptance Criteria** - Defines what "done" means
3. **Solution** - Technical approach and implementation details

### Priority 2: Important Supporting Information
4. **Summary** - Story title
5. **Description** - Detailed requirements
6. **Status** - Current state (To Do, In Progress, Done, etc.)
7. **Components** - Affected Salesforce components
8. **Story Points** - Complexity estimate
9. **Assignee** - Who is working on it

### Priority 3: Nice to Have
- Reporter, Created Date, Updated Date
- Labels, Fix Version, Sprint name
- Comments, Sub-tasks
- Time tracking fields

## Parsing Guidelines

### When Reading HTML Files

1. **Look for the main table**: `<table id="issuetable">`
2. **Parse header row**: Identify column positions (AC, Solution, etc. may be in different positions)
3. **Iterate through rows**: Each `<tr>` in `<tbody>` is a story
4. **Extract cell contents**: Match cells to column headers
5. **Clean up HTML**: Remove `<br>` tags, decode entities, strip markup

### Handling JIRA Markup in Cells

Content may include:
- `*text*` - Bold (Jira markup)
- `_text_` - Italic (Jira markup)
- `-` or `*` - Bullet points
- `#` - Numbered lists
- `<br/>` or `<br>` - Line breaks

### Common Patterns in Acceptance Criteria

```
*-Suggested Acceptance Criteria:-*
1. AC text here
2. AC text here

OR

Acceptance Criteria:
- AC 1: Description
- AC 2: Description

OR

Given [context]
When [action]
Then [expected result]
```

### Common Patterns in Solution Field

```
Technical Approach:
- Component X will be modified to...
- New flow Y will handle...
- Integration with Z via...

OR

Solution: [Detailed paragraph describing implementation]

OR

- Apex class changes
- Flow modifications
- Object/field updates
```

## Querying Stories

### Single Story Lookup (Preferred: Per-Story Markdown)
```
User: "What are the acceptance criteria for story PROJ-1234?"

Process:
1. Find the per-story file: knowledge/sprints/Sprint N/stories/PROJ-1234.md
2. Read the markdown file — AC, Solution, and Description are already extracted
3. Return the Acceptance Criteria section
Cost: ~200-500 tokens
```

### Single Story Lookup (Fallback: HTML)
```
User: "What are the acceptance criteria for story PROJ-1234?"

Process (only if per-story file does not exist):
1. Search sprint HTML files for Issue Key = PROJ-1234
2. Find the matching row
3. Extract the Acceptance Criteria cell
4. Parse and format the AC content
5. Return clean, readable acceptance criteria
Cost: ~500-2000 tokens
```

### Sprint Summary
```
User: "Summarize all stories in Sprint 14"

Process:
1. Open /knowledge/sprints/Sprint 14/Your Program Sprint 14.html
2. Count total rows in issuetable
3. Group by Status
4. Extract Issue Keys and Summaries
5. Return structured summary
```

### Component Search
```
User: "What stories modified the Account object?"

Process:
1. Search all sprint HTML files
2. Check Components column for "Account"
3. Also check Description and Solution columns for mentions
4. Collect matching Issue Keys
5. Return list with sprint context
```

### Acceptance Criteria Extraction
```
User: "List all acceptance criteria from Sprint 14"

Process:
1. Open Sprint 14 HTML
2. For each row, extract Issue Key and AC column
3. Parse AC content (may have markup)
4. Format as structured list
5. Return organized by story
```

### Solution Analysis
```
User: "What technical approaches were used in Sprint 13?"

Process:
1. Open Sprint 13 HTML
2. For each row, extract Issue Key and Solution column
3. Parse Solution content
4. Categorize by approach (Apex, Flow, Config, etc.)
5. Summarize patterns and techniques
```

### When Quoting Acceptance Criteria
```markdown
**Story PROJ-1234: [Summary]**

Acceptance Criteria:
1. AC 1 text here
2. AC 2 text here
3. AC 3 text here

(From: Sprint 14)
```

### When Quoting Solution
```markdown
**Story PROJ-1234: [Summary]**

Solution Approach:
[Clean, formatted solution text from Solution column]

Technical Components:
- Component A
- Component B

(From: Sprint 14)
```

### When Comparing Stories
```markdown
**Comparison: Story A vs Story B**

Story PROJ-1234 (Sprint 13):
- AC: [summary of AC]
- Solution: [summary of solution]
- Components: X, Y

Story PROJ-1234 (Sprint 14):
- AC: [summary of AC]
- Solution: [summary of solution]
- Components: Y, Z

Overlap: Both modify Component Y
Potential Conflict: [analysis]
```

## Edge Cases

### Empty Fields
- If AC is empty: Note that acceptance criteria are not defined
- If Solution is empty: Note that solution approach is not documented
- Don't make up content - explicitly state what's missing

### Multiple Sprints for One Story
- If a story appears in multiple sprint files, note the progression
- Compare Status changes across sprints
- Track how AC or Solution evolved

### Malformed HTML
- If table structure is unexpected, note the issue
- Attempt best-effort parsing
- Request user to verify the HTML file

### Large Sprint Files
- HTML files may have 50+ stories
- Be selective about what to return
- Offer to filter or summarize
- Don't return entire HTML table contents

## Performance Tips

1. **Don't read all sprints unless necessary**: Ask user which sprint or search selectively
2. **Cache sprint data**: If analyzing multiple stories from same sprint, keep data in context
3. **Use targeted searches**: grep for story numbers before parsing full HTML
4. **Summarize when appropriate**: Don't dump 50 stories - provide structured summaries

## Reminding Users About Code Repository

When discussing implementation:
```
"Based on the Solution field in Sprint 14, this story requires [technical changes]. 
The actual implementation of these changes lives in your Salesforce code repository. 
This workspace contains the requirements and solution design; the code itself is 
version-controlled separately."
```

Always remind users that:
- HTML files = requirements and planning
- Code repository = actual implementation
- Story numbers bridge the two
