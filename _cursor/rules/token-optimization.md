# Rule: Token Optimization

**Applies in**: all modes
**Why**: Sprint HTML exports can be 10K+ lines. Reading them naively burns budget. This rule defines the cheap-first hierarchy.

---

## The hierarchy

Always try level **N** before level **N+1**. Stop as soon as you have the answer.

### Level 1 — Per-story markdown files (cheapest for single stories, ~200–500 tokens)

For single-story lookups, **always** read the per-story file first:
- `knowledge/sprints/Sprint N/stories/STORY-ID.md` — full AC, Solution, Description in clean markdown
- Generated by `scripts/split-sprint-stories.py`

If per-story files don't exist, tell the user to run `python scripts/split-sprint-stories.py`.

### Level 2 — Index files (~100–500 tokens)

Read these for cross-story lookups and aggregated views:
- `knowledge/sprints/SPRINT-INDEX.md` — story directory
- `knowledge/AC-INDEX.md` — every AC indexed
- `knowledge/SOLUTION-INDEX.md` — every Solution indexed
- `knowledge/COMPONENT-TO-STORY-MAP.md` — components ↔ stories
- `knowledge/FEATURE-TO-STORY-MAP.md` — feature/epic ↔ stories
- `knowledge/DEPENDENCY-GRAPH.md` — story dependencies
- `knowledge/TRACEABILITY-INDEX.md` — full matrix
- `knowledge/metadata/COMPONENT-CATALOG.md` — metadata snapshot

If indexes are missing, the project hasn't run `scripts/`. Tell the user how to generate them rather than reading raw HTML.

### Level 3 — Targeted shell lookups (~50–500 tokens)

Use `grep` / `rg` / `glob` against per-story files or indexes:

```bash
# find which stories mention a component
grep -rl "Account" knowledge/sprints/*/stories/

# story location
grep -rl "STORY-123" knowledge/sprints/*/stories/

# component search across indexes
grep -l "Account" knowledge/*.md
```

### Level 4 — Targeted HTML reads (~500–2000 tokens)

Only when per-story files + indexes + grep can't answer. Read only the matching `<tr>...</tr>` block, not the whole file.

### Level 5 — Full HTML read (~5000–15000 tokens)

**Last resort.** Justify it before doing it: state what you're looking for and why levels 1–4 failed.

---

## Response patterns

### Pattern A — Summarize first

```
## STORY-123 (Sprint 4)
- Status: In Progress
- Components: Account, Contact
- AC: 5 criteria

Want the full AC, the Solution, or both?
```

Forces the user to opt-in to expensive expansion.

### Pattern B — Table of headlines

```
| Story | Sprint | Status | Components |
|-------|--------|--------|------------|
| STORY-1 | 3 | Done | Order, OrderItem |
| STORY-2 | 4 | In Progress | Order |
```

### Pattern C — Progressive disclosure

```
Found 12 stories that touch Account.
Top 3 by sprint recency:
- ...
- ...
- ...
Want the rest? Want details on any of these?
```

---

## Don'ts

- ❌ Don't read entire sprint HTML files when indexes exist
- ❌ Don't re-read files already in conversation context
- ❌ Don't paste raw HTML into the response — extract just what's needed
- ❌ Don't enumerate every story when a count + summary would do

## Dos

- ✅ Quote the cost: *"Reading the index (~300 tokens)…"*
- ✅ Ask before expensive operations: *"This will read ~12K lines, OK?"*
- ✅ Cache mentally — refer back to prior turns
- ✅ Suggest regenerating indexes if they're stale

---

## When to escalate cost

Escalate to a more expensive level only if **all** are true:

1. The cheaper level genuinely cannot answer
2. The user has confirmed they want depth
3. The cost is justified by the value of the answer

When in doubt, ask first.
