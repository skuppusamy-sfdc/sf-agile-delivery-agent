## Description

**Bug Report – Fee Schedule Rate Lookup Ambiguity**

**Environment:** Acme Insurance – Full Sandbox (FS1)
**Reported by:** Emily Watson (Claims Operations QA)
**Date Observed:** 2024-11-04

**Steps to Reproduce:**
1. Ensure two Fee_Schedule__c records exist for the same LOB (e.g., Medical) with overlapping Effective_Date__c / Expiration_Date__c ranges and both records have Status__c = 'Active'.
   - Fee Schedule A: Effective 2024-01-01 to 2024-12-31, CPT 99213 rate = $85.00
   - Fee Schedule B: Effective 2024-06-01 to 2025-05-31, CPT 99213 rate = $92.00
2. Create a Claim__c with Service_Date__c = 2024-08-15 linked to a provider with a PNA referencing Fee Schedule A.
3. Submit the Claim_Line__c for adjudication (set Claim_Line__c.Adjudication_Status__c = 'Ready for Adjudication').
4. Observe Claim_Line__c.Allowed_Amount__c after the adjudication flow runs.

**Expected Behavior:**
Claim_Line__c.Allowed_Amount__c = $85.00 (PNA-linked Fee Schedule A should take precedence; the lookup should respect the PNA.Fee_Schedule__c foreign key and not scan all active schedules).

**Actual Behavior:**
Claim_Line__c.Allowed_Amount__c = $92.00 — the system is returning the rate from Fee Schedule B (the most recently created record) because the FeeScheduleLookupService SOQL query does not filter by PNA.Fee_Schedule__c and instead retrieves the first result ordered by CreatedDate DESC.

**Impact:** High — incorrect reimbursement amounts on in-network claims; potential overpayment exposure estimated at $12,000 in FS1 test data over 3-day regression window.

**Root Cause (Preliminary):** FeeScheduleLookupService.cls line 47 — SOQL query missing WHERE Fee_Schedule__c.Id = :pnaFeeScheduleId filter clause. Query relies solely on date range and CPT code match.

## Acceptance Criteria

As per Description.

Additional validation:
- After fix, regression test suite (FeeScheduleLookupServiceTest.cls) must pass all 24 existing test methods plus 3 new overlap-scenario test methods.
- Claim_Line__c.Rate_Source__c must correctly reflect 'PNA' and display the Fee_Schedule__c.Name used for the rate.
- No Claim_Line__c records with Adjudication_Status__c = 'Adjudicated' and incorrect Allowed_Amount__c should exist in FS1 post-fix deployment (data fix script required).

## Solution

Fix FeeScheduleLookupService.cls line 47 to add explicit filter: WHERE Id = :linkedFeeScheduleId in the SOQL for Fee_Schedule_Line__c lookup, where linkedFeeScheduleId is resolved from the PNA.Fee_Schedule__c lookup field. Add a guard clause to throw a custom FeeScheduleAmbiguityException if more than one rate row is returned for the same CPT code on the same fee schedule (should be impossible post data-model constraint but defensive coding required). Add a unique compound index (via unique External ID workaround) on Fee_Schedule_Line__c combining Fee_Schedule__c + CPT_Code__c to prevent duplicate entries at the data layer. Write data fix Apex script to re-adjudicate affected Claim_Line__c records in FS1.
