# INS-0405: BUG: Fee_Schedule__c Rate_Amount__c Incorrectly Populated as $0.00 When PNA Has Multiple LOB Assignments

- **Resolution**: Unresolved
- **Assignee**: Kevin Nguyen
- **Reporter**: Rachel Torres

- **Epic**: Provider Network Contracting

- **Labels**: Phase1, BugFix, FeeSchedule, DataIntegrity

## Description

**Bug Summary**: When a Provider_Network_Agreement__c record has more than one Line_of_Business__c (LOB) junction record (e.g., both 'Commercial' and 'Medicare Advantage'), the Fee_Schedule__c.Rate_Amount__c field is saved as $0.00 on all Fee_Schedule__c child records created via the 'Add Fee Schedule' OmniScript, regardless of the rate entered by the user.

**Environment**: Acme Insurance Full Sandbox (CS-47)
**Build**: Sprint 5 Day 3 deployment (changeset AcmeIns_S5_D3)
**Severity**: High — blocks Provider Network Specialists from loading contracted rates ahead of go-live UAT

**Steps to Reproduce**:
1. Navigate to any Provider_Network_Agreement__c record that has 2 or more PNA_LOB_Assignment__c child records (LOB values: 'Commercial' AND 'Medicare Advantage').
2. Click the 'Add Fee Schedule' action button, which launches OmniScript OS_AddFeeSchedule.
3. In Step 2 of the OmniScript, enter a CPT Code (e.g., '99213'), an Effective Date, an Expiration Date, and a Rate Amount (e.g., $125.00).
4. Click 'Save Fee Schedule' on the final OmniScript step.
5. Navigate to the Fee_Schedule__c related list on the PNA record.
6. Open the newly created Fee_Schedule__c record.

**Expected Result**: Fee_Schedule__c.Rate_Amount__c = $125.00 (as entered in the OmniScript).

**Actual Result**: Fee_Schedule__c.Rate_Amount__c = $0.00. All other fields (CPT_Code__c, Effective_Date__c, Expiration_Date__c, Provider_Network_Agreement__c lookup) are correctly populated.

**Additional Observations**:
- Bug does NOT reproduce when the PNA has exactly ONE PNA_LOB_Assignment__c record.
- Checked DataRaptor Load debug logs: the Rate_Amount field value is correctly passed into the DR input JSON as '125.00' (string). The $0.00 value appears to be written at the DML layer.
- Suspected cause: DataRaptor Load DR_SaveFeeSchedule may have a duplicate mapping row for Rate_Amount__c — one mapping correctly references the OmniScript input node, and a second (likely a stale copy-paste row) maps Rate_Amount__c to a null LOB aggregate field that evaluates to 0 when multiple LOB records exist. The second mapping appears to overwrite the first.
- No Apex trigger on Fee_Schedule__c currently deployed that would reset the field post-insert.

## Acceptance Criteria

As per Description.

Additional verification steps post-fix:
1. Reproduce original scenario (PNA with 2 LOBs, enter $125.00) — confirm Fee_Schedule__c.Rate_Amount__c = $125.00 after save.
2. Verify single-LOB PNA scenario still saves Rate_Amount__c correctly (regression).
3. Verify PNA with 3 LOBs (Commercial, Medicare Advantage, Medicaid) also saves correctly.
4. Confirm no other Fee_Schedule__c currency fields (e.g., Fee_Schedule__c.Override_Rate__c, Fee_Schedule__c.Blended_Rate__c) are affected by the same duplicate mapping pattern.

## Solution

**Root Cause Investigation (in progress)**:
1. Open DataRaptor Load DR_SaveFeeSchedule in OmniStudio DataRaptor designer.
2. Navigate to the 'Output' tab and filter mappings for target field 'Rate_Amount__c'.
3. Identify and delete the duplicate/stale mapping row that sources from the LOB aggregate node (suspected node path: `PNALOBList.AggregateRate` or similar) — this node resolves to null/0 when iterated over multiple LOB records and is overwriting the correct input mapping.
4. Retain only the mapping: Input Path = `FeeScheduleStep.RateAmountInput` → Output Object = `Fee_Schedule__c` → Output Field = `Rate_Amount__c`, with correct data type set to 'Decimal'.
5. Verify DataRaptor Preview mode with a mock payload containing 2 LOB entries returns correct Rate_Amount.
6. Deploy fix via OmniStudio export/import to sandbox and re-run reproduction steps.
7. If duplicate mapping removal is insufficient, add a debug step in the Integration Procedure IP_SaveFeeSchedule to log the resolved Rate_Amount value immediately before the DataRaptor Load call to confirm value integrity at each orchestration layer.
8. Post-fix: add a Validation Rule on Fee_Schedule__c: `AND(Rate_Amount__c = 0, NOT(ISPICKVAL(Rate_Type__c, 'No Charge')))` to block future $0.00 saves as a safety net, with error message 'Rate Amount cannot be $0.00 unless Rate Type is No Charge. Please review the fee schedule entry.'
