## Description

**Bug Report**

**Summary:** The Fee Schedule lookup introduced in INS-0358 is incorrectly returning expired Fee_Schedule__c records when a Claim__c.Date_of_Service__c is set to a future date (e.g., pre-authorized elective procedures).

**Steps to Reproduce:**
1. Create a Fee_Schedule__c record with Effective_Date__c = 01/01/2024, Expiration_Date__c = 12/31/2024, CPT_Code__c = '99213', Allowed_Amount__c = 125.00.
2. Create a second Fee_Schedule__c record with Effective_Date__c = 01/01/2025, Expiration_Date__c = 12/31/2025, CPT_Code__c = '99213', Allowed_Amount__c = 132.00.
3. Create a Claim__c with Date_of_Service__c = 03/15/2025 and a child Claim_Line__c with CPT_Code__c = '99213'.
4. Observe Claim_Line__c.Contracted_Rate__c after save.

**Expected Result:** Claim_Line__c.Contracted_Rate__c = 132.00 (2025 rate applies).

**Actual Result:** Claim_Line__c.Contracted_Rate__c = 125.00 (2024 expired rate is returned).

**Root Cause (Preliminary):** The SOQL filter in `FeeScheduleLookupService` uses `Effective_Date__c <= :today` instead of `Effective_Date__c <= :dateOfService`, causing the query to anchor on the current system date rather than the claim's date of service.

**Environment:** Full Sandbox (Acme-FS1)
**Severity:** High – financial impact on pre-auth claims

## Acceptance Criteria

as per Description

## Solution

Fix the SOQL bind variable in `FeeScheduleLookupService.resolveContractedRate()`: replace `Effective_Date__c <= :today AND Expiration_Date__c >= :today` with `Effective_Date__c <= :dateOfService AND Expiration_Date__c >= :dateOfService` where `dateOfService` is extracted from `claimLine.Claim__r.Date_of_Service__c`. Add a unit test in `FeeScheduleLookupServiceTest` covering the future date of service scenario with assertions on Claim_Line__c.Contracted_Rate__c. Deploy fix to Full Sandbox for regression testing before promoting to UAT. Update the Apex test class to achieve >= 90% coverage on the modified method.
