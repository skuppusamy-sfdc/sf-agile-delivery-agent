## Description

**Bug Report — FeeScheduleRateLookupService returning expired rates**

**Steps to Reproduce:**
1. Navigate to an existing Claim__c record with Date_of_Service__c = TODAY().
2. Add a Claim_Line__c with CPT_Code__c = '99213' and a Provider__c that has two Fee_Schedule__c records: one expired (Rate_Expiration_Date__c = 30 days ago) and one active (Rate_Effective_Date__c = 60 days ago, Rate_Expiration_Date__c = 1 year from today).
3. Save the Claim_Line__c record.
4. Observe Claim_Line__c.Allowed_Amount__c.

**Expected Behavior:**
Claim_Line__c.Allowed_Amount__c is populated using the currently active Fee_Schedule__c rate ($145.00).

**Actual Behavior:**
Claim_Line__c.Allowed_Amount__c is populated using the expired Fee_Schedule__c rate ($132.00), resulting in underpayment.

**Root Cause (Preliminary):**
The SOQL query in FeeScheduleRateLookupService.cls is ordering by CreatedDate ASC instead of Rate_Effective_Date__c DESC, causing the oldest (expired) record to be selected when multiple fee schedule versions exist for the same CPT code.

**Environment:** Acme Insurance - Full Sandbox (FS1)
**Severity:** High — affects active adjudication pipeline

## Acceptance Criteria

As per Description. Verify fix by confirming Claim_Line__c.Allowed_Amount__c = $145.00 when two Fee_Schedule__c versions exist and Date_of_Service__c falls within the active version's effective date range. Regression test with single fee schedule version to confirm no regression.

## Solution

Fix the SOQL ORDER BY clause in FeeScheduleRateLookupService.cls from 'ORDER BY CreatedDate ASC LIMIT 1' to 'ORDER BY Rate_Effective_Date__c DESC LIMIT 1'. Add a WHERE clause condition: Rate_Expiration_Date__c >= :serviceDate OR Rate_Expiration_Date__c = null to handle open-ended fee schedules. Add unit test method testExpiredVsActiveRateSelection() covering the multi-version scenario. Deploy to FS1 for QA validation, then promote to UAT.
