## Description

As a Provider Network Specialist, I need the Fee Schedule Engine to automatically resolve allowable reimbursement rates by CPT code and effective date when a Provider_Network_Agreement__c is active, so that Claim_Line__c adjudication uses the correct contracted rate rather than defaulting to billed charges or requiring manual rate entry.

Background: Currently, claims processors are manually cross-referencing PDF fee schedules attached to PNA records to determine the contracted rate for each CPT code on a Claim_Line__c. This is error-prone and adds 2-3 minutes of handle time per claim line. This story builds the core rate lookup capability on Fee_Schedule__c and wires it into the Claim_Line__c adjudication path.

Extends INS-0387 (Provider_Network_Agreement__c contract lifecycle automation).

## Acceptance Criteria

Given a Fee_Schedule__c record associated to an active Provider_Network_Agreement__c,
When a Claim_Line__c is created or updated with a CPT_Code__c value and a Date_of_Service__c,
Then the system queries Fee_Schedule__c.Rate_Amount__c where Fee_Schedule__c.CPT_Code__c = Claim_Line__c.CPT_Code__c AND Fee_Schedule__c.Effective_Date__c <= Claim_Line__c.Date_of_Service__c AND Fee_Schedule__c.Expiration_Date__c >= Claim_Line__c.Date_of_Service__c AND Fee_Schedule__c.Provider_Network_Agreement__c = the resolved PNA for the servicing provider.

Given a matched Fee_Schedule__c rate record exists,
When the rate lookup succeeds,
Then Claim_Line__c.Allowed_Amount__c is populated with Fee_Schedule__c.Rate_Amount__c and Claim_Line__c.Rate_Source__c is set to 'Fee Schedule' and Claim_Line__c.Fee_Schedule__c (lookup) is stamped with the matched record Id.

Given a CPT code exists on the Claim_Line__c but NO matching Fee_Schedule__c record is found for the effective date range,
When the lookup returns zero results,
Then Claim_Line__c.Rate_Source__c is set to 'No Contract Rate Found', Claim_Line__c.Allowed_Amount__c is left null, and a Task is created and assigned to the owning Provider Network Specialist queue with Subject = 'Missing Fee Schedule Rate — CPT [code] for PNA [PNA Number]'.

Given a Provider_Network_Agreement__c.Status__c = 'Terminated' or 'Expired',
When the rate lookup is triggered for a Claim_Line__c with Date_of_Service__c falling outside the PNA active window,
Then the system must NOT apply the contracted rate; instead Claim_Line__c.Rate_Source__c = 'PNA Inactive — Manual Review' and a Case of RecordType 'Provider Contract Dispute' is auto-created linked to the Claim__c.

Given multiple Fee_Schedule__c records match the same CPT code and date range (e.g., overlapping amendments),
When conflict is detected (record count > 1),
Then the system selects the record with the most recent Fee_Schedule__c.Amendment_Version__c (highest integer) and logs a warning on Claim__c.Processing_Notes__c.

-Given a modifier code exists on Claim_Line__c.Modifier__c, the system shall apply a modifier multiplier stored on Fee_Schedule__c.Modifier_Multiplier__c- (DEPRECATED — modifier logic deferred to INS-0421; modifier field should remain on object but not drive rate calculation in this story)

Given a successful rate resolution,
When Claim_Line__c.Allowed_Amount__c is set,
Then Claim__c.Total_Allowed_Amount__c (roll-up summary or triggered recalculation) is updated to reflect the sum of all child Claim_Line__c.Allowed_Amount__c values.

## Solution

1. **Fee_Schedule__c Schema**: Confirm or add fields: CPT_Code__c (Text), Rate_Amount__c (Currency), Effective_Date__c (Date), Expiration_Date__c (Date), Amendment_Version__c (Number), Provider_Network_Agreement__c (Lookup), Rate_Type__c (Picklist: 'Per Unit','Per Diem','Case Rate'), Modifier_Multiplier__c (reserved, not active this sprint).

2. **Apex Service Class — FeeScheduleResolverService**: Implement `resolveRate(Id claimLineId)` method. Executes a SOQL query on Fee_Schedule__c with compound WHERE clause on CPT code, date range, and PNA. Returns a FeeScheduleResult wrapper. Handles tie-breaking by Amendment_Version__c DESC LIMIT 1 after conflict detection.

3. **Record-Triggered Flow on Claim_Line__c** (After Insert / After Update, optimized run): Calls the Apex Action wrapping FeeScheduleResolverService. Passes Claim_Line__c.CPT_Code__c, Date_of_Service__c, and resolved PNA Id (sourced from parent Claim__c → Provider__c → active PNA via custom metadata or relationship query). Updates Claim_Line__c fields via Flow Update Records element.

4. **Task / Case Auto-Creation**: Handled inside the Apex service using Database.insert with a SObjectUnitOfWork pattern (fflib) to batch DML. Task assignment uses a Custom Metadata record mapping LOB to Queue Id to avoid hardcoded Ids.

5. **Claim__c Total Recalculation**: Record-Triggered Flow on Claim_Line__c (After Insert/Update/Delete) invokes an Apex-calculated rollup (due to cross-object complexity) updating Claim__c.Total_Allowed_Amount__c.

6. **Unit Tests**: Minimum 90% coverage on FeeScheduleResolverService. Test scenarios: happy path, no match, expired PNA, multi-version conflict.

7. **Permission Set Group**: FeeSchedule_ReadWrite PSG grants CRUD on Fee_Schedule__c to Provider Network Specialist and Claims Processor profiles.
