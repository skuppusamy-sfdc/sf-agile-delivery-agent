# INS-0344: Implement Fee Schedule versioning and effective-date-based rate lookup for Claim Line adjudication

## Description

As a Claims Operations Manager, I need the Fee Schedule Engine to support versioned Fee_Schedule__c records with effective and expiration dates so that Claim_Line__c adjudication always applies the rate that was in force on the Date of Service (DOS), preventing retroactive rate misapplication and resulting overpayments or underpayments.

Extends INS-0343 — relies on Fee_Schedule__c being linked to an active PNA before rate lookup is invoked.

## Acceptance Criteria

Given multiple Fee_Schedule__c records exist for the same Provider__c and CPT code (e.g., 99213) with overlapping version history,
When a Claim_Line__c is created with Service_Date__c = '2025-03-15' and CPT_Code__c = '99213',
Then the adjudication engine queries Fee_Schedule__c where Effective_Date__c <= 2025-03-15 AND (Expiration_Date__c >= 2025-03-15 OR Expiration_Date__c = null) and returns exactly one active rate record; if multiple are found, the record with the latest Effective_Date__c is selected.

Given a Fee_Schedule__c record is updated with a new Expiration_Date__c that retroactively closes a rate period,
When the update is saved,
Then a Validation Rule prevents Expiration_Date__c from being set earlier than any Claim_Line__c.Service_Date__c already adjudicated under that schedule, and an error message 'Cannot expire a fee schedule with adjudicated claims dated within the active period' is displayed.

Given a CPT code on a Claim_Line__c has no matching Fee_Schedule__c rate for the DOS,
When adjudication is attempted,
Then Claim_Line__c.Adjudication_Status__c is set to 'Pended - Missing Rate', Claim__c.Status__c rolls up to 'Pended', and a Task is created assigned to the Provider Network Specialist queue with Subject = 'Missing Fee Schedule Rate: [CPT Code]'.

Given a Fee_Schedule__c version is created with Version_Number__c auto-incremented,
When the record is saved,
Then the previous version's Expiration_Date__c is automatically set to (new Effective_Date__c - 1 day) via a Record-Triggered Flow, ensuring no gaps or overlaps in the rate timeline.

Given a Claims Operations Manager views a Claim_Line__c record,
When they inspect the Fee Schedule panel,
Then they see a read-only FlexCard displaying Fee_Schedule__c.Version_Number__c, Rate_Amount__c, Effective_Date__c, and Expiration_Date__c that was applied at adjudication time, sourced from Claim_Line__c.Applied_Fee_Schedule_ID__c (lookup field).

## Solution

1. Apex Service Class 'FeeScheduleLookupService' with method getRateForDOS(claimLineId) — performs SOQL with date-range filter and version tiebreaker ORDER BY Effective_Date__c DESC LIMIT 1.
2. Record-Triggered Flow on Fee_Schedule__c (before insert) auto-increments Version_Number__c using a custom metadata counter and sets previous version Expiration_Date__c.
3. Validation Rule on Fee_Schedule__c using SOQL-based formula cross-referencing Claim_Line__c adjudicated dates (implemented as Apex-based Custom Validation via trigger due to SOQL in VR limitation).
4. Apex Trigger on Claim_Line__c (after insert/update) invokes FeeScheduleLookupService and stamps Applied_Fee_Schedule_ID__c.
5. FlexCard 'Applied_Fee_Schedule_FC' on Claim_Line__c record page surfacing version metadata via DataRaptor Extract.
6. Custom Metadata Type 'Fee_Schedule_Config__mdt' stores adjudication behavior flags (e.g., allow_null_expiry = true).
