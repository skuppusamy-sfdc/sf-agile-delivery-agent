## Description

**Bug Report — INS-0357**

**Environment:** Acme Insurance Full Sandbox (FS1)
**Severity:** High — Causes incorrect Claim_Line__c.Allowed_Amount__c values, directly impacting payment accuracy
**Related Story:** INS-0351 (Fee Schedule Engine implementation)

**Steps to Reproduce:**
1. Navigate to a Fee_Schedule__c record that has two or more Fee_Schedule_Line__c child records sharing the same CPT_Code__c value (e.g., CPT 99213) but with different Place_of_Service__c values (e.g., '11 - Office' and '22 - Outpatient Hospital').
2. Create a Claim__c record in 'In Adjudication' status linked to a provider with an active PNA referencing the above Fee_Schedule__c.
3. Create a Claim_Line__c with CPT_Code__c = '99213' and Place_of_Service__c = '22 - Outpatient Hospital'.
4. Observe Claim_Line__c.Allowed_Amount__c after the Fee Schedule Engine processes the record.

**Expected Result:** Claim_Line__c.Allowed_Amount__c should reflect the rate from the Fee_Schedule_Line__c record where Place_of_Service__c = '22 - Outpatient Hospital'.

**Actual Result:** Claim_Line__c.Allowed_Amount__c is populated with the rate from the first Fee_Schedule_Line__c record returned by the SOQL query (CPT 99213, Place_of_Service__c = '11 - Office'), which is incorrect. The SOQL in FeeScheduleResolutionService is missing the Place_of_Service__c filter predicate — the query filters on CPT_Code__c only.

**Root Cause (Identified):** In FeeScheduleResolutionService.cls, line 47, the SOQL query reads:
`SELECT Id, Reimbursement_Rate__c FROM Fee_Schedule_Line__c WHERE Fee_Schedule__c = :feeScheduleId AND CPT_Code__c = :cptCode LIMIT 1`
The `AND Place_of_Service__c = :placeOfService` predicate is absent.

**Impact Assessment:** Any Claim_Line__c adjudicated against a fee schedule with multi-POS CPT entries since INS-0351 deployment is potentially mispriced. Claims team to run data audit query provided in comments.

## Acceptance Criteria

As per Description. Post-fix, the SOQL query in FeeScheduleResolutionService must include AND Place_of_Service__c = :placeOfService in the WHERE clause. Unit tests must assert correct rate selection when two Fee_Schedule_Line__c records share a CPT code but differ on Place_of_Service__c. A data remediation script must re-process all affected Claim_Line__c records where Claim__c.Status__c != 'Paid' and Claim_Line__c.Adjudication_Date__c >= Sprint 5 deployment date.

## Solution

Update FeeScheduleResolutionService.cls line 47 to add Place_of_Service__c filter: `SELECT Id, Reimbursement_Rate__c FROM Fee_Schedule_Line__c WHERE Fee_Schedule__c = :feeScheduleId AND CPT_Code__c = :cptCode AND Place_of_Service__c = :placeOfService LIMIT 1`. Add a new @isTest method in FeeScheduleResolutionServiceTest asserting multi-POS disambiguation. Deploy fix via changeset to FS1, validate unit tests pass at 90%+ coverage. Execute Anonymous Apex remediation script to re-invoke FeeScheduleResolutionService for all open Claim_Line__c records with Adjudication_Date__c >= 2024-11-01. Coordinate with Claims Operations Manager to hold payment batch until remediation is confirmed complete.
