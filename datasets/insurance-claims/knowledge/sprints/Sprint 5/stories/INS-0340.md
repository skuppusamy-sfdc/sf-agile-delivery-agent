## Description

**Bug Summary:** The Fee_Schedule__c effective date overlap validation rule is not triggering when a user clones an existing Fee_Schedule__c record, allowing duplicate date-range entries for the same CPT code and PNA combination to be saved. This causes the rate resolution logic (INS-0335) to apply an unpredictable rate (whichever record has the most recent CreatedDate) rather than the correct contracted rate.

**Steps to Reproduce:**
1. Navigate to any active Fee_Schedule__c record with CPT_Code__c = '99213' linked to a PNA.
2. Click the standard Salesforce 'Clone' button.
3. Do NOT change Effective_Start_Date__c or Effective_End_Date__c.
4. Click Save.
5. Observe: Record saves successfully with no validation error.

**Expected Behavior:** Validation rule 'No_Overlapping_Fee_Schedule_Dates' fires and prevents save with message: 'A Fee Schedule entry for this CPT code already exists within the specified date range for this PNA.'

**Actual Behavior:** Record saves without triggering the validation rule. A duplicate Fee_Schedule__c record is created with identical PNA__c, CPT_Code__c, Effective_Start_Date__c, and Effective_End_Date__c values.

**Environment:** Acme Insurance - Full Sandbox (ACM-FS1)
**Regression:** Introduced in Sprint 4 deployment (INS-0305 validation rule change)
**Severity:** High — directly impacts claims adjudication rate accuracy.

## Acceptance Criteria

As per Description.

Additional verification: After fix, cloning a Fee_Schedule__c record with overlapping dates must throw the validation error in both UI (Clone button) and API insert contexts. Bulk data load via DataLoader must also be blocked by the validation.

## Solution

Root cause: The existing validation rule uses `ISNEW()` in its criteria, which evaluates to TRUE on a Clone operation (since Clone creates a net-new record), but the SOQL query within the validation rule formula is not executing correctly due to a formula character limit workaround that was introduced in Sprint 4. Fix: Replace the formula-based validation rule with an Apex Trigger `FeeScheduleOverlapValidator` on before-insert and before-update that performs a SOQL query for overlapping records: SELECT Id FROM Fee_Schedule__c WHERE PNA__c = :newRecord.PNA__c AND CPT_Code__c = :newRecord.CPT_Code__c AND Effective_Start_Date__c <= :newRecord.Effective_End_Date__c AND Effective_End_Date__c >= :newRecord.Effective_Start_Date__c AND Id != :newRecord.Id. If results returned, call addError() on the record. Add test class with 95%+ coverage including clone simulation, bulk insert of 200 records, and API context test.
