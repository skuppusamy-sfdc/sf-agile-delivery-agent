## Description

As a Claims Operations Manager, I need a fee schedule rate lookup engine that automatically determines the allowed amount on each Claim_Line__c based on the provider's active PNA and associated Fee_Schedule__c so that claims are adjudicated consistently and without manual rate entry. The engine must resolve CPT code, modifier, place of service, and effective date to return the correct contracted rate. Extends INS-0289.

## Acceptance Criteria

Given a Claim_Line__c record is created with a valid CPT code (Claim_Line__c.CPT_Code__c) and an associated Provider__c with an active PNA,
When the adjudication engine runs,
Then Claim_Line__c.Allowed_Amount__c is populated from the matching Fee_Schedule__c rate row where CPT, modifier, and place of service match.

Given a CPT code exists in the Fee_Schedule__c with a modifier,
When Claim_Line__c.Modifier__c is blank,
Then the engine applies the base CPT rate without modifier adjustment.

Given multiple Fee_Schedule__c versions exist for a provider,
When the Claim__c.Date_of_Service__c falls within a specific Fee_Schedule__c.Rate_Effective_Date__c and Fee_Schedule__c.Rate_Expiration_Date__c,
Then only the rate-effective version is used for adjudication.

Given a CPT code is not found in the Fee_Schedule__c,
When the lookup engine executes,
Then Claim_Line__c.Adjudication_Status__c is set to 'Pended - Missing Rate' and a task is created for the Claims Operations Manager.

Given the provider is out-of-network (no active PNA),
When the rate lookup runs,
Then Claim_Line__c.Allowed_Amount__c is calculated using the default UCR (Usual, Customary, and Reasonable) rate from the Fee_Schedule__c record where Fee_Schedule__c.Type__c = 'UCR'.

-Given ICD-10 diagnosis codes drive rate selection, the engine cross-references ICD-10 to CPT mapping table-
Note: ICD-10 to CPT mapping deferred to Sprint 7.

## Solution

Implement an Apex service class (FeeScheduleRateLookupService.cls) with a static method invoked from a Record-Triggered Flow on Claim_Line__c (before save). The service queries Fee_Schedule_Line__c child records filtered by CPT_Code__c, Modifier__c, Place_of_Service__c, and effective date range. Results are cached using Platform Cache (Org partition) keyed by Fee_Schedule__c Id to reduce SOQL calls during bulk adjudication. An OmniStudio DataRaptor Extract is used for the rate lookup UI component on the Claim_Line__c FlexCard. Bulk adjudication scenarios handled via Queueable Apex with chunking at 200 records. Unit tests achieve >85% coverage.
