## Description

**Bug Report**

**Summary:** The Fee_Schedule__c validation rule 'Effective_Date_Cannot_Be_Past' is incorrectly triggering a save error when a read-only user (Provider Relations Specialist profile) simply opens and views an existing Fee_Schedule__c record — even though no edits are being made.

**Steps to Reproduce:**
1. Log in as a user with the 'Provider_Relations_Specialist' profile (read-only access to Fee_Schedule__c).
2. Navigate to any existing Fee_Schedule__c record where Effective_Date__c is in the past (e.g., Effective_Date__c = 2024-01-01).
3. Observe the record detail page — no edits made.
4. A validation error banner appears: 'Effective Date cannot be in the past.'

**Expected Behavior:** Validation rules should only fire on record save/edit operations. A read-only view of a record with a past Effective_Date__c should display without error.

**Actual Behavior:** The validation error banner renders on page load for records with Effective_Date__c < TODAY, even with no user-initiated save action.

**Environment:** Partial Sandbox (PS1)
**Affected Profiles:** Provider_Relations_Specialist, Claims_Operations_Manager (read-only on Fee_Schedule__c)
**Regression:** Introduced after INS-0366 validation rule deployment on 2025-06-10.

## Acceptance Criteria

As per Description. Validation rule 'Effective_Date_Cannot_Be_Past' must only evaluate during insert/update DML operations and must not surface on record view for historical records.

## Solution

Root cause: the validation rule formula does not account for the ISNEW() / ISCHANGED() context, causing it to evaluate as true on any record load where the date is past. Fix: update the validation rule formula to wrap the condition with OR(ISNEW(), ISCHANGED(Effective_Date__c)) so the rule only fires when the field is being set or changed. Additionally, add a PRIORVALUE guard: AND(Effective_Date__c < TODAY(), OR(ISNEW(), ISCHANGED(Effective_Date__c))). Deploy fix to PS1 for regression testing before promoting to PS2 and Production.
