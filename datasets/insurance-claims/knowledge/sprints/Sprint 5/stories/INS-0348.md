## Description

As a Credentialing Specialist, I need automated NPI validation against the NPPES (National Plan and Provider Enumeration System) registry and TIN format validation during Provider__c record creation and update so that invalid provider identifiers are caught at data entry, preventing downstream claim rejections and ERA matching failures.

The NPPES integration uses the public CMS NPI Registry API (https://npiregistry.cms.hhs.gov/api). TIN validation is a format check (9-digit EIN pattern) with a future hook for IRS TIN Matching (Phase 2 scope).

## Acceptance Criteria

Given a Credentialing Specialist enters an NPI__c value on a new Provider__c record,
When the record is saved,
Then an Apex callout to the NPPES API is made with the NPI as the query parameter; if the API returns a valid result, Provider__c.NPI_Verified__c is set to true and Provider__c.NPI_Verification_Date__c is stamped with today's date.

Given the NPPES API returns no results for the entered NPI,
When the save is attempted,
Then a save-blocking error is displayed: 'NPI [value] not found in NPPES registry. Please verify and re-enter.' and the record is not committed.

Given the NPPES API returns a result where the provider's taxonomy code does not match Provider__c.Primary_Specialty__c,
When the record is saved,
Then a non-blocking warning toast is shown: 'NPPES taxonomy mismatch — please confirm specialty' and Provider__c.NPPES_Taxonomy_Match__c is set to false for Credentialing Specialist review.

Given a TIN__c value is entered on a Provider__c record,
When the record is saved,
Then a Validation Rule enforces the pattern XX-XXXXXXX (2 digits, hyphen, 7 digits); invalid formats display 'TIN must be in format XX-XXXXXXX'.

Given the NPPES API is unavailable (timeout or HTTP 5xx),
When a Provider__c save is attempted,
Then the record saves successfully with Provider__c.NPI_Verified__c = false and Provider__c.NPI_Verification_Pending__c = true, and a retry Platform Event (NPI_Validation_Retry__e) is published for async re-validation within 1 hour.

-Given a bulk data import of providers via Data Loader, NPI validation fires synchronously for each record.- (Superseded — bulk import uses async validation batch; see DataIntegration team backlog)

## Solution

1. Apex Class 'NPPESValidationService' with @future(callout=true) method validateNPI(providerId, npi) — calls NPPES REST API via Named Credential 'NPPES_CMS_NC', parses JSON response, updates Provider__c fields.
2. Record-Triggered Flow on Provider__c (after insert/update, NPI__c changed) calls Invocable Apex 'InvocableNPIValidation' which enqueues Queueable Apex for callout (to avoid mixed DML).
3. Validation Rule on Provider__c for TIN format: NOT(REGEX(TIN__c, '[0-9]{2}-[0-9]{7}')).
4. Platform Event NPI_Validation_Retry__e with fields Provider_Id__c (text), NPI__c (text); a Scheduled-Triggered Flow processes retry events hourly.
5. Custom Metadata Type 'NPPES_Config__mdt' stores API endpoint, timeout (ms), and retry max attempts for environment-specific configuration.
6. Permission Set 'NPI_Validation_Override_PS' allows Credentialing Specialists to manually set NPI_Verified__c = true with a mandatory override reason field.
