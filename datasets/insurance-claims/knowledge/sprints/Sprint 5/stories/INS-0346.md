## Description

**Bug Summary:** The Fee Schedule Engine is returning a stale (prior-version) rate for CPT code 99214 after a mid-month Fee_Schedule__c update, resulting in Claim_Line__c underpayments of approximately $18.50 per line.

**Environment:** Acme Insurance Full Sandbox (AcmeFSSB1)

**Steps to Reproduce:**
1. Navigate to Fee_Schedule__c record for Provider NPI 1234567890, CPT 99214.
2. Update Rate_Amount__c from $85.00 to $103.50 with Effective_Date__c = today's date and save.
3. Create a new Claim__c with Service_Date__c = today and one Claim_Line__c with CPT_Code__c = '99214'.
4. Trigger adjudication by setting Claim__c.Status__c = 'Ready for Adjudication'.
5. Observe Claim_Line__c.Allowed_Amount__c.

**Expected Result:** Claim_Line__c.Allowed_Amount__c = $103.50 (new rate).

**Actual Result:** Claim_Line__c.Allowed_Amount__c = $85.00 (prior rate). The stale rate persists for approximately 10–15 minutes post-update before resolving, suggesting a platform cache or static variable holding the previous rate lookup result.

**Impact:** Claims Operations team has identified 47 affected Claim_Line__c records in the past billing cycle. Total underpayment exposure: ~$869.50. Downstream Payment__c and EOB records are also incorrect.

**Root Cause Hypothesis:** FeeScheduleLookupService Apex class uses a static Map variable to cache rate results within a transaction context; suspect the cache is being populated by a scheduled batch job that runs every 15 minutes and does not invalidate on Fee_Schedule__c updates.

## Acceptance Criteria

As per Description. Additionally: after fix deployment, re-run adjudication on the 47 identified Claim_Line__c records and confirm Allowed_Amount__c = $103.50 for all lines with Service_Date__c on or after the rate Effective_Date__c.

## Solution

1. Investigate FeeScheduleLookupService static Map cache — clear static cache variable on Fee_Schedule__c update via a Record-Triggered Flow that calls an @InvocableMethod to invalidate cache, or remove static caching entirely in favor of per-transaction SOQL with selectivity optimization.
2. Add Platform Cache (Org Cache) partition 'FeeScheduleCache' with TTL = 0 (no cache) for rate lookups, replacing static variable approach.
3. Create a compensating batch job 'BatchReadjudicateStaleRates' implementing Database.Batchable to re-process the 47 identified Claim_Line__c records by re-invoking FeeScheduleLookupService.
4. Add unit test in FeeScheduleLookupServiceTest asserting that a rate update followed immediately by a new claim line returns the updated rate within the same transaction.
