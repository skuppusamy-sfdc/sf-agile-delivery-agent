## Description

As a Provider Network Specialist, I need all legacy Provider_Network_Agreement__c records (approximately 14,200 records) migrated from the legacy contract management system (Conga CLM export) into Salesforce with full referential integrity to Provider__c, Facility__c, and Fee_Schedule__c objects so that the network contracting team operates exclusively from Salesforce starting in Sprint 14. Extends INS-1081.

## Acceptance Criteria

Given the source data extract from Conga CLM is available as a validated CSV,
When the migration script executes via Salesforce Data Loader in upsert mode using External_ID__c as the match key,
Then all 14,200 PNA records are loaded with zero data loss and error rate below 0.5%.

Given a migrated PNA record is created,
When the record is inspected post-load,
Then Provider_Network_Agreement__c.Provider__c correctly references the matching Provider__c record by NPI (NPI__c external ID), Provider_Network_Agreement__c.Facility__c references Facility__c by TIN (TIN__c external ID), and Provider_Network_Agreement__c.LOB__c is mapped per the agreed translation table.

Given PNA records with legacy Status values of 'TERM', 'EXP', or 'INACT',
When the transformation rules are applied,
Then Provider_Network_Agreement__c.Status__c is set to 'Terminated', 'Expired', or 'Inactive' respectively per the mapping specification.

Given the migration is complete,
When a reconciliation report is run,
Then a row-count comparison between the source system and Salesforce shows ≤ 0.5% variance and all records with Status__c = 'Active' have a non-null Provider_Network_Agreement__c.Effective_Date__c.

-All PNA documents will be migrated as Salesforce Files in the same load- (deprecated: document migration deferred to Sprint 14 due to storage quota concerns)

Given migration validation passes,
When the cutover is approved by the Provider Network Specialist,
Then legacy system write access is disabled and Salesforce becomes the system of record.

## Solution

Multi-phase migration approach: Phase 1 uses a Python transformation script to clean and map legacy CSV fields to Salesforce API names, applying status translation logic and NPI/TIN-based foreign key resolution. Phase 2 uses Salesforce Data Loader CLI in upsert mode with External_ID__c as the external key, executed in batches of 200. A custom Migration_Log__c object captures per-record success/failure with error messages. Post-load validation uses a SOQL-based Apex script comparing record counts and spot-checking 500 random records. Validation results surface in a custom Migration Dashboard LWC built on top of Migration_Log__c.
