## Description

**Bug Report — PCN Generation Skipped on Bulk PNA Updates**

**Steps to Reproduce:**
1. Prepare a Data Loader CSV with 50+ Provider_Network_Agreement__c records updating PNA__c.Effective_Date__c to a future date.
2. Execute bulk upsert via Salesforce Data Loader (API mode).
3. Navigate to any updated PNA__c record and verify Provider_Change_Notification__c child records.
4. Observe that no Provider_Change_Notification__c records have been created for any of the updated PNA records.

**Expected Behavior:** For every PNA__c where Effective_Date__c is changed, a Provider_Change_Notification__c record should be auto-generated with PCN__c.Change_Type__c = 'Effective Date Update' and PCN__c.Provider__c linked to the parent provider, as defined in the 'PNAEffectiveDateChange_RTF' Record-Triggered Flow.

**Actual Behavior:** PCN records are generated correctly when PNA__c.Effective_Date__c is updated via the UI (single record edit) but are completely skipped during bulk API upsert operations.

**Root Cause Hypothesis:** The Record-Triggered Flow 'PNAEffectiveDateChange_RTF' is configured with 'Run Asynchronously' in the after-save path, which may be getting suppressed due to the bulk API context exceeding the async Apex queue depth, or the flow entry criteria are not evaluating correctly when ISCHANGED(Effective_Date__c) is assessed in bulk context.

**Environment:** Acme Insurance Full Sandbox (FS1) and Partial Sandbox (PS2)
**Frequency:** 100% reproducible with batch size > 10 records via API
**Impact:** Medium — PCN audit trail is incomplete; compliance reporting gap for network change notifications.

## Acceptance Criteria

As per Description.

Given 50 PNA__c records are updated via Data Loader bulk upsert with changed Effective_Date__c values,
When the bulk operation completes,
Then exactly 50 Provider_Change_Notification__c records must be created, each with PCN__c.Change_Type__c = 'Effective Date Update' and PCN__c.PNA__c linked correctly.

Given the fix is deployed,
When a single-record UI update to PNA__c.Effective_Date__c is performed,
Then PCN generation behavior is unchanged from pre-fix behavior (regression check).

## Solution

Investigate 'PNAEffectiveDateChange_RTF' async path behavior under bulk API context. Convert the PCN creation logic from a Record-Triggered Flow asynchronous path to a synchronous after-save path, or migrate the logic to an Apex Trigger 'PNATrigger' with a handler class 'PNATriggerHandler' that uses a static set to prevent recursion. Use Trigger.isInsert / Trigger.isUpdate context checks and ISCHANGED equivalent via comparing Trigger.new vs Trigger.oldMap for Effective_Date__c. Ensure PCN DML is bulkified (list-based insert). Add test class 'PNATriggerHandlerTest' with bulk test method inserting 200 records. Coordinate with DataIntegration team to confirm Data Loader API version compatibility.
