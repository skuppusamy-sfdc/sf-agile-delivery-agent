## Description

As an Underwriting Analyst, I need historical Payment__c records migrated from the legacy TPA system (covering claim payments from 2021–2023) to be validated, transformed, and loaded into Salesforce with full referential integrity to Claim__c and Provider__c records, so that claims history is available for actuarial reporting and COB lookups without dual-system queries. This story covers the validation framework and initial load of 250,000 payment records.

## Acceptance Criteria

Given the legacy TPA payment extract file (CSV, ~250K rows) is staged in the Salesforce Files / MuleSoft SFTP landing zone
When the DataMigration_PaymentValidation batch job runs
Then each row is validated for: non-null Payment__c.Claim__c lookup (matched on Legacy_Claim_ID__c), non-null Payment__c.Provider__c lookup (matched on NPI__c), Payment_Amount__c > 0, Payment_Date__c BETWEEN 2021-01-01 AND 2023-12-31, and Payment_Method__c IN ('EFT', 'Check', 'ERA')

Given the validation batch completes
When validation errors are found
Then failed rows are written to Migration_Error_Log__c with fields: Source_Record_ID__c, Error_Type__c, Error_Message__c, and Migration_Batch_ID__c; a summary report is emailed to the data migration team

Given all validation checks pass for a row
When the load batch executes
Then Payment__c records are inserted with Migration_Source__c = 'Legacy TPA', Migration_Batch_ID__c populated, and Payment_Status__c = 'Historical'

Given the load batch completes
When total inserted records are confirmed
Then a reconciliation report is generated comparing source row count vs. Salesforce insert count vs. error count, with variance <= 0.5% acceptable threshold

Given historical Payment__c records are loaded
When a Claim__c record is opened that has historical payments
Then the Payment__c related list displays all records including those with Migration_Source__c = 'Legacy TPA', sorted by Payment_Date__c DESC

-Given ERA file attachments from the legacy system are migrated as ContentDocuments linked to Payment__c-
(ERA document migration deferred to post-go-live Phase 3 — storage cost review pending)

Given the full load completes
When the Claims Operations Manager runs the 'Historical Payment Summary' report
Then report shows total Payment_Amount__c by LOB__c, Payment_Year__c, and Payment_Method__c with no null Claim__c or Provider__c lookups in the migrated dataset

## Solution

Build a two-phase Apex batch architecture: Phase 1 — DataMigration_PaymentValidationBatch.cls reads staged records from a Migration_Staging__c custom object (pre-populated via MuleSoft CSV ingestion), applies validation logic, and writes errors to Migration_Error_Log__c. Phase 2 — DataMigration_PaymentLoadBatch.cls processes only validated staging records, constructs Payment__c sObjects with all required fields, and uses Database.insert() with allOrNone=false for partial success handling. Claim__c and Provider__c lookups resolved using External_ID fields (Legacy_Claim_ID__c and NPI__c respectively) for upsert key matching. Batch job chaining implemented via Database.executeBatch() in the finish() method. Reconciliation report built as a Salesforce Report on Migration_Batch_Summary__c object. All migration activity gated behind 'Data_Migration_Admin' Permission Set Group.
