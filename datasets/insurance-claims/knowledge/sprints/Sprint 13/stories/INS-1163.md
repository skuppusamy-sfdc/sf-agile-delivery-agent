## Description

**Bug Report — COB Claim Premature Closure**

**Steps to Reproduce:**
1. Create a Claim__c record with Claim__c.COB_Indicator__c = TRUE and Claim__c.Secondary_Payer__c populated.
2. Post a primary Payment__c record with Payment__c.Remittance_Status__c = 'Posted' and Payment__c.Payer_Type__c = 'Primary'.
3. Observe Claim__c.Status__c after the ERA reconciliation Record-Triggered Flow executes.

**Expected Behavior:**
When COB_Indicator__c = TRUE, Claim__c.Status__c should transition to 'Pending COB — Secondary' and must NOT advance to 'Closed' until a secondary Payment__c with Payer_Type__c = 'Secondary' is also posted.

**Actual Behavior:**
Claim__c.Status__c transitions directly to 'Closed' after primary Payment__c posting, bypassing the COB secondary adjudication check. This causes EOB__c records to be generated with incorrect EOB__c.Patient_Responsibility__c values and prevents the secondary payer from submitting an 837 crossover claim.

**Environment:** Production sandbox (Full Copy — Sprint 13 refresh)
**Frequency:** 100% reproducible when COB_Indicator__c = TRUE
**Impact:** ~340 claims affected in UAT data set; financial exposure estimated at $220K in incorrect patient responsibility calculations.

## Acceptance Criteria

As per Description.

Given Claim__c.COB_Indicator__c = TRUE and a primary Payment__c with Payer_Type__c = 'Primary' is posted,
When the Record-Triggered Flow on Payment__c evaluates the parent Claim__c,
Then Claim__c.Status__c must be set to 'Pending COB — Secondary' and must NOT be set to 'Closed'.

Given Claim__c.Status__c = 'Pending COB — Secondary',
When a secondary Payment__c with Payment__c.Payer_Type__c = 'Secondary' and Payment__c.Remittance_Status__c = 'Posted' is created,
Then Claim__c.Status__c must transition to 'Adjudicated' and subsequently to 'Closed' only after EOB__c.Patient_Responsibility__c is recalculated correctly.

## Solution

Root cause identified in the Record-Triggered Flow 'Claim_Status_Transition_Flow' — the decision node evaluating COB_Indicator__c was using a stale formula field reference resolved at flow save time rather than at runtime. Fix: Add an explicit Get Records element to re-query Claim__c.COB_Indicator__c within the flow execution context before the status transition decision. Add a new flow path for COB = TRUE that sets Status__c = 'Pending COB — Secondary'. Add a separate entry criteria condition on the flow to handle secondary Payment__c posting. Update the flow version, deactivate v1.4, activate v1.5. Add a Apex test class assertion for COB scenario with 95%+ coverage. Backfill affected Claim__c records via Data Loader using exported UAT exception report.
