## Description

As a Credentialing Specialist, I need historical credentialing records migrated from the legacy CAQH flat-file extract (CSV format, ~42,000 provider records) into Salesforce Credentialing_Case__c objects so that the credentialing team can manage all provider credential lifecycle activities natively in Salesforce from go-live.

The migration must preserve the full credentialing history per provider (initial credentialing, re-credentialing cycles) and correctly link each Credentialing_Case__c to its parent Provider__c (Person Account) via NPI matching.

## Acceptance Criteria

Given the legacy CAQH extract CSV is loaded into the Salesforce migration staging org (Data_Migration_Staging__c custom object),
When the migration Apex batch (CredentialingMigrationBatch) executes,
Then each row must produce one Credentialing_Case__c record with the following field mappings:
  - CAQH_Provider_ID → Credentialing_Case__c.CAQH_Profile_ID__c
  - NPI → used to look up and link Provider__c via Provider__c.NPI__c
  - Credential_Type → Credentialing_Case__c.Credential_Type__c
  - Initial_Approval_Date → Credentialing_Case__c.Approval_Date__c
  - Expiration_Date → Credentialing_Case__c.Expiration_Date__c
  - Credentialing_Status → Credentialing_Case__c.Case_Status__c (mapped via Custom Metadata CAQH_Status_Map__mdt)

Given a CAQH record's NPI cannot be matched to an existing Provider__c.NPI__c,
When the batch processes the record,
Then the record must be written to a Migration_Exception__c staging record with Migration_Exception__c.Exception_Type__c = 'NPI Match Failure' and Migration_Exception__c.Source_Row__c populated with the raw CSV row data.

Given a provider has multiple credentialing cycles in the CAQH extract (identified by same NPI, different credential cycle dates),
When the batch processes these records,
Then each cycle must create a distinct Credentialing_Case__c record; the most recent active record must have Credentialing_Case__c.Is_Primary__c = TRUE.

Given the full migration batch completes,
When a post-migration validation report is run,
Then:
  - Total migrated Credentialing_Case__c count must be within 0.5% of source row count (excluding documented exceptions)
  - Zero Credentialing_Case__c records may have a null Provider__c lookup
  - Migration_Exception__c records must not exceed 2% of total source rows

-Given DEA license data is present in the CAQH extract, migrate to Credentialing_Case__c.DEA_Number__c- (deprecated: DEA data migrated separately via INS-1089 DEA license object)

Given migration is complete,
When a Credentialing Specialist opens a Provider__c record,
Then the Related List for Credentialing_Case__c must display all migrated cases sorted by Credentialing_Case__c.Approval_Date__c descending.

## Solution

Build Apex Batch class CredentialingMigrationBatch (implements Database.Batchable, Database.Stateful) processing Data_Migration_Staging__c records in chunks of 200. NPI lookup uses a pre-built Map<String,Id> keyed on Provider__c.NPI__c loaded in the start() method for governor limit efficiency. CAQH status mapping via CAQH_Status_Map__mdt Custom Metadata queried once in start(). Migration exceptions written to Migration_Exception__c using a separate Database.insert with allOrNone=false. Post-migration validation implemented as a scheduled Apex report class (CredentialingMigrationValidator) that queries counts and publishes results to a custom Migration_Dashboard__c record. Use Data Loader for initial CSV staging load. Full migration dry-run in ACME_FS1 before production execution.
