## Description

As a Claims Operations Manager, I need the Salesforce platform to automatically ingest inbound Electronic Remittance Advice (ERA) files from the CMS HIPAA 835 transaction feed and create corresponding Payment__c and Explanation_of_Benefits__c records so that manual ERA reconciliation effort is eliminated and payment posting accuracy improves to 99%+.

Inbound ERA files arrive via a secure SFTP endpoint every business day at 06:00 CST. Each ERA file may contain remittance data for 1 to 5,000 individual Claim_Line__c records.

## Acceptance Criteria

Given an ERA 835 file is deposited on the SFTP endpoint,
When the scheduled MuleSoft integration job polls the endpoint (every 15 minutes on business days),
Then a Platform Event (ERA_Inbound__e) must be published with fields: File_Name__c, Record_Count__c, Source_Payer_ID__c, and raw payload reference.

Given an ERA_Inbound__e Platform Event is received by Salesforce,
When the Apex trigger on ERA_Inbound__e fires,
Then one Payment__c record must be created per unique check/EFT number in the ERA with Payment__c.ERA_Reference__c populated, Payment__c.Payment_Date__c set from the ERA BPR segment, and Payment__c.Amount__c set to the total paid amount.

Given a Payment__c record is created from an ERA,
When Claim_Line__c matching occurs using Claim_Line__c.Claim_Line_Number__c and Claim__c.Claim_Number__c from the ERA CLP segment,
Then each matched Claim_Line__c must have Claim_Line__c.Paid_Amount__c updated, Claim_Line__c.Adjustment_Reason_Code__c populated from the CAS segment, and Claim_Line__c.Status__c set to 'Paid'.

Given a Claim_Line__c in the ERA cannot be matched to an existing Salesforce record,
When the reconciliation Apex batch completes,
Then the unmatched line must be written to an ERA_Exception__c record with ERA_Exception__c.Reason__c = 'Claim Line Not Found' and assigned to the Claims Operations queue.

Given all Claim_Line__c records on a Claim__c are in Status__c = 'Paid',
When the final line is updated,
Then a Record-Triggered Flow must set Claim__c.Status__c = 'Closed - Paid' and generate an Explanation_of_Benefits__c record linked to the Claim__c.

Given an ERA file contains duplicate check numbers already present in Salesforce,
When the Apex ERA processor detects the duplicate via Payment__c.ERA_Reference__c uniqueness check,
Then the duplicate file must be skipped, logged to an Integration_Log__c record with Integration_Log__c.Error_Type__c = 'Duplicate ERA', and a notification email sent to the Claims Ops distribution list.

## Solution

MuleSoft Anypoint integration polls SFTP and publishes ERA_Inbound__e Platform Events to Salesforce. An Apex Trigger on ERA_Inbound__e invokes an async Queueable Apex class (ERAProcessorQueueable) that parses the 835 payload (stored in Salesforce Files / ContentVersion), performs bulk DML for Payment__c and Claim_Line__c updates using Database.update with allOrNone=false for partial success handling. A Record-Triggered Flow on Claim_Line__c handles the rollup status check and EOB generation. Custom metadata (ERA_Config__mdt) stores SFTP credentials reference and batch size thresholds. Deploy via unlocked package.
