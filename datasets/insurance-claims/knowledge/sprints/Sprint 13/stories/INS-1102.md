## Description

**Bug Report – COB Secondary Flag Zeroing Allowed Amount**

**Environment:** Acme Insurance Full Sandbox (ACME_FS1)
**Severity:** High – Financial Impact

**Steps to Reproduce:**
1. Open an existing Claim__c record where Claim__c.COB_Type__c = 'Secondary'.
2. Add a new Claim_Line__c record with a valid CPT code (e.g., CPT 99213) linked to an active Fee_Schedule__c record where Fee_Schedule__c.Allowed_Amount__c = 150.00.
3. Set Claim_Line__c.COB_Secondary__c = TRUE on the Claim_Line__c record.
4. Save the Claim_Line__c record and observe the adjudication calculation.
5. Navigate to the saved Claim_Line__c record.

**Expected Behavior:** Claim_Line__c.Allowed_Amount__c should be populated from the linked Fee_Schedule__c.Allowed_Amount__c (e.g., 150.00) and COB secondary logic should apply the primary payer's paid amount as a credit offset, not zero out the allowed amount.

**Actual Behavior:** Claim_Line__c.Allowed_Amount__c is set to 0.00 and Claim_Line__c.COB_Calculated_Amount__c is also 0.00, resulting in a $0 payment being generated. This causes Payment__c records to be created with Payment__c.Amount__c = 0.00 and EOB records with incorrect patient liability calculations.

**Frequency:** Reproducible 100% of the time when Claim_Line__c.COB_Secondary__c = TRUE.
**Impact:** All secondary COB claims adjudicated since Sprint 11 deploy (est. 340 affected Claim_Line__c records) may have incorrect Payment__c amounts.

## Acceptance Criteria

As per Description.

Given the bug fix is deployed,
When a Claim_Line__c is saved with COB_Secondary__c = TRUE and a valid Fee_Schedule__c lookup,
Then Claim_Line__c.Allowed_Amount__c must equal Fee_Schedule__c.Allowed_Amount__c and Claim_Line__c.COB_Calculated_Amount__c must equal (Allowed_Amount__c minus Primary_Payer_Paid_Amount__c), not 0.00.

Given the fix is in place,
When a data fix script is run against the 340 affected records in production,
Then all impacted Claim_Line__c.Allowed_Amount__c values are recalculated and associated Payment__c.Amount__c records are corrected with a Payment__c.Adjustment_Reason__c = 'COB Recalculation – Bug INS-1102'.

## Solution

Root cause identified in the AdjudicationService Apex class (AdjudicationService.cls, line 247): a conditional branch checking `if (claimLine.COB_Secondary__c == true)` incorrectly reassigns `allowedAmount = 0` before the Fee Schedule lookup executes, due to an ordering bug introduced in the Sprint 11 refactor. Fix involves reordering the Fee Schedule lookup to occur before the COB offset calculation block. A data fix Apex script will be written to recalculate the 340 affected Claim_Line__c records in production via an anonymous Apex execution window after deployment. Unit tests will be updated in AdjudicationServiceTest.cls to cover COB secondary scenarios. Change set deployment to production requires CAB approval given financial impact.
