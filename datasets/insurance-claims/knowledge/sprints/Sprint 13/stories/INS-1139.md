## Description

As a Provider Network Specialist, I need all legacy fee schedule records (sourced from the decommissioned ProviderPro system) to be migrated into Salesforce Fee_Schedule__c objects with CPT code mappings, effective date ranges, and LOB associations so that claims adjudication can reference accurate contracted rates without manual lookups in the legacy system. Extends INS-1106.

## Acceptance Criteria

Given the source migration file contains 142,000 fee schedule rows exported from ProviderPro,
When the Apex Data Loader batch job executes,
Then all rows must be upserted into Fee_Schedule__c using Fee_Schedule__c.Legacy_ID__c as the external ID, with zero duplicate Fee_Schedule__c records allowed post-migration.

Given each source row includes a CPT code,
When the row is loaded,
Then Fee_Schedule__c.CPT_Code__c must be validated against the CPT_Reference__mdt custom metadata table; rows with unrecognized CPT codes must be written to a Migration_Exception__c record with Migration_Exception__c.Reason__c = 'Invalid CPT Code'.

Given Fee_Schedule__c.Effective_End_Date__c is present in the source,
When the record is inserted,
Then Fee_Schedule__c.Status__c must be auto-set to 'Expired' if Effective_End_Date__c < TODAY(), or 'Active' if Effective_End_Date__c >= TODAY().

Given the migration completes,
When a post-load validation report is run,
Then at least 99.5% of source rows must have a corresponding Fee_Schedule__c record, and a Migration_Summary__c record must be created capturing total records attempted, succeeded, and failed.

-Given Fee_Schedule__c.Rate_Override__c is populated, When adjudication runs, Then use Rate_Override__c instead of base rate — DEPRECATED: rate override logic moved to FeeScheduleEngine Apex service per INS-1129-

Given a Fee_Schedule__c record is inserted or updated via migration,
When the record is saved,
Then a Field History Tracking entry must be captured for Fee_Schedule__c.Contracted_Rate__c, Fee_Schedule__c.Effective_Start_Date__c, and Fee_Schedule__c.Status__c.

## Solution

Build a two-phase migration: Phase 1 uses a Python pre-processing script to cleanse and map ProviderPro CSV exports to Salesforce field schema, outputting a validated CSV with Legacy_ID__c populated. Phase 2 uses Salesforce Data Loader CLI in upsert mode with external ID Fee_Schedule__c.Legacy_ID__c. An Apex Trigger 'FeeScheduleStatusTrigger' (before insert, before update) auto-sets Status__c based on date comparison. CPT validation uses a Custom Metadata query in the trigger. Migration_Exception__c and Migration_Summary__c records are inserted via a post-load Apex batch job 'MigrationValidationBatch'. Field History Tracking enabled on Fee_Schedule__c via Setup.
