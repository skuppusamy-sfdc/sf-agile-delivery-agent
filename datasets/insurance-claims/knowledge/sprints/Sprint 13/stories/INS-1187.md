## Description

As a Provider Network Specialist, I need all 47,000+ legacy provider (Person Account) and facility (Business Account) records migrated from the legacy ProviderMaster system into Salesforce with NPI and TIN-based deduplication, NPPES validation, and CAQH profile linkage so that the Salesforce platform becomes the system of record for all provider data ahead of Phase 2 go-live.

## Acceptance Criteria

Given the migration ETL process runs for Provider__c (Person Account) records,
When a legacy provider record is processed,
Then Provider__c.NPI__c must be validated against NPPES API (HTTP 200 with matching provider name) before insert; records failing NPPES validation must be written to Migration_Exception__c with Migration_Exception__c.Reason__c = 'NPPES Validation Failed'.

Given two legacy records share the same Provider__c.NPI__c value,
When deduplication logic runs,
Then only the record with the more recent Last_Updated_Date from the source system must be inserted; the duplicate must be logged in Migration_Exception__c with Reason__c = 'Duplicate NPI'.

Given a Provider__c record is successfully inserted,
When the CAQH linkage step runs,
Then Provider__c.CAQH_Provider_ID__c must be populated via CAQH ProView API lookup by NPI; if no CAQH record found, Provider__c.CAQH_Status__c = 'Not Enrolled'.

Given Facility__c (Business Account) records are migrated,
When a facility record is processed,
Then Facility__c.TIN__c must be unique within the org; duplicate TINs must trigger a merge candidate record in Migration_Merge_Candidate__c for manual review by Provider Network Specialist.

Given all migration batches complete,
When the migration summary report is generated,
Then it must show total records inserted, total exceptions, NPPES validation pass rate (target ≥ 95%), and CAQH linkage rate (target ≥ 80%) exported to a Migration_Run__c summary record.

-Given providers with Provider__c.Credentialing_Status__c = 'Lapsed', exclude from migration- (deprecated — Compliance Officer confirmed lapsed providers must be migrated with status preserved for audit history)

Given the full migration completes,
When post-migration validation runs,
Then a reconciliation count between legacy ProviderMaster total and Salesforce Provider__c + Facility__c total must be within a 0.5% variance threshold.

## Solution

Python-based ETL pipeline orchestrated via MuleSoft Anypoint. Stage 1: Extract from legacy ProviderMaster SQL database into staging S3 bucket. Stage 2: NPPES REST API validation per NPI (batched at 100 req/min to respect rate limits). Stage 3: CAQH ProView API enrichment. Stage 4: Salesforce bulk upsert via Bulk API 2.0 using External ID field Provider__c.NPI__c for providers and Facility__c.TIN__c for facilities. Deduplication via pre-load Python Pandas groupby on NPI/TIN. Migration_Exception__c and Migration_Merge_Candidate__c records created via separate Bulk API insert job. Migration_Run__c summary record auto-created by post-migration Apex batch MigrationReconciliationBatch. All jobs logged in custom Migration_Job_Log__c object.
