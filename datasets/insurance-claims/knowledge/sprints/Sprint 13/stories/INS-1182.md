## Description

**Bug Report — PCN Auto-Creation Failure on Provider Network Termination**

**Environment:** Acme Insurance Full Sandbox (FS1) and Partial Sandbox (PS2)
**Discovered by:** Priya Patel (Provider Relations Specialist) during regression testing for Sprint 13 provider termination stories
**Related Story:** INS-1168 (Network Termination Case automation)

**Steps to Reproduce:**
1. Open an existing Provider__c (Person Account) record where Provider__c.Network_Status__c = 'Active'.
2. Open a linked Network_Termination_Case__c record and set Network_Termination_Case__c.Status__c = 'Approved'.
3. Observe that the Record-Triggered Flow on Network_Termination_Case__c fires and updates Provider__c.Network_Status__c from 'Active' to 'Terminated'.
4. Navigate to the Provider__c record and check for a newly created Provider_Change_Notification__c child record.

**Expected Behavior:**
A Provider_Change_Notification__c record should be automatically created with PCN.Provider__c = Provider__c.Id, PCN.Change_Type__c = 'Network Termination', PCN.Effective_Date__c = Network_Termination_Case__c.Termination_Effective_Date__c, and PCN.Status__c = 'Pending Distribution'. This PCN is required for CMS and downstream MCO notification compliance.

**Actual Behavior:**
No Provider_Change_Notification__c record is created. The Provider__c.Network_Status__c field updates correctly to 'Terminated', but the subsequent PCN creation step in the flow does not execute. No error appears in Debug Logs — the Create Records element appears to be silently skipped.

**Impact:** Compliance breach risk — CMS requires PCN distribution within 30 days of network termination. Currently 7 providers terminated in PS2 testing have no associated PCN records.

## Acceptance Criteria

As per Description.

Additional regression criteria: Verify that PCN creation fires correctly for all three termination paths: (1) Manual status update on Provider__c, (2) Network_Termination_Case__c approval flow, (3) Bulk update via Data Loader. Confirm PCN.Status__c = 'Pending Distribution' and PCN.Effective_Date__c matches Network_Termination_Case__c.Termination_Effective_Date__c in all three paths.

## Solution

Root cause investigation points to a Flow fault path misconfiguration: the 'Create PCN Record' Create Records element in the Network_Termination_Case_Approval_Flow is placed after a Decision element that evaluates Provider__c.Network_Status__c BEFORE the update DML commits, causing the condition to read the old value ('Active') and route to the wrong path. Fix: reorder the Flow to execute the Create Records (PCN) element in the same transaction path as the Provider__c update, using the $Record.Network_Termination_Case__c fields directly rather than re-querying Provider__c. Add a null-check condition on Termination_Effective_Date__c to prevent PCN creation with blank dates. Add Flow fault path error logging to a custom Flow_Error_Log__c object. Write Flow test coverage using Apex FlowInterviewTestHelper for all three termination paths.
