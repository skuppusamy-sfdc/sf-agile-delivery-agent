## Description

**Bug Report — PCN Not Triggered on TIN Change**

**Environment:** Acme Insurance Partial Sandbox (Sprint 13)
**Severity:** Medium — compliance risk; TIN changes on active providers must generate PCN records for CMS reporting

**Steps to Reproduce:**
1. Open an existing Provider__c (Person Account) record where at least one Provider_Network_Agreement__c exists with Status__c = 'Active'.
2. Edit the Provider__c record and change the value of Provider__c.TIN__c to any new valid TIN (format: XX-XXXXXXX).
3. Save the record.
4. Navigate to the Provider_Change_Notification__c related list on the Provider__c record.
5. Observe: No PCN record is created.

**Expected Behavior:** When Provider__c.TIN__c is updated and the provider has at least one active PNA (Provider_Network_Agreement__c.Status__c = 'Active'), a Provider_Change_Notification__c record must be auto-created with:
  - PCN__c.Change_Type__c = 'TIN Update'
  - PCN__c.Previous_Value__c = old TIN
  - PCN__c.New_Value__c = new TIN
  - PCN__c.Provider__c = Provider__c.Id
  - PCN__c.Status__c = 'Pending Review'
  - PCN__c.Notification_Date__c = today

**Actual Behavior:** No Provider_Change_Notification__c record is created. The TIN updates silently with no audit trail or compliance notification.

**Root Cause (Preliminary):** The Record-Triggered Flow 'Provider_Data_Change_Notification_Flow' has an entry condition checking ISCHANGED({!$Record.NPI__c}) but does not include an ISCHANGED condition for {!$Record.TIN__c}. TIN change detection was omitted during the original flow build (Sprint 9).

**Regression Risk:** Other demographic fields (Provider__c.Primary_Specialty__c, Provider__c.License_Number__c) should also be evaluated for PCN trigger coverage.

## Acceptance Criteria

As per Description.

Additional regression check: Verify that changes to Provider__c.NPI__c, Provider__c.Primary_Specialty__c, and Provider__c.License_Number__c each independently trigger a Provider_Change_Notification__c record with the appropriate PCN__c.Change_Type__c value ('NPI Update', 'Specialty Update', 'License Update' respectively) when an active PNA exists.

## Solution

Edit the 'Provider_Data_Change_Notification_Flow' Record-Triggered Flow entry conditions to add ISCHANGED({!$Record.TIN__c}) using an OR condition alongside the existing NPI change check. Within the flow, add a new Decision branch for TIN changes that creates a Provider_Change_Notification__c record via a Create Records element with the field mappings specified in the bug description. Also add equivalent Decision branches for Primary_Specialty__c and License_Number__c changes. Store Change_Type__c values in a Custom Metadata type (PCN_Change_Type__mdt) keyed by field API name to avoid hardcoding. Conduct regression testing across all four field change scenarios in Partial Sandbox before promoting to Production.
