## Description

**Bug Report**

**Summary:** The FeeScheduleEngine returns the wrong contracted rate for Claim_Line__c adjudication when two Fee_Schedule__c records for the same CPT code, TIN, and LOB have overlapping effective date ranges. The system picks the oldest record instead of the most recently effective one, causing underpayment or overpayment on adjudicated claims.

**Steps to Reproduce:**
1. Create two Fee_Schedule__c records for CPT code 99213, TIN 123456789, LOB = 'Commercial':
   - Record A: Fee_Schedule__c.Effective_Date__c = 2024-01-01, Fee_Schedule__c.End_Date__c = 2025-06-30, Contracted_Rate__c = $85.00
   - Record B: Fee_Schedule__c.Effective_Date__c = 2025-01-01, Fee_Schedule__c.End_Date__c = 2025-12-31, Contracted_Rate__c = $92.50
2. Submit a Claim_Line__c with Service_Date__c = 2025-03-15, CPT_Code__c = '99213', Provider_TIN__c = '123456789'.
3. Observe Claim_Line__c.Contracted_Rate__c after adjudication.

**Expected Result:** Claim_Line__c.Contracted_Rate__c = $92.50 (Record B is the most recently effective record covering the service date).

**Actual Result:** Claim_Line__c.Contracted_Rate__c = $85.00 (Record A is returned — SOQL ORDER BY is missing, returning the first inserted record).

**Environment:** Acme Insurance Full Sandbox (CS47)
**Frequency:** Reproducible 100% when overlapping fee schedule records exist.
**Impact:** 340 Claim_Line__c records in Sprint 12 UAT incorrectly adjudicated; financial exposure estimated at $14,200.

## Acceptance Criteria

As per Description.

Additionally: the SOQL query in FeeScheduleLookupService.cls must include ORDER BY Fee_Schedule__c.Effective_Date__c DESC LIMIT 1 to ensure the most recently effective non-expired record is returned for any given CPT + TIN + LOB + Service Date combination.

## Solution

Fix FeeScheduleLookupService.cls SOQL query to add ORDER BY Fee_Schedule__c.Effective_Date__c DESC LIMIT 1. Add a validation rule on Fee_Schedule__c to warn (not block) when a new record's effective date range overlaps an existing active record for the same CPT + TIN + LOB combination. Add Apex unit test covering the overlap scenario with assertion on correct rate selection. Re-adjudicate the 340 impacted Claim_Line__c records via a one-time Data Loader update in Full Sandbox before promoting the fix to Production.
