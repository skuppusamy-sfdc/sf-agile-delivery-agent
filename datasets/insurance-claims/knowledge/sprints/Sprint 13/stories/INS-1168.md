## Description

As a Claims Operations Manager, I need inbound 835 ERA files from the CMS integration endpoint to be automatically parsed and reconciled against adjudicated Claim__c and Claim_Line__c records, with Payment__c records created and EOB__c records generated, so that manual payment posting is eliminated and remittance turnaround meets our 24-hour SLA.

## Acceptance Criteria

Given a valid 835 ERA file is received on the CMS MFT endpoint,
When the scheduled Apex batch job (ERA_Ingestion_Batch) runs every 4 hours,
Then each payment loop in the file must produce one Payment__c record with:
  - Payment__c.Amount__c populated from CLP03 segment
  - Payment__c.Check_EFT_Number__c from CLP01
  - Payment__c.Payment_Date__c from BPR16
  - Payment__c.Status__c = 'Posted'
  - Payment__c.Claim__c lookup resolved by matching CLP01 to Claim__c.Claim_Number__c

Given a Claim_Line__c exists with Claim_Line__c.CPT_Code__c matching the SVC01 segment,
When the ERA line-level reconciliation runs,
Then Claim_Line__c.Paid_Amount__c is updated and Claim_Line__c.Adjustment_Reason_Code__c is populated from CAS02.

Given all Claim_Line__c records for a Claim__c are reconciled,
When the final line is processed,
Then Claim__c.Status__c must transition to 'Payment Posted' via a Record-Triggered Flow and an Explanation_of_Benefits__c record is auto-generated with EOB__c.Total_Paid__c summed from child Claim_Line__c.Paid_Amount__c values.

Given an ERA file contains a TIN that does not match any active Provider__c.TIN__c,
When the batch processes that payment loop,
Then the Payment__c record is created with Payment__c.Status__c = 'Unmatched' and a Case is auto-created assigned to the Claims Operations queue with Subject = 'ERA TIN Mismatch — [TIN value]'.

Given a duplicate ERA file is received (matching existing Payment__c.Check_EFT_Number__c),
When the batch attempts ingestion,
Then the file is rejected, no duplicate records are created, and an error entry is written to Integration_Log__c with Status__c = 'Duplicate Rejected'.

## Solution

Build ERA_Ingestion_Batch as a Salesforce Apex Batchable class with a Queueable chain for line-level processing. Use a custom metadata type (ERA_Config__mdt) to store CMS endpoint credentials and file path patterns. File retrieval via Named Credential + Apex HttpCallout to the CMS MFT REST wrapper. 835 parsing handled by a dedicated ERA_Parser Apex utility class mapping ISA/GS/ST/CLP/SVC/CAS segments. Payment__c and EOB__c creation in after-insert context. Record-Triggered Flow on Claim__c fires on Status__c change to 'Payment Posted' to publish a Platform Event (Claim_Status_Changed__e) consumed by downstream notification flows. Integration_Log__c records written via a separate logging utility to avoid transaction rollback contamination.
