## Description

As a Credentialing Specialist, I need historical provider credentialing records migrated from the Acme legacy mainframe (COBOL flat files) into Salesforce Credentialing_Case__c objects, enriched with live CAQH ProView and NPPES NPI Registry data, so that the credentialing team operates from a single system of record starting in Sprint 14 and the legacy mainframe can be decommissioned on schedule.

## Acceptance Criteria

Given the legacy mainframe export contains 12,400 provider credentialing records in pipe-delimited flat file format,
When the DataIntegration migration pipeline executes,
Then each record is transformed and loaded as a Credentialing_Case__c with Credentialing_Case__c.Legacy_System_ID__c preserving the source system key for audit traceability.

Given a Credentialing_Case__c is created from the migration,
When the CAQH ProView API callout is made using the provider's CAQH_ID__c,
Then Credentialing_Case__c.CAQH_Profile_Status__c, Credentialing_Case__c.Last_Attestation_Date__c, and Credentialing_Case__c.CAQH_Authorization_On_File__c are populated from the CAQH response.

Given the CAQH API returns a 404 for a provider,
When the enrichment step processes that record,
Then Credentialing_Case__c.CAQH_Profile_Status__c = 'Not Found' and a Migration_Exception__c record is created with the provider NPI and reason for manual follow-up by the Credentialing team.

Given the NPPES NPI Registry is queried for each migrated provider,
When the NPPES response is received,
Then Provider__c.NPI__c is validated against NPPES and Provider__c.NPI_Validation_Status__c is set to 'Verified' or 'Mismatch'; any 'Mismatch' records are flagged on a Credentialing Exception Report.

Given the full migration batch completes,
When the post-migration validation report is generated,
Then record count reconciliation shows ≤ 0.1% variance between source flat file row count and Credentialing_Case__c records created, and zero records with null Credentialing_Case__c.Provider__c lookup.

Given migrated records include credentialing decisions from the last 5 years,
When Credentialing_Case__c.Decision__c = 'Approved',
Then Credentialing_Case__c.Stage__c = 'Credentialed' and Credentialing_Case__c.Credential_Effective_Date__c is mapped from the legacy 'CRED_EFF_DT' field.

## Solution

Three-phase migration: (1) ETL using MuleSoft DataWeave transforms legacy pipe-delimited flat files to Salesforce bulk API upsert payloads, keyed on Legacy_System_ID__c external ID. (2) Post-load Apex Batch (CAQH_Enrichment_Batch) performs sequential CAQH ProView REST API callouts per provider with Named Credential authentication, running at 50 records/batch to respect CAQH rate limits. (3) NPPES_Validation_Batch performs NPI Registry REST callouts and updates Provider__c.NPI_Validation_Status__c. Migration_Exception__c records auto-assigned to Credentialing queue. Full reconciliation report delivered as a Salesforce Report (tabular) exported to SharePoint for sign-off by Credentialing Manager.
