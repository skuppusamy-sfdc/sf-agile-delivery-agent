## Description

As a Claims Operations Manager, I need the system to automatically ingest Electronic Remittance Advice (ERA) files from our clearinghouse integration and reconcile them against open Payment__c records on adjudicated Claim__c objects, so that EOB documents are generated and distributed to policyholders without manual intervention, reducing payment cycle time and eliminating reconciliation errors that currently require analyst rework.

Background: Currently, ERA files arrive via SFTP in 835 EDI format and are manually keyed into Salesforce by claims staff. This process introduces a 2–3 day lag in payment posting and a ~4% error rate in remittance matching. This story implements an end-to-end automated pipeline covering ERA ingestion, Payment__c matching, Claim_Line__c status updates, and Explanation_of_Benefits__c record creation with DocuSign-triggered delivery.

Extends INS-1160 (Payment__c schema extensions for ERA metadata fields).

## Acceptance Criteria

Given an 835 EDI ERA file is deposited to the designated SFTP endpoint by the clearinghouse,
When the scheduled DataRaptor Extract job runs (every 15 minutes during business hours, hourly off-hours),
Then the file is parsed and a Platform Event (ERA_Received__e) is published with fields: ERA_Transaction_Control_Number__c, Payer_ID__c, Total_Claim_Count__c, Total_Payment_Amount__c, and File_Received_Datetime__c.

Given an ERA_Received__e Platform Event is consumed by the ERA Processing Apex handler,
When the handler attempts to match each ERA claim segment (CLP loop) to an existing Claim__c record,
Then matching must succeed on at least two of the following: Claim__c.Payer_Claim_Control_Number__c, Claim__c.Member_ID__c + Claim__c.Date_of_Service__c, or Claim__c.NPI__c + Claim__c.TIN__c, and Claim__c.Status__c must equal 'Adjudicated' for the match to be accepted.

Given a successful Claim__c match is identified,
When the ERA line-item detail (SVC loops) is processed,
Then each corresponding Claim_Line__c record must be updated with: Claim_Line__c.ERA_Paid_Amount__c, Claim_Line__c.Adjustment_Reason_Code__c (CARC), Claim_Line__c.Remark_Code__c (RARC), Claim_Line__c.ERA_Processed_Date__c = TODAY(), and Claim_Line__c.Status__c transitioned from 'Approved' to 'Payment Posted'.

Given all Claim_Line__c records under a Claim__c have Status__c = 'Payment Posted',
When the Record-Triggered Flow on Claim__c detects all child lines are reconciled,
Then Claim__c.Status__c must be updated to 'Closed – Paid', Claim__c.Final_Payment_Date__c must be set to the ERA payment date (BPR segment date), and a Payment__c record must be created with: Payment__c.Amount__c = sum of ERA paid amounts, Payment__c.Payment_Method__c = mapped from BPR02 transaction handling code, Payment__c.Check_EFT_Number__c = BPR09, Payment__c.Remittance_Trace_Number__c = TRN02, and Payment__c.Claim__c = parent Claim__c.Id.

Given a Payment__c record is created with Payment__c.Payment_Method__c = 'EFT',
When the record-triggered flow evaluates the payment method,
Then Payment__c.EFT_Effective_Date__c must be populated from the ERA BPR16 field and Payment__c.Bank_Account_Verified__c must be TRUE before the payment record status is set to 'Disbursed'; if Bank_Account_Verified__c = FALSE, status must be set to 'On Hold – Bank Verification Required' and a Task must be created assigned to the owning Claims Operations queue.

Given a Claim__c.Status__c transitions to 'Closed – Paid',
When the EOB generation process is triggered via Conga CLM template merge,
Then an Explanation_of_Benefits__c record must be created with: EOB__c.Claim__c = Claim__c.Id, EOB__c.Policy__c = Claim__c.Policy__c, EOB__c.EOB_Generated_Date__c = TODAY(), EOB__c.Total_Billed_Amount__c, EOB__c.Total_Allowed_Amount__c, EOB__c.Total_Paid_Amount__c, EOB__c.Member_Responsibility__c, EOB__c.Status__c = 'Generated', and the merged PDF must be attached to the EOB record as a Salesforce File (ContentVersion).

Given an Explanation_of_Benefits__c record with Status__c = 'Generated' and a valid ContentVersion attachment,
When the DocuSign Send flow is invoked (no signature required — informational delivery only),
Then the EOB PDF must be delivered to the email address on the associated Policy__c.Policyholder__c (Person Account — PersonEmail field) within 2 hours, EOB__c.Delivery_Method__c must be set to 'Email', EOB__c.Delivery_Timestamp__c must be populated upon successful send callback, and EOB__c.Status__c must transition to 'Delivered'.

-Given a paper EOB preference is flagged on Policy__c.Communication_Preference__c = 'Mail', the system must enqueue a print job via the legacy print vendor API.- (Deprecated: print vendor API integration deferred to Phase 3 — see INS-1205)

Given an ERA claim segment cannot be matched to any Claim__c record after exhausting all matching strategies,
When the unmatched segment is logged,
Then an ERA_Exception__c record must be created with: ERA_Exception__c.Exception_Type__c = 'Unmatched Claim', ERA_Exception__c.ERA_Control_Number__c, ERA_Exception__c.Raw_Segment_Data__c (truncated to 32,000 chars), ERA_Exception__c.Assigned_To__c = Claims Operations Manager queue, and a high-priority Case must be opened linked to the ERA_Exception__c with Subject = 'ERA Reconciliation Exception – Manual Review Required'.

Given the ERA ingestion pipeline has completed processing a full ERA file,
When the batch summary is compiled,
Then a Reconciliation_Summary__c record must be created capturing: total segments received, total matched, total unmatched, total Payment__c records created, total EOB__c records generated, processing duration in seconds, and pipeline completion status (Success / Partial / Failed); this record must be visible on the Claims Operations Manager home page dashboard via a dedicated FlexCard component.

## Solution

**Architecture Overview:**
End-to-end pipeline implemented across MuleSoft (SFTP polling → Salesforce Platform Event publication), Apex (ERA parsing and matching engine), Record-Triggered Flows (status orchestration), Conga CLM (EOB document generation), and DocuSign (delivery).

**1. ERA Ingestion (DataRaptor + Platform Event):**
- MuleSoft integration (managed separately by integration team) polls SFTP every 15 min and calls a REST endpoint backed by an Apex @RestResource class `ERAIngestionController` that parses the 835 EDI payload using a custom `EDI835Parser` utility class.
- Upon successful parse, `ERAIngestionController` publishes `ERA_Received__e` Platform Events for each functional group.

**2. ERA Matching Engine (Apex):**
- `ERAReconciliationHandler` subscribes to `ERA_Received__e` via a Platform Event-triggered Apex handler.
- Implements a three-tier matching strategy with configurable priority via Custom Metadata Type `ERA_Matching_Strategy__mdt`.
- Bulkified to process up to 200 CLP loops per transaction; overflow queued via Queueable Apex chain (`ERAReconciliationQueueable`).
- Matching results persisted to `ERA_Match_Result__c` junction object for auditability.

**3. Claim & Claim Line Updates (Record-Triggered Flow + Apex):**
- Apex DML updates `Claim_Line__c` records in bulk within the handler.
- Record-Triggered Flow on `Claim__c` (After Update, optimized with entry conditions: Status__c Changed) invokes a sub-flow `SF_ClaimAllLinesPosted_Check` that queries child `Claim_Line__c` records via a Get Records element and evaluates aggregate status; triggers Payment__c creation via an Apex Action `CreatePaymentFromERA` to enforce field-level logic cleanly outside flow DML limits.

**4. EOB Generation (Conga CLM):**
- Record-Triggered Flow on `Payment__c` (After Insert, Status__c = 'Disbursed') calls Conga CLM via an Invocable Apex wrapper `CongaEOBMergeAction`, passing Claim__c.Id and template ID stored in Custom Setting `EOB_Config__c.Conga_Template_Id__c`.
- Merged PDF returned as base64 and inserted as `ContentVersion` linked to `Explanation_of_Benefits__c`.

**5. DocuSign Delivery (Flow + DocuSign API):**
- Record-Triggered Flow on `Explanation_of_Benefits__c` (After Insert, Status__c = 'Generated') calls Invocable Action `DocuSignSendInfoOnly` using the DocuSign for Salesforce managed package send envelope API.
- Callback handled by a `@future` callout in `DocuSignCallbackHandler` updating EOB__c delivery fields.

**6. Exception Handling:**
- Unmatched segments processed in `ERAExceptionService` Apex class; creates `ERA_Exception__c` and associated Case via a single DML transaction with partial-save error handling.

**7. Reconciliation Dashboard:**
- `Reconciliation_Summary__c` created by `ERAReconciliationSummaryBuilder` Apex class at pipeline completion.
- FlexCard `ERA_ReconciliationSummary_FC` built in OmniStudio, surfaced on Claims Operations Manager App Home page via Lightning App Builder.

**Permission Model:**
- New Permission Set `ERA_Pipeline_Integration_PS` grants CRUD on `ERA_Exception__c`, `ERA_Match_Result__c`, `Reconciliation_Summary__c`; added to `Claims_Operations_PSG` Permission Set Group.
- Field-Level Security on `Payment__c.Check_EFT_Number__c` and `Payment__c.Remittance_Trace_Number__c` restricted to `ERA_Pipeline_Integration_PS` and `Claims_Manager_PS` only.

**Testing:**
- Apex test class `ERAReconciliationHandlerTest` targeting ≥85% coverage with mock EDI payloads covering matched, unmatched, partial-match, and malformed file scenarios.
- UAT scripts provided for Claims Operations Manager role in sandbox org `ACM-STG2`.
