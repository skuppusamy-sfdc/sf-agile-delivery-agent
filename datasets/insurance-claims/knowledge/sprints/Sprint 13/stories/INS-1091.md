## Description

**Bug Report — INS-1091**

**Summary:** When a Fee_Schedule__c has two records for the same CPT_Code__c with overlapping Effective_Date__c ranges, the FeeScheduleResolutionService returns the lower allowed amount instead of the most recently effective record, resulting in underpayment on Claim_Line__c.

**Steps to Reproduce:**
1. Create two Fee_Schedule__c records for CPT Code '99213': Record A with Effective_Date__c = 2024-01-01 and Allowed_Amount__c = 85.00; Record B with Effective_Date__c = 2024-06-01 and Allowed_Amount__c = 92.50. Both records have Termination_Date__c = NULL.
2. Create a Claim__c with Service_Date__c = 2024-08-15.
3. Add a Claim_Line__c with CPT_Code__c = '99213'.
4. Save the Claim_Line__c and observe Allowed_Amount__c on the record.

**Expected Result:** Claim_Line__c.Allowed_Amount__c = 92.50 (most recently effective Fee Schedule record applies)

**Actual Result:** Claim_Line__c.Allowed_Amount__c = 85.00 (earliest record is selected due to missing ORDER BY in SOQL query)

**Impact:** Claims Operations Manager (Rachel Torres) reports 47 claims in UAT have underpaid allowed amounts. Potential compliance exposure if released to production.

**Root Cause (preliminary):** FeeScheduleResolutionService.cls SOQL query missing 'ORDER BY Effective_Date__c DESC LIMIT 1' — returns first record inserted rather than most recent.

## Acceptance Criteria

As per Description. Fix verified when Claim_Line__c.Allowed_Amount__c = 92.50 for the reproduction scenario above. Regression test suite must pass for all CPT fee schedule resolution scenarios in the FeeScheduleResolutionService_Test.cls class.

## Solution

Update FeeScheduleResolutionService.cls SOQL query to include 'ORDER BY Effective_Date__c DESC LIMIT 1' when resolving active Fee_Schedule__c records by CPT_Code__c and Service_Date__c. Add a validation rule on Fee_Schedule__c to warn (not block) when a new record is created with an overlapping date range for the same CPT_Code__c and Network__c combination. Update unit test class to add test methods covering overlapping date range scenarios. Deploy via change set after peer code review.
