## Description

**Bug Report**

**Summary:** When a Broker Relations Coordinator clones an existing Group_Service_Agreement__c (GSA) record in Salesforce to create a renewal, the effective date validation rule (GSA_EffectiveDate_Validation) incorrectly fires and blocks the save — even when the cloned record's dates are valid. This forces coordinators to manually re-enter all GSA fields from scratch, causing significant data entry errors and delays in group employer onboarding.

**Steps to Reproduce:**
1. Navigate to any existing Group_Service_Agreement__c record where GSA__c.Status__c = 'Active' and GSA__c.End_Date__c is in the past 30 days (recently expired).
2. Click 'Clone' from the record action menu.
3. On the clone form, update GSA__c.Effective_Date__c to a future date (e.g., 2025-07-01) and GSA__c.End_Date__c to one year later (2026-06-30).
4. Leave all other fields (LOB, Employer_Group__c, TPA__c, contracted tiers) as cloned.
5. Click Save.

**Expected Result:** The cloned GSA record saves successfully with the new effective dates. Validation rule should only fire if End_Date__c < Effective_Date__c or if a duplicate active GSA exists for the same Employer_Group__c + LOB combination.

**Actual Result:** Validation rule error: 'A Group Service Agreement already exists for this Employer Group and LOB within the specified date range.' The rule is comparing the cloned record's dates against the SOURCE record (which has the same RecordId context during clone), not against other existing records.

**Environment:** Acme Insurance Full Sandbox (CS47) and Production (confirmed by Lisa Park, Broker Relations Coordinator)
**Frequency:** 100% reproducible on clone of any recently expired GSA.
**Impact:** 8 group employer renewals blocked in the past 2 weeks; coordinators resorting to full manual re-entry, introducing field-mapping errors on GSA__c.Fee_Schedule__c and GSA__c.TPA__c lookups.

## Acceptance Criteria

As per Description.

Additionally:
- The validation rule logic must be updated to exclude the current record's own ID when checking for duplicate active GSA records (i.e., the SOQL-based duplicate check in the validation formula must include AND Id != Id condition or be refactored to an Apex Trigger with isClone() check).
- After fix, a cloned GSA with non-overlapping future dates must save without error in both Sandbox and Production.
- Existing validation behavior (blocking genuinely duplicate active GSAs for same Employer_Group__c + LOB) must remain intact and be covered by updated Apex unit tests.

## Solution

Refactor GSA_EffectiveDate_Validation from a formula-based validation rule to an Apex Before Insert/Before Update trigger (GSA_Validation_Trigger) on Group_Service_Agreement__c. In the trigger, use SObject.isClone() to detect clone context and skip the duplicate date-range check on insert when the record originates from a clone. For non-clone inserts and updates, perform a SOQL duplicate check excluding the current record ID. Update existing GSA_ValidationTrigger_Test Apex test class to add clone scenario coverage (minimum 85% coverage maintained). Deploy validation rule deactivation and trigger activation in a single changeset to avoid gap window.
