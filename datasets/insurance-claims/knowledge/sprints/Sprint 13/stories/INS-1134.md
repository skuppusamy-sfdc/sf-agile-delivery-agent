## Description

**Bug Summary:** The effective date overlap validation for Provider_Network_Agreement__c (PNA) records is enforced correctly through the UI but is completely bypassed when records are upserted via the Salesforce REST API (e.g., from the Conga CLM integration). This allows duplicate active PNA agreements for the same Provider__c and LOB__c combination to exist simultaneously, causing incorrect fee schedule resolution.

**Steps to Reproduce:**
1. Confirm an active PNA__c exists: Provider__c = 'Dr. John Doe (NPI: 1234567890)', LOB__c = 'Commercial', PNA.Effective_Date__c = '2024-01-01', PNA.Termination_Date__c = '2024-12-31', PNA.Status__c = 'Active'.
2. Using Postman, send a REST API PATCH request to /services/data/v59.0/sobjects/Provider_Network_Agreement__c with the same Provider__c, LOB__c, and an overlapping Effective_Date__c = '2024-06-01'.
3. Observe the response.

**Expected Result:** API returns a 400 error with a descriptive message: 'An active PNA already exists for this provider and LOB within the specified date range.'

**Actual Result:** API returns HTTP 200 and creates a second active PNA record, bypassing the validation rule entirely.

**Root Cause Hypothesis:** The existing overlap check is implemented only as a Lightning validation rule with $User.UITheme context, making it ineffective for API and programmatic DML operations.

**Impact:** 3 duplicate PNA records confirmed in Full Sandbox. Potential production impact if Conga CLM integration is promoted before fix.

## Acceptance Criteria

as per Description

## Solution

Replace or supplement the Lightning validation rule with an Apex Trigger 'PNATrigger' (before insert, before update) that performs a SOQL overlap query against existing Provider_Network_Agreement__c records for the same Provider__c and LOB__c. If an overlap is detected, call trigger.addError() to block the DML regardless of context (UI or API). Ensure the SOQL query uses selective filters on Provider__c (indexed) to avoid full table scans at scale. Add Apex test class 'PNATrigger_Test' with API-context test scenarios using Test.startTest()/stopTest(). Remove the UI-only validation rule to avoid redundancy post-fix.
