## Description

**Bug Report — PNA Termination Date Not Cascading to Fee Schedules**

**Steps to Reproduce:**
1. Open an existing Provider_Network_Agreement__c record with Status__c = 'Active' and one or more linked Fee_Schedule__c records (Fee_Schedule__c.PNA__c = PNA Id).
2. Create a Network_Termination_Case__c record linked to the PNA.
3. Set Network_Termination_Case__c.Status__c = 'Approved' and populate Network_Termination_Case__c.Termination_Date__c = [future date].
4. Close the Network_Termination_Case__c (Status__c = 'Closed').
5. Verify PNA: PNA.Termination_Date__c is correctly stamped (this works as expected).
6. Navigate to linked Fee_Schedule__c records.
7. Observe Fee_Schedule__c.End_Date__c is NULL and Fee_Schedule__c.Status__c remains 'Active'.

**Expected Result:** All Fee_Schedule__c records linked to the terminated PNA should have Fee_Schedule__c.End_Date__c set to the PNA.Termination_Date__c value and Fee_Schedule__c.Status__c updated to 'Terminated'.

**Actual Result:** The Record-Triggered Flow (RTF_PNATerminationCascade) updates PNA fields correctly but does not traverse the PNA-to-Fee_Schedule__c relationship. Fee schedules remain active, causing incorrect payment calculations for post-termination claims.

**Environment:** Full Sandbox (Acme-FS1) and UAT Sandbox (Acme-UAT2)
**Severity:** Medium — financial impact on post-termination claim payments; workaround is manual fee schedule update

## Acceptance Criteria

As per Description.

Given Network_Termination_Case__c.Status__c is set to 'Closed' with a valid Termination_Date__c,
When RTF_PNATerminationCascade completes,
Then ALL Fee_Schedule__c records where Fee_Schedule__c.PNA__c = the terminated PNA's Id must have Fee_Schedule__c.End_Date__c = PNA.Termination_Date__c and Fee_Schedule__c.Status__c = 'Terminated'.

Given a Fee_Schedule__c record is already in Status__c = 'Terminated' or 'Expired',
When the cascade runs,
Then those records must not be updated (no unnecessary DML).

## Solution

Extend RTF_PNATerminationCascade flow with a 'Get Records' element to retrieve all Fee_Schedule__c records where Fee_Schedule__c.PNA__c = triggering PNA Id AND Fee_Schedule__c.Status__c = 'Active'. Add a loop with an 'Update Records' element inside to set End_Date__c and Status__c. If Fee Schedule volume per PNA can exceed 200 records, replace the flow loop with an invocable Apex method (CascadeFeeScheduleTermination) called from the flow to handle bulk DML safely within governor limits. Add test coverage for bulk scenario (200+ fee schedules per PNA).
