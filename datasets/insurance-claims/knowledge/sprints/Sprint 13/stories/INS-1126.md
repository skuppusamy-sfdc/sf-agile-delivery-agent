## Description

As a Claims Operations Manager, I need Agentforce to automatically analyze new FNOL submissions and generate a fraud risk score and triage recommendation so that high-risk claims are immediately flagged for SIU review while low-risk claims are fast-tracked for straight-through processing, improving adjudication cycle time and reducing fraudulent payouts.

This capability leverages the Agentforce Claims AI agent configured with a custom Prompt Template and connected to the Einstein Trust Layer for PII-safe inference.

## Acceptance Criteria

Given a new Claim__c record is created with Claim__c.Status__c = 'FNOL Received',
When the Record-Triggered Flow (RTF_AgentforceClaimsTriage) fires after save,
Then an Agentforce agent action (InvokeClaimsTriageAgent) must be called asynchronously with the following context fields: Claim__c.Claim_Type__c, Claim__c.Loss_Date__c, Claim__c.Reported_Date__c, Claim__c.Total_Incurred__c, Policy__c.Policy_Type__c, Policy__c.Inception_Date__c, and Policyholder loss history count (last 36 months).

Given the Agentforce agent returns a triage response,
When the response is parsed,
Then Claim__c.AI_Fraud_Risk_Score__c (Number, 0–100) must be populated, Claim__c.AI_Triage_Recommendation__c (Text) must be set to one of: 'Straight-Through Process', 'Manual Review', or 'SIU Referral', and Claim__c.AI_Triage_Timestamp__c must be stamped.

Given Claim__c.AI_Triage_Recommendation__c = 'SIU Referral',
When the field is set,
Then a Record-Triggered Flow must automatically create a Case record linked to the Claim__c with Case.RecordType = 'SIU Investigation', Case.Priority = 'High', and assign it to the SIU queue (Queue: SIU_Review_Queue).

Given Claim__c.AI_Triage_Recommendation__c = 'Straight-Through Process',
When the field is set,
Then Claim__c.Status__c must advance to 'In Adjudication' and Claim__c.Assigned_Adjudicator__c must be auto-populated via round-robin assignment using a Custom Permission Set Group 'Claims_Adjudicator'.

Given the Agentforce inference call fails or times out,
When the async action catches the exception,
Then Claim__c.AI_Triage_Status__c must be set to 'AI Triage Failed', a Platform Event (AI_Triage_Error__e) must be published for monitoring, and the claim must default to 'Manual Review' routing.

Given any Agentforce inference involving Claim__c data,
When the prompt is constructed,
Then the Einstein Trust Layer must mask Policyholder PII fields (SSN, DOB, Address) before sending to the LLM, verified by audit log entries in Einstein_Trust_Log__c.

## Solution

Configure Agentforce Agent 'ClaimsTriageAgent' in Salesforce Agent Builder with a custom Prompt Template referencing a grounding DataRaptor that retrieves claim context. Invoke agent via Apex Action from an asynchronous Record-Triggered Flow (scheduled path, 0 minutes after FNOL save) to avoid governor limit issues. Parse agent response JSON in Apex class (AgentforceClaimsTriageHandler) and perform DML updates on Claim__c. SIU Case creation and straight-through routing handled in a separate After-Save Record-Triggered Flow on Claim__c watching AI_Triage_Recommendation__c. Einstein Trust Layer data masking configured via Data Mask Policy in Agentforce setup. Platform Event AI_Triage_Error__e subscribed by a monitoring Flow that creates an Integration_Log__c record.
