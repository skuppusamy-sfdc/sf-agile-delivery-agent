## Description

As a Credentialing Specialist, I need all historical provider credentialing records migrated from the legacy mainframe system (CRED-MAIN v4.2) into Salesforce Credentialing_Case__c objects, with each record validated against live CAQH and NPPES endpoints, so that credentialing history is fully accessible in Salesforce and re-credentialing workflows can be initiated without referencing the decommissioned mainframe.

## Acceptance Criteria

Given the data migration ETL job executes against the legacy CRED-MAIN extract (CSV format, ~48,000 records),
When records are loaded into Salesforce via the migration Apex Batchable,
Then each Credentialing_Case__c must be created with Credentialing_Case__c.Provider__c (matched by NPI), Credentialing_Case__c.Original_Credential_Date__c, Credentialing_Case__c.Expiration_Date__c, Credentialing_Case__c.Specialty_Code__c, and Credentialing_Case__c.Migration_Source__c = 'CRED-MAIN-v4.2'.

Given a migrated Credentialing_Case__c has Credentialing_Case__c.NPI__c populated,
When the post-migration NPPES validation job runs,
Then the NPPES API must be queried for each NPI and Credentialing_Case__c.NPPES_Validated__c must be set to TRUE if the NPI status is 'Active', or FALSE with Credentialing_Case__c.Validation_Error__c populated if inactive or not found.

Given a Credentialing_Case__c.NPPES_Validated__c = TRUE,
When the CAQH ProView validation step executes,
Then Credentialing_Case__c.CAQH_Provider_ID__c must be populated from the CAQH response and Credentialing_Case__c.CAQH_Status__c must reflect the current attestation status ('Complete', 'Incomplete', 'Expired').

Given a migrated record has Credentialing_Case__c.Expiration_Date__c < TODAY() + 90 days,
When the migration validation report is generated,
Then the record must be flagged in Migration_Exception__c with Migration_Exception__c.Exception_Type__c = 'Expiring Credential' and assigned to the Credentialing queue for re-credentialing action.

Given the full migration batch completes,
When the Migration_Summary__c report is generated,
Then it must show total records processed, success count, NPPES validation pass/fail counts, CAQH match rate, and count of records requiring manual review — all reconciled against the source extract row count.

-Given duplicate NPI records from the mainframe are merged automatically during migration,-
-Auto-merge is disabled per data governance policy DG-112; duplicates must be routed to Migration_Exception__c for manual review instead.-

## Solution

Build an Apex Batchable class 'CredentialingMigrationBatch' consuming a CSV file loaded to Salesforce Files (ContentVersion). Use external ID field Provider__c.NPI__c for upsert matching of Provider__c (Person Account) records. Post-load, execute a chained Apex Queueable 'NPPESValidationQueueable' that calls the NPPES NPI Registry REST API (Named Credential: NPPES_Registry) in batches of 50 with 100ms delay between callouts to respect rate limits. Chain a second Queueable 'CAQHValidationQueueable' for CAQH ProView API calls (Named Credential: CAQH_ProView_Prod). Create Migration_Exception__c records via a dedicated Apex helper class. Build a Salesforce Report (Migration_Summary__c custom object) auto-populated by the batch completion handler. All migration activity logged to a custom Migration_Audit_Log__c object for compliance traceability.
