## Description

**Bug Report — Silent FNOL Submission Failure**

**Environment:** Acme Insurance UAT Org (Sandbox)
**Discovered By:** Rachel Torres (QA) during Sprint 6 regression

**Steps to Reproduce:**
1. Log in to the Agent/Broker Portal as a broker user with linked Broker Account.
2. Navigate to the FNOL Submission OmniScript (BrokerFNOLSubmission).
3. In Step 1 'Policy Lookup', search for and select a Policy__c record that has 2 or more active Endorsement__c child records (Endorsement__c.Status__c = 'Active').
4. Complete all required fields in Steps 2 and 3.
5. Click 'Submit' on the confirmation step.

**Expected Result:** A new Claim__c record is created with Status__c = 'FNOL Received' and the FNOL_Submitted__e Platform Event fires. Broker sees success confirmation screen.

**Actual Result:** The OmniScript spinner runs for approximately 8 seconds and then the screen returns to Step 1 with no error message displayed. No Claim__c record is created. No Platform Event is fired. No Custom Log__c error record is written. The issue does NOT reproduce when the selected policy has 0 or 1 active endorsements.

**Root Cause Hypothesis:** The Integration Procedure 'FNOL_Create_IP' DataRaptor Turbo Action may be hitting a SOQL row limit or a duplicate Claim__c check query is returning multiple rows when multiple endorsements exist, causing an unhandled exception that is swallowed before the error toast logic executes.

**Impact:** High — brokers cannot submit FNOLs for complex policies with endorsements, which represent approximately 34% of the active policy portfolio per Underwriting Analyst estimates.

## Acceptance Criteria

As per Description. Fix verified when: FNOL OmniScript successfully creates Claim__c and fires FNOL_Submitted__e Platform Event for a Policy__c with 2, 3, and 5 active Endorsement__c records without error.

## Solution

Investigate Integration Procedure 'FNOL_Create_IP': (1) Review the DataRaptor Extract step that queries Policy__c with child Endorsement__c — add LIMIT 1 or restructure to avoid multi-row return feeding into single-record DML action. (2) Add explicit try/catch error handling in the Apex Action 'CreateFNOLClaim' to ensure all exceptions surface to the OmniScript error handler and write to Custom Log__c regardless of endorsement count. (3) Add debug logging to Integration Procedure in UAT to capture exact failure step. (4) Unit test with @IsTest data sets including 1, 2, 5, and 10 Endorsement__c child records. Fix must not break existing FNOL flow for policies with 0-1 endorsements.
