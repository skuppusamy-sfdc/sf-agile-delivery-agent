## Description

As a Broker Relations Coordinator, I need a guided OmniScript on the Agent/Broker Portal to submit a First Notice of Loss (FNOL) on behalf of a policyholder so that claims are initiated accurately and immediately routed to the Claims Operations team without manual data re-entry.

Extends INS-0454 (Policy Search LWC) — the policy lookup step reuses the policy search component built in that story.

## Acceptance Criteria

Given a broker initiates a new FNOL from the portal,
When the OmniScript Step 1 'Policy Lookup' loads,
Then the broker can search by Policy__c.Policy_Number__c or Policy__c.Insured_Name__c and the matching Policy__c record auto-populates Policy__c.LOB__c, Policy__c.Effective_Date__c, and Policy__c.Expiration_Date__c into read-only display fields.

Given the policy is located and has Status__c = 'Lapsed',
When the broker attempts to proceed to Step 2,
Then an inline error message displays: 'Policy is not currently active. FNOL cannot be submitted for a lapsed policy.' and navigation is blocked.

Given the broker completes all required FNOL fields including Claim__c.Date_of_Loss__c, Claim__c.Loss_Description__c, Claim__c.Loss_Location__c, and Claim__c.Estimated_Loss_Amount__c,
When the broker submits the OmniScript,
Then a new Claim__c record is created with Claim__c.Status__c = 'FNOL Received', Claim__c.Source__c = 'Broker Portal', and Claim__c.Submitted_By__c populated with the broker's User Id.

Given the Claim__c record is successfully created,
When the OmniScript completes,
Then a Platform Event 'FNOL_Submitted__e' is fired with payload fields: Claim_Id__c, Policy_Id__c, LOB__c, and Broker_Account_Id__c, triggering downstream assignment routing.

Given the FNOL submission fails due to a DML exception,
When the error is caught in the Integration Procedure,
Then a user-friendly error toast is displayed and the error details are written to a Custom Log__c record with Log_Type__c = 'FNOL Error'.

Given the broker has partially completed the OmniScript,
When the broker navigates away and returns within the same session,
Then OmniScript resume functionality restores previously entered field values via session storage.

## Solution

Build OmniStudio OmniScript 'BrokerFNOLSubmission' with 4 steps: (1) Policy Lookup using existing LWC from INS-0454 embedded as a custom LWC element; (2) Loss Details — standard OmniScript fields mapped to Claim__c; (3) Document Upload — File Upload element writing to ContentVersion with LinkedEntityId = new Claim__c; (4) Confirmation screen. Backend Integration Procedure 'FNOL_Create_IP' handles Claim__c DML via DataRaptor Turbo Action. Platform Event published via Apex Action called from Integration Procedure. Error handling via try/catch in Apex with Custom Log__c insert. OmniScript save-resume enabled via OmniScript Resume settings.
