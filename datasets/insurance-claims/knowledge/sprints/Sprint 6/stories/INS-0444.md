## Description

**Bug Report**

**Summary:** The effective date overlap validation on Group_Service_Agreement__c (GSA) records is bypassed when a user clones an existing GSA record using the standard Salesforce 'Clone' button. The validation rule 'GSA_EffectiveDateOverlap_VR' — which prevents two active GSA records for the same Group Account from having overlapping Effective_Date__c and Termination_Date__c ranges — does not fire during the clone operation, allowing duplicate overlapping agreements to be saved.

**Steps to Reproduce:**
1. Navigate to any Group_Service_Agreement__c record with GSA.Status__c = 'Active'.
2. Click the standard 'Clone' button from the record detail page.
3. In the cloned record form, do NOT change GSA.Effective_Date__c or GSA.Termination_Date__c.
4. Click Save.
5. Observe: the cloned record saves successfully with identical dates to the original, bypassing 'GSA_EffectiveDateOverlap_VR'.

**Expected Behavior:** The validation rule 'GSA_EffectiveDateOverlap_VR' should fire on clone-save and prevent overlapping date ranges, returning the error: 'An active Group Service Agreement already exists for this Group Account within the specified date range.'

**Actual Behavior:** The cloned GSA record is saved without triggering the validation rule, resulting in two active GSA records with identical date ranges for the same Group Account.

**Root Cause (Suspected):** The validation rule formula references RecordType and may be short-circuiting due to a null RecordTypeId on the cloned record before full page load. Alternatively, the IsClone() formula function may need to be explicitly handled.

**Environment:** Acme Insurance Full Sandbox (ACME_FS1) and Production
**Frequency:** 100% reproducible
**Impact:** Medium — Underwriting Analysts report duplicate GSA records causing incorrect premium calculation in downstream Group/Employer onboarding workflows.

## Acceptance Criteria

As per Description.

Given the fix is deployed,
When a user clones a GSA record and attempts to save with overlapping GSA.Effective_Date__c and GSA.Termination_Date__c for the same GSA.Group_Account__c,
Then the save is blocked and the error message 'An active Group Service Agreement already exists for this Group Account within the specified date range.' is displayed.

Given the fix is deployed,
When a user clones a GSA record and changes the date range to a non-overlapping period before saving,
Then the record saves successfully without error.

## Solution

Investigate validation rule 'GSA_EffectiveDateOverlap_VR' formula for IsClone() or RecordType null handling issues. Primary fix: replace or supplement the declarative validation rule with an Apex Trigger 'GSATrigger' (before insert, before update) that executes overlap detection SOQL: SELECT Id FROM Group_Service_Agreement__c WHERE Group_Account__c = :newGSA.Group_Account__c AND Status__c = 'Active' AND Id != :newGSA.Id AND Effective_Date__c <= :newGSA.Termination_Date__c AND Termination_Date__c >= :newGSA.Effective_Date__c. If results returned, call addError() on the record. Handler class 'GSATriggerHandler' follows trigger framework pattern. Add Apex test class 'GSATriggerHandler_Test' with clone scenario coverage, targeting 95%+ code coverage. Retain original validation rule as secondary guard but update formula to explicitly handle ISCLONE() = TRUE condition.
