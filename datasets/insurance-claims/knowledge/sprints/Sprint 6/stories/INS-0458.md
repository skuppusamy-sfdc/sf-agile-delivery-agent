## Description

**Bug Report — INS-0458**

Related to INS-0454 (Broker FNOL OmniScript).

**Steps to Reproduce:**
1. Log in to the Agent/Broker Portal as a broker user whose Salesforce User.TimeZoneSidKey = 'America/Los_Angeles' (UTC-8).
2. Navigate to 'File a Claim' and select an active Policy__c record.
3. On Step 2 (Loss Details), enter a Date_of_Loss__c value of today's date at 11:00 PM local time (e.g., 2025-07-14T23:00:00-08:00).
4. Click 'Next' to proceed past Step 2 validation.

**Expected Behavior:**
The OmniScript validation should accept the entered date as today (valid) and allow progression to Step 3.

**Actual Behavior:**
The OmniScript validation rejects the date with the error message 'Date of Loss cannot be a future date', because the DataRaptor Transform date comparison uses the Salesforce server timezone (UTC) rather than the user's local timezone, causing the local date (11:00 PM PST) to be interpreted as the following UTC day (07:00 AM UTC next day).

**Impact:** Brokers in Pacific, Mountain, and Central time zones are unable to submit same-day FNOL entries during evening hours. Workaround: none identified; brokers must contact Claims Operations directly.

**Environment:** Production Sandbox (Full), OmniStudio version 248.6.

## Acceptance Criteria

As per Description. Fix verified when a broker in America/Los_Angeles timezone can successfully submit a Date_of_Loss__c of the current local date at 11:59 PM without a future-date validation error. Claim__c.Date_of_Loss__c must be stored in UTC but validated against the submitting user's User.TimeZoneSidKey offset.

## Solution

Update the DataRaptor Transform date validation node in the BrokerFNOL_OS OmniScript to retrieve the current user's timezone offset via an Integration Procedure call (SOQL: SELECT TimeZoneSidKey FROM User WHERE Id = :UserInfo.getUserId()). Convert the server-side 'today' timestamp to the user's local date before comparison using a custom Apex utility method (TimezoneUtil.toLocalDate(DateTime serverNow, String tzSidKey)). Update the OmniScript Step 2 validation formula to reference the timezone-adjusted date. Add a unit test in OmniScriptDateValidationTest covering UTC-8, UTC-5, and UTC+0 edge cases.
