## Description

**Bug Report — Incorrect Fee Schedule Tier on Provider 360 FlexCard**

**Environment:** Acme Insurance UAT Org
**Discovered By:** Anika Gupta during Provider 360 FlexCard UAT (INS-0486)
**Severity:** Medium — incorrect data displayed to Provider Network Specialists during contract review calls

**Steps to Reproduce:**
1. Identify a Provider__c record that has two Provider_Network_Agreement__c records where both have PNA__c.Status__c = 'Active' (this can occur during a contract renewal overlap period where the new PNA is activated before the old one is terminated).
2. Ensure the two PNA records are linked to different Fee_Schedule__c records with different Fee_Schedule__c.Rate_Tier__c values (e.g., PNA-1 → 'Tier 1', PNA-2 → 'Tier 2').
3. Open the Provider__c record page as a Provider Network Specialist.
4. Observe the Provider 360 FlexCard 'ContractingState' section.

**Expected Result:** The FlexCard displays the Fee Schedule tier from the PNA with the most recent PNA__c.Contract_Start_Date__c (i.e., the newest active contract).

**Actual Result:** The FlexCard displays the Fee Schedule tier from the PNA with the oldest PNA__c.Contract_Start_Date__c (i.e., the earliest active contract). This is because the DataRaptor Extract 'DR_GetActivePNA' lacks an ORDER BY clause and returns whichever PNA record Salesforce retrieves first, which in practice is the oldest due to record creation order.

**Frequency:** Reproducible 100% of the time when a provider has ≥ 2 active PNA records.

**Business Impact:** Provider Network Specialists may quote incorrect reimbursement rates to providers during network calls, creating contractual disputes and potential overpayment or underpayment on Claim_Line__c adjudication.

## Acceptance Criteria

As per Description. Fix verified when: Provider 360 FlexCard consistently displays the Fee_Schedule__c.Rate_Tier__c associated with the PNA__c record having the maximum PNA__c.Contract_Start_Date__c when multiple active PNA records exist for the same Provider__c.

## Solution

Update DataRaptor Extract 'DR_GetActivePNA' in OmniStudio: add ORDER BY PNA__c.Contract_Start_Date__c DESC and set LIMIT to 1 on the primary PNA query to ensure only the most recent active PNA record is returned. Validate that the Fee_Schedule__c lookup join still resolves correctly after ordering change. Regression test with Provider__c records having 1, 2, and 3 active PNA records. Also add a validation rule or duplicate-check alert on Provider_Network_Agreement__c to warn users when creating a new Active PNA for a provider that already has an Active PNA, to prevent this data state from occurring in production. Deploy DataRaptor version update via OmniStudio version activation in UAT then Production.
