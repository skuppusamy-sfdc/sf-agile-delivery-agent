## Description

As a Claims Operations Manager, I need an automated inbound ERA (Electronic Remittance Advice) 835 file processing pipeline that parses incoming EDI 835 files from MCO and TPA partners, maps remittance data to Payment__c and Claim_Line__c records, and flags discrepancies for manual review, so that payment reconciliation cycle time is reduced from 5 business days to same-day processing and write-off decisions are data-driven.

## Acceptance Criteria

Given an EDI 835 file is deposited into the designated SFTP inbound folder,
When the scheduled MuleSoft integration job runs (every 4 hours),
Then the file is retrieved, parsed, and a batch of Payment__c staging records (Payment_Staging__c) are created in Salesforce with Payment_Staging__c.ERA_File_Name__c, Payment_Staging__c.Payer_ID__c, Payment_Staging__c.Check_Number__c, Payment_Staging__c.Check_Amount__c, Payment_Staging__c.Processing_Status__c = 'Pending'.

Given Payment_Staging__c records exist with Processing_Status__c = 'Pending',
When the Batch Apex job ERA835ReconciliationBatch.cls runs,
Then each staging record is matched to an existing Payment__c record via Payment__c.Check_Number__c and Payment__c.Payer__c; if a match is found, Payment__c.Reconciliation_Status__c is updated to 'Matched' and Payment__c.ERA_Received_Date__c is stamped.

Given a Payment_Staging__c record cannot be matched to a Payment__c record,
When reconciliation processing completes,
Then Payment_Staging__c.Processing_Status__c = 'Unmatched' is set, a Case is created with RecordType = 'ERA_Discrepancy', Case.Subject = 'Unmatched ERA Payment — {Check_Number}', and the Case is assigned to the 'Claims_Payment_Review' queue.

Given a matched Payment__c record has Payment__c.Amount__c that differs from the ERA remitted amount by more than $0.01,
When the discrepancy is detected,
Then Payment__c.Reconciliation_Status__c = 'Amount Variance', a Payment_Discrepancy__c child record is created capturing Payment_Discrepancy__c.Expected_Amount__c, Payment_Discrepancy__c.Remitted_Amount__c, and Payment_Discrepancy__c.Variance_Amount__c.

Given ERA 835 loop 2110 CLP/SVC segments are present,
When Claim_Line__c matching is performed,
Then each SVC segment is matched to Claim_Line__c via Claim_Line__c.CPT_Code__c and Claim_Line__c.Claim__c, and Claim_Line__c.Remitted_Amount__c and Claim_Line__c.Adjustment_Reason_Code__c are updated from the CAS segment data.

Given the full ERA file is processed,
When all Payment_Staging__c records reach a terminal status ('Matched', 'Unmatched', 'Amount Variance'),
Then a summary Platform Event 'ERA_Processing_Complete__e' is published with payload including file name, total records, matched count, unmatched count, and variance count, triggering a notification email to the Claims Operations Manager.

## Solution

MuleSoft integration parses EDI 835 using MuleSoft EDI connector and POSTs parsed JSON payload to a Salesforce REST API endpoint (ERA835InboundController.cls, @RestResource). Apex controller bulk-inserts Payment_Staging__c records. Batch Apex class ERA835ReconciliationBatch.cls (implements Database.Batchable, Database.Stateful) processes staging records with batch size 200, performs SOQL matching on Payment__c, updates Claim_Line__c records, and creates discrepancy records. Platform Event ERA_Processing_Complete__e published from batch finish() method. A Process Builder (legacy — migrate to Flow in Phase 2) or Record-Triggered Flow on Payment__c handles Reconciliation_Status__c-based notifications. Custom Metadata Type 'ERA_Partner_Config__mdt' stores payer-specific matching rules.
