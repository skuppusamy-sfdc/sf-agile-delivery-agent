## Description

As a Broker Relations Coordinator, I need a guided OmniScript on the Agent/Broker Portal that allows me to submit a First Notice of Loss (FNOL) on behalf of a policyholder, so that claims intake is accelerated and manual data entry errors in Claim__c creation are reduced. Extends INS-0399 (FNOL baseline Claim__c creation flow).

## Acceptance Criteria

Given a broker initiates FNOL from the portal,
When Step 1 of the OmniScript loads,
Then the broker must be able to search and select an existing Policy__c by Policy__c.Policy_Number__c or policyholder name, and the form must auto-populate Policy__c.LOB__c, Policy__c.Effective_Date__c, and Policy__c.Expiration_Date__c as read-only fields.

Given the policy is selected and is within its effective date range,
When the broker proceeds to Step 2 (Incident Details),
Then required fields must include: Claim__c.Date_of_Loss__c, Claim__c.Loss_Description__c (minimum 20 characters), Claim__c.Loss_Type__c (picklist), and Claim__c.Reported_By__c pre-filled with the broker's Contact ID.

Given the broker completes all required steps,
When the OmniScript is submitted,
Then a new Claim__c record must be created with Claim__c.Status__c = 'FNOL Received', Claim__c.FNOL_Submission_Channel__c = 'Agent Portal', and a unique Claim__c.Claim_Number__c auto-generated via custom Apex auto-number logic.

Given the Claim__c record is successfully created,
When the OmniScript submission completes,
Then a Platform Event 'FNOL_Submitted__e' must be published containing Claim__c.Id, Policy__c.Id, and timestamp, triggering downstream assignment routing.

Given the selected Policy__c.Status__c is NOT 'Active' (e.g., 'Lapsed', 'Cancelled'),
When the broker attempts to proceed past Step 1,
Then a blocking validation message 'Policy [Policy_Number] is not currently active. FNOL cannot be submitted.' must be displayed and submission must be prevented.

Given a broker submits FNOL successfully,
When the confirmation screen renders,
Then the broker must see the newly generated Claim__c.Claim_Number__c and an estimated acknowledgement timeline based on Claim__c.Loss_Type__c.

## Solution

Develop OmniScript 'AgentPortal_FNOL_OS' with 5 steps: Policy Lookup, Incident Details, Claimant Info, Document Upload, Review & Submit. Use DataRaptor Transform to map OmniScript inputs to Claim__c fields on submit. Invoke Apex class 'ClaimNumberGeneratorService' for auto-numbering. Publish Platform Event 'FNOL_Submitted__e' via Apex invoked from OmniScript Integration Procedure. Record-Triggered Flow on Claim__c (after-insert) handles assignment queue routing. Integrate with Salesforce Files for document attachment. Apply field-level security via Broker_Portal_PSG Permission Set Group.
