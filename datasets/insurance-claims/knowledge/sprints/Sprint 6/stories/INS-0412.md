## Description

**Bug Report**

**Summary:** When a Policy__c record's Status__c is updated (e.g., from 'Active' to 'Cancelled') by a Claims Operations user, the Agent Portal policy search OmniScript continues to return the stale status value for up to 15 minutes due to a Platform Cache (session partition) TTL that is not invalidated on record update. Brokers are incorrectly seeing 'Active' policies that have been cancelled, which is causing erroneous FNOL submissions.

**Steps to Reproduce:**
1. Log in as a broker user in the Agent Portal and search for Policy Number 'ACM-2024-009823' — note Status = 'Active'.
2. As a Claims Operations Manager in the internal org, update Policy__c.Status__c to 'Cancelled' and save.
3. Return to the Agent Portal broker session and re-search for the same policy number within 15 minutes.
4. Observe the returned Policy__c.Status__c value.

**Expected Result:** Policy__c.Status__c = 'Cancelled' should be returned immediately after the internal update.

**Actual Result:** Policy__c.Status__c = 'Active' is returned from cache; the cancellation is not reflected until the cache TTL expires.

**Environment:** Acme Insurance - Partial Sandbox (CS42) and confirmed in UAT Sandbox (CS61)
**Impact:** High — brokers submitted 3 FNOL claims against cancelled policies in UAT testing cycle.

## Acceptance Criteria

As per Description. Post-fix, confirm that a Policy__c.Status__c update is reflected in Agent Portal search results within 30 seconds. Validate in both CS42 and CS61 sandbox environments. Regression test FNOL OmniScript (INS-0407) to confirm no performance degradation from cache invalidation change.

## Solution

Root cause: the DataRaptor Extract caching strategy in the Policy Search Integration Procedure uses a static cache key based on Policy_Number__c only, with no invalidation hook. Fix: implement a Record-Triggered Flow on Policy__c (after update, all changes to Status__c) that calls an Apex invocable method PolicyCacheInvalidationService.invalidateCache(policyNumber) to programmatically clear the Platform Cache session partition entry using Cache.Session.remove(). Reduce TTL from 900s to 120s as a secondary safeguard. Update Integration Procedure to include a cache-miss fallback direct SOQL path. Load test cache invalidation throughput before deploying to production.
