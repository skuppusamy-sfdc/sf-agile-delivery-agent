## Description

As a Claims Operations Manager, I need the system to automatically generate an Electronic Remittance Advice (ERA) file in ANSI X12 835 format when a Payment__c batch record is marked as disbursed, so that providers receive timely, accurate remittance data that can be ingested into their practice management systems without manual file generation by the claims operations team.

ERA generation must account for Coordination of Benefits (COB) adjustments and include all related Claim_Line__c and Explanation_of_Benefits__c (EOB) detail segments.

## Acceptance Criteria

Given a Payment__c record has Payment__c.Status__c changed to 'Disbursed' by an authorized Claims Operations user,
When the Record-Triggered Flow 'PaymentDisbursed_ERA_RTF' fires on the Payment__c update,
Then an Apex queueable job 'ERAGenerationQueueable' is enqueued to process the ERA file asynchronously within 5 minutes of the status change.

Given the ERAGenerationQueueable job executes,
When it queries related Claim_Line__c records (Claim_Line__c.Payment__c = Payment__c.Id) and their parent Claim__c records,
Then the generated 835 file includes: ST segment with transaction set ID '835', BPR segment with Payment__c.Payment_Amount__c and Payment__c.Payment_Date__c, CLP segment per Claim__c.Claim_Number__c, SVC segment per Claim_Line__c with CPT code (Claim_Line__c.CPT_Code__c), billed amount (Claim_Line__c.Billed_Amount__c), and paid amount (Claim_Line__c.Paid_Amount__c).

Given a Claim__c record has COB adjustments (Claim__c.COB_Applicable__c = TRUE),
When the ERA file is generated,
Then the CAS segment includes adjustment reason codes sourced from Explanation_of_Benefits__c.Adjustment_Reason_Code__c for each related EOB record, and EOB.Primary_Payer_Paid__c is included in the AMT segment.

Given the ERA file is successfully generated,
When the Apex job completes without exception,
Then a ContentVersion record is created with ContentVersion.Title = 'ERA_' + Payment__c.Payment_Batch_Number__c + '_' + Payment__c.Payment_Date__c, ContentVersion.FileExtension = '835', and the file is linked to the Payment__c record via ContentDocumentLink.

Given the ERA file ContentVersion is created,
When the Payment__c.Provider__c (lookup to Provider__c) has Provider__c.ERA_Delivery_Method__c = 'SFTP',
Then a Platform Event 'ERA_Ready_For_Delivery__e' is published with payload fields ERA_File_Id__c and Provider_SFTP_Config__c to trigger the outbound SFTP delivery integration.

Given the ERAGenerationQueueable job encounters an unhandled exception,
When the error is caught in the try/catch block,
Then Payment__c.ERA_Generation_Status__c is set to 'Failed', Payment__c.ERA_Error_Message__c is populated with the exception message (truncated to 255 chars), and a Case is auto-created with Subject = 'ERA Generation Failure — ' + Payment__c.Payment_Batch_Number__c assigned to the 'Claims Operations' queue.

## Solution

Implement a Record-Triggered Flow 'PaymentDisbursed_ERA_RTF' on Payment__c, triggering After Update when Payment__c.Status__c changes to 'Disbursed'. The flow calls an Apex action 'EnqueueERAGeneration' which enqueues the 'ERAGenerationQueueable' Apex class. The Queueable class queries Payment__c, related Claim__c, Claim_Line__c, and Explanation_of_Benefits__c records in bulk. An 835 builder utility class 'X12_835_Builder' assembles the EDI string using a StringBuilder pattern. ContentVersion is inserted via Apex DML. Platform Event 'ERA_Ready_For_Delivery__e' is published via EventBus.publish(). Error handling uses a dedicated 'ERAErrorHandler' utility class that performs the Payment__c update and Case insert in a separate Queueable to avoid mixed DML. Custom Metadata Type 'ERA_Config__mdt' stores adjustable parameters (file prefix, SFTP flag, segment delimiters) without code changes.
