## Description

As a Provider Network Specialist, I need the system to automatically validate a provider's NPI against the NPPES registry and their TIN against CMS records when a Provider__c record is created or when Provider__c.NPI__c or Provider__c.TIN__c is updated so that we maintain data integrity and avoid claims adjudication errors caused by invalid provider identifiers.

## Acceptance Criteria

Given a new Provider__c record is saved with a populated Provider__c.NPI__c,
When the record-triggered automation fires,
Then an asynchronous callout is made to the NPPES NPI Registry API (https://npiregistry.cms.hhs.gov/api/) and the response is stored in Provider__c.NPPES_Validation_Status__c as 'Valid', 'Invalid', or 'Not Found'.

Given the NPPES response returns 'Valid',
When the response is processed,
Then Provider__c.NPPES_Last_Validated_Date__c is set to TODAY() and Provider__c.NPPES_Provider_Name__c is populated from the API response 'basic.first_name' + 'basic.last_name' fields.

Given the NPPES response returns 'Invalid' or 'Not Found',
When the response is processed,
Then Provider__c.NPPES_Validation_Status__c is set accordingly, a Task is created assigned to the Provider Network Specialist queue with Subject = 'NPI Validation Failed — Review Required', and Provider__c.Active_in_Network__c is set to FALSE.

Given a Provider__c.TIN__c value is entered or updated,
When the CMS TIN validation callout completes,
Then Provider__c.TIN_Validation_Status__c is set to 'Verified' or 'Unverified' and Provider__c.TIN_Last_Verified_Date__c is updated.

Given either NPI or TIN validation fails,
When any associated Credentialing_Case__c is in status 'Approved',
Then Credentialing_Case__c.Status__c is automatically updated to 'Suspended — Pending Re-Validation' and a PCN record (Provider_Change_Notification__c) is created with PCN.Change_Type__c = 'Identifier Validation Failure'.

Given the NPPES API is unreachable (timeout or HTTP 5xx),
When the callout fails,
Then the failure is logged to a custom object IntegrationError__c with fields: IntegrationError__c.Source__c = 'NPPES', IntegrationError__c.Provider__c = record ID, IntegrationError__c.Error_Message__c, and a retry is scheduled via Schedulable Apex within 4 hours.

## Solution

Implement Apex Queueable class 'NPPESValidationQueueable' and 'CMSTINValidationQueueable' invoked from a Record-Triggered Flow 'RTF_ProviderIdentifierValidation' on Provider__c after insert/update when NPI__c or TIN__c is changed. Named Credentials 'NPPES_Registry' and 'CMS_Provider_API' manage endpoint auth. Response parsing handled in 'ProviderValidationResponseParser' Apex class. Retry logic uses Schedulable Apex 'ProviderValidationRetryScheduler'. PCN creation and Credentialing_Case__c status update handled in a separate Apex method called post-validation. Custom metadata type 'Validation_Config__mdt' stores retry interval and timeout thresholds. All callouts must respect governor limits via async pattern.
