## Description

**Bug Report — Agent Portal Data Isolation Failure**

**Environment:** Acme Insurance Developer Sandbox (sprint6-dev2)
**Severity:** Critical — potential data leakage between broker books of business
**Discovered By:** Sofia Hernandez during INS-0414 UAT execution

**Steps to Reproduce:**
1. Log into the Agent Portal as Broker User A (NPN: 1234567, associated with Account 'Sunrise Brokerage LLC').
2. Navigate to the 'My Claims' list view on the Agent Portal dashboard.
3. Observe the list of Claim__c records displayed.
4. Note that Claim__c records where Claim__c.Broker_of_Record__c references Broker User B (NPN: 7654321, 'Pacific Coast Insurance Group') are visible in the list.

**Expected Behavior:** Broker User A should only see Claim__c records where Claim__c.Broker_of_Record__c = Broker User A's Contact ID. Zero records from other brokers should be visible.

**Actual Behavior:** Broker User A can see Claim__c records belonging to at least 2 other broker agencies. The 'My Claims' list view SOQL query in the DataRaptor does not apply the broker filter when the Sharing Rule 'Claim_Broker_Sharing_Rule' is in a 'Draft' (inactive) state, which occurred after the last sandbox refresh.

**Root Cause Hypothesis:** The DataRaptor Extract backing the FlexCard list view relies on an OWD (Organization-Wide Default) + Sharing Rule combination for Claim__c visibility rather than enforcing an explicit WHERE Claim__c.Broker_of_Record__c = :currentUserId filter in the SOQL. When the Sharing Rule is inactive (Draft), all Claim__c records become visible to all portal users.

**Risk:** If this misconfiguration were to occur in production (e.g., post-deployment sharing rule deactivation), brokers would have cross-visibility into competitor claim data. Must be fixed before portal go-live.

## Acceptance Criteria

As per Description.

Additional verification steps:
1. Confirm DataRaptor Extract 'DR_Get_Broker_Claims' includes explicit WHERE Claim__c.Broker_of_Record__c = '{UserContact.Id}' filter regardless of sharing rule state.
2. Confirm that even with 'Claim_Broker_Sharing_Rule' set to 'Draft', Broker User A sees zero claims belonging to other brokers.
3. Confirm fix is regression-tested with 3 broker users in isolated test data set.
4. Confirm a Salesforce Shield Platform Encryption or Field Audit Trail entry is not required for this fix (Compliance Officer sign-off captured in comments).

## Solution

Fix DataRaptor Extract 'DR_Get_Broker_Claims' to add an explicit SOQL WHERE clause: WHERE Broker_of_Record__c = '{UserContact.Id}' using the DataRaptor formula token for the authenticated user's linked Contact ID. This ensures defense-in-depth: the filter is enforced at the data retrieval layer independently of sharing rule configuration. Additionally, activate the 'Claim_Broker_Sharing_Rule' Criteria-Based Sharing Rule in all sandboxes and production via a post-deployment script in the release runbook. Add a Salesforce Health Check monitoring step to the weekly ops checklist to verify Claim__c OWD = 'Private' and sharing rule is Active. Update Permission Set Group 'Broker_Portal_User' to remove any inadvertent 'View All' permission on Claim__c object.
