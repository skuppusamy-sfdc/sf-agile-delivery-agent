## Description

As a Broker Relations Coordinator, I need a guided FNOL submission OmniScript accessible from the Agent Portal so that brokers can initiate a First Notice of Loss on behalf of a policyholder without requiring a phone call to the claims operations center, reducing FNOL intake cycle time from an average of 48 hours to under 4 hours. Extends INS-0415 (FNOL base data model).

## Acceptance Criteria

Given a broker with the 'Broker_Portal_User' Permission Set Group is on the portal,
When they click 'File a New Claim' from the dashboard FlexCard (INS-0422),
Then a multi-step OmniScript titled 'Broker_FNOL_Submission' launches with a minimum of 4 steps: (1) Policy Lookup, (2) Loss Details, (3) Claimant Information, (4) Review & Submit.

Given the broker is on Step 1 (Policy Lookup),
When they enter a Policy__c.Policy_Number__c value,
Then a DataRaptor Lookup validates the policy exists, Policy__c.Status__c = 'Active', and Policy__c.Broker__c matches the logged-in broker's Contact; if validation fails, an inline error message reads 'Policy not found or not associated with your account.'

Given a valid policy is confirmed,
When the broker advances to Step 2 (Loss Details),
Then the following fields are required: Claim__c.Date_of_Loss__c (must not be a future date), Claim__c.Loss_Type__c (picklist), Claim__c.Loss_Description__c (textarea, max 2000 chars), Claim__c.LOB__c (auto-populated from Policy__c.LOB__c, read-only).

Given the broker completes Step 3 (Claimant Information),
When they submit the OmniScript,
Then a Claim__c record is created with Claim__c.Status__c = 'FNOL Received', Claim__c.Source__c = 'Broker Portal', Claim__c.Policy__c populated, and a Platform Event 'FNOL_Received__e' is published to trigger downstream assignment routing.

Given the Claim__c record is created successfully,
When the OmniScript reaches the confirmation step,
Then the broker sees the generated Claim__c.Name (e.g., CLM-000456), a summary of submitted fields, and a 'Download Confirmation PDF' button rendered via a Conga Composer integration action.

Given the broker's session times out mid-OmniScript (after 15 minutes of inactivity),
When they return and resume,
Then OmniScript session state is restored from a draft Claim__c record with Claim__c.Status__c = 'FNOL Draft' created at Step 2 completion.

## Solution

Develop OmniScript 'Broker_FNOL_Submission' with 4 steps using DataRaptor Lookup for policy validation and DataRaptor Transform for field mapping. On final step, invoke an Apex action (FNOLSubmissionController.cls) that performs DML to insert Claim__c and publishes FNOL_Received__e Platform Event. A separate Record-Triggered Flow on Claim__c (trigger: after insert, Status__c = 'FNOL Received') handles assignment routing via Assignment Coordinator queue. Draft save uses a pre-step Apex action to upsert a draft Claim__c. Conga Composer template 'FNOL_Broker_Confirmation' triggered via REST callout on confirmation step.
