## Description

As a Broker Relations Coordinator, I need an OmniScript-driven FNOL submission form accessible from the Agent/Broker Portal so that brokers can initiate a new Claim__c on behalf of a policyholder without requiring internal Claims Operations staff involvement, reducing FNOL intake cycle time. Extends INS-0445.

## Acceptance Criteria

Given a broker user clicks 'Submit New Claim' from the Broker Dashboard,
When the FNOL OmniScript launches,
Then Step 1 presents a Policy lookup restricted to Policy__c records in the broker's book of business where Policy__c.Status__c = 'Active'.

Given the broker selects a Policy__c record,
When Step 2 loads,
Then the OmniScript auto-populates Claim__c.Insured_Name__c, Claim__c.Policy_Number__c, and Claim__c.LOB__c from the selected Policy__c via DataRaptor Transform.

Given the broker completes all required FNOL fields (Date of Loss, Loss Type, Description),
When the broker clicks 'Submit',
Then a new Claim__c record is created with Claim__c.Status__c = 'FNOL Received', Claim__c.Submission_Source__c = 'Broker Portal', and Claim__c.FNOL_Date__c = today's date.

Given the Claim__c record is created,
When creation is confirmed,
Then a Platform Event (FNOL_Submitted__e) is published to trigger downstream assignment routing.

Given the FNOL submission fails due to a duplicate Claim__c detected (matching Policy__c + Date of Loss + Loss Type),
When the broker attempts to submit,
Then the OmniScript displays an inline error: 'A claim with similar details already exists: [Claim__c.Name]. Please review before submitting.' and blocks creation.

Given a broker partially completes the OmniScript and navigates away,
When the broker returns within 24 hours,
Then the OmniScript resumes from the last completed step using OmniScript Resume functionality.

-Given Step 3 includes Provider lookup for treating provider- (deprecated — provider association moved to post-FNOL adjudication workflow per Claims Operations Manager request)

## Solution

Build a multi-step OmniScript ('BrokerFNOLSubmission') with DataRaptor Extract for Policy lookup and DataRaptor Transform for field auto-population. Use an Integration Procedure for duplicate detection logic querying Claim__c with SOQL on Policy__c, Loss_Date__c, and Loss_Type__c. On submit, an Apex Action within the OmniScript creates the Claim__c and publishes FNOL_Submitted__e Platform Event. A Record-Triggered Flow on Claim__c (After Insert, FNOL Received) handles initial SIU screening flag. OmniScript save state enabled for resume functionality. Broker access gated via Permission Set Group.
