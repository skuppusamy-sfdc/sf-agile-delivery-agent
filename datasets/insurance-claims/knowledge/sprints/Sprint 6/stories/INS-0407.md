## Description

As a Broker Relations Coordinator, I need brokers to be able to submit a First Notice of Loss (FNOL) directly through the Agent Portal via a guided OmniScript so that claim intake is accelerated and manual data re-entry by Claims Operations staff is eliminated. Extends INS-0389 (Claims Intake Apex Service).

## Acceptance Criteria

Given a broker is authenticated in the Agent Portal,
When they initiate a new FNOL submission,
Then the OmniScript must first validate that a matching Policy__c record exists where Policy__c.Policy_Number__c matches the entered value and Policy__c.Status__c = 'Active'; if not found, display inline error 'No active policy found for this policy number'.

Given a valid policy is confirmed,
When the broker completes Step 2 (Loss Details),
Then Claim__c.Date_of_Loss__c must be required and must not be a future date; Claim__c.Loss_Type__c picklist must be dynamically filtered by Policy__c.LOB__c.

Given all required fields are complete,
When the broker submits the FNOL,
Then a Claim__c record must be created with Claim__c.Status__c = 'FNOL Received', Claim__c.FNOL_Submitted__c = TRUE, Claim__c.Source__c = 'Broker Portal', and Claim__c.Submitted_By__c = broker's Contact ID.

Given the Claim__c record is created,
When creation succeeds,
Then a DocuSign envelope must be triggered via the DocuSign for Salesforce API to send the FNOL acknowledgment template to the broker's email on file.

Given the FNOL is submitted,
When the confirmation screen renders,
Then Claim__c.Claim_Number__c (auto-generated) must be displayed to the broker with a copy-to-clipboard action.

-Given broker uploads supporting documents, files must be stored in Salesforce Files- (deprecated: document upload deferred to Sprint 7 per backlog grooming)

Given a submission error occurs at any step,
When the OmniScript catches an exception,
Then the error must be logged to a custom object Error_Log__c with Error_Log__c.Source__c = 'FNOL OmniScript' and the broker must see a user-friendly message.

## Solution

Build a multi-step OmniScript 'FNOL_Broker_Submission_OS' with DataRaptor Transform steps for policy lookup and Claim__c creation. Use an Integration Procedure to orchestrate: (1) policy validation SOQL via DRE, (2) Claim__c upsert via DataRaptor Load, (3) DocuSign envelope trigger via named credential REST callout. An Apex-invocable method from INS-0389's ClaimsIntakeService class handles auto-number generation for Claim__c.Claim_Number__c. Deploy via OmniStudio version activation; wrap in a Permission Set Group 'Broker_FNOL_Access'.
