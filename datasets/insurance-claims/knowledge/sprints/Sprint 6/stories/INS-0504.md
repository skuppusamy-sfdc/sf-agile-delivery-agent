## Description

**Bug Report — INS-0504**

The Broker FNOL OmniScript (BrokerFNOLSubmission_OS, built in INS-0501) throws a null pointer error and halts on Step 2 ('Loss Details') when the selected Policy__c record has no child Coverage_Line__c records. This prevents brokers from completing FNOL submission for policies that were migrated from the legacy system without coverage line detail records.

**Environment:** Acme Insurance Full Sandbox (FS1)
**Found by:** QA during Sprint 6 regression testing
**Affected User Profile:** Broker Community User

**Steps to Reproduce:**
1. Log in to the Broker Portal Experience Cloud site as a broker test user (e.g., testbroker@acmeins.test.portal).
2. Click 'Submit a Claim' from the Broker Dashboard.
3. In the Policy Lookup step, search for and select Policy Number 'POL-MIGR-00847' (a migrated policy with zero Coverage_Line__c child records).
4. Click 'Next' to advance to the Loss Details step.
5. Observe: OmniScript displays a generic error banner 'An error has occurred. Please try again.' and the step does not render. Browser console shows: 'TypeError: Cannot read properties of undefined (reading "Coverage_Type__c")' originating from DataRaptor Transform 'DR_Transform_CoverageLines'.

**Expected Behavior:** When a policy has no Coverage_Line__c records, the Loss Details step should render with a warning message 'Coverage detail unavailable for this policy. You may proceed with FNOL submission.' and allow the broker to continue without coverage-line-dependent fields.

**Actual Behavior:** OmniScript throws a null pointer error in the DataRaptor Transform, halts step rendering, and the broker cannot proceed.

**Impact:** Approximately 340 migrated policies (identified via SOQL: SELECT COUNT() FROM Policy__c WHERE Migration_Source__c = 'Legacy_PAS' AND Id NOT IN (SELECT Policy__c FROM Coverage_Line__c)) are affected. Brokers cannot file FNOL for these policies via the portal.

**Workaround:** Brokers must call the claims operations center directly. Estimated 15–20 calls/day impacted.

## Acceptance Criteria

As per Description.

Additional verification steps:
- Confirm that selecting a policy with zero Coverage_Line__c records no longer throws an error and Step 2 renders with the warning message.
- Confirm that the DataRaptor Transform 'DR_Transform_CoverageLines' handles an empty/null response array without exception (null-safe mapping applied).
- Confirm that policies WITH Coverage_Line__c records continue to function correctly (no regression).
- Confirm fix is validated against Policy POL-MIGR-00847 and at least 2 additional migrated policies in FS1 sandbox.

## Solution

Root cause: DataRaptor Transform 'DR_Transform_CoverageLines' maps Coverage_Line__c.Coverage_Type__c without a null-safe default, causing a JavaScript TypeError when the input array is empty. Fix: (1) Add a null-safe default value ('') to the Coverage_Type__c mapping node in the DataRaptor Transform. (2) Add an OmniScript 'Set Values' action before the Loss Details step group that checks if the 'CoverageLines' DataRaptor response array length === 0 and sets a boolean OmniScript element 'NoCoverageLines' = true. (3) Use a conditional view on the Loss Details step group to display the warning message block when 'NoCoverageLines' = true. (4) Ensure coverage-line-dependent fields are conditionally required (Required = false) when 'NoCoverageLines' = true so the broker can advance. Deploy fix to FS1 for QA re-validation before promoting to UAT.
