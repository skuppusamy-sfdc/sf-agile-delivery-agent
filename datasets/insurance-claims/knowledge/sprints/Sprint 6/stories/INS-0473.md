## Description

**Bug Report — Broker Portal FNOL OmniScript Null Pointer Error**

**Environment:** Acme Insurance Full Sandbox (FS1)
**Discovered by:** Emily Watson (QA) during Sprint 6 regression testing

**Steps to Reproduce:**
1. Log into the Agent Portal as a broker user with Permission Set 'Broker_FNOL_Submit'
2. Click 'Submit New Claim' to launch the BrokerFNOL_OS OmniScript
3. In Step 1, search for and select a Policy__c record that has Status__c = 'Active' but has zero associated Claim_Line__c child records (e.g., Policy Number TEST-POL-00042)
4. Advance to Step 3 (Coverage Verification)
5. Observe the OmniScript Step 3 DataRaptor Extract (DR_GetPolicyCoverageLines) execute

**Expected Behavior:**
Step 3 should render an empty coverage lines section with the message 'No existing claim lines found for this policy. Proceed to enter loss details.' and allow the broker to continue the FNOL submission.

**Actual Behavior:**
The OmniScript throws an unhandled NullPointerException in the Integration Procedure 'IP_CreateFNOLClaim', crashing the OmniScript session and displaying a generic 'An error has occurred' toast. The broker is unable to complete FNOL submission. The error is logged in Salesforce debug logs as: 'System.NullPointerException: Attempt to de-reference a null object at line 47 of FNOLClaimCreator.cls'.

**Root Cause (Initial Analysis):**
The Apex invocable method 'FNOLClaimCreator.buildCoverageLinesList()' does not perform a null check on the List<Claim_Line__c> returned by the DataRaptor before iterating, causing a null dereference when the list is empty.

**Frequency:** Reproducible 100% with policies having no Claim_Line__c records. Affects approximately 12% of active policies in the broker portfolio based on data audit.

**Severity:** High — blocks broker self-service FNOL submission for a significant subset of policies.

## Acceptance Criteria

As per Description. Verify fix by executing the reproduction steps against Policy TEST-POL-00042 in FS1 sandbox. Confirm OmniScript Step 3 renders without error and FNOL submission completes successfully, creating a Claim__c record with Status__c = 'FNOL Submitted'.

## Solution

Add null check in 'FNOLClaimCreator.buildCoverageLinesList()' at line 47: replace direct iteration with a null-safe pattern (e.g., 'if (coverageLines != null && !coverageLines.isEmpty())'). Additionally update DR_GetPolicyCoverageLines DataRaptor to return an empty array node rather than null when no Claim_Line__c records exist, preventing the null propagation upstream. Add a conditional element in the OmniScript Step 3 to display the empty-state message when the DataRaptor response collection size = 0. Deploy fix to FS1 for QA validation, then promote to UAT. Add unit test coverage in FNOLClaimCreatorTest.cls for the null Claim_Line__c scenario targeting 95%+ code coverage on the affected method.
