## Description

As a licensed broker, I need a guided self-service FNOL submission form within the Agent Portal so that I can initiate a new insurance claim on behalf of a policyholder without requiring internal Claims Operations staff involvement for intake. The OmniScript must collect all required FNOL data fields, validate policy eligibility, and create a Claim__c record in 'FNOL Submitted' status upon completion.

## Acceptance Criteria

Given a broker accesses the 'File a New Claim' action from the Agent Portal,
When the OmniScript launches,
Then Step 1 presents a policy search by Policy__c.Policy_Number__c or insured name, and only policies where Policy__c.Status__c = 'Active' and Policy__c.Broker__c = current broker are returned.

Given a broker selects a valid policy,
When Step 2 (Loss Details) is displayed,
Then the following fields are required: Claim__c.Date_of_Loss__c, Claim__c.Loss_Type__c (picklist), Claim__c.Loss_Description__c (textarea, min 50 chars), Claim__c.Reported_By__c.

Given the broker enters a Claim__c.Date_of_Loss__c that is earlier than Policy__c.Effective_Date__c or later than Policy__c.Expiration_Date__c,
When the Step 2 validation runs,
Then an inline error is shown: 'Date of Loss falls outside the active policy period. Please verify loss date or contact Claims Operations.'

Given all required FNOL fields are valid,
When the broker clicks 'Submit Claim' on the confirmation step,
Then a Claim__c record is created with Claim__c.Status__c = 'FNOL Submitted', Claim__c.Source__c = 'Broker Portal', Claim__c.Assigned_Policy__c populated, and a unique Claim__c.Claim_Number__c auto-generated via custom auto-number format 'CLM-{YYYY}-{0000000}'.

Given the Claim__c record is successfully created,
When the OmniScript reaches the confirmation screen,
Then the broker is shown the new Claim__c.Claim_Number__c and a summary, and a confirmation email is dispatched to the policyholder's Contact.Email using the 'FNOL_Confirmation_Broker' email template.

Given the OmniScript submission fails due to a DML error,
When the error is caught,
Then the broker sees a user-friendly error message and the error is logged to a custom object Claim_Submission_Error_Log__c with fields: Error_Message__c, Stack_Trace__c, Broker_User__c, Timestamp__c.

## Solution

Build a multi-step OmniScript (FNOL_BrokerSubmission_OS) with steps: PolicySearch (DataRaptor Extract on Policy__c), LossDetails (custom validation formula for date range check), ClaimantInfo, DocumentUpload (Salesforce Files via OmniScript File component), Confirmation. On Submit, use a DataRaptor Transform + Integration Procedure to upsert Claim__c. Auto-number format configured on Claim__c object. Email dispatch via Apex EmailService invoked from Integration Procedure. Error logging via Apex try/catch in the Integration Procedure's Apex Action step writing to Claim_Submission_Error_Log__c. Broker user access gated by BrokerPortal_PSG Permission Set Group with Create on Claim__c.
