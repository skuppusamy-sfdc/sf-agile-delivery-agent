## Description

**Bug Report**

**Summary:** Claim__c.Total_Incurred__c roll-up summary field is including Claim_Line__c records with Status__c = 'Voided', causing overstated claim liability figures on EOBs and in Payment__c disbursement calculations.

**Steps to Reproduce:**
1. Navigate to any Claim__c record (e.g., CLM-000312) that has at least one Claim_Line__c with Status__c = 'Voided'.
2. Observe Claim__c.Total_Incurred__c value on the record detail page.
3. Manually sum only the Claim_Line__c records where Status__c != 'Voided' and compare to the displayed Total_Incurred__c.
4. Note the discrepancy — the roll-up includes voided lines.

**Expected Behavior:** Claim__c.Total_Incurred__c should only aggregate Claim_Line__c.Allowed_Amount__c for lines where Claim_Line__c.Status__c NOT IN ('Voided', 'Reversed').

**Actual Behavior:** All Claim_Line__c records regardless of Status__c are included in the roll-up, inflating Total_Incurred__c.

**Impact:** Payment__c records generated from this field are overpaying providers. EOB__c.Total_Plan_Paid__c is also incorrectly calculated downstream. Identified in UAT on 2024-12-03 by Claims Operations Manager.

**Environment:** Full Sandbox (ACME_FS1)
**Frequency:** Reproducible 100% on claims with voided lines (approx. 340 affected Claim__c records identified via SOQL audit).

## Acceptance Criteria

As per Description.

Given a Claim__c record has Claim_Line__c children with mixed statuses including 'Voided' and 'Reversed',
When Claim__c.Total_Incurred__c is evaluated,
Then the value equals the SUM of Claim_Line__c.Allowed_Amount__c WHERE Claim_Line__c.Status__c NOT IN ('Voided', 'Reversed') only.

Given the fix is deployed,
When a batch recalculation job runs against the 340 identified affected Claim__c records,
Then all Claim__c.Total_Incurred__c values are corrected and a reconciliation report is generated comparing old vs. new values.

## Solution

Modify the existing Roll-Up Summary Field (RUSF) filter on Claim__c.Total_Incurred__c to add filter criteria: Claim_Line__c.Status__c NOT EQUAL TO 'Voided' AND Claim_Line__c.Status__c NOT EQUAL TO 'Reversed'. If RUSF filter limitations prevent multi-value exclusion, replace with an Apex-managed roll-up via a Trigger on Claim_Line__c (after insert, after update, after delete, after undelete) calling ClaimLineRollupService.cls. Run a one-time Batch Apex job (RecalculateTotalIncurredBatch.cls) scoped to affected 340 records. Validate downstream EOB__c.Total_Plan_Paid__c recalculation via existing Payment Recalculation Flow.
