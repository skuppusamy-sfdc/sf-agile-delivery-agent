## Description

As a Compliance Officer, I need every status transition on Claim__c to be recorded in an immutable audit log with timestamp, actor identity, previous value, and new value so that the organization can satisfy state DOI examination requests and internal SIU investigations without manual reconstruction of claim history.

## Acceptance Criteria

Given a Claim__c record whose Status__c field changes from any value to any other value,
When the DML operation is committed,
Then a new Claim_Audit_Log__c record must be created with: Claim__c (lookup), Field_Name__c = 'Status__c', Old_Value__c, New_Value__c, Changed_By__c (User lookup), Changed_DateTime__c (UTC), Change_Source__c ('User'|'Automation'|'Integration'), and IP_Address__c captured from UserInfo.

Given a Claim_Audit_Log__c record has been created,
When any user or process attempts to update or delete that record,
Then the operation must be blocked with a custom exception message 'Audit log records are immutable and cannot be modified.'

Given the Compliance Officer navigates to the Claim__c record page,
When they view the 'Adjudication Audit Trail' Lightning component,
Then all Claim_Audit_Log__c entries for that claim must be displayed in reverse chronological order with columns: Date/Time, Changed By, Field, From, To, Source.

Given a bulk adjudication run processes more than 200 Claim__c records,
When status changes occur across the batch,
Then all corresponding Claim_Audit_Log__c records must be inserted without hitting governor limits, using chunked Apex batch processing.

-Given audit logs older than 3 years are archived to external storage automatically- (deprecated: retention policy moved to INS-1612)

Given Change_Source__c = 'Integration' is set,
When the record is created,
Then Integration_System__c must also be populated (e.g., 'CMS', 'ERA Feed', 'TPA Gateway') and must not be null.

## Solution

Implement an Apex After Trigger on Claim__c that fires on insert and update, comparing old vs. new Status__c values and inserting Claim_Audit_Log__c records via a dedicated AuditLogService Apex class invoked asynchronously via Platform Event (Claim_Status_Change__e) to decouple from the triggering transaction and avoid mixed DML issues. The Claim_Audit_Log__c object will have OWD set to Private with no sharing rules and will use a custom Permission Set Group 'Compliance_ReadOnly' granting read access to Compliance Officers. A Record-Triggered Flow is explicitly excluded to preserve governor limit headroom in bulk scenarios. The LWC 'claimAuditTrail' will be placed on the Claim__c Lightning record page using a Tab component. Validation Rule on Claim_Audit_Log__c will block edits; a separate Apex Trigger will block deletes by throwing a custom AuditImmutabilityException.
