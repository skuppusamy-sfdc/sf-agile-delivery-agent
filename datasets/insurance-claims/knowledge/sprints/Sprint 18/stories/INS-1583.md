## Description

**Bug Report — PNA Effective Date Validation Bypass via Bulk API**

**Environment:** Production, Staging
**Reported By:** Emily Watson (Provider Network Specialist)
**Severity:** High — data integrity and compliance risk; invalid PNA records with back-dated effective dates are being created by the nightly Workday integration load, resulting in Fee_Schedule__c lookups resolving to incorrect rate periods.

**Steps to Reproduce:**
1. Using Salesforce Bulk API v1 (or Data Loader) logged in as the Workday_Integration_User profile, attempt to insert a Provider_Network_Agreement__c record with PNA_Effective_Date__c set to a date more than 365 days in the past.
2. Observe that the insert succeeds without error.
3. Navigate to the newly created PNA record and confirm PNA_Effective_Date__c reflects the back-dated value.
4. Note that the same insert attempted via the Salesforce UI correctly triggers validation rule 'PNA_EffectiveDate_NotBackdated' and blocks the save.

**Expected Behavior:**
The validation rule 'PNA_EffectiveDate_NotBackdated' on Provider_Network_Agreement__c should fire for all DML operations regardless of source (UI, API, Apex, Bulk API), blocking PNA records with PNA_Effective_Date__c more than 365 days prior to today.

**Actual Behavior:**
Validation rule is bypassed when records are inserted via Bulk API. Investigation shows the Workday_Integration_User profile has the 'Bypass_PNA_Validation' custom permission, which was originally granted to allow legitimate historical data migration during Phase 1 onboarding (Sprint 8). This custom permission is referenced in the validation rule formula as an exclusion condition: OR($Permission.Bypass_PNA_Validation, ...). The migration is long complete but the permission was never revoked.

**Impact:** 47 PNA records in Production have invalid back-dated effective dates as of last audit. Fee_Schedule__c.Effective_Date__c lookups on these PNAs are resolving to expired rate schedules, causing underpayment on approximately 120 Claim_Line__c records.

**Fix Required:** Remove 'Bypass_PNA_Validation' custom permission from Workday_Integration_User profile/permission set. Validate that the Workday integration does not legitimately require back-dating beyond 365 days. Remediate the 47 affected PNA records.

## Acceptance Criteria

As per Description. Additionally: after fix deployment, a data remediation script must correct PNA_Effective_Date__c on the 47 identified records and trigger re-adjudication of the ~120 affected Claim_Line__c records by setting Claim_Line__c.Adjudication_Status__c = 'Pending Re-Adjudication'.

## Solution

Immediate fix: Remove 'Bypass_PNA_Validation' custom permission from the Workday_Integration_Permission_Set assigned to Workday_Integration_User. Confirm with the Workday integration team that no current integration use case requires PNA back-dating beyond 365 days. Update the validation rule formula to remove the $Permission.Bypass_PNA_Validation exclusion clause. Data remediation: develop an anonymous Apex script (reviewed and approved via change management) to query the 47 affected PNA records, correct PNA_Effective_Date__c to the nearest valid date per contract documentation, and set Claim_Line__c.Adjudication_Status__c = 'Pending Re-Adjudication' on related claim lines. Long-term: implement an Apex Trigger on Provider_Network_Agreement__c to enforce effective date constraints as a code-level safeguard independent of validation rules, ensuring bypass via custom permission cannot recur.
