## Description

Steps to Reproduce:
1. Log in as the Workday Integration service account user (profile: 'Integration Service Profile').
2. Via the Workday-to-Salesforce middleware, send a REST API PATCH request updating Fee_Schedule__c.Effective_Date__c to a date in the past (e.g., 2022-01-01) on an active Fee_Schedule__c record where Fee_Schedule__c.Status__c = 'Active'.
3. Confirm the update succeeds with HTTP 200 and no error is returned.
4. Navigate to the Fee_Schedule__c record in Salesforce and observe that Effective_Date__c has been backdated successfully.
5. Attempt the same update as a standard internal user via the UI — the validation rule fires correctly and prevents the backdating.

Expected Behavior: The validation rule 'No_Backdating_Active_Fee_Schedule' on Fee_Schedule__c must enforce that Fee_Schedule__c.Effective_Date__c cannot be set to a date more than 7 days in the past for records with Status__c = 'Active', regardless of the running user's profile or whether the update originates from UI, REST API, or Workday integration.

Actual Behavior: The validation rule is bypassed when the Workday Integration service account user performs the update, allowing backdated fee schedule effective dates that create compliance and claims payment accuracy risks — adjudicated Claim_Line__c records may be repriced incorrectly against a backdated fee schedule.

Environment: Production and Full Sandbox.
Discovered during Sprint 18 compliance audit of fee schedule change history.

## Acceptance Criteria

As per Description. Verify fix by running the Workday integration user test scenario in Full Sandbox and confirming the validation rule fires and returns an appropriate error message. Confirm that legitimate Workday fee schedule updates with future or current Effective_Date__c are not blocked. Add a Compliance_Audit_Event__c record creation to log any attempted backdating violation with the integration user's name, attempted date value, and timestamp. Regression test must cover: UI user (blocked), REST API internal user (blocked), Workday integration user (blocked), and Workday integration user with valid future date (allowed).

## Solution

Investigate the 'No_Backdating_Active_Fee_Schedule' validation rule formula. The likely root cause is a $Profile.Name or $Permission check in the validation rule formula that inadvertently exempts the Workday Integration service account's profile from the rule. Remove any profile-based bypass logic from the validation rule formula to make it universally applicable. If the Workday integration requires bypassing validation for legitimate operational reasons, implement a dedicated Custom Permission 'Fee_Schedule_Date_Override' with a separate approval-gated process rather than blanket profile exemption. Update the validation rule formula to use NOT($Permission.Fee_Schedule_Date_Override) as the bypass condition. Add an Apex Trigger (FeeScheduleComplianceTrigger) as a defense-in-depth measure to create Compliance_Audit_Event__c records on attempted backdating violations regardless of whether the validation rule fires. Update the Workday-Salesforce middleware error handling to surface validation rule failures back to the Workday audit log.
