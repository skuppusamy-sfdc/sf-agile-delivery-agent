## Description

As a Credentialing Specialist, I need a Provider_Change_Notification__c (PCN) record to be automatically generated whenever a Credentialing_Case__c transitions to a terminal status (Approved, Denied, Suspended, Lapsed) so that the audit trail required by NCQA credentialing standards is maintained and downstream network agreement updates are triggered in a traceable manner. Extends INS-1577.

## Acceptance Criteria

Given a Credentialing_Case__c record's Status__c changes to 'Approved', 'Denied', 'Suspended', or 'Lapsed',
When the update is saved,
Then a Provider_Change_Notification__c record must be auto-created with: PCN.Provider__c = Credentialing_Case__c.Provider__c, PCN.Notification_Type__c = 'Credentialing Status Change', PCN.Effective_Date__c = today, PCN.Previous_Status__c, PCN.New_Status__c, PCN.Triggered_By_Case__c (lookup to Credentialing_Case__c), and PCN.Status__c = 'Pending Review'.

Given a PCN is created with New_Status__c = 'Approved',
When PCN.Status__c = 'Pending Review',
Then the assigned Provider Network Specialist must receive a Bell notification and an email with a link to the PCN and the associated Provider__c record.

Given a PCN is created with New_Status__c = 'Denied' or 'Suspended',
When PCN.Status__c = 'Pending Review',
Then all active Provider_Network_Agreement__c records for that provider where PNA.Status__c = 'Active' must have PNA.Pending_Review_Flag__c set to true and a related Task created for the Provider Network Specialist.

Given CAQH integration is enabled for the provider (Provider__c.CAQH_Integration_Active__c = true),
When a PCN with New_Status__c = 'Approved' is created,
Then an outbound callout to the CAQH ProView API must be initiated to confirm attestation currency, and CAQH_Last_Verified__c on Provider__c must be updated with the response timestamp.

Given a Credentialing_Case__c update occurs via bulk data load,
When more than 50 records are updated in a single transaction,
Then PCN creation must be handled asynchronously via a Queueable Apex chain without dropping records.

## Solution

Implement a Record-Triggered Flow on Credentialing_Case__c that fires After Save on Update, with an entry condition checking that Status__c changed to one of the four terminal values using the {!$Record.Status__c} vs {!$Record__Prior.Status__c} comparison. The flow calls an Apex Action 'CreatePCNAndNotify' which handles PCN insertion, PNA flag updates, and Task creation within a single Apex transaction. CAQH callout is handled via a separate Queueable Apex class 'CAQHVerificationQueueable' enqueued from the Apex Action to respect callout-after-DML restrictions. Bulk safety is ensured by a custom object 'PCN_Creation_Queue__c' used as a staging table when the flow detects bulk context via a formula checking System.isBatch(). Bell notifications use the ConnectApi.ChatterFeeds.postFeedElement method.
