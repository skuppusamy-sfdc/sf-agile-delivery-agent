## Description

**Bug Report — INS-1569**

**Summary:** When a Network_Termination_Case__c is closed with Termination_Outcome__c = 'Terminated', only the first related Provider_Network_Agreement__c record is updated to PNA_Status__c = 'Terminated'. Any additional PNA records linked to the same Provider (Person Account) via Provider__c lookup remain in 'Active' status, creating a compliance gap where a terminated provider appears still active under secondary agreements.

**Steps to Reproduce:**
1. Create a Provider (Person Account) with 3 active PNA records (PNA_Status__c = 'Active') across different LOBs (e.g., Commercial, Medicare Advantage, Medicaid).
2. Create a Network_Termination_Case__c linked to this provider.
3. Close the case by setting Termination_Outcome__c = 'Terminated' and Status__c = 'Closed'.
4. Navigate to the Provider record and review all related Provider_Network_Agreement__c records.

**Expected Behavior:** All 3 PNA records for the provider should have PNA_Status__c updated to 'Terminated' and Termination_Effective_Date__c populated with the value from Network_Termination_Case__c.Effective_Date__c.

**Actual Behavior:** Only the PNA record stored in Network_Termination_Case__c.Primary_PNA__c (a direct lookup field) is updated. The remaining 2 PNA records remain 'Active'.

**Environment:** Partial Sandbox (ACME-PS1) and Production
**Salesforce Version:** Winter '25
**Frequency:** 100% reproducible when provider has more than 1 active PNA
**Regulatory Impact:** HIGH — terminated providers may continue to receive claim payments against active secondary PNAs.

## Acceptance Criteria

As per Description. Post-fix verification: repeat steps 1–4 with 3 PNA records and confirm all 3 PNA records have PNA_Status__c = 'Terminated' and Termination_Effective_Date__c = Network_Termination_Case__c.Effective_Date__c. Confirm existing single-PNA provider termination scenarios are unaffected (regression). Confirm Compliance_Event__c records are created for each terminated PNA.

## Solution

Root cause: the existing Record-Triggered Flow on Network_Termination_Case__c uses a 'Get Records' element scoped to Network_Termination_Case__c.Primary_PNA__c (single lookup) rather than querying all PNA records for the provider. Fix: update the Flow's 'Get Records' element to query Provider_Network_Agreement__c WHERE Provider__c = Network_Termination_Case__c.Provider__c AND PNA_Status__c = 'Active', returning a collection. Replace the single 'Update Records' element with a loop over the collection, updating each PNA's PNA_Status__c = 'Terminated' and Termination_Effective_Date__c = Effective_Date__c. Add a Create Records element inside the loop to generate Compliance_Event__c per PNA. Add bulk test data (200+ providers with 3 PNAs each) to unit tests to verify governor limit compliance.
