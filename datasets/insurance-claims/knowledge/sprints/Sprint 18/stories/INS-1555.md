## Description

As a Compliance Officer, I need a tamper-evident audit trail that captures every status transition on Claim__c and Claim_Line__c records so that we can satisfy state DOI examination requests and demonstrate adjudication integrity during regulatory reviews.

## Acceptance Criteria

Given a Claim__c record undergoes any Status__c transition (e.g., 'Submitted' → 'In Review' → 'Adjudicated' → 'Paid'), When the transition is saved by any user or automated process, Then a Claim_Audit_Event__c record must be created capturing: Claim__c lookup, Old_Status__c, New_Status__c, Changed_By__c (User), Changed_DateTime__c (UTC), Change_Source__c ('User'|'Flow'|'Apex'|'Integration'), and IP_Address__c.

Given a Claim_Line__c record has its Adjudication_Status__c field changed, When the record is committed, Then a corresponding Claim_Line_Audit_Event__c record must be created with Claim_Line__c lookup, CPT_Code__c, Billed_Amount__c, Allowed_Amount__c, and the actor's User ID.

Given a Compliance Officer views a Claim__c record, When they open the Audit Trail tab on the Claims 360 FlexCard, Then all Claim_Audit_Event__c records for that claim must be displayed in reverse-chronological order with no ability to edit or delete entries.

Given an audit trail record is created, When any DML operation attempts to update or delete a Claim_Audit_Event__c record, Then the operation must be blocked by an Apex Trigger with error message: 'Audit trail records are immutable and cannot be modified or deleted.'

-Given audit events are written synchronously in the same transaction- (deprecated: replaced by Platform Event async pattern per INS-1531)

Given the Claims Operations Manager runs a compliance summary, When they filter Claim_Audit_Event__c by Date_Range__c and LOB__c, Then the report must return all matching records within 10 seconds for datasets up to 500,000 rows.

Given a TPA submits a batch adjudication file via integration, When Claim__c statuses are bulk-updated, Then audit events must be generated for every record in the batch without exceeding governor limits (chunked via Queueable Apex).

## Solution

Implement a Platform Event (Claim_State_Change__e) published from a Record-Triggered Flow on Claim__c (after-save, all status changes) and a separate Apex Trigger on Claim_Line__c. A dedicated Apex Platform Event subscriber (ClaimAuditEventSubscriber) inserts Claim_Audit_Event__c records in an async context to avoid mixed DML. Claim_Audit_Event__c is configured with OWD = Private, no sharing rules, and a dedicated Permission Set Group 'Compliance_ReadOnly' grants read-only access to Compliance Officers. A before-delete and before-update Apex Trigger on Claim_Audit_Event__c enforces immutability. FlexCard 'ClaimsAuditTrailCard' built in OmniStudio surfaces the timeline. Custom indexes on Claim__c (lookup) and Changed_DateTime__c support report performance. Queueable Apex chaining handles bulk TPA batch scenarios.
