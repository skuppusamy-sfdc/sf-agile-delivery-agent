# INS-1559: Credentialing Case Compliance Hold: Block Claims Payment When Credentialing_Case__c is Suspended

## Description

As a Credentialing Specialist, I need the system to automatically place a compliance payment hold on all pending Payment__c records associated with a provider when their Credentialing_Case__c status transitions to 'Suspended' so that we do not disburse funds to providers who have failed credentialing review, ensuring regulatory compliance with NCQA standards and state licensure requirements.

## Acceptance Criteria

Given a Credentialing_Case__c record has its Case_Status__c updated to 'Suspended', When the record is saved, Then all Payment__c records where Payment__c.Provider__c = Credentialing_Case__c.Provider__c and Payment__c.Payment_Status__c IN ('Pending', 'Approved') must have Payment_Status__c set to 'Compliance Hold' and Compliance_Hold_Reason__c = 'Provider Credentialing Suspended'.

Given Payment__c records are placed on Compliance Hold, When the hold is applied, Then a Compliance_Hold_Log__c record must be created for each affected payment capturing: Payment__c lookup, Hold_Applied_Date__c, Hold_Reason__c, Credentialing_Case__c lookup, and Applied_By__c = 'System'.

Given a Credentialing_Case__c transitions from 'Suspended' to 'Active' (credentialing reinstated), When the status change is saved, Then the system must NOT automatically release the payment holds — a Credentialing Specialist must manually review and release each Payment__c hold individually via a 'Release Hold' quick action, which logs Compliance_Hold_Log__c.Release_Notes__c.

Given the Payment Engine attempts to process a Payment__c with Payment_Status__c = 'Compliance Hold', When the disbursement batch job runs, Then the record must be skipped with a batch log entry and no ERA file must be generated for that payment.

Given a Claims Operations Manager views the Provider 360 FlexCard for a suspended provider, When the card loads, Then a red compliance banner must display: 'Active Credentialing Suspension — [Count] payments on hold' with a link to the open Credentialing_Case__c.

Given the CAQH integration returns a provider sanction flag during nightly sync, When the sync updates Provider__c.CAQH_Sanction_Flag__c = true, Then if no open Credentialing_Case__c exists for that provider, the system must auto-create one with Case_Status__c = 'Suspended' and Source__c = 'CAQH Integration'.

## Solution

Record-Triggered Flow on Credentialing_Case__c (after-update, Case_Status__c changed to 'Suspended') calls an Apex invocable method 'CredentialingComplianceHoldService.applyPaymentHolds()' which bulk-queries and updates Payment__c records and inserts Compliance_Hold_Log__c entries in a single transaction using Database.update/insert. A separate before-execute check in the Payment disbursement Apex batch class 'PaymentDisbursementBatch' filters out Payment_Status__c = 'Compliance Hold' records. The 'Release Hold' quick action is a Screen Flow with a mandatory Release_Notes__c text input, updating Payment__c and inserting a release Compliance_Hold_Log__c. CAQH nightly DataRaptor integration updated to check for existing open Credentialing_Case__c before auto-creating to prevent duplicates. FlexCard expression updated with conditional banner rendering.
