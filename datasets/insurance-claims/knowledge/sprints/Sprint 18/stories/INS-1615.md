## Description

**Bug Report — Regulatory Reporting Dashboard: Incorrect Claim Count for Workers Comp LOB**

**Environment:** Acme Insurance Production (also reproducible in Full Sandbox)
**Reported by:** Amanda Foster (Compliance Officer)
**Severity:** High — affects monthly CMS submission accuracy

**Summary:**
The 'Regulatory Claims Summary' dashboard (Dashboard API name: Regulatory_Claims_Summary_Dash) displays an inflated Claim__c count when the LOB filter is set to 'Workers Comp'. The count shown in the 'Total Adjudicated Claims' dashboard component is approximately 2–3x the expected value, causing the Compliance team to nearly submit incorrect figures to CMS for the November 2024 reporting period.

**Steps to Reproduce:**
1. Navigate to the 'Regulatory Claims Summary' dashboard in the Compliance app.
2. Apply the global filter: Line_of_Business__c = 'Workers Comp'.
3. Observe the 'Total Adjudicated Claims' tile — note the count displayed (e.g., 4,847).
4. Open the underlying 'Adjudicated Claims by LOB' report directly and apply the same filter.
5. Observe the report count (e.g., 1,623) — this matches manual SOQL count: SELECT COUNT() FROM Claim__c WHERE Line_of_Business__c = 'Workers Comp' AND Status__c = 'Adjudicated'.
6. The dashboard tile inflates the count by roughly the number of Claim_Line__c child records per Claim__c (average ~3), strongly suggesting the report driving the dashboard tile is joining at the Claim_Line__c level without a distinct Claim__c count.

**Expected Behavior:**
The 'Total Adjudicated Claims' dashboard tile must display the count of distinct Claim__c records where Claim__c.Line_of_Business__c = 'Workers Comp' AND Claim__c.Status__c = 'Adjudicated', matching the underlying report and direct SOQL.

**Actual Behavior:**
The tile displays a count inflated by the average number of child Claim_Line__c records, indicating the report uses a 'Claim Lines with Claims' Custom Report Type and counts rows rather than distinct parent Claim__c records.

**Root Cause Hypothesis:**
The dashboard component 'Total Adjudicated Claims' references report 'Adjudicated_Claims_LOB_Detail' which was recently updated (INS-1601 sprint 17) to switch its Custom Report Type from 'Claims' (primary only) to 'Claim Lines with Claims' to support line-level drill-down. This change caused the row count metric to reflect Claim_Line__c rows rather than parent Claim__c records. A RECORD_COUNT metric on this report type without a distinct parent grouping will always over-count.

**Impact:**
- Compliance Officer cannot trust dashboard figures for CMS regulatory submissions.
- Risk of regulatory penalty if incorrect data is submitted.
- Affects all LOBs but is most visible for Workers Comp due to higher average lines-per-claim (avg 3.1 vs 1.4 for Medical).

## Acceptance Criteria

As per Description.

Additional verification steps post-fix:
- Confirm 'Total Adjudicated Claims' tile count matches SELECT COUNT() FROM Claim__c WHERE Line_of_Business__c = 'Workers Comp' AND Status__c = 'Adjudicated' within a tolerance of 0 (exact match).
- Confirm fix does not break the line-level drill-down requirement introduced in INS-1601 (the underlying detail report must still show Claim_Line__c rows when a user clicks through from the dashboard tile).
- Confirm all other LOB filter values (Medical, Dental, Vision, Auto, General Liability) also return correct distinct Claim__c counts on the dashboard.

## Solution

1. **Root cause confirmed**: Report 'Adjudicated_Claims_LOB_Detail' uses Custom Report Type 'Claim Lines with Claims' and the dashboard component metric is set to 'Record Count' which counts Claim_Line__c rows, not distinct Claim__c parents.

2. **Fix approach**: Create a new summary report 'Adjudicated_Claims_LOB_Summary' using the standard 'Claims' Custom Report Type (Claim__c primary, no child join) with a Row Count metric and LOB grouping. Update the dashboard tile 'Total Adjudicated Claims' to point to this new summary report.

3. **Preserve drill-down**: Keep the existing 'Adjudicated_Claims_LOB_Detail' report (Claim Lines with Claims type) as the drill-through target by configuring the dashboard component's 'Drill Down to Report' setting to reference the detail report, while the displayed metric is sourced from the summary report. This satisfies both the accurate count display and the INS-1601 line-level drill-down requirement.

4. **Dashboard filter alignment**: Ensure the global dashboard filter on Line_of_Business__c maps correctly to Claim__c.Line_of_Business__c in both the summary and detail reports to prevent filter mismatch.

5. **Validation**: Run SOQL count queries in Developer Console against Production data snapshot for all 6 LOB values and compare to dashboard tile values post-deployment; document results in the release notes.

6. **Regression test**: Add a Compliance dashboard QA checklist item to the sprint Definition of Done for any future report type changes that feed regulated dashboard components.
