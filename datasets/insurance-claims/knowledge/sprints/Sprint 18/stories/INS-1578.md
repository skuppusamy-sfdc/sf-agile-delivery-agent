## Description

As a Compliance Officer, I need every status transition on a Claim__c record to be captured in an immutable audit trail object so that I can demonstrate to state regulators and internal auditors a complete, tamper-proof history of claim lifecycle events including who made each change, when, and from which system or integration. This extends INS-1547 which established the base Claim__c status workflow. The audit trail must capture both human-initiated and system-initiated (e.g., ERA auto-adjudication, SIU referral triggers) transitions.

## Acceptance Criteria

Given any change to Claim__c.Status__c (e.g., from 'Pending' to 'Adjudicated', 'Adjudicated' to 'Appealed'),
When the record is saved by any user, Apex trigger, Flow, or integration user,
Then a Claim_Audit_Trail__c child record is created with: Claim__c (lookup), Old_Status__c, New_Status__c, Changed_By__c (User lookup), Changed_Date__c (DateTime, system-set), Change_Source__c ('User'|'Flow'|'Integration'|'Apex'), and IP_Address__c where available via UserInfo.

Given a Claim_Audit_Trail__c record is created,
When any user (including System Administrator) attempts to edit or delete it,
Then the operation is blocked by an Apex trigger with error message: 'Claim audit trail records are immutable and cannot be modified or deleted.'

Given Claim__c.Status__c transitions to 'SIU Referral',
When the Claim_Audit_Trail__c record is written,
Then Change_Source__c = 'Flow' and a related Platform Event (Claim_Status_Change__e) is published with payload fields: Claim_Id__c, New_Status__c, Timestamp__c for downstream SIU system consumption.

Given a Claims Operations Manager views a Claim__c record,
When they navigate to the 'Audit Trail' related list,
Then Claim_Audit_Trail__c records are displayed in reverse chronological order with all fields visible, and no inline edit is available on any field.

Given an ERA auto-adjudication integration updates Claim__c.Status__c = 'Adjudicated',
When the integration user (ERA_Integration_User) saves the record,
Then Claim_Audit_Trail__c.Change_Source__c = 'Integration' and Changed_By__c reflects the named integration user account.

Given more than 500 audit trail records exist on a single Claim__c,
When the related list is rendered,
Then pagination is applied and performance remains under 3 seconds load time.

## Solution

Implement an Apex Trigger (ClaimAuditTrailTrigger) on Claim__c, firing on before/after update, capturing old and new Status__c values from Trigger.oldMap and Trigger.newMap. Use a TriggerHandler pattern with a ClaimAuditTrailService class. Immutability enforced via a separate Apex Trigger on Claim_Audit_Trail__c blocking update and delete DML operations. Platform Event (Claim_Status_Change__e) published via EventBus.publish() for SIU transitions. Change_Source__c determined by inspecting the running user profile and a custom 'Integration User' custom permission. Related list displayed via a custom LWC (claimAuditTrailList) with server-side pagination using OFFSET-based SOQL. Field-level security on Claim_Audit_Trail__c enforced via Permission Set Group.
