## Description

**Bug Report**

**Summary:** During bulk adjudication batch runs that process more than 500 Claim__c records in a single execution context, Claim_Audit_Log__c records are silently dropped and no error is surfaced to the operations team. This was discovered during UAT regression testing of the INS-1607 audit trail feature.

**Steps to Reproduce:**
1. Execute the 'BulkAdjudication_Batch' Apex job with a scope of 500+ Claim__c records in a sandbox.
2. Trigger a status change from 'Under Review' to 'Adjudicated' across all records in the batch scope.
3. After batch completion, query Claim_Audit_Log__c for the affected Claim IDs.
4. Observe that only the first 200-250 Claim_Audit_Log__c records are created; the remainder are missing with no exception in the Apex Jobs log.

**Expected Behavior:** All Claim__c status changes within the batch, regardless of batch scope size, must produce a corresponding Claim_Audit_Log__c record as specified in INS-1607.

**Actual Behavior:** Claim_Audit_Log__c inserts silently fail for records beyond approximately the 200-record mark within a single batch chunk due to Platform Event publish limits being reached (2,000 events per transaction with high payload sizes causing the buffer to be exhausted). The Platform Event 'Claim_Status_Change__e' publish call returns false for overflow records but the return value is not being checked.

**Environment:** Acme Insurance Full Sandbox (CS112)
**Severity:** High — compliance audit trail is incomplete, regulatory risk.
**Detected By:** Sofia Hernandez during Sprint 18 regression cycle.

## Acceptance Criteria

As per Description. Additional verification: After fix deployment, execute the bulk adjudication batch with 1,000 records and confirm COUNT(Claim_Audit_Log__c) WHERE Claim__c IN [batch record set] = 1,000 with zero missing records. Confirm no unhandled exceptions in Apex Jobs log.

## Solution

Root cause: The Apex After Trigger on Claim__c publishes Claim_Status_Change__e Platform Events synchronously within the trigger context, and the EventBus.publish() return value (a List<Database.SaveResult>) is not inspected. When publish partially fails due to event buffer limits, failed records are silently skipped. Fix: (1) Inspect the SaveResult list from EventBus.publish() and collect failed Claim IDs. (2) For failed publishes, fall back to inserting Claim_Audit_Log__c records directly via a separate Queueable Apex job 'AuditLogFallback_Queueable' that chunks inserts into groups of 200. (3) Add a custom Big Object 'Audit_Publish_Failure__b' to log any secondary failures for forensic review. (4) Update the BulkAdjudication_Batch chunk size in the Schedulable wrapper to 200 to align with DML governor limits and reduce per-chunk event volume. Unit tests must include a @TestSetup scenario generating 600 Claim__c records to validate fix at scale.
