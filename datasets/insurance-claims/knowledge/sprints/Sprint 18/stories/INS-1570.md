## Description

As a Compliance Officer, I need a comprehensive audit trail capturing every state transition on the Claim__c record so that we can satisfy state regulatory examination requests and demonstrate adherence to prompt-pay statutes without manual reconstruction of claim history.

## Acceptance Criteria

Given a Claim__c record undergoes any Status__c change (e.g., 'Submitted' → 'In Review' → 'Adjudicated' → 'Paid'),
When the transition is saved by any user or automated process,
Then a Claim_Audit_Event__c child record must be created capturing: Claim__c (lookup), Previous_Status__c, New_Status__c, Changed_By__c (User lookup), Changed_At__c (DateTime), Change_Reason__c, and IP_Address__c.

Given a Claim__c.Status__c is set to 'Denied',
When the audit record is created,
Then Denial_Reason_Code__c must be populated from Claim__c.Denial_Reason__c and must not be null; if null, a validation error must block the save.

Given an automated process (e.g., Payment Engine batch job) triggers the status change,
When the Claim_Audit_Event__c record is written,
Then Changed_By__c must reference the named integration user 'PaymentEngine_Integration' and Source_System__c must equal 'PaymentEngine'.

Given a Compliance Officer queries audit history,
When they open the Claim 360 page,
Then a timeline LWC component must display all Claim_Audit_Event__c records in descending Changed_At__c order with no more than 200ms render time for up to 500 audit records.

Given audit records are written,
When any user (including System Administrator) attempts to edit or delete a Claim_Audit_Event__c record,
Then the operation must be blocked by an Apex-enforced restriction and a custom error message 'Audit records are immutable per compliance policy' must be returned.

-Given audit records are archived after 90 days to a BigObjects store- (superseded: retention policy updated to 7 years per state mandate; see INS-1545)

Given the volume of claims exceeds 50,000 per day,
When audit records are inserted,
Then the solution must use Platform Events (Claim_Audit_PE__e) published asynchronously to avoid adding latency to the primary claim save transaction.

## Solution

Implement an Apex Trigger (after insert, after update) on Claim__c that publishes a Claim_Audit_PE__e Platform Event containing all delta fields. A separate Platform Event-Triggered Flow subscribes to Claim_Audit_PE__e and inserts Claim_Audit_Event__c records via a Record-Create element. Immutability enforced via a before-delete and before-update Apex Trigger on Claim_Audit_Event__c that throws AuraHandledException. The Claim 360 timeline LWC uses a @wire adapter calling a custom Apex controller with SOQL on Claim_Audit_Event__c with indexed fields (Claim__c, Changed_At__c). Permission Set Group 'Compliance_ReadOnly_PSG' grants read-only access to Claim_Audit_Event__c for Compliance Officers. Deployment includes a Custom Metadata Type Audit_Config__mdt to configure retention thresholds without code changes.
