## Description

As a Compliance Officer, I need a centralized, searchable audit log viewer within Salesforce that surfaces field-level change history across Policy__c and Claim__c records so that during regulatory examinations or internal audits I can quickly retrieve a complete chronological history of data modifications without relying on Salesforce Setup > Field History Tracking exports or ad-hoc SOQL queries. The viewer must support filtering by date range, object type, field name, changed-by user, and record ID, and must display both old and new values in a human-readable format. This complements the immutable audit trail built in INS-1578 by providing a unified search interface across both standard Field History and the custom Claim_Audit_Trail__c object.

## Acceptance Criteria

Given a Compliance Officer opens the Audit Log Viewer (accessible from the Compliance app navigation),
When they enter search criteria (date range, Object = 'Claim__c', Field = 'Status__c'),
Then results are returned within 5 seconds showing all matching Claim_Audit_Trail__c records and Claim__c standard field history records, displayed in a unified table with columns: Record_ID, Object_Type, Field_Name, Old_Value, New_Value, Changed_By, Changed_Date.

Given the user filters by Changed_By user and a date range spanning 90 days,
When the query executes,
Then results are paginated at 50 records per page with total result count displayed, and the user can export results to CSV.

Given a Policy__c record ID is entered as a filter,
When the search runs,
Then all field history entries from PolicyHistory (standard Salesforce field history) for that Policy__c are returned alongside any custom audit records, with Object_Type clearly labeled.

Given the audit log viewer is accessed by a user without the 'Compliance_Audit_Viewer' custom permission,
When they attempt to load the component,
Then the component renders a permission-denied message and no audit data is exposed.

-Given the viewer also surfaces Setup Audit Trail (login history, permission changes) via Salesforce Tooling API, When queried, Then setup changes by administrators are included in results.- (Descoped — Setup Audit Trail access requires additional security review; deferred to Sprint 20)

## Solution

Build a custom LWC (complianceAuditLogViewer) embedded in a dedicated 'Compliance Audit' tab within the Compliance App. The component uses a dual-source Apex controller (AuditLogSearchController) that queries Claim_Audit_Trail__c via standard SOQL and Policy__c field history via the standard PolicyHistory object (queryable via SOQL). Results are merged and sorted client-side by Changed_Date. Search filters implemented as reactive LWC properties bound to a filter panel. CSV export uses a ContentVersion Blob approach. Access gated by a 'Compliance_Audit_Viewer' custom permission checked via FeatureManagement.checkPermission() in the Apex controller. Pagination implemented server-side using LIMIT/OFFSET SOQL with total count from a COUNT() aggregate query.
