## Description

As a Claims Operations Manager, I need a pre-disbursement compliance validation checklist on the Explanation_of_Benefits__c (EOB) record so that every payment disbursement is blocked until all mandatory regulatory fields are confirmed complete, reducing the risk of improper payments and state DOI penalties. This story extends INS-1558 which established the base Payment__c disbursement workflow.

## Acceptance Criteria

Given an EOB record (Explanation_of_Benefits__c) is associated with a Payment__c record in Payment__c.Status__c = 'Approved for Disbursement',
When a Claims Operations Manager attempts to advance Payment__c.Status__c to 'Disbursed',
Then the system must execute a compliance checklist validation confirming all of the following fields are populated on the EOB: EOB_Issue_Date__c, Member_ID__c, Provider_NPI__c, Provider_TIN__c, Service_Date_From__c, Service_Date_To__c, Total_Billed_Amount__c, Total_Allowed_Amount__c, Total_Member_Responsibility__c, Denial_Reason_Code__c (if any Claim_Line__c.Adjudication_Status__c = 'Denied').

Given the compliance checklist validation fails (one or more required fields missing),
When the disbursement attempt is made,
Then the system must block the status transition, display a validation error listing each missing field by name, and create a Compliance_Hold__c record linked to Payment__c with Hold_Reason__c = 'EOB Incomplete — Pre-Disbursement Check' and Hold_Date__c = today.

Given all checklist validations pass,
When Payment__c.Status__c transitions to 'Disbursed',
Then EOB_Compliance_Verified__c (checkbox) must be stamped true, EOB_Verified_By__c must capture the running user, and EOB_Verified_DateTime__c must capture the UTC timestamp.

Given a Compliance_Hold__c record exists on a Payment__c,
When the Claims Operations Manager resolves the hold and re-attempts disbursement,
Then the checklist must re-execute from scratch (not cache prior results).

-Given COB (Coordination of Benefits) scenarios, a secondary payer EOB is also validated simultaneously- (deprecated — COB EOB validation deferred to Sprint 20 per product decision)

Given an EOB fails validation 3 or more times for the same Payment__c,
When the third failure occurs,
Then an escalation Task must be auto-assigned to the Claims Operations Manager's supervisor (User.ManagerId) with Subject = 'Escalation: Repeated EOB Compliance Failures — [Claim Number]' and Priority = 'High'.

## Solution

Implement a before-update Apex Trigger on Payment__c that invokes EOBComplianceValidationService.validate(paymentId) when Status__c changes to 'Disbursed'. The service queries the related Explanation_of_Benefits__c and evaluates the required field list defined in a Custom Metadata Type (EOB_Required_Field__mdt) for maintainability. On failure, throw an AuraHandledException with a structured error payload listing missing fields. Use a Record-Triggered Flow (after update) on Payment__c to create Compliance_Hold__c and escalation Tasks, keeping DML out of the Apex trigger context. Stamp EOB_Compliance_Verified__c, EOB_Verified_By__c, and EOB_Verified_DateTime__c via the Flow's Update Records element. Track failure count using Payment__c.EOB_Validation_Failure_Count__c (Number field, auto-incremented by Flow).
