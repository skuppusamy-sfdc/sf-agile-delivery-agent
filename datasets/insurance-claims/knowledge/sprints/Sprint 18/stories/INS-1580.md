## Description

**Bug Report — EOB PDF Generation Failure for High-Volume Claim Lines**

**Environment:** Production (also reproducible in Staging)
**Reported By:** Amanda Foster (Claims Operations Manager)
**Severity:** High — impacts facility claims with complex surgical procedures routinely exceeding 25 CPT line items

**Steps to Reproduce:**
1. Navigate to a Claim__c record where the count of related Claim_Line__c records exceeds 25.
2. Click the 'Generate EOB' action button on the Claim__c record.
3. The Conga CLM template merge is initiated via the GenerateEOBController Apex class.
4. Observe the result after approximately 8–12 seconds.

**Expected Behavior:**
An Explanation_of_Benefits__c record is created with Status__c = 'Generated', and a PDF rendition is attached as a ContentVersion linked to both the EOB and parent Claim__c records. All Claim_Line__c records (regardless of count) are rendered in the EOB line detail table.

**Actual Behavior:**
An unhandled Apex exception is thrown: 'System.LimitException: Apex CPU time limit exceeded' in GenerateEOBController.generateEOBDocument() at line 187. No Explanation_of_Benefits__c record is created. The user sees a generic error toast: 'An error occurred while generating the EOB. Please contact your administrator.' Claims Operations staff are manually producing EOBs in Excel for affected facility claims as a workaround, introducing compliance risk.

**Root Cause Hypothesis:**
The GenerateEOBController performs a nested SOQL query inside a for-loop iterating over Claim_Line__c records to fetch related Fee_Schedule__c pricing data. With 25+ lines, this results in 25+ SOQL queries within a single transaction, consuming CPU budget before the Conga merge call is made.

**Affected Records:** Approximately 340 Claim__c records in Production with 25+ Claim_Line__c records. Claim__c.LOB__c = 'Facility' accounts for ~90% of affected claims.

**Regression Note:** This worked correctly prior to the Sprint 16 deployment of the Fee_Schedule__c lookup enhancement (INS-1521).

## Acceptance Criteria

As per Description. Fix must be verified against a Claim__c record with exactly 25, 50, and 100 Claim_Line__c records in Staging before promotion to Production.

## Solution

Refactor GenerateEOBController to bulkify the Fee_Schedule__c lookup: collect all Fee_Schedule__c IDs from Claim_Line__c records in a Set, execute a single SOQL query outside the loop, and use a Map<Id, Fee_Schedule__c> for O(1) lookups within the iteration. If CPU time remains a concern at 100+ lines, move the Conga merge invocation to an asynchronous Queueable Apex job (EOBGenerationQueueable) triggered from the controller, with a Platform Event (EOB_Generation_Complete__e) notifying the UI upon completion. Add a SOQL governor limit check (Limits.getQueries()) as a defensive guard with a graceful user-facing error if thresholds are approached. Unit tests must cover bulk scenarios with 200 Claim_Line__c records.
