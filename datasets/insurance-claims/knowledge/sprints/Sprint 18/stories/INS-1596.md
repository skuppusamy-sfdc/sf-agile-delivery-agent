## Description

As a Compliance Officer, I need every change to Fee_Schedule__c and its child Fee_Schedule_Line__c records to be captured with full version history, effective dates, and the identity of the approving user so that Acme Insurance can demonstrate to state regulators and CMS that provider reimbursement rates were applied correctly for any given date of service during an audit lookback period.

## Acceptance Criteria

Given a Fee_Schedule__c record is modified (any field), When the change is committed, Then a Fee_Schedule_Version__c child record is created capturing: Fee_Schedule_Version__c.Fee_Schedule__c (lookup), Fee_Schedule_Version__c.Version_Number__c (auto-incremented integer), Fee_Schedule_Version__c.Snapshot_JSON__c (serialized JSON of all Fee_Schedule__c field values at time of change), Fee_Schedule_Version__c.Changed_By__c, Fee_Schedule_Version__c.Change_Timestamp__c, Fee_Schedule_Version__c.Approved_By__c, Fee_Schedule_Version__c.Change_Reason__c (required text field).

Given a Fee_Schedule_Line__c record is created, updated, or deleted (CPT code rate changes), When the DML operation is committed, Then the parent Fee_Schedule__c.Last_Modified_Reason__c is stamped and a new Fee_Schedule_Version__c snapshot is triggered capturing the full state of all child Fee_Schedule_Line__c records as a JSON array in Fee_Schedule_Version__c.Line_Items_Snapshot_JSON__c.

Given a Claim_Line__c is adjudicated and Claim_Line__c.Allowed_Amount__c is calculated by the Fee Schedule Engine, When the calculation occurs, Then Claim_Line__c.Applied_Fee_Schedule_Version__c (lookup to Fee_Schedule_Version__c) is populated with the version active on Claim_Line__c's parent Claim__c.Date_of_Service__c, ensuring point-in-time rate auditability.

Given a Compliance Officer needs to audit a historical rate, When they query 'What rate was applied for CPT 99213 under Fee Schedule FS-0042 on 2024-03-15?', Then the Fee Schedule Version History LWC on the Fee_Schedule__c record page displays a timeline view of all Fee_Schedule_Version__c records, and selecting a version renders the full Fee_Schedule_Line__c snapshot as of that date.

-Given fee schedule changes are approved via email- (deprecated: replaced by Salesforce Approval Process with two-tier approval: Provider Network Specialist → Compliance Officer)

Given a Fee_Schedule__c change is submitted for approval, When the Approval Process is triggered, Then Fee_Schedule__c.Status__c transitions to 'Pending Approval' and the record is locked against further edits until the approval is granted or rejected.

## Solution

Apex Trigger (FeeScheduleTrigger) on Fee_Schedule__c captures before-image using a static map pattern and inserts Fee_Schedule_Version__c after-save with JSON.serialize() for snapshot fields. A separate trigger on Fee_Schedule_Line__c calls a parent-rollup Apex class to regenerate the line items snapshot. Custom auto-number field on Fee_Schedule_Version__c for Version_Number__c. The Fee Schedule Engine (FeeScheduleCalculationService Apex class) is updated to perform a SOQL query against Fee_Schedule_Version__c filtered by effective date range to resolve the correct version for a given date of service. LWC 'feeScheduleVersionTimeline' uses a custom Apex controller returning versioned data. Approval Process configured with record lock via standard Salesforce approval settings. Permission Set 'Fee_Schedule_Approver' granted to Compliance Officer profile.
