## Description

**Bug Report — Extends audit trail work from INS-1555**

**Steps to Reproduce:**
1. Log in as a Claims Operations Manager (non-integration user).
2. Open any Explanation_of_Benefits__c (EOB) record where EOB_Status__c = 'Draft'.
3. Manually update the Allowed_Amount__c field and save the record.
4. Navigate to the related EOB_Audit_Event__c list.
5. Observe the Changed_By__c field on the newly created audit record.

**Expected Behavior:** Changed_By__c should reference the Claims Operations Manager's User record who performed the manual edit.

**Actual Behavior:** Changed_By__c is stamped with the Integration Service Account user ('svc_claims_integration@acmeins.com') regardless of which user made the change. This causes compliance failures during DOI audits because manual adjustments appear to be system-generated, obscuring human accountability.

**Environment:** Production sandbox (Partial Copy). Reproducible 100% of the time for manual edits on EOB records. Integration-triggered updates correctly stamp the integration user (that behavior is correct and should not be broken).

**Root Cause Hypothesis:** The Platform Event subscriber Apex class (ClaimAuditEventSubscriber) that processes Claim_State_Change__e events likely runs in the context of the integration user who published the event, not the user who triggered the originating Flow. The UserInfo.getUserId() call inside the subscriber is capturing the wrong execution context.

## Acceptance Criteria

As per Description. Additionally: Changed_By__c on EOB_Audit_Event__c must reflect the actual initiating user for both manual UI edits and Flow-triggered updates. Integration user attribution must be preserved only when the originating action is a genuine API/integration call as determined by Change_Source__c = 'Integration'.

## Solution

Fix: Modify the Record-Triggered Flow on Explanation_of_Benefits__c to capture {!$User.Id} at flow execution time and include it as a field (Initiating_User_Id__c) in the published Claim_State_Change__e Platform Event payload — before the async handoff loses execution context. Update the ClaimAuditEventSubscriber Apex class to read Initiating_User_Id__c from the event payload instead of calling UserInfo.getUserId() (which returns the automated process user in async context). Add a null-check: if Initiating_User_Id__c is blank on the event, fall back to UserInfo.getUserId() to preserve integration user attribution. Deploy updated Flow and Apex class together. Add unit test assertions verifying Changed_By__c for both manual-user and integration-user scenarios with System.runAs().
