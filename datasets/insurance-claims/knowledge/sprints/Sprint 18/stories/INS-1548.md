## Description

As a Compliance Officer, I need a tamper-evident audit trail capturing every status transition on Claim__c records so that we can satisfy state DOI examination requests and demonstrate adherence to prompt-pay regulations without manual reconstruction of claim history.

## Acceptance Criteria

Given a Claim__c record transitions from any status to another (e.g., 'Submitted' → 'In Review' → 'Adjudicated' → 'Paid'),
When the status change is saved,
Then a Claim_Audit_Event__c child record must be created containing: Claim__c (lookup), Previous_Status__c, New_Status__c, Changed_By__c (User lookup), Change_Timestamp__c (datetime, UTC), Change_Reason__c, and IP_Address__c of the acting user.

Given a Compliance Officer opens the Claim 360 page,
When they view the Audit Trail component,
Then all Claim_Audit_Event__c records for that claim are displayed in reverse-chronological order with no ability to edit or delete any record (OWD = Private, no delete permission in Permission Set Group).

Given a Claim_Audit_Event__c record is created,
When a malicious or accidental DML delete is attempted by any non-System Admin profile,
Then the operation must be blocked by an Apex Trigger (before delete) and a platform-level restriction, and an error message 'Audit records cannot be deleted' is surfaced.

Given the Claim__c.Status__c is set to 'Void',
When the transition is saved,
Then Claim_Audit_Event__c.Change_Reason__c must be populated (required field enforced via validation rule) and Claim__c.Void_Authorized_By__c must reference a user with the 'Claims_Supervisor' Permission Set.

Given a bulk adjudication batch job processes more than 500 claims in a single execution,
When audit records are generated,
Then all Claim_Audit_Event__c records must be inserted via Platform Event (Claim_Status_Changed__e) to avoid governor limit breaches, with a subscriber Apex trigger persisting them asynchronously.

-Given audit records are exported nightly to an S3 bucket via MuleSoft- (deprecated; replaced by Salesforce Shield Event Monitoring export in INS-1552).

## Solution

Implement a Record-Triggered Flow on Claim__c (fires on update, before-save to capture old values via {!$Record__Prior}) that publishes a Claim_Status_Changed__e Platform Event containing all required audit fields. An Apex Trigger subscriber on the Platform Event inserts Claim_Audit_Event__c records in a separate transaction, bypassing the originating transaction's governor limits. A custom Permission Set Group 'Compliance_ReadOnly' grants Read access to Claim_Audit_Event__c with no Create/Edit/Delete. Apex before-delete trigger on Claim_Audit_Event__c throws a System.AuraHandledException for all non-System-Admin profiles. An LWC 'claimAuditTimeline' is added to the Claim__c Lightning Record Page, querying audit events via @wire and rendering them in a sorted lightning-datatable.
