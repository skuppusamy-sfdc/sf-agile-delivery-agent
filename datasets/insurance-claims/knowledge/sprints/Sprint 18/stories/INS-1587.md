## Description

**Bug Report**

**Summary:** When an EOB (Explanation_of_Benefits__c) document is regenerated after an initial generation (e.g., due to a claim correction or COB adjustment), the second and subsequent generation events are not recorded in the EOB_Audit_Trail__c object, leaving a compliance gap.

**Steps to Reproduce:**
1. Navigate to an Adjudicated Claim__c record with an existing Explanation_of_Benefits__c child record.
2. Click 'Regenerate EOB' button (triggers Conga Composer template via custom LWC action).
3. Observe that EOB_Audit_Trail__c shows only the original generation event; the regeneration event is absent.
4. Repeat regeneration a second time — still only one EOB_Audit_Trail__c record exists.

**Expected Behavior:** Every EOB generation or regeneration event should create a new EOB_Audit_Trail__c record with Generation_Type__c = 'Initial' or 'Regeneration', Generated_By__c (User), Generation_Timestamp__c, and Trigger_Reason__c.

**Actual Behavior:** EOB_Audit_Trail__c.Generation_Type__c is only populated on initial generation. The Apex method `EOBGenerationService.generateEOB()` contains a conditional guard `if (existingAuditRecord == null)` that prevents subsequent audit writes.

**Root Cause (suspected):** EOBGenerationService.cls line ~87 — null-check guard on existingAuditRecord bypasses audit insert for regeneration scenarios.

**Affected Environments:** Full Sandbox, Production
**Regression introduced in:** INS-1541 (EOB auto-generation feature)

## Acceptance Criteria

As per Description. Fix must ensure EOB_Audit_Trail__c records are created for every generation event regardless of prior records existing. EOB_Audit_Trail__c.Generation_Type__c must correctly distinguish 'Initial' vs 'Regeneration'. All existing unit tests in EOBGenerationServiceTest.cls must pass; new test method must cover regeneration scenario with ≥ 75% code coverage on modified lines.

## Solution

1. In EOBGenerationService.cls, remove the `if (existingAuditRecord == null)` guard around the EOB_Audit_Trail__c insert DML. 2. Replace with logic that always constructs a new EOB_Audit_Trail__c record; set Generation_Type__c = 'Initial' if no prior records exist for the Claim__c, else 'Regeneration'. 3. Add Trigger_Reason__c population from the calling context parameter (already passed but unused). 4. Write new @isTest method `testEOBRegenerationAuditTrail()` asserting two EOB_Audit_Trail__c records exist after two generateEOB() calls. 5. Deploy fix to Full Sandbox for QA regression before Production push.
