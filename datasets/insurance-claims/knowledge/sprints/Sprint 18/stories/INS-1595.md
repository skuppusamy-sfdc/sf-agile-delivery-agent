## Description

**Bug Report — Audit Trail Gap for ERA Batch Updates**

**Environment:** Full Sandbox (CS-07)
**Discovered by:** Priya Patel during INS-1592 UAT

**Steps to Reproduce:**
1. Trigger the ERA inbound batch job (ERAInboundBatchProcessor) via the scheduled job or manual Execute Anonymous invocation.
2. Allow the batch to process a set of 835 ERA files, updating Claim__c.Status__c from 'Adjudicated' to 'Paid' and populating Payment__c records.
3. After batch completion, navigate to any Claim__c record updated by the batch.
4. Open the 'Audit Trail' LWC tab (INS-1592).
5. Observe Claim_Audit_Log__c entries for the Status__c change from 'Adjudicated' to 'Paid'.

**Expected Result:** A Claim_Audit_Log__c record exists for each Claim__c updated by the ERA batch, with Claim_Audit_Log__c.Change_Source__c = 'Integration' and Claim_Audit_Log__c.Changed_By__c = the ERA Integration User.

**Actual Result:** No Claim_Audit_Log__c records are created for claims updated via the ERA batch. Audit trail is completely absent for ERA-sourced payment postings, creating a compliance gap for CMS audit purposes.

**Root Cause Hypothesis:** The Record-Triggered Flow on Claim__c that publishes Claim_Audit_Event__e Platform Events is configured with 'Run Asynchronously' and appears to be suppressed when the context user is the ERA_Integration_User profile, likely due to a missing Permission Set assignment or the batch context exceeding Platform Event publish limits (150,000/hour) during high-volume ERA processing windows.

**Impact:** HIGH — compliance audit trail is incomplete for all ERA-processed claims since Sprint 17 deployment (approx. 14,200 claims affected in Full Sandbox).

## Acceptance Criteria

As per Description. Verify fix by re-running ERAInboundBatchProcessor in CS-07 sandbox against a 500-claim ERA file and confirming 500 corresponding Claim_Audit_Log__c records are created with Change_Source__c = 'Integration'. Confirm no Platform Event governor limit errors appear in debug logs.

## Solution

Investigate two parallel root causes: (1) Assign the 'Claim_Audit_Platform_Events' Permission Set to the ERA_Integration_User profile to ensure Platform Event publish access. (2) Refactor ERAInboundBatchProcessor to directly call an Apex invocable method 'CreateClaimAuditLog' as a fallback within the execute() method, bypassing the Record-Triggered Flow for batch contexts and directly inserting Claim_Audit_Log__c records. This dual-path approach ensures audit coverage regardless of Platform Event availability. Add a Batch_Context__c field to Claim_Audit_Log__c to flag records created via direct Apex vs. Platform Event path. Backfill script to be run by Kevin Nguyen for the 14,200 affected claims using a one-time Apex Execute Anonymous script.
