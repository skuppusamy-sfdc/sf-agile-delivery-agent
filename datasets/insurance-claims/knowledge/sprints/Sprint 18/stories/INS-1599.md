## Description

As a Compliance Officer, I need a comprehensive audit trail that captures every status transition on Claim__c and Claim_Line__c records so that we can demonstrate regulatory compliance during state DOI examinations and internal audits without relying solely on Salesforce native field history tracking.

## Acceptance Criteria

Given a Claim__c record undergoes any status transition (e.g., Claim__c.Status__c changes from 'Pending' to 'Under Review', 'Under Review' to 'Adjudicated', or 'Adjudicated' to 'Reopened'), When the record is saved, Then a Compliance_Audit_Event__c record must be created with fields: Claim__c (lookup), Old_Status__c, New_Status__c, Changed_By__c (User lookup), Change_Timestamp__c (datetime), Change_Reason__c, and LOB__c populated from Claim__c.Line_of_Business__c.

Given a Claim_Line__c record has its Adjudication_Status__c or Allowed_Amount__c modified by any user or automated process, When the DML operation completes, Then a corresponding Compliance_Audit_Event__c child record must be created referencing both Claim_Line__c and its parent Claim__c.

Given an audit trail record is created, When a Compliance Officer views the Claim__c record page, Then a 'Compliance Audit Trail' tab on the Lightning record page must display all Compliance_Audit_Event__c records in reverse chronological order with user name, old value, new value, timestamp, and change reason visible without requiring additional clicks.

Given the audit trail volume, When records are older than 7 years, Then an archival process must flag Compliance_Audit_Event__c.Archive_Eligible__c = true and publish a Platform Event (Audit_Archive_Event__e) for downstream processing — -records must be hard deleted after 7 years per original spec superseded by legal hold requirement-.

Given a TPA-submitted claim adjudication, When the status change originates via the Claims API integration user, Then Compliance_Audit_Event__c.Source__c must be set to 'TPA Integration' rather than the generic API user name to preserve meaningful audit context.

## Solution

Implement an After-Save Record-Triggered Flow on Claim__c to detect Status__c changes and create Compliance_Audit_Event__c records via a Create Records element. Use a separate Apex Trigger (ClaimLineAuditTrigger) on Claim_Line__c for bulk-safe handling of Claim_Line__c field changes, leveraging a ClaimLineAuditService Apex class with @InvocableMethod support. Publish Audit_Archive_Event__e Platform Events via Apex for the 7-year archival flag process. Build an LWC (complianceAuditTrail) embedded in the Claim__c Lightning record page using a @wire adapter against a custom AuditTrailController Apex class with SOQL ordering by Change_Timestamp__c DESC. Apply a Permission Set Group 'Compliance_Readonly' to restrict Compliance_Audit_Event__c deletion to System Administrators only. Add a Custom Metadata Type (Audit_Config__mdt) to externalize field-tracking configuration without code deployments.
