## Description

**Bug Report — Agentforce Triage Silent Failure on Dental LOB**

**Environment:** Acme Insurance Full Sandbox (FS1)
**Detected by:** Emily Watson (QA) during Sprint 14 regression cycle
**Related Story:** INS-1282

**Steps to Reproduce:**
1. Navigate to the Claims tab and create a new Claim__c record.
2. Set Claim__c.LOB__c = 'Dental' and Claim__c.Stage__c = 'FNOL Received'.
3. Populate all required fields (Policy__c lookup, Claimant__c, Date_of_Loss__c) and save the record.
4. Wait 60 seconds and refresh the record page.
5. Observe Claim__c.AI_Severity_Score__c, Claim__c.AI_Triage_Timestamp__c, and Claim__c.AI_Triage_Status__c.

**Expected Behavior:**
The Agentforce Triage Agent should invoke within 30 seconds of record creation (per INS-1282 ACs), populate Claim__c.AI_Severity_Score__c with a valid severity tier, and set Claim__c.AI_Triage_Status__c to 'Completed' or 'Failed - Manual Review Required' on timeout.

**Actual Behavior:**
All three fields remain null after 60+ seconds. No Platform Event (Claim_Triage_Alert__e) is published. No Apex exception is logged in Setup > Apex Jobs or the custom AI_Audit_Log__c object. The claim silently bypasses triage and proceeds to the standard adjudication queue as if triage was never attempted. The issue is 100% reproducible for LOB__c = 'Dental' and does not occur for 'Medical', 'Vision', or 'Workers Compensation' LOBs.

**Root Cause Hypothesis:**
The AgentforceTriage_Queueable job contains a LOB-to-AgentTopic routing Map. The 'Dental' key is absent from the Map, causing the queueable to exit the routing block without executing the callout or setting AI_Triage_Status__c to the failure state. No exception is thrown because the missing-key path has no else/catch branch.

**Impact:**
Dental claims are entering the adjudication queue without severity scoring, violating the SLA routing rules defined in INS-1282. Approximately 340 Dental FNOL claims per day are affected in production-equivalent volume testing.

## Acceptance Criteria

As per Description. Fix must ensure: (1) 'Dental' LOB is added to the AgentTopic routing Map with value 'FNOL_Severity_Triage_Dental'; (2) a defensive else branch sets Claim__c.AI_Triage_Status__c = 'Failed - Manual Review Required' and logs to AI_Audit_Log__c for any unmapped LOB value; (3) regression tests pass for all existing LOBs (Medical, Vision, Workers Compensation, Dental, Property).

## Solution

Fix the AgentforceTriage_Queueable Apex class: (1) Add 'Dental' → 'FNOL_Severity_Triage_Dental' entry to the LOB_TO_TOPIC_MAP static final Map. (2) Wrap the map lookup in a containsKey check; if the LOB is not in the map, set AI_Triage_Status__c = 'Failed - Manual Review Required', insert an AI_Audit_Log__c record with error detail 'Unmapped LOB: [value]', and return early without throwing an unhandled exception. (3) Deploy the 'FNOL_Severity_Triage_Dental' AgentTopic in Agentforce Agent Builder (mirrors Medical topic with Dental-specific CPT/ADA code context). (4) Add an Apex test method in AgentforceTriageQueueable_Test covering the Dental LOB path and the unmapped LOB defensive branch, targeting ≥ 90% class coverage. (5) Backfill affected Claim__c records in FS1 via an anonymous Apex script that re-enqueues triage for records where LOB__c = 'Dental' AND AI_Triage_Status__c = null AND Stage__c != 'Closed'.
