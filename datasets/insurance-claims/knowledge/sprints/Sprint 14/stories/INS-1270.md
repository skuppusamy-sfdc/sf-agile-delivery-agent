## Description

**Bug Report – COB Priority Field Not Saving via OmniScript**

**Steps to Reproduce:**
1. Open an existing Claim__c record with Claim__c.COB_Flag__c = true.
2. Launch the 'COB_Intake_OmniScript' from the Claim__c record page action button.
3. On Step 3 ('Secondary Payer Details'), populate the Secondary_Payer_Name, Secondary_Policy_Number, and set COB_Priority to 'Secondary'.
4. Complete the OmniScript and click 'Submit'.
5. Navigate back to the Claim__c record.

**Expected Result:** Claim__c.COB_Priority__c = 'Secondary' and Claim__c.Secondary_Payer_Name__c reflect the values entered in the OmniScript.

**Actual Result:** Claim__c.COB_Priority__c remains null or retains its previous value. Claim__c.Secondary_Payer_Name__c is correctly updated. The field save silently fails with no user-facing error.

**Environment:** Acme Insurance – Full Sandbox (Sprint 14 refresh)
**Frequency:** Reproducible 100% of the time when COB_Priority is changed from null to any picklist value.
**Impact:** Claims Operations staff are manually correcting COB_Priority__c post-submission, introducing data integrity risk and delaying COB adjudication logic downstream.

**Suspected Cause:** The DataRaptor Transform feeding the OmniScript 'Submit' action maps COB_Priority to a formula field alias rather than the API field name Claim__c.COB_Priority__c. Possible field API name mismatch introduced during the Sprint 13 schema migration (INS-1248).

## Acceptance Criteria

As per Description.

Additionally: After fix is deployed, regression test must confirm that all five COB_Priority__c picklist values ('Primary', 'Secondary', 'Tertiary', 'Unknown', 'Not Applicable') persist correctly when set via the OmniScript.

No Claim__c records with COB_Flag__c = true should have COB_Priority__c = null after the OmniScript submission completes successfully.

## Solution

Inspect the DataRaptor Transform 'DRT_COB_Intake_SaveClaim' — correct the Output field mapping for COB Priority from the erroneous formula alias ('COB_Priority_Formula__c') to the correct writable field API name ('COB_Priority__c'). Validate that the DataRaptor Output JSON structure matches the Claim__c SObject field API names exactly. After correction, re-activate the DataRaptor and re-publish the COB_Intake_OmniScript. Run a data fix script (anonymous Apex) to backfill COB_Priority__c on the estimated 340 affected Claim__c records in production using audit log data from the OmniScript session logs. Deploy fix to Full Sandbox first, execute regression suite, then promote via change set.
