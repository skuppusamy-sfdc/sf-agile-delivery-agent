## Description

As a Claims Operations Manager, I need inbound ERA (Electronic Remittance Advice) files in ANSI X12 835 format to be automatically ingested, parsed, and reconciled against open Payment__c records in Salesforce so that payment posting is automated, manual ERA entry is eliminated, and claim balances are updated in real time. ERA files will be delivered by payers to an SFTP endpoint and must be processed within a defined SLA.

## Acceptance Criteria

Given an ERA file in ANSI X12 835 format is deposited to the configured SFTP endpoint,
When the scheduled MuleSoft integration job polls the SFTP every 15 minutes,
Then the file is ingested and a Platform Event (ERA_File_Received__e) is published to Salesforce containing the raw 835 payload and a unique ERA_File_Reference__c identifier.

Given the ERA_File_Received__e Platform Event is consumed by Salesforce,
When the ERA parsing Apex class (ERA835ParserService.cls) processes the payload,
Then for each CLP segment, a ERA_Transaction__c record is created with fields: Payer_Claim_Control_Number__c, Patient_Control_Number__c, Claim_Payment_Amount__c, Claim_Status_Code__c, and Payer_ID__c.

Given an ERA_Transaction__c record is created,
When the reconciliation engine attempts to match it to an existing Payment__c,
Then the match is performed on Payment__c.Claim__r.Claim_Number__c = ERA_Transaction__c.Patient_Control_Number__c AND Payment__c.Status__c = 'Pending'.

Given a successful match between ERA_Transaction__c and Payment__c,
When reconciliation completes,
Then Payment__c.Status__c = 'Posted', Payment__c.ERA_Reference__c = ERA_Transaction__c.ERA_File_Reference__c, Payment__c.Posted_Amount__c = ERA_Transaction__c.Claim_Payment_Amount__c, and the parent Claim__c.Balance_Due__c is recalculated.

Given no matching Payment__c is found for an ERA_Transaction__c,
When the reconciliation engine exhausts match attempts,
Then ERA_Transaction__c.Reconciliation_Status__c = 'Unmatched' and a Case is created in the 'ERA Exceptions' queue with the ERA transaction details attached.

Given ERA_Transaction__c.Claim_Status_Code__c = '4' (Denial),
When the denial is detected during reconciliation,
Then Claim__c.Stage__c = 'Denied', EOB__c is generated with denial reason code populated from ERA CAS segment, and a denial workflow task is created for the assigned claims examiner.

-Given ERA files may arrive in HL7 FHIR format- (deprecated — FHIR ERA format deferred to Phase 3 per architecture decision ADR-042).

## Solution

Implement a MuleSoft SFTP Listener flow that polls the ERA SFTP endpoint every 15 minutes, reads 835 files, and publishes a Platform Event (ERA_File_Received__e) to Salesforce via the Salesforce Connector. On the Salesforce side, a Platform Event-Triggered Flow subscribes to ERA_File_Received__e and invokes an Apex action (ProcessERAFileAction.cls) which calls ERA835ParserService.cls. The parser uses a custom X12 835 parsing library (deployed as an Apex utility class) to extract CLP/CAS/SVC segments and bulk-inserts ERA_Transaction__c records. Reconciliation is handled by a second Apex Batch class (ERAReconciliationBatch.cls) triggered by the Platform Event Flow after parsing. Denial processing triggers a Record-Triggered Flow on Claim__c when Stage__c changes to 'Denied'. EOB generation reuses the EOB generation service from INS-1215. All file processing activity logged to ERA_Processing_Log__c with file reference, record counts, and exception details.
