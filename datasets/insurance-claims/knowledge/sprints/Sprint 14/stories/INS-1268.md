## Description

As a Claims Operations Manager, I need the system to automatically generate ANSI X12 835 Electronic Remittance Advice files for all adjudicated claims and transmit them to the billing provider's preferred delivery endpoint, so that providers receive timely ERA notifications that reconcile against their submitted claims and reduce manual payment inquiries to Provider Relations.

## Acceptance Criteria

Given a Claim__c record transitions to Claim__c.Status__c = 'Adjudicated' and Payment__c.Status__c = 'Issued',
When the nightly ERA generation batch job executes (scheduled 11:00 PM ET),
Then an ERA_File__c record is created linked to Payment__c with ERA_File__c.Format__c = '835', ERA_File__c.Generated_Date__c = TODAY(), and ERA_File__c.Status__c = 'Pending Transmission'.

Given ERA_File__c.Status__c = 'Pending Transmission',
When the file is transmitted to the provider's endpoint defined in Provider__c.ERA_Delivery_Method__c ('SFTP' | 'Clearinghouse' | 'Direct'),
Then ERA_File__c.Status__c = 'Transmitted' and ERA_File__c.Transmission_Timestamp__c is stamped.

Given a Provider__c record has Provider__c.ERA_Delivery_Method__c = null or Provider__c.NPI__c is blank,
When ERA generation is attempted,
Then the ERA_File__c record is created with ERA_File__c.Status__c = 'On Hold – Missing Provider Config' and a Case is opened against the Provider__c record with Subject = 'ERA Delivery Configuration Incomplete'.

Given Claim__c contains multiple Claim_Line__c records,
When the 835 file is generated,
Then each Claim_Line__c.Adjudicated_Amount__c, Claim_Line__c.CPT_Code__c, Claim_Line__c.Denial_Reason_Code__c, and Claim_Line__c.Allowed_Amount__c are reflected in the corresponding CLP/SVC loop segments of the 835 file.

Given ERA transmission fails due to SFTP timeout or clearinghouse rejection,
When the transmission error is received,
Then ERA_File__c.Status__c = 'Transmission Failed', ERA_File__c.Error_Detail__c is populated, a Platform Event (ERA_Transmission_Failed__e) fires, and the file is queued for retry within 4 hours (max 3 retries).

## Solution

Implement a scheduled Apex batch class (GenerateERABatch.cls, runs nightly) that queries Payment__c records joined to adjudicated Claim__c records and constructs 835 EDI strings using a custom EDI835Builder utility class. ERA_File__c is the Salesforce record representing each file. Transmission logic is handled by a separate queueable Apex class (TransmitERAQueueable.cls) that calls a named credential-secured external service (clearinghouse API or SFTP via MuleSoft integration middleware). Retry logic is managed via a Record-Triggered Flow on ERA_File__c watching Status__c = 'Transmission Failed' and scheduling a Platform Event replay. A Permission Set Group 'ERA_Operations' controls visibility of ERA_File__c records. Monitoring dashboard built as a CRM Analytics (CRMA) lens on ERA_File__c.
