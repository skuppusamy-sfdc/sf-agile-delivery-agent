## Description

**Bug Report**

**Environment:** Full Sandbox (Sprint 14)
**Reported by:** Carlos Rodriguez (Provider Network Specialist)

**Summary:** Provider_Change_Notification__c (PCN) records with PCN_Type__c = 'Address Change' are not being routed to the 'Provider_Data_Maintenance_Queue' as expected. Instead, they are landing in the default 'Provider_General_Queue', causing SLA breaches for address update processing.

**Steps to Reproduce:**
1. Create a Provider_Change_Notification__c record with PCN_Type__c = 'Address Change' and Status__c = 'New'.
2. Save the record.
3. Navigate to the 'Provider_Data_Maintenance_Queue' list view.
4. Observe the new PCN record is NOT present in the queue.
5. Navigate to 'Provider_General_Queue' – the record IS present there.

**Expected Behavior:** PCN records with PCN_Type__c = 'Address Change' or 'NPI Change' or 'TIN Change' should route to 'Provider_Data_Maintenance_Queue'. PCN records with PCN_Type__c = 'Termination Request' should route to 'Network_Termination_Queue'.

**Actual Behavior:** All PCN record types are routing to 'Provider_General_Queue' regardless of PCN_Type__c value.

**Root Cause (suspected):** The Assignment Rule 'PCN_Routing_Assignment_Rule' has an entry order conflict introduced during the Sprint 13 hotfix for INS-1214. The catch-all rule entry (PCN_Type__c != NULL → Provider_General_Queue) was accidentally moved to position 1, overriding all specific type-based entries.

**Impact:** MEDIUM – Approximately 47 unrouted PCN records since Sprint 13 deployment. Provider data update SLA of 3 business days is being breached.

## Acceptance Criteria

As per Description.

Given a Provider_Change_Notification__c record is created with PCN_Type__c IN ('Address Change', 'NPI Change', 'TIN Change'),
When the Assignment Rule evaluates the record,
Then OwnerId must resolve to 'Provider_Data_Maintenance_Queue'.

Given a Provider_Change_Notification__c record is created with PCN_Type__c = 'Termination Request',
When the Assignment Rule evaluates the record,
Then OwnerId must resolve to 'Network_Termination_Queue'.

## Solution

Correct the Assignment Rule 'PCN_Routing_Assignment_Rule' entry order: (1) move the specific type-based entries (Address Change, NPI Change, TIN Change → Provider_Data_Maintenance_Queue; Termination Request → Network_Termination_Queue) to positions 1–2; (2) demote the catch-all entry to the last position. Verify rule entries in Setup > Assignment Rules. Additionally, run a one-time data fix script (Anonymous Apex) to re-evaluate and re-assign the 47 backlogged PCN records to the correct queues. Add a unit test to the existing 'PCN_AssignmentRule_Test' class covering all PCN_Type__c routing scenarios to prevent regression.
