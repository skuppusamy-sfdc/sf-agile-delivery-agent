## Description

**Bug Report**

**Summary:** When a Provider__c has multiple Fee_Schedule__c records with overlapping effective date ranges for the same CPT code, the adjudication engine selects the wrong (older) fee schedule version, resulting in incorrect Claim__c.Adjudicated_Amount__c and underpayment/overpayment of claims.

**Steps to Reproduce:**
1. Create a Provider__c linked to a Provider_Network_Agreement__c.
2. Create two Fee_Schedule__c records for the same PNA and CPT code '99213':
   - FS-A: Fee_Schedule__c.Effective_Start_Date__c = 01/01/2023, Fee_Schedule__c.Effective_End_Date__c = 12/31/2024, Allowed_Amount__c = $85.00
   - FS-B: Fee_Schedule__c.Effective_Start_Date__c = 01/01/2024, Fee_Schedule__c.Effective_End_Date__c = 12/31/2025, Allowed_Amount__c = $92.00
3. Submit a Claim__c with Claim__c.Service_Date__c = 06/15/2024 and Claim_Line__c.CPT_Code__c = '99213'.
4. Trigger adjudication.
5. Observe Claim__c.Adjudicated_Amount__c.

**Expected Result:** Claim__c.Adjudicated_Amount__c = $92.00 (FS-B selected as most specific/recent effective record for service date 06/15/2024).

**Actual Result:** Claim__c.Adjudicated_Amount__c = $85.00 (FS-A incorrectly selected — SOQL ORDER BY clause missing, first record returned by insertion order).

**Impact:** High — affects all claims adjudicated since 01/01/2024 where a fee schedule amendment exists. Finance estimates ~1,200 claims may be mispriced.

**Root Cause (Preliminary):** FeeScheduleLookupService.cls line 47 executes SOQL without ORDER BY Effective_Start_Date__c DESC LIMIT 1, returning the oldest matching record.

## Acceptance Criteria

As per Description.

Additional regression criteria:
Given multiple Fee_Schedule__c records exist for the same PNA + CPT code with overlapping date ranges
When adjudication runs for a Claim__c.Service_Date__c that falls within multiple schedules
Then the Fee_Schedule__c with the latest Effective_Start_Date__c that is <= Claim__c.Service_Date__c must be selected
And Claim__c.Applied_Fee_Schedule__c must be stamped with the selected Fee_Schedule__c.Id for audit purposes
And a bulk remediation report of affected claims (Jan 2024–present) must be produced via RemediateMispricedClaimsBatch.cls

## Solution

Fix FeeScheduleLookupService.cls line 47: add ORDER BY Effective_Start_Date__c DESC LIMIT 1 to the SOQL query. Add a unique validation rule or duplicate management rule on Fee_Schedule__c to warn on exact-duplicate CPT + date range combinations. Stamp Claim__c.Applied_Fee_Schedule__c (new lookup field) at adjudication time. Deploy RemediateMispricedClaimsBatch.cls to identify and re-adjudicate affected claims; output CSV report to Salesforce Files for Finance review. Add Apex unit tests (FeeScheduleLookupServiceTest.cls) covering overlapping date range scenarios to prevent regression. Code review required before deploy to production.
