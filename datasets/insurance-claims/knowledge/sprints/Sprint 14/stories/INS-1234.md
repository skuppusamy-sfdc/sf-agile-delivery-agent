## Description

As a Credentialing Specialist, I need Salesforce to automatically query the CAQH ProView API when a Credentialing_Case__c reaches the verification stage so that provider attestation data, license status, and malpractice history are pulled directly into the credentialing record without manual data entry, reducing credentialing cycle time by an estimated 40%. This builds on the Provider 360 work completed in INS-1232.

## Acceptance Criteria

Given a Credentialing_Case__c record transitions to Credentialing_Case__c.Status__c = 'CAQH Verification Pending',
When the Record-Triggered Flow fires,
Then an Apex callout to the CAQH ProView REST API must be initiated using the provider's Provider__c.CAQH_Provider_ID__c as the query parameter within 5 minutes of the status change.

Given a successful CAQH API response (HTTP 200),
When the response payload is parsed,
Then the following Credentialing_Case__c fields must be populated: Credentialing_Case__c.CAQH_Attestation_Date__c, Credentialing_Case__c.License_State__c, Credentialing_Case__c.License_Number__c, Credentialing_Case__c.License_Expiration_Date__c, Credentialing_Case__c.Malpractice_Claims_Flag__c (Boolean), and Credentialing_Case__c.CAQH_Last_Synced__c = NOW.

Given Credentialing_Case__c.License_Expiration_Date__c < TODAY or Credentialing_Case__c.Malpractice_Claims_Flag__c = true,
When the sync completes,
Then Credentialing_Case__c.Status__c must be set to 'Flagged — Manual Review' and the assigned Credentialing Specialist must receive a Salesforce In-App Notification and an email alert.

Given a CAQH API response of HTTP 404 (provider not found) or HTTP 429 (rate limit),
When the error is caught,
Then the Apex callout must implement exponential backoff retry logic (max 3 attempts) and on final failure, Credentialing_Case__c.CAQH_Sync_Error__c must be populated with the error code and message.

Given Provider__c.CAQH_Provider_ID__c is null,
When the verification flow attempts to initiate the callout,
Then the flow must branch to a sub-flow that creates a Task assigned to the Credentialing queue with Subject = 'CAQH ID Missing — Manual Lookup Required' and Credentialing_Case__c.Status__c must remain 'CAQH Verification Pending'.

Given a CAQH sync has completed successfully,
When the Credentialing Specialist views the Credentialing_Case__c record,
Then a read-only 'CAQH Verification Summary' Lightning component must display all synced fields with a timestamp of last sync.

-Given the provider is registered with NPPES, the NPI registry should also be queried in the same callout batch- (descoped — NPPES integration tracked separately in INS-1251).

## Solution

Create a Record-Triggered Flow on Credentialing_Case__c that fires on Status__c change to 'CAQH Verification Pending' and calls an Apex invocable action (CAQHVerificationService.cls). The service class uses a Named Credential ('CAQH_ProView_API') for authentication and implements a retry wrapper (ExponentialBackoffRetry utility class). Response parsing maps JSON fields to Credentialing_Case__c via a structured DTO class. A separate LWC (CAQHVerificationSummary.lwc) is placed on the Credentialing_Case__c record page as a read-only display component. Notifications are sent via Salesforce Custom Notification Type ('Credentialing_Alert') using the ConnectApi.Notifications class. Store CAQH API credentials in Protected Custom Metadata (CAQH_Config__mdt) with field-level encryption for the API key.
