## Description

As a Credentialing Specialist, I need the system to automatically query the CAQH ProView API when a Credentialing_Case__c is opened or reaches the 'Primary Source Verification' stage, and sync the returned provider attestation data back to the Credentialing_Case__c and Provider__c records, so that manual data entry from the CAQH portal is eliminated and credentialing cycle times are reduced from an average of 22 days to under 14 days. Extends INS-1237 (Provider Credentialing baseline object model).

## Acceptance Criteria

Given a Credentialing_Case__c record is created with Credentialing_Case__c.Status__c = 'Initiated' and Provider__c.CAQH_Provider_ID__c is populated,
When the record-triggered automation fires,
Then a callout is made to the CAQH ProView API within 5 minutes and Credentialing_Case__c.CAQH_Last_Queried__c is stamped with the current datetime.

Given the CAQH API returns a successful response (HTTP 200),
When the response is processed,
Then the following fields are updated on Credentialing_Case__c: CAQH_Profile_Status__c, CAQH_Attestation_Date__c, CAQH_License_Expiry__c, CAQH_DEA_Status__c, and CAQH_Malpractice_Coverage_Verified__c.

Given the CAQH API returns Provider__c data that differs from existing Provider__c field values (e.g., specialty, address, board certification),
When discrepancies are detected,
Then a Provider_Change_Notification__c record is auto-created with PCN__c.Change_Type__c = 'CAQH Data Discrepancy', PCN__c.Source__c = 'System – CAQH Sync', and linked to both the Credentialing_Case__c and Provider__c.

Given Provider__c.CAQH_Provider_ID__c is null when the Credentialing_Case__c is created,
When the automation attempts to fire,
Then the callout is suppressed, Credentialing_Case__c.Status__c is set to 'On Hold – Missing CAQH ID', and a Task is created for the assigned Credentialing Specialist with Subject = 'Obtain CAQH Provider ID' and Due Date = TODAY() + 3.

Given the CAQH API returns an error (HTTP 4xx or 5xx),
When the error is received,
Then Credentialing_Case__c.CAQH_Sync_Error__c is populated with the error code and message, a retry is scheduled for 2 hours later, and after 3 consecutive failures the Credentialing Specialist is notified via Salesforce notification and email.

Given a Credentialing_Case__c.Status__c transitions to 'Active – Credentialed',
When the status change is saved,
Then Provider__c.Credentialing_Status__c = 'Credentialed', Provider__c.Credentialing_Expiry_Date__c is set from CAQH_License_Expiry__c, and Provider__c.Last_Credentialed_Date__c = TODAY().

## Solution

Use a Record-Triggered Flow (after-save, async path) on Credentialing_Case__c to publish a Platform Event (CAQH_Verification_Requested__e) containing the Credentialing_Case__c Id and CAQH_Provider_ID__c. An Apex trigger subscribes to this Platform Event and invokes a Queueable Apex class (CAQHProViewSyncQueueable.cls) that makes a callout to the CAQH ProView REST API using a Named Credential ('CAQH_ProView_NC'). Response parsing and field mapping are handled within the Queueable class, with discrepancy detection comparing API response values against current Provider__c field values. PCN creation and Credentialing_Case__c field updates are performed in the same transaction. Retry logic uses a custom object (API_Retry_Queue__c) processed by a scheduled Apex job every 2 hours. CAQH API credentials are stored in Salesforce Named Credentials with OAuth 2.0. A custom Permission Set ('Credentialing_CAQH_Admin') controls access to CAQH sync fields.
