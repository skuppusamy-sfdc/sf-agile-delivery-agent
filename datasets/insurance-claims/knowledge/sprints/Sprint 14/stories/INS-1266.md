## Description

**Bug Report**

**Summary:** The Fee Schedule rate lookup logic in the FeeScheduleEngine returns null for Claim_Line__c.Allowed_Amount__c when a multi-specialty provider bills a CPT code that maps to their secondary specialty taxonomy rather than their primary taxonomy. This causes Claim_Line__c.Adjudication_Status__c to be set to 'Pended – Rate Not Found' and stalls payment processing unnecessarily, as valid contracted rates exist in Fee_Schedule__c for the secondary taxonomy.

**Steps to Reproduce:**
1. Identify a Provider__c record with two active Provider_Network_Agreement__c records: one with PNA__c.Specialty_Taxonomy__c = '207Q00000X' (Primary) and one with PNA__c.Specialty_Taxonomy__c = '261QM1300X' (Secondary).
2. Create a Claim__c with this provider and add a Claim_Line__c with CPT_Code__c = '99285' (maps to taxonomy '261QM1300X').
3. Trigger adjudication.
4. Observe Claim_Line__c.Allowed_Amount__c = null and Adjudication_Status__c = 'Pended – Rate Not Found'.

**Expected Behavior:** The FeeScheduleEngine should perform a fallback lookup against all active PNA specialty taxonomies for the provider, find the matching rate under the secondary taxonomy PNA, and populate Claim_Line__c.Allowed_Amount__c correctly.

**Actual Behavior:** The engine queries Fee_Schedule__c only against the primary PNA taxonomy (PNA__c.Is_Primary__c = true) and returns null when no match is found, without attempting secondary taxonomy lookups.

**Environment:** Acme-UAT and Acme-PROD orgs, API v59.0
**Frequency:** Occurs on approximately 12% of multi-specialty provider claims per ops team estimate.
**Severity:** Medium – causes claim pend backlog but does not result in incorrect payments

## Solution

Root cause: FeeScheduleService.cls method getRateForClaimLine() constructs its SOQL query with a hard filter WHERE PNA__c.Is_Primary__c = true, preventing secondary taxonomy evaluation. Fix: Refactor getRateForClaimLine() to first attempt primary taxonomy lookup; if null result, iterate through all active PNA records for the provider ordered by PNA__c.Payer_Order__c ascending and return the first non-null Fee_Schedule__c match. Cache the full PNA list per provider NPI within the transaction using a Map to avoid redundant SOQL in bulk adjudication scenarios. Add a Claim_Line__c field Fee_Schedule_Match_Type__c (picklist: 'Primary Taxonomy', 'Secondary Taxonomy', 'Not Found') to record which lookup path resolved the rate for audit purposes. Update FeeScheduleService unit test class to include multi-specialty provider test scenarios achieving 95%+ code coverage.
