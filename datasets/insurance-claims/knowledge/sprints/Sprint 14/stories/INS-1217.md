## Description

As a Claims Operations Manager, I need the Fee Schedule Engine to automatically look up the applicable allowed amount for each Claim_Line__c based on CPT code, provider network tier, and effective date so that adjudication is accurate, consistent, and does not require manual rate entry. The engine must respect both Fee_Schedule__c line-level rates and any GSA override rates where a Group_Service_Agreement__c is active for the policy group.

## Acceptance Criteria

Given a Claim__c record transitions to Claim__c.Stage__c = 'Adjudication',
When the Fee Schedule Engine processes each child Claim_Line__c,
Then for each Claim_Line__c, the engine queries Fee_Schedule__c where Fee_Schedule__c.CPT_Code__c = Claim_Line__c.CPT_Code__c AND Fee_Schedule__c.Network_Tier__c = Claim__c.Provider_Network_Tier__c AND Fee_Schedule__c.Effective_Date__c <= Claim__c.Date_of_Service__c AND (Fee_Schedule__c.Expiration_Date__c >= Claim__c.Date_of_Service__c OR Fee_Schedule__c.Expiration_Date__c = null).

Given a matching Fee_Schedule__c record is found,
When the allowed amount is calculated,
Then Claim_Line__c.Allowed_Amount__c is set to Fee_Schedule__c.Allowed_Amount__c and Claim_Line__c.Rate_Source__c = 'Fee Schedule'.

Given the Policy__c associated with the Claim__c has an active Group_Service_Agreement__c (GSA__c.Status__c = 'Active' and GSA__c.Effective_Date__c <= Date_of_Service__c),
When the engine checks for GSA overrides,
Then if a GSA line item exists for the same CPT code, Claim_Line__c.Allowed_Amount__c is overridden with the GSA rate and Claim_Line__c.Rate_Source__c = 'GSA Override'.

Given no matching Fee_Schedule__c record exists for a CPT code,
When the engine fails to find a rate,
Then Claim_Line__c.Adjudication_Status__c = 'Pending Rate Review', Claim_Line__c.Rate_Source__c = 'Not Found', and the parent Claim__c.Stage__c does not advance past 'Adjudication' until all lines are resolved.

Given Claim_Line__c.Adjudication_Status__c = 'Approved' for all child lines,
When all lines are processed,
Then Claim__c.Total_Allowed_Amount__c is updated as the sum of all Claim_Line__c.Allowed_Amount__c values and Claim__c.Stage__c advances to 'Approved'.

Given the Fee Schedule Engine runs,
When processing completes (success or partial failure),
Then a Fee_Schedule_Audit_Log__c record is created capturing: Claim__c Id, number of lines processed, number of lines with 'Not Found' status, timestamp, and running user.

## Solution

Implement a Fee Schedule Engine as an Apex Batch class (FeeScheduleAdjudicationBatch.cls) invoked by a Record-Triggered Flow on Claim__c when Stage__c changes to 'Adjudication'. The batch processes Claim_Line__c records in chunks of 200. A FeeScheduleLookupService.cls handles the tiered lookup logic: first checking GSA overrides via a SOQL query on Group_Service_Agreement__c line items, then falling back to Fee_Schedule__c. All rate decisions are logged to Fee_Schedule_Audit_Log__c via a separate Apex insert. A Platform Event (Fee_Schedule_Complete__e) is published on batch finish, consumed by a downstream Flow that evaluates whether all Claim_Line__c records are resolved before advancing Claim__c.Stage__c. CPT code validation uses a Custom Metadata Type (Valid_CPT_Codes__mdt) to reject invalid codes before lookup.
