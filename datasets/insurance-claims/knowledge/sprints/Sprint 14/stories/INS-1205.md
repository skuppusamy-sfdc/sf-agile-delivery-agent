## Description

**Bug Report – INS-1205**

**Summary:** When a Provider_Change_Notification__c (PCN) record is created via the inbound CMS data feed integration, the PCN.Provider__c lookup field is not being auto-populated even when an existing Provider__c (Person Account) record with a matching NPI exists in the system. This causes PCNs to appear as orphaned records, requiring manual association by Provider Network Specialists and creating a backlog.

**Steps to Reproduce:**
1. Confirm an existing Provider__c (Person Account) record exists with Provider__c.NPI__c = '1234567890'.
2. Trigger the CMS inbound data feed integration (or manually insert a PCN record via Apex anonymous) with PCN.Incoming_NPI__c = '1234567890'.
3. Navigate to the newly created Provider_Change_Notification__c record.
4. Observe that PCN.Provider__c is null.
5. Manually search for the provider by NPI to confirm the matching record exists.

**Expected Behavior:** Upon PCN creation, an Apex Trigger or Flow should query Provider__c records where NPI__c = PCN.Incoming_NPI__c and auto-populate PCN.Provider__c with the matching record Id. PCN.Auto_Match_Status__c should be set to 'Matched' and PCN.Match_Confidence__c = 'High – NPI Exact Match'.

**Actual Behavior:** PCN.Provider__c remains null. PCN.Auto_Match_Status__c = 'Unmatched' regardless of NPI match availability.

**Root Cause Hypothesis:** The Record-Triggered Flow 'PCN_AutoMatch_Flow' has a filter condition checking PCN.Incoming_NPI__c != null but the CMS integration is populating PCN.NPI_Raw__c instead of PCN.Incoming_NPI__c due to a field mapping error introduced in the Sprint 12 DataRaptor update.

**Environment:** Full Sandbox and Production.
**Frequency:** 100% of CMS-sourced PCN records affected since Sprint 12 deployment.
**Impact:** Medium-High – approximately 200+ orphaned PCN records in production backlog.

## Acceptance Criteria

As per Description. Fix must also include a one-time data remediation script to back-fill PCN.Provider__c for all existing orphaned PCN records where a matching NPI can be found. Regression test must confirm both manual and CMS-integration-sourced PCN creation paths correctly auto-match.

## Solution

Correct the DataRaptor Load field mapping in the CMS inbound integration Integration Procedure to map the incoming NPI value to PCN.Incoming_NPI__c (not PCN.NPI_Raw__c). Update the PCN_AutoMatch_Flow Record-Triggered Flow entry condition to also handle PCN.NPI_Raw__c as a fallback lookup field during a one-release transition period. Write a one-time Apex Data Remediation batch class (PCNBackfillBatch) that queries all PCN records where Provider__c = null and Incoming_NPI__c != null or NPI_Raw__c != null, performs NPI-based matching against Provider__c, and updates PCN.Provider__c, PCN.Auto_Match_Status__c, and PCN.Match_Confidence__c. Schedule the batch to run once in a maintenance window after deployment.
