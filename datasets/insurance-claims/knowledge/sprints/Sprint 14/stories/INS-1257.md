## Description

As a Policyholder, I need to ask natural language questions about my claim status through the self-service portal (e.g., 'What is the status of my knee surgery claim?' or 'When will I receive my EOB for claim #CLM-00452?') and receive accurate, real-time responses from an Agentforce AI agent so that I can get claim information 24/7 without calling the claims service center.

The agent must enforce strict data isolation so policyholders only see their own Claim__c and EOB records. The agent must be able to handle at minimum: claim status queries, EOB availability, payment status, and next steps guidance.

## Acceptance Criteria

Given an authenticated policyholder accesses the self-service portal and initiates a chat with the Agentforce Claims Status Agent,
When the agent session starts,
Then the agent greets the user by first name (from Contact.FirstName linked to Policy__c.Policyholder__c) and presents a prompt suggesting available inquiry types.

Given a policyholder asks 'What is the status of my claim?',
When the Agentforce agent processes the natural language input,
Then the agent queries Claim__c records WHERE Policy__c.Policyholder__c = current user Contact Id, returns Claim__c.Status__c, Claim__c.Date_of_Service__c, and Claim__c.Claim_Number__c, and presents a human-readable response such as 'Your claim for services on [Date] is currently [Status].'

Given a policyholder asks about EOB availability,
When the agent queries Explanation_of_Benefits__c,
Then the agent checks EOB__c.Available_to_Policyholder__c = true AND EOB__c.Claim__c.Policy__c.Policyholder__c = current user, returns EOB__c.Document_URL__c as a clickable link, and confirms the mailing date if EOB__c.Mailed_Date__c is populated.

Given a policyholder asks a question that requires human intervention (e.g., 'Why was my claim denied?') and Claim__c.Status__c = 'Denied',
When the agent identifies the escalation trigger,
Then the agent responds with a denial reason summary derived from Claim__c.Denial_Reason_Code__c mapped to a human-readable description via Custom Metadata Type (Denial_Reason_Description__mdt), offers to connect to a live agent, and if accepted, creates a Case linked to the Claim__c with Case.Origin = 'Policyholder Portal Chat'.

Given any policyholder session,
When the agent attempts to retrieve data,
Then all SOQL queries executed by the agent run in user context (WITH USER_MODE) and the agent cannot return Claim__c records where Policy__c.Policyholder__c != current authenticated user under any prompt injection scenario.

Given the agent cannot find any Claim__c records for the authenticated user,
When the query returns zero results,
Then the agent responds with 'I don't see any claims on file for your policy. If you believe this is an error, I can connect you with our claims team.' and does NOT expose any system error messages or Salesforce record IDs.

## Solution

Configure an Agentforce Agent (type: External-Facing / Experience Cloud) with a custom Agent Topic 'Claims Status Inquiry' and four Agent Actions: (1) GetClaimStatus – Apex invocable querying Claim__c WITH USER_MODE, (2) GetEOBDetails – Apex invocable querying EOB__c WITH USER_MODE, (3) GetPaymentStatus – Apex invocable querying Payment__c, (4) EscalateToLiveAgent – Flow-based action creating a Case and triggering Omni-Channel routing. Natural language understanding is handled by the Einstein LLM layer configured in the Agent Builder with a system prompt enforcing data isolation rules and response tone guidelines. The agent is embedded in the Experience Cloud Policyholder Portal via the Agentforce Messaging component. A custom Permission Set 'Agentforce_Policyholder_Agent' grants read-only FLS on Claim__c, EOB__c, Payment__c with Sharing Rules enforcing policyholder-level record access. All agent conversations are logged to AgentWork and MessagingSession objects for audit purposes.
