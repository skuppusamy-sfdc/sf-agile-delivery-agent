## Description

**Bug Report – Claim_Line__c Allowed Amount Miscalculation with Stacked Modifiers**

**Environment:** Acme Insurance Production Sandbox (Sprint 14 refresh)
**Severity:** High – Financial impact on claim payments

**Steps to Reproduce:**
1. Create a Claim__c with LOB__c = 'Medical' linked to a valid Policy__c.
2. Add a Claim_Line__c with CPT__c = '27447' (total knee replacement), Billed_Amount__c = 15000.00.
3. Attach two modifiers: Modifier_1__c = '62' (two surgeons) and Modifier_2__c = 'AS' (assistant surgeon).
4. Link a Fee_Schedule__c record where the base allowed amount for CPT 27447 = 8500.00, modifier 62 applies a 125% multiplier, and modifier AS applies a 16% of allowed rate.
5. Trigger adjudication by setting Claim__c.Status__c = 'Ready for Adjudication'.
6. Observe Claim_Line__c.Allowed_Amount__c after adjudication completes.

**Expected Result:**
Claim_Line__c.Allowed_Amount__c = 8500.00 × 1.25 = 10625.00 (modifier 62 applied first per CMS modifier stacking rules), then modifier AS is calculated as a separate line at 16% of 10625.00 = 1700.00. Total allowed = 12325.00 across two Claim_Line__c records.

**Actual Result:**
The FeeScheduleCalculationService applies both multipliers additively: 8500.00 × (1.25 + 0.16) = 11985.00 on a single Claim_Line__c, which is incorrect per CMS modifier stacking rules and overpays by $340.00 per claim line.

**Additional Context:**
This was introduced in the Sprint 12 refactor of FeeScheduleCalculationService (commit ref: FSE-refactor-sprint12). Affects all claims with two or more modifiers where at least one modifier triggers a percentage-of-allowed calculation. Estimated financial exposure: ~$34,000 across 100 affected claims in UAT dataset.

## Acceptance Criteria

As per Description. Verified fix must pass regression test suite FeeSchedule_Modifier_Stack_Tests (Apex test class) with 100% pass rate and demonstrate correct stacking order per CMS modifier hierarchy rules for the following modifier combinations: [62+AS], [80+AS], [62+80], [TC+26].

## Solution

Fix FeeScheduleCalculationService.applyModifiers() method to implement sequential modifier application using a priority-ordered List<Fee_Schedule_Modifier__mdt> sorted by Modifier_Priority__c (Custom Metadata Type field). Modifiers that produce a new 'allowed base' (type = 'Multiplier') are applied first and update the running allowed amount; modifiers of type 'Percentage_Of_Allowed' are applied to the post-multiplier base and generate a new Claim_Line__c child record rather than modifying the original. Add a Custom Metadata Type field Modifier_Generates_New_Line__c (Boolean) to control this behavior per modifier code. Update Apex test class FeeSchedule_Modifier_Stack_Tests with parameterized test methods covering all four failing modifier combinations. Deploy fix with a Data Fix script to recalculate Allowed_Amount__c on affected Claim_Line__c records in UAT before promoting to production.
