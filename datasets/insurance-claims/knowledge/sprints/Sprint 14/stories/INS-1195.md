## Description

**Bug Summary:** Claim_Line__c.Allowed_Amount__c is not recalculated when the associated Fee_Schedule__c record is updated while a Claim__c is in 'In Adjudication' status, resulting in incorrect payment amounts being posted.

**Steps to Reproduce:**
1. Create a Claim__c record with Status__c = 'In Adjudication'.
2. Add a Claim_Line__c with CPT_Code__c = '99213' linked to an active Fee_Schedule__c where Fee_Schedule__c.Allowed_Amount__c = 150.00.
3. Update the Fee_Schedule__c record: change Fee_Schedule__c.Allowed_Amount__c to 175.00 and save.
4. Navigate back to the Claim_Line__c record.
5. Observe that Claim_Line__c.Allowed_Amount__c still displays 150.00.

**Expected Result:** Claim_Line__c.Allowed_Amount__c should recalculate to 175.00 to reflect the updated fee schedule.

**Actual Result:** Claim_Line__c.Allowed_Amount__c remains at 150.00 (stale value). Payment__c is subsequently generated with the incorrect allowed amount.

**Impact:** High — incorrect ERA amounts are being sent to providers and overpayments/underpayments are occurring in production.

**Regression:** Introduced in Sprint 12 deployment (INS-1143 fee schedule engine refactor).

## Acceptance Criteria

As per Description. Verify Claim_Line__c.Allowed_Amount__c reflects updated Fee_Schedule__c.Allowed_Amount__c within 60 seconds of Fee_Schedule__c save. Confirm no Payment__c records are generated with stale allowed amounts when Claim__c.Status__c = 'In Adjudication'.

## Solution

Root cause: the Fee Schedule lookup recalculation logic in FeeScheduleRecalcTrigger.cls has a missing cross-object trigger path — the trigger fires on Claim_Line__c DML but not on parent Fee_Schedule__c updates. Fix: add an Apex Trigger on Fee_Schedule__c (After Update) that queries all related Claim_Line__c records where the parent Claim__c.Status__c = 'In Adjudication' and re-invokes the FeeScheduleCalculatorService.recalculate() method. Use a Trigger Handler pattern to avoid recursive execution. Add a bypass Custom Setting (TriggerBypass__c.FeeSchedule_Bypass__c) for controlled rollback. Unit tests to cover mid-adjudication fee schedule change scenario with 95%+ coverage.
