## Description

As a Provider Network Specialist, I need the system to automatically generate a Network_Termination_Case__c record when a Provider_Network_Agreement__c record's termination trigger date is reached or the PNA Status is set to 'Pending Termination', so that the network exit workflow is initiated without manual intervention and no providers slip through the cracks during offboarding.

This story extends the foundational PNA lifecycle work completed in INS-0842 and builds the automated exit ramp for both voluntary and involuntary terminations across all LOBs.

## Acceptance Criteria

Given a Provider_Network_Agreement__c record where PNA_Status__c transitions to 'Pending Termination' via any channel (manual update, API, or upstream Flow),
When the Record-Triggered Flow evaluates the record on save,
Then a new Network_Termination_Case__c is created with:
  - Network_Termination_Case__c.Provider__c = PNA.Provider__c
  - Network_Termination_Case__c.PNA_Reference__c = PNA.Id
  - Network_Termination_Case__c.Termination_Type__c derived from PNA.Termination_Reason__c (mapping: 'Voluntary', 'Involuntary', 'Mutual Agreement')
  - Network_Termination_Case__c.Effective_Termination_Date__c = PNA.Contract_End_Date__c
  - Network_Termination_Case__c.Status__c = 'Open'
  - Network_Termination_Case__c.OwnerId = Queue 'Provider Network Terminations'

Given a Network_Termination_Case__c is successfully created,
When the record is inserted,
Then a Platform Event (Provider_Exit_Event__e) is published with payload fields: provider_id, pna_id, lob, effective_termination_date, termination_type — to notify downstream systems including ClaimsProcessing and FeeScheduleEngine.

Given the PNA_Status__c is already 'Terminated' or 'Expired' at the time of evaluation,
When the Flow checks for an existing Network_Termination_Case__c linked to the PNA,
Then no duplicate Network_Termination_Case__c is created and a warning log entry is written to the Integration_Log__c object with Log_Type__c = 'Duplicate Prevention'.

Given a PNA with PNA_Status__c = 'Pending Termination' and associated LOB = 'Medicare Advantage',
When the Network_Termination_Case__c is created,
Then Network_Termination_Case__c.CMS_Notification_Required__c = TRUE and a task is auto-created assigned to the Compliance Officer queue with Subject = 'CMS 60-Day Notice Required'.

-Given termination is triggered by a Credentialing_Case__c adverse action, auto-populate Network_Termination_Case__c.Credentialing_Case__c lookup — DEPRECATED: credentialing linkage moved to INS-0874-

Given any failure in Network_Termination_Case__c creation,
When the Flow fault path is triggered,
Then an email alert is dispatched to the Provider Network Specialist assigned to the PNA and an error record is written to Integration_Log__c with Log_Type__c = 'Termination Automation Error', Status__c = 'Failed'.

## Solution

Implement a Record-Triggered Flow on Provider_Network_Agreement__c firing on Update (entry condition: PNA_Status__c CHANGED TO 'Pending Termination'). Flow uses a Decision element to check for existing Network_Termination_Case__c via SOQL (Get Records). If none found, Create Records action instantiates the Network_Termination_Case__c with mapped fields. A subsequent Publish Platform Event action fires Provider_Exit_Event__e for downstream subscribers. An Apex-invocable method handles the CMS notification flag logic and LOB-specific branching to keep Flow complexity manageable. Fault path routes to a Send Email action and a Create Records action writing to Integration_Log__c. A custom Permission Set Group 'Network_Termination_Automation' grants FLS on all relevant fields. Unit tests cover happy path, duplicate prevention, and fault path with minimum 85% coverage.
