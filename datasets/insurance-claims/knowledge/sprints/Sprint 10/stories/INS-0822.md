## Description

**Bug Report**

**Summary:** During the Network Exit Checklist OmniScript (INS-0818) Step 2 execution, when a provider has multiple active Provider_Network_Agreement__c records across different Lines of Business (e.g., Commercial, Medicare Advantage, Medicaid), only the Fee_Schedule__c records linked to the PNA specified in the Network_Termination_Case__c are suspended. Fee schedules associated with the provider's other active PNAs in different LOBs remain Active, causing potential incorrect payment processing for those LOBs.

**Steps to Reproduce:**
1. Create a Provider__c with three active PNA records: PNA-A (LOB = Commercial), PNA-B (LOB = Medicare Advantage), PNA-C (LOB = Medicaid).
2. Create one Fee_Schedule__c per PNA, all with Status__c = 'Active'.
3. Create a Network_Termination_Case__c referencing PNA-A only.
4. Launch the Network Exit Checklist OmniScript and complete Step 2 (Fee Schedule Suspension).
5. Check all three Fee_Schedule__c records.

**Expected Result:** Only FS linked to PNA-A (the terminating PNA) should be suspended. FS records for PNA-B and PNA-C should remain Active (provider is only exiting the Commercial network).

**Actual Result:** The DataRaptor Turbo Action in Step 2 is using Provider__c Id as the filter key instead of PNA__c Id, causing ALL Fee_Schedule__c records for the provider across all LOBs to be updated to Status__c = 'Suspended' regardless of the terminating PNA.

**Environment:** Full Sandbox (FS-01)
**Frequency:** 100% reproducible when provider has >1 active PNA
**Severity:** Medium — data integrity issue; overly broad suspension impacts non-terminating LOBs

## Acceptance Criteria

As per Description. Fix must update the DataRaptor Turbo Action filter from Provider__c = [provider_id] to PNA__c = [pna_id] so that only Fee_Schedule__c records with FS.PNA__c = NTC.PNA__c are suspended. Regression test with a provider having 3 active PNAs must confirm only the target LOB fee schedules are affected.

## Solution

Open the DataRaptor Turbo Action 'SuspendFeeSchedules_DTA' in OmniStudio. Locate the WHERE clause filter and change the filter key from 'Provider__c' to 'PNA__c', binding the value to the OmniScript context variable 'pnaId' (already available in the OmniScript data model from Step 1 DataRaptor Extract). Update the DataRaptor Input JSON mapping accordingly. Re-activate the DataRaptor and re-test in FS-01. Update the OmniScript unit test data to include multi-PNA scenarios.
