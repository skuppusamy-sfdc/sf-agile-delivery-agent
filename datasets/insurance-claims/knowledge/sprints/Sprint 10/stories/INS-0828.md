## Description

As a Claims Operations Manager, I need Fee_Schedule__c records associated with a terminating provider to be automatically deactivated on the termination effective date, so that no new Claim_Line__c records are priced against an invalid fee schedule after the provider exits the network.

Fee schedules must remain readable for historical adjudication lookups on claims with Date_of_Service__c prior to the termination effective date.

## Acceptance Criteria

Given a Network_Termination_Case__c transitions to Status__c = 'Approved' with a valid Effective_Date__c
When the Effective_Date__c is reached (evaluated by nightly scheduled job)
Then all Fee_Schedule__c records where Fee_Schedule__c.Provider__c = terminating Provider__c.Id AND Fee_Schedule__c.Status__c = 'Active' are updated to Status__c = 'Inactive' and Fee_Schedule__c.Deactivation_Reason__c = 'Provider Network Termination'

Given Fee_Schedule__c records are deactivated
When a new Claim_Line__c is submitted for the terminated provider with Service_Date__c >= Termination Effective_Date__c
Then the FeeScheduleEngine returns a 'No Active Fee Schedule' error code and Claim_Line__c.Pricing_Status__c = 'Fee Schedule Not Found' and the parent Claim__c.Status__c is set to 'Pended - Provider Eligibility'

Given a Claim_Line__c with Service_Date__c < Termination Effective_Date__c is submitted after termination
When the FeeScheduleEngine processes the claim line
Then the deactivated Fee_Schedule__c is still accessible in read-only mode for pricing lookup and Claim_Line__c.Pricing_Status__c = 'Priced' with the historical rate applied

Given Fee_Schedule__c deactivation completes
When the batch job finishes
Then an Integration_Log__c record is created with Log_Type__c = 'Fee Schedule Deactivation', Records_Processed__c count, and any failed record IDs in Error_Detail__c

-Fee_Schedule__c records linked to Group_Service_Agreement__c should also be deactivated in this flow- (deprecated: GSA fee schedule deactivation is handled separately in the GSA amendment process per legal review)

## Solution

Scheduled Apex Batch 'FeeScheduleDeactivationBatch' runs nightly, queries Network_Termination_Case__c records where Status__c = 'Approved' AND Effective_Date__c = TODAY. For each, queries related Fee_Schedule__c records. Batch size = 200. Uses Database.update with allOrNone = false; failures captured in Integration_Log__c. Historical access preserved by NOT deleting records — Status__c field drives FeeScheduleEngine SOQL filter (WHERE Status__c = 'Active'). Engine modified to include fallback query without Status filter when Service_Date__c < termination date, gated by custom metadata flag 'FSE_Config__mdt.Allow_Historical_Inactive_Lookup__c'.
