# INS-0853: Suspend active Claim__c lines for terminated providers and route to Claims Operations review queue

## Description

As a Claims Operations Manager, I need all in-flight Claim__c records associated with a provider whose Network_Termination_Case__c has reached 'Terminated' status to be automatically suspended and routed to a dedicated Claims Operations review queue so that payments are not erroneously disbursed to out-of-network providers and each claim can be adjudicated under the correct out-of-network fee schedule.

This story is triggered downstream of the Platform Event fired in INS-0848.

## Acceptance Criteria

Given a Platform Event 'ProviderTerminationInitiated__e' is received with payload containing Provider__c Id and Termination_Effective_Date__c
When the Platform Event-Triggered Flow 'PEF_SuspendProviderClaims' processes the event
Then all Claim__c records where Claim__c.Rendering_Provider__c = event.Provider_Id__c AND Claim__c.Status__c IN ('Submitted', 'In Adjudication', 'Pending Payment') AND Claim__c.Service_Date__c >= event.Termination_Effective_Date__c are updated to Claim__c.Status__c = 'Suspended - Provider Termination'

Given Claim__c records are suspended
When the update completes
Then each suspended Claim__c is assigned to OwnerId = [Claims_Ops_Termination_Review Queue Id] and Claim__c.Suspension_Reason__c = 'Rendering Provider Network Termination' and Claim__c.Linked_NTC__c = the originating Network_Termination_Case__c Id

Given a suspended Claim__c has associated Payment__c records in Payment__c.Status__c = 'Pending'
When the suspension flow processes
Then Payment__c.Status__c is updated to 'On Hold' and Payment__c.Hold_Reason__c = 'Provider Termination Review' to prevent ERA disbursement

Given a Claim__c where Service_Date__c < Termination_Effective_Date__c
When the flow evaluates the record
Then the claim is NOT suspended and a Claim__c note is added: 'Provider terminated post-service date; adjudication continues under original PNA terms'

Given more than 200 Claim__c records are returned for a single termination event
When the Platform Event-Triggered Flow invokes the update
Then processing is delegated to a Queueable Apex job 'SuspendProviderClaimsQueueable' to handle bulk DML within governor limits, and a log entry is written to Termination_Processing_Log__c

-Automatically generate an EOB for each suspended claim- (deferred to Sprint 11; EOB generation requires manual review completion first)

## Solution

Platform Event-Triggered Flow subscribes to 'ProviderTerminationInitiated__e'. Flow uses Get Records to retrieve matching Claim__c records using the provider Id from the event payload. Decision element checks record count: if <= 200, use loop + Update Records within Flow; if > 200, call Apex invocable 'EnqueueClaimSuspensionJob' which dispatches a Queueable Apex class. Queueable Apex performs chunked DML in batches of 200. Payment__c hold logic implemented in an Apex Trigger (ClaimAfterUpdate) that fires after Claim__c.Status__c changes to 'Suspended - Provider Termination', using a SOQL query on related Payment__c records and performing a bulk update. Custom Label stores the Claims_Ops_Termination_Review Queue Id for environment portability. All processing steps write to Termination_Processing_Log__c for audit trail.
