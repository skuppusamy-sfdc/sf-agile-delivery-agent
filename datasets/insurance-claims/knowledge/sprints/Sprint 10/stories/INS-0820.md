## Description

As a Claims Operations Manager, I need all in-flight Claim_Line__c records associated with a terminated provider to be automatically suspended for manual review upon confirmation of the provider's network exit so that we do not disburse payments at in-network rates to providers who are no longer contracted, reducing financial leakage and ensuring EOB accuracy.

## Acceptance Criteria

Given a Provider_Termination_Event__e Platform Event is received (published by INS-0818 Step 5)
When the Platform Event-Triggered Flow processes the event
Then all Claim_Line__c records where:
  - CL.Provider__c = event.Provider_Id__c
  - CL.Status__c IN ('Pending Adjudication', 'In Adjudication')
  - CL.Service_Date__c >= event.Termination_Effective_Date__c
Are updated to CL.Status__c = 'Suspended - Provider Termination' and CL.Suspension_Reason__c = 'Provider Network Exit'

Given Claim_Line__c records are suspended
When the update completes
Then the parent Claim__c.Status__c is updated to 'On Hold' if ALL child Claim_Line__c records are suspended, and a Case is created for the Claims Operations queue with Subject = 'Provider Termination - Claim Review Required'

Given a Claim_Line__c with CL.Service_Date__c BEFORE the termination effective date
When the Platform Event is processed
Then that Claim_Line__c is NOT suspended and continues adjudication at the contracted rate

-Given suspended Claim_Line__c records, automatically reroute to out-of-network fee schedule-
-This requirement is deferred to Sprint 11; out-of-network repricing will be handled manually by Claims Ops in this sprint-

Given more than 500 Claim_Line__c records match the suspension criteria for a single provider
When the Platform Event-Triggered Flow hits governor limits
Then processing is chunked via an Apex Queueable job to handle bulk volumes without hitting DML limits

## Solution

Create a Platform Event-Triggered Flow subscribing to Provider_Termination_Event__e. Use Get Records to retrieve matching Claim_Line__c records in batches. For volumes under 200, use Update Records in the flow directly. For larger volumes, invoke an Apex Queueable class 'ClaimLineSuspensionQueueable' via an Apex Action element, passing the Provider Id and Effective Date. The Queueable processes records in chunks of 200. After suspension, a second flow path checks parent Claim__c child aggregate and updates Status__c. Case creation handled via Create Records. Add a Custom Metadata Type 'Termination_Config__mdt' to configure the chunk size and queue name without code changes.
