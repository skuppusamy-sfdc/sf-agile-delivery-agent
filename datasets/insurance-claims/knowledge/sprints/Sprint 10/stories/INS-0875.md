## Description

As a Claims Operations Manager, I need the system to automatically sunset the active Fee_Schedule__c records associated with a terminated provider and block any new Claim_Line__c records from being priced against those fee schedules after the Network_Termination_Case__c is approved and the effective termination date is reached, so that out-of-network claims are correctly routed for OON adjudication rather than being incorrectly processed at in-network contracted rates.

This story depends on the Network_Termination_Case__c automation delivered in INS-0872 and the date-fix resolved in INS-0874. It represents the financial control layer of the provider network exit flow and is critical for payment integrity across all LOBs.

## Acceptance Criteria

Given a Network_Termination_Case__c record where Status__c transitions to 'Approved' and Effective_Termination_Date__c is a future date,
When the Record-Triggered Flow evaluates on Status__c change,
Then all Fee_Schedule__c records related to the provider via Fee_Schedule__c.Provider__c = Network_Termination_Case__c.Provider__c are updated:
  - Fee_Schedule__c.End_Date__c = Network_Termination_Case__c.Effective_Termination_Date__c
  - Fee_Schedule__c.Status__c = 'Pending Sunset'
  - Fee_Schedule__c.Sunset_Reason__c = 'Provider Network Exit'

Given a Fee_Schedule__c record with Status__c = 'Pending Sunset' and End_Date__c = TODAY(),
When a scheduled Apex batch job (BatchFeeScheduleSunset) runs at 00:01 AM daily,
Then Fee_Schedule__c.Status__c is updated to 'Inactive' and Fee_Schedule__c.Active__c = FALSE for all matching records.

Given a Claim_Line__c record is submitted for a provider whose Fee_Schedule__c.Status__c = 'Inactive' OR Fee_Schedule__c.Active__c = FALSE,
When the FeeScheduleEngine Apex service attempts to price the Claim_Line__c,
Then:
  - Claim_Line__c.Pricing_Status__c = 'Blocked — Provider Terminated'
  - Claim_Line__c.OON_Routing_Flag__c = TRUE
  - The parent Claim__c.Adjudication_Queue__c is set to 'OON Adjudication'
  - An EOB line-level remark code 'T01 — Provider Not In Network at DOS' is written to Explanation_of_Benefits__c.Remark_Code__c

Given a Claim_Line__c with a Date_of_Service__c that falls BEFORE the Fee_Schedule__c.End_Date__c (i.e., services rendered while provider was still in-network),
When the FeeScheduleEngine evaluates the Claim_Line__c,
Then the Claim_Line__c is priced at the contracted rate (no block applied) and Claim_Line__c.Pricing_Status__c = 'Priced — Historical In-Network'.

Given a Fee_Schedule__c sunset is triggered for a provider with an active Group_Service_Agreement__c linking the provider to an employer group,
When the sunset Flow processes,
Then a Provider_Change_Notification__c (PCN) record is auto-created with:
  - PCN.Notification_Type__c = 'Provider Network Exit'
  - PCN.GSA_Reference__c = Group_Service_Agreement__c.Id
  - PCN.Status__c = 'Pending Distribution'
  - PCN.Target_Audience__c = 'Employer Group'
And a task is assigned to the Broker Relations Coordinator queue with Subject = 'PCN Distribution Required — Provider Exit'.

-Given Fee_Schedule__c sunset should also trigger ERA file regeneration for open Payment__c records — DEPRECATED: ERA regeneration descoped to Phase 3 per Compliance Officer review 2024-11-14-

Given the BatchFeeScheduleSunset job encounters a DML exception on any Fee_Schedule__c record,
When the exception is caught,
Then the failed record Id, error message, and stack trace are written to Integration_Log__c with Log_Type__c = 'Fee Schedule Sunset Error' and the batch continues processing remaining records without full abort.

Given the entire sunset and repricing block automation is deployed,
When a Provider Network Specialist reviews a terminated provider's record,
Then the Provider 360 FlexCard (INS-0873) 'Network Exit Status' panel displays Fee_Schedule__c.Status__c = 'Inactive' with the sunset date visible in the panel.

## Solution

Implement a Record-Triggered Flow on Network_Termination_Case__c (on Update, entry condition: Status__c CHANGED TO 'Approved') that uses a Get Records node to retrieve all related Fee_Schedule__c records and a Loop + Update Records pattern to stamp End_Date__c and Status__c = 'Pending Sunset'. To avoid governor limit issues with large fee schedule volumes, the Flow invokes an Apex-invocable method (FeeScheduleSunsetService.scheduleForSunset()) which performs a bulk DML update via a Queueable Apex chain. The daily BatchFeeScheduleSunset Apex batch (Database.Batchable, scope 200) queries Fee_Schedule__c WHERE Status__c = 'Pending Sunset' AND End_Date__c <= TODAY() and sets Active__c = FALSE, Status__c = 'Inactive'. The FeeScheduleEngine pricing service (existing Apex class) is updated with a pre-pricing guard clause checking Fee_Schedule__c.Active__c and comparing Claim_Line__c.Date_of_Service__c against Fee_Schedule__c.End_Date__c to implement the historical in-network carve-out. PCN creation logic is added to the same Record-Triggered Flow via a Decision branch checking for active GSA linkage. A Permission Set Group 'Fee_Schedule_Sunset_Automation' governs FLS. All Apex classes require 85%+ test coverage with dedicated test data factory methods for terminated provider scenarios.
