## Description

**Bug Report — Provider Termination Date Contamination**

**Environment:** Full Sandbox (ACME-FS01)
**Discovered by:** Rachel Torres, Provider Network Specialist, during UAT Sprint 10

**Steps to Reproduce:**
1. Navigate to a Provider__c (Person Account) record that has BOTH an active Credentialing_Case__c (with Credential_Expiry_Date__c populated) AND an active Provider_Network_Agreement__c (with Contract_End_Date__c populated, where Contract_End_Date__c ≠ Credential_Expiry_Date__c).
2. Manually set PNA_Status__c on the Provider_Network_Agreement__c to 'Pending Termination' and save.
3. Observe the auto-created Network_Termination_Case__c record (created by the Record-Triggered Flow from INS-0872).
4. Inspect Network_Termination_Case__c.Effective_Termination_Date__c.

**Expected Result:**
Network_Termination_Case__c.Effective_Termination_Date__c should equal Provider_Network_Agreement__c.Contract_End_Date__c for the triggering PNA record.

**Actual Result:**
Network_Termination_Case__c.Effective_Termination_Date__c is populated with the value from Credentialing_Case__c.Credential_Expiry_Date__c, producing an incorrect termination date. This causes downstream payment suppression in ClaimsProcessing to activate on the wrong date, potentially denying valid claims.

**Root Cause Hypothesis:**
The Record-Triggered Flow introduced in INS-0872 contains a Get Records node that retrieves the related Credentialing_Case__c to check for adverse actions. A subsequent Assign element appears to be incorrectly mapping Credentialing_Case__c.Credential_Expiry_Date__c to the Network_Termination_Case__c date field instead of using the PNA's Contract_End_Date__c from the triggering record ($Record.Contract_End_Date__c).

**Impact:**
- Severity: High — affects all providers with co-existing Credentialing and PNA records
- Affected records identified in sandbox: 14 Network_Termination_Case__c records with incorrect dates
- Potential production impact if not resolved before Sprint 10 deployment

## Acceptance Criteria

As per Description.

Additional verification steps post-fix:
- Confirm Network_Termination_Case__c.Effective_Termination_Date__c = PNA.Contract_End_Date__c for all 14 affected sandbox records after data correction script runs.
- Confirm the Flow Assign element references $Record.Contract_End_Date__c and NOT {!Get_Credentialing_Case.Credential_Expiry_Date__c} for the Effective_Termination_Date__c mapping.
- Confirm that a provider with NO active Credentialing_Case__c still creates a correct Network_Termination_Case__c (regression check).
- Confirm that a provider with NO active PNA Contract_End_Date__c (null) results in Network_Termination_Case__c.Effective_Termination_Date__c = NULL and triggers the 'Missing Termination Date' validation alert task.

## Solution

Fix the Assign element in the 'Create_Network_Termination_Case' Record-Triggered Flow (INS-0872 flow) to explicitly reference {!$Record.Contract_End_Date__c} for the Effective_Termination_Date__c field assignment, replacing the erroneous reference to {!Get_Credentialing_Case.Credential_Expiry_Date__c}. The Get Records node retrieving Credentialing_Case__c should be scoped solely to the adverse action flag check and its output variables should be renamed with a 'CredCase_' prefix to prevent future variable collision. After Flow fix is deployed to sandbox, run a one-time Apex data fix script (AnonymousApex or Batch Apex) to correct the 14 affected Network_Termination_Case__c records by re-querying their linked PNA Contract_End_Date__c. All changes to be peer-reviewed by Marcus Johnson before sandbox regression test. Flow version history to be preserved; prior version deactivated but not deleted per audit policy.
