# INS-0791: Bug — Fee_Schedule__c lookup returns incorrect rate for Claim_Line__c when provider has multiple overlapping termination-era schedules

## Description

Steps to Reproduce:
1. Create a Provider__c with two Fee_Schedule__c records: (A) Fee_Schedule__c.Effective_Start_Date__c = 2024-01-01, Effective_End_Date__c = 2024-12-31, Type__c = 'In-Network Standard'; (B) Fee_Schedule__c.Effective_Start_Date__c = 2024-09-01, Effective_End_Date__c = 2025-03-31, Type__c = 'In-Network Tail Period'.
2. Submit a Claim__c with Date_of_Service__c = 2024-10-15 for this provider who has Network_Status__c = 'Termination Pending'.
3. Observe the Fee Schedule applied to Claim_Line__c records.

Expected Behavior: The 'In-Network Tail Period' Fee_Schedule__c (Schedule B) is selected as it is the most recently effective schedule and matches Type__c = 'In-Network Tail Period' per the tail period adjudication rule.

Actual Behavior: The SOQL query in FeeScheduleLookupService returns Schedule A ('In-Network Standard') because the ORDER BY clause sorts by Effective_Start_Date__c ASC and returns the first result, ignoring the Type__c priority ranking. This results in Claim_Line__c.Allowed_Amount__c being underpaid by an average of 12–18% based on QA test data, causing incorrect Payment__c disbursements and EOB__c generation with wrong member cost-share figures.

Impact: High — affects all tail-period claims for providers with overlapping fee schedules. Estimated 340 Claim__c records in UAT are mispriced. Linked to INS-0786 tail period logic.

## Acceptance Criteria

As per Description. Additionally:
Given multiple Fee_Schedule__c records exist for a provider covering the same DOS,
When the FeeScheduleLookupService executes,
Then schedules are prioritized: Type__c = 'In-Network Tail Period' > 'In-Network Standard' > 'Out-of-Network', and within the same type, the record with the latest Effective_Start_Date__c is selected.

Given the corrected lookup logic is deployed,
When the 340 mispriced Claim__c records in UAT are reprocessed via ClaimReprocessingBatch,
Then Claim_Line__c.Allowed_Amount__c, Payment__c.Amount__c, and EOB__c.Member_Cost_Share__c are recalculated correctly and audit records are written to Claim_Reprocessing_Log__c.

## Solution

Fix the SOQL query in FeeScheduleLookupService.lookupSchedule() method: replace ORDER BY Effective_Start_Date__c ASC LIMIT 1 with a prioritized query using a CASE-based priority field. Introduce a new Fee_Schedule__c.Priority_Rank__c formula field (numeric) that assigns: Tail Period = 1, In-Network Standard = 2, Out-of-Network = 3. Update SOQL to ORDER BY Priority_Rank__c ASC, Effective_Start_Date__c DESC LIMIT 1. Write a ClaimReprocessingBatch (implements Database.Batchable) to reprocess affected Claim__c records identified by a custom report snapshot saved as Claim_Reprocess_Queue__c records. Batch updates Claim_Line__c.Allowed_Amount__c, triggers Payment__c recalculation via PaymentRecalculationService, and regenerates EOB__c via EOBGenerationService. All reprocessed records stamped with Reprocessed_Date__c and Reprocessing_Batch_ID__c for audit traceability. Deploy fix to FULL sandbox for regression before hotfix to UAT.
