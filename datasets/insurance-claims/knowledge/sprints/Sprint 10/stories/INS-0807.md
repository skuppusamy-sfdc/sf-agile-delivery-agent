# INS-0807: Create NPPES and CAQH data sync validation checkpoint during Network_Termination_Case__c closure

## Description

As a Credentialing Specialist, I need the system to perform a real-time NPPES and CAQH data validation when a Network_Termination_Case__c is being closed, so that I can confirm the provider's NPI status and CAQH profile reflect the network exit before the case is marked 'Closed – Complete' and downstream credentialing records are archived. This ensures our Provider__c master data stays synchronized with federal and industry credentialing registries at the point of termination closure.

## Acceptance Criteria

Given a Network_Termination_Case__c with Status__c = 'Effective' and all PCN acknowledgments received
When a Credentialing Specialist attempts to transition Status__c to 'Closed – Complete'
Then a validation checkpoint screen flow launches automatically before the status write is committed

Given the validation checkpoint flow is triggered
When the NPPES integration callout is made using Provider__c.NPI__c
Then the system retrieves and displays NPPES_Enumeration_Status__c and NPPES_Last_Updated__c on the checkpoint screen; if NPPES_Enumeration_Status__c = 'Deactivated' a green confirmation badge is shown; if 'Active' a yellow warning is shown with message: 'NPI still active in NPPES – confirm provider has been notified'

Given the NPPES callout completes
When the CAQH integration callout is made using Provider__c.CAQH_Provider_ID__c
Then CAQH_Profile_Status__c and CAQH_Last_Attestation_Date__c are retrieved and displayed; if CAQH_Profile_Status__c = 'Inactive' a confirmation badge is shown

Given either the NPPES or CAQH callout returns a timeout or HTTP error
When the checkpoint screen renders
Then the integration error is logged to Custom_Log__c with HTTP status code and response body, a warning banner displays 'External validation unavailable – manual verification required', and the Credentialing Specialist can override and proceed with Override_Reason__c mandatory text entry

Given the checkpoint is passed (either via clean validation or documented override)
When Status__c is set to 'Closed – Complete'
Then Credentialing_Case__c.Status__c linked to the provider is updated to 'Archived', Provider__c.Network_Status__c is set to 'Terminated', and a final audit ContentNote is created on the Network_Termination_Case__c summarizing validation results and override reason (if applicable)

Given Provider__c.Network_Status__c = 'Terminated'
When any future Credentialing_Case__c is created for the same provider
Then a warning banner displays on the new Credentialing_Case__c: 'Provider has prior termination record – supervisor approval required' and Credentialing_Case__c.Supervisor_Approval_Required__c = TRUE

## Solution

Implement a Screen Flow 'TerminationClosureValidation' invoked via a Before-Save Record-Triggered Flow on Network_Termination_Case__c when Status__c changes to 'Closed – Complete'. The Screen Flow calls two Apex invocable actions: NPPESValidationAction and CAQHValidationAction, each making Named Credential-secured HTTP callouts to NPPES NPI Registry REST API and CAQH ProView API respectively. Timeout handling via Continuation (async callout pattern) for LWC-based screen if latency is a concern. Results displayed on a custom LWC screen component embedded in the flow. Override captured via a Flow text input and written to Network_Termination_Case__c.Override_Reason__c. Final status updates and ContentNote creation handled via Apex invocable post-flow action. Custom Metadata (Integration_Endpoint__mdt) stores NPPES and CAQH endpoint URLs for environment-agnostic configuration.
