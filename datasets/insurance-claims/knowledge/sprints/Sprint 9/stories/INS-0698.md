## Description

**Bug Report**

**Summary:** When multiple active Fee_Schedule__c records exist for the same CPT code and LOB, the Claim_Line__c fee schedule lookup Apex logic is returning the wrong rate — specifically the oldest effective record rather than the most recently effective record — causing underpayment or overpayment on adjudicated claims.

**Steps to Reproduce:**
1. Ensure two Fee_Schedule__c records exist with the same CPT code (e.g., '99213'), same LOB__c ('Medical'), same Network__c ('Preferred'), and overlapping EffectiveDate__c ranges (e.g., Record A: 01/01/2023–12/31/2024, Record B: 01/01/2024–12/31/2025).
2. Create a new Claim_Line__c with CPTCode__c = '99213', LOB__c = 'Medical', NetworkStatus__c = 'In-Network', DateOfService__c = 06/01/2024 (falls within both records).
3. Save the Claim_Line__c and observe the auto-populated Claim_Line__c.FeeSchedule__c lookup and Claim_Line__c.AllowedAmount__c.

**Expected Behavior:** The lookup should resolve to Fee_Schedule__c Record B (most recently effective, highest EffectiveDate__c start within the date of service range) and populate the correct AllowedAmount__c.

**Actual Behavior:** The lookup resolves to Fee_Schedule__c Record A (oldest record by CreatedDate) due to a missing ORDER BY clause in the SOQL query within 'FeeScheduleLookupService.cls', line 47. The AllowedAmount__c is populated with the stale rate from Record A.

**Financial Impact:** Estimated $14,200 in payment discrepancies identified across 38 claims in UAT environment.
**Environment:** Acme Insurance - UAT Sandbox
**Affected Release:** Sprint 8 deploy, persists in Sprint 9
**Frequency:** Reproducible whenever overlapping fee schedule records exist for the same CPT/LOB/Network combination.

## Acceptance Criteria

As per Description.

Fix must add ORDER BY Fee_Schedule__c.EffectiveDate__c DESC, Fee_Schedule__c.CreatedDate DESC with LIMIT 1 to the SOQL query in 'FeeScheduleLookupService.cls'.

Post-fix validation must confirm that for all 38 impacted Claim_Line__c records in UAT, Claim_Line__c.AllowedAmount__c is recalculated using the correct Fee_Schedule__c record.

A data fix script must be prepared and reviewed by the Claims Operations Manager before execution in UAT and Production.

Unit test coverage for 'FeeScheduleLookupService.cls' must be updated to include a test scenario with overlapping fee schedule records, asserting the correct (most recent) rate is returned — coverage must reach minimum 85%.

## Solution

Update 'FeeScheduleLookupService.cls' line 47 SOQL to add 'ORDER BY EffectiveDate__c DESC, CreatedDate DESC LIMIT 1'. Add a unique validation rule or duplicate rule on Fee_Schedule__c to warn when overlapping effective date ranges exist for the same CPT + LOB + Network combination to prevent recurrence. Write an anonymous Apex data fix script to re-invoke the fee schedule lookup for all affected Claim_Line__c records in UAT (identified by a custom report). Peer review required before Production deployment. Update 'FeeScheduleLookupServiceTest.cls' with overlap scenario. Schedule a hotfix deploy to UAT for validation within current sprint.
