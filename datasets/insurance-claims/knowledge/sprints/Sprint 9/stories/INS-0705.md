## Description

**Bug Report — PCN Auto-Creation: Missing Provider_Change_Notification__c on Batch NPI Updates**

**Environment:** Full Sandbox (Sprint 9 UAT)
**Severity:** High — Compliance risk; PCN records are required for audit trail per CMS guidelines

**Steps to Reproduce:**
1. Execute the nightly DataIntegration batch job 'NPPES_Provider_Sync_Batch' which updates Provider__c records from the NPPES feed.
2. Confirm that one or more Provider__c records have their Provider__c.NPI__c field updated by the batch.
3. Navigate to the updated Provider__c record.
4. Check the Provider_Change_Notification__c related list.

**Expected Behavior:**
Whenever Provider__c.NPI__c is modified, a Provider_Change_Notification__c (PCN) record should be automatically created with PCN__c.Change_Type__c = 'NPI Update', PCN__c.Old_Value__c = previous NPI, PCN__c.New_Value__c = updated NPI, PCN__c.Change_Source__c = 'NPPES Sync', and PCN__c.Provider__c = the updated Provider__c Id. This behavior is defined by the 'Provider_NPI_Change' Record-Triggered Flow.

**Actual Behavior:**
No Provider_Change_Notification__c records are created following batch NPI updates. Manual NPI updates via the UI correctly trigger PCN creation. The issue is isolated to batch-context updates.

**Root Cause Hypothesis:**
The 'Provider_NPI_Change' Record-Triggered Flow is configured with 'Run Asynchronously' on the After-Save path. Batch Apex jobs that update Provider__c records via Database.update() may be bypassing the Flow trigger due to a known Salesforce limitation where asynchronous Flow paths do not fire reliably from batch context when the batch already consumes async governor limits. Additionally, the batch class may be setting a custom bypass flag 'Flow_Bypass__c' = true on Provider__c records to improve performance, inadvertently suppressing the PCN flow.

## Acceptance Criteria

As per Description. Verify fix by running NPPES_Provider_Sync_Batch in sandbox and confirming Provider_Change_Notification__c records are created for all Provider__c records where NPI__c was modified. Confirm PCN__c.Change_Source__c = 'NPPES Sync' and PCN__c.Old_Value__c captures the pre-update NPI value correctly.

## Solution

Refactor PCN creation logic out of the Record-Triggered Flow and into an After-Insert/After-Update Apex Trigger 'ProviderChangeTrigger' on Provider__c, using a TriggerHandler pattern. The trigger handler class 'ProviderChangeTriggerHandler' compares Trigger.oldMap.get(id).NPI__c vs Trigger.newMap.get(id).NPI__c and bulk-inserts Provider_Change_Notification__c records. Remove the Flow_Bypass__c check from the batch class or ensure it is reset to false before DML. Retire the 'Provider_NPI_Change' Record-Triggered Flow after confirming the Apex trigger covers all change scenarios. Update the batch class to NOT set Flow_Bypass__c = true for NPI-related updates.
