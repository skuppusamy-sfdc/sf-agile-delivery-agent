## Description

**Bug Report — Coverage History Timeline DST Date Shift**

**Steps to Reproduce:**
1. Navigate to a Policy__c record where Coverage_Period__c.End_Date__c falls on or around November 5, 2023 (US DST fall-back date).
2. Open the Coverage History Timeline LWC (INS-0724).
3. Observe the rendered end date label on the timeline node.

**Expected Behavior:** Coverage_Period__c.End_Date__c = 2023-11-05 should display as '11/05/2023' in the user's local timezone.

**Actual Behavior:** The timeline renders the end date as '11/04/2023' — one day behind — because the LWC is converting the Date field value through a JavaScript Date object without anchoring to UTC midnight, causing the DST offset to roll the date back by one day.

**Environment:** Production sandbox (CS92). Reproducible for all Coverage_Period__c records with End_Date__c between 2023-11-05 and 2023-11-06. Not observed for Spring DST dates.

**Impact:** Claims Operations Managers are incorrectly identifying coverage lapses for members whose coverage ended on November 5, potentially triggering erroneous claim denials.

## Solution

Fix the JavaScript date parsing in the 'coverageHistoryTimeline' LWC by replacing 'new Date(dateString)' with a UTC-safe parser: parse the ISO date string by splitting on '-' and constructing 'new Date(Date.UTC(year, month-1, day))' before formatting with Intl.DateTimeFormat. Add a Jest unit test covering dates spanning both DST transition windows (March and November). Deploy fix to sandbox and validate against the 5 identified Policy__c records in CS92 before promoting to production.
