## Description

**Bug Report — Fee Schedule Engine / PNA Resolution**

**Environment:** Acme Insurance Full Sandbox (ACME_FS1) and Partial Sandbox (ACME_PS2)
**Severity:** High — blocking claims adjudication for multi-PNA providers; financial impact on Payment__c calculation

**Steps to Reproduce:**
1. Identify a Provider__c record that has 2 or more active Provider_Network_Agreement__c records with different PNA.Network_Tier__c values (e.g., 'Tier 1' and 'Tier 2') and different linked Fee_Schedule__c records.
2. Create a new Claim__c record linked to this provider (Claim__c.Rendering_Provider__c = Provider Id).
3. Add a Claim_Line__c record with a valid CPT code.
4. Trigger the fee schedule resolution process (either via the 'Calculate Reimbursement' button or the automated Record-Triggered Flow on Claim_Line__c insert).
5. Observe the value of Claim_Line__c.Applied_Fee_Schedule__c and Claim_Line__c.Allowed_Amount__c.

**Expected Behavior:** The fee schedule resolution logic should select the PNA with the highest Network_Tier__c priority (Tier 1 > Tier 2) and populate Claim_Line__c.Applied_Fee_Schedule__c with the corresponding Fee_Schedule__c Id and calculate Claim_Line__c.Allowed_Amount__c accordingly.

**Actual Behavior:** Claim_Line__c.Applied_Fee_Schedule__c is null and Claim_Line__c.Allowed_Amount__c = 0.00 for providers with more than one active PNA. Providers with a single active PNA are unaffected.

**Root Cause Hypothesis:** The Apex class 'FeeScheduleResolutionService.cls' method resolveFeeSchedule(providerId, claimLineId) likely uses a SOQL query with LIMIT 1 on Provider_Network_Agreement__c without an ORDER BY clause, causing non-deterministic record selection; when the non-priority PNA is selected, the associated Fee_Schedule__c may be inactive or mismatched to the claim's LOB, causing the null return.

**Impacted Claims:** 12 Claim__c records in ACME_FS1 with Claim_Line__c.Allowed_Amount__c = 0.00 identified via report 'FS_Null_Resolution_Report'.

**Attachments:** FeeScheduleResolutionService_DebugLog.txt, Impacted_Claims_List.csv

## Acceptance Criteria

As per Description. Fix verified when: Claim_Line__c.Applied_Fee_Schedule__c is correctly populated for all providers with 1 or more active PNAs. Claim_Line__c.Allowed_Amount__c must be > 0 for any Claim_Line__c with a valid CPT code and an active fee schedule. Regression test must cover single-PNA provider, multi-PNA same tier, and multi-PNA different tier scenarios.

## Solution

Fix 'FeeScheduleResolutionService.cls' resolveFeeSchedule() method: update SOQL on Provider_Network_Agreement__c to include ORDER BY Network_Tier_Priority__c ASC LIMIT 1 to enforce deterministic tier selection. Add null-check guard before accessing Fee_Schedule__c lookup. Add a custom field PNA.Network_Tier_Priority__c (Number) to encode tier ranking (Tier 1 = 1, Tier 2 = 2, etc.) and populate via data migration script. Update Apex unit tests in 'FeeScheduleResolutionServiceTest.cls' to cover multi-PNA scenarios. Re-run 'FS_Null_Resolution_Report' post-fix to confirm 0 impacted records. Backfill the 12 identified Claim_Line__c records via Data Loader after fix deployment.
