## Description

**Bug Report**

**Summary:** The Fee_Schedule__c lookup logic returns the base allowed amount without applying modifier-based reimbursement reductions, causing Claim_Line__c.Allowed_Amount__c to be overstated for CPT codes submitted with modifiers 51, 59, and 80.

**Steps to Reproduce:**
1. Create a Claim__c with Claim__c.LOB__c = 'Commercial' and at least two Claim_Line__c records.
2. Set Claim_Line__c.CPT_Code__c = '99213' and Claim_Line__c.Modifier_1__c = '51' on the second line.
3. Trigger the fee schedule lookup by changing Claim__c.Status__c to 'Pending Adjudication'.
4. Observe Claim_Line__c.Allowed_Amount__c on the second line.

**Expected Result:** Claim_Line__c.Allowed_Amount__c should reflect the modifier-adjusted rate from Fee_Schedule__c.Modifier_Rate__c (e.g., 50% reduction for modifier 51).

**Actual Result:** Claim_Line__c.Allowed_Amount__c is populated with the base Fee_Schedule__c.Base_Allowed_Amount__c, ignoring the modifier entirely. This results in an overpayment of up to 50% on affected claim lines.

**Environment:** Acme Insurance – Full Sandbox (Sprint 9 deploy)
**Frequency:** Reproducible 100% of the time when Modifier_1__c is populated.
**Impact:** High — affects all commercial claims with surgical or assistant-surgeon modifiers.

## Acceptance Criteria

As per Description.

Given a Claim_Line__c with Claim_Line__c.Modifier_1__c = '51', '59', or '80',
When the fee schedule lookup Apex method 'FeeScheduleService.resolveAllowedAmount()' executes,
Then it must query Fee_Schedule__c WHERE CPT_Code__c = :cptCode AND Modifier__c = :modifier AND Effective_Date__c <= :dateOfService AND (End_Date__c >= :dateOfService OR End_Date__c = null) and apply Fee_Schedule__c.Modifier_Rate__c to calculate the final Claim_Line__c.Allowed_Amount__c.

Given no modifier-specific Fee_Schedule__c record exists for the CPT/modifier combination,
When the lookup runs,
Then the system must fall back to the base Fee_Schedule__c.Base_Allowed_Amount__c and log a warning in Claim_Line__c.Adjudication_Notes__c = 'No modifier rate found — base rate applied'.

## Solution

Fix the Apex class 'FeeScheduleService' method 'resolveAllowedAmount()'. Current logic performs a single SOQL query filtering only on CPT_Code__c and Effective_Date__c, ignoring Modifier__c. Updated logic: first attempt a query with all three filters (CPT, modifier, date range); if no result, fall back to base rate query (CPT + date only, Modifier__c = null). Apply Fee_Schedule__c.Modifier_Rate__c as a decimal multiplier to Base_Allowed_Amount__c when a modifier record is found. Add a unit test class 'FeeScheduleServiceTest' with test methods covering modifier 51, 59, 80, and no-modifier scenarios. Deploy fix to Full Sandbox for regression testing before promoting to UAT.
