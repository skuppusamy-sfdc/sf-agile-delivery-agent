## Description

**Bug Report – INS-0738**

An Apex validation rule on Provider_Network_Agreement__c (PNA) is incorrectly blocking record creation when PNA renewal records are pushed from the Conga CLM integration via the REST API. The validation is intended to prevent backdated PNA effective dates for manually created records, but it is also triggering for renewal records where PNA.EffectiveDate__c is set to a future date by the CLM system.

**Steps to Reproduce:**
1. Complete a PNA renewal contract execution in Conga CLM for any provider.
2. Observe the Conga CLM outbound webhook firing the POST request to the Salesforce REST API endpoint /services/apexrest/PNAIntegration/.
3. The integration user (named credential 'Conga_CLM_Integration_User') attempts to create a new Provider_Network_Agreement__c record with PNA.EffectiveDate__c = [contract execution date + 1 day] and PNA.RecordOrigin__c = 'Conga CLM'.
4. The record creation fails with HTTP 400 and error body: 'FIELD_CUSTOM_VALIDATION_EXCEPTION: PNA effective date cannot precede today's date for new agreements.'

**Expected Behavior:** PNA records created via the Conga CLM integration (identifiable by PNA.RecordOrigin__c = 'Conga CLM') should bypass the backdating validation rule OR the validation logic should correctly evaluate the EffectiveDate__c as future-dated and allow the record creation to succeed.

**Actual Behavior:** The validation fires regardless of PNA.RecordOrigin__c value and regardless of whether EffectiveDate__c is in the past or future. All Conga CLM-originated PNA renewals are blocked. Approximately 23 PNA renewal records are currently queued and failing in the integration error log.

**Environment:** Acme Insurance – Full Sandbox (FS1) and Production
**Affected Validation Rule:** PNA_EffectiveDateBackdatingCheck on Provider_Network_Agreement__c
**Affected Integration:** Conga CLM outbound webhook → Salesforce REST Apex endpoint
**Regression:** Introduced when Apex validation was tightened in Sprint 8 deployment on 2024-10-21.

## Solution

Review the validation rule formula 'PNA_EffectiveDateBackdatingCheck' on Provider_Network_Agreement__c. Modify the formula to add a bypass condition: wrap the existing logic with AND(NOT(ISPICKVAL(RecordOrigin__c, 'Conga CLM')), [existing date comparison logic]). This ensures the backdating check only fires for manually created records. Additionally, fix the secondary issue: the current date comparison formula uses DATE(YEAR(TODAY()), MONTH(TODAY()), DAY(TODAY())) which appears to be evaluating incorrectly — replace with direct TODAY() reference. For the 23 queued failed records, create a Data Loader re-import CSV from the Conga CLM integration error log and re-process them via the REST endpoint after the validation fix is deployed. Add a test method in PNAIntegrationServiceTest to cover the Conga CLM record origin bypass scenario and confirm validation does not fire when RecordOrigin__c = 'Conga CLM'. Deploy validation rule change to FS1 for QA sign-off before promoting to Production.
