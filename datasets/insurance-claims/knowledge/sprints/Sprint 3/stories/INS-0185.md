## Description

As a Credentialing Specialist, I need the system to perform an automated TIN (Tax Identification Number) verification check during the credentialing workflow for both individual providers (Provider__c) and facilities (Facility__c / Business Account), so that we remain compliant with IRS B-Notice requirements and prevent improper payments to unverified tax entities.

## Acceptance Criteria

Given a Credentialing_Case__c record advances to Status__c = 'TIN Verification Pending',
When the status transition is saved,
Then the system initiates a TIN match request using the value stored in Provider__c.TIN__c (or Facility__c.TIN__c for facility credentialing cases) against the configured IRS TIN Matching API endpoint.

Given the TIN match returns a 'Match' result,
When the response is processed,
Then Credentialing_Case__c.TIN_Verification_Status__c is set to 'Verified', Credentialing_Case__c.TIN_Verified_Date__c is stamped, and the workflow automatically advances Status__c to 'Background Check Pending'.

Given the TIN match returns a 'Mismatch' or 'Not Found' result,
When the response is processed,
Then Credentialing_Case__c.TIN_Verification_Status__c is set to 'Failed', Status__c is set to 'Pending – TIN Review', a high-priority Task is created for the Credentialing Specialist, and a Provider_Change_Notification__c (PCN) record is auto-created with PCN.Type__c = 'TIN Discrepancy' and PCN.Status__c = 'Open'.

Given a Facility__c (Business Account) is the subject of the Credentialing_Case__c (identified by Credentialing_Case__c.Entity_Type__c = 'Facility'),
When TIN verification runs,
Then the system reads TIN from Facility__c.TIN__c on the linked Business Account and uses the legal entity name from Facility__c.Legal_Entity_Name__c in the IRS match request.

Given TIN verification has already been completed successfully within the current calendar year for the same TIN,
When a new Credentialing_Case__c triggers TIN verification,
Then the prior result is reused, Credentialing_Case__c.TIN_Verification_Status__c = 'Verified (Prior Year)' is set, and a note is added to the case chatter feed.

Given a Compliance Officer reviews a TIN Failed case,
When they update Credentialing_Case__c.TIN_Override_Approved__c = true and provide Credentialing_Case__c.TIN_Override_Justification__c,
Then the workflow is permitted to advance past TIN Verification with Status__c = 'TIN Override – Approved' and the override is logged in the Compliance_Audit_Log__c object.

## Solution

A Record-Triggered Flow on Credentialing_Case__c fires on status change to 'TIN Verification Pending'. The flow calls an Apex invocable method (TINVerificationService.cls) which performs an async Queueable callout to the IRS TIN Matching API via a Named Credential. Entity type branching (Provider vs. Facility) is handled in the Apex service layer. PCN auto-creation uses a separate Apex helper to maintain transaction boundaries. An override path is handled by a second Record-Triggered Flow monitoring TIN_Override_Approved__c field changes, with a custom Permission Set restricting the override field to Compliance Officers only. All TIN data is encrypted at rest using Salesforce Shield Platform Encryption on TIN__c fields.
