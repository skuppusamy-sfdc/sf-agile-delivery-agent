## Description

As a Credentialing Specialist, I need the system to automatically validate a provider's NPI against the NPPES registry when a new Credentialing_Case__c record is created or updated, so that I can ensure only providers with active, correctly classified NPIs are admitted to the network and reduce manual lookup effort.

## Acceptance Criteria

Given a Credentialing_Case__c record is saved with a populated Provider__c lookup and a non-null NPI__c field on the associated Provider__c (Person Account),
When the record is created or the NPI__c field is modified,
Then an asynchronous callout to the NPPES NPI Registry REST API is triggered within 30 seconds and the results are written back to Provider__c.NPI_Validation_Status__c (values: 'Valid', 'Invalid', 'Not Found', 'Pending').

Given the NPPES response returns an active NPI record,
When the response is parsed,
Then Provider__c.NPI_Taxonomy_Code__c, Provider__c.NPI_Entity_Type__c ('Individual' or 'Organization'), and Provider__c.NPI_Last_Validated_Date__c are populated with values from the response payload.

Given the NPPES response indicates the NPI is deactivated or not found,
When the validation completes,
Then Credentialing_Case__c.Status__c is set to 'Pending – NPI Issue', a Task is created and assigned to the owning Credentialing Specialist with Subject = 'NPI Validation Failed – Manual Review Required', and Provider__c.NPI_Validation_Status__c = 'Invalid' or 'Not Found' respectively.

Given the NPPES API is unavailable (HTTP 5xx or timeout),
When the callout fails,
Then a Platform Event (NPI_Validation_Event__e) is published with Retry_Count__c incremented, the job is retried up to 3 times with exponential backoff, and after 3 failures Credentialing_Case__c.NPI_Validation_Status__c is set to 'Pending' with a warning banner surfaced on the record page.

-Given NPI validation is triggered on every Credentialing_Case__c save regardless of field change, Then the callout fires unconditionally.- (Superseded: callout only fires when NPI__c is new or changed to reduce API consumption.)

Given a bulk import of more than 200 Credentialing_Case__c records via Data Loader,
When records are inserted,
Then NPI validation callouts are queued via a Queueable Apex chain respecting governor limits and no more than 100 callouts are issued per transaction batch.

## Solution

Implement a Record-Triggered Flow on Credentialing_Case__c (entry criteria: NPI__c is changed OR record is new) that invokes an Apex Action (InvocableMethod) to enqueue a Queueable Apex class (NPPESValidationQueueable). The Queueable class performs an HTTP callout to the NPPES REST endpoint (https://npiregistry.cms.hhs.gov/api/). Named Credential 'NPPES_Registry' stores the endpoint. Response parsing is handled in a dedicated NPPESResponseParser Apex class. Retry logic is managed via a Platform Event (NPI_Validation_Event__e) consumed by a separate Trigger-on-Platform-Event handler. A Custom Metadata Type (NPI_Validation_Config__mdt) stores timeout thresholds and retry limits for environment-specific tuning. LWC toast notifications surface validation status on the Credentialing_Case__c record page via a FlexCard overlay.
