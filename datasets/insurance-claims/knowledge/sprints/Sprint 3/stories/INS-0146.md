## Description

**Bug Report**

**Summary:** A Credentialing Specialist can manually set Credentialing_Case__c.Status__c to 'Approved' even when Credentialing_Case__c.NPI_Validation_Status__c is 'Failed' or 'Pending — API Unavailable', bypassing the NPI validation gate and potentially allowing unapproved providers into the network.

**Steps to Reproduce:**
1. Navigate to any Credentialing_Case__c record where NPI_Validation_Status__c = 'Failed'.
2. Click Edit on the record.
3. Manually change Status__c from 'On Hold' to 'Approved' using the picklist.
4. Click Save.
5. Observe: Record saves successfully with Status__c = 'Approved' and NPI_Validation_Status__c = 'Failed'.

**Expected Behavior:** Salesforce should throw a validation error preventing Status__c from being set to 'Approved' unless NPI_Validation_Status__c = 'Validated'.

**Actual Behavior:** No validation fires; record saves with contradictory field values. This can lead to provider activation in the network without a valid NPI, creating compliance and claim payment risk.

**Environment:** Acme Insurance Full Sandbox (CS47)
**Reproducible in Production:** Not yet tested — blocked pending fix.
**Regression:** Introduced after the record-triggered flow deployed in Sprint 2 (INS-0131) was modified to remove a now-deprecated validation node.

## Acceptance Criteria

As per Description. Validation rule must prevent Credentialing_Case__c.Status__c = 'Approved' when NPI_Validation_Status__c is not 'Validated'. Fix must be verified in Full Sandbox before production deployment.

## Solution

Add a Salesforce Validation Rule on Credentialing_Case__c: ISPICKVAL(Status__c, 'Approved') && NOT(ISPICKVAL(NPI_Validation_Status__c, 'Validated')). Error message: 'Provider NPI must be successfully validated before this Credentialing Case can be Approved. Current NPI Validation Status: {!NPI_Validation_Status__c}'. Additionally, reinstate the removed validation node in the Record-Triggered Flow (INS-0131 flow version) as a defense-in-depth measure. Deploy validation rule to sandbox, execute regression test suite, then promote to production via change set.
