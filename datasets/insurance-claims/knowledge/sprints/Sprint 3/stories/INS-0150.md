## Description

**Bug Report**

**Summary:** When a DocuSign envelope for a Provider_Network_Agreement__c reaches 'Completed' status for an NPI-2 Organizational provider (stored as a Facility__c / Business Account), the Provider__c.Network_Status__c field is NOT updated to 'In-Network'. The same workflow functions correctly for NPI-1 Individual providers (Person Accounts). This is a regression introduced during the PNA workflow deployment in Sprint 3 (INS-0148).

**Steps to Reproduce:**
1. Identify or create a Facility__c (Business Account) record with a valid NPI-2.
2. Complete the credentialing intake OmniScript (INS-0147) selecting 'Organizational Provider (NPI-2)'.
3. Approve the resulting Credentialing_Case__c — a PNA__c record is created and a DocuSign envelope is sent.
4. Complete the DocuSign signature as the recipient.
5. Observe the DocuSign Connect callback fires: PNA__c.Status__c updates to 'Active' and PNA__c.Signed_Date__c is stamped correctly.
6. Navigate to the related Facility__c (Business Account) record.
7. Observe: Account.Network_Status__c remains unchanged (still 'Pending' or prior value); it is NOT set to 'In-Network'.

**Expected Behavior:** Facility__c (Business Account) Account.Network_Status__c should be set to 'In-Network' upon DocuSign completion, mirroring the behavior for Provider__c (Person Account).

**Root Cause Hypothesis:** The Apex REST callback handler in 'PNADocuSignCallbackController' references Provider__c (Person Account record type) via a hardcoded RecordType check when resolving the parent account from PNA__c.Provider__c. Facility__c records use the 'Business_Account' RecordType and are not matched by this condition, causing the Network_Status__c update DML to be silently skipped.

**Environment:** Acme Insurance Full Sandbox (CS47)
**Frequency:** 100% reproducible for all NPI-2 Organizational providers.
**Business Impact:** HIGH — Organizational providers completing credentialing are not being activated in-network, blocking claim adjudication for facility-based claims.

## Acceptance Criteria

As per Description. Fix must handle both Person Account (NPI-1) and Business Account (NPI-2) record types in the DocuSign callback handler. Regression tests must cover both provider types. Verified in Full Sandbox before production deployment.

## Solution

Update 'PNADocuSignCallbackController.handleEnvelopeComplete()' to remove the hardcoded RecordType check and instead query the parent Account from PNA__c.Provider__c regardless of RecordType, then perform the Network_Status__c = 'In-Network' DML update on the resolved Account record. Add a null-safe check for the Account lookup. Update existing Apex unit test class 'PNADocuSignCallbackControllerTest' to add test methods covering the Business Account / NPI-2 scenario using a @TestSetup method that creates both Person Account and Business Account provider records. Deploy via change set to Full Sandbox for QA validation, then promote to production. Add a regression test case to the Sprint 3 regression suite.
