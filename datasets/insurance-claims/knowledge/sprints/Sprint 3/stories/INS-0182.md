## Description

**Bug Report — Credentialing_Case__c Due_Date__c off by one day for Pacific timezone users**

**Steps to Reproduce:**
1. Log in as a user whose Salesforce profile timezone is set to 'America/Los_Angeles' (Pacific Time).
2. Create a new Provider__c record with Provider_Status__c = 'Pending Credentialing' and a valid NPI__c.
3. Save the record. The Record-Triggered Flow from INS-0175 fires and creates a Credentialing_Case__c.
4. Open the newly created Credentialing_Case__c and observe Due_Date__c.
5. Note: Due_Date__c is set to TODAY() + 29 days instead of the expected TODAY() + 30 days.

**Expected Behavior:**
Credentialing_Case__c.Due_Date__c = the date 30 calendar days from the date of Provider__c creation, regardless of the running user's timezone.

**Actual Behavior:**
For users in Pacific Time (UTC-8), the Flow formula TODAY() evaluates to the previous calendar day during the hours of 00:00–08:00 UTC, causing Due_Date__c to be one day short.

**Environment:** Full Sandbox (FS-01) and Production
**Regression:** Present since INS-0175 go-live on 2024-10-28
**Frequency:** Reproducible consistently between 4:00 PM–8:00 PM Pacific (00:00–04:00 UTC next day)
**Severity:** Medium — affects SLA tracking and specialist workload planning accuracy.

## Solution

Root cause: Salesforce Flow formula TODAY() returns a date value based on the running user's timezone locale, not UTC. Fix: Replace the Flow formula TODAY() + 30 in the Create Records element (INS-0175 flow 'Provider_Credentialing_Case_AutoCreate') with an Apex-invocable method 'DateCalculationService.getUTCDatePlusDays(30)' that uses System.today() in an Apex context, which always returns the server-side UTC date unaffected by user timezone settings. Alternatively, use a before-save Apex Trigger on Credentialing_Case__c to set Due_Date__c = System.today() + 30 on insert, overriding the Flow-set value. Preferred approach: Apex Trigger for reliability across bulk inserts. Update unit tests in CredentialingCaseAutoCreate_Test to include assertions for timezone-boundary scenarios. Validate fix in FS-01 by simulating record creation at 07:00 UTC with a Pacific timezone test user.
