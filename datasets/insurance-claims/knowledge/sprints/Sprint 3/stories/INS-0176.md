## Description

As a Provider Network Specialist, I need NPI numbers entered on Provider__c and Claim__c records to be validated in real-time against the CMS NPPES registry API so that only legitimate, active NPIs are accepted into our system and downstream claims adjudication errors caused by invalid provider identifiers are eliminated.

Extends INS-0175 (Credentialing Case auto-creation flow).

NPPES REST endpoint: https://npiregistry.cms.hhs.gov/api/?number={NPI}&version=2.1

## Acceptance Criteria

Given a user enters a 10-digit value in Provider__c.NPI__c and saves the record,
When the NPPES callout is executed,
Then if the NPI is found and result_count = 1, Provider__c.NPI_Validated__c = TRUE, Provider__c.NPI_Validation_Date__c = TODAY(), and Provider__c.NPI_Status__c = 'Active'.

Given the NPPES API returns result_count = 0 for the submitted NPI,
When validation completes,
Then Provider__c.NPI_Validated__c = FALSE, Provider__c.NPI_Status__c = 'Invalid', and a validation warning toast is displayed to the user: 'NPI not found in NPPES registry. Please verify before proceeding.'

Given a Claim__c record is submitted with Billing_Provider_NPI__c populated,
When the claim moves to Status__c = 'Submitted',
Then an asynchronous NPPES validation is triggered and Claim__c.Provider_NPI_Valid__c is set accordingly before adjudication begins.

Given the NPPES API is unreachable (timeout or HTTP 5xx),
When the callout fails,
Then the record save is NOT blocked, NPI_Status__c = 'Validation Pending', and a retry Platform Event 'NPI_Validation_Retry__e' is published for processing within 15 minutes.

Given an NPI that is valid but taxonomy code does not match Provider__c.Specialty__c,
When validation returns,
Then a non-blocking warning is logged on Provider__c.NPI_Taxonomy_Warning__c = TRUE and a chatter post is made on the record tagging the assigned Credentialing Specialist.

-Given NPI length != 10, display error 'NPI must be exactly 10 digits'- (handled by existing field-level validation rule NPI_Format_Check, deprecated from this AC)

## Solution

Build an Apex class 'NPPESValidationService' implementing the HttpCallout interface against the NPPES REST API. Use a Named Credential 'NPPES_Registry' to manage endpoint and auth. Invoke synchronously from a Before-Save Record-Triggered Flow on Provider__c using an Apex Action, and asynchronously via a Queueable Apex job for Claim__c NPI validation to avoid governor limit issues on bulk claim submissions. Implement exponential backoff retry logic using Platform Events and a scheduled Apex job 'NPPESRetryBatch'. Store raw API response JSON in Provider__c.NPI_Raw_Response__c (Long Text Area) for audit purposes. Create a Connected App and configure Remote Site Settings for the NPPES endpoint. Unit test coverage target: 90%, including mock HttpCalloutMock implementations for success, not-found, and timeout scenarios.
