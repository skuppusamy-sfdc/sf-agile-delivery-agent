## Description

**Bug Report**

**Summary:** When a Provider__c NPI is corrected and re-validated (NPI_Validation_Status__c changes to 'Valid'), associated Claim__c records that were pended due to 'Invalid Billing Provider NPI' are not automatically released from pend status. The Claim__c.Status__c remains 'Pended' indefinitely and requires manual intervention by Claims Operations.

**Steps to Reproduce:**
1. Create a Provider__c record with an intentionally invalid NPI__c value (e.g., '1234567890').
2. Submit a Claim__c record with Billing_Provider_NPI__c = '1234567890'. Observe Claim__c.Status__c = 'Pended' and Claim__c.Pend_Reason__c = 'Invalid Billing Provider NPI'.
3. Correct the Provider__c.NPI__c to a valid NPI and trigger NPPES re-validation (INS-0137 flow).
4. Observe Provider__c.NPI_Validation_Status__c updates to 'Valid'.
5. Navigate to the associated Claim__c record.

**Expected Result:** Claim__c.Status__c automatically transitions from 'Pended' to 'Resubmitted' (or equivalent re-queue status) and Claim__c.Pend_Reason__c is cleared.

**Actual Result:** Claim__c.Status__c remains 'Pended'. Claim__c.Pend_Reason__c = 'Invalid Billing Provider NPI' is not cleared. No automated re-adjudication is triggered.

**Environment:** Full Sandbox (FS-01)
**Frequency:** 100% reproducible
**Impact:** Claims Operations Manager reports approximately 40+ claims per week requiring manual pend release, creating SLA breach risk.

## Acceptance Criteria

As per Description.

Additional verification:
Given Provider__c.NPI_Validation_Status__c transitions to 'Valid'
When the Record-Triggered Flow (After Update) on Provider__c fires
Then all related Claim__c records where Claim__c.Billing_Provider_NPI__c = Provider__c.NPI__c AND Claim__c.Pend_Reason__c = 'Invalid Billing Provider NPI' AND Claim__c.Status__c = 'Pended'
Are updated to Claim__c.Status__c = 'Resubmitted' and Claim__c.Pend_Reason__c = NULL within the same transaction or via queued batch job if record volume > 200

## Solution

Root cause: The NPPES validation Record-Triggered Flow on Provider__c (INS-0137) updates Provider__c fields but contains no downstream logic to re-evaluate pended Claim__c records. Fix: Add a decision node to the existing Provider__c After-Update Flow that checks if NPI_Validation_Status__c changed to 'Valid'. If true and related Claim count <= 200, use a Loop + Update Records element to release pended claims in-transaction. If count > 200, invoke a Queueable Apex job (NPIValidationClaimRelease_Queueable.cls) via Flow's Apex Action element to process in bulk with proper governor limit handling. Add regression test class (NPIValidationClaimReleaseTest.cls) with 95%+ coverage.
