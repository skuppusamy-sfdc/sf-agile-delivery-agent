## Description

As a Credentialing Specialist, I need NPI numbers entered on Provider__c and Claim__c records to be validated in real time against the CMS NPPES registry, so that we prevent credentialing and claims processing errors caused by invalid, deactivated, or mismatched NPI values. This story extends INS-0106 (Provider data model build-out) by adding live external validation to the existing NPI__c field.

## Acceptance Criteria

Given a user enters or updates a value in Provider__c.NPI__c
When the record is saved
Then an asynchronous callout is made to the NPPES NPI Registry REST API (https://npiregistry.cms.hhs.gov/api/) with the entered NPI
And the response is parsed and stored in Provider__c.NPI_Validation_Status__c ('Valid', 'Invalid', 'Deactivated', 'Not Found')
And Provider__c.NPI_Last_Validated_Date__c is stamped with the current datetime

Given the NPPES API returns a valid NPI record
When the API response Provider_First_Name and Provider_Last_Name are compared to Provider__c.FirstName and Provider__c.LastName
Then if mismatch is detected, Provider__c.NPI_Name_Mismatch__c = TRUE and a warning banner is displayed on the Provider__c record page via LWC
And Credentialing_Case__c.Flags__c is updated to include 'NPI Name Mismatch'

Given a Claim__c record is submitted with Billing_Provider_NPI__c populated
When the claim reaches Status__c = 'Submitted'
Then the same NPPES validation logic is invoked via Platform Event (NPI_Validation_Request__e)
And if NPI_Validation_Status__c = 'Invalid' or 'Deactivated', Claim__c.Status__c is set to 'Pended' and Claim__c.Pend_Reason__c = 'Invalid Billing Provider NPI'

Given the NPPES API is unavailable or times out (>5 seconds)
When the callout fails
Then Provider__c.NPI_Validation_Status__c = 'Validation Pending' and a retry is queued via Schedulable Apex within 15 minutes
And no hard block is placed on record save during API downtime

Given NPI_Validation_Status__c = 'Deactivated'
When the Provider__c record is part of an active Provider_Network_Agreement__c
Then the PNA.Status__c is flagged as 'Requires Review' and the assigned Provider Relations Specialist is notified via Salesforce In-App Notification

## Solution

Build an Apex class (NPPESIntegrationService.cls) implementing HttpCallout to the NPPES REST API. Invoke asynchronously via a @future(callout=true) method triggered from a Record-Triggered Flow (After Save) on Provider__c and via Platform Event subscriber (NPI_Validation_Request__e) for Claim__c. Store Named Credential 'NPPES_Registry' for endpoint management. Parse JSON response with a wrapper class (NPPESResponseWrapper.cls). LWC component (npiValidationBanner) reads NPI_Validation_Status__c and NPI_Name_Mismatch__c to render contextual warnings on the Provider__c record page. Retry logic implemented as SchedulableApex (NPPESRetryScheduler.cls). Custom metadata type (NPI_Validation_Config__mdt) stores timeout thresholds and retry intervals for admin configurability.
