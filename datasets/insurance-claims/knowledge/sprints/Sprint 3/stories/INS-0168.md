## Description

As a Credentialing Specialist, I need the system to validate an entered NPI against the CMS NPPES registry and auto-populate verified demographic fields on the Provider__c record so that credentialing data entry is accurate, consistent, and reduces manual re-keying errors. Extends INS-0167 by ensuring NPI integrity prior to Credentialing_Case__c creation.

## Acceptance Criteria

Given a user enters a 10-digit value in Provider__c.NPI__c and saves the record
When the NPPES API call is made
Then the system returns a 200 response and populates Provider__c.NPPES_Verified__c = true, Provider__c.NPPES_Last_Verified_Date__c = TODAY(), and Provider__c.Provider_Type__c from the enumeration type returned by NPPES

Given the NPPES API returns an 'entity not found' response for the submitted NPI
When the validation completes
Then Provider__c.NPPES_Verified__c = false, a warning banner is displayed on the record page: 'NPI not found in NPPES registry. Please verify before proceeding.', and the Network_Participation_Status__c field is locked from advancing to 'Pending Credentialing'

Given a valid NPI is confirmed by NPPES
When the response is parsed
Then the following fields are auto-populated if blank: Provider__c.FirstName, Provider__c.LastName, Provider__c.Primary_Taxonomy_Code__c, Provider__c.Practice_State__c, Provider__c.Enumeration_Date__c — existing non-blank values are NOT overwritten

Given the NPPES API is unavailable (timeout or 5xx response)
When the callout fails
Then the system logs an error to the Integration_Error_Log__c object with fields: Error_Source__c = 'NPPES', Error_Message__c = [response body], Related_Record_Id__c = Provider__c.Id, and the user sees a non-blocking toast: 'NPI verification temporarily unavailable. Validation will retry automatically.'

Given a Provider__c.NPI__c value is not exactly 10 numeric digits
When the record is saved
Then a Validation Rule fires with message: 'NPI must be exactly 10 numeric digits per CMS standards.'

-Given NPI verification, also validate TIN against IRS TIN Matching API in the same transaction- (deprecated — TIN matching deferred to Phase 2)

## Solution

Apex Queueable class (NPPESValidationQueueable) invoked from a Record-Triggered Flow (after-save, async path) on Provider__c insert/update where NPI__c is changed. Queueable performs HTTP callout to NPPES public API (https://npiregistry.cms.hhs.gov/api/). Response parsed via JSON deserialization into a wrapper class. Field mapping handled by a dedicated NPPESResponseMapper Apex class. Integration errors written to Integration_Error_Log__c via a separate Apex utility. Custom Metadata Type (NPPES_Config__mdt) stores endpoint URL and timeout values for environment flexibility. Retry logic implemented via Platform Event (NPI_Validation_Retry__e) published on callout failure, consumed by a trigger handler to re-enqueue after 15 minutes.
