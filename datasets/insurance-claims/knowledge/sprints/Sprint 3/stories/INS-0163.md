## Description

**Bug Report — NPI Validation Callout Fires on Every Provider__c Save**

**Environment:** Full Sandbox (CS-47)
**Reported By:** Rachel Torres (Provider Network Specialist)
**Related Story:** INS-0160

**Steps to Reproduce:**
1. Open an existing Provider__c record where Provider__c.NPI__c is already populated and Provider__c.NPI_Status__c = 'Valid'.
2. Edit any unrelated field — e.g., update Provider__c.Phone or Provider__c.MailingCity.
3. Save the record.
4. Observe: The NPPES API callout fires again, overwriting Provider__c.NPI_Validated_Date__c with today's date and resetting Provider__c.NPI_Status__c to 'Validation Pending' momentarily before resolving back to 'Valid'.

**Expected Behavior:** The NPPES callout should only fire when Provider__c.NPI__c value changes.

**Actual Behavior:** The callout fires on every save of the Provider__c record regardless of whether NPI__c has changed, causing unnecessary API consumption, potential NPPES rate-limit errors, and confusing status flicker on the Provider 360 FlexCard (INS-0161).

**Impact:** High — NPPES has a rate limit of 20 requests/second per IP. In bulk data loads (e.g., Workday provider imports), this causes HTTP 429 errors and failed validations across hundreds of records.

## Solution

Fix the ProviderNPITrigger (from INS-0160) to include an NPI change detection guard. In the trigger handler class, compare Trigger.new NPI__c value against Trigger.oldMap NPI__c value before enqueuing NPPESValidationQueueable. Add conditional: if (newProvider.NPI__c == oldProvider.NPI__c) continue; to skip unchanged records. Also add a static boolean flag in the trigger handler to prevent recursive re-entry. Update the trigger unit test class (ProviderNPITriggerTest) to add a test method asserting zero callout invocations when NPI__c is unchanged. Deploy via change set to Full Sandbox for regression before production push.
