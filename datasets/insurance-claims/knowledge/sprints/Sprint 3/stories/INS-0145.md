## Description

As a Credentialing Specialist, I need Salesforce to automatically create a Credentialing_Case__c record when a provider completes or re-attests their CAQH ProView profile, so that credentialing workflows are initiated without manual intervention and cycle times are reduced. This story extends the NPI validation groundwork established in INS-0143.

## Acceptance Criteria

Given Acme's CAQH integration middleware posts a webhook payload to the Salesforce Experience Cloud API endpoint upon provider attestation completion,
When the payload contains a valid caqh_provider_id and npi fields,
Then Salesforce matches the NPI to an existing Provider__c.NPI__c and creates a Credentialing_Case__c with Credentialing_Case__c.Origin__c = 'CAQH Attestation', Credentialing_Case__c.CAQH_Provider_ID__c populated, and Credentialing_Case__c.Status__c = 'New'.

Given the CAQH webhook payload is received but no matching Provider__c.NPI__c exists,
When the lookup returns zero results,
Then a new Provider__c (Person Account) is created in 'Pending Verification' status, the Credentialing_Case__c is created with Credentialing_Case__c.Provider_Match_Status__c = 'Unmatched — Review Required', and an email alert is sent to the Credentialing queue.

Given a Credentialing_Case__c with the same CAQH_Provider_ID__c and Status__c not in ('Closed', 'Withdrawn') already exists,
When a duplicate webhook is received within 30 days,
Then no new Credentialing_Case__c is created; instead, Credentialing_Case__c.Last_CAQH_Attestation_Date__c is updated on the existing record.

Given the Credentialing_Case__c is successfully created from CAQH attestation,
When the record is committed,
Then the NPI validation flow from INS-0143 is triggered automatically via the Record-Triggered Flow entry criteria.

Given the inbound webhook payload fails schema validation,
When required fields (caqh_provider_id, npi, attestation_date) are missing,
Then the request returns HTTP 400, an error log is written to Integration_Error_Log__c, and no Salesforce records are created.

## Solution

Expose a custom Apex REST resource (@RestResource '/caqh/attestation/v1') to receive the CAQH webhook POST. Apex parses the JSON payload, performs a SOQL lookup on Provider__c by NPI__c, and uses a CredentialingCaseService Apex class to handle create/update logic with duplicate detection. New Provider__c auto-creation uses Account record type 'Provider_Person_Account'. Integration errors persisted to Integration_Error_Log__c custom object. Unit tests cover all three matching scenarios with mock HTTP responses. Connected App with IP whitelisting restricts inbound calls to CAQH middleware IP range. Platform Event published post-creation to trigger downstream NPI validation flow.
