## Description

As a Credentialing Specialist, I need the system to validate a provider's NPI against the CMS NPPES registry in real time when a Provider__c record is saved so that only legitimate, active NPIs are accepted and provider demographic data is automatically populated, reducing manual data entry and credentialing errors.

Extends INS-0159 — NPI validation must complete successfully before a Credentialing_Case__c can be auto-created.

## Acceptance Criteria

Given a user enters a 10-digit value in Provider__c.NPI__c and saves the record,
When the record save triggers the NPPES callout,
Then the system calls the NPPES NPI Registry REST API (https://npiregistry.cms.hhs.gov/api/) with the supplied NPI.

Given the NPPES API returns a valid, active provider record,
When the callout response is received,
Then the following fields are populated on Provider__c:
  - Provider__c.NPI_Status__c = 'Valid'
  - Provider__c.NPI_Validated_Date__c = TODAY()
  - Provider__c.Provider_Type__c mapped from NPPES enumeration_type ('NPI-1' → 'Individual', 'NPI-2' → 'Organization')
  - Provider__c.Primary_Taxonomy_Code__c from NPPES taxonomy[0].code
  - Provider__c.FirstName, Provider__c.LastName auto-populated if blank

Given the NPPES API returns no results or an inactive NPI,
When the callout response is received,
Then Provider__c.NPI_Status__c = 'Invalid' and a screen error is surfaced: 'NPI not found or inactive in NPPES registry. Please verify and re-enter.'

Given the NPPES API is unavailable (timeout or HTTP 5xx),
When the callout fails,
Then Provider__c.NPI_Status__c = 'Validation Pending' and a Platform Event (NPI_Validation_Retry__e) is published for async retry.

Given Provider__c.NPI__c is not exactly 10 numeric digits,
When the user attempts to save,
Then a Validation Rule blocks save with message: 'NPI must be exactly 10 numeric digits (e.g., 1234567890).'

-Given NPI-2 (organizational) NPIs should be stored on Facility__c instead of Provider__c- (deprecated: org NPIs now supported on both objects per updated data model)

Given a valid NPI callout succeeds,
When Provider__c.Provider_Type__c = 'Organization',
Then a related Facility__c lookup (Facility__c.NPI__c) is checked and linked if a matching record exists.

## Solution

Apex Trigger (ProviderNPITrigger) on Provider__c fires after insert/update when NPI__c is changed. Trigger enqueues a Queueable Apex class (NPPESValidationQueueable) to perform the REST callout asynchronously to avoid mixed-DML issues. The Queueable class uses a Named Credential ('NPPES_Registry') for endpoint management. Response JSON is deserialized into a wrapper class. Field mapping logic updates Provider__c fields via a separate DML operation. On timeout/failure, a Platform Event (NPI_Validation_Retry__e) is published; a subscriber Flow retries up to 3 times with exponential backoff tracked in a custom object NPI_Retry_Log__c. A Validation Rule enforces 10-digit numeric format client-side before trigger fires.
