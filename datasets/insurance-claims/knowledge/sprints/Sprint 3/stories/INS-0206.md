## Description

As a Credentialing Specialist, I need a Credentialing_Case__c record to be automatically created whenever a new Provider__c (Person Account) is submitted for network participation, so that no provider application falls through the cracks and every credentialing workflow is traceable from day one.

## Acceptance Criteria

Given a new Provider__c record is created with Provider_Status__c = 'Pending Credentialing'
When the record is saved
Then a Credentialing_Case__c is auto-created with:
  - Credentialing_Case__c.Status__c = 'Initiated'
  - Credentialing_Case__c.Provider__c = [linked Provider__c Id]
  - Credentialing_Case__c.LOB__c inherited from Provider__c.Line_of_Business__c
  - Credentialing_Case__c.Assigned_Specialist__c populated via round-robin queue assignment
  - Credentialing_Case__c.Due_Date__c = TODAY() + 45 days

Given a Provider__c record is updated (not created)
When Provider_Status__c changes to 'Re-Credentialing Required'
Then a new Credentialing_Case__c is created with Type__c = 'Re-Credentialing' and prior case linked via Parent_Case__c

Given a Credentialing_Case__c is successfully created
When the record is saved
Then an email notification is dispatched to the assigned Credentialing Specialist via the 'Credentialing Case Assigned' email alert template

Given a Provider__c already has an open Credentialing_Case__c with Status__c != 'Closed'
When a duplicate submission is attempted
Then no new Credentialing_Case__c is created and a warning toast is displayed: 'An active credentialing case already exists for this provider.'

-Given Provider__c.NPI__c is null, block case creation and surface a validation error- (superseded by INS-0208 NPI validation story)

## Solution

Implement a Record-Triggered Flow on Provider__c object firing on Create and Update. Use a Decision element to check for existing open Credentialing_Case__c via a Get Records element (filter: Provider__c = {!$Record.Id} AND Status__c != 'Closed'). On the create path, use a Create Records element to instantiate Credentialing_Case__c with all required field mappings. For round-robin assignment, invoke an Apex action (CredentialingQueueAssignmentService.cls) that queries the active queue members and rotates based on a Custom Metadata Type (Credentialing_Assignment_Config__mdt) counter. Deploy a separate Flow path for the Re-Credentialing scenario using a scheduled-path or update-triggered branch. Email alert leverages an existing Workflow Email Alert migrated to Flow Send Email action.
