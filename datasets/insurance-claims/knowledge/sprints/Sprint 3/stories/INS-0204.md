## Description

**Bug Report — relates to INS-0200**

**Summary:** The npiValidationPanel LWC component throws an unhandled JavaScript TypeError and the entire Provider__c record page becomes unresponsive when a user enters non-numeric characters (e.g., letters, hyphens, spaces) in the NPI__c field before clicking 'Validate NPI'.

**Steps to Reproduce:**
1. Navigate to any Provider__c (Person Account) record page in the Credentialing Specialist app.
2. Locate the 'NPI Validation' panel (npiValidationPanel LWC).
3. Enter a non-numeric value such as 'ABC-123-456X' or '12 345 678 90' in the NPI input field.
4. Click the 'Validate NPI' button.
5. Observe the browser console and the page behavior.

**Expected Result:** A user-friendly inline validation message displays beneath the NPI input field: 'NPI must be a 10-digit numeric value.' The 'Validate NPI' button remains disabled until a valid 10-digit numeric string is entered. No API callout is made.

**Actual Result:** A JavaScript TypeError ('Cannot read properties of undefined — reading length') is thrown in the browser console. The LWC component enters an error state and renders a blank panel. The rest of the Provider__c record page becomes unresponsive until a full page refresh. In two observed instances, a partial NPPES API callout was still fired with a malformed NPI parameter, returning an HTTP 400 error that was not gracefully handled, resulting in an unresolved spinner state.

**Environment:** Full Sandbox (CS-47), API version 59.0, Chrome 124, Lightning Experience.

**Severity:** High — Credentialing Specialists are unable to use the NPI validation feature without risk of losing unsaved record edits on page crash.

## Acceptance Criteria

As per Description.

Additional verification:
- Confirm no NPPES API callout is fired when client-side validation fails
- Confirm spinner state is never left unresolved regardless of input or API response
- Confirm fix is regression-tested against valid 10-digit NPI inputs from INS-0200 ACs

## Solution

Fix the npiValidationPanel LWC JavaScript controller to add client-side input validation before any Apex callout is invoked. Add a regex validation function: /^[0-9]{10}$/.test(npiValue) — if false, set an inline error message using a reactive errorMessage property bound to a template conditional and do not proceed to the Apex call. Wrap the entire handleValidateNPI async function in a try-catch-finally block where the finally block always sets isLoading = false to prevent unresolved spinner states. In the Apex controller NppesIntegrationService.validateNPI(), add a server-side guard: if (!npi.matches('[0-9]{10}')) throw new AuraHandledException('Invalid NPI format.') as a defense-in-depth measure. Add Jest unit tests covering: (1) non-numeric input blocked client-side, (2) spinner resolves on callout exception, (3) valid input proceeds to callout mock.
