## Description

As a Credentialing Specialist, I need the system to validate a provider's NPI against the CMS NPPES registry in real time when a Provider__c record is saved so that only legitimate, active NPIs are accepted into the network and downstream claims adjudication errors are avoided.

## Acceptance Criteria

Given a Provider__c record is saved with a 10-digit NPI__c value,
When the NPPES API call is made (GET https://npiregistry.cms.hhs.gov/api/?number={NPI}),
Then:
  - If the API returns a valid active provider match, Provider__c.NPI_Validation_Status__c is set to 'Validated' and Provider__c.NPI_Validated_Date__c = TODAY()
  - If the API returns no results or an inactive record, Provider__c.NPI_Validation_Status__c is set to 'Invalid' and a page-level error is shown: 'NPI {value} could not be validated against NPPES. Please verify and resubmit.'

Given a validated NPPES response,
When the API returns provider taxonomy codes,
Then Provider__c.Primary_Taxonomy_Code__c is auto-populated with the primary taxonomy from the response and Provider__c.Provider_Type__c is mapped accordingly.

Given the NPPES API is unreachable (timeout > 5 seconds or HTTP 5xx),
When the callout fails,
Then Provider__c.NPI_Validation_Status__c = 'Pending Validation', a Platform Event (NPI_Validation_Retry__e) is published for async retry, and the record save is allowed to proceed with a warning banner.

Given a Provider__c NPI__c field is not exactly 10 numeric digits,
When the user attempts to save,
Then a field-level validation rule fires before any API callout: 'NPI must be exactly 10 numeric digits.'

-Given NPI validation should block record save on API timeout- (deprecated — stakeholder decision Sprint 2 retro: save should be allowed with 'Pending Validation' status to avoid workflow stoppage)

Given a Provider__c record with NPI_Validation_Status__c = 'Validated' is edited but NPI__c is unchanged,
When the record is saved,
Then no new NPPES callout is made.

Given NPI__c is changed on an existing validated Provider__c,
When the record is saved,
Then re-validation is triggered and NPI_Validated_Date__c is updated.

## Solution

Implement an Apex Trigger (ProviderTrigger.cls) with a TriggerHandler pattern (ProviderTriggerHandler.cls) calling an async Queueable Apex class (NPPESValidationQueueable.cls) to perform the NPPES REST callout and avoid mixed DML issues. Named Credential 'NPPES_Registry' stores the endpoint. Response parsing handled in NPPESResponseParser.cls. Taxonomy-to-Provider_Type__c mapping stored in a Custom Metadata Type (NPI_Taxonomy_Mapping__mdt). Retry logic via Platform Event (NPI_Validation_Retry__e) consumed by a separate Apex trigger. A Screen Flow on the Provider__c record page surfaces inline validation status via an LWC (npiValidationBadge) using a @wire adapter. All callouts mocked in test class using HttpCalloutMock.
