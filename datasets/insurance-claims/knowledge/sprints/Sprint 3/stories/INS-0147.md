## Description

As a Credentialing Specialist, I need a guided OmniScript intake form for initiating provider credentialing that dynamically adapts its steps based on whether the provider is an individual practitioner (NPI-1) or an organizational/facility provider (NPI-2), so that I capture the correct data elements for each provider type and reduce rework caused by incomplete submissions. This story supersedes the separate NPI-1/NPI-2 flow paths referenced in INS-0143.

## Acceptance Criteria

Given a Credentialing Specialist launches the 'New Credentialing Intake' OmniScript from the Provider__c record or App Launcher,
When Step 1 renders,
Then the specialist must select Provider Type: 'Individual Practitioner (NPI-1)' or 'Organizational Provider (NPI-2)' before proceeding.

Given the specialist selects 'Individual Practitioner (NPI-1)',
When Step 2 renders,
Then the OmniScript displays fields: First Name, Last Name, NPI, TIN, Primary Specialty (NUCC taxonomy lookup), DEA Number, State License Number, State License Expiry Date, CAQH Provider ID, and Board Certification fields.

Given the specialist selects 'Organizational Provider (NPI-2)',
When Step 2 renders,
Then the OmniScript displays fields: Organization Name, NPI, TIN, Facility Type, Group NPI, Primary Practice Address, CLIA Number (if applicable), and Credentialing Contact Name/Email.

Given the NPI field is populated and the specialist clicks 'Validate NPI',
When the OmniScript calls the NPPES Integration Procedure,
Then the NPI type returned must match the selected Provider Type; a mismatch displays an inline warning 'NPI type does not match selected provider type — please verify.'

Given all required fields are complete and NPI is validated,
When the specialist submits the OmniScript on the final Review & Submit step,
Then a Provider__c record (or match to existing) is created/updated, a Credentialing_Case__c is created with Credentialing_Case__c.Provider_Type__c set appropriately, and the specialist is navigated to the new Credentialing_Case__c record page.

Given the OmniScript session is abandoned mid-way,
When the specialist returns within 24 hours,
Then OmniScript resume functionality restores the partially completed data via OmniScript Save State.

-Given a separate OmniScript exists for NPI-2 Facility credentialing- (deprecated: unified OmniScript per this story replaces standalone facility intake).

## Solution

Build a single OmniStudio OmniScript ('Provider_Credentialing_Intake_OS') with conditional step visibility driven by the Provider Type selection in Step 1. Integration Procedure ('NPPES_NPI_Validate_IP') wraps the NPPES API callout Apex action established in INS-0143. DataRaptor Transform maps OmniScript output to Provider__c and Credentialing_Case__c field mappings. OmniScript Save State enabled using OmniScript.vSaveState with a 24-hour TTL. NUCC taxonomy code lookup implemented as a DataRaptor Extract against a custom Taxonomy_Code__c object pre-loaded with CMS taxonomy data. OmniScript deployed as an LWC component embedded on the Provider__c Lightning Record Page and accessible via a custom App Launcher tile for standalone launch.
