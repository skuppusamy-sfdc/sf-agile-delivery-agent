## Description

As a Credentialing Specialist, I need the system to automatically validate a provider's NPI against the NPPES registry when a new Credentialing_Case__c is submitted, so that we eliminate manual lookup errors and ensure only legitimate providers advance through the credentialing workflow.

## Acceptance Criteria

Given a Credentialing_Case__c record is created with Provider__c populated and Credentialing_Case__c.NPI__c is non-null,
When the record is saved and transitions to Status__c = 'Intake Submitted',
Then an outbound callout is made to the NPPES NPI Registry API using the value in Credentialing_Case__c.NPI__c.

Given the NPPES API returns a valid, active NPI record,
When the response is received,
Then Credentialing_Case__c.NPI_Validation_Status__c is set to 'Verified', Credentialing_Case__c.NPI_Verified_Date__c is stamped with today's date, and Provider__c.NPI_Status__c on the linked Person Account is updated to 'Active'.

Given the NPPES API returns no matching record or returns a deactivated NPI,
When the response is received,
Then Credentialing_Case__c.NPI_Validation_Status__c is set to 'Failed', a Task is auto-created and assigned to the owning Credentialing Specialist with Subject = 'NPI Validation Failed – Manual Review Required', and the Credentialing_Case__c.Status__c is set to 'Pending – NPI Review'.

Given the NPPES API is unreachable or returns an HTTP 5xx error,
When the callout times out or fails,
Then Credentialing_Case__c.NPI_Validation_Status__c is set to 'Indeterminate', a Platform Event (NPI_Validation_Event__e) is published for retry processing, and the Credentialing Specialist is notified via in-app notification.

-Given NPI validation is triggered on every save of Credentialing_Case__c regardless of status change- (superseded — validation only fires on initial Intake Submitted transition to avoid redundant callouts)

Given a provider's NPI has already been verified within the last 90 days (Credentialing_Case__c.NPI_Verified_Date__c within range),
When a new Credentialing_Case__c is submitted for the same Provider__c,
Then the system skips the NPPES callout, copies the prior verification result, and sets Credentialing_Case__c.NPI_Validation_Status__c = 'Verified (Cached)' with a tooltip note indicating the cached date.

## Solution

Implement a Record-Triggered Flow on Credentialing_Case__c that fires After Save on create/update when Status__c changes to 'Intake Submitted'. The flow invokes an Apex Action (NPPESValidationService.cls) which performs a named-credential-secured HTTP callout to the NPPES REST API (api.cms.gov). Response parsing populates NPI_Validation_Status__c and related fields via a follow-up DML update in a separate Queueable Apex class to respect callout-then-DML governor limits. A Platform Event (NPI_Validation_Event__e) is used for retry logic via a Trigger on the event bus. Provider__c (Person Account) NPI_Status__c is updated via a separate invocable method to maintain separation of concerns. Cache check logic is handled in the flow via a Get Records node querying recent Credentialing_Case__c records for the same Provider__c.
