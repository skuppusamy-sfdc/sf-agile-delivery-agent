## Description

As a Credentialing Specialist, I need a Credentialing_Case__c record to be automatically created whenever a Provider__c (Person Account) is submitted for network participation so that credentialing workflows are initiated without manual intervention and no applications fall through the cracks.

## Acceptance Criteria

Given a Provider__c record has Network_Participation_Status__c set to 'Pending Credentialing' and NPI__c is populated,
When the record is saved or updated to that status,
Then a new Credentialing_Case__c is created with:
  - Credentialing_Case__c.Status__c = 'Open'
  - Credentialing_Case__c.Provider__c = the triggering Provider__c Id
  - Credentialing_Case__c.Assigned_To__c = round-robin assigned Credentialing Specialist queue
  - Credentialing_Case__c.LOB__c inherited from Provider__c.Primary_LOB__c
  - Credentialing_Case__c.Application_Received_Date__c = TODAY()

Given a Credentialing_Case__c already exists in a non-terminal status for the same Provider__c,
When the provider is re-submitted,
Then no duplicate Credentialing_Case__c is created and a warning toast is displayed to the user.

Given the Provider__c.NPI__c field is blank at time of submission,
When the status change is attempted,
Then the save is blocked with validation message: 'NPI is required before submitting for credentialing.'

-Given Provider__c.TIN__c is also required at submission- (deprecated: TIN validation moved to INS-0162)

Given a Credentialing_Case__c is successfully created,
When the case is created,
Then an email notification is sent to the assigned Credentialing Specialist using the 'New Credentialing Application' email template.

## Solution

Implement a Record-Triggered Flow on Provider__c (Person Account) firing on Create and Update when Network_Participation_Status__c changes to 'Pending Credentialing'. Flow uses a Get Records element to check for existing open Credentialing_Case__c records. If none found, a Create Records element provisions the Credentialing_Case__c. Round-robin assignment handled via a custom Apex Action invoked from the Flow referencing the Credentialing_Queue custom setting. Duplicate guard uses a Decision element checking for existing open cases. Validation Rule on Provider__c enforces NPI__c presence. Email alert action fires post-case creation using an OrgWide Email Address.
