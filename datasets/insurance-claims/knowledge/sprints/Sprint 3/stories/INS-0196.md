## Description

**Bug Report**

**Summary:** The NPI validation badge on the Provider 360 FlexCard (INS-0193) and the Provider__c record page incorrectly displays 'Validated' (green) for providers whose NPPES registry record has a deactivation date in the past. This causes Credentialing Specialists to incorrectly assume an NPI is active when it has been deactivated by CMS, creating compliance risk.

**Steps to Reproduce:**
1. Navigate to a Provider__c record where NPI_Validation_Status__c = 'Validated' and NPI_Validated_Date__c is older than 180 days.
2. Manually verify the NPI against NPPES (https://npiregistry.cms.hhs.gov) — confirm the NPI shows 'Deactivation Date' in the past.
3. Observe the Provider 360 FlexCard and Provider__c record page header.
4. Badge displays green 'Validated' despite the NPI being deactivated in NPPES.

**Expected Behavior:** The NPI validation badge should reflect the current NPPES status. If the NPPES registry returns a deactivation date <= TODAY(), the badge should display 'Deactivated' (red) and Provider__c.NPI_Validation_Status__c should be updated to 'Deactivated'.

**Actual Behavior:** Badge shows 'Validated' (green) because NPPESValidationQueueable.cls does not parse or store the 'deactivation_date' field from the NPPES API JSON response. The field NPI_Deactivation_Date__c does not exist on Provider__c.

**Root Cause (Preliminary):** NPPESResponseParser.cls (built in INS-0192) maps only 'enumeration_type', 'number', 'basic.status', and 'taxonomies' fields from the NPPES response. The 'basic.deactivation_date' JSON node is not mapped, so deactivated providers pass validation as active.

**Environment:** Full Sandbox (CS47)
**Severity:** High — compliance risk; deactivated NPI providers must not be credentialed.

## Acceptance Criteria

As per Description.

Additional fix acceptance:
Given NPPESResponseParser.cls is updated to parse 'basic.deactivation_date' from the NPPES API response,
When deactivation_date is present and <= TODAY(),
Then Provider__c.NPI_Validation_Status__c is set to 'Deactivated', NPI_Deactivation_Date__c is populated, and the FlexCard badge renders red with label 'Deactivated'.

Given a nightly scheduled Apex job re-validates all Provider__c records with NPI_Validation_Status__c = 'Validated' and NPI_Validated_Date__c > 180 days ago,
When the job runs,
Then deactivated NPIs are flagged and Provider Relations Specialists receive a daily digest email listing all newly deactivated providers.

## Solution

Fix NPPESResponseParser.cls to include mapping of 'basic.deactivation_date' JSON field. Add NPI_Deactivation_Date__c (Date) custom field to Provider__c. Update NPPESValidationQueueable.cls logic: if deactivation_date != null AND deactivation_date <= System.today(), set NPI_Validation_Status__c = 'Deactivated'. Update FlexCard (Provider360_FC) DataRaptor and FlexCard State conditions to handle 'Deactivated' status with red color class. Implement a Scheduled Apex job (NPIRevalidationScheduler.cls) running nightly querying stale validated providers and republishing NPI_Validation_Retry__e events. Add digest email via Apex using custom email template 'NPI_Deactivation_Digest_ET'. Deploy fix to Full Sandbox first, regression test INS-0192 and INS-0193 scenarios.
