## Description

**Bug Report — Hypercare P1**

**Summary:** When a Claims Examiner triggers EOB PDF generation from an Explanation_of_Benefits__c record linked to a Claim__c that has more than 15 associated Claim_Line__c child records, the Conga CLM merge job completes with a status of 'Success' in the Conga activity log but no PDF is attached to the EOB record and no error is surfaced to the user. The examiner has no indication the document failed to generate.

**Environment:** Production (post-Sprint 19 go-live)

**Steps to Reproduce:**
1. Navigate to any Claim__c record where the count of related Claim_Line__c records > 15 (e.g., Claim__c record Id: 'a0B9X000003TzKL' — sanitised test clone available in Full Sandbox).
2. Open the related Explanation_of_Benefits__c record.
3. Click the 'Generate EOB' quick action button (LWC).
4. Observe the success toast notification: 'EOB document generated successfully'.
5. Refresh the Files related list on the EOB record — no PDF is attached.
6. Check Conga Activity Log — job status shows 'Completed' with 0 bytes output.

**Expected Behaviour:** A fully populated EOB PDF is generated and attached to the Explanation_of_Benefits__c record as a Salesforce File (ContentDocument), with all Claim_Line__c rows rendered in the line-item table of the document template.

**Actual Behaviour:** No PDF is attached. The Conga merge silently produces a 0-byte output when the Claim_Line__c repeating table in the Conga template exceeds the configured row limit. The LWC success toast fires based on the HTTP 200 response from the Conga API, not on document content validation.

**Root Cause (Preliminary):** Conga template 'EOB_Standard_Template_v3' has a hardcoded repeating table row limit of 15 in the DOCX template configuration. Rows 16+ are silently truncated, causing Conga to output an empty document body. Additionally, the LWC callout does not validate ContentDocumentLink creation after the Conga job completes.

**Impact:** High — affects approximately 12% of adjudicated claims in production based on Claims Operations Manager report. ERA files to payers are being generated without corresponding EOB PDFs, creating COB reconciliation failures downstream.

**Workaround:** Claims Examiners are manually splitting claims with > 15 lines into supplemental EOB records — unsustainable at volume.

## Solution

Two-part fix:

1. **Conga Template Fix:** Update 'EOB_Standard_Template_v3' DOCX template to remove the hardcoded 15-row repeating table cap. Replace static table with a Conga dynamic table construct ({{TableStart:ClaimLines}} / {{TableEnd:ClaimLines}}) that iterates over all related Claim_Line__c records without an upper bound. Re-test template rendering with 5, 15, 20, and 50 Claim_Line__c records in Full Sandbox. Upload corrected template version as 'EOB_Standard_Template_v3.1' and update the Conga Solution record to reference the new template.

2. **LWC Callout Validation Fix:** Modify the 'GenerateEOBAction' LWC to invoke a post-generation Apex validation method (EobDocumentValidator.verifyContentDocumentLink) after the Conga API call returns HTTP 200. The Apex method queries ContentDocumentLink WHERE LinkedEntityId = :eobRecordId AND ContentDocument.FileExtension = 'pdf' AND ContentDocument.CreatedDate = LAST_N_MINUTES:2. If no record is found, the LWC surfaces an error toast ('EOB generation failed — no document attached. Please contact Claims Support.') and logs a Platform Event (EOB_Generation_Failure__e) with the Claim__c Id and EOB Id for monitoring. Add a named credential timeout override to 30 seconds on the Conga callout to prevent premature HTTP 200 responses on large payloads. Deploy via change set to Production under emergency change advisory board approval given P1 hypercare priority.
