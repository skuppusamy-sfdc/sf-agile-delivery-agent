## Description

As a Provider Network Specialist, I need to validate and stabilize the Provider_Change_Notification__c (PCN) cascade automation that fires when a Network_Termination_Case__c reaches 'Approved' status, ensuring that all downstream objects — including PNA status update, Fee_Schedule__c deactivation, open Claim__c flagging, and broker notification — execute correctly and in the expected sequence during the hypercare observation window.

Several edge cases were identified in UAT (Sprint 18) where the cascade partially completed when a provider had more than 10 active PNA records, causing orphaned Fee_Schedule__c records to remain active post-termination.

## Acceptance Criteria

Given a Network_Termination_Case__c.Status__c transitions to 'Approved',
When the Record-Triggered Flow executes,
Then ALL of the following cascade actions complete within 30 seconds without governor limit exceptions:
  1. All related Provider_Network_Agreement__c records where PNA.Provider__c = terminated provider and PNA.Agreement_Status__c = 'Active' are updated to PNA.Agreement_Status__c = 'Terminated' and PNA.Termination_Date__c = Network_Termination_Case__c.Effective_Termination_Date__c
  2. All related Fee_Schedule__c records where Fee_Schedule__c.Provider__c = terminated provider and Fee_Schedule__c.Status__c = 'Active' are updated to Fee_Schedule__c.Status__c = 'Inactive'
  3. All open Claim__c records where Claim__c.Billing_Provider__c = terminated provider and Claim__c.Status__c NOT IN ('Paid', 'Denied', 'Closed') have Claim__c.Termination_Review_Flag__c = true and Claim__c.Assigned_Queue__c = 'Termination_Review_Queue'
  4. A Provider_Change_Notification__c record is created with PCN.Change_Type__c = 'Network Termination', PCN.Effective_Date__c populated, and PCN.Notification_Status__c = 'Pending Dispatch'

Given the provider has more than 10 active PNA records (edge case from UAT),
When the termination cascade runs,
Then all PNA records are updated without hitting DML governor limits (solution must use bulkified Apex invocable, not inline Flow DML).

Given the cascade completes successfully,
When the Network_Termination_Case__c record is reviewed,
Then Network_Termination_Case__c.Cascade_Status__c = 'Completed' and Network_Termination_Case__c.Cascade_Completion_Timestamp__c is populated.

Given the cascade encounters any DML error,
When the error occurs,
Then Network_Termination_Case__c.Cascade_Status__c = 'Failed', an error log entry is written to DataIntegration_Error__c, and the assigned Provider Network Specialist receives an in-app notification.

## Solution

Refactor the existing Record-Triggered Flow ('Network_Termination_Cascade_Flow') to replace inline DML collection updates for PNA and Fee_Schedule__c with an Apex Invocable Method (TerminationCascadeService.cls) that processes all related records in a single bulkified DML transaction using Database.update() with allOrNone = false and partial success logging. The Flow calls the Invocable Action post-approval and passes the Network_Termination_Case__c Id. Claim__c flagging uses a separate Flow-called Apex method to avoid mixed DML. PCN creation remains in Flow using a Create Records element. Cascade status updates use a final Flow element. Error handling uses a Fault Path in Flow that creates DataIntegration_Error__c and fires a Custom Notification via the Notification Builder. Add Apex test class (TerminationCascadeServiceTest.cls) with test data covering 15+ active PNA records.
