## Description

**Bug Report — Production Incident P1-2024-0892**

**Steps to Reproduce:**
1. Navigate to a Claim__c record where the count of related Claim_Line__c records exceeds 25 (e.g., Claim ID: CLM-009142 in Production).
2. Click the 'Generate EOB' action button on the Claim__c record page.
3. The Conga CLM merge job is triggered via the 'Generate_EOB_OmniScript'.
4. Observe the OmniScript Step 3 spinner — it runs indefinitely (>120 seconds) then silently completes with no error message displayed to the user.
5. Navigate to the related Explanation_of_Benefits__c record — no new EOB__c record is created and no PDF attachment exists on the Claim__c ContentDocumentLink.

**Expected Behavior:** EOB PDF is generated via Conga CLM, attached to Claim__c as a ContentDocument, and EOB__c.Status__c is set to 'Generated' with EOB__c.Generated_Date__c stamped.

**Actual Behavior:** EOB generation silently fails. No EOB__c record is created. No error surfaced to user. Conga API response log shows HTTP 413 'Payload Too Large' when Claim_Line__c SOQL subquery returns >25 rows, exceeding the Conga template merge field limit configured in the 'EOB_Standard_Template_v3' template.

**Environment:** Production. Reproducible in Full Sandbox with >25 Claim_Line__c records.

**Impact:** Affects approximately 340 Claim__c records in Production with high line-item counts (facility claims, inpatient stays). Compliance risk — EOB delivery is required within CMS mandated timelines.

## Acceptance Criteria

As per Description. Additionally:

Given a Claim__c record with more than 25 Claim_Line__c child records,
When 'Generate EOB' is triggered,
Then the OmniScript calls a revised DataRaptor Integration action that paginates Claim_Line__c retrieval in batches of 25 and the Conga merge completes successfully without HTTP 413 error.

Given the Conga API returns any non-200 HTTP response,
When the OmniScript processes the response,
Then a user-facing error toast displays: 'EOB generation failed — please contact Claims Support (Ref: EOB-ERR-[ClaimNumber])' and a Salesforce Case__c is auto-created and assigned to the 'Claims_EOB_Error_Queue'.

Given the fix is deployed,
When regression tested against the 340 impacted Production Claim__c records via a Batch Apex job (EOBRetryBatch),
Then all records produce a valid EOB__c with Status__c = 'Generated' and a ContentDocument PDF attachment within 4 hours of batch execution.

## Solution

Root cause: Conga CLM template 'EOB_Standard_Template_v3' has a single merge field block for Claim_Line__c that issues one SOQL query returning all child records. Conga's REST API enforces a 1MB payload cap causing HTTP 413 on high line-count claims. Fix approach: (1) Refactor the DataRaptor Integration action in 'Generate_EOB_OmniScript' to paginate Claim_Line__c queries using OFFSET/LIMIT 25 and aggregate line data into a JSON array passed to a new Apex invocable method (EOBLineAggregator.aggregateLines()) before the Conga API call. (2) Update Conga template to accept pre-aggregated JSON array. (3) Add HTTP response error handling in OmniScript Step 3 with conditional branching on Conga response code. (4) Create EOBRetryBatch Apex class to reprocess the 340 impacted records. (5) Add Named Credential timeout setting increase from 10s to 30s for Conga endpoint.
