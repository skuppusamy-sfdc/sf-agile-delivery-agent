## Description

As a Provider Network Specialist and future system administrator at Acme Insurance, I need a comprehensive knowledge transfer package — including Salesforce Flow architecture diagrams, field mapping references, and an operational runbook — for the Provider Network Contracting automation suite delivered in Phase 2 so that the internal team can independently support, troubleshoot, and extend the contracting workflows without SI vendor dependency after hypercare ends.

The contracting automation suite includes: (1) the PNA lifecycle Record-Triggered Flows, (2) the Conga CLM contract generation OmniScript, (3) the DocuSign eSignature routing logic on Provider_Network_Agreement__c, and (4) the automated Provider_Change_Notification__c (PCN) creation process triggered on PNA termination.

This story covers documentation and knowledge transfer sessions only — no new development is in scope.

## Acceptance Criteria

Given the completion of this story, When a Provider Network Specialist accesses the Acme Insurance internal Confluence space, Then a 'Provider Network Contracting — Phase 2 Architecture Guide' page is published containing: (a) a data flow diagram showing the end-to-end PNA lifecycle from creation through termination, referencing Provider_Network_Agreement__c.Status__c state transitions ('Draft' → 'Pending Signature' → 'Active' → 'Pending Termination' → 'Terminated'), (b) a field mapping table cross-referencing PNA fields to corresponding Conga CLM template merge fields and DocuSign envelope recipient roles, and (c) a list of all active Flows on Provider_Network_Agreement__c with their trigger criteria, entry conditions, and downstream object updates documented.

Given a PNA Status__c transition to 'Pending Termination', When the runbook section 'PNA Termination Troubleshooting' is consulted, Then it provides step-by-step instructions for: verifying Provider_Change_Notification__c auto-creation (PCN.Trigger_Source__c = 'Automated — PNA Termination'), manually re-triggering the PCN creation Flow if the automation fails, and confirming Network_Termination_Case__c is linked to the originating PNA record via Network_Termination_Case__c.Related_PNA__c.

-Given a GSA record linked to a terminating PNA, When the runbook is consulted, the admin is directed to manually archive the GSA — this step was automated in INS-1681 and the manual instruction is now deprecated-

Given the knowledge transfer session schedule, When sprint review occurs, Then evidence is provided that at least two live KT sessions (recorded to SharePoint) have been conducted with Acme's internal Provider Network team covering: (a) Flow debug walkthrough using Salesforce Flow Debug Logs for a sample PNA record, and (b) Conga CLM template editing basics for updating the 'PNA_Standard_Contract_v2' template.

Given a future administrator needing to add a new LOB to the contracting process, When they follow the runbook section 'Adding a New Line of Business to PNA Contracting', Then the documented steps reference the specific Flow 'PNA_LOB_Routing_Flow' (Record-Triggered, on Provider_Network_Agreement__c), the picklist field PNA.LOB__c, the Conga Solution record 'PNA_Contract_Generation', and the DocuSign Template mapping table — enabling the change without SI assistance.

## Solution

No Salesforce development required. Deliverables are documentation and enablement assets:

1. **Architecture Guide:** Authored in Confluence using the Acme Insurance 'Technical Architecture' page template. Flow diagrams created in Lucidchart (embedded via Confluence macro) using exported Flow metadata from the sandbox org as source of truth. Field mapping tables exported from a DataRaptor JSON definition and formatted as Confluence tables.

2. **Operational Runbook:** Word document converted to Confluence page. Includes screenshots from Full Sandbox of: Flow debug logs, Conga Activity Log, DocuSign Envelope Status on PNA record, and PCN record auto-creation. Runbook sections indexed by failure scenario (Conga timeout, DocuSign bounce, PCN not created, NPI mismatch on NPPES validation).

3. **KT Sessions:** Two 90-minute sessions recorded via MS Teams, uploaded to SharePoint 'Acme Insurance — SI Handoff' folder. Session 1: Flow architecture walkthrough with live debug. Session 2: Conga CLM template editing and DocuSign routing configuration. Attendance sheet signed by Provider Network Specialist, Credentialing Specialist, and Claims Operations Manager as acceptance of knowledge transfer.

4. **Acceptance Sign-off:** KT completion sign-off captured as a DocuSign envelope sent to James Wright (Acme engagement lead) referencing this story ID INS-1705 as the formal hypercare handoff milestone.
