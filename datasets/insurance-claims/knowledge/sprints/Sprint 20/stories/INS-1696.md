## Description

As a Provider Relations Specialist, I need a consolidated Provider 360 FlexCard on the Provider__c (Person Account) record page that surfaces the provider's active Provider_Network_Agreement__c (PNA), associated Fee_Schedule__c effective dates, NPI validation status from NPPES, and CAQH credentialing attestation date, so that during hypercare triage I can quickly validate provider eligibility without navigating across multiple objects.

This addresses a volume of hypercare tickets raised in Sprint 19 where specialists were manually cross-referencing 4–5 tabs to confirm network status before approving claim lines.

## Acceptance Criteria

Given a Provider Relations Specialist opens a Provider__c (Person Account) record,
When the Provider 360 FlexCard renders,
Then the following data is displayed in a single card with expandable sections:
  - Provider__c.NPI__c with a NPPES validation badge (Valid / Invalid / Pending)
  - Provider__c.CAQH_Attestation_Date__c
  - Active PNA: Provider_Network_Agreement__c.Agreement_Status__c = 'Active', displaying PNA Name, PNA.Effective_Date__c, PNA.Termination_Date__c, and PNA.Network_Tier__c
  - Linked Fee_Schedule__c: Fee_Schedule__c.Effective_Date__c, Fee_Schedule__c.Expiration_Date__c, Fee_Schedule__c.Schedule_Type__c

Given the provider has no active PNA record,
When the FlexCard renders,
Then the PNA section displays a warning banner: 'No Active Network Agreement on File — Claims may be subject to out-of-network adjudication.'

Given the NPPES integration callout fails (timeout or HTTP 5xx),
When the FlexCard loads,
Then the NPI badge renders as 'Validation Unavailable' in amber and logs a DataIntegration_Error__c record with Error_Source__c = 'NPPES' and Provider_Id__c populated.

-Given the FlexCard also shows QGenda schedule availability (deprecated — QGenda integration descoped from Phase 2)-

Given a Provider Relations Specialist has the 'Provider_Network_ReadOnly' Permission Set,
When they view the FlexCard,
Then all PNA and Fee Schedule fields are read-only and no edit actions are available.

## Solution

Implement an OmniStudio FlexCard ('Provider_360_Summary') on the Provider__c record page using a DataRaptor Turbo Extract to fetch related PNA and Fee Schedule records in a single SOQL query. NPPES NPI validation is called via an Integration Procedure that wraps an Apex REST callout to the NPPES API endpoint, with a 5-second timeout and fallback handling. CAQH attestation date is sourced from a nightly DataRaptor batch sync stored on Provider__c.CAQH_Attestation_Date__c. FlexCard states (Active, Warning, Error) are driven by conditional visibility rules on card sections. Permission Set 'Provider_Network_ReadOnly' controls FLS. Error logging uses a Record-Triggered Flow on DataIntegration_Error__c creation.
