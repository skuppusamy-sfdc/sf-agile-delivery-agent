## Description

**Bug Report — Production Regression identified during Sprint 20 Hypercare monitoring.**

**Steps to Reproduce:**
1. Navigate to an existing Claim__c record with LOB__c = 'Medical' and at least two Claim_Line__c child records.
2. Ensure the linked Provider_Network_Agreement__c has a Fee_Schedule__c with more than one CPT code range entry (e.g., CPT range 99201–99215 AND a second range 70000–79999).
3. Set Claim_Line__c.CPT_Code__c to a value that falls in the SECOND range (e.g., '70553').
4. Save the Claim_Line__c record and observe the Claim_Line__c.Allowed_Amount__c field.

**Expected Result:** Claim_Line__c.Allowed_Amount__c populates with the correct allowed amount from Fee_Schedule__c matching CPT range 70000–79999 (e.g., $425.00).

**Actual Result:** Claim_Line__c.Allowed_Amount__c = $0.00. The fee schedule lookup Apex method returns null when iterating beyond the first CPT range entry, causing the amount to default to zero.

**Impact:** Affects approximately 340 Claim_Line__c records processed since Sprint 19 deployment. Payment__c disbursements for affected claims are underpaying providers. SIU flag risk if pattern continues.

**Root Cause (preliminary):** The Apex method `FeeScheduleCalculator.getMatchingFeeSchedule()` uses a `break` statement after evaluating the first Fee_Schedule__c range entry, preventing iteration to subsequent ranges. Introduced in INS-1644 refactor.

## Acceptance Criteria

As per Description.

Additional verification: After fix deployment, execute data fix script to recalculate Claim_Line__c.Allowed_Amount__c for all records where Allowed_Amount__c = 0.00 AND Claim_Line__c.Fee_Schedule_Lookup_Attempted__c = TRUE AND CreatedDate >= 2024-11-01. Confirm Payment__c adjustment records are created for underpaid claims per the Payment Correction runbook.

## Solution

Fix the `FeeScheduleCalculator.getMatchingFeeSchedule()` Apex method to remove the erroneous `break` statement and correctly iterate all Fee_Schedule__c range entries associated with the PNA. Replace the loop logic with a SOQL query filtered by Fee_Schedule__c.CPT_Range_Start__c <= :cptCode AND Fee_Schedule__c.CPT_Range_End__c >= :cptCode to leverage indexed fields rather than in-memory range evaluation. Add a bulkified Apex test class with @isTest coverage for multi-range fee schedules at ≥95% coverage. Deploy data remediation anonymous Apex script in sandbox first, then production with Claims Operations Manager sign-off. Post-fix: update the Record-Triggered Flow on Claim_Line__c that invokes the Apex action to pass the full PNA Id rather than relying on a formula field lookup chain that may resolve to null.
