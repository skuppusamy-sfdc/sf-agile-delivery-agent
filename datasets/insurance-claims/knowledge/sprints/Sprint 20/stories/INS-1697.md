## Description

**Bug Report — Hypercare P1**

**Summary:** When a Provider__c has two or more active Fee_Schedule__c records covering overlapping effective date ranges for the same CPT code, the Claim_Line__c adjudication engine applies the lower-value fee schedule rate rather than the most recently effective one, resulting in underpayment to providers and downstream ERA reconciliation discrepancies.

**Steps to Reproduce:**
1. Navigate to a Provider__c record with NPI = 1234567890 (test provider 'Dr. Jane Fictional, MD' in Acme Sandbox).
2. Confirm two Fee_Schedule__c records exist: FSC-0042 (Effective 01/01/2024, Rate = $150.00 for CPT 99213) and FSC-0078 (Effective 07/01/2024, Rate = $175.00 for CPT 99213), both with Fee_Schedule__c.Status__c = 'Active'.
3. Create a new Claim__c with Service_Date__c = 08/15/2024 and add a Claim_Line__c with CPT_Code__c = '99213'.
4. Trigger adjudication via the 'Adjudicate Claim' quick action.
5. Observe Claim_Line__c.Allowed_Amount__c is set to $150.00 instead of the expected $175.00.

**Expected Result:** Claim_Line__c.Allowed_Amount__c = $175.00 (most recently effective Fee_Schedule__c as of Service_Date__c).

**Actual Result:** Claim_Line__c.Allowed_Amount__c = $150.00 (earliest effective Fee_Schedule__c is applied).

**Impact:** Affects approximately 340 Claim_Line__c records adjudicated since go-live (identified via SOQL audit). Provider payment shortfall estimated at $8,500 across impacted claims. ERA files already transmitted require correction.

**Root Cause (Preliminary):** The SOQL query in FeeScheduleLookupService.cls uses ORDER BY Fee_Schedule__c.Effective_Date__c ASC and LIMIT 1, returning the oldest matching record instead of the most recent.

## Acceptance Criteria

As per Description. Fix verified when Claim_Line__c.Allowed_Amount__c correctly reflects the Fee_Schedule__c with the latest Effective_Date__c that is less than or equal to Claim__c.Service_Date__c for the matching CPT_Code__c and Provider__c combination.

## Solution

Fix the SOQL query in FeeScheduleLookupService.cls to use ORDER BY Fee_Schedule__c.Effective_Date__c DESC LIMIT 1, ensuring the most recently effective fee schedule is selected. Add a query filter: Fee_Schedule__c.Effective_Date__c <= :serviceDate to prevent future-dated schedules from being applied. Deploy a post-fix data remediation script (Apex anonymous) to recalculate Claim_Line__c.Allowed_Amount__c for all 340 impacted records and regenerate affected Payment__c and EOB__c records. Add a unit test in FeeScheduleLookupServiceTest.cls covering the overlapping-schedule scenario with >= 90% code coverage. Update the Integration Procedure used by the adjudication OmniScript to re-invoke the corrected service class.
