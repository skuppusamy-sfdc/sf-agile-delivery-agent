## Description

**Bug Report — PNA Effective Date Validation Bypass via Data Loader**

**Environment:** Production (Hypercare) and Full Sandbox

**Steps to Reproduce:**
1. Identify a Provider_Network_Agreement__c record where PNA.Status__c = 'Active'.
2. Using Salesforce Data Loader (or any API-based tool such as Workbench), update PNA.Contract_End_Date__c to a date that is BEFORE PNA.Contract_Effective_Date__c.
3. Observe the result in Salesforce.

**Expected Behavior:** The validation rule 'PNA_EndDate_After_EffectiveDate' (active on the object) should fire and reject the update with the error: 'Contract End Date must be on or after Contract Effective Date.'

**Actual Behavior:** The update succeeds without error. The PNA record is saved with Contract_End_Date__c < Contract_Effective_Date__c, creating an invalid date range. Downstream Fee_Schedule__c lookups that rely on the PNA effective date window return incorrect results, leading to incorrect Claim_Line__c allowed amount calculations.

**Root Cause (Confirmed):** The validation rule 'PNA_EndDate_After_EffectiveDate' has 'Enforce Validation Rules' unchecked in the rule definition — this was intentionally disabled during a data migration sprint (Sprint 14) and was never re-enabled. Additionally, the System Administrator profile has 'Bypass Validation Rules' enabled via a legacy permission.

**Frequency:** 12 PNA records in production currently have invalid date ranges as a result.

## Acceptance Criteria

As per Description.

Given the validation rule 'PNA_EndDate_After_EffectiveDate' is re-enabled,
When any user (including System Administrator and integration users) attempts to save a Provider_Network_Agreement__c record where Contract_End_Date__c < Contract_Effective_Date__c via any channel (UI, API, Data Loader, Flow),
Then the save is rejected with the error message: 'Contract End Date must be on or after Contract Effective Date.'

Given the 12 existing invalid PNA records in production,
When a data correction script is run,
Then each record's Contract_End_Date__c is updated to the correct value confirmed with the Provider Network Specialist, and an audit entry is logged in Claim__c-related records that referenced the invalid PNA during the affected window.

## Solution

1. Re-enable the validation rule 'PNA_EndDate_After_EffectiveDate' in both sandbox and production. 2. Remove the 'Bypass Validation Rules' legacy permission from the System Administrator profile and from the MuleSoft integration user profile — replace with a dedicated integration Permission Set that does NOT include this bypass. 3. Add an equivalent Apex trigger validation in 'PNATriggerHandler.validateEffectiveDates()' as a defense-in-depth measure that fires on before insert and before update, ensuring the rule cannot be bypassed even if the declarative validation is accidentally disabled in future. 4. Run a SOQL query to identify all 12 invalid PNA records; schedule a data correction session with Provider Network Specialist (Rachel Torres) to confirm correct end dates before bulk update via Data Loader with logging. 5. Add a post-deployment validation test in the CI/CD pipeline (Apex test class 'PNAValidationTest') that explicitly asserts the validation fires on API-context DML.
