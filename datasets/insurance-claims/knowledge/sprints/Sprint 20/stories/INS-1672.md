# INS-1672: BUG: Group_Service_Agreement__c Effective Date Validation Incorrectly Blocks Retroactive GSA Amendments for TPA Accounts

## Description

**Bug Report — Identified during Sprint 20 hypercare UAT session with Broker Relations team.**

**Steps to Reproduce:**
1. Log in as a user with the 'Broker_Relations_Coordinator' Permission Set Group.
2. Open an existing Group_Service_Agreement__c record where GSA.Account_Type__c = 'TPA'.
3. Navigate to the 'Amendment' quick action and create a new GSA Amendment record.
4. Set GSA_Amendment__c.Amendment_Effective_Date__c to any date in the past (e.g., 90 days ago) — a legitimate retroactive correction scenario.
5. Click Save.

**Expected Result:** The retroactive amendment saves successfully. Retroactive amendments for TPA accounts are permitted per business rule BR-GSA-014 when GSA.Retroactive_Amendment_Allowed__c = TRUE.

**Actual Result:** Validation error fires: 'Amendment Effective Date cannot be in the past.' The validation rule 'VR_GSA_Amendment_FutureDate' does not check the GSA.Retroactive_Amendment_Allowed__c flag before evaluating, blocking all retroactive saves regardless of the parent GSA configuration.

**Impact:** Broker Relations Coordinators cannot process approximately 12 pending TPA retroactive amendments, causing downstream Group/Employer onboarding delays and incorrect Payment__c proration calculations.

**Root Cause:** Validation Rule 'VR_GSA_Amendment_FutureDate' formula missing the bypass condition: `AND(NOT(GSA__r.Retroactive_Amendment_Allowed__c), ...)`.

## Acceptance Criteria

As per Description.

Post-fix verification: Confirm that saving a GSA_Amendment__c with Amendment_Effective_Date__c = TODAY() - 90 succeeds when parent GSA.Retroactive_Amendment_Allowed__c = TRUE. Confirm the validation rule still fires correctly when GSA.Retroactive_Amendment_Allowed__c = FALSE and the date is in the past.

## Solution

Update Validation Rule 'VR_GSA_Amendment_FutureDate' on GSA_Amendment__c to add the bypass condition. Revised formula: `AND( NOT(GSA__r.Retroactive_Amendment_Allowed__c), DATEVALUE(Amendment_Effective_Date__c) < TODAY() )`. Deploy via Change Set 'CS_Sprint20_HypercareFixes' to sandbox for regression testing, then production. Add two additional Apex test methods to 'GSAAmendmentTriggerTest' class covering retroactive-allowed and retroactive-blocked scenarios. Update the field-level description on GSA.Retroactive_Amendment_Allowed__c to reference validation rule 'VR_GSA_Amendment_FutureDate' for future maintainer awareness.
