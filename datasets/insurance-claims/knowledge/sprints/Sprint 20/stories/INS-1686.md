## Description

**Bug Report — Production Incident P1-2024-0901**

**Steps to Reproduce:**
1. In Workday, update a Provider's Tax Identification Number (TIN) for an existing provider record that is mapped to a Provider__c (Person Account) in Salesforce via the Workday-Salesforce integration middleware (MuleSoft).
2. The MuleSoft flow pushes the TIN update to Salesforce via a REST API PATCH call, updating Provider__c.TIN__c field.
3. Observe the Provider__c record in Salesforce — TIN__c is correctly updated.
4. Navigate to the related Provider_Change_Notification__c (PCN) related list on the Provider__c record.
5. No new PCN record has been created to document the TIN change.

**Expected Behavior:** Whenever Provider__c.TIN__c is modified — regardless of the update source (UI or integration) — a Provider_Change_Notification__c record should be auto-created with PCN__c.Change_Type__c = 'TIN Update', PCN__c.Previous_Value__c = old TIN, PCN__c.New_Value__c = new TIN, PCN__c.Status__c = 'Pending Review', and PCN__c.Provider__c lookup populated.

**Actual Behavior:** PCN is NOT created when TIN update originates from the Workday MuleSoft integration. Manual TIN updates via Salesforce UI correctly trigger PCN creation (via the existing Record-Triggered Flow 'Provider_TIN_Change_Flow'). Root cause suspected: MuleSoft REST API call uses a dedicated integration user ('mulesoft_integration@acmeins.com') which is excluded from the Record-Triggered Flow entry criteria via a filter on LastModifiedById.

**Environment:** Production and Full Sandbox. First observed Sprint 19 Day 3 post go-live.

**Impact:** TIN changes from Workday are not generating PCNs, creating a compliance gap — Provider Relations Specialists are unaware of TIN changes that may affect 1099 reporting and claim payment routing to correct TIN.

## Acceptance Criteria

As per Description.

Given Provider__c.TIN__c is updated by any user or integration (including the MuleSoft integration user),
When the Record-Triggered Flow 'Provider_TIN_Change_Flow' evaluates the record,
Then a Provider_Change_Notification__c is created regardless of the LastModifiedById value.

Given the PCN is created following a Workday-sourced TIN update,
When the Provider Relations Specialist opens the PCN record,
Then PCN__c.Source__c = 'Workday Integration' and PCN__c.Integration_Reference_ID__c is populated with the MuleSoft correlation ID passed in the API payload custom header.

## Solution

Remove the incorrect 'LastModifiedById != mulesoft_integration@acmeins.com' filter from the entry criteria of Record-Triggered Flow 'Provider_TIN_Change_Flow'. This filter was added during Sprint 17 to prevent loop scenarios but inadvertently blocked integration-sourced updates. Introduce loop prevention via a Process_Guard__c custom field (Provider__c.TIN_Flow_Guard__c checkbox) toggled by the flow itself rather than filtering by user. Coordinate with MuleSoft team to pass correlation ID in a custom HTTP header (X-Mulesoft-Correlation-Id); update the Salesforce REST API receiver to map this header to PCN__c.Integration_Reference_ID__c via a before-insert Apex Trigger (ProviderChangeTrigger). Deploy flow change to sandbox for regression testing against 5 Workday integration test scenarios before Production push.
