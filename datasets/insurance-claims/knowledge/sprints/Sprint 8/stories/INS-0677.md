## Description

As a Claims Operations Manager, I need the system to automatically validate and populate the allowed amount on each Claim_Line__c record by looking up the applicable Fee_Schedule__c rate for the submitted CPT code and provider contract so that adjudication is consistent, reduces manual override rates, and ensures payments align with contracted rates in the Provider_Network_Agreement__c.

This story covers the core lookup and validation logic. ERA generation and EOB population will be addressed in INS-0679.

## Acceptance Criteria

**AC1 — CPT Code Fee Schedule Lookup on Claim Line Save**
Given a Claim_Line__c record is created or updated with CPT_Code__c populated and the parent Claim__c.Provider__c has an Active Provider_Network_Agreement__c,
When the record is saved,
Then the system queries Fee_Schedule__c WHERE Fee_Schedule__c.CPT_Code__c = Claim_Line__c.CPT_Code__c AND Fee_Schedule__c.PNA__c = matching active PNA AND Fee_Schedule__c.Effective_Date__c <= Claim__c.Date_of_Service__c AND Fee_Schedule__c.End_Date__c >= Claim__c.Date_of_Service__c,
And populates Claim_Line__c.Allowed_Amount__c with Fee_Schedule__c.Allowed_Amount__c.

**AC2 — Billed Amount Exceeds Allowed Amount Warning**
Given Claim_Line__c.Billed_Amount__c > Claim_Line__c.Allowed_Amount__c after fee schedule lookup,
When the record is saved,
Then Claim_Line__c.Variance_Flag__c is set to true and Claim_Line__c.Variance_Amount__c = Billed_Amount__c - Allowed_Amount__c.

**AC3 — No Matching Fee Schedule Found**
Given no Fee_Schedule__c record matches the CPT code and active PNA for the date of service,
When the Claim_Line__c is saved,
Then Claim_Line__c.Adjudication_Status__c is set to 'Pended — No Fee Schedule' and a Case is auto-created with Subject = 'Missing Fee Schedule: CPT [code]' assigned to the Provider Network queue.

**AC4 — Out-of-Network Provider Handling**
Given the Claim__c.Provider__c has no Active Provider_Network_Agreement__c,
When Claim_Line__c is saved,
Then Claim_Line__c.Network_Status__c = 'Out of Network' and Claim_Line__c.Allowed_Amount__c is calculated using the Fee_Schedule__c linked to the default OON GSA (Group_Service_Agreement__c.Type__c = 'OON Default').

**AC5 — ICD-10 Diagnosis Code Compatibility Check**
Given Claim_Line__c.ICD10_Code__c is populated,
When fee schedule lookup executes,
Then the system validates that the ICD10_Code__c is present in the Fee_Schedule__c.Covered_Diagnoses__c (multi-select picklist) or that Fee_Schedule__c.Diagnosis_Restriction__c = false; if incompatible, set Claim_Line__c.Adjudication_Status__c = 'Pended — Diagnosis Mismatch'.

-**AC6 — Populate Claim_Line__c.UCR_Amount__c from external UCR database API** (deferred to Phase 2 — UCR integration not in scope for Phase 1)-

**AC6 — Bulk Processing**
Given a batch of up to 200 Claim_Line__c records are inserted simultaneously (e.g., via EDI 837 inbound batch),
When the fee schedule lookup logic executes,
Then all 200 records are processed within governor limits without hitting SOQL query limits (logic must be bulkified using Maps keyed on CPT_Code__c + PNA Id).

## Solution

1. Create Apex class 'FeeScheduleLookupService' with a static method 'resolveAllowedAmounts(List<Claim_Line__c> lines)' called from an Apex Trigger 'ClaimLineTrigger' (after insert, after update on CPT_Code__c or Claim__c changes).
2. Bulkify by collecting all unique CPT codes and PNA Ids from the trigger set, then executing a single SOQL query against Fee_Schedule__c using IN clauses.
3. Build a Map<String, Fee_Schedule__c> keyed on CPT_Code__c + ':' + PNA_Id__c for O(1) lookup per Claim_Line__c.
4. Implement OON fallback logic using a separate query for GSA OON Default records.
5. Auto-Case creation for missing fee schedules via a separate invocable Apex method called from a Record-Triggered Flow to keep DML separation clean.
6. Write Apex test class 'FeeScheduleLookupService_Test' with 90%+ coverage including bulk (200 record) test scenarios, OON path, and pended diagnosis mismatch scenarios.
