## Description

**Bug Report — UAT Sprint 8**

During UAT testing of the Group/Employer Onboarding OmniScript (OS-GroupOnboarding-GSAExecution), the DocuSign envelope for Group_Service_Agreement__c (GSA) countersignature is not routing to the second signer (Acme Insurance authorized signatory) after the employer/broker completes their signature. The envelope status in DocuSign remains 'Waiting for Others' indefinitely, and GSA.DocuSign_Status__c on the Salesforce record is never updated beyond 'Sent'.

**Steps to Reproduce:**
1. Launch the Group Onboarding OmniScript as a Broker Relations Coordinator.
2. Complete all onboarding steps through to the GSA Execution step.
3. Trigger the DocuSign Send action — envelope is dispatched and employer receives signature request.
4. Complete the employer signature in DocuSign.
5. Observe DocuSign envelope status — shows 'Waiting for Others' (second signer: Acme countersignatory).
6. Check the second signer's DocuSign inbox — no envelope received.
7. Check Salesforce GSA record — GSA.DocuSign_Status__c = 'Sent' (not updated to 'Partially Signed').

**Expected Behavior:** After the first signer (employer/broker) completes signing, DocuSign should automatically route the envelope to the configured second signer (Acme Insurance countersignatory role) and update GSA.DocuSign_Status__c = 'Partially Signed'. Upon full completion, GSA.DocuSign_Status__c should update to 'Executed' and GSA.Status__c should update to 'Active'.

**Root Cause Hypothesis:** The DocuSign template (TMPL-GSA-MultiSigner) may have the second recipient's routing order incorrectly configured as parallel rather than sequential, or the DocuSign Integration User's permission profile in the UAT sandbox is missing the 'Send on Behalf of Others' permission, preventing envelope routing continuation.

**Environment:** Acme Insurance UAT Sandbox. DocuSign for Salesforce package v23.2.

## Solution

Audit the DocuSign template (TMPL-GSA-MultiSigner) in the DocuSign admin console to verify recipient routing order: Signer 1 (Employer/Broker) = Routing Order 1, Signer 2 (Acme Countersignatory) = Routing Order 2 (sequential, not parallel). Confirm the DocuSign Integration User in UAT sandbox has 'Send on Behalf of Others' and 'Manage Envelope Templates' permissions enabled. Review the OmniScript DocuSign action configuration in OS-GroupOnboarding-GSAExecution to ensure the RecipientType for Signer 2 is set to 'signer' with the correct routing order integer. Implement a Platform Event subscriber (DocuSignStatusUpdateHandler) listening for DocuSign Connect webhook callbacks to update GSA.DocuSign_Status__c field via a Record-Triggered Flow or Apex handler. Add an Apex test method simulating the multi-step callback sequence. Validate fix end-to-end in UAT sandbox before promoting to regression suite.
