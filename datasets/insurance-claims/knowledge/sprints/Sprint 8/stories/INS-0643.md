## Description

As an Assignment Coordinator, I need newly created Credentialing_Case__c records to be automatically routed to the appropriate Credentialing Specialist based on the provider's primary specialty and the current open case workload per specialist so that credentialing cases are distributed equitably and specialty-matched without manual triage.

The routing logic must integrate with QGenda to check credentialing specialist availability (out-of-office / on-leave) before assignment. Extends the CAQH integration baseline established in INS-0621.

## Acceptance Criteria

Given a new Credentialing_Case__c record is created with Credentialing_Case__c.Status__c = 'New' and a linked Provider__c record,
When the auto-assignment routing logic executes,
Then the system must:
  1. Read Provider__c.Primary_Specialty__c from the linked provider
  2. Query the Specialty_Assignment_Matrix__mdt custom metadata to identify which Credentialing Specialist queue is mapped to that specialty
  3. Query all active Salesforce Users in that queue and retrieve their current open Credentialing_Case__c count (Status__c NOT IN ('Closed', 'Withdrawn'))
  4. Assign Credentialing_Case__c.OwnerId to the specialist with the lowest open case count (round-robin tiebreaker)
  5. Set Credentialing_Case__c.Assignment_Method__c = 'Auto-Routed'

Given the QGenda integration is enabled (QGenda_Integration_Active__c = TRUE in org-wide custom setting),
When the auto-assignment evaluates candidate specialists,
Then any specialist flagged as unavailable in QGenda (via QGenda REST API /api/schedule/availability endpoint) must be excluded from the candidate pool.

Given no available specialist is found for the provider's specialty after QGenda filtering,
When the routing logic exhausts all candidates,
Then Credentialing_Case__c.OwnerId must be set to the 'Credentialing — Unassigned' queue and a High-priority Task must be created for the Assignment Coordinator with Subject = 'Manual Assignment Required: [Case Number] — No Available Specialist'.

Given Credentialing_Case__c.Provider__c has Provider__c.CAQH_Verified__c = FALSE,
When the case is created,
Then in addition to assignment routing, an automated outreach Task must be created with Subject = 'CAQH Profile Incomplete — Provider Action Required' assigned to the routed specialist.

Given a Credentialing Specialist manually changes Credentialing_Case__c.OwnerId after auto-assignment,
When the record is saved,
Then Credentialing_Case__c.Assignment_Method__c must update to 'Manually Reassigned' and the previous assignee must be logged in Credentialing_Case__c.Previous_Assignee__c (lookup to User).

-Given providers with Primary_Specialty__c = 'General Practice', route to any available specialist regardless of specialty matrix- *(deprecated: specialty matrix now covers all specialties including General Practice per updated staffing model)*

## Solution

Auto-assignment logic implemented in an Apex after-insert trigger on Credentialing_Case__c (CredentialingCaseAssignmentTrigger) calling CredentialingCaseAssignmentService. The service reads Specialty_Assignment_Matrix__mdt to identify the target queue, then queries User records in the queue with an aggregate SOQL COUNT on open Credentialing_Case__c records. QGenda availability check is a synchronous callout via QGendaIntegrationService (HttpRequest to QGenda REST API) wrapped in a Future method to avoid mixed DML/callout restrictions; a Platform Event (Credentialing_Assignment_Ready__e) is published from the trigger and consumed by the Future-based callout handler to complete assignment asynchronously. Manual reassignment detection implemented via a Record-Triggered Flow on Credentialing_Case__c (Update path) comparing OwnerId to its prior value and writing to Assignment_Method__c and Previous_Assignee__c. All routing configuration stored in Specialty_Assignment_Matrix__mdt and CredentialingRouting_Settings__c custom setting for admin maintainability without code deployment.
