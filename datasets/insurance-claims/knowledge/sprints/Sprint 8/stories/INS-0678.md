## Description

As a Credentialing Specialist, I need the Salesforce Credentialing_Case__c lifecycle to automatically reflect a provider's current CAQH ProView attestation status so that I can make timely credentialing decisions without manually checking the CAQH portal, reducing credentialing cycle time and ensuring compliance with NCQA credentialing standards.

The CAQH REST API should be polled on a scheduled basis and on-demand from the Credentialing_Case__c record, updating relevant attestation fields and triggering workflow actions based on status changes.

## Acceptance Criteria

**AC1 — Scheduled CAQH Status Sync**
Given a Credentialing_Case__c record has Credentialing_Case__c.CAQH_Provider_ID__c populated and Case Status__c IN ('In Progress', 'Awaiting CAQH Attestation'),
When the nightly CAQH sync batch job runs,
Then the system calls the CAQH ProView API endpoint '/api/v1/provider/{caqhId}/attestation-status' and updates Credentialing_Case__c.CAQH_Attestation_Status__c, Credentialing_Case__c.CAQH_Last_Attested_Date__c, and Credentialing_Case__c.CAQH_Reattestaton_Due_Date__c with values from the API response.

**AC2 — On-Demand Refresh from Record Page**
Given a Credentialing Specialist is on a Credentialing_Case__c record page,
When they click the 'Refresh CAQH Status' quick action button,
Then the system makes a real-time callout to CAQH API and updates the same fields as AC1 within 5 seconds, displaying a toast notification 'CAQH status refreshed successfully' or an error message if the callout fails.

**AC3 — Auto-Advance Case on Complete Attestation**
Given Credentialing_Case__c.CAQH_Attestation_Status__c transitions to 'Complete',
When the sync job or on-demand refresh updates the field,
Then Credentialing_Case__c.Status__c automatically advances to 'CAQH Verified' via a Record-Triggered Flow, and a notification is sent to the case owner.

**AC4 — Flag Expired Attestation**
Given CAQH_Reattestaton_Due_Date__c < TODAY and CAQH_Attestation_Status__c != 'Complete',
When the sync job runs,
Then Credentialing_Case__c.Status__c is set to 'On Hold — CAQH Expired' and a Task is created for the Credentialing Specialist with ActivityDate = TODAY + 3.

**AC5 — API Error Handling and Retry**
Given the CAQH API callout returns a non-200 HTTP response,
When the batch job or on-demand refresh executes,
Then the error is logged to Integration_Error_Log__c with fields: Error_Source__c = 'CAQH', HTTP_Status_Code__c, Error_Message__c, Related_Record_Id__c = Credentialing_Case__c.Id, and the batch job retries up to 3 times with exponential backoff before marking the case with CAQH_Sync_Error__c = true.

## Solution

1. Create Named Credential 'CAQH_ProView_NC' for secure API authentication (OAuth 2.0 client credentials flow).
2. Build Apex class 'CAQHIntegrationService' with methods: syncAttestationStatus(List<Credentialing_Case__c>) for batch use and refreshSingleCase(Id caseId) for on-demand use.
3. Implement 'CAQHSyncBatch' (Database.Batchable, Database.AllowsCallouts) scheduled via CronTrigger to run nightly at 2:00 AM.
4. Create LWC quick action 'caqhRefreshAction' on Credentialing_Case__c page layout calling refreshSingleCase via @wire or imperative Apex.
5. Record-Triggered Flow 'Credentialing_CAQH_Status_Advance_Flow' on Credentialing_Case__c fires on CAQH_Attestation_Status__c update to 'Complete' to advance Status__c.
6. Integration_Error_Log__c custom object with retry counter field; retry logic implemented via Queueable Apex chain with max 3 iterations.
7. Remote Site Settings and CSP Trusted Sites configured for CAQH API domain.
