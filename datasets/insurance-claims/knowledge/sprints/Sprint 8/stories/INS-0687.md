## Description

As a Claims Operations Manager, I need inbound Electronic Remittance Advice (ERA) 835 transaction files from payers to be automatically parsed and mapped to Payment__c and Claim_Line__c records in Salesforce so that payment reconciliation is automated, manual ERA posting is eliminated, and EOB generation is triggered without human intervention.

ERA files are delivered via SFTP by three payer partners (BlueCross, Aetna, Cigna) on a nightly schedule. File volumes: up to 50,000 claim line transactions per file.

## Acceptance Criteria

Given an ERA 835 file is deposited on the SFTP endpoint,
When the MuleSoft integration poll runs every 15 minutes,
Then the file is ingested, parsed into individual CLP (Claim Payment) segments, and a Platform Event 'ERA_Batch_Received__e' is published with ERA_File_Name__c, Payer_ID__c, and Total_CLP_Count__c.

Given the Platform Event is received in Salesforce,
When the Apex Platform Event trigger processes the event,
Then a Payment__c record is created per CLP segment with fields: Payment__c.Payer__c (matched via Payer_ID__c → Account), Payment__c.Check_EFT_Number__c (CLP03), Payment__c.Payment_Date__c (CLP04), Payment__c.Payment_Amount__c (CLP02), Payment__c.ERA_File_Reference__c.

Given a Payment__c record is created from ERA,
When SVC (Service Payment) segments within the CLP are processed,
Then each SVC segment creates or updates a Claim_Line__c record: Claim_Line__c.Paid_Amount__c (SVC02), Claim_Line__c.Adjustment_Reason_Code__c (CAS segment reason code), Claim_Line__c.Line_Status__c = 'ERA Posted', Claim_Line__c.CPT_Code__c (SVC01 composite).

Given a CLP segment references a Claim__c.Claim_Number__c that cannot be matched in Salesforce,
When the mapping fails,
Then an ERA_Exception__c record is created with Exception_Type__c = 'Unmatched Claim', Source_CLP_Segment__c (raw segment text), and the record is assigned to the Claims Operations queue for manual resolution.

Given all Claim_Line__c records on a Claim__c have Line_Status__c = 'ERA Posted',
When the final SVC segment for that claim is processed,
Then Claim__c.Status__c is updated to 'Adjudicated' and a Platform Event 'Claim_Adjudicated__e' is published to trigger EOB generation.

-Given ERA file volume exceeds 50,000 CLP records, processing must complete within 30 minutes using synchronous Apex.- (Revised: async Queueable Apex chain required for volumes above 10,000 records per compliance with governor limits)

Given ERA processing completes for a full file,
When all CLP segments are processed,
Then a Payment_Reconciliation_Summary__c record is created with: Total_CLP_Processed__c, Total_Matched__c, Total_Exceptions__c, Processing_Duration_Seconds__c, and Status__c = 'Complete' or 'Complete with Exceptions'.

## Solution

MuleSoft integration layer handles SFTP polling, X12 835 EDI parsing (using MuleSoft EDI connector), and publishes Platform Events to Salesforce via Connected App. Salesforce-side: Apex trigger on ERA_Batch_Received__e invokes Queueable Apex class 'ERA835ProcessingQueueable' which processes CLP segments in batches of 200 using chained Queueable pattern to respect governor limits. SOQL queries Claim__c by Claim_Number__c with compound index on Claim_Number__c + Status__c for performance. Unmatched CLPs create ERA_Exception__c via a separate Apex method. Platform Event 'Claim_Adjudicated__e' published via EventBus.publish() after final SVC processing. Payment_Reconciliation_Summary__c created in the finalizer Queueable step. All objects use external ID fields (ERA_CLP_Reference__c) for idempotent upsert operations to handle file redelivery scenarios. Load tested in sandbox at 60,000 CLP records — processing time 22 minutes.
