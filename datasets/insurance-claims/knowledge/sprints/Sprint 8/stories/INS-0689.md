## Description

As a Claims Operations Manager, I need a consolidated Provider 360 FlexCard on the Provider__c (Person Account) and Facility__c (Business Account) record pages that surfaces the provider's active Provider_Network_Agreement__c (PNA) details, associated Fee_Schedule__c tier, current network participation status, and any open Provider_Change_Notification__c (PCN) records, so that I can make accurate in-network vs. out-of-network adjudication decisions at the point of claim review without navigating to multiple related lists or running separate SOQL queries.

This story extends the foundational Provider data model work completed in INS-0658 and incorporates the credentialing status feed established in INS-0663.

**Business Context:** Currently, claims adjusters must open 3–5 separate tabs to confirm whether a rendering provider is in-network for the specific LOB on the claim, what fee schedule tier applies (e.g., Tier 1 PPO vs. Tier 2 EPO), and whether any active PCN or Network_Termination_Case__c might affect eligibility. This causes an average of 4.2 minutes of additional handle time per claim and has resulted in 18 documented incorrect in-network determinations in Q3.

## Acceptance Criteria

**AC1 — FlexCard Displays Active PNA Summary**
Given a Claims Operations Manager views a Provider__c or Facility__c record page,
When the Provider 360 FlexCard loads,
Then the card must display the active Provider_Network_Agreement__c where PNA.Status__c = 'Active' AND PNA.Effective_Date__c <= TODAY AND (PNA.Termination_Date__c = NULL OR PNA.Termination_Date__c >= TODAY),
And the following PNA fields must be visible: PNA.Contract_Number__c, PNA.Network_Tier__c, PNA.LOB__c, PNA.Effective_Date__c, PNA.Termination_Date__c.
If multiple active PNAs exist (multi-LOB), all must be displayed in a child card list grouped by PNA.LOB__c.

**AC2 — Fee Schedule Tier Display**
Given an active PNA is present,
When the FlexCard renders the PNA section,
Then the associated Fee_Schedule__c record (linked via PNA.Fee_Schedule__c lookup) must display: Fee_Schedule__c.Schedule_Name__c, Fee_Schedule__c.Tier__c, Fee_Schedule__c.Effective_Date__c, and Fee_Schedule__c.CPT_Modifier_Override__c (if populated).
If Fee_Schedule__c.Status__c = 'Expired', the tier display must render with a warning badge labeled 'Fee Schedule Expired'.

**AC3 — Network Participation Status Badge**
Given the FlexCard is loaded for any provider record,
When the DataRaptor retrieves participation data,
Then a color-coded status badge must appear at the top of the card:
- Green badge: 'In-Network' when PNA.Status__c = 'Active' and no open Network_Termination_Case__c exists.
- Yellow badge: 'Pending Termination' when a Network_Termination_Case__c exists with Status__c IN ('Open', 'In Review') linked to the provider.
- Red badge: 'Out-of-Network' when no active PNA exists or PNA.Status__c = 'Terminated'.
- -Blue badge: 'Provisional' when Credentialing_Case__c.Status__c = 'Provisional' — deprecated per compliance review 2024-11-02, removed from scope-

**AC4 — Open PCN Alerts**
Given one or more Provider_Change_Notification__c records exist for the provider with PCN.Status__c IN ('Submitted', 'Under Review'),
When the FlexCard loads,
Then an alert banner must appear below the status badge displaying: 'X Open Change Notification(s)' with a link to the PCN list view filtered to that provider.
And each PCN must show PCN.Change_Type__c, PCN.Submitted_Date__c, and PCN.Assigned_To__c.

**AC5 — Credentialing Status Integration**
Given the provider has an associated Credentialing_Case__c record (linked via Credentialing_Case__c.Provider__c),
When the FlexCard loads,
Then the most recent Credentialing_Case__c must display: Credentialing_Case__c.Status__c, Credentialing_Case__c.Expiration_Date__c, and Credentialing_Case__c.CAQH_Profile_Status__c.
If Credentialing_Case__c.Expiration_Date__c < TODAY + 90 days, display a 'Credentials Expiring Soon' warning.

**AC6 — Performance**
Given a Claims Operations Manager loads the Provider 360 FlexCard,
When the OmniStudio DataRaptor executes its SOQL retrievals,
Then the FlexCard must fully render within 3 seconds for providers with up to 10 active PNAs and 50 Claim_Line__c history records.

**AC7 — Access Control**
Given a user with the 'Claims_Adjuster' Permission Set Group,
When they view the Provider 360 FlexCard,
Then they must see all AC1–AC5 data in read-only mode with no edit actions exposed.
Given a user with the 'Provider_Network_Specialist' Permission Set Group,
When they view the same FlexCard,
Then an 'Edit PNA' action button must be visible that launches the PNA OmniScript.

## Solution

**OmniStudio Implementation:**
1. **DataRaptor Turbo Extract (DR-ProviderPNA-Extract):** Build a multi-object DataRaptor that retrieves Provider__c / Facility__c base data, joins to Provider_Network_Agreement__c (filtered by active status and date), joins to Fee_Schedule__c via PNA lookup, joins to Provider_Change_Notification__c (open statuses), and joins to Credentialing_Case__c (most recent by CreatedDate DESC LIMIT 1). Use DataRaptor Input parameters for RecordId to support both Person Account and Business Account record types.
2. **FlexCard (FC-Provider360):** Build a parent FlexCard with conditional rendering states (In-Network / Pending Termination / Out-of-Network) driven by formula fields derived from the DR output. Embed a child FlexCard (FC-PNAList) for multi-PNA scenarios using a repeater block grouped by LOB__c. Use OmniStudio custom CSS classes for badge color-coding per AC3.
3. **OmniScript (OS-EditPNA):** Wire the 'Edit PNA' action button (AC7) to an existing OmniScript for PNA updates — reference the OmniScript built in INS-0658. Pass PNA.Id as input parameter.
4. **Permission Set Group Visibility:** Use FlexCard conditional actions (showAction formula) referencing {UserPermissions.Claims_Adjuster} vs. {UserPermissions.Provider_Network_Specialist} to show/hide the Edit action per AC7.
5. **Record Page Deployment:** Deploy FC-Provider360 to both the Provider__c (Person Account) Lightning Record Page and the Facility__c (Business Account) Lightning Record Page via App Builder, positioned in the right-hand sidebar at 30% width.
6. **Performance Optimization:** Enable DataRaptor caching (TTL 300s) for Fee_Schedule__c data which is low-volatility; disable caching for PCN and Credentialing_Case__c data which requires real-time accuracy.
