## Description

As a Claims Operations Manager, I need the FNOL intake OmniScript to automatically trigger a Coordination of Benefits (COB) verification check against the policyholder's active Policy__c records so that duplicate or overlapping coverage is identified at the point of first notice, reducing downstream payment errors and manual rework by the adjudication team.

When a claimant reports a new loss, the system should query all Policy__c records associated with the claimant's Contact to detect secondary or tertiary coverage. If COB is detected, the claim intake agent must acknowledge and record the primary payer order before the Claim__c record is created.

## Acceptance Criteria

Given a Claims Operations Manager or intake agent is completing the FNOL OmniScript step 'Policyholder Verification',
When the agent enters the policyholder's Member_ID__c or SSN_Encrypted__c,
Then the OmniScript executes an Integration Procedure (IP_COB_PolicyLookup) that queries all Policy__c records where Policy__c.Insured_Contact__c matches the claimant and Policy__c.Status__c = 'Active'.

Given the IP_COB_PolicyLookup returns 2 or more active Policy__c records,
When the COB check step renders,
Then a COB warning panel displays all active policies with Policy__c.Policy_Number__c, Policy__c.Carrier_Name__c, Policy__c.LOB__c, and Policy__c.Group_Number__c.

Given COB is detected,
When the agent proceeds past the COB step,
Then the agent must select a Primary_Payer_Policy__c and Secondary_Payer_Policy__c from the displayed list before the 'Next' button is enabled; the OmniScript must not allow advancement without this selection.

Given the agent has selected payer order,
When the Claim__c record is created at OmniScript completion,
Then Claim__c.COB_Flag__c = true, Claim__c.Primary_Policy__c is populated with the selected primary Policy__c ID, and Claim__c.Secondary_Policy__c is populated if applicable.

Given the IP_COB_PolicyLookup returns exactly 1 active Policy__c record,
When the COB step renders,
Then no COB warning is shown, Claim__c.COB_Flag__c = false, and the OmniScript advances automatically.

Given the COB check Integration Procedure call fails or times out,
When the error is caught,
Then a non-blocking warning message is displayed ('COB check unavailable — please verify manually') and the agent may proceed; Claim__c.COB_Manual_Review__c is set to true.

## Solution

Extend the existing 'FNOL_Intake_OmniScript' by adding a new step group 'COB_Verification' between steps 'Policyholder_Verification' and 'Loss_Details'. Create Integration Procedure 'IP_COB_PolicyLookup' using a DataRaptor Turbo Extract to query Policy__c with a dynamic SOQL filter on Insured_Contact__c and Status__c. Use OmniScript Set Values element to store returned policy list in a repeatable block. Add a Conditional View block using the JSON path response count to show/hide the COB warning panel. Implement a custom LWC 'cobPayerOrderSelector' embedded in the OmniScript for the payer priority selection UX. Map final selections to Claim__c fields via the OmniScript's DataRaptor Post action 'DRP_Create_Claim_With_COB'. Add error handling via OmniScript HTTP Action timeout configuration (10s) with fallback Set Values to COB_Manual_Review__c = true.
