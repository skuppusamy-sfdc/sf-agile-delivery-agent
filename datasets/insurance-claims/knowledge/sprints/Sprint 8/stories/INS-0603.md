## Description

As a Credentialing Specialist, I need the system to automatically create and route a Provider_Change_Notification__c (PCN) record whenever key credentialing or demographic fields are updated on a Provider__c (Person Account) record — such as NPI, TIN, license number, or board certification status — so that downstream network contracting, claims adjudication, and compliance teams are proactively notified of material provider data changes without relying on manual notification processes.

This story addresses a UAT finding where testers updated Provider__c.NPI__c and no PCN was generated, causing downstream Claim__c adjudication to reference stale provider data.

## Acceptance Criteria

Given a Provider__c record where any of the following fields are modified: NPI__c, TIN__c, License_Number__c, License_Expiration_Date__c, Board_Certification_Status__c, or Primary_Specialty__c,
When the record is saved,
Then a Provider_Change_Notification__c record is automatically created with PCN.Provider__c = Provider__c.Id, PCN.Change_Type__c populated from a picklist mapping the changed field, PCN.Status__c = 'Pending Review', PCN.Changed_By__c = running user, and PCN.Change_Date__c = TODAY().

Given a PCN is created with PCN.Change_Type__c = 'NPI Change' or 'TIN Change',
When the PCN record is saved,
Then the PCN is auto-assigned to the queue 'Credentialing_High_Priority_Queue' and PCN.Priority__c is set to 'High'.

Given a PCN is created with PCN.Change_Type__c = 'License Update' or 'Specialty Change',
When the PCN record is saved,
Then the PCN is assigned to the queue 'Provider_Data_Review_Queue' and PCN.Priority__c is set to 'Medium'.

Given an active PCN (PCN.Status__c != 'Closed') already exists for the same Provider__c and Change_Type__c,
When a new change to the same field is saved,
Then no duplicate PCN is created; instead, PCN.Last_Updated__c and PCN.Change_Details__c on the existing open PCN are updated with the latest change information.

Given a PCN is created,
When PCN.Status__c = 'Pending Review' for more than 5 business days,
Then an escalation email is sent to the Credentialing Specialist's manager using the 'PCN_Escalation_Alert' email template and PCN.Escalated__c is set to TRUE.

Given any PCN creation or update event,
When the PCN is persisted,
Then a Platform Event (ProviderDataChangeEvent__e) is published containing Provider__c.Id, changed field API name, old value, and new value to notify downstream claims adjudication subscribers.

## Solution

Implement a Record-Triggered Flow (RTF-Provider-PCNCreation) on Provider__c (after update) that uses a Decision element to detect changes to monitored fields (NPI__c, TIN__c, License_Number__c, License_Expiration_Date__c, Board_Certification_Status__c, Primary_Specialty__c) using ISCHANGED() formula references. Use a Get Records element to check for existing open PCNs for the same provider and change type to enforce deduplication. Create PCN via Create Records element with appropriate field mappings. Invoke a separate Record-Triggered Flow (RTF-PCN-QueueRouting) on Provider_Change_Notification__c (after insert) to handle queue assignment logic based on Change_Type__c picklist value. Build a Scheduled Flow (SF-PCN-EscalationCheck) running nightly to identify PCNs exceeding the 5-business-day SLA and send alerts via EmailAlert action. Publish ProviderDataChangeEvent__e Platform Event from an Apex invocable action called within the RTF to ensure governor limit-safe async publishing. Update Field History Tracking on Provider__c for all monitored fields.
