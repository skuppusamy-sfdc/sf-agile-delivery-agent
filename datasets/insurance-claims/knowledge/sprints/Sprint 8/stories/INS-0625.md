## Description

As a Claims Operations Manager, I need the system to automatically validate each Claim_Line__c CPT code against the active Fee_Schedule__c effective date range at the time of claim submission so that we prevent incorrect reimbursement rates from being applied to claims where the service date falls outside the contracted fee schedule window, reducing manual rework and overpayment risk.

Fee schedules are stored in Fee_Schedule__c with Effective_Date__c, Expiration_Date__c, LOB__c, and Fee_Schedule_Type__c (e.g., 'Medicare Equivalent', 'Contracted Rate'). Each Claim_Line__c carries CPT_Code__c and Date_of_Service__c.

## Acceptance Criteria

Given a Claim_Line__c is created or updated with CPT_Code__c and Date_of_Service__c populated,
When the system looks up the associated Fee_Schedule__c via the parent Claim__c.LOB__c and Facility__c.Network_Tier__c,
Then the Fee_Schedule__c returned must have Effective_Date__c <= Claim_Line__c.Date_of_Service__c AND Expiration_Date__c >= Claim_Line__c.Date_of_Service__c.

Given no active Fee_Schedule__c is found for the CPT/LOB/date combination,
When the Claim_Line__c is saved,
Then a validation error is thrown: 'No active fee schedule found for CPT [code] on date of service [date]. Please review Fee Schedule configuration.' and Claim_Line__c.Status__c is set to 'Pended — Fee Schedule Missing'.

Given a valid Fee_Schedule__c is found,
When the Claim_Line__c is saved,
Then Claim_Line__c.Allowed_Amount__c is auto-populated from Fee_Schedule_Line__c.Allowed_Amount__c where Fee_Schedule_Line__c.CPT_Code__c = Claim_Line__c.CPT_Code__c.

Given a Fee_Schedule__c with Fee_Schedule_Type__c = 'Medicare Equivalent',
When Allowed_Amount__c is calculated,
Then the system applies the Facility__c.Medicare_Conversion_Factor__c multiplier: Allowed_Amount__c = Fee_Schedule_Line__c.RVU__c * Facility__c.Medicare_Conversion_Factor__c.

Given a Claim__c with Claim_Source__c = 'ERA' (Electronic Remittance Advice inbound),
When fee schedule validation runs,
Then validation is bypassed and Claim_Line__c.Fee_Schedule_Override__c = true is set to indicate external adjudication.

-Given CPT code not found in Fee_Schedule_Line__c, default to UCR (Usual, Customary, Reasonable) rate lookup- (deprecated — UCR fallback removed per compliance review, pend instead).

Given a bulk import of more than 200 Claim_Line__c records in a single transaction,
When fee schedule validation runs,
Then all records are validated without hitting Salesforce governor limits (SOQL < 100, heap < 10MB) using a bulkified Apex approach.

## Solution

Implement Apex Trigger 'ClaimLineFeeScheduleTrigger' on Claim_Line__c (before insert, before update) delegating to FeeScheduleValidationService.validate(List<Claim_Line__c>). Service performs a single bulkified SOQL query joining Fee_Schedule__c and Fee_Schedule_Line__c using a Map<String, Fee_Schedule_Line__c> keyed by CPT_Code__c + LOB__c. Medicare Equivalent calculation handled in FeeScheduleCalculatorStrategy class implementing IFeeCalculationStrategy interface (Strategy pattern). ERA bypass logic checks Claim__c.Claim_Source__c via a parent-field map populated in the trigger handler. Custom Metadata Type 'Fee_Schedule_Config__mdt' stores LOB-to-fee-schedule-type mappings to avoid hardcoding. Deploy with associated Apex test class achieving >85% coverage including bulk test of 200 records.
