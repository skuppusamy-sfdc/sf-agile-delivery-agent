# INS-0610: Implement automated Fee Schedule eligibility validation on Claim_Line__c submission

## Description

As a Claims Operations Manager, I need the system to automatically validate each Claim_Line__c record against the applicable Fee_Schedule__c at the time of submission so that allowed amounts are calculated consistently, manual adjudication errors are reduced, and payout variances are flagged before payment disbursement.

The fee schedule lookup must account for the rendering provider's active PNA, the CPT code on the claim line, the date of service, and the member's LOB.

## Acceptance Criteria

Given a Claim_Line__c record is created or updated with Status__c = 'Submitted',
When the fee schedule validation process executes,
Then the system must identify the applicable Fee_Schedule__c by matching: Fee_Schedule__c.LOB__c = parent Claim__c.LOB__c AND Fee_Schedule__c.Effective_Date__c <= Claim_Line__c.Date_of_Service__c AND Fee_Schedule__c.Expiration_Date__c >= Claim_Line__c.Date_of_Service__c AND Fee_Schedule__c.Provider_Network_Agreement__c = rendering provider's active PNA Id.

Given a matching Fee_Schedule__c is found,
When the CPT code on Claim_Line__c.CPT_Code__c matches a Fee_Schedule_Line__c.CPT_Code__c record,
Then Claim_Line__c.Allowed_Amount__c is set to Fee_Schedule_Line__c.Allowed_Amount__c and Claim_Line__c.Fee_Schedule_Applied__c is populated with the Fee_Schedule__c.Id.

Given a matching Fee_Schedule__c is found but the CPT code does not match any Fee_Schedule_Line__c,
When validation runs,
Then Claim_Line__c.Status__c is set to 'Pended' and Claim_Line__c.Pend_Reason__c = 'CPT Code Not Found in Fee Schedule' and a Task is created assigned to the Claims Operations Manager queue.

Given no matching Fee_Schedule__c exists for the provider/LOB/date combination,
When validation runs,
Then Claim_Line__c.Status__c = 'Pended', Claim_Line__c.Pend_Reason__c = 'No Active Fee Schedule Found', and parent Claim__c.Requires_Manual_Review__c = true.

Given Claim_Line__c.Billed_Amount__c exceeds Fee_Schedule_Line__c.Allowed_Amount__c,
When the allowed amount is applied,
Then Claim_Line__c.Variance_Amount__c = Claim_Line__c.Billed_Amount__c - Claim_Line__c.Allowed_Amount__c and Claim_Line__c.Variance_Flag__c = true.

-Given COB applies and the member has a secondary payer, reduce Allowed_Amount__c by the primary payer's paid amount before storing.-
(COB logic deferred to INS-0631)

Given the parent Claim__c.LOB__c = 'Medicare Advantage',
When fee schedule validation runs,
Then the system must additionally verify that the CPT code is valid per the current CMS fee schedule integration (InsCAQHIntegrationService) and log the CMS reference rate in Claim_Line__c.CMS_Reference_Rate__c.

## Solution

1. Create Apex class 'FeeScheduleValidationService' with method validateClaimLine(Id claimLineId) containing the full lookup and calculation logic.
2. Implement a Record-Triggered Flow 'RT_ClaimLine_FeeScheduleValidation' on Claim_Line__c, triggering on Create and Update when Status__c changes to 'Submitted', which calls the Apex action via an Apex Action element (to support complex SOQL across Fee_Schedule__c and Fee_Schedule_Line__c objects).
3. FeeScheduleValidationService executes a two-step SOQL: first query Fee_Schedule__c filtered by LOB, date range, and PNA; then query Fee_Schedule_Line__c by CPT code within the matched schedule.
4. Use a Custom Metadata Type 'Fee_Schedule_Config__mdt' to store LOB-to-fee-schedule-type mappings for easy maintenance without code deployments.
5. For Medicare Advantage LOB, call InsCAQHIntegrationService.getCMSReferenceRate(cptCode, dateOfService) via Named Credential to retrieve CMS rate and populate Claim_Line__c.CMS_Reference_Rate__c.
6. Task creation for pended lines uses a Flow subflow 'SF_CreateClaimsManagerTask' to route to the Claims_Operations_Manager queue.
7. Add Apex test class 'FeeScheduleValidationServiceTest' with minimum 90% coverage including positive, negative, and Medicare Advantage scenarios.
