## Description

**Bug Report — UAT Sprint 8**

During UAT regression testing of the ERA bulk import process, the parent Claim__c.Status__c field fails to roll up to 'Adjudicated' after all child Claim_Line__c records are individually set to 'Adjudicated' via the ERA 835 file processing job.

**Steps to Reproduce:**
1. Navigate to Claims Processing > ERA Import and upload a valid 835 ERA file containing 3+ claim lines for a single Claim__c record.
2. Confirm the DataRaptor Transform job completes without errors.
3. Open the parent Claim__c record.
4. Observe Claim__c.Status__c — it remains 'In Adjudication' rather than rolling up to 'Adjudicated'.
5. Manually edit any single Claim_Line__c record and save — the parent Claim__c.Status__c then correctly updates.

**Expected Behavior:** After all Claim_Line__c records for a given Claim__c are set to Status__c = 'Adjudicated' via bulk ERA import, the parent Claim__c.Status__c should automatically update to 'Adjudicated' and trigger downstream Payment__c generation.

**Actual Behavior:** Parent Claim__c.Status__c remains 'In Adjudication'. Payment__c records are not generated. EOB dispatch is blocked.

**Root Cause Hypothesis:** The Record-Triggered Flow on Claim_Line__c that performs the roll-up check is not firing during bulk DML operations initiated by the DataRaptor batch process, likely due to the DataRaptor executing DML in a context that bypasses standard trigger invocation or the flow's bulk recursion guard is short-circuiting after the first record.

as per Description

## Solution

Investigate the DataRaptor Transform (DR-ERA-835-LineUpdate) bulk DML execution context. Replace the roll-up logic currently embedded in the Record-Triggered Flow (RTF-ClaimLine-StatusRollup) with an Apex Trigger (ClaimLineTrigger, after insert/update) that uses a TriggerHandler pattern to aggregate Claim_Line__c status values per parent Claim__c.Id in a Set, then performs a single SOQL query and bulk update on affected Claim__c records. Ensure the Apex handler fires correctly in async/batch contexts. Add a Platform Event (ClaimAdjudicatedEvent__e) publish step post-update to trigger downstream Payment__c creation flow. Deploy updated DataRaptor and deactivate the conflicting RTF to prevent double-processing.
