## Description

As a Broker Relations Coordinator, I need the Group/Employer Onboarding OmniScript to automatically pre-populate Group_Service_Agreement__c (GSA) fields from the executed Conga CLM agreement data so that I do not have to re-enter contract terms that were already negotiated and signed, reducing data entry errors and onboarding cycle time.

Extends INS-0629 (Group Employer Onboarding OmniScript base build).

## Acceptance Criteria

Given a Conga CLM agreement record has a status of 'Executed' and is linked to an Account (employer group) via the CLM_Agreement__c.Account__c lookup,
When the Broker Relations Coordinator launches the Group Employer Onboarding OmniScript for that Account,
Then the OmniScript Step 2 'Contract Terms' fields are pre-populated from the Conga CLM agreement: GSA.Effective_Start_Date__c, GSA.Effective_End_Date__c, GSA.Benefit_Plan_Type__c, GSA.Premium_Rate__c, and GSA.Billing_Frequency__c.

Given the CLM agreement data is successfully retrieved,
When the coordinator reviews pre-populated fields in the OmniScript,
Then each pre-populated field displays a read-only indicator icon and tooltip: 'Pre-populated from executed CLM Agreement [Agreement Name]' to distinguish auto-filled from manually entered data.

Given the linked Conga CLM agreement does not have status = 'Executed' (e.g., 'In Negotiation'),
When the OmniScript attempts to retrieve CLM data,
Then no pre-population occurs, fields remain blank and editable, and an informational message displays: 'No executed CLM Agreement found. Please enter contract terms manually.'

-Given multiple executed CLM agreements exist for the same Account, pre-populate from the oldest executed agreement.- (Deprecated — updated rule: pre-populate from the most recently executed agreement by CLM_Agreement__c.Executed_Date__c DESC, see UAT feedback ticket INS-0641)

Given pre-populated data is present,
When the coordinator modifies any pre-populated field,
Then GSA.CLM_Data_Overridden__c = true is set and the field loses the read-only indicator, allowing free-form edit.

Given the OmniScript completes and the coordinator clicks 'Submit',
When the GSA record is created,
Then GSA.Source_CLM_Agreement_Id__c is stamped with the Conga CLM agreement record ID for audit traceability.

## Solution

Implement a DataRaptor Extract ('DR_Fetch_Executed_CLM_Agreement') that queries CLM_Agreement__c filtered by Account__c = current Account Id AND Status__c = 'Executed' ORDER BY Executed_Date__c DESC LIMIT 1. The DataRaptor is invoked at OmniScript Step 2 entry via an Integration Procedure ('IP_Prefill_GSA_From_CLM'). Pre-populated field state managed via OmniScript conditional read-only expressions bound to a JSON flag 'isPrefilled'. Override tracking handled by OmniScript onChange events setting a JSON property 'clmDataOverridden' = true, mapped to GSA.CLM_Data_Overridden__c on save. Final GSA record creation handled by a DataRaptor Upsert action at OmniScript submission, stamping GSA.Source_CLM_Agreement_Id__c. Integration with Conga CLM uses the existing named credential 'Conga_CLM_NC' established in INS-0629.
