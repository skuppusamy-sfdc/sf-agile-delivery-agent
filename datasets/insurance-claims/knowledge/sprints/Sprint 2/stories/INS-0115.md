## Description

As a Provider Network Specialist, I need the system to automatically create a Provider_Change_Notification__c (PCN) record whenever a provider's TIN (Provider__c.TIN__c) or primary practice address (Provider__c.BillingStreet, Provider__c.BillingCity, Provider__c.BillingState, Provider__c.BillingPostalCode) is updated — whether via the Workday integration feed or a manual Salesforce edit — so that downstream teams (Claims Operations, Credentialing, Compliance) are notified of material provider data changes within the required 24-hour SLA. Extends INS-0109.

## Acceptance Criteria

Given a Provider__c record's Provider__c.TIN__c field value changes (old value != new value),
When the record is saved (via UI, API, or Workday integration Platform Event),
Then a new Provider_Change_Notification__c must be automatically created with: PCN__c.Provider__c = Provider__c.Id, PCN__c.Change_Type__c = 'TIN Change', PCN__c.Previous_Value__c = old TIN value, PCN__c.New_Value__c = new TIN value, PCN__c.Status__c = 'Open', PCN__c.Source__c = 'Workday' or 'Manual' based on the originating context, PCN__c.Effective_Date__c = TODAY().

Given a Provider__c record's BillingStreet, BillingCity, BillingState, or BillingPostalCode changes,
When the record is saved,
Then a new PCN must be created with PCN__c.Change_Type__c = 'Address Change' and PCN__c.Previous_Value__c = concatenated prior address string and PCN__c.New_Value__c = concatenated new address string.

Given a PCN is created,
When PCN__c.Status__c = 'Open',
Then a Task must be auto-created and assigned to the provider's owning Provider Relations Specialist (Provider__c.OwnerId) with Subject = 'Review PCN: [Change_Type__c] for [Provider Name]', ActivityDate = TODAY() + 1 (24-hour SLA), and Priority = 'High'.

Given both TIN and address change simultaneously in the same save transaction,
When the automation fires,
Then two separate PCN records must be created — one per change type — not a single combined record.

Given the Workday integration publishes a 'Provider_Data_Update__e' Platform Event with PayloadType = 'TIN_UPDATE' or 'ADDRESS_UPDATE',
When the Platform Event is consumed by the Salesforce subscriber,
Then the Provider__c record must be updated first, triggering the PCN creation logic as described above, and the Platform Event must be acknowledged with a success response.

-Given the provider has an active Network_Termination_Case__c, suppress PCN creation- (deprecated — legal confirmed PCNs must still be created even during termination proceedings per compliance policy update 2024-11)

Given a PCN is created with Change_Type__c = 'TIN Change',
When the record is saved,
Then an email alert must be dispatched to the Compliance Officer distribution group (EmailTemplate: 'PCN_TIN_Change_Compliance_Alert') within 15 minutes.

Given a duplicate PCN already exists (same Provider__c.Id, same Change_Type__c, PCN__c.Status__c = 'Open', created within the last 24 hours),
When a new change event fires for the same field,
Then no duplicate PCN must be created; instead, the existing open PCN's PCN__c.Last_Updated__c must be stamped and a note appended to PCN__c.Notes__c.

## Solution

Implement a Record-Triggered Flow 'RTF_Provider_PCN_Creation' on the Provider__c object, triggered After Save on Update. Flow uses formula resources to detect field changes on TIN__c and Billing address fields using {!$Record.TIN__c} != {!$Record__Prior.TIN__c} conditions. Separate decision branches for TIN vs Address change create PCN records via Create Records elements. Source__c determination uses a custom Flow variable populated by a before-save flow that stamps a hidden 'Last_Modified_Source__c' field based on API user profile (Workday integration user vs standard user). Platform Event 'Provider_Data_Update__e' consumed via a Platform Event-Triggered Flow 'PETF_Workday_ProviderDataUpdate' that upserts Provider__c via external ID (Provider__c.Workday_Worker_ID__c) before the RTF fires. Duplicate suppression implemented in the RTF via a Get Records check for existing open PCNs within 24 hours before the Create Records element. Task creation handled within the same RTF. Email alert for TIN changes uses a Send Email action with 'PCN_TIN_Change_Compliance_Alert' template. All flows governed by a custom metadata type 'PCN_Automation_Config__mdt' to allow field-level toggle without deployment.
