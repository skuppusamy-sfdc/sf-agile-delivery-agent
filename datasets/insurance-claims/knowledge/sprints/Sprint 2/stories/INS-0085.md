## Description

As a Claims Operations Manager, I need a unified Policyholder 360 FlexCard on the Contact/Person Account record so that I can view all relevant policy, claim, and payment information in a single screen without navigating to multiple objects.

This card should surface active Policy__c records, open or in-progress Claim__c records, recent Payment__c disbursements, and any pending EOB__c documents linked to the policyholder. The intent is to reduce average handle time during inbound FNOL calls by eliminating tab-switching.

## Acceptance Criteria

Given a Claims Operations Manager opens a Policyholder Person Account record,
When the Policyholder 360 FlexCard loads,
Then the 'Active Policies' panel displays all Policy__c records where Policy__c.Status__c = 'Active' AND Policy__c.PolicyHolder__c = current Contact Id, showing fields: Policy__c.Policy_Number__c, Policy__c.LOB__c, Policy__c.Effective_Date__c, Policy__c.Expiration_Date__c.

Given the policyholder has one or more claims,
When the 'Open Claims' panel renders,
Then it displays Claim__c records where Claim__c.Status__c IN ('FNOL Received','In Review','Pending Documentation') linked via Claim__c.Policyholder__c, showing Claim__c.Claim_Number__c, Claim__c.Date_of_Loss__c, Claim__c.Total_Incurred__c, and Claim__c.Assigned_Adjuster__c.

Given the policyholder has recent payments,
When the 'Payment Summary' panel renders,
Then it displays the last 5 Payment__c records ordered by Payment__c.Payment_Date__c DESC, showing Payment__c.Amount__c, Payment__c.Payment_Method__c, and Payment__c.Status__c.

Given a policyholder has no active policies,
When the Active Policies panel loads,
Then an inline empty-state message reads 'No active policies found for this policyholder.'

-Given the card loads, a COB Alerts section displays any Coordination_of_Benefits__c flags- (deprecated — moved to INS-0091)

Given the user does not have the 'Claims_Ops_User' Permission Set Group,
When they navigate to the Policyholder record,
Then the FlexCard is not visible and no underlying data is exposed.

## Solution

Build an OmniStudio FlexCard named 'Policyholder_360_Card' with three child card panels: ActivePolicies, OpenClaims, PaymentSummary. Each panel uses a dedicated DataRaptor Extract to query respective objects with field-level filtering. The FlexCard is surfaced via a Lightning App Builder component drop on the Person Account page layout, gated by Permission Set Group 'Claims_Ops_User'. Conditional rendering states handle empty datasets. DataRaptor Turbo Extract used for Payment__c panel to support LIMIT 5 ordering. FlexCard deployed to the 'Claims Operations' Lightning App.
