## Description

**Bug Report — Provider 360 FlexCard null error on providers with no active PNA**

**Environment:** Full Sandbox (Acme-FS1)
**Build:** OmniStudio Package v1.1 (INS-0102 deployment)

**Steps to Reproduce:**
1. Navigate to any Provider__c Person Account record where no Provider_Network_Agreement__c records exist OR all PNA records have Network_Status__c = 'Terminated'.
2. Observe the Provider 360 FlexCard loading on the record page.
3. The FlexCard spinner runs for approximately 8 seconds then displays the error: 'DataRaptor extract returned null — DR_Get_Active_PNA'.
4. The entire FlexCard fails to render; no panels are visible.

**Expected Behavior:** The FlexCard should render successfully with all panels visible. The Network Agreement Summary panel should display an empty-state message such as 'No active network agreements on file' when no active PNA exists.

**Actual Behavior:** The FlexCard throws a null reference error originating from DR_Get_Active_PNA and the entire component fails to render, blocking the Provider Relations Specialist from viewing any provider data.

**Frequency:** Reproducible 100% of the time on providers with no active PNA. Affects approximately 340 provider records in sandbox based on data audit.

**Impact:** High — Provider Relations team cannot view any Provider 360 data for out-of-network or terminated providers, which represent a significant portion of their daily workflow.

**Related Story:** INS-0102

## Acceptance Criteria

As per Description.

Additional verification:
- Given a Provider__c record with zero PNA records, the FlexCard renders all panels without error and the Network Agreement Summary panel shows 'No active network agreements on file'.
- Given a Provider__c record where all PNAs have Network_Status__c = 'Terminated', the FlexCard renders and the panel shows 'No active network agreements on file — [count] terminated agreement(s) on record. View All'.
- No browser console errors are thrown during FlexCard load for null-PNA providers.

## Solution

Root cause: DR_Get_Active_PNA DataRaptor Extract does not handle a zero-row result set gracefully — the FlexCard's downstream DataRaptor Transform DR_Get_Fee_Schedule_Tier attempts to read PNA Id from the null extract output, causing a cascade failure. Fix: (1) Add a null-check condition in the FlexCard state between the PNA extract and the fee schedule transform — if DR_Get_Active_PNA returns empty, set a FlexCard state variable 'hasPNA' = false and short-circuit subsequent DataRaptor calls. (2) Add an empty-state conditional view block in the Network Agreement Summary panel that renders when hasPNA = false. (3) Update DR_Get_Active_PNA to return a default empty object {} instead of null when no rows are found, using the DataRaptor 'Return Empty Response' output option. Deploy fix to Acme-FS1 sandbox for QA re-test before promoting to Acme-UAT.
