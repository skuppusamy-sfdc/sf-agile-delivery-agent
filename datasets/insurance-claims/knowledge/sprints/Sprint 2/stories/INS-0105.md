# INS-0105: Provider Change Notification (PCN) auto-creation when Provider NPI or TIN is updated on Provider__c record

## Description

As a Provider Relations Specialist, I need the system to automatically create a Provider_Change_Notification__c (PCN) record whenever a provider's NPI or TIN is updated on their Provider__c Person Account so that downstream compliance, claims adjudication, and network contracting teams are alerted to data changes that may affect claim routing and payment accuracy.

This automation must capture both the old and new values for audit trail purposes and assign the PCN to the Provider Relations queue for review.

## Acceptance Criteria

Given a Provider__c Person Account record exists with Provider__c.NPI__c and Provider__c.TIN__c populated,
When an authorized user updates Provider__c.NPI__c to a new value and saves the record,
Then a Provider_Change_Notification__c record is automatically created with:
  - PCN__c.Provider__c = the updated Provider__c record Id
  - PCN__c.Change_Type__c = 'NPI Update'
  - PCN__c.Old_Value__c = the previous NPI value
  - PCN__c.New_Value__c = the new NPI value
  - PCN__c.Status__c = 'Pending Review'
  - PCN__c.Created_Date__c = system timestamp
  - PCN__c.OwnerId = 'Provider Relations Queue'

Given a Provider__c TIN is updated,
When the record is saved,
Then a PCN is created with PCN__c.Change_Type__c = 'TIN Update' and corresponding old/new values.

Given both NPI and TIN are updated in the same save operation,
When the trigger fires,
Then two separate PCN records are created — one for each change type — not a single combined record.

Given a PCN is created,
When the Provider Relations queue member opens the PCN record,
Then a chatter post is visible on the PCN record reading: 'PCN auto-generated: [Change_Type__c] for provider [Provider__c.Name]. Review required before next claims adjudication cycle.'

Given an integration user (Workday sync profile) updates the NPI or TIN field,
When the trigger fires,
Then the PCN is created with PCN__c.Change_Source__c = 'Integration' to distinguish from manual updates.

Given no change is made to NPI or TIN (other fields updated only),
When the Provider__c record is saved,
Then no PCN record is created.

## Solution

Implement a Record-Triggered Flow on Provider__c object, triggering on 'A record is updated' with entry conditions: (NPI__c IS CHANGED) OR (TIN__c IS CHANGED). Use two separate decision branches for NPI and TIN changes. Each branch invokes a Create Records element to instantiate Provider_Change_Notification__c with old/new values captured using the {!$Record__Prior} global variable. Change_Source__c logic determined by checking running user profile name against known integration profile. Chatter post created via a Post to Chatter flow action element. Assign OwnerId dynamically by querying the 'Provider Relations Queue' Group Id in a Get Records element at flow start. Unit tested with 95%+ coverage via Apex test class TestProviderPCNFlow. Flow activated and included in Phase 1 unlocked package v1.2.
