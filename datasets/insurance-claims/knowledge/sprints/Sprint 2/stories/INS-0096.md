## Description

**Bug Report**

**Environment:** Acme Insurance Full Sandbox (FS1)
**Discovered During:** Sprint 2 UAT – Policyholder 360 FlexCard testing (INS-0093)

**Steps to Reproduce:**
1. Navigate to any Contact record where Contact.RecordType = 'Policyholder'.
2. Ensure the contact has zero Policy__c records (e.g., a newly migrated policyholder pending policy assignment).
3. Observe the Policyholder 360 FlexCard on the Lightning Record Page.

**Expected Behavior:** FlexCard renders with an empty state message 'No active policies found. Please contact your broker.' in the Policy Summary panel. All other panels (Claims, Payments, EOB) render normally or show their own empty states.

**Actual Behavior:** The entire FlexCard throws a JavaScript error: 'Cannot read properties of undefined (reading 0)' and the full card goes blank. The browser console shows an uncaught OmniStudio DataRaptor response parsing error when the DR_GetActivePolicies DataRaptor returns an empty array.

**Impact:** High – any newly onboarded policyholder or lapsed policyholder with no active policies causes a complete FlexCard failure, blocking Claims Operations staff from viewing any policyholder data.

**Frequency:** Reproducible 100% of the time with zero-policy policyholders. Estimated 340 affected Contact records in production data.

## Acceptance Criteria

as per Description

## Solution

Investigate DR_GetActivePolicies DataRaptor Turbo Extract — add null/empty array guard in the DataRaptor output transform so that an empty result set returns a structured empty object rather than null. Update the Policyholder360Card FlexCard Policy Summary child card to handle empty data state using FlexCard conditional visibility (show empty-state block when data.policies.length === 0). Add similar null guards to all 4 DataRaptors in the FlexCard (DR_GetOpenClaims, DR_GetRecentPayments, DR_GetEOBHistory). Write OmniStudio unit test scenarios for zero-record responses. Regression test all 5 FlexCard panels post-fix in FS1 before promoting to UAT.
