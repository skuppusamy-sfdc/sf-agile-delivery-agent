## Description

As a Claims Operations Manager, I need a 'File New Claim (FNOL)' action button on the Policyholder 360 FlexCard that launches a pre-populated FNOL OmniScript using the policyholder's active policy context so that intake staff can initiate a First Notice of Loss in under 2 minutes without re-entering known policyholder and policy data.

## Acceptance Criteria

Given the Policyholder 360 FlexCard is displayed for a policyholder with at least one active Policy__c,
When the user clicks 'File New Claim (FNOL)',
Then the FNOL OmniScript (OS_FNOL_Intake) must launch in a modal and pre-populate the following fields from the selected Policy__c: Policy__c.Policy_Number__c → FNOL Step 1 'Policy Number', Policy__c.LOB__c → FNOL Step 1 'Line of Business', Policy__c.PolicyHolder__c → FNOL Step 1 'Policyholder Name', Policy__c.Effective_Date__c and Policy__c.Expiration_Date__c → read-only display fields.

Given the FNOL OmniScript is pre-populated,
When the user completes and submits the script,
Then a new Claim__c record must be created with Claim__c.Claim_Status__c = 'FNOL Received', Claim__c.Policy__c = source Policy__c.Id, Claim__c.FNOL_Received_Date__c = TODAY(), and Claim__c.Intake_Channel__c = 'Agent Portal'.

Given the policyholder has multiple active policies,
When the FNOL button is clicked,
Then a policy selector step must appear first, listing all active Policy__c records by Policy__c.Policy_Number__c and Policy__c.Product_Name__c, before proceeding to the intake form.

Given the FNOL OmniScript is submitted successfully,
When the modal closes,
Then the Policyholder 360 FlexCard must automatically refresh and the new Claim__c must appear in the Open Claims panel with status 'FNOL Received'.

Given a user without the 'Claims_FNOL_Create' Permission Set Group,
When the FlexCard loads,
Then the 'File New Claim (FNOL)' button must not be rendered.

## Solution

Add a FlexCard Action of type 'OmniScript' on the Policyholder360Card (INS-0108) that invokes OmniScript 'OS_FNOL_Intake' (Type: FNOL, SubType: Intake). Pass Policy__c context via FlexCard action input parameters using DataRaptor Extract DR_GetActivePoliciesForPolicyholder. Multi-policy selector implemented as OmniScript Step 1 with a DataRaptor Turbo Extract populating a Select component. Claim__c creation handled by Integration Procedure IP_CreateFNOLClaim with a DataRaptor Load. FlexCard refresh post-submission uses the OmniScript 'pubsub' event to trigger a FlexCard data refresh action. Button visibility controlled via FlexCard Conditional Visibility bound to Permission Set Group check via custom $User merge field.
