## Description

**Bug Report — Fee Schedule Panel Load Failure on Provider 360**

**Steps to Reproduce:**
1. Navigate to a Provider__c (Person Account) record for a high-volume specialist (e.g., orthopedic surgeon or multi-specialty group) that has an active PNA with more than 200 associated Fee_Schedule__c records (one per CPT code).
2. Observe the 'Fee Schedules' panel in the Provider 360 page (delivered in INS-0117).
3. Wait for the component to fully render or time out.

**Expected Behavior:**
The Fee Schedules panel should load and display paginated Fee_Schedule__c records with Fee_Schedule__c.CPT_Code__c, Fee_Schedule__c.Allowed_Amount__c, and Fee_Schedule__c.Effective_Date__c visible. Pagination should handle large record sets gracefully.

**Actual Behavior:**
The 'Fee Schedules' panel displays a blank white box with a spinning indicator that never resolves. Browser console shows: 'Apex heap size limit exceeded' and the @wire getRelatedListRecords call returns a 'READ_TOO_MANY_ROWS' error for providers with >200 Fee_Schedule__c CPT entries. Providers with ≤200 records are unaffected.

**Environment:** Full Sandbox (FS1), also reproducible in Partial Sandbox PS1.
**Frequency:** 100% reproducible for providers with >200 Fee_Schedule__c records. Affects approximately 340 provider records in current data set.
**Impact:** Provider Network Specialists cannot review fee schedule data for high-volume providers on the 360 page, forcing them to use report workarounds.

## Acceptance Criteria

As per Description.

Given a Provider__c record has more than 200 Fee_Schedule__c records linked via an active PNA,
When the Fee Schedules panel renders on the Provider 360 page,
Then the panel must load successfully without error and display records in paginated format (25 records per page) with Previous/Next navigation controls.

Given the paginated Fee Schedules panel is loaded,
When a Provider Network Specialist uses the CPT Code search/filter input,
Then the panel must filter displayed rows client-side or via a fresh server-side query without re-triggering the heap overflow.

Given the fix is deployed to FS1,
When the 340 affected provider records are tested,
Then all must load the Fee Schedules panel without 'READ_TOO_MANY_ROWS' or heap errors in debug logs.

## Solution

Root cause: @wire getRelatedListRecords retrieves all Fee_Schedule__c child records in a single call without pagination, causing heap overflow for large datasets. Fix: Replace @wire getRelatedListRecords with an imperative Apex @AuraEnabled(cacheable=true) method 'FeeScheduleService.getPagedFeeSchedules(providerId, pageNumber, pageSize)' that uses SOQL LIMIT/OFFSET pagination (pageSize = 25). LWC 'providerFeeSchedulePanel' refactored to maintain currentPage and totalPages reactive properties, with Previous/Next button handlers calling the imperative method. CPT Code filter uses a debounced input that passes a search string to the Apex method as an additional SOQL WHERE clause (Fee_Schedule__c.CPT_Code__c LIKE :searchTerm%). Apex method enforces a hard LIMIT of 25 and maximum OFFSET of 2000 to stay within governor limits. Add test coverage for boundary cases: 0, 1, 200, 201, and 500 Fee_Schedule__c records.
