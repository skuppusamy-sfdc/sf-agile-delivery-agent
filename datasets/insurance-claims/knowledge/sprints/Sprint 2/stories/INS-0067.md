## Description

**Bug Report — Policyholder 360 Duplicate Policy Records**

**Environment:** Acme Insurance Developer Sandbox (DEV2)
**Discovered by:** Emily Watson during integration testing on 2024-12-04

**Steps to Reproduce:**
1. Log in as a Claims Operations Manager.
2. Navigate to a Person Account (policyholder) record that is linked to BOTH an individual Policy__c record (Policy__c.Policy_Type__c = 'Individual') AND a Group_Service_Agreement__c-linked Policy__c record (Policy__c.Policy_Type__c = 'Group') for the same LOB.
3. Example test record: Contact 'Maria Gonzalez' (Id: 003XX000007DUPTEST) — linked to Policy #IND-2024-0091 and Policy #GRP-2024-0012.
4. Open the Policyholder 360 FlexCard and observe the Active Policies section.

**Expected Result:** Two distinct policy records are shown — one individual, one group — each appearing exactly once.

**Actual Result:** The group policy (Policy #GRP-2024-0012) appears twice in the Active Policies list. The DataRaptor Extract 'DR_GetActivePolicies' appears to be performing a JOIN across Policy__c and Group_Service_Agreement__c that produces a Cartesian product when the policyholder is a member of a GSA with multiple Group_Service_Agreement__c.Member_Count__c entries.

**Expected Result:** Each Policy__c record appears exactly once regardless of GSA membership.

**Actual Result:** Duplicate rows rendered for group policies — one row per GSA member record rather than per policy.

**Frequency:** Reproducible 100% for policyholders with group policy + GSA linkage. Affects approximately 12% of policyholders in the current dataset based on data analysis by Carlos Rodriguez.

## Acceptance Criteria

As per Description. Given the fix is deployed, When a policyholder with both Individual and Group Policy__c records views the Policyholder 360 FlexCard, Then each Policy__c record appears exactly once in the Active Policies section with no duplicate rows, verified against policyholders with 1-to-many GSA relationships.

## Solution

Fix DataRaptor Extract 'DR_GetActivePolicies' SOQL query to remove the erroneous JOIN on Group_Service_Agreement__c member records. Rewrite query to: SELECT Id, Policy_Number__c, LOB__c, Policy_Type__c, Effective_Date__c, Expiration_Date__c, Premium_Amount__c FROM Policy__c WHERE Policyholder__c = :recordId AND Status__c = 'Active' — removing the GROUP BY clause that was referencing GSA member count. Add DISTINCT equivalent via query-level deduplication in DRTransform if GSA metadata is still needed for display. Add DataRaptor unit test covering policyholders with mixed Individual/Group policies. Regression test INS-0061 FlexCard post-fix.
