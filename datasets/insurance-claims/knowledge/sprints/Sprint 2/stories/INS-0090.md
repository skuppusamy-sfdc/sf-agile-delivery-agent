## Description

As a Provider Relations Specialist, I need a guided OmniScript to update a provider's NPI, TIN, and specialty information so that data changes are validated against NPPES and CAQH before being written to the Provider Person Account record, ensuring downstream claims processing accuracy.

Currently, provider data edits are performed directly on the record with no validation layer, leading to NPI format errors and TIN mismatches that cause ERA rejection and claim payment failures. The OmniScript must enforce NPI format validation (10-digit numeric), trigger a real-time NPPES lookup, and create a Provider_Change_Notification__c (PCN) record to log the data change for audit purposes.

## Acceptance Criteria

Given a Provider Relations Specialist launches the 'Update Provider Demographics' OmniScript from the Provider 360 action bar,
When Step 1 loads,
Then the OmniScript pre-populates current values from Provider__c.NPI__c, Provider__c.TIN__c, Provider__c.Primary_Specialty__c using a DataRaptor Load.

Given the specialist enters a new NPI value,
When they attempt to advance to Step 2,
Then the OmniScript validates that the NPI is exactly 10 numeric digits; if invalid, an inline error displays: 'NPI must be a 10-digit numeric value per CMS NPI standards.'

Given a valid NPI is entered,
When the specialist advances to Step 2,
Then the OmniScript calls the NPPES Integration Procedure 'IP_NPPES_NPI_Lookup' and displays the returned provider name, entity type, and enumeration date for confirmation.

Given the NPPES lookup returns a provider name that does NOT match Provider__c.LastName + Provider__c.FirstName (>80% string similarity),
When the result is displayed,
Then a warning banner displays: 'NPPES name does not match Salesforce record. Please verify before saving.' The specialist may proceed but must check an acknowledgment checkbox.

Given all validations pass and the specialist clicks 'Save Changes',
When the OmniScript DataRaptor Transform executes,
Then Provider__c.NPI__c, Provider__c.TIN__c, and Provider__c.Primary_Specialty__c are updated AND a Provider_Change_Notification__c record is created with PCN__c.Change_Type__c = 'Demographic Update', PCN__c.Changed_By__c = running user, PCN__c.Status__c = 'Open', PCN__c.Provider__c = current Account Id.

-Given the TIN is changed, automatically trigger a Fee Schedule re-evaluation- (deprecated — Fee Schedule re-evaluation to be handled by a separate scheduled batch process per architecture review)

Given the NPPES API is unavailable (timeout or HTTP 500),
When the lookup step is reached,
Then the OmniScript displays: 'NPPES verification is temporarily unavailable. You may save with manual acknowledgment.' and logs an error to the Integration_Error_Log__c object.

## Solution

Build OmniScript 'Update_Provider_Demographics_OS' with 3 steps: (1) Pre-populate using DataRaptor Load 'DR_Load_ProviderDemographics'; (2) NPI validation via OmniScript formula element (REGEX check) + Integration Procedure 'IP_NPPES_NPI_Lookup' calling NPPES REST API via Named Credential 'NPPES_API'; (3) Save step using DataRaptor Transform 'DR_Transform_ProviderDemographics' to update Provider__c fields and create PCN__c record in a single transaction. String similarity check implemented as an Apex Action called from OmniScript using Jaro-Winkler algorithm. API failure fallback handled via Integration Procedure error step with conditional branching. OmniScript exposed as a Quick Action on the Provider Person Account page layout.
