## Description

**Bug Report — Duplicate Claim Creation on FNOL OmniScript Back Navigation**

**Environment:** Acme Insurance Full Sandbox (SF-FULL-01)
**Discovered by:** Kevin Nguyen (QA) during Sprint 4 regression
**Related Story:** INS-0268

**Steps to Reproduce:**
1. Log in as a Claims Operations Manager user.
2. Launch the FNOL_Intake OmniScript from the Claims tab.
3. Complete all steps through the final 'Review & Submit' step.
4. Click the 'Back' button to return to the previous step.
5. Re-enter or leave unchanged the data on the previous step.
6. Click 'Next' and then 'Submit' on the Review step.
7. Observe the Claim__c records list view.

**Expected Behavior:** Only one Claim__c record is created per FNOL submission session. If the user navigates back and resubmits, the existing draft record is updated, not duplicated.

**Actual Behavior:** Two Claim__c records are created with identical Policy__c, Date_of_Loss__c, and LOB__c values. Both records have Status__c = 'FNOL Received' and are published as separate FNOL_Submitted__e Platform Events, causing duplicate downstream adjudication queue entries.

**Frequency:** 100% reproducible.
**Impact:** High — duplicate claims enter the adjudication queue, causing double-payment risk and adjuster confusion. Duplicates must currently be manually merged by Claims Operations.

## Acceptance Criteria

As per Description.

Additionally:
Given a Claim__c record has already been created in the current OmniScript session (session token stored),
When the user navigates Back and then resubmits,
Then the DataRaptor Post performs an upsert on the existing Claim__c.Id rather than an insert, and no new record is created.

Given the fix is deployed,
When regression testing runs the back-and-resubmit flow 10 consecutive times,
Then exactly one Claim__c record exists per session and exactly one FNOL_Submitted__e event is published.

## Solution

Root cause: The DataRaptor Post action on the OmniScript final step always performs an INSERT because no record Id is passed on re-submission. Fix: After the first successful Claim__c creation, store the returned Claim__c.Id in an OmniScript session variable (claimId_session). Modify the DataRaptor Post action to check for claimId_session; if populated, set the DRInputMap Id field to trigger an upsert. Additionally, add a duplicate check Apex method 'ClaimDuplicateGuard.findExistingFNOL(policyId, dateOfLoss, lob)' called as a Remote Action before the DataRaptor Post to catch edge cases where session state is lost. Deploy a Duplicate Rule on Claim__c as a secondary safeguard matching on Policy__c + Date_of_Loss__c + LOB__c with Alert action.
