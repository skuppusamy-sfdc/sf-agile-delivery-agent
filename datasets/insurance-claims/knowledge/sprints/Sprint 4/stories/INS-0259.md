## Description

As a Claims Operations Manager, I need the adjudication engine to validate the rendering provider's Provider_Network_Agreement__c (PNA) status at the time of the claim's date of loss so that claims are not incorrectly processed as in-network when the provider's agreement was expired, terminated, or not yet effective on the date of service.

Validation rules:
- PNA.Status__c must be 'Active'
- Claim__c.DateOfLoss__c must fall between PNA.EffectiveDate__c and PNA.TerminationDate__c (inclusive)
- If PNA is expired or terminated: set Claim__c.NetworkStatus__c = 'Out-of-Network' and apply out-of-network benefit rules
- If no PNA exists for the provider: set Claim__c.NetworkStatus__c = 'Non-Participating'
- If a Network_Termination_Case__c exists for the provider with TerminationDate__c <= Claim__c.DateOfLoss__c and Status__c = 'Closed — Terminated': treat as terminated

## Acceptance Criteria

Given a Claim__c record references a Provider__c who has an active PNA where PNA.EffectiveDate__c <= Claim__c.DateOfLoss__c <= PNA.TerminationDate__c,
When the PNA validation service runs,
Then Claim__c.NetworkStatus__c = 'In-Network' and adjudication proceeds with in-network benefit rules.

Given the provider's PNA.TerminationDate__c is prior to Claim__c.DateOfLoss__c,
When the validation service runs,
Then Claim__c.NetworkStatus__c = 'Out-of-Network' and Claim__c.AdjudicationStatus__c = 'Pending — OON Review'.

Given no PNA record exists for the rendering Provider__c,
When the validation service runs,
Then Claim__c.NetworkStatus__c = 'Non-Participating' and a Task is created on the Claim__c assigned to the Provider Relations queue.

Given a Network_Termination_Case__c exists for the provider with TerminationDate__c = 2024-09-30 and Status__c = 'Closed — Terminated', and Claim__c.DateOfLoss__c = 2024-10-05,
When validation runs,
Then Claim__c.NetworkStatus__c = 'Out-of-Network' even if a PNA record with a later date exists.

-Given PNA.ContractType__c = 'TPA Delegated', skip in-network validation and defer to TPA- (removed from Phase 1 scope — TPA delegation logic moved to Phase 2)

Given validation completes for any network status outcome,
When the Claim__c record is updated,
Then an audit record is written to Claim_Validation_Log__c with fields: ClaimId__c, ValidationRule__c = 'PNA_Check', Outcome__c, and RunTimestamp__c.

## Solution

Implement Apex class 'PNAValidationService' invoked from the FNOL_Submitted__e Platform Event subscriber Flow:
- SOQL: query Provider_Network_Agreement__c WHERE Provider__c = :claim.RenderingProvider__c AND Status__c = 'Active' ORDER BY EffectiveDate__c DESC LIMIT 1
- Secondary SOQL: query Network_Termination_Case__c WHERE Provider__c = :providerId AND Status__c = 'Closed — Terminated'
- Apply date comparison logic in Apex
- Update Claim__c.NetworkStatus__c via DML
- Write to Claim_Validation_Log__c via insert DML
- Expose as InvocableMethod for Flow invocation
- Record-Triggered Flow on Claim__c (after update, when RenderingProvider__c changes) also calls this service to handle provider corrections post-FNOL
- Custom Label 'PNA_Validation_OON_Message' used for Task description text to ensure consistency
