## Description

As a Provider Relations Specialist, I need the system to verify the rendering provider's NPI against the CMS NPPES registry when a new Claim__c is submitted so that claims with invalid, deactivated, or mismatched NPI numbers are flagged before adjudication, reducing compliance risk and preventing fraudulent provider billing. Referenced by INS-0265 (network determination).

## Acceptance Criteria

Given a Claim__c record is created with a value in Claim__c.Rendering_Provider_NPI__c,
When the record is saved,
Then an asynchronous callout is made to the NPPES NPI Registry API (https://npiregistry.cms.hhs.gov/api/) using the NPI value as the search parameter.

Given the NPPES API returns a valid, active NPI record,
When the response is processed,
Then Provider__c.NPI_Verified__c = TRUE, Provider__c.NPI_Verification_Date__c = today, and Provider__c.NPI_Status__c = 'Active — Verified'.

Given the NPPES API returns no results for the submitted NPI,
When the response is processed,
Then Claim__c.NPI_Verification_Status__c = 'Invalid NPI', Claim__c.Status__c = 'Pended — NPI Review', and a Task is created and assigned to the 'Provider Data Maintenance' queue with Subject = 'NPI Verification Failed — Manual Review Required'.

Given the NPPES API returns a deactivated NPI (deactivation_date is populated),
When the response is processed,
Then Claim__c.NPI_Verification_Status__c = 'Deactivated NPI', Claim__c.Status__c = 'Pended — NPI Review', and Claim__c.Adjudication_Notes__c is appended with 'NPI deactivated on {deactivation_date}. Claim pended for compliance review.'

Given the NPPES API is unavailable (HTTP timeout or 5xx response),
When the callout fails,
Then the system retries up to 3 times with exponential backoff, and if all retries fail, Claim__c.NPI_Verification_Status__c = 'Verification Pending — API Unavailable' and an alert email is sent to the Claims Integration monitoring DL.

Given NPI verification completes successfully,
When Provider__c.NPI_Status__c = 'Active — Verified',
Then the adjudication engine proceeds without interruption.

## Solution

Implement an Apex Queueable class (NPPESVerificationQueueable) invoked by a Record-Triggered Flow (after insert on Claim__c where Rendering_Provider_NPI__c is not null). The Queueable performs an HTTP callout to the NPPES REST API, parses the JSON response, and updates Provider__c and Claim__c fields via DML. Retry logic implemented using a counter field on a custom NPI_Verification_Request__c tracking object. Named Credential (NPPES_Registry) stores the endpoint URL and manages authentication. Remote Site Setting configured for the NPPES host. Mock HTTP callout classes provided for test coverage. A Platform Event (NPI_Verification_Result__e) publishes the outcome for downstream consumption by the adjudication batch.
