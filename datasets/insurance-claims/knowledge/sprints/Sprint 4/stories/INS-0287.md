## Description

**Bug Report — FNOL OmniScript Null Pointer on Missing GSA**

**Steps to Reproduce:**
1. Log in as a Claims Rep user with PSG_Claims_Intake permission set.
2. Navigate to the Claims console and click 'New FNOL'.
3. In Step 1, search for and select a Policy__c record that has Policy__c.Group_Service_Agreement__c = null (i.e., no GSA linked — common for individual/non-group policies).
4. Click 'Next' to proceed to Step 2.

**Expected Result:**
The OmniScript should proceed to Step 2 and leave the GSA field blank/unpopulated with no error.

**Actual Result:**
The OmniScript throws a generic error: 'An error occurred. Please contact your administrator.' The browser console shows: 'TypeError: Cannot read properties of null (reading Group_Service_Agreement__c)' originating from the DataRaptor Extract DR_GetPolicyByNumber response mapping.

**Environment:** Full Sandbox (ACME-FS1)
**Reproducibility:** 100% when policy has no GSA
**Related Story:** INS-0284

## Acceptance Criteria

As per Description.

Given a Policy__c record with Group_Service_Agreement__c = null is selected in the FNOL OmniScript,
When the DataRaptor Extract DR_GetPolicyByNumber executes,
Then the OmniScript must handle the null GSA gracefully, leave Claim__c.Group_Service_Agreement__c unpopulated, and proceed to Step 2 without error.

## Solution

1. Update DataRaptor Extract (DR_GetPolicyByNumber) output JSON mapping to use a null-safe path expression for Group_Service_Agreement__c (e.g., use IF/ELSE transform node to default to empty string when null).
2. Add a null-check conditional element in the OmniScript Step 2 to only display the GSA field as read-only if the value is non-empty.
3. Add a DataRaptor Transform node between the Extract and the OmniScript state to sanitize all nullable lookup fields (Group_Service_Agreement__c, TPA__c) before they are bound to OmniScript elements.
4. Regression test: execute FNOL flow against 3 policies — one with GSA, one without, one with a deactivated GSA.
