## Description

As a Claims Operations Manager, I need the FNOL OmniScript (INS-0276) extended with LOB-specific sub-steps for Auto and Property claims so that adjusters capture VIN, vehicle damage description, or property address and damage type during initial intake, ensuring complete loss data is available for adjudication without a follow-up call.

This story picks up the VIN capture requirement deprecated from INS-0276 and adds the Property loss sub-flow.

## Acceptance Criteria

Given Claim__c.LOB__c = 'Auto' is selected in Step 1 of the FNOL OmniScript,
When the user reaches the LOB Detail step,
Then fields Claim__c.Vehicle_VIN__c, Claim__c.Vehicle_Make__c, Claim__c.Vehicle_Model__c, and Claim__c.Vehicle_Year__c are displayed and VIN is required.

Given Claim__c.LOB__c = 'Property' is selected,
When the user reaches the LOB Detail step,
Then fields Claim__c.Loss_Location_Address__c, Claim__c.Property_Damage_Type__c (picklist: Fire, Water, Wind, Theft, Other), and Claim__c.Estimated_Damage_Amount__c are displayed.

Given Claim__c.LOB__c = 'Medical',
When the LOB Detail step renders,
Then no vehicle or property fields are displayed and the step is skipped automatically.

Given a VIN is entered for an Auto claim,
When the user tabs out of the VIN field,
Then an Integration Procedure calls the external VIN decode service and auto-populates Claim__c.Vehicle_Make__c, Claim__c.Vehicle_Model__c, and Claim__c.Vehicle_Year__c if a match is found.

Given VIN decode returns no match,
When the user attempts to proceed,
Then a non-blocking warning 'VIN not verified — please confirm vehicle details manually' is displayed and manual entry is permitted.

## Solution

Extend the existing FNOL OmniScript (INS-0276) by adding a conditional Step 3b 'LOB Details' that uses a Conditional View element branching on LOB__c picklist value. Auto sub-step includes a Text Input for VIN wired to an Integration Procedure (IP_VINDecode) that calls an external REST API via Named Credential. IP response maps to OmniScript elements via a DataRaptor Transform. Property sub-step uses standard OmniStudio input elements. Conditional display logic uses OmniScript Show/Hide conditions. The main OmniScript DataRaptor Turbo Action on final submit maps all LOB-specific fields to the Claim__c upsert payload. Unit tests cover all three LOB branching paths.
