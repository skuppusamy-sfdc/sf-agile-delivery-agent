## Description

As a Credentialing Specialist, I need a Credentialing_Case__c to be automatically created whenever a new Provider__c (Person Account) record is created with a valid NPI populated so that the credentialing workflow is initiated without manual intervention and no provider slips through without review.

The NPI must be validated in real time against the NPPES registry before the Credentialing_Case__c is created.

## Acceptance Criteria

Given a new Provider__c (Person Account) record is saved with NPI__c populated,
When the Record-Triggered Flow fires on the After Insert event,
Then an HTTP callout must be made to the NPPES NPI Registry API (https://npiregistry.cms.hhs.gov/api/) using the NPI value.

Given the NPPES API returns a valid result (result_count >= 1),
When the callout response is processed,
Then Provider__c.NPI_Verified__c must be set to true and Provider__c.NPI_Verification_Date__c stamped with the current date.

Given NPI_Verified__c = true,
When the verification completes,
Then a Credentialing_Case__c must be created with Credentialing_Case__c.Provider__c = the new provider, Credentialing_Case__c.Status__c = 'Initiated', Credentialing_Case__c.Source__c = 'NPPES Auto-Registration', and Credentialing_Case__c.Assigned_To__c = the Credentialing queue.

Given the NPPES API returns result_count = 0 or an HTTP error (4xx/5xx),
When the callout fails,
Then Provider__c.NPI_Verified__c must be set to false, Provider__c.NPI_Verification_Status__c = 'Failed — NPPES Lookup', and a Task must be created for the Credentialing queue with Subject = 'Manual NPI Verification Required: [Provider Name]'.

Given an NPI__c value is not exactly 10 digits,
When the record is saved,
Then a validation rule must block save and display: 'NPI must be exactly 10 numeric digits per CMS requirements.'

Given a Provider__c record is updated (not inserted) with a changed NPI__c,
When the flow evaluates,
Then a new NPPES verification must be triggered and a new Credentialing_Case__c created only if no open Credentialing_Case__c already exists for the provider.

## Solution

1. Record-Triggered Flow (RTF_ProviderNPIVerification) on Provider__c (Person Account), After Save, fires on Insert and on Update where NPI__c IS CHANGED.
2. Flow calls Apex invocable method (NPPESVerificationService.callNPPES) which performs the NPPES REST callout asynchronously via a Queueable Apex chain (NPPESCalloutQueueable.cls) to avoid mixed-DML and callout-in-trigger restrictions.
3. Queueable updates Provider__c fields and invokes a second Queueable to create Credentialing_Case__c if verification passes.
4. Named Credential (NPPES_Registry) configured in Setup to store the NPPES endpoint; Remote Site Setting enabled.
5. Custom Metadata (NPPES_Config__mdt) stores API version and timeout threshold (default 10s).
6. Validation Rule on Provider__c: NOT(REGEX(NPI__c, '[0-9]{10}')) with appropriate error message.
7. Apex test uses HttpCalloutMock to simulate NPPES success and failure responses.
