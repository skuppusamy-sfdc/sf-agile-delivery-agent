## Description

As a Claims Operations Manager, I need a guided FNOL intake OmniScript that captures all required first-notice data for Auto and Medical LOBs so that a Claim__c record is created with accurate loss details, linked policy, and claimant information without manual data re-entry.

The OmniScript must support both agent-assisted and self-service channels. It should branch logic based on LOB (Auto vs. Medical) and pre-populate policyholder data from the matched Policy__c record.

## Acceptance Criteria

Given a logged-in Claims Operations user launches the FNOL OmniScript,
When they enter a valid Policy Number,
Then the OmniScript calls a DataRaptor Extract to populate Policy__c.PolicyNumber__c, Policy__c.EffectiveDate__c, Policy__c.ExpirationDate__c, and the associated Provider__c (Person Account) name on-screen without requiring manual lookup.

Given the LOB field is set to 'Medical',
When the user advances past Step 1,
Then the OmniScript displays ICD-10 diagnosis code capture fields and hides the Vehicle Damage section; Claim__c.LOB__c is set to 'Medical'.

Given the LOB field is set to 'Auto',
When the user advances past Step 1,
Then the OmniScript displays Vehicle Damage fields (make, model, VIN) and hides ICD-10 fields; Claim__c.LOB__c is set to 'Auto'.

Given all required fields are completed,
When the user clicks Submit,
Then a Claim__c record is created with Claim__c.Status__c = 'New', Claim__c.DateOfLoss__c populated, Claim__c.ClaimantName__c populated, and Claim__c.Policy__c lookup set to the matched Policy__c record.

Given a duplicate policy number is entered (same loss date, same claimant),
When the OmniScript runs its duplicate-check DataRaptor,
Then a warning banner displays 'Potential duplicate claim detected — Claim ID: [existing Claim__c.Name]' and requires the user to confirm intent to proceed.

-Given submission, auto-assign to available adjuster via round-robin queue- (superseded by INS-0258 assignment logic)

Given the FNOL is submitted successfully,
When the Claim__c record is saved,
Then a Platform Event (FNOL_Submitted__e) is fired containing Claim__c.Id, Claim__c.LOB__c, and Policy__c.Id for downstream processing.

## Solution

Build a multi-step OmniStudio OmniScript ('FNOL_Intake_OS') with the following structure:
- Step 1: Policy lookup via DataRaptor Extract querying Policy__c by PolicyNumber__c
- Step 2: LOB-branched conditional blocks using OmniScript Show/Hide conditions
- Step 3: Claimant & loss detail capture
- Step 4: Review & Submit

On submit, use an Integration Procedure ('IP_CreateFNOLClaim') that orchestrates:
1. DataRaptor Transform to map OmniScript JSON to Claim__c fields
2. Apex callout to duplicate-check service
3. Platform Event publish (FNOL_Submitted__e) via Apex action

Deploy OmniScript as a Lightning Web Component embed on the Claims Console App home page. Assign access via 'Claims_Intake_User' Permission Set Group.
