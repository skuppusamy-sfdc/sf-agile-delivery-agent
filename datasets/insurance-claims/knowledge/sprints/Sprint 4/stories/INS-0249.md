## Description

**Bug Report — FNOL OmniScript Policy Pre-Population Failure**

**Environment:** Acme Insurance Full Sandbox (FS1)
**Build:** Sprint 4 Day 3 deployment
**Related Story:** INS-0245

**Steps to Reproduce:**
1. Log in as a Claims Adjuster user profile.
2. Navigate to the FNOL Intake OmniScript (Type: FNOL_Intake, Sub-Type: Auto_Health).
3. On Step 1 (Policy Lookup), enter a Policy Number in the format 'POL-2024-0087432' (containing hyphens).
4. Click the 'Look Up Policy' button to trigger the DataRaptor Extract.
5. Observe the policy data fields on the OmniScript form.

**Expected Behavior:**
Policy__c.PolicyHolder__c, Policy__c.Effective_Date__c, Policy__c.Expiration_Date__c, and Policy__c.Coverage_Type__c should pre-populate in the read-only fields as defined in INS-0245 AC2.

**Actual Behavior:**
All pre-population fields remain blank. No error message is displayed to the user. The DataRaptor Extract silently returns zero records. The issue does not occur when a policy number without hyphens (e.g., 'POL20240087432') is entered manually.

**Root Cause Hypothesis:**
The DataRaptor Extract filter on Policy__c.Policy_Number__c appears to apply URL encoding or escaping to the hyphen character ('-'), causing the SOQL WHERE clause to fail the exact-match lookup. Suspect the OmniScript input element's 'Trim Input' or 'Sanitize' property is stripping or encoding the hyphen before passing it to the DataRaptor.

**Impact:** High — the majority of legacy policy numbers in the migrated dataset use the hyphenated format 'POL-YYYY-XXXXXXX'. Adjusters are unable to pre-populate policy data for approximately 94% of existing policyholders.

## Acceptance Criteria

As per Description. Verify fix against Policy__c records with Policy_Number__c values in all three known formats: 'POL-YYYY-XXXXXXX', 'POLYYYY-XXXXXXX', and 'POLXXXXXXXXX'. Regression test full FNOL submission flow post-fix.

## Solution

Investigate the DataRaptor Extract (DR: Get_Policy_By_Number) input parameter mapping for the Policy_Number__c filter. Disable any input sanitization on the OmniScript Text Input element for the Policy Number field. If the DataRaptor is applying LIKE instead of exact-match, update the filter operator to '=' and ensure the input value is passed as a raw string without encoding. Add a Debug step in the Integration Procedure to log the raw input value and DataRaptor response for QA validation. Regression test in FS1 before promoting to UAT.
