## Description

**Bug Report — FNOL OmniScript Duplicate Claim Creation**

**Environment:** Full Sandbox (Acme-FS1)
**Build:** Sprint 4 Day 3 deployment

**Steps to Reproduce:**
1. Log in as a Claims Operations Manager user.
2. Open the FNOL OmniScript from the Claims App.
3. Complete Steps 1–3 (Policy lookup, Incident Details, LOB Details).
4. On Step 4 (Summary), click the 'Back' button to return to Step 3.
5. Make no changes on Step 3 and click 'Next' to return to Step 4.
6. Click 'Submit FNOL'.

**Expected Result:** A single Claim__c record is created with Status__c = 'FNOL Received'.

**Actual Result:** Two Claim__c records are created with identical Policy__c lookup, Date_of_Loss__c, and Claimant_Name__c values. Both records have Status__c = 'FNOL Received' and fire duplicate FNOL_Submitted__e Platform Events, triggering duplicate adjuster assignment tasks.

**Impact:** High — data integrity issue causing duplicate claims in production-like environment. SIU team flagged duplicate records as potential fraud indicators, generating false positive investigations.

**Frequency:** Reproducible 100% of the time when Back button is used at least once before final submit.

## Acceptance Criteria

As per Description. 

Additional verification: After fix deployment, execute the reproduction steps 10 consecutive times and confirm only one Claim__c record is created per FNOL submission. Verify Claim__c.External_ID__c (OmniScript Session Token) is populated and used as the upsert key in the DataRaptor Turbo Action to prevent duplicate inserts.

## Solution

Root cause: the DataRaptor Turbo Action on the OmniScript 'Submit' step performs an INSERT operation each time the final step is rendered, including on Back/Next navigation returns. Fix: (1) Generate a unique OmniScript Session Token (UUID) in Step 1 using a Set Values element and store it in Claim__c.External_ID__c. (2) Change the DataRaptor Turbo Action DML operation from INSERT to UPSERT using External_ID__c as the External ID field. (3) Add a conditional check in the Integration Procedure: if a Claim__c with the same External_ID__c already exists, skip Platform Event publication to prevent duplicate FNOL_Submitted__e events. Deploy fix to sandbox, execute regression test suite (10 iterations of reproduction steps), then promote to production via change set.
