## Description

As a Claims Operations Manager, I need a guided FNOL intake OmniScript that captures initial loss details for Auto and Medical LOBs so that a Claim__c record is created with all mandatory fields populated and routed to the correct adjudication queue without manual data entry errors.

## Acceptance Criteria

Given a logged-in Claims Operations user opens the FNOL OmniScript,
When they search by Policy__c.Policy_Number__c,
Then the script auto-populates Policy__c.Policyholder__c, Policy__c.Effective_Date__c, Policy__c.Expiration_Date__c, and Policy__c.LOB__c on the intake form.

Given the policy is active (Policy__c.Status__c = 'Active'),
When the user completes all required FNOL fields (Date_of_Loss__c, Loss_Description__c, Loss_Type__c, Reported_By__c),
Then a new Claim__c record is created with Claim__c.Status__c = 'FNOL Received' and Claim__c.FNOL_Received_Date__c stamped to the current datetime.

Given the LOB is 'Medical',
When the FNOL is submitted,
Then Claim__c.Assigned_Queue__c is set to 'Medical Claims Intake' and a Task is created for the assigned adjuster with Subject = 'Review New FNOL — Medical'.

Given the LOB is 'Auto',
When the FNOL is submitted,
Then Claim__c.Assigned_Queue__c is set to 'Auto Claims Intake' and Claim__c.Estimated_Loss_Amount__c field is visible and required.

-Given the policy is expired, When FNOL is submitted, Then auto-create a retroactive coverage review case- (superseded by INS-0248 retroactive flow).

Given any required field is left blank,
When the user attempts to advance to the next OmniScript step,
Then an inline validation error displays the field label and the script does not advance.

Given the FNOL is successfully submitted,
When the record is created,
Then a Confirmation_Number__c is generated using a custom auto-number format 'FNOL-{YYYY}-{0000000}' and displayed on the final OmniScript confirmation screen.

## Solution

Build a multi-step OmniScript (Type: FNOL, Sub-Type: Intake) using OmniStudio. Step 1 uses a DataRaptor Extract to query Policy__c by Policy_Number__c and populate form fields. Step 2 branches via a Conditional View based on Policy__c.LOB__c value. Step 3 uses a DataRaptor Transform to map OmniScript data to Claim__c fields. On submit, an Integration Procedure calls a DataRaptor Post to insert the Claim__c record and fires a Platform Event (FNOL_Submitted__e) consumed by a Record-Triggered Flow that assigns the queue and creates the follow-up Task. Auto-number format configured on Claim__c.Confirmation_Number__c field.
