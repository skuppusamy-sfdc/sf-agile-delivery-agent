## Description

**Bug Report — Claim Header Status Not Cascading from Pended Claim Lines**

**Environment:** Acme Insurance Full Sandbox (FS1)
**Build:** Sprint 4 Day 5 deployment
**Related Stories:** INS-0247, INS-0250

**Steps to Reproduce:**
1. Log in as a Claims Manager user (Claims_Manager_PSG).
2. Open an existing Claim__c record with Claim__c.Status__c = 'Under Review' and Claim__c.Eligibility_Validated__c = true.
3. Add two Claim_Line__c records, each with a CPT_Code__c value that does not exist in the active Fee_Schedule__c (e.g., CPT '99ZZZ' and '88YYY').
4. Save both Claim_Line__c records.
5. Observe that both Claim_Line__c records correctly show Status__c = 'Pended — Invalid CPT' as per INS-0250 AC4.
6. Navigate back to the parent Claim__c record header and observe Claim__c.Status__c.

**Expected Behavior:**
When all child Claim_Line__c records are in a pended state (Status__c = 'Pended — Invalid CPT'), the parent Claim__c.Status__c should automatically update to 'Pended — Line Review' to reflect that the claim cannot proceed to payment.

**Actual Behavior:**
Claim__c.Status__c remains 'Under Review' regardless of all child lines being pended. The Claim__c.All_Lines_Validated__c checkbox also remains false (correct) but no status cascade logic fires. This causes the claim to appear eligible for payment processing in queue reports, creating a risk of premature payment disbursement.

**Root Cause Hypothesis:**
The Record-Triggered Flow on Claim_Line__c (after update) described in INS-0250 solution appears to only handle the 'all validated' roll-up scenario (setting All_Lines_Validated__c = true). There is no corresponding logic path that evaluates whether all lines are in a pended state and cascades a 'Pended' status to the parent Claim__c header.

**Impact:** High — claims with fully pended line items are incorrectly surfacing in the 'Ready for Payment' work queue, risking erroneous Payment__c record creation.

## Acceptance Criteria

As per Description. Additionally verify: (1) A Claim__c with a mix of 'Validated' and 'Pended — Invalid CPT' lines does NOT trigger the parent status cascade — only when ALL lines are pended should the parent update. (2) When a previously pended Claim_Line__c is corrected and set to 'Validated', and all other lines are also 'Validated', Claim__c.Status__c should revert to 'Under Review' and Claim__c.All_Lines_Validated__c should be set to true. (3) Regression test INS-0250 full AC suite after fix is applied.

## Solution

Update the existing Record-Triggered Flow (RTF: ClaimLine_StatusRollup_Flow) on Claim_Line__c (after insert/update) to add a second decision branch evaluating whether COUNT of related Claim_Line__c records WHERE Status__c NOT IN ('Pended — Invalid CPT', 'Pended — Coverage Mismatch') = 0 (i.e., all lines are pended). If true, update parent Claim__c.Status__c = 'Pended — Line Review' via a related record update flow element. Add a third branch for the mixed-state scenario to ensure no erroneous cascade occurs. Consider consolidating roll-up logic into ClaimLineTrigger / FeeScheduleValidationService.cls (see INS-0250) to avoid split logic between Apex and Flow — confirm approach with tech lead before implementation.
