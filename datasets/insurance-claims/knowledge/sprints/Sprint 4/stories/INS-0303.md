## Description

**Bug Summary:** When a user navigates backward from Step 3 (Loss Details) to Step 1 (Policy Search) in the FNOL OmniScript (INS-0300) and re-enters a different policy number, the OmniScript JSON state retains the original Policy__c.Id from the first lookup. The Claim__c record created on final submission is incorrectly associated with the first policy, not the updated one.

**Environment:** Full Sandbox (CS47)
**Build:** OmniScript FNOL_Intake v1.3, deployed 2024-11-14

**Steps to Reproduce:**
1. Open the FNOL OmniScript from the Claims tab.
2. On Step 1, enter Policy Number 'POL-00012345' and click 'Search'. Confirm policy details populate.
3. Advance to Step 2 (Claimant Info) and Step 3 (Loss Details).
4. Click the 'Back' button twice to return to Step 1.
5. Clear the policy number field and enter a different policy number, e.g., 'POL-00067890'. Click 'Search'.
6. Advance through all steps and click 'Submit FNOL'.
7. Open the newly created Claim__c record and inspect the Claim__c.Policy__c lookup field.

**Expected Result:** Claim__c.Policy__c should reference the Policy__c record for 'POL-00067890'.

**Actual Result:** Claim__c.Policy__c references the Policy__c record for 'POL-00012345' (the original lookup). The FNOL_Submitted__e Platform Event also carries the incorrect Policy__c.Id.

**Impact:** High — Claims are being filed against incorrect policies. Two confirmed production-equivalent incidents found in regression testing.

## Acceptance Criteria

As per Description.

Additionally: Given the user re-searches a policy on Step 1 after backward navigation, When the new policy search DataRaptor fires, Then all OmniScript JSON nodes scoped to Policy__c (PolicyId, PolicyNumber, PolicyStatus, LOB, EffectiveDate, ExpirationDate) must be cleared and re-populated with values from the new policy lookup result before the user can advance to Step 2.

## Solution

Root cause: The DataRaptor Extract on Step 1 appends to the OmniScript JSON document rather than overwriting existing policy-related nodes when the step is revisited. Fix by adding an OmniScript Set Values element at the beginning of the Step 1 'Search' action to explicitly null out all Policy__c-related JSON nodes (PolicyId, PolicyNumber, PolicyHolderId, LOBValue, EffectiveDateValue, ExpirationDateValue) before the DataRaptor Extract fires. Additionally, add an OmniScript Conditional View on the 'Next' button of Step 1 that validates PolicyId is not null and matches the most recent search result. Re-deploy OmniScript FNOL_Intake v1.4 to sandbox and run regression suite covering back-navigation scenarios.
