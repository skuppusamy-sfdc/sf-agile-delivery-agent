## Description

**Bug Report — Duplicate Claim Creation via FNOL OmniScript**

**Environment:** Acme Insurance Full Sandbox (SF-FULL-01)
**Related Story:** INS-0237

**Steps to Reproduce:**
1. Open an active Policy__c record (e.g., Policy Number: POL-2024-00412, LOB: Medical).
2. Launch the FNOL OmniScript from the 'File a Claim' quick action.
3. Complete all steps through Step 4 (Incident Details).
4. On Step 5 (Review & Submit), click the browser Back button or the OmniScript's built-in 'Previous' chevron.
5. Re-enter Step 5 and click 'Submit'.
6. Observe Claim__c records created.

**Expected Behavior:** Only one Claim__c record is created per FNOL submission session, regardless of navigation within the OmniScript.

**Actual Behavior:** Two Claim__c records are created with identical Policy__c, Date_of_Service__c, and ICD-10 codes but different auto-generated Claim_Number__c values (e.g., CLM-2024-00891 and CLM-2024-00892). Both records have Status__c = 'FNOL Received'.

**Impact:** HIGH — Duplicate claims inflate claim counts, trigger duplicate adjudication runs, and create erroneous Payment__c obligations. Identified in UAT by Claims Operations Manager persona testing.

**Frequency:** Reproducible 100% of the time when the back-navigation path is followed.

## Acceptance Criteria

As per Description. Verified fix must demonstrate that navigating back and resubmitting within the same OmniScript session results in exactly one Claim__c record. Regression test must cover: (a) normal submission path, (b) back-then-submit path, (c) page refresh mid-script.

## Solution

Root cause: the DataRaptor Post on Step 5 fires on each 'Submit' action without idempotency check. Fix: introduce a session token (SetValues: FNOL_Session_ID__c = {OmniScript GUID}) stored on the Claim__c record on first creation. Before the DataRaptor Post executes, add a DataRaptor Extract lookup to check for an existing Claim__c with matching Policy__c + FNOL_Session_ID__c; if found, skip creation and route to confirmation step. Additionally, disable the OmniScript 'Previous' button on the final Review step via Step Condition to prevent re-navigation post-draft save. Deploy fix to sandbox for UAT re-validation before promoting to production.
