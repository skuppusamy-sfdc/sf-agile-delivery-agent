## Description

**Bug Report — Fee Schedule Duplicate Record Returns Wrong Allowed Amount**

**Environment:** Acme Insurance Full Sandbox (CS47)
**Build:** Sprint 4 deployment — 2024-11-14
**Related Story:** INS-0294 (Basic Adjudication Rule Engine)

**Steps to Reproduce:**
1. Ensure two Fee_Schedule__c records exist for the same CPT_Code__c (e.g., '99213') and the same LOB__c (e.g., 'Medical'), both with Status__c = 'Active', but with overlapping Effective_Date__c ranges (e.g., one effective 2024-01-01 and one effective 2024-06-01, both without a termination date).
2. Submit a FNOL claim via the OmniScript (INS-0292) for LOB = 'Medical' with a Claim_Line__c containing CPT_Code__c = '99213' and Date_of_Service__c = 2024-09-15.
3. Allow the AdjudicationEngine.cls to process the claim.
4. Observe Claim_Line__c.Allowed_Amount__c after adjudication.

**Expected Behavior:** The adjudication engine should select the Fee_Schedule__c record with the most recent Effective_Date__c that is ≤ Date_of_Service__c (i.e., the 2024-06-01 record) and apply its Allowed_Amount__c.

**Actual Behavior:** The engine applies the Allowed_Amount__c from the 2024-01-01 Fee_Schedule__c record (the older record), resulting in a lower reimbursement amount. Root cause: the SOQL query in AdjudicationEngine.cls uses 'LIMIT 1' without an ORDER BY clause, causing non-deterministic record selection.

**Financial Impact:** Underpayments to providers for affected CPT codes. At least 3 Claim__c records in sandbox have been identified with incorrect Allowed_Amount__c values.
**Severity:** High — financial accuracy defect.

## Acceptance Criteria

As per Description.

Given multiple Fee_Schedule__c records exist for the same CPT_Code__c and LOB__c with overlapping effective dates,
When the AdjudicationEngine processes a Claim_Line__c,
Then the SOQL query selects the Fee_Schedule__c record with the greatest Effective_Date__c that is less than or equal to Claim_Line__c.Date_of_Service__c, using ORDER BY Effective_Date__c DESC LIMIT 1.

Given the fix is deployed,
When a Claim_Line__c with CPT_Code__c = '99213' and Date_of_Service__c = 2024-09-15 is adjudicated,
Then Claim_Line__c.Allowed_Amount__c reflects the value from the Fee_Schedule__c record effective 2024-06-01.

Given the fix is deployed,
When no Fee_Schedule__c record exists for a given CPT/LOB combination,
Then Claim__c.Status__c = 'Pended' and Claim__c.Pend_Reason__c = 'Fee Schedule Not Found' rather than throwing an unhandled exception.

## Solution

Update the SOQL query in AdjudicationEngine.cls fee schedule lookup from:
SELECT Id, Allowed_Amount__c FROM Fee_Schedule__c WHERE CPT_Code__c = :cptCode AND LOB__c = :lob AND Status__c = 'Active' LIMIT 1
to:
SELECT Id, Allowed_Amount__c FROM Fee_Schedule__c WHERE CPT_Code__c = :cptCode AND LOB__c = :lob AND Status__c = 'Active' AND Effective_Date__c <= :dateOfService ORDER BY Effective_Date__c DESC LIMIT 1
Add a null-check on the query result and set Claim__c.Pend_Reason__c = 'Fee Schedule Not Found' if no record is returned. Add a Validation Rule on Fee_Schedule__c to warn (via a duplicate rule) when a new Active record is inserted for a CPT/LOB combination that already has an Active record with an overlapping date range. Write regression unit tests covering: single active record, multiple active records (assert newest selected), and no matching record (assert pend). Backfill the 3 identified sandbox Claim__c records via an anonymous Apex data fix script.
