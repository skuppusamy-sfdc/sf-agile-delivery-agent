## Description

As a Claims Operations Manager, I need a guided FNOL intake OmniScript that captures all required first-notice-of-loss data for Auto and Health LOB claims so that claims are created with complete, validated information and routed correctly for adjudication without manual re-entry.

## Acceptance Criteria

Given a logged-in Claims Operations user navigates to the FNOL intake screen,
When they select LOB = 'Auto' or LOB = 'Health',
Then the OmniScript dynamically renders LOB-specific input steps and suppresses irrelevant fields.

Given the user enters a Policy Number,
When the DataRaptor lookup executes against Policy__c,
Then Policy__c.Status__c must equal 'Active' and Policy__c.Effective_Date__c must be on or before the Date_of_Loss__c; if validation fails, an inline error message is displayed and submission is blocked.

Given all required FNOL fields are completed,
When the user clicks 'Submit Claim',
Then a new Claim__c record is created with Claim__c.Status__c = 'FNOL Received', Claim__c.LOB__c populated, Claim__c.Date_of_Loss__c set, and Claim__c.Policy__c lookup populated.

Given the Claim__c record is created,
When the record-triggered Flow on Claim__c fires (after insert),
Then a Platform Event (Claim_Created__e) is published and the claim is auto-assigned to the appropriate Claims queue based on Claim__c.LOB__c.

Given the claim is of LOB = 'Health',
When the OmniScript step for Provider information is reached,
Then the user must supply a valid NPI; the script validates against the Provider__c (Person Account) object using Provider__c.NPI__c and blocks submission if no matching active provider is found.

-Given LOB = 'Workers Comp', the OmniScript renders a sub-step for employer data capture.- (deprecated — Workers Comp FNOL deferred to Phase 2, see INS-0198)

Given the claim is successfully submitted,
When the confirmation step renders,
Then the generated Claim__c.Claim_Number__c (auto-number) is displayed to the user and a confirmation email is sent via an Apex-invocable email action.

## Solution

Build a multi-step OmniScript (Type: FNOL_Intake, Sub-Type: Auto_Health) in OmniStudio. Use DataRaptor Transform to pre-populate policyholder data from Policy__c and related Contact. Conditional rendering via OmniScript conditional views driven by LOB selection. On submit, use an Integration Procedure to (1) create Claim__c via DataRaptor Load, (2) publish Claim_Created__e Platform Event. A separate Record-Triggered Flow on Claim__c [After Insert] consumes the event context and assigns to queue via Assignment Rules. NPI validation calls an Apex REST callout to NPPES integration service. Deploy OmniScript as an Aura-wrapped LWC tab in the Claims Console app.
