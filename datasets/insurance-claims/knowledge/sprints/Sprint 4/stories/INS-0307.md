## Description

As a Claims Operations Manager, I need a guided FNOL intake OmniScript that captures initial loss details for Auto and Medical LOBs so that a Claim__c record is created with all required fields populated, reducing manual data entry errors and accelerating the claim intake SLA from 48 hours to under 4 hours.

This story extends INS-0277 (Policy lookup integration) by consuming the Policy__c lookup DataRaptor built in Sprint 3 and pre-populating policyholder details on the FNOL form.

## Acceptance Criteria

Given a logged-in Claims Operations user launches the FNOL OmniScript from the Policy__c record,
When the user completes Step 1 (Loss Information),
Then the following fields must be captured and validated before proceeding:
  - Claim__c.Date_of_Loss__c (required, cannot be future date)
  - Claim__c.Loss_Description__c (required, min 20 characters)
  - Claim__c.LOB__c (picklist: Auto | Medical | Property — Auto and Medical only in Phase 1)
  - Claim__c.FNOL_Channel__c (picklist: Phone | Web | Mobile | TPA)

Given the user selects LOB__c = 'Medical',
When Step 2 (Provider Information) is rendered,
Then the OmniScript must perform a real-time NPI lookup via the NPPES DataRaptor integration and auto-populate:
  - Claim__c.Billing_Provider__c (lookup to Provider__c Person Account)
  - Claim__c.Facility__c (lookup to Facility__c Business Account)
  - Claim__c.Rendering_NPI__c

Given the user completes all required steps,
When the OmniScript is submitted,
Then a Claim__c record is created with:
  - Claim__c.Status__c = 'FNOL Received'
  - Claim__c.Intake_Timestamp__c = system datetime of submission
  - Claim__c.Assigned_Queue__c = 'Medical Claims Intake' or 'Auto Claims Intake' based on LOB__c
  - Claim__c.Policy__c populated from the originating Policy__c record

Given the Policy__c.Status__c = 'Lapsed' or 'Cancelled' at time of FNOL submission,
When the OmniScript performs the policy validation step,
Then the user must be presented with a warning banner: 'Policy is not active — claim will be flagged for review' and Claim__c.Requires_Coverage_Review__c must be set to TRUE.

-Given the user selects FNOL_Channel__c = 'TPA', the TPA code field is optional-
Given the user selects FNOL_Channel__c = 'TPA',
When the TPA information step is rendered,
Then Claim__c.TPA_Reference_Number__c is required and must be alphanumeric, 6–20 characters.

Given the Claim__c record is successfully created,
When the OmniScript completion screen is shown,
Then the user sees the auto-generated Claim__c.Claim_Number__c (format: CLM-{YYYY}-{00000}) and a 'Confirmation Sent' indicator confirming that a Case confirmation email was dispatched to the policyholder email on Policy__c.Policyholder_Email__c.

## Solution

Implement a 6-step OmniScript (Type: FNOL, Sub-Type: ClaimsIntake) in OmniStudio. Use DataRaptors for: (1) Policy lookup by policy number against Policy__c, (2) NPPES NPI REST callout wrapped in an Integration Procedure (IP: NPPES_NPI_Lookup) to hydrate Provider__c and Facility__c fields. Step branching controlled by LOB__c selection using OmniScript Conditional Views. On submit, invoke Integration Procedure IP_FNOL_CreateClaim which calls an Apex invocable method (ClaimIntakeService.createFNOLClaim()) to create Claim__c with all mapped fields, assign to the appropriate Salesforce Queue via Group lookup, and fire a Platform Event (FNOL_Submitted__e) consumed by a downstream Record-Triggered Flow that sends the policyholder confirmation email via an Email Alert. Claim_Number__c auto-number format configured on the Claim__c object. OmniScript deployed as an LWC embedded on the Policy__c Lightning Record Page via a custom tab.
