## Description

As a Claims Operations Manager, I need the system to automatically match each Claim_Line__c record's CPT code against the applicable Fee_Schedule__c entry so that the AllowedAmount__c is calculated and populated without manual adjuster intervention, enabling straight-through adjudication for in-network medical claims.

Fee schedule matching logic:
1. Match on Fee_Schedule__c.CPTCode__c = Claim_Line__c.CPTCode__c
2. Within matched CPT codes, filter by Fee_Schedule__c.EffectiveDate__c <= Claim__c.DateOfLoss__c <= Fee_Schedule__c.ExpirationDate__c
3. Further filter by Fee_Schedule__c.LOB__c = Claim__c.LOB__c
4. If the rendering provider has an active Provider_Network_Agreement__c (PNA), apply PNA.DiscountRate__c to the schedule rate
5. If no matching Fee_Schedule__c record is found, set Claim_Line__c.AllowedAmount__c = NULL and Claim_Line__c.AdjudicationStatus__c = 'Pending Manual Review'

Extends base adjudication framework established in INS-0223.

## Acceptance Criteria

Given a Claim_Line__c record is created with CPTCode__c = '99213' and parent Claim__c.DateOfLoss__c = 2024-10-15 and LOB__c = 'Medical',
When the fee schedule matching Apex service runs,
Then the system finds the Fee_Schedule__c record where CPTCode__c = '99213', EffectiveDate__c <= 2024-10-15, ExpirationDate__c >= 2024-10-15, and LOB__c = 'Medical', and sets Claim_Line__c.ScheduledRate__c to the matched Fee_Schedule__c.Rate__c value.

Given the rendering provider has an active PNA with PNA.DiscountRate__c = 15%,
When the allowed amount is calculated,
Then Claim_Line__c.AllowedAmount__c = ScheduledRate__c * (1 - 0.15) and Claim_Line__c.AdjudicationStatus__c = 'Auto-Adjudicated'.

Given the rendering provider does NOT have an active PNA,
When the allowed amount is calculated,
Then Claim_Line__c.AllowedAmount__c = ScheduledRate__c (no discount applied) and Claim_Line__c.NetworkStatus__c = 'Out-of-Network'.

Given no Fee_Schedule__c record matches the CPT code and date range,
When the matching service completes,
Then Claim_Line__c.AllowedAmount__c = NULL, Claim_Line__c.AdjudicationStatus__c = 'Pending Manual Review', and a Task is created on the parent Claim__c assigned to the Claims Operations queue.

Given multiple Fee_Schedule__c records match the same CPT code and date range (data conflict),
When the matching service runs,
Then the record with the most recent Fee_Schedule__c.LastModifiedDate is selected and Claim_Line__c.FeeScheduleSource__c is populated with that record's Id.

Given a batch of 500 Claim_Line__c records is processed,
When the Apex batch job runs,
Then all 500 records are processed within governor limits (no CPU timeout, no SOQL limit breach) and results are logged to a Custom Metadata audit object.

## Solution

Implement Apex class 'FeeScheduleMatchingService' with the following design:
- Public static method: matchClaimLines(List<Claim_Line__c> lines)
- Bulk-safe: collect all CPT codes and loss dates from input list, execute single SOQL query against Fee_Schedule__c with IN filter on CPT codes
- Apply date range filter and LOB filter in-memory to avoid additional SOQL queries
- Secondary SOQL: query Provider_Network_Agreement__c WHERE Provider__c IN (set of rendering provider Ids) AND Status__c = 'Active'
- Calculate AllowedAmount__c using discount logic in-memory
- Perform bulk DML update on Claim_Line__c list
- Invoke from Record-Triggered Flow ('RTF_ClaimLine_FeeMatch') on Claim_Line__c after insert, calling Apex action
- Also expose as Apex Batch ('FeeScheduleMatchBatch') for retroactive processing
- Custom Metadata Type 'Fee_Schedule_Config__mdt' stores tie-breaking rules and fallback behavior flags
- Deploy via unlocked package 'AcmeFeeScheduleEngine_v1.0'
