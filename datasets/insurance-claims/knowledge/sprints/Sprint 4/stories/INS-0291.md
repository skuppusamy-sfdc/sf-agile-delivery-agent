## Description

**Bug Report — Duplicate Claim_Line__c on FNOL Multi-Submit**

**Steps to Reproduce:**
1. Log in as Claims Rep with PSG_Claims_Intake permission set.
2. Launch the FNOL Intake OmniScript (INS-0284).
3. Complete all steps through the Claim Lines entry step, adding 2 Claim_Line__c entries.
4. Reach the Review & Submit step and click 'Submit'.
5. Observe the confirmation screen briefly loads.
6. Click the browser back button or the OmniScript's built-in 'Back' navigation button.
7. Click 'Submit' a second time without modifying any data.

**Expected Result:**
The second submission should be idempotent — either updating the existing Claim__c record or presenting an error message indicating the claim was already submitted, with no additional Claim_Line__c records created.

**Actual Result:**
A second set of Claim_Line__c records (duplicates of the original 2) is created and associated with the same Claim__c record, resulting in 4 Claim_Line__c children. Claim__c.Total_Billed_Amount__c is doubled as a result, causing downstream adjudication to incorrectly route the claim to 'Pending Manual Review' due to the inflated total.

**Environment:** Full Sandbox (ACME-FS1) and Developer Sandbox (ACME-DEV2)
**Reproducibility:** ~80% reproducible; timing-dependent
**Severity:** High — data integrity impact on adjudication
**Related Stories:** INS-0284, INS-0285

## Acceptance Criteria

As per Description.

Given a Claim__c record has already been created with Status__c = 'FNOL Received' for a given session,
When the FNOL OmniScript is re-submitted with the same Claim_Number__c (or session token),
Then no additional Claim_Line__c records must be created, and the existing Claim__c record must remain unchanged.

Given the duplicate submission scenario,
When the second submit is attempted,
Then the OmniScript must display an informational message: 'This claim has already been submitted. Claim Number: [Claim_Number__c]' and disable the Submit button.

## Solution

1. Add an idempotency check in the DataRaptor Post (DR_PostFNOLClaim): before inserting Claim_Line__c records, query for existing children using Claim__c Id stored in OmniScript state; skip insert if count > 0.
2. Introduce a hidden OmniScript element (Submitted_Flag) set to 'true' after first successful submission; add a conditional step at the start of the Submit action that halts processing if Submitted_Flag = 'true'.
3. Disable the OmniScript 'Back' button on the Review & Submit step using the OmniScript Step property 'Hide Back Button' = true.
4. Implement an External ID upsert (Claim_Number__c as External ID) in the DataRaptor Post for Claim__c so that re-submission updates rather than inserts the parent record.
5. Add a Duplicate Rule on Claim_Line__c matching on Claim__c + CPT_Code__c + Service_Date__c with Block action to serve as a safety net at the data layer.
6. Regression test script to cover single submit, back-and-resubmit, and browser-refresh scenarios.
