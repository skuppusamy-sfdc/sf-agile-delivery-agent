## Description

**Bug Report — FNOL OmniScript Auto-Population Failure**

**Steps to Reproduce:**
1. Navigate to the FNOL Intake OmniScript (INS-0260).
2. In the Policy Number search field, enter a policy number containing hyphens, e.g., 'ACM-2024-00445-A'.
3. Click the 'Find Policy' action button.
4. Observe the policy lookup result — the record is found and displayed in the dropdown.
5. Select the policy from the dropdown.

**Expected Result:** Policy__c.Policyholder__c, Policy__c.Effective_Date__c, Policy__c.Expiration_Date__c, and Policy__c.LOB__c auto-populate in the OmniScript form fields as designed in INS-0260.

**Actual Result:** All auto-populate fields remain blank. The policy record is selected (highlighted in the dropdown) but no DataRaptor Extract response values are mapped to the form. No error message is displayed to the user.

**Environment:** Full Sandbox (ACME-FS1)
**Frequency:** 100% reproducible with hyphenated policy numbers; policy numbers without hyphens work correctly.
**Impact:** Claims staff cannot complete FNOL intake for any policy with a hyphenated policy number, which represents approximately 68% of the active policy book.

## Acceptance Criteria

As per Description. Verify fix against policy numbers with hyphens (e.g., 'ACM-2024-00445-A'), numbers only (e.g., '20240044500'), and alphanumeric without hyphens (e.g., 'ACM202400445A') — all must auto-populate correctly.

## Solution

Root cause identified: the DataRaptor Extract input key for Policy_Number__c was using an exact-match filter that did not URL-encode or escape the hyphen character before passing the value to the SOQL WHERE clause, causing a zero-result return. Fix: update the DataRaptor Extract 'Input Filter' for Policy__c.Policy_Number__c to use a sanitized input mapping that preserves special characters. Additionally, add a Set Values element in the OmniScript before the DataRaptor call to trim and escape the search input. Regression test with all policy number formats in ACME-FS1 before promoting to UAT.
