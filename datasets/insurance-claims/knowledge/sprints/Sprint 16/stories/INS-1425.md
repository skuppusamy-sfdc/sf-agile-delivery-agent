## Description

As a Credentialing Specialist, I need a dedicated dashboard that visualizes the full Credentialing_Case__c pipeline — from application receipt through CAQH verification, committee review, and final approval — so that I can identify aging cases, track CAQH integration sync status, and ensure credentialing turnaround times meet NCQA-mandated 60-day windows.

CAQH integration pulls provider attestation data via the DataIntegration component established in Phase 1. This story focuses solely on the reporting surface.

## Acceptance Criteria

Given the Credentialing Specialist opens the 'Credentialing Pipeline Dashboard',
When the dashboard loads,
Then a funnel chart displays Credentialing_Case__c counts at each Status__c stage: 'Application Received' → 'Primary Source Verification' → 'CAQH Sync Complete' → 'Committee Review' → 'Approved' → 'Denied'.

Given the aging analysis section,
When a Credentialing_Case__c has Credentialing_Case__c.Days_Open__c > 45,
Then it appears in the 'At Risk' list and Credentialing_Case__c.Days_Open__c > 60 cases appear in the 'Overdue / NCQA Breach' list with red visual treatment.

Given the CAQH sync status tile,
When rendered,
Then it displays counts of Credentialing_Case__c records grouped by Credentialing_Case__c.CAQH_Sync_Status__c IN ('Pending', 'Synced', 'Failed', 'Attestation Expired') sourced from the last CAQH API poll timestamp stored in Credentialing_Case__c.CAQH_Last_Sync_Datetime__c.

Given a CAQH_Sync_Status__c = 'Failed' record,
When the specialist clicks it from the dashboard tile,
Then they are routed to the Credentialing_Case__c record where the CAQH_Error_Message__c field displays the failure reason returned by the CAQH integration.

Given the dashboard filter bar,
When the specialist filters by Credentialing_Case__c.Credentialing_Type__c IN ('Initial', 'Re-credentialing', 'Provisional'),
Then all dashboard components update to reflect only the selected credentialing type.

Given the dashboard is viewed by a user with the 'Credentialing_Read_Only' Permission Set,
When they attempt to click through to a Credentialing_Case__c record,
Then they can view but not edit the record, and sensitive fields (Credentialing_Case__c.SSN_Last4__c, Credentialing_Case__c.DEA_Number__c) are masked per field-level security.

## Solution

Build dashboard using CRM Analytics with a Credentialing_Cases dataset joining Credentialing_Case__c to Provider__c (Person Account) via the Provider__c lookup. Funnel chart implemented as a custom SAQL Steps-based widget. Aging buckets computed via a SAQL CASE expression on Days_Open__c. CAQH sync status tile uses a Donut widget. Dataset refresh triggered by a Platform Event (CAQH_Sync_Complete__e) published by the CAQH integration Apex callout class (CAQHIntegrationService.cls), ensuring near-real-time dashboard accuracy. Field masking enforced via Field-Level Security on the Credentialing_Case__c object for the Credentialing_Read_Only Permission Set. Dashboard app access restricted via Permission Set Group 'Credentialing_Ops_PSG'.
