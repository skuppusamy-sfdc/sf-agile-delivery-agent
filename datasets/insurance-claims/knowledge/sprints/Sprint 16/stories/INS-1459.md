## Description

**Bug Report — Provider Change Notification Dashboard Widget**

**Environment:** Acme Insurance Full Sandbox (Sprint 16)

**Steps to Reproduce:**
1. Navigate to the 'Provider Network Ops' CRM Analytics dashboard.
2. Note the count shown in the 'Open PCNs' KPI tile (e.g., 47).
3. Open a Provider_Change_Notification__c record with Status__c = 'Open'.
4. Update PCN Status__c from 'Open' to 'Resolved' and save.
5. Return to the 'Provider Network Ops' dashboard within 60 seconds and observe the 'Open PCNs' KPI tile.

**Expected Behavior:**
The 'Open PCNs' KPI tile count should decrement by 1 (e.g., show 46) within the dashboard's declared 15-minute refresh window, or immediately if using a live direct data connection.

**Actual Behavior:**
The tile continues to show the pre-update count (47) for up to 4 hours, significantly exceeding the expected refresh window. Investigation indicates the CRM Analytics dataflow for the PCN dataset is scheduled every 4 hours and is not triggered by PCN record updates. The dashboard was designed to reflect near-real-time data per the original requirements in INS-1431 but the dataflow schedule was misconfigured during deployment.

**Impact:** Provider Network Specialists and Provider Relations Specialists are making routing decisions based on stale PCN queues, leading to duplicate outreach to providers.

**Reproducibility:** 100% reproducible in Full Sandbox.

## Acceptance Criteria

As per Description.

Given the fix is deployed,
When a Provider_Change_Notification__c record's Status__c is updated to any value,
Then the CRM Analytics PCN dataset reflects the change within 15 minutes.

Given the fix is deployed,
When the 'Provider Network Ops' dashboard 'Open PCNs' tile is viewed after a PCN status update,
Then the displayed count matches a simultaneously-run Salesforce report filtering Provider_Change_Notification__c.Status__c = 'Open' with zero variance.

## Solution

Root cause: CRM Analytics dataflow 'PCN_Dataset_Dataflow' is on a 4-hour schedule with no event-driven refresh. Fix options evaluated: (1) Switch PCN dataset to CRM Analytics Direct Data (live Salesforce connection) — preferred if record volume < 100K; (2) If Direct Data is not viable due to SAQL complexity, implement a Platform Event (PCN_Status_Change__e) published by a Record-Triggered Flow on Provider_Change_Notification__c AFTER UPDATE WHERE Status__c IS CHANGED, with a subscribing Apex trigger that calls the CRM Analytics REST API endpoint POST /wave/dataflowjobs to trigger an on-demand dataflow run. Confirm with data architecture team which approach is approved before implementation. Update dashboard documentation to reflect new refresh SLA of 15 minutes.
