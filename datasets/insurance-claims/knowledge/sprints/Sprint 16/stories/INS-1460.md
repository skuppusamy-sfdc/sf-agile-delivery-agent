## Description

As a Claims Operations Manager, I need a comprehensive Payment Disbursement Analytics Dashboard surfaced within Salesforce so that I can monitor real-time payment cycle performance, identify disbursement bottlenecks, track EOB issuance rates, and ensure compliance with state-mandated prompt payment regulations across all Lines of Business (LOBs).

This dashboard must aggregate data from Payment__c, Claim__c, Claim_Line__c, Explanation_of_Benefits__c, and Policy__c objects to provide actionable KPIs. The dashboard will be the primary reporting tool used during weekly Claims Operations review meetings and will replace the current manual Excel-based reporting process.

Extends INS-1430 (Claims Payment Reconciliation Flow) — the analytics layer should consume payment status events produced by the reconciliation flow introduced in that story.

Key metrics required:
- Total disbursements by LOB (Medical, Dental, Vision, Workers Comp)
- Average days to payment (FNOL to Payment__c.Disbursement_Date__c)
- ERA acceptance/rejection rates from clearinghouse integration
- Pending EOB issuance queue volume
- COB recovery amounts by payer
- SIU-flagged claims pending payment hold
- TPA-managed vs. direct-pay split by dollar volume

## Acceptance Criteria

Given a Claims Operations Manager with the 'Claims_Analytics_Viewer' Permission Set Group assigned,
When they navigate to the 'Payment Disbursement Analytics' Lightning App Page,
Then they should see a fully rendered CRM Analytics (Tableau CRM) dashboard with no more than 8-second load time for datasets up to 500,000 Payment__c records.

Given the dashboard header filter bar,
When the user selects LOB, Date Range (defaulting to current calendar month), and Claim__c.Status__c values,
Then all dashboard widgets must refresh synchronously and reflect only records matching the selected filter criteria, with LOB values sourced from Policy__c.Line_of_Business__c picklist.

Given the 'Disbursement Summary' KPI tile,
When the dashboard loads or filters are applied,
Then the tile must display:
  - Total Payment__c.Amount__c summed for Payment__c.Status__c = 'Disbursed' within the selected period
  - Count of distinct Claim__c records with at least one disbursed Payment__c child record
  - A period-over-period variance percentage compared to the prior equivalent date range
  - Values must match within ±$0.01 of a parallel SOQL query run against live org data during UAT validation.

Given the 'Average Days to Payment' trend chart,
When rendered,
Then the chart must calculate the mean of (Payment__c.Disbursement_Date__c - Claim__c.FNOL_Date__c) in calendar days, grouped by week,
And display a reference line at the state prompt-payment threshold (configurable via Custom Metadata Type 'Prompt_Payment_Rule__mdt' keyed by State__c and LOB__c),
And claims where Claim__c.SIU_Hold__c = TRUE must be excluded from this calculation and counted separately in an adjacent 'SIU Hold Impact' sub-tile.

Given the 'EOB Issuance Status' donut chart,
When rendered,
Then it must segment Explanation_of_Benefits__c records by EOB__c.Status__c values: 'Issued', 'Pending', 'Failed', and -'Draft' (deprecated — EOBs no longer enter Draft status as of Sprint 14 data model change; filter must exclude this legacy value)-
And clicking any segment must open a filtered list view of the underlying EOB__c records in a modal panel.

Given the 'ERA Acceptance Rate' bar chart,
When the clearinghouse DataIntegration feed has posted ERA transaction records via Platform Event 'ERA_Transaction__e' within the selected date range,
Then the chart must display acceptance vs. rejection counts grouped by ERA_Transaction__e.Rejection_Reason_Code__c,
And a drill-down click on any rejection reason bar must navigate to a pre-filtered Claim_Line__c list view where Claim_Line__c.ERA_Rejection_Code__c matches the selected value.

Given the 'COB Recovery' summary table,
When Payment__c.COB_Recovery_Amount__c > 0,
Then the table must display recovery totals grouped by Payment__c.Secondary_Payer_Name__c,
And include columns for: Payer Name, Recovery Amount, Number of Claims, and Average Recovery per Claim,
-And include a 'Recovery Pending Subrogation' column sourced from Payment__c.Subrogation_Status__c (removed from scope per INS-1455 deferral decision)-
And the table must be exportable to CSV via a standard Lightning 'Export' button respecting field-level security.

Given the 'TPA vs. Direct Pay Split' pie chart,
When rendered,
Then payments where Claim__c.TPA_Managed__c = TRUE must be segmented as 'TPA-Managed' and all others as 'Direct Pay',
And both segments must display total Payment__c.Amount__c and percentage of total disbursement volume,
And hovering over each segment must display a tooltip with the top 3 TPA names (sourced from Claim__c.TPA_Account__r.Name) by disbursement volume.

Given a user with only the 'Claims_Read_Only' Permission Set assigned (no 'Claims_Analytics_Viewer'),
When they attempt to access the Payment Disbursement Analytics Lightning App Page,
Then they must receive a permission error message: 'You do not have access to this dashboard. Contact your system administrator.' and no Payment__c or Claim__c data must be exposed.

Given the CRM Analytics dataset refresh schedule,
When the nightly dataflow job 'Claims_Payment_Disbursement_Dataflow' completes,
Then the dashboard dataset must reflect data as of no more than 1 hour prior to the scheduled 2:00 AM PST run,
And a 'Last Refreshed' timestamp widget on the dashboard must display the most recent CRM Analytics dataflow completion timestamp pulled from the Analytics dataset metadata.

## Solution

Technical approach:

1. **CRM Analytics (Tableau CRM) Dashboard**: Build a multi-page CRM Analytics dashboard 'Claims_Payment_Disbursement' using SAQL queries against a dedicated dataflow. The dataflow 'Claims_Payment_Disbursement_Dataflow' will join Payment__c, Claim__c, Claim_Line__c, Explanation_of_Benefits__c, Policy__c, and Account (Provider__c) datasets using Augment nodes keyed on Claim__c.Id and Policy__c.Id.

2. **Custom Metadata for Prompt Payment Thresholds**: Extend existing 'Prompt_Payment_Rule__mdt' Custom Metadata Type with State__c and Line_of_Business__c fields. A SAQL CASE expression will join threshold values dynamically into the 'Average Days to Payment' dataset column for reference line rendering.

3. **ERA Platform Event Integration**: An Apex Trigger on the 'ERA_Transaction__e' Platform Event (published by the existing DataIntegration middleware layer per INS-1430) will upsert a custom object 'ERA_Summary__c' that acts as a near-real-time staging table. The CRM Analytics dataflow will include an SFDC Digest node against ERA_Summary__c to populate the ERA Acceptance Rate chart dataset.

4. **Record-Triggered Flow for SIU Hold Flagging**: A Record-Triggered Flow on Payment__c (trigger: before save, on update) will stamp Payment__c.SIU_Excluded_From_Metrics__c = TRUE when Claim__c.SIU_Hold__c = TRUE via a Get Records lookup, ensuring the SAQL average days calculation can filter cleanly using this denormalized flag without a cross-object SAQL join.

5. **Permission Model**: A new Permission Set 'Claims_Analytics_Viewer' will be created granting Read access to the CRM Analytics app asset, with Object-level Read on Payment__c, Claim__c, EOB__c, Claim_Line__c. This Permission Set will be bundled into the existing 'Claims_Operations_Manager_PSG' Permission Set Group via the Permission Set Group configuration. Field-Level Security on Payment__c.COB_Recovery_Amount__c and Payment__c.Secondary_Payer_Name__c will be restricted from profiles below Claims Operations Manager level.

6. **Lightning App Page**: A new Lightning App Page 'Payment_Disbursement_Analytics_Page' will embed the CRM Analytics dashboard component alongside a custom LWC 'eraRejectionModal' that renders the drill-down Claim_Line__c filtered list in a lightning-modal when a rejection reason bar is clicked. The LWC will use @wire(getRelatedListRecords) to fetch Claim_Line__c records filtered by ERA_Rejection_Code__c passed as a property from the CRM Analytics dashboard via cross-filter events.

7. **CSV Export**: A standard CRM Analytics table widget 'Export to CSV' action will be enabled on the COB Recovery table. Field-level security enforcement is handled natively by CRM Analytics row-level security dataset predicates tied to the user's Profile and Permission Set.

8. **Dataflow Scheduling**: The dataflow will be scheduled via CRM Analytics Schedule node for 2:00 AM PST daily. A companion Apex Scheduled Job 'RefreshTimestampJob' will query the CRM Analytics REST API endpoint '/wave/dataflows/{id}/histories' post-completion and write the latest run timestamp to a Custom Setting 'Analytics_Refresh_Metadata__c.Last_Refresh_Timestamp__c', which the 'Last Refreshed' LWC widget on the dashboard will display via a @wire(getRecord) call.
