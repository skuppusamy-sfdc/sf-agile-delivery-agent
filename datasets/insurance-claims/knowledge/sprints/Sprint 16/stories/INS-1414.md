## Description

**Bug Report — COB Priority Field Not Populated During FNOL OmniScript Intake**

**Steps to Reproduce:**
1. Launch the 'FNOL_Intake_OmniScript' from the Policyholder__c record page for a policyholder with Policy__c.COB_Indicator__c = true (i.e., member has both a primary and secondary policy on file).
2. On Step 3 ('Coordination of Benefits'), select 'Secondary Payer' from the COB_Role__c picklist.
3. Complete all remaining OmniScript steps and click 'Submit Claim'.
4. Navigate to the newly created Claim__c record.

**Expected Behavior:** Claim__c.COB_Priority__c = 'Secondary' and Claim__c.Primary_Payer_Claim_ID__c is populated with the value entered in Step 3 of the OmniScript.

**Actual Behavior:** Claim__c.COB_Priority__c remains blank (null). Claim__c.Primary_Payer_Claim_ID__c is also null. The DataRaptor Transform action 'DR_Transform_FNOL_to_Claim' at OmniScript Step 6 is not mapping the COB_Role__c element value to the Claim__c.COB_Priority__c target field — confirmed via OmniScript debug mode showing the element value present in the JSON payload but absent from the DataRaptor output map.

**Environment:** Full Sandbox (API v59.0). Regression — worked in Sprint 14 prior to the DataRaptor Transform version upgrade (v7 → v8) performed in INS-1391.

**Impact:** Medium-High — secondary COB claims are being processed as primary, resulting in incorrect allowed amount calculations and potential overpayment on Fee_Schedule__c-adjudicated lines.

## Acceptance Criteria

As per Description. Regression test must confirm Claim__c.COB_Priority__c = 'Secondary' AND Claim__c.Primary_Payer_Claim_ID__c is non-null after FNOL OmniScript submission for a secondary payer scenario. Additionally, Claim__c.COB_Priority__c = 'Primary' must remain correctly set for primary-only submissions (no regression on primary flow).

## Solution

Investigate the DataRaptor Transform 'DR_Transform_FNOL_to_Claim' v8 mapping configuration. The likely root cause is a breaking change in OmniStudio v238+ where conditional element paths in OmniScript no longer automatically flatten nested JSON nodes into the DataRaptor input context — the COB_Role__c element sits inside a conditional Step 3 block and its path key changed from 'COBRole' to 'Step3:COBDetails:COBRole' in the serialized JSON. Fix: update the DataRaptor Transform input field mapping from 'COBRole' to 'Step3:COBDetails:COBRole' and add a Set Values action immediately before the DataRaptor action to explicitly normalize the COB element values into a flat context variable. Add OmniScript unit test scenarios (OmniStudio Test Framework) for primary, secondary, and tertiary COB paths. Coordinate with INS-1391 author (Ryan Mitchell) to review all other DataRaptor Transforms upgraded in that story for similar path-flattening regressions.
