## Description

**Bug Report: EOB Issuance Volume Report double-counts Claim_Line__c records for split-payment claims**

**Steps to Reproduce:**
1. Navigate to Reports > Claims Analytics > 'Monthly EOB Issuance Volume' report.
2. Filter report to date range: 2024-10-01 to 2024-10-31.
3. Observe the 'Total EOB Records Issued' count.
4. Identify a Claim__c record (e.g., Claim #CLM-2024-009821) that has 2 associated Payment__c records (split payment scenario where Payment__c.Payment_Type__c = 'Primary' and 'COB Secondary').
5. Navigate to the Explanation_of_Benefits__c object and confirm only 1 EOB record exists for CLM-2024-009821.
6. Return to the report — CLM-2024-009821 is counted twice in the EOB volume total.

**Expected Behavior:** Each Claim__c should be counted once in the EOB Issuance Volume report regardless of how many Payment__c child records are associated.

**Actual Behavior:** Claims with multiple Payment__c records (split/COB payments) are counted once per Payment__c record, inflating the EOB issuance count. In October data, this caused a 14% overcount (1,243 reported vs 1,091 actual EOBs issued).

**Environment:** Production & Full Sandbox
**Affected Report:** 'Monthly EOB Issuance Volume' (Report ID: 00O8Z000004xKPQ)
**Salesforce Objects Involved:** Explanation_of_Benefits__c, Payment__c, Claim__c

## Acceptance Criteria

As per Description.

Additionally:
Given the corrected report runs for October 2024,
When results are validated against Explanation_of_Benefits__c record count (WHERE EOB__c.Issue_Date__c >= 2024-10-01 AND <= 2024-10-31),
Then the report total matches the direct EOB record count with 0% variance.

Given a Claim__c with Payment__c.Payment_Type__c = 'COB Secondary' exists,
When the report runs,
Then the parent Claim__c is counted only once in the EOB volume metric.

## Solution

Root cause: The 'Monthly EOB Issuance Volume' report was built on a custom Report Type joining Explanation_of_Benefits__c (primary) → Payment__c (related), causing fan-out on the Payment__c relationship. Fix: Rebuild the report using Explanation_of_Benefits__c as the sole primary object with no Payment__c join. EOB-to-Claim relationship should be surfaced via the EOB__c.Claim__c lookup field directly. Add a unique count on Explanation_of_Benefits__c.Id as the primary metric. Validate corrected report output against a SOQL count query in Developer Console: SELECT COUNT(Id) FROM Explanation_of_Benefits__c WHERE Issue_Date__c = THIS_MONTH. Deploy report fix to sandbox for QA validation before production push.
