## Description

As a Compliance Officer, I need a CRM Analytics dashboard that identifies Fee_Schedule__c utilization rates across active Group_Service_Agreement__c (GSA) contracts and flags Claim_Line__c records where the billed CPT code amount exceeds the contracted Fee_Schedule__c rate by more than 15% so that I can refer potential billing anomalies to the SIU team and ensure GSA contract compliance. The dashboard must also surface providers with a pattern of anomalous billing across multiple GSAs.

## Acceptance Criteria

Given the Compliance Officer opens the 'Fee Schedule Compliance Dashboard' CRM Analytics app,
When the dashboard loads,
Then a summary KPI tile must display the count of Claim_Line__c records where (Claim_Line__c.Billed_Amount__c - Fee_Schedule__c.Contracted_Rate__c) / Fee_Schedule__c.Contracted_Rate__c > 0.15 and Claim_Line__c.Claim__r.Status__c != 'Denied' for the current quarter.

Given the anomaly detection data table,
When rendered,
Then each row must display: Claim_Line__c.CPT_Code__c, Claim_Line__c.Billed_Amount__c, Fee_Schedule__c.Contracted_Rate__c, calculated overage percentage, Provider__c.NPI__c, Group_Service_Agreement__c.Name, and an action button 'Refer to SIU' that creates a new Case record with Case.RecordType = 'SIU_Referral' and pre-populates Case.Related_Claim__c and Case.Provider__c.

Given the GSA utilization chart,
When rendered,
Then a horizontal bar chart must display each active Group_Service_Agreement__c.Name on the Y-axis with the X-axis showing percentage of Claim_Line__c records billed within contracted Fee_Schedule__c rates (compliant vs. anomalous split as stacked bars).

Given a Provider__c has Claim_Line__c anomalies across 3 or more distinct Group_Service_Agreement__c records in the current quarter,
When the 'Multi-GSA Anomaly Providers' widget renders,
Then the provider must appear in the widget with: Provider__c.Name, Provider__c.NPI__c, count of affected GSAs, and total overage amount in dollars.

Given the Compliance Officer applies the CPT code filter,
When a specific CPT code is selected (e.g., '99213'),
Then all dashboard widgets must filter to Claim_Line__c records with Claim_Line__c.CPT_Code__c = selected value and the anomaly threshold calculation must remain applied.

Given the 'Refer to SIU' action is clicked on an anomalous Claim_Line__c row,
When the Case is created,
Then a Salesforce Notification must be sent to the SIU queue ('SIU_Review_Queue') and the Claim__c.SIU_Referral_Flag__c must be set to true via a Record-Triggered Flow.

Given the dashboard dataset is refreshed,
When the scheduled dataflow runs,
Then the Fee_Schedule__c.Contracted_Rate__c used in calculations must reflect the rate effective for the Claim_Line__c.Service_Date__c using Fee_Schedule__c.Effective_Date__c and Fee_Schedule__c.Termination_Date__c range logic, not the current rate.

## Solution

Build CRM Analytics app 'Fee Schedule Compliance Dashboard' with a custom dataset joining Claim_Line__c, Fee_Schedule__c (date-effective rate lookup resolved in DataRaptor Transform using Service_Date__c range match), Claim__c, Provider__c, and Group_Service_Agreement__c. Overage percentage computed as a SAQL derived measure. Anomaly threshold (15%) stored as a CRM Analytics dashboard variable 'Anomaly_Threshold_Pct' for configurability. 'Refer to SIU' action implemented as a Salesforce Action (Quick Action) on Claim_Line__c invoked from the CRM Analytics List widget via a Salesforce action URL; a Record-Triggered Flow on Case creation (RecordType = SIU_Referral) fires to set Claim__c.SIU_Referral_Flag__c = true and sends a Custom Notification to the SIU_Review_Queue. Multi-GSA anomaly widget uses a SAQL GROUP BY Provider__c.NPI__c, GSA.Name with HAVING count(distinct GSA.Id) >= 3. Dataset scheduled dataflow runs nightly at 2 AM.
