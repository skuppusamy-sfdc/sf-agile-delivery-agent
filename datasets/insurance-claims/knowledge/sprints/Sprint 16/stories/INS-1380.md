## Description

As a Credentialing Specialist, I need a dedicated analytics dashboard that shows CAQH verification turnaround time and the aging distribution of open Credentialing_Case__c records in the re-credentialing queue so that I can prioritize workloads, meet NCQA credentialing cycle standards, and proactively flag cases at risk of SLA breach.

CAQH integration data is sourced via the existing DataIntegration component syncing with the CAQH ProView API (established in Phase 1).

## Acceptance Criteria

Given the Credentialing Specialist navigates to the 'Credentialing Analytics' tab,
When the dashboard loads,
Then a CAQH Verification Turnaround KPI tile displays the average number of days between Credentialing_Case__c.CAQH_Verification_Requested_Date__c and Credentialing_Case__c.CAQH_Verified_Date__c for cases closed in the current quarter.

Given the Re-Credentialing Queue Aging chart renders,
When displayed,
Then it shows a stacked bar chart bucketing open Credentialing_Case__c records (Type__c = 'Re-Credentialing', Status__c NOT IN ('Approved','Denied','Withdrawn')) into aging bands: 0–30 days, 31–60 days, 61–90 days, 90+ days, based on DATEDIFF(TODAY(), Credentialing_Case__c.CreatedDate).

Given a case falls into the '90+ days' aging band,
When the dashboard refreshes,
Then those records are highlighted in red and a count badge labeled 'At Risk' is displayed.

Given the dashboard filter for Credentialing Type (Initial, Re-Credentialing, CAQH Attestation) is applied,
When a filter value is selected,
Then all dashboard components filter to Credentialing_Case__c records matching Credentialing_Case__c.Type__c = selected value.

-Given the dashboard, When a Credentialing Specialist clicks a case row, Then an inline edit panel allows updating Credentialing_Case__c.Status__c.- (deprecated — inline edit removed; all status updates must go through the Credentialing OmniScript workflow per governance decision)

Given the dashboard is accessed,
When rendered for a user with the 'Credentialing_Specialist' Permission Set,
Then only Credentialing_Case__c records assigned to that specialist (OwnerId = running user) are visible unless the user also holds the 'Credentialing_Supervisor' Permission Set, in which case all records are visible.

## Solution

Build CRM Analytics (Tableau CRM) dashboard 'Credentialing_Analytics_Dash' with two SAQL-based widgets: (1) KPI widget computing AVG of CAQH_Verified_Date__c - CAQH_Verification_Requested_Date__c using a date difference measure on Credentialing_Case__c dataset; (2) Stacked bar chart using SAQL CASE WHEN bucketing on CreatedDate delta. Dataset refreshed nightly via scheduled dataflow joining Credentialing_Case__c with Provider__c (Person Account). Row-level security enforced via a CRM Analytics Security Predicate: 'OwnerId' == "$User.Id" OR "$User.ProfileName" == 'Credentialing Supervisor'. At-Risk badge implemented as a conditional formatting rule on the chart widget. Dashboard embedded on the Credentialing App Home page via Lightning App Builder.
