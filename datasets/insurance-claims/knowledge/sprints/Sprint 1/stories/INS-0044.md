## Description

**Bug Report**

**Summary:** When a new Credentialing_Case__c record is created — either manually by a Credentialing Specialist, via the CAQH integration DataRaptor write, or through the Provider 360 FlexCard 'Initiate Re-credentialing' action (see INS-0039) — the Owner field defaults to the creating user instead of the 'Credentialing_Assignment_Coordinator' queue as designed.

**Steps to Reproduce:**
1. Log in as any user with PSG_Credentialing_Specialist permission set group.
2. Navigate to any Provider__c (Person Account) record.
3. Click 'New Credentialing Case' from the Related List or from the Provider 360 FlexCard quick action.
4. Fill in required fields: Credentialing_Type__c = 'Initial', Provider__c = current provider, Submission_Date__c = today.
5. Click Save.
6. Observe the Owner field on the newly created Credentialing_Case__c record.

**Expected Result:** Credentialing_Case__c.OwnerId is set to the 'Credentialing_Assignment_Coordinator' queue ID.

**Actual Result:** Credentialing_Case__c.OwnerId is set to the ID of the logged-in user who performed the save action.

**Environment:** Acme-INS-DEV1 and Acme-INS-QA1 sandboxes — reproducible in both. API v59.0.

**Root Cause Hypothesis:** The before-save Record-Triggered Flow 'RTF_CredCase_SetOwnerToQueue' appears to be inactive in both environments. The flow was likely deactivated during the Sprint 1 metadata refresh and not reactivated. Additionally, the Queue 'Credentialing_Assignment_Coordinator' has a different ID in QA1 vs DEV1, suggesting the Flow references a hardcoded Queue ID rather than a dynamic lookup.

**Impact:** All new credentialing cases are landing in individual user inboxes, bypassing the Assignment Coordinator queue and breaking the triage workflow. Credentialing SLA tracking is being skewed as a result.

## Acceptance Criteria

As per Description.

Additional verification: After fix is deployed, confirm Credentialing_Case__c.OwnerId resolves to the 'Credentialing_Assignment_Coordinator' queue when records are created via (a) manual UI entry, (b) DataRaptor Turbo Action from OmniScript, and (c) direct Apex insert in an anonymous script. Confirm fix uses a dynamic Queue lookup (by DeveloperName = 'Credentialing_Assignment_Coordinator') rather than a hardcoded ID so the fix is environment-agnostic.

## Solution

Reactivate the before-save Record-Triggered Flow 'RTF_CredCase_SetOwnerToQueue' in all sandboxes and production via Change Set or SFDX deployment. Refactor the Flow to replace the hardcoded Queue ID with a Get Records element that queries the Group object where Type = 'Queue' AND DeveloperName = 'Credentialing_Assignment_Coordinator', returning the Id dynamically. Assign the retrieved Id to the {!$Record.OwnerId} before-save assignment. Add a Flow fault path that logs a Platform Event if the Queue is not found, preventing silent failures. Update the CI pipeline deployment checklist to include a post-deploy Flow activation verification step to prevent recurrence during future metadata refreshes.
