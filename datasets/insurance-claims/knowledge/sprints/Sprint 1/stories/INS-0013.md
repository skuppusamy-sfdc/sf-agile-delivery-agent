## Description

As a Credentialing Specialist, I need a structured Credentialing_Case__c object with an intake OmniScript so that I can initiate, track, and manage provider credentialing applications end-to-end within Salesforce, replacing the current manual spreadsheet process. The intake form must capture all data required for CAQH ProView lookup and primary source verification (PSV) initiation.

## Acceptance Criteria

Given the Credentialing_Case__c object is deployed,
When a Credentialing Specialist creates a new record,
Then the object includes: Case_Number__c (Auto Number format CRED-{0000}), Provider__c (Lookup to Account, required), Credentialing_Type__c (Picklist: Initial|Re-credentialing|Delegated), Credentialing_Status__c (Picklist: Intake|PSV In Progress|Committee Review|Approved|Denied|Withdrawn), Application_Received_Date__c (Date), CAQH_Profile_ID__c (Text 10), CAQH_Last_Attested_Date__c (Date), NPI__c (Text 10), TIN__c (Text 9), Expiration_Date__c (Date), Assigned_To__c (Lookup to User).

Given the OmniScript 'Credentialing_Intake_OS' is launched from a Provider Account,
When the Credentialing Specialist completes Step 1 (Provider Identification),
Then the script pre-populates NPI__c and CAQH_Profile_ID__c from the parent Provider Account fields if they exist, with an option to override.

Given the OmniScript Step 2 (Credentialing Type Selection) is reached,
When 'Re-credentialing' is selected,
Then an additional field 'Previous_Credentialing_Case__c' (Lookup to Credentialing_Case__c) appears and is required, linking the new case to the prior approved case.

Given the OmniScript is submitted,
When the Credentialing_Case__c record is created,
Then Credentialing_Status__c is set to 'Intake', a Task is auto-created assigned to Assigned_To__c with Subject 'Complete CAQH Verification' and Due Date = TODAY() + 5, and a Chatter post is added to the record: 'New credentialing case initiated via intake form by {CurrentUser}'.

Given a Credentialing_Case__c record where Expiration_Date__c is within 90 days,
When a scheduled batch job runs nightly,
Then the record's Credentialing_Status__c remains unchanged but a Platform Event 'Credentialing_Expiry_Alert__e' is published with fields: Case_Id__c, Provider_Name__c, Days_To_Expiry__c for downstream notification handling.

-Given CAQH integration is live, When OmniScript Step 1 runs, Then auto-call CAQH ProView REST API to validate CAQH_Profile_ID__c in real time- (deprecated: CAQH API integration moved to Phase 2 per integration dependency review; Phase 1 uses manual entry only)

## Solution

Deploy Credentialing_Case__c custom object via SFDX metadata. Build OmniScript 'Credentialing_Intake_OS' with 3 steps using OmniStudio Designer: Step 1 (Provider ID with pre-population via DataRaptor Extract on Account), Step 2 (Credentialing Type with conditional element for Re-credentialing lookup), Step 3 (Review & Submit with DataRaptor Upsert to create Credentialing_Case__c). Post-save automation handled by Record-Triggered Flow 'RTF_Credentialing_Case_AfterCreate' (After Save, New Records only) that creates Task and posts to Chatter via standard Flow actions. Nightly expiry scan implemented as Apex Scheduled Batch 'CredentialingExpiryBatch' querying Credentialing_Case__c WHERE Expiration_Date__c <= NEXT_N_DAYS:90 AND Credentialing_Status__c = 'Approved', publishing Platform Events in execute() method. Platform Event subscriber Flow routes alerts to Credentialing Specialist Chatter group.
