## Description

As a Claims Operations Manager, I need a Fee Schedule Engine on the Salesforce platform that can look up the allowed amount for a given CPT code, service date, provider network tier, and LOB so that Claim_Line__c records are automatically populated with Allowed_Amount__c during adjudication, reducing manual rate research and improving payment accuracy.

## Acceptance Criteria

Given a Fee_Schedule__c object is configured
When a Fee Schedule record is created
Then it must contain: CPT_Code__c (Text), Modifier__c (Text, optional), LOB__c (Picklist), Network_Tier__c (Picklist: Tier1|Tier2|Out-of-Network), Effective_Date__c (Date), End_Date__c (Date), Allowed_Amount__c (Currency), Rate_Basis__c (Picklist: Flat|Percentage_of_Billed|RVU_Based)

Given a Claim_Line__c record is in status 'Pending' and CPT_Code__c is populated
When the adjudication engine is triggered
Then a SOQL lookup finds the most specific matching Fee_Schedule__c record using: CPT_Code__c exact match → Modifier__c match (if present) → LOB__c match → Network_Tier__c match → Effective_Date__c <= Service_Date__c <= End_Date__c

Given a matching Fee_Schedule__c record is found
When Rate_Basis__c = 'Flat'
Then Claim_Line__c.Allowed_Amount__c = Fee_Schedule__c.Allowed_Amount__c

Given Rate_Basis__c = 'Percentage_of_Billed'
When the lookup resolves
Then Claim_Line__c.Allowed_Amount__c = Claim_Line__c.Billed_Amount__c * (Fee_Schedule__c.Allowed_Amount__c / 100), rounded to 2 decimal places

Given no matching Fee_Schedule__c record is found
When the adjudication engine completes the lookup
Then Claim_Line__c.Line_Status__c is set to 'Pending Manual Review', a flag field Fee_Schedule_Not_Found__c (Checkbox) is set to true, and a Task is created for the Claims Operations queue

Given multiple Fee_Schedule__c records match the lookup criteria
When tie-breaking is required
Then the record with the most recent Effective_Date__c is selected; if still tied, the record with the non-null Modifier__c takes precedence over a null Modifier__c record

Given the fee schedule lookup is invoked
When more than 500 Claim_Line__c records are processed in a single transaction
Then the engine operates in bulk-safe mode with no SOQL queries inside loops and all lookups resolved via a Map pre-loaded from a single SOQL query

## Solution

Apex invocable class `FeeScheduleLookupService` with `@InvocableMethod` annotation, callable from both Flow and direct Apex. Core logic uses a bulkified Map<String, Fee_Schedule__c> keyed on composite key (CPT_Code__c + '_' + LOB__c + '_' + Network_Tier__c) loaded in a single SOQL query filtered by date range. Modifier matching applied as a secondary filter on the pre-loaded map values. Rate calculation logic encapsulated in a `RateCalculator` inner class. Service invoked from a Record-Triggered Flow on Claim_Line__c (After Save, when Line_Status__c changes to 'Pending' and CPT_Code__c is not null). Task creation for unmatched lines handled via Flow action post-invocable response. Custom metadata type `Fee_Schedule_Config__mdt` stores configurable parameters (rounding precision, fallback behavior). Unit tests cover 95%+ code coverage with bulk test of 200 Claim_Line__c records.
