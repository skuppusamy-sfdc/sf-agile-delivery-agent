## Description

As a Credentialing Specialist, I need an OmniScript-guided workflow to initiate a new Credentialing_Case__c for a provider, automatically fetch the provider's CAQH profile data via the CAQH ProView API, and pre-populate the credentialing checklist fields so that I can reduce manual data entry, decrease credentialing turnaround time, and ensure the most current provider attestation data is used during the credentialing review.

The OmniScript must be launchable from the Provider (Account) record page via a Quick Action. CAQH API credentials are stored in Salesforce Named Credentials (NC_CAQH_ProView). The Credentialing_Case__c record must be created before the CAQH call is made so that a case ID exists for audit logging.

## Acceptance Criteria

Given a Credentialing Specialist is on an Account record where Account.CAQH_Provider_ID__c is populated,
When they click the 'Initiate Credentialing' Quick Action,
Then the OmniScript (OS_InitiateCredentialing) must launch in a modal and display Step 1: 'Confirm Provider Identity' showing Account.Name, Account.NPI__c, Account.TIN__c (masked as XXX-XX-XXXX), and Account.CAQH_Provider_ID__c.

Given the Credentialing Specialist confirms provider identity on Step 1 and clicks Next,
When Step 2 renders,
Then a new Credentialing_Case__c record must be created via a DataRaptor Turbo Action with:
  - CredentialingCase__c.Provider__c = current Account ID
  - CredentialingCase__c.Status__c = 'Initiated'
  - CredentialingCase__c.Initiated_By__c = running user ID
  - CredentialingCase__c.Initiated_Date__c = today

Given the Credentialing_Case__c is created,
When the OmniScript proceeds to Step 2,
Then an HTTP callout must be made to the CAQH ProView API (GET /provider/{caqhProviderId}/profile) using Named Credential NC_CAQH_ProView, and the following fields must be mapped from the response to the Credentialing_Case__c record:
  - CredentialingCase__c.CAQH_Last_Attestation_Date__c ← caqhResponse.lastAttestationDate
  - CredentialingCase__c.Board_Certification__c ← caqhResponse.boardCertification.status
  - CredentialingCase__c.DEA_Number__c ← caqhResponse.deaNumber
  - CredentialingCase__c.Malpractice_Insurance_Expiry__c ← caqhResponse.malpracticeInsurance.expirationDate
  - CredentialingCase__c.License_State__c ← caqhResponse.stateLicense.state
  - CredentialingCase__c.License_Expiry__c ← caqhResponse.stateLicense.expirationDate

Given the CAQH API call returns a 200 response,
When Step 2 of the OmniScript renders,
Then all fetched fields must be displayed as read-only pre-filled values with a banner: 'Data retrieved from CAQH ProView as of [timestamp]. Please verify before submitting.'

Given the CAQH API call returns a non-200 response or times out,
When the error occurs,
Then the OmniScript must display an error message: 'Unable to retrieve CAQH profile. Please enter credentialing information manually.' and allow the Credentialing Specialist to proceed with manual entry.

Given the OmniScript reaches the final step and the specialist clicks 'Submit',
When submission completes,
Then CredentialingCase__c.Status__c must update to 'Under Review' and a Task must be assigned to the Credentialing Specialist with Subject = 'Review Credentialing Documents — [Provider Name]' and Due Date = today + 10 business days.

-Given the CAQH profile shows an attestation date older than 120 days, When the OmniScript renders, Then auto-deny the credentialing case — removed per compliance review, replaced with manual review flag only-

## Solution

OmniScript (OS_InitiateCredentialing) built with OmniStudio, using a multi-step layout with Step components for identity confirmation, CAQH data fetch, document checklist, and submission. Credentialing_Case__c creation handled via DataRaptor Turbo Action (DRT_CreateCredentialingCase) in Step 2 before the CAQH callout. CAQH ProView API callout implemented as an Integration Procedure (IP_CAQH_FetchProviderProfile) using HTTP Action element pointing to Named Credential NC_CAQH_ProView. Response mapped to OmniScript data JSON via Set Values elements. Error handling via Try/Catch in the Integration Procedure with conditional branching. Task creation on submission handled by a Record-Triggered Flow (RTF_CredentialingCaseOnCreate) firing on CredentialingCase__c Status__c change to 'Under Review'. Business days calculation for due date uses a custom Apex invocable action (BusinessDayCalculator.addBusinessDays).
