## Description

As a Credentialing Specialist, I need an automated intake process for new provider credentialing requests that creates a Credentialing_Case__c record, validates the provider's NPI against the NPPES registry, and retrieves CAQH ProView attestation status so that I can begin primary source verification with confirmed identity data and reduce manual data entry errors during the credentialing lifecycle.

## Acceptance Criteria

Given a Credentialing Specialist submits a new credentialing request via the 'New Credentialing Case' screen flow, when all required fields are entered, then:

1. The screen flow must capture: Provider__c lookup (or create new Person Account), Credentialing_Case__c.NPI_Submitted__c (Text, 10-digit validation), Credentialing_Case__c.TIN_Submitted__c (Text, 9-digit validation), Credentialing_Case__c.Credentialing_Type__c (Picklist: Initial|Re-Credentialing|Delegated), Credentialing_Case__c.Requested_Effective_Date__c (Date).

2. Given NPI_Submitted__c is entered, when the flow advances, then an Integration Procedure must call the NPPES NPI Registry API and return: Provider first/last name, Provider Type, Primary Practice Address, Enumeration Date. If NPPES returns no match, the flow must display 'NPI not found in NPPES registry — verify and re-enter' and block progression.

3. Given NPPES validation succeeds, when the flow calls CAQH, then Integration Procedure 'IP_CAQH_Status_Check' must return: CAQH_ID, Attestation_Status (Complete|Incomplete|Expired), Last_Attestation_Date. These values must be stored on: Credentialing_Case__c.CAQH_ID__c, Credentialing_Case__c.CAQH_Attestation_Status__c, Credentialing_Case__c.CAQH_Last_Attested__c.

4. Given CAQH_Attestation_Status = 'Expired' or 'Incomplete', when the Credentialing_Case__c is created, then Credentialing_Case__c.Status__c must be set to 'Pending_Provider_Action' and a Task must be auto-created assigned to the provider's linked Contact with Subject = 'CAQH Attestation Required — Action Needed'.

5. Given all validations pass, when Credentialing_Case__c is created, then: Status__c = 'Intake_Complete', Created_Date__c = TODAY(), SLA_Due_Date__c = TODAY() + 30 (calendar days), and a notification email must be sent to the assigned Credentialing_Specialist using Email Template 'Credentialing_Case_Opened'.

6. Given a Credentialing_Case__c record is created, when viewed on the Provider__c related list, then it must appear under 'Credentialing Cases' and display Credentialing_Case__c.Status__c, Credentialing_Case__c.Credentialing_Type__c, Credentialing_Case__c.SLA_Due_Date__c.

## Solution

Build Screen Flow 'SF_Credentialing_Intake' with 3 screens: Provider Lookup, NPI/TIN Entry, Review & Submit. Use Apex action 'NPPESValidationAction' to call NPPES REST API (https://npiregistry.cms.hhs.gov/api/) with NPI input and parse JSON response. Build Integration Procedure 'IP_CAQH_Status_Check' using HTTP Action to call CAQH ProView SOAP API with Basic Auth credentials stored in Named Credential 'CAQH_ProView_NC'. Use DataRaptor Transform to create Credentialing_Case__c record with mapped fields. Record-Triggered Flow on Credentialing_Case__c fires on create where CAQH_Attestation_Status__c IN ('Expired','Incomplete') to create follow-up Task. SLA_Due_Date__c calculated via Formula Field. Flow launched from 'New Credentialing Case' Quick Action on Provider__c page layout.
