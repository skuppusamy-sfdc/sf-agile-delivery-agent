# INS-0049: Implement Provider__c (Person Account) and Facility__c (Business Account) record types with NPI and TIN fields

## Description

As a Provider Network Specialist, I need Provider and Facility records to be modeled using Salesforce Person Accounts and Business Accounts respectively, with all required NPI, TIN, specialty, and credentialing status fields, so that the provider data model supports downstream PNA contracting, credentialing workflows, and claims adjudication lookups.

This story establishes the Account record type configuration and custom field schema for providers and facilities. Provider 360 FlexCards, CAQH/NPPES integration, and credentialing case automation are in later stories.

## Acceptance Criteria

Given Person Accounts are enabled in the org,
When a new Account is created with RecordType = 'Provider',
Then the record is a Person Account and the following custom fields are available on the layout:
  - Account.NPI__c (Text 10, unique, external ID)
  - Account.TIN__c (Text 9, encrypted)
  - Account.Provider_Type__c (Picklist: MD, DO, NP, PA, LCSW, DDS, OD, PharmD, Other)
  - Account.Primary_Specialty__c (Text 100)
  - Account.Secondary_Specialty__c (Text 100)
  - Account.Credentialing_Status__c (Picklist: Not Started, In Progress, Credentialed, Recredentialing Due, Suspended, Terminated)
  - Account.CAQH_Provider_ID__c (Text 20)
  - Account.DEA_Number__c (Text 20, encrypted)
  - Account.License_State__c (Picklist: all 50 US states + DC)
  - Account.License_Number__c (Text 30)
  - Account.License_Expiration_Date__c (Date)
  - Account.Board_Certification__c (Text 100)
  - Account.Accepts_New_Patients__c (Checkbox)
  - Account.In_Network__c (Formula Checkbox — true if active PNA exists)

Given a Business Account is created with RecordType = 'Facility',
When a user inspects the record,
Then the following fields are present:
  - Account.NPI__c (Text 10, unique, external ID — shared field with Provider RecordType)
  - Account.TIN__c (Text 9, encrypted — shared field)
  - Account.Facility_Type__c (Picklist: Hospital, ASC, Urgent Care, Lab, Imaging Center, SNF, Home Health, Pharmacy)
  - Account.CMS_Certification_Number__c (Text 20)
  - Account.JCAHO_Accredited__c (Checkbox)
  - Account.Network_Participation_Status__c (Picklist: Participating, Non-Participating, Pending)
  - Account.Credentialing_Status__c (shared picklist field)

Given duplicate NPI values must be prevented,
When a user attempts to save a Provider or Facility record with an NPI__c that already exists on another Account,
Then a duplicate rule fires and blocks the save with the message: 'A provider or facility with this NPI already exists in the system.'

Given the provider's license is expiring,
When Account.License_Expiration_Date__c is within 90 days of today,
Then Account.Credentialing_Status__c displays a visual warning indicator on the record page (addressed via page layout formula field in a follow-on story).

-Given providers are stored as Contacts linked to a generic Account,
Then the Contact object is the primary provider record.- (Superseded — Person Account model adopted per architecture decision ADR-003)

## Solution

1. Enable Person Accounts in org settings (requires Salesforce Support ticket if not already enabled — tracked as prerequisite).
2. Create 'Provider' (Person Account) and 'Facility' (Business Account) Record Types with associated page layouts.
3. Custom fields added to Account object via metadata; encrypted fields (TIN__c, DEA_Number__c) use Classic Encryption (Shield Platform Encryption evaluated for Phase 2).
4. NPI__c set as External ID and Unique to support future NPPES integration upsert operations.
5. Duplicate Rule created for NPI__c using standard Duplicate Management (DuplicateRule + MatchingRule metadata).
6. In_Network__c formula field references a related list count of active Provider_Network_Agreement__c records (PNA.Status__c = 'Active') — cross-object formula.
7. All metadata in ins-provider-core package; deployed via Salesforce CLI pipeline.
