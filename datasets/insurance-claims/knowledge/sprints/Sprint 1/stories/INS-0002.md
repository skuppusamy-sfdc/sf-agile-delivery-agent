## Description

As a Compliance Officer, I need a well-structured Salesforce Role Hierarchy and Permission Set Group model implemented so that each user persona (Claims Operations Manager, Provider Network Specialist, Credentialing Specialist, Broker Relations Coordinator, SIU Analyst, Underwriting Analyst) has access only to the objects, fields, and records appropriate for their job function, ensuring regulatory compliance with HIPAA minimum necessary standards.

This story establishes the foundational security model for all of Phase 1 and Phase 2 work. All subsequent user stories assume this permission model is in place. Extends INS-0001 (requires core objects to exist before OLS/FLS can be configured).

## Acceptance Criteria

Given the Role Hierarchy is configured,
When a System Administrator reviews Setup > Roles,
Then the following role tree must exist:
  - Acme Insurance (Root)
    - Claims Division
      - Claims Operations Manager
        - Claims Adjudicator
        - SIU Analyst
    - Provider Relations Division
      - Provider Network Specialist
        - Provider Relations Specialist
      - Credentialing Specialist
    - Broker & Underwriting Division
      - Underwriting Analyst
      - Broker Relations Coordinator

Given the Permission Set Groups are configured,
When reviewing Setup > Permission Set Groups,
Then the following PSGs must exist, each composed of the listed Permission Sets:
  - PSG_Claims_Adjudicator: [PS_Claim_Read_Write, PS_ClaimLine_Read_Write, PS_Payment_Read, PS_EOB_Read, PS_Policy_Read]
  - PSG_Claims_Operations_Manager: [PS_Claim_Full_Access, PS_ClaimLine_Full_Access, PS_Payment_Full_Access, PS_EOB_Full_Access, PS_Policy_Read_Write, PS_SIU_Escalation]
  - PSG_Provider_Network_Specialist: [PS_Provider_Read_Write, PS_PNA_Read_Write, PS_FeeSchedule_Read, PS_GSA_Read_Write]
  - PSG_Credentialing_Specialist: [PS_Provider_Read_Write, PS_CredentialingCase_Full_Access, PS_CAQH_Integration_Access]
  - PSG_SIU_Analyst: [PS_Claim_Read, PS_ClaimLine_Read, PS_SIU_Case_Full_Access, PS_Policy_Read]
  - PSG_Underwriting_Analyst: [PS_Policy_Full_Access, PS_GroupServiceAgreement_Read_Write]

Given the Field-Level Security (FLS) is configured for the PSG_Claims_Adjudicator group,
When a user with PSG_Claims_Adjudicator attempts to view Claim__c records,
Then the field Claim__c.TIN__c must be masked (read = false, edit = false) and Claim__c.SIU_Referral_Flag__c must be read-only (read = true, edit = false).

Given the Sharing Model is configured,
When reviewing Setup > Sharing Settings,
Then:
  - Claim__c OWD = 'Private'
  - Policy__c OWD = 'Private'
  - Account (Provider) OWD = 'Public Read Only'
  - A Criteria-Based Sharing Rule must exist: share Claim__c records where Claim__c.LOB__c = 'TPA' with the role 'Claims Operations Manager' with 'Read/Write' access.

Given a test user is assigned PSG_SIU_Analyst,
When that user attempts to edit Claim__c.Status__c,
Then the edit must be blocked and the user receives a permission error (field is read-only for this PSG).

## Solution

All Permission Sets, Permission Set Groups, and Role Hierarchy deployed via SFDX metadata (permissionset-meta.xml, permissionsetgroup-meta.xml, role-meta.xml). FLS configured within each Permission Set's fieldPermissions node. OWD and Sharing Rules deployed via sharingRules-meta.xml and deployed as part of the security metadata package. Criteria-Based Sharing Rule for TPA claims implemented via sharingCriteriaRule in Claim__c sharing rules metadata. Post-deployment validation performed using a custom Apex test class (SecurityModelValidationTest) that programmatically asserts FLS for key fields across all PSGs using Schema.DescribeFieldResult. Muting Permission Sets used within PSGs where baseline profile grants excess access.
