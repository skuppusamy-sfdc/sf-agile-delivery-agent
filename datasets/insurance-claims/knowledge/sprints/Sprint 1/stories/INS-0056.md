## Description

As a Credentialing Specialist, I need a structured Credentialing_Case__c object and a guided OmniScript intake form so that new provider credentialing requests are captured consistently, routed to the correct specialist queue, and pre-populated with data retrieved from CAQH and NPPES integrations — reducing manual data entry and ensuring data accuracy at the point of intake.

This story covers: (1) Credentialing_Case__c object field configuration, (2) OmniScript 'OS_New_Credentialing_Request' build, (3) CAQH DataRaptor integration stub (live integration in Phase 2), (4) assignment to Credentialing Queue.

## Acceptance Criteria

Given the Credentialing_Case__c object is deployed,
When a record is created,
Then the following fields are available and required where noted: Provider__c (Lookup, required), Case_Type__c (Picklist: Initial|Re-credentialing|Provisional, required), Submitted_Date__c (Date, required), Status__c (Picklist: Received|In Review|Pending Provider Info|Approved|Denied|Withdrawn, default='Received'), CAQH_Provider_ID__c (Text), NPI__c (Text, 10 chars), TIN__c (Text, encrypted), Expiration_Date__c (Date), Assigned_Specialist__c (Lookup to User), Decision_Date__c (Date), Denial_Reason__c (Text Area, visible only when Status__c = 'Denied').

Given the OmniScript 'OS_New_Credentialing_Request' is launched from a Provider__c record,
When Step 1 (Provider Lookup) completes,
Then the script pre-populates NPI__c and Provider_Name from the parent Account record via DataRaptor Extract.

Given the specialist enters a valid CAQH_Provider_ID__c on Step 2,
When they click 'Retrieve CAQH Data',
Then a DataRaptor Turbo Action calls the CAQH stub (Named Credential: NC_CAQH_Sandbox) and returns: Board_Certification__c, DEA_Number__c, Malpractice_Insurance_Carrier__c — populating the form fields (stub returns mock data in Phase 1).

Given all required fields are completed,
When the specialist submits the OmniScript,
Then a Credentialing_Case__c record is created with Status__c = 'Received', a Case Number auto-generated (format CRED-{0000000}), and the record is assigned to Queue 'Q_Credentialing_Review'.

Given the Credentialing_Case__c is created,
When assignment to Q_Credentialing_Review occurs,
Then an email notification is sent to the queue members using Email Template 'ET_New_Credentialing_Case' with the provider name and CRED case number.

-Given the OmniScript is submitted, When Status__c = 'Received', Then automatically trigger CAQH primary source verification API call- (deprecated: full CAQH PSV deferred to Phase 2 integration sprint)

Given a Credentialing Specialist opens an existing Credentialing_Case__c,
When Status__c is updated to 'Approved',
Then Decision_Date__c is auto-stamped with TODAY and a Record-Triggered Flow sets the linked Provider__c.Credentialing_Status__c = 'Credentialed'.

## Solution

Credentialing_Case__c deployed via metadata package (extends INS-0051 object foundation). OmniScript built in OmniStudio Designer with 4 steps: (1) Provider Lookup, (2) CAQH Retrieval, (3) Credentialing Details, (4) Review & Submit. DataRaptor Extract 'DR_Extract_ProviderForCredentialing' queries Account fields. DataRaptor Turbo Action 'DTA_CAQH_Lookup' calls Named Credential NC_CAQH_Sandbox via HTTP GET (Phase 1: returns static mock JSON from a Custom Metadata Type 'CAQH_Mock_Response__mdt'). OmniScript Submit action uses DataRaptor Load 'DR_Load_CredentialingCase' to upsert record. Record-Triggered Flow 'RTF_Credentialing_Approval_Sync' fires on Status__c change to 'Approved' to update Provider__c. Queue assignment via Flow Create Records with OwnerId = queue ID stored in Custom Label 'CL_Credentialing_Queue_ID'.
