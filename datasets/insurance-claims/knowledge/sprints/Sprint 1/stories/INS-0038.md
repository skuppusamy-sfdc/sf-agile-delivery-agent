## Description

As a Compliance Officer, I need a well-structured Salesforce role hierarchy and permission set groups defined for all claims processing personas so that data visibility, field-level security, and object access are enforced according to our regulatory and operational requirements from day one.

## Acceptance Criteria

Given the role hierarchy is configured
When a user with the role 'Claims_Operations_Manager' is logged in
Then they have Read/Write access to Claim__c, Claim_Line__c, Payment__c, and EOB__c; and Read-Only access to Policy__c and Provider__c records within their business unit

Given Permission Set Groups are defined
When the 'PSG_Claims_Adjudicator' group is assigned to a user
Then the user inherits permissions from: PS_Claims_Base_Access, PS_Payment_View, PS_EOB_Generate

Given the 'PSG_SIU_Investigator' group is assigned
When the user accesses a Claim__c record with SIU_Referral__c = true
Then the user can edit SIU_Notes__c and SIU_Status__c fields; non-SIU users see these fields as blank and read-only

Given Sharing Rules are required for multi-region visibility
When a Claim__c record has Region__c = 'Northeast'
Then a criteria-based Sharing Rule grants Read access to the 'Northeast_Claims_Team' public group

Given the Credentialing Specialist role is configured
When assigned 'PSG_Credentialing_Specialist'
Then the user has Read/Write on Credentialing_Case__c and Provider__c (Person Account) but NO access to Claim__c or Payment__c objects

Given compliance audit requirements
When any permission set group assignment is made or removed
Then a Platform Event (Permission_Change_Event__e) is fired and logged to a custom Audit_Log__c record with fields: User__c, Changed_By__c, PSG_Name__c, Action__c (Assigned|Removed), Timestamp__c

## Solution

Role hierarchy built via Setup > Roles with five tiers: CEO > Regional VP > Business Unit Manager > Team Lead > Individual Contributor. Permission Sets created as granular units (PS_Claims_Base_Access, PS_Payment_View, PS_EOB_Generate, PS_SIU_Full_Access, PS_Credentialing_Full_Access) and grouped into Permission Set Groups using permissionsetgroup-meta.xml. Muting Permission Sets used within PSGs to suppress over-permissive base profile access. Criteria-based Sharing Rules deployed for Claim__c and Policy__c. Platform Event `Permission_Change_Event__e` published via Apex Trigger on PermissionSetAssignment object (after insert/delete). Subscribe logic handled by a Record-Triggered Flow on Audit_Log__c creation.
