## Description

As a Compliance Officer, I need a Permission Set Group-based access model for Claims Processing roles so that each team member has the minimum necessary access to PHI-adjacent claims data in accordance with HIPAA minimum necessary standards and Acme Insurance's internal data governance policy.

This story covers the creation of Permission Sets, Permission Set Groups, and Muting Permission Sets for the following roles:
- Claims Operations Manager
- Claims Adjudicator (read/edit Claim__c, Claim_Line__c)
- SIU Analyst (read-only on Claim__c with SIU_Referral_Flag__c visibility)
- Read-Only Auditor

Does NOT cover Provider Network or Credentialing roles — those are in INS-0050.

## Acceptance Criteria

Given the Permission Set Group 'PSG_Claims_Operations_Manager' is created,
When it is assigned to a test user in a sandbox,
Then the user can Create, Read, Edit, Delete on Claim__c and Claim_Line__c, and has access to all fields including SIU_Referral_Flag__c and Total_Paid_Amount__c.

Given the Permission Set Group 'PSG_Claims_Adjudicator' is created,
When it is assigned to a test user,
Then the user can Create, Read, and Edit Claim__c and Claim_Line__c but CANNOT Delete, and CANNOT view or edit SIU_Referral_Flag__c (muted via Muting Permission Set 'MPS_Hide_SIU_Fields').

Given the Permission Set Group 'PSG_SIU_Analyst' is created,
When assigned to a test user,
Then the user has Read-Only access on Claim__c including SIU_Referral_Flag__c, and cannot edit any Claim__c or Claim_Line__c record.

Given the Permission Set Group 'PSG_Read_Only_Auditor' is created,
When assigned to a test user,
Then the user has Read access to Claim__c, Claim_Line__c, Policy__c, and Payment__c objects but no Create, Edit, or Delete permissions on any object.

Given all Permission Set Groups are deployed,
When a Compliance Officer reviews the access matrix in a documented spreadsheet,
Then each PSG maps correctly to the documented RACI and no PSG grants access beyond its documented scope.

-Given profile-based permissions are used as the primary access model,
Then standard profiles are modified directly.- (Superseded — all access is Permission Set Group-based; standard profiles set to minimum/no object access)

## Solution

1. Create base Permission Sets (PS_Claims_Read, PS_Claims_Edit, PS_Claims_Delete, PS_SIU_Fields_Access) as granular building blocks.
2. Create Muting Permission Set 'MPS_Hide_SIU_Fields' that removes FLS on SIU_Referral_Flag__c and SIU_Notes__c.
3. Assemble Permission Set Groups in Setup > Permission Set Groups, adding appropriate base PSs and Muting PSs.
4. All PSGs, PSs, and Muting PSs deployed as Metadata API source (permissionsets/, permissionsetgroups/, mutingpermissionsets/ directories).
5. Assign PSGs to named test users in Full Sandbox using a Salesforce CLI data script for validation.
6. Document access matrix in Confluence and link to this story.
7. No Apex required. Object-level and FLS controlled entirely through declarative permission model.
