## Description

As a Claims Operations Manager, I need role-appropriate Permission Set Groups assigned to my team so that Claims Operations staff, Provider Network Specialists, and Credentialing Specialists each have least-privilege access to Salesforce objects relevant to their function — and cannot read or modify records outside their domain without explicit escalation.

## Acceptance Criteria

Given the Permission Set Group 'PSG_Claims_Operations' exists,
When assigned to a Claims Operations Manager profile user,
Then the user has Read/Create/Edit on Claim__c, Claim_Line__c, Payment__c, and Explanation_of_Benefits__c, and has NO access to Provider_Network_Agreement__c or Fee_Schedule__c.

Given the Permission Set Group 'PSG_Provider_Network_Specialist' exists,
When assigned to a Provider Network Specialist user,
Then the user has Read/Create/Edit on Provider_Network_Agreement__c, Group_Service_Agreement__c, Fee_Schedule__c, and Provider__c (Person Account), and Read-Only on Claim__c.

Given the Permission Set Group 'PSG_Credentialing_Specialist' exists,
When assigned to a Credentialing Specialist user,
Then the user has Read/Create/Edit on Credentialing_Case__c and Provider__c, and has NO Create/Edit on Payment__c or Claim__c.

Given the Permission Set Group 'PSG_SIU_Analyst' exists,
When assigned to an SIU team member,
Then the user has Read-Only access to Claim__c, Claim_Line__c, and Payment__c, and can create/edit a custom SIU_Flag__c junction object.

Given any of the above PSGs are assigned,
When the user navigates to Setup > Permission Set Groups,
Then the muting permission set within each PSG correctly suppresses any conflicting permissions inherited from the base profile.

-Given a custom Profile 'Claims_Ops_Profile' is used as the base, When PSG is assigned, Then profile-level FLS is overridden- (Superseded: all users will use the 'Minimum Access - Salesforce' base profile per updated security design.)

## Solution

Create three Permission Sets — PS_Claims_Core, PS_Provider_Network_Core, PS_Credentialing_Core — and one muting Permission Set per group. Bundle into Permission Set Groups via Setup > Permission Set Groups. Use the Permission Set Group Assignment screen to validate FLS and CRUD. Automated assignment will be handled by a Record-Triggered Flow on the User object that fires on creation, reading the User.Department field to map to the correct PSG via a custom metadata type (PSG_Mapping__mdt). Manual override available to System Admins via a custom LWC component on the User record page.
