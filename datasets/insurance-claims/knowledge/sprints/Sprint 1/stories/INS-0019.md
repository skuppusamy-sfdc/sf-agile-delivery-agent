## Description

As a Provider Network Specialist, I need the Provider_Network_Agreement__c (PNA) object deployed and related to both Provider__c and Fee_Schedule__c so that I can track the contractual relationship between Acme Insurance and each in-network provider, including effective dates, reimbursement basis, and the applicable fee schedule — enabling accurate claims adjudication in later sprints. Extends INS-0018 (Provider__c object must be deployed first).

## Acceptance Criteria

Given the Provider_Network_Agreement__c object is deployed,
When a PNA record is created,
Then the following fields are available: PNA_Number__c (Auto Number, format PNA-{000000}), Provider__c (Lookup to Account [Provider record type], Required), Facility__c (Lookup to Account [Facility record type]), Effective_Date__c (Date, Required), Termination_Date__c (Date), Status__c (Picklist: Draft, Active, Expired, Terminated, Pending Renewal), Reimbursement_Basis__c (Picklist: Fee Schedule, Percent of Billed Charges, Capitation, Case Rate, DRG), LOB__c (Multi-select Picklist: Medical, Dental, Vision, Workers Comp), Fee_Schedule__c (Lookup to Fee_Schedule__c), MCO__c (Lookup to Account [MCO record type]), Credentialing_Required__c (Checkbox, default true).

Given a PNA record where Termination_Date__c is populated and is in the past,
When the record is saved,
Then Status__c is automatically set to 'Expired' and a read-only banner displays: 'This agreement has expired. A renewal PNA must be created.'

Given a PNA record with Status__c = 'Active',
When the related Provider__c.Network_Status__c is not 'In-Network',
Then a validation rule blocks the PNA activation with the error: 'Provider must have an In-Network status before a PNA can be set to Active.'

Given the Fee_Schedule__c object,
When a Fee_Schedule__c record is created,
Then it must have: Fee_Schedule_Name__c (Text, Required), Effective_Date__c (Date, Required), Expiration_Date__c (Date), LOB__c (Picklist), Version__c (Text 10), and a child object Fee_Schedule_Line__c with CPT_Code__c, Allowed_Amount__c, and Modifier__c fields.

Given a PNA record is set to Status__c = 'Active',
When Credentialing_Required__c = true and the related Provider__c.Credentialing_Status__c != 'Active',
Then the activation is blocked with: 'Provider credentialing must be Active before the PNA can be activated.'

Given the PNA record page,
When viewed by a Provider Network Specialist,
Then the Fee Schedule Lines related list is visible showing CPT_Code__c and Allowed_Amount__c for the linked Fee_Schedule__c.

## Solution

Deploy Provider_Network_Agreement__c, Fee_Schedule__c, and Fee_Schedule_Line__c objects via Salesforce CLI metadata deployment. Validation rules implemented declaratively using cross-object formula fields referencing Provider__c.Credentialing_Status__c and Provider__c.Network_Status__c. Auto-expiry logic (setting Status__c to 'Expired' when Termination_Date__c < TODAY()) implemented via a Scheduled Record-Triggered Flow running nightly. Fee Schedule Lines related list configured on the PNA page layout via a custom lookup from Fee_Schedule_Line__c to Fee_Schedule__c. Conga CLM integration for PNA document generation scoped for Sprint 3. Custom metadata type (LOB_Reimbursement_Config__mdt) created to store LOB-specific reimbursement rules for use in adjudication engine in later sprints.
