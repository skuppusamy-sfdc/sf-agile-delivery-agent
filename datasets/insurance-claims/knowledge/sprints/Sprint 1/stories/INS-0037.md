## Description

As a Claims Operations Manager, I need the core Salesforce custom objects (Policy__c, Claim__c, Claim_Line__c, Payment__c) to be deployed with all required fields, record types, and page layouts so that the platform is ready to support end-to-end claims processing workflows in subsequent sprints.

## Acceptance Criteria

Given the Salesforce org is a fresh sandbox provisioned for Phase 1
When the deployment package is executed
Then Policy__c exists with fields: Policy_Number__c (Text, Unique, External ID), LOB__c (Picklist: Medical|Dental|Vision|Behavioral Health), Effective_Date__c (Date), Termination_Date__c (Date), TPA__c (Lookup: Account), Policy_Status__c (Picklist: Active|Lapsed|Cancelled|Pending)

Given Claim__c is created
When a new claim record is inserted
Then Claim__c has the following fields: Claim_Number__c (Auto Number), FNOL_Date__c (DateTime), Claim_Status__c (Picklist: New|In Review|Adjudicated|Denied|Paid|Closed|Voided), Policy__c (Master-Detail: Policy__c), Total_Billed_Amount__c (Currency), Total_Allowed_Amount__c (Currency), Total_Paid_Amount__c (Currency), COB_Flag__c (Checkbox), SIU_Referral__c (Checkbox)

Given Claim_Line__c is configured
When a claim line is associated to a Claim__c record
Then Claim_Line__c includes: CPT_Code__c (Text, 5-char), ICD10_Primary__c (Text), Service_Date__c (Date), Billed_Amount__c (Currency), Allowed_Amount__c (Currency), Line_Status__c (Picklist: Pending|Approved|Denied|Adjusted), Rendering_Provider__c (Lookup: Provider__c)

Given record types are required per LOB
When a Claim__c record is created
Then record types exist for: Medical_Claim, Dental_Claim, Vision_Claim, Behavioral_Health_Claim

-Given Payment__c uses a legacy junction object model
Then Payment__c should relate to both Claim__c and Policy__c via two Master-Detail fields- (superseded — Payment__c relates to Claim__c only via Master-Detail; Policy__c relationship is a Lookup)

Given the deployment is complete
When a System Administrator reviews the object manager
Then all objects pass Salesforce Field Audit Trail enablement check and no required fields are missing from compact layouts

## Solution

Deploy metadata via Salesforce DX scratch org pipeline. Custom objects created using object-meta.xml definitions. Record types defined per LOB in recordtype-meta.xml. Page layouts assigned per record type via profileObjectPermissions. Compact layouts configured for Claim__c and Policy__c to surface key fields in mobile and related list views. Field History Tracking enabled on Claim__c.Claim_Status__c, Claim__c.Total_Paid_Amount__c, and Policy__c.Policy_Status__c. Deployment validated against a CI pipeline using PMD static analysis and Salesforce CLI `sf project deploy validate`.
