## Description

As a Claims Operations Manager, I need the Claim__c and Claim_Line__c custom objects to be created in Salesforce with all Phase 1 required fields, validation rules, and record types so that downstream configuration for FNOL intake, adjudication workflows, and EOB generation can be built on a stable data model.

This story establishes the foundational schema only. Automation, page layouts, and permission assignments are covered in subsequent stories (INS-0047, INS-0049).

## Acceptance Criteria

Given the Claim__c object is created,
When a developer inspects the object in Setup,
Then the following fields exist with correct data types:
  - Claim__c.Claim_Number__c (Auto Number, format CLM-{0000000})
  - Claim__c.Status__c (Picklist: Draft, Submitted, In Review, Pending Info, Adjudicated, Denied, Closed, Voided)
  - Claim__c.LOB__c (Picklist: Medical, Dental, Vision, Pharmacy, Behavioral Health)
  - Claim__c.Date_of_Service__c (Date)
  - Claim__c.Date_Received__c (Date)
  - Claim__c.FNOL_Date__c (Date)
  - Claim__c.Policy__c (Lookup to Policy__c)
  - Claim__c.Provider__c (Lookup to Account where RecordType = Provider)
  - Claim__c.Facility__c (Lookup to Account where RecordType = Facility)
  - Claim__c.Total_Billed_Amount__c (Currency)
  - Claim__c.Total_Allowed_Amount__c (Currency)
  - Claim__c.Total_Paid_Amount__c (Currency, Read-Only via formula roll-up)
  - Claim__c.COB_Indicator__c (Checkbox)
  - Claim__c.TPA_Reference_Number__c (Text 50)
  - Claim__c.SIU_Referral_Flag__c (Checkbox, default false)
  - Claim__c.ICD10_Primary_Diagnosis__c (Text 10)
  - Claim__c.Claim_Source__c (Picklist: EDI 837, Portal, Manual, Fax, Phone)

Given the Claim_Line__c object is created,
When a developer inspects the object,
Then the following fields exist:
  - Claim_Line__c.Claim__c (Master-Detail to Claim__c)
  - Claim_Line__c.Line_Number__c (Number 3,0)
  - Claim_Line__c.CPT_Code__c (Text 10)
  - Claim_Line__c.ICD10_Diagnosis_Pointer__c (Text 20)
  - Claim_Line__c.Units__c (Number 5,2)
  - Claim_Line__c.Billed_Amount__c (Currency)
  - Claim_Line__c.Allowed_Amount__c (Currency)
  - Claim_Line__c.Paid_Amount__c (Currency)
  - Claim_Line__c.Denial_Reason_Code__c (Picklist: CO-4, CO-11, CO-16, CO-22, CO-97, PR-1, PR-2, OA-23)
  - Claim_Line__c.Status__c (Picklist: Pending, Approved, Denied, Adjusted)
  - Claim_Line__c.Fee_Schedule__c (Lookup to Fee_Schedule__c)
  - Claim_Line__c.Modifier_1__c (Text 2)
  - Claim_Line__c.Modifier_2__c (Text 2)

Given record types are required for Claim__c,
When record types are inspected,
Then two record types exist: 'Professional_Claim' and 'Institutional_Claim', each with an active page layout assignment.

Given validation rules are required,
When a user attempts to set Claim__c.Status__c to 'Adjudicated' without a value in Total_Allowed_Amount__c,
Then a validation error fires: 'Total Allowed Amount is required before a claim can be adjudicated.'

Given audit requirements,
When Field History Tracking is enabled on Claim__c,
Then Status__c, Total_Paid_Amount__c, SIU_Referral_Flag__c, and Total_Allowed_Amount__c are tracked.

## Solution

1. Create Claim__c and Claim_Line__c objects via Metadata API / scratch org source-tracked development in the ins-claims-core package.
2. All fields defined in YAML field definition spec and deployed via 'sf project deploy start'.
3. Master-Detail relationship on Claim_Line__c enables roll-up summary fields on Claim__c for Total_Billed_Amount__c and Total_Paid_Amount__c.
4. Record Types created with profiles mapped in recordTypeVisibilities metadata.
5. Validation Rules deployed as metadata (ValidationRule XML).
6. Field History Tracking enabled via object metadata historyRetentionPolicy.
7. No Apex in this story — pure declarative schema. Apex triggers for status transitions covered in INS-0049.
