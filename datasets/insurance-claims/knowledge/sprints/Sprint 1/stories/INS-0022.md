## Description

**Bug Report — Permission Model Defect**

**Environment:** Full Sandbox (Acme-INS-FSB)
**Discovered by:** Rachel Torres (QA) during Sprint 1 permission model UAT on 2024-01-18

**Steps to Reproduce:**
1. Log in to the Full Sandbox as a test user assigned only the 'PSG_Claims_Operations' Permission Set Group (base profile: Minimum Access - Salesforce).
2. Navigate to the Provider_Network_Agreement__c tab.
3. Open any existing PNA record.
4. Observe the Edit button is present and functional.
5. Modify the PNA record's Status__c field and click Save — save succeeds.

**Expected Behavior:** A user assigned only 'PSG_Claims_Operations' should have NO access (not even Read) to Provider_Network_Agreement__c records, as defined in INS-0016 acceptance criteria. The Edit button should not appear, and any direct URL navigation to a PNA record should return an 'Insufficient Privileges' error.

**Actual Behavior:** Claims Operations users can Edit Provider_Network_Agreement__c records, including changing Status__c to 'Active' or 'Terminated', which could result in unauthorized contract modifications and a potential compliance violation.

**Root Cause Hypothesis:** The base Permission Set 'PS_Claims_Core' may have inadvertently inherited object permissions for Provider_Network_Agreement__c from a cloned permission set that previously included PNA access. The muting permission set within PSG_Claims_Operations does not appear to be suppressing PNA CRUD permissions.

**Impact:** High — unauthorized contract modification risk. SIU and Compliance notified.

## Acceptance Criteria

As per Description.

Given a user assigned only 'PSG_Claims_Operations' (no other PSGs or permission sets),
When they navigate to any Provider_Network_Agreement__c record URL directly,
Then they receive 'Insufficient Privileges' and cannot view or edit the record.

Given the fix is deployed,
When the QA team re-runs the full permission matrix test suite in the Full Sandbox,
Then all 47 permission matrix test cases in the UAT script pass with no regressions to Claim__c, Claim_Line__c, or Payment__c access for Claims Operations users.

## Solution

Audit PS_Claims_Core permission set by exporting its metadata XML and reviewing the objectPermissions node for Provider_Network_Agreement__c. Remove any CRUD permissions for PNA from PS_Claims_Core. Verify the muting permission set 'MPS_Claims_Operations_Mute' has an explicit deny (allowCreate=false, allowRead=false, allowEdit=false, allowDelete=false) for Provider_Network_Agreement__c. Redeploy the corrected Permission Set and Permission Set Group metadata via Salesforce CLI to the Full Sandbox. Re-run the automated permission matrix test suite (Apex test class: PermissionMatrix_ClaimsOps_Test) and confirm all assertions pass. Promote fix to all downstream sandboxes after QA sign-off. Add a regression test case to the CI/CD pipeline to catch PNA access on the Claims Operations PSG in future deployments.
