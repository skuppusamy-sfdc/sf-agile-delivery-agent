## Description

**Bug Report — Claim__c Status Picklist Bypass via REST API**

**Environment:** Acme Insurance Sprint 1 Sandbox (API v59.0)

**Severity:** High — data integrity risk; invalid statuses break downstream adjudication Flow logic and EOB generation.

**Steps to Reproduce:**
1. Authenticate to the org via Connected App using OAuth 2.0 JWT flow (integration user: svc_claims_integration).
2. Identify an existing Claim__c record with Status__c = 'Submitted'.
3. Send a PATCH request to /services/data/v59.0/sobjects/Claim__c/{recordId} with body: { "Status__c": "INVALID_STATUS_XYZ" }
4. Observe the HTTP response.

**Expected Behavior:**
The API should return HTTP 400 with an INVALID_OR_NULL_FOR_RESTRICTED_PICKLIST error, rejecting the update because Claim__c.Status__c is defined as a Restricted Picklist in the object definition.

**Actual Behavior:**
The API returns HTTP 200 and the record is saved with Status__c = 'INVALID_STATUS_XYZ'. The restricted picklist constraint is not being enforced for the svc_claims_integration API user.

**Root Cause Hypothesis:**
The svc_claims_integration integration user's Permission Set (PS_Claims_API_Write) may have 'Restricted Picklist — Allow Any Value' implicitly enabled, or the Claim__c.Status__c field was not correctly flagged as Restricted during the Sprint 1 object build (see INS-0051 for original field setup). Additionally, the Record-Triggered Flow (INS_Claim_Status_Transition_Flow) that enforces valid status transitions only fires on UI-originated saves; API updates with AllOrNone=false bypass the before-save Flow entry criteria.

**Impact:**
- Claim__c records with invalid Status__c values cause NullPointerException in the INS_EOB_Generation_Flow when it evaluates the decision element branching on Status__c = 'Adjudicated'.
- Downstream Payment__c record creation via Platform Event (Claim_Adjudicated__e) fails silently because the event is never published for non-standard status values.
- Compliance Officer dashboard report (filtered on Status__c IN ('Submitted','In Review','Adjudicated','Paid','Denied','Closed')) omits affected records, creating a gap in LOB reporting.

## Solution

**Proposed Fix (two-pronged):**

1. **Field Correction:** Verify and re-deploy the Claim__c.Status__c field metadata to confirm Restricted='true' is set in the field XML (fields/Status__c.field-meta.xml). If the flag was dropped during a sandbox refresh or partial deployment, re-deploy via SFDX. Validate in Setup > Object Manager > Claim__c > Fields > Status__c that 'Restrict picklist to the values defined in the value set' checkbox is enabled.

2. **Apex Validation Layer:** Implement a before-insert/before-update Apex Trigger (ClaimStatusValidationTrigger) on Claim__c that maintains a static Set<String> of valid Status__c values sourced from a Custom Metadata Type (Claim_Status_Config__mdt) so valid values are configurable without code deployment. The trigger calls trigger.newMap records' Status__c against the allowed set and invokes addError() for any violation, ensuring API, Flow, and UI paths are all covered regardless of picklist restriction enforcement gaps.

3. **Integration User PSG Audit:** Review PS_Claims_API_Write for any 'Edit' access on restricted picklist fields that could override field-level restrictions; apply a Muting Permission Set if over-broad access is confirmed.

4. **Regression Test:** Add a Apex test class (ClaimStatusValidationTriggerTest) with a test method that asserts a DmlException is thrown when an invalid Status__c value is set via direct Apex DML, simulating the API path. Target 95%+ code coverage on the trigger handler class.
