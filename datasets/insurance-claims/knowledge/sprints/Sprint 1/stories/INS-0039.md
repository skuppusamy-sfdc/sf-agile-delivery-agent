## Description

As a Provider Relations Specialist, I need a Provider 360 FlexCard displayed on the Provider__c (Person Account) record page that surfaces key network participation status, active PNA details, credentialing case summary, and NPI/TIN verification badges so that I can resolve provider inquiries without navigating across multiple objects.

## Acceptance Criteria

Given a Provider__c record is open in the Lightning Experience
When the Provider 360 FlexCard loads
Then it displays: Provider Full Name, NPI (from NPI__c), TIN (masked to last 4 digits), Primary Specialty (Specialty__c), Network Status badge (In-Network|Out-of-Network|Pending|Terminated) derived from active Provider_Network_Agreement__c records

Given the provider has one or more active PNA records
When the FlexCard Network Panel is rendered
Then it lists each Provider_Network_Agreement__c with: PNA_Number__c, Effective_Date__c, Termination_Date__c, Network_Name__c, and a colored status pill (Green=Active, Yellow=Expiring within 90 days, Red=Terminated)

Given the provider has a Credentialing_Case__c record
When the Credentialing Summary section loads
Then it shows the most recent Credentialing_Case__c with fields: Case_Status__c, Credentialing_Type__c (Initial|Re-credentialing|CAQH_Verification), Expiration_Date__c, and a 'Days Until Expiry' calculated field

Given the provider's NPI is verified against NPPES
When the NPPES DataRaptor returns a match
Then a green 'NPI Verified' badge is displayed; if unverified, a yellow 'NPI Pending Verification' badge is shown with a timestamp of last check

Given a Provider Relations Specialist clicks 'Initiate Re-credentialing' on the FlexCard
When the action is triggered
Then a new Credentialing_Case__c record is created with Credentialing_Type__c = 'Re-credentialing', Status__c = 'Initiated', and Related_Provider__c populated with the current provider's ID

-Given the FlexCard includes a Claims History panel showing last 10 Claim__c records
Then claims data is fetched via inline SOQL on the FlexCard- (superseded — Claims History panel moved to separate INS-0051 story to manage page load performance)

## Solution

OmniStudio FlexCard built with multiple child cards (Network Status Card, Credentialing Card, NPI Verification Card) using DataRaptor Extract actions to query Provider_Network_Agreement__c and Credentialing_Case__c. TIN masking implemented via a DataRaptor Transform formula. NPPES integration DataRaptor calls an Integration Procedure that invokes a Named Credential HTTP callout to the NPPES API (GET /v2.1/providers?number={NPI}). Network status badge color logic handled via FlexCard State configuration with conditional CSS classes. 'Initiate Re-credentialing' action triggers an OmniScript that pre-populates and saves Credentialing_Case__c via a DataRaptor Turbo Action. FlexCard deployed to the Provider__c Lightning Record Page via App Builder.
