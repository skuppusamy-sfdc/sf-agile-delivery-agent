## Description

As a Claims Operations Manager, I need a structured Permission Set Group model deployed in the Salesforce org so that each role in the claims lifecycle has appropriately scoped access to objects, fields, and records without over-provisioning sensitive data.

This story covers the foundational permission architecture for Phase 1 roles: Claims Operations Manager, Provider Network Specialist, Credentialing Specialist, and Compliance Officer. Subsequent phases will layer in Broker Relations Coordinator, Assignment Coordinator, and SIU roles.

Extends INS-0041 (Org baseline configuration and profile strategy).

## Acceptance Criteria

Given the Claims Operations Manager Permission Set Group is assigned to a user,
When that user navigates to Claim__c records,
Then they have Read, Create, Edit access on Claim__c and can view all fields including Claim__c.Status__c, Claim__c.Total_Billed_Amount__c, Claim__c.LOB__c, and Claim__c.Adjudication_Date__c.

Given the Claims Operations Manager Permission Set Group is assigned,
When the user attempts to delete a Claim__c record,
Then the delete action is denied and a permission error is surfaced (delete is reserved for System Administrator profile only).

Given the Provider Network Specialist Permission Set Group is assigned,
When the user accesses Provider_Network_Agreement__c (PNA) records,
Then they have Read and Edit access on PNA fields including PNA.Effective_Date__c, PNA.Termination_Date__c, PNA.Contract_Status__c, and PNA.Fee_Schedule__c (lookup),
And they do NOT have visibility into Payment__c.Check_Amount__c or Payment__c.Bank_Routing_Number__c (field-level security enforced at PSG level).

Given the Credentialing Specialist Permission Set Group is assigned,
When the user opens a Credentialing_Case__c record,
Then they have full Read/Create/Edit access on Credentialing_Case__c including fields Credentialing_Case__c.CAQH_Profile_ID__c, Credentialing_Case__c.NPI__c, Credentialing_Case__c.TIN__c, Credentialing_Case__c.Credentialing_Status__c, and Credentialing_Case__c.Expiration_Date__c.

Given the Compliance Officer Permission Set Group is assigned,
When the user runs a report on Claim__c with filters on Claim__c.SIU_Referral_Flag__c = TRUE,
Then the report returns results and the user can view the Claim__c.SIU_Notes__c field,
And the Compliance Officer CANNOT edit Claim__c.Status__c directly (read-only FLS on that field for this PSG).

-Given a legacy 'Claims_Read_Only' profile exists, users on that profile should be migrated to the new PSG model in Sprint 1- (deprecated — legacy profile migration deferred to INS-0072 in Sprint 3).

Given all four Permission Set Groups are deployed to the sandbox,
When a System Administrator runs the Permission Set Group assignment validation Flow (INS_PSG_Validation_Flow),
Then the Flow confirms zero conflicts between muting permission sets and base PSG entitlements and posts a summary record to the Permission_Audit_Log__c object.

Given any of the Phase 1 PSGs are assigned to a test user in the UAT sandbox,
When the Compliance Officer reviews the Permission_Audit_Log__c,
Then each log entry captures Permission_Audit_Log__c.Assigned_User__c, Permission_Audit_Log__c.PSG_Name__c, Permission_Audit_Log__c.Assignment_Date__c, and Permission_Audit_Log__c.Assigned_By__c.

## Solution

1. Create four Permission Set Groups (PSGs) in the org: PSG_Claims_Ops_Manager, PSG_Provider_Network_Specialist, PSG_Credentialing_Specialist, PSG_Compliance_Officer.
2. Each PSG is composed of granular Permission Sets (e.g., PS_Claim_Read, PS_Claim_Edit, PS_PNA_Edit, PS_Payment_ReadOnly) to allow flexible assembly and future reuse across Phase 2 roles.
3. Muting Permission Sets are applied within each PSG to suppress over-broad permissions inherited from the base Salesforce Standard User profile (e.g., mute delete on Claim__c, mute edit on Payment__c.Bank_Routing_Number__c via FLS muting).
4. A Record-Triggered Flow on the PermissionSetAssignment standard object fires on insert to write a child Permission_Audit_Log__c record capturing assignment metadata for Compliance Officer reporting.
5. A separate scheduled Apex class (PSGConflictValidator) runs nightly in sandbox to assert no muting conflicts exist and logs results to Permission_Audit_Log__c with a Validation_Result__c field.
6. All PSGs and constituent Permission Sets are deployed via Salesforce DX metadata (force-app/main/default/permissionsets and permissionsetgroups directories) and included in the Phase 1 CI/CD pipeline validation.
7. FLS for sensitive Payment__c fields is enforced at the PSG level using Muting Permission Sets rather than profile-level restrictions to maintain profile simplicity.
