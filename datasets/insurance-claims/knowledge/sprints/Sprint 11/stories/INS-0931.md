## Description

As a Claims Operations Manager, I need the system to automatically validate that the active Fee_Schedule__c record associated with a Claim_Line__c falls within the service date range so that adjudicated payment amounts always reflect the correct contracted rate and we eliminate manual rate lookup errors during high-volume processing.

## Acceptance Criteria

Given a Claim_Line__c record is submitted for adjudication,
When the adjudication engine evaluates the Claim_Line__c.Service_Date__c,
Then the system must query Fee_Schedule__c where Fee_Schedule__c.Effective_Date__c <= Claim_Line__c.Service_Date__c AND Fee_Schedule__c.Termination_Date__c >= Claim_Line__c.Service_Date__c AND Fee_Schedule__c.LOB__c = Claim__c.Line_of_Business__c.

Given a matching Fee_Schedule__c record is found,
When the CPT code on Claim_Line__c.CPT_Code__c matches a Fee_Schedule_Line__c.CPT_Code__c,
Then Claim_Line__c.Allowed_Amount__c must be populated with Fee_Schedule_Line__c.Allowed_Rate__c and Claim_Line__c.Rate_Source__c must be set to 'Fee Schedule'.

Given no active Fee_Schedule__c record exists for the given service date and LOB,
When adjudication is attempted,
Then Claim_Line__c.Adjudication_Status__c must be set to 'Pended' and a Task must be created and assigned to the Claims Operations queue with Subject = 'Missing Fee Schedule – Manual Rate Review Required'.

Given a Fee_Schedule__c record has Fee_Schedule__c.Status__c = 'Draft',
When the adjudication engine evaluates rate applicability,
Then the Draft record must be excluded from rate resolution and a warning must be logged in Claim__c.Adjudication_Notes__c.

-Given multiple overlapping Fee_Schedule__c records exist for the same LOB and date range, the system selects the record with the lowest Allowed_Rate__c- (superseded: system now selects the most recently approved record per Fee_Schedule__c.Approved_Date__c descending).

Given a Claim_Line__c.CPT_Code__c is flagged as a bundled code,
When Fee Schedule lookup is performed,
Then the system must apply the bundled rate from Fee_Schedule_Line__c.Bundled_Rate__c and set Claim_Line__c.Bundling_Applied__c = true.

## Solution

Implement an Apex invocable method 'FeeScheduleResolutionService' called from a Record-Triggered Flow on Claim_Line__c (trigger: before update, when Adjudication_Status__c changes to 'In Review'). The Apex class performs a SOQL query against Fee_Schedule__c and child Fee_Schedule_Line__c with date range and LOB filters. Results are cached using Platform Cache (Org partition, key = LOB + ServiceDate) to reduce query volume during bulk adjudication runs. On resolution failure, a Platform Event (Fee_Schedule_Resolution_Failed__e) is published to trigger Task creation via a separate Automation Flow. Unit test coverage target: 90%+. Existing Fee Schedule data model extended with Fee_Schedule__c.Approved_Date__c field (new custom field, migration script required for historical records).
