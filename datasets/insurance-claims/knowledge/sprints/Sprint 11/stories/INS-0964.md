## Description

**Bug Report — Fee Schedule CPT Lookup Collision**

**Steps to Reproduce:**
1. Ensure two Fee_Schedule__c records exist with Status__c = 'Active', the same LOB__c (e.g., 'Commercial HMO'), and overlapping CPT code entries (e.g., CPT 99213) but different Provider_Network_Agreement__c lookups.
2. Submit a Claim__c with a Claim_Line__c where CPT_Code__c = '99213' and the associated Policy__c.LOB__c = 'Commercial HMO'.
3. Trigger adjudication by setting Claim__c.Status__c = 'In Adjudication'.
4. Observe Claim_Line__c.Applied_Fee_Schedule__c and Claim_Line__c.Allowed_Amount__c after adjudication completes.

**Expected Behavior:**
The adjudication engine resolves the fee schedule by matching Fee_Schedule__c to the provider's active Provider_Network_Agreement__c (PNA.Provider__c = Claim__c.Rendering_Provider__c) and applies the rate from that specific schedule.

**Actual Behavior:**
The SOQL query in FeeScheduleLookupService.cls uses ORDER BY CreatedDate DESC LIMIT 1, which selects the most recently created Fee_Schedule__c record regardless of PNA association. This causes Claim_Line__c.Allowed_Amount__c to reflect the wrong contracted rate — in one production case observed by Rachel Torres (Claims Operations Manager), a $145.00 allowed amount was applied instead of the correct $112.50, resulting in a $32.50 overpayment on Payment__c INS-PAY-004821.

**Environment:** Production & Full Sandbox
**Frequency:** Reproducible 100% when two active schedules share a CPT code under the same LOB.

## Acceptance Criteria

As per Description.

Additional validation: After fix deployment, re-adjudicate Claim__c INS-CLM-008834 in sandbox and confirm Claim_Line__c.Allowed_Amount__c = $112.50 and Claim_Line__c.Applied_Fee_Schedule__c references the PNA-matched schedule.

## Solution

1. Update FeeScheduleLookupService.cls to include a JOIN condition on Provider_Network_Agreement__c: SOQL updated to filter WHERE Fee_Schedule__c.Provider_Network_Agreement__c IN (SELECT Id FROM Provider_Network_Agreement__c WHERE Provider__c = :renderingProviderId AND Status__c = 'Active').
2. Add a secondary sort: ORDER BY Fee_Schedule__c.Effective_Start_Date__c DESC LIMIT 1 to handle edge cases where a provider has multiple historical PNA-linked schedules.
3. Add a unit test (FeeScheduleLookupServiceTest) with a scenario covering dual-active-schedule CPT collision — assert correct PNA-matched schedule is returned.
4. Deploy fix to Full Sandbox for regression testing; Rachel Torres to validate against cloned production data set.
5. Post-deploy: identify and queue any Claim_Line__c records in 'Adjudicated' status within the last 90 days where Applied_Fee_Schedule__c may be incorrect — create a Data Fix Report for Claims Operations Manager review.
