## Description

As a Claims Operations Manager, I need the system to automatically validate that the active Fee_Schedule__c record in effect on the date of service is applied to each Claim_Line__c during adjudication, so that reimbursement calculations reflect the correct contracted rates and manual rate-lookup errors are eliminated.

Background: Currently, adjusters manually cross-reference Fee_Schedule__c effective dates against Claim_Line__c.Date_of_Service__c, leading to an estimated 4% mis-pricing rate on professional claims. This story extends the adjudication engine introduced in INS-0846.

## Acceptance Criteria

Given a Claim_Line__c record enters adjudication (Claim__c.Status__c transitions to 'In Adjudication'),
When the system evaluates the applicable Fee_Schedule__c,
Then it must select the Fee_Schedule__c record where Fee_Schedule__c.Effective_Date__c <= Claim_Line__c.Date_of_Service__c AND Fee_Schedule__c.Expiration_Date__c >= Claim_Line__c.Date_of_Service__c AND Fee_Schedule__c.LOB__c = Claim__c.Line_of_Business__c.

Given a matching Fee_Schedule__c is found,
When the CPT code on Claim_Line__c.CPT_Code__c is looked up in the fee schedule rate table,
Then Claim_Line__c.Allowed_Amount__c must be populated with the Fee_Schedule_Rate__c.Allowed_Amount__c value and Claim_Line__c.Fee_Schedule__c (lookup) must be stamped with the resolved Fee_Schedule__c record ID.

Given no active Fee_Schedule__c exists for the date of service and LOB combination,
When adjudication attempts to resolve the rate,
Then Claim__c.Status__c must be set to 'Pended' with Claim__c.Pend_Reason__c = 'No Active Fee Schedule' and a Task must be created and assigned to the Claims Operations queue.

Given a Fee_Schedule__c record has Fee_Schedule__c.Schedule_Type__c = 'Medicare-Based' (percent-of-Medicare),
When the allowed amount is calculated,
Then Claim_Line__c.Allowed_Amount__c = CMS_Medicare_Rate__c.Base_Rate__c * Fee_Schedule__c.Medicare_Percentage__c / 100, rounded to two decimal places.

-Given multiple overlapping Fee_Schedule__c records exist for the same LOB and date range, the system selects the record with the highest Fee_Schedule__c.Priority__c value.- (Superseded: priority field deprecated in favor of Fee_Schedule__c.Precedence_Rank__c per data model update INS-0861)

Given multiple Fee_Schedule__c records match after precedence evaluation,
When Claim_Line__c adjudication runs,
Then the record with the lowest Fee_Schedule__c.Precedence_Rank__c integer value is selected (1 = highest precedence).

Given adjudication completes successfully,
When all Claim_Line__c records on a Claim__c are adjudicated,
Then Claim__c.Total_Allowed_Amount__c must equal the sum of all child Claim_Line__c.Allowed_Amount__c values and a Platform Event (Fee_Schedule_Applied__e) must be published with the Claim ID and Fee Schedule ID.

## Solution

Implement an Apex invocable method (FeeScheduleResolver.cls) called from a Record-Triggered Flow on Claim_Line__c (after update, when Status__c changes to 'Adjudication Ready'). The resolver performs a SOQL query with date-range filtering on Fee_Schedule__c and Fee_Schedule_Rate__c using CPT code and LOB as composite keys. A separate Record-Triggered Flow on Claim__c aggregates child Claim_Line__c.Allowed_Amount__c into Claim__c.Total_Allowed_Amount__c via a roll-up triggered after all lines are processed. Platform Event (Fee_Schedule_Applied__e) published via Apex to downstream Payment Engine. CMS Medicare base rates are synced nightly via a scheduled DataRaptor Extract integration pulling from the CMS pricing API. Unit test coverage target: 90%+.
