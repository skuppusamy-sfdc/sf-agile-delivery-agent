## Description

**Bug Report — Fee Schedule LOB Mismatch on Claim Line Adjudication**

**Environment:** Acme Insurance Production Sandbox (Sprint 11 Refresh)

**Steps to Reproduce:**
1. Create two Fee_Schedule__c records for the same Provider_Network_Agreement__c: one with LOB__c = 'Commercial' and one with LOB__c = 'Medicare Advantage', both containing CPT code 99213, both with overlapping effective date ranges.
2. Create a Claim__c with Line_of_Business__c = 'Medicare Advantage'.
3. Add a Claim_Line__c with CPT_Code__c = '99213' and Date_of_Service__c within both fee schedule effective ranges.
4. Save the Claim_Line__c and observe Claim_Line__c.Applied_Fee_Schedule__c and Claim_Line__c.Allowed_Amount__c.

**Expected Behavior:** Claim_Line__c.Applied_Fee_Schedule__c resolves to the Fee_Schedule__c with LOB__c = 'Medicare Advantage' and Allowed_Amount__c reflects the Medicare Advantage rate.

**Actual Behavior:** Claim_Line__c.Applied_Fee_Schedule__c resolves to the Fee_Schedule__c with LOB__c = 'Commercial' (the record with the lower CreatedDate), and Allowed_Amount__c is populated with the Commercial rate — resulting in incorrect adjudication and potential overpayment.

**Root Cause (Preliminary):** The SOQL query in FeeScheduleRateResolver Apex class is missing the LOB__c filter in the WHERE clause. The ORDER BY clause sorts by CreatedDate ASC LIMIT 1, which returns the first-created record regardless of LOB.

**Impact:** HIGH — any multi-LOB provider with shared CPT codes is affected. Estimated 340 Claim_Line__c records adjudicated incorrectly in the last 14 days.

## Acceptance Criteria

As per Description. Post-fix verification: repeat reproduction steps and confirm Claim_Line__c.Applied_Fee_Schedule__c.LOB__c = Claim__c.Line_of_Business__c for all test scenarios including Commercial, Medicare Advantage, and Medicaid LOBs. Run data remediation script on affected Claim_Line__c records and confirm re-adjudication queue is populated.

## Solution

Fix the SOQL query in FeeScheduleRateResolver.cls to include AND LOB__c = :claimLOB in the WHERE clause. Update ORDER BY to use Effective_Date__c DESC LIMIT 1 to prefer the most recently effective schedule. Add a unit test method in FeeScheduleRateResolverTest.cls covering multi-LOB same-CPT scenarios with assertions on Applied_Fee_Schedule__c.LOB__c. Deploy a one-time data fix Apex script to re-queue affected Claim_Line__c records (Status__c reset to 'Pending Adjudication') identified by a SOQL query joining Applied_Fee_Schedule__c.LOB__c != Claim__r.Line_of_Business__c.
