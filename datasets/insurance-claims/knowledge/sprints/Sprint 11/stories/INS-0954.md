## Description

As a Claims Operations Manager, I need the system to automatically apply the correct Fee_Schedule__c record to each Claim_Line__c based on the service date and provider's active PNA effective dates, so that adjudication always uses the rate in force at the time of service and eliminates manual rate lookups that cause payment errors.

## Acceptance Criteria

Given a Claim_Line__c record is saved with a Service_Date__c value,
When the adjudication engine evaluates the line,
Then the system queries Fee_Schedule__c where Fee_Schedule__c.Effective_Date__c <= Claim_Line__c.Service_Date__c AND Fee_Schedule__c.Expiration_Date__c >= Claim_Line__c.Service_Date__c AND Fee_Schedule__c.LOB__c = Claim__c.Line_of_Business__c,
And sets Claim_Line__c.Applied_Fee_Schedule__c = the matched Fee_Schedule__c record ID.

Given multiple Fee_Schedule__c records overlap for the same CPT code and date range,
When the system resolves the applicable rate,
Then the record with the highest Fee_Schedule__c.Priority__c value is selected and Claim_Line__c.Rate_Conflict_Flag__c is set to TRUE with a warning surfaced on the adjudicator's FlexCard.

Given no matching Fee_Schedule__c record exists for the Claim_Line__c.CPT_Code__c on the service date,
When adjudication attempts to price the line,
Then Claim_Line__c.Status__c is set to 'Pended - Missing Rate' and a Case is auto-created with Subject = 'Fee Schedule Gap: [CPT Code]' assigned to the Provider Network Specialist queue.

Given a Fee_Schedule__c record has Fee_Schedule__c.Rate_Type__c = 'Percentage of Medicare',
When the allowed amount is calculated,
Then Claim_Line__c.Allowed_Amount__c = CMS_Medicare_Rate__c * Fee_Schedule__c.Percentage_Factor__c and the source is logged in Claim_Line__c.Rate_Calculation_Notes__c.

-Given the provider NPI is not matched to a PNA, default to the base fee schedule- (superseded by INS-0941 network gap logic)

Given a Claim_Line__c is reprocessed after a retroactive Fee_Schedule__c rate change,
When the recalculation runs,
Then Claim_Line__c.Reprocessed_Flag__c = TRUE, the prior allowed amount is stored in Claim_Line__c.Prior_Allowed_Amount__c, and a Payment_Adjustment__c child record is created.

## Solution

Implement a Record-Triggered Flow on Claim_Line__c (trigger: before save, on create and update) that invokes an Apex invocable method 'FeeScheduleResolutionService.resolve()'. The Apex class queries Fee_Schedule__c with a SOQL covering index on Effective_Date__c, Expiration_Date__c, CPT_Code__c, and LOB__c. A custom metadata type (Fee_Schedule_Config__mdt) controls priority tie-breaking rules. Platform Events (Fee_Schedule_Applied__e) are fired post-resolution for downstream ERA generation. Unit test coverage target: 95%. Reprocessing batch job uses Apex Batch with scope 200, invoked via Scheduled Flow nightly.
