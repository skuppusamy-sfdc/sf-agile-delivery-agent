## Description

**Bug Report – Fee Schedule Boundary Date Defect**

**Environment:** Acme Insurance Sandbox (Sprint 11 Full Copy)
**Severity:** High – Financial Impact

**Steps to Reproduce:**
1. Create a Fee_Schedule__c record with Effective_Date__c = 01/01/2024 and Expiration_Date__c = 03/31/2024, Status__c = 'Expired'.
2. Create a second Fee_Schedule__c record for the same CPT code with Effective_Date__c = 04/01/2024, Status__c = 'Active'.
3. Create a Claim__c and add a Claim_Line__c with Date_of_Service__c = 03/31/2024.
4. Trigger adjudication by setting Claim__c.Status__c = 'Ready for Adjudication'.
5. Observe Claim_Line__c.Allowed_Amount__c and Fee_Schedule_Lookup_Status__c after processing.

**Expected Behavior:** The system should match the Fee_Schedule__c record with Expiration_Date__c = 03/31/2024 (inclusive boundary), populate the correct allowed amount from that record, and set Fee_Schedule_Lookup_Status__c = 'Matched'.

**Actual Behavior:** The SOQL query in FeeScheduleLookupService.cls uses a strict less-than operator (`< Date_of_Service__c`) on Expiration_Date__c, causing the boundary date record to be excluded. The system falls through to 'No Match Found' and pends the claim incorrectly, resulting in Claim__c.Status__c = 'Pended – Fee Schedule Review' and downstream payment delay.

**Root Cause (Preliminary):** Line 47 of FeeScheduleLookupService.cls: `WHERE Expiration_Date__c < :dos` should be `WHERE Expiration_Date__c >= :dos`.

**Impact:** Any Claim_Line__c with a DOS equal to a fee schedule expiration date will be incorrectly pended. Estimated 120–200 claims per month affected based on schedule turnover cadence.

## Acceptance Criteria

As per Description. Additional verification: after fix, re-process a Claim_Line__c with Date_of_Service__c exactly equal to Fee_Schedule__c.Expiration_Date__c and confirm Claim_Line__c.Fee_Schedule_Lookup_Status__c = 'Matched' and Claim_Line__c.Allowed_Amount__c is correctly populated. Regression test: DOS one day after expiration must still return 'No Match Found' if no subsequent schedule exists.

## Solution

Fix SOQL predicate in FeeScheduleLookupService.cls line 47: change `Expiration_Date__c < :dos` to `Expiration_Date__c >= :dos`. Add unit test method `testBoundaryDateExact()` and `testBoundaryDatePlusOne()` to FeeScheduleLookupServiceTest.cls to cover inclusive and exclusive boundary scenarios. Validate fix in Full Copy sandbox against historical claim data before promoting to production. Coordinate with Claims Operations Manager (Sarah Chen) to identify and re-trigger all claims pended since 04/01/2024 due to this defect using a Data Loader re-processing script.
