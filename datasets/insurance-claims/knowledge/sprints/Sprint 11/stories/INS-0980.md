## Description

**Bug Report**

**Summary:** When a Fee_Schedule__c record's Status__c is updated to 'Approved' and one or more Claim_Line__c records are adjudicated within the same Apex transaction (e.g., via a batch process or data load), the FeeScheduleResolver invocable method (introduced in INS-0977) retrieves the pre-commit state of the Fee_Schedule__c from the SOQL query cache, causing Claim_Line__c.Allowed_Amount__c to be populated with the prior (Draft) rate rather than the newly approved rate.

**Environment:** Full Sandbox (ACM-FS-02), API v59.0

**Steps to Reproduce:**
1. Ensure a Fee_Schedule__c record exists with Status__c = 'Draft' and at least one Fee_Schedule_Line__c with Allowed_Rate__c = 85.00.
2. Via Apex Execute Anonymous, update Fee_Schedule__c.Status__c = 'Approved' and simultaneously trigger adjudication on a Claim_Line__c with a matching CPT code and service date in the same transaction scope.
3. After the transaction commits, inspect Claim_Line__c.Allowed_Amount__c.

**Expected Behavior:** Claim_Line__c.Allowed_Amount__c = 85.00 (the approved rate from the now-active Fee_Schedule__c).

**Actual Behavior:** Claim_Line__c.Allowed_Amount__c = 0.00 or retains a previously cached/draft value, and Claim_Line__c.Fee_Schedule_Applied__c remains null.

**Frequency:** Reproducible 100% in batch adjudication scenarios; intermittent (~40%) in single-record UI updates.

**Impact:** Claim lines are being pended with Pend_Reason__c = 'No Active Fee Schedule' incorrectly, creating false-positive work items in the Provider Network Specialist queue. Approximately 230 Claim_Line__c records in the ACM-FS-02 sandbox are affected. Production risk flagged as HIGH pending fix.

## Solution

Root cause: SOQL query inside FeeScheduleResolver is executing against the database snapshot prior to the Fee_Schedule__c DML commit due to Apex's within-transaction read consistency. Fix approach: Refactor FeeScheduleResolver to use 'FOR UPDATE' SOQL locking hint on the Fee_Schedule__c query to force a fresh read, OR restructure the batch process to enforce a two-phase commit — Phase 1 commits Fee Schedule approval, Phase 2 (separate transaction/queueable) triggers Claim_Line__c adjudication. Recommended approach: Implement a Queueable Apex class 'ClaimLineAdjudicationQueueable' that is enqueued from the Fee Schedule approval Record-Triggered Flow (after-save context), ensuring adjudication always runs post-commit. Add a custom setting 'FeeSchedule_Adjudication_Async__c' (checkbox, default TRUE) to gate the async behavior for controlled rollout. Backfill script (Apex batch) to re-adjudicate the ~230 affected Claim_Line__c records in sandbox; production backfill to be executed during off-peak window with Claims Operations Manager sign-off.
