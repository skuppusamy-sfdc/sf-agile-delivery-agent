## Description

**Bug Report**

**Summary:** When a Fee_Schedule__c record's Status__c is changed to 'Published' and the record is saved more than once within a short window (e.g., due to a user double-clicking Save or an automated process re-saving the record), duplicate Provider_Change_Notification__c records are created for each affected provider, resulting in providers receiving multiple identical rate change notifications.

**Steps to Reproduce:**
1. Open an existing Fee_Schedule__c record with Status__c = 'Draft' that has at least 3 active PNA records linked.
2. Change Status__c to 'Published'.
3. Rapidly click Save twice in quick succession (or simulate via API by sending two PATCH requests within 2 seconds).
4. Navigate to a related provider's Provider_Change_Notification__c related list.
5. Observe the number of PCN records created.

**Expected Behavior:** Exactly one Provider_Change_Notification__c record per affected provider is created regardless of how many times the Fee_Schedule__c record is saved with Status__c = 'Published'.

**Actual Behavior:** Two or more duplicate PCN records are created per provider, each triggering a separate email notification. Providers are reporting receiving 2-4 copies of the same rate change notice.

**Environment:** Production, Full Sandbox (FS-01)
**Frequency:** Intermittent — reproducible ~60% of the time under rapid save conditions
**Impact:** Medium — provider relations complaints; potential compliance concern if duplicate notifications imply multiple rate changes

## Acceptance Criteria

As per Description.

Additionally: Given a Fee_Schedule__c record is saved with Status__c = 'Published' N times within a 60-second window, When PCN generation logic evaluates the record, Then exactly one PCN record per affected provider must exist after all saves complete. A duplicate prevention mechanism must check for existing PCN records with PCN.Fee_Schedule__c = the current record Id and PCN.Notification_Type__c = 'Rate Change' before inserting new records.

## Solution

Root cause: The Record-Triggered Flow on Fee_Schedule__c fires on every save where Status__c = 'Published', without checking if PCN records already exist for the same Fee_Schedule__c. Fix: Add a Decision element at the start of the flow that queries COUNT of existing Provider_Change_Notification__c records where Fee_Schedule__c = {!$Record.Id} AND Notification_Type__c = 'Rate Change'. If count > 0, exit the flow without creating new PCNs. Additionally, implement an Apex duplicate rule using a before-insert trigger on Provider_Change_Notification__c that checks for existing records with the same Fee_Schedule__c + Provider__c combination and throws a handled exception with a user-friendly error message. Consider adding a Fee_Schedule__c.PCN_Generated__c checkbox field as a fast-check gate to short-circuit the flow before hitting SOQL.
