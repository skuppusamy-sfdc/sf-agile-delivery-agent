## Description

**Bug Report**

**Summary:** The fee schedule resolution logic introduced in INS-0954 incorrectly applies Fee_Schedule__c records whose Effective_Date__c is in the future when no current-dated record exists, causing Claim_Line__c records to be priced at incorrect future rates.

**Steps to Reproduce:**
1. Create a Fee_Schedule__c record with Effective_Date__c = TODAY + 30 days, Status__c = 'Published', CPT_Code__c = '99213', LOB__c = 'Commercial'.
2. Ensure no other Fee_Schedule__c record exists for CPT 99213 / Commercial with a current effective date.
3. Create a Claim__c with Line_of_Business__c = 'Commercial' and a Claim_Line__c with Service_Date__c = TODAY and CPT_Code__c = '99213'.
4. Trigger adjudication by setting Claim__c.Status__c = 'In Adjudication'.
5. Observe Claim_Line__c.Applied_Fee_Schedule__c — it is populated with the future-dated record.

**Expected Result:** Claim_Line__c.Status__c = 'Pended - Missing Rate' and no Fee_Schedule__c applied, per AC #3 of INS-0954.

**Actual Result:** Claim_Line__c.Applied_Fee_Schedule__c is set to the future-dated Fee_Schedule__c record and Claim_Line__c.Allowed_Amount__c is calculated using the future rate.

**Environment:** Full Sandbox (AcmeIns-FS1)
**Frequency:** Reproducible 100%
**Impact:** High — claims are being overpaid or underpaid depending on rate direction.

## Acceptance Criteria

As per Description. The SOQL in FeeScheduleResolutionService.resolve() must include the condition Fee_Schedule__c.Effective_Date__c <= TODAY in all execution paths, including the fallback query branch introduced in the 11/14 hotfix commit.

## Solution

Fix the SOQL fallback branch in FeeScheduleResolutionService.resolve() Apex class — the secondary query executed when the primary date-range query returns zero results is missing the Effective_Date__c <= :serviceDate filter. Add the missing WHERE clause predicate. Add a regression test method 'testFutureDatedFeeScheduleNotApplied()' to FeeScheduleResolutionServiceTest. Deploy fix to sandbox for QA validation before promoting to UAT. Update the custom metadata type (Fee_Schedule_Config__mdt) field 'Allow_Future_Rate_Fallback__c' default to FALSE to prevent recurrence via configuration.
