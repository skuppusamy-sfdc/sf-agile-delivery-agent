## Description

**Bug Report — Fee Schedule Expiration Date Not Auto-Stamped on Supersession**

**Environment:** Acme Insurance Full Sandbox (FS-01), API v59.0

**Steps to Reproduce:**
1. Navigate to an active Fee_Schedule__c record (e.g., FS-2024-PROF-001) with Status__c = 'Active' and Expiration_Date__c = null.
2. Create a new Fee_Schedule__c record with the same LOB__c, Schedule_Type__c = 'Standard', and Effective_Date__c = today + 1 day.
3. Set the new record's Status__c = 'Active' and save.
4. Observe the original Fee_Schedule__c record (FS-2024-PROF-001).

**Expected Behavior:** The original Fee_Schedule__c.Expiration_Date__c should be automatically set to (new record Effective_Date__c - 1 day) and Fee_Schedule__c.Status__c should transition to 'Superseded'.

**Actual Behavior:** The original record retains Status__c = 'Active' and Expiration_Date__c remains null. Both records show as 'Active' simultaneously, causing the FeeScheduleResolver (INS-0876) to return duplicate matches and throw an unhandled AuraHandledException on Claim_Line__c adjudication.

**Impact:** P1 adjudication failures observed on 14 Claim__c records in the sandbox regression suite. Potential production impact if fee schedule versioning is used before fix is deployed.

**Root Cause (preliminary):** The Record-Triggered Flow (RTF-FeeSchedule-Supersession) has a missing entry condition — it fires only when Effective_Date__c is changed, but not when Status__c is directly set to 'Active' on a new record insert.

## Acceptance Criteria

As per Description. Verify that after fix: (1) Original Fee_Schedule__c.Status__c = 'Superseded' and Expiration_Date__c = new record Effective_Date__c - 1 day. (2) FeeScheduleResolver returns exactly one Fee_Schedule__c match for a given LOB and date. (3) Regression: no existing Active fee schedules with null Expiration_Date__c remain in org (data fix script required).

## Solution

Fix the Record-Triggered Flow (RTF-FeeSchedule-Supersession) entry condition to trigger on BOTH field changes: Effective_Date__c IS CHANGED OR Status__c IS CHANGED TO 'Active'. Add a second flow path for new record inserts (triggerType = CREATE) with the same supersession logic. Additionally, add a SOQL-based duplicate detection in the FeeScheduleResolver Apex class to throw a descriptive ClaimsProcessingException (rather than AuraHandledException) if multiple active schedules are found, logging the conflict to a custom Fee_Schedule_Conflict__c object for ops team review. Deploy a one-time data fix script (Anonymous Apex) to back-fill Expiration_Date__c and Status__c on all currently conflicting records in production.
