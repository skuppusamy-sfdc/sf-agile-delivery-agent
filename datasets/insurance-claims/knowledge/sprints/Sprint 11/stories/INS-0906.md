## Description

**Bug Report — PCN Not Generated on Batch Fee Schedule Expiration Updates**

**Environment:** Acme Insurance Full Sandbox

**Steps to Reproduce:**
1. Prepare a CSV file with 50+ Fee_Schedule__c records, updating Expiration_Date__c to a date within 60 days from today.
2. Import the CSV via Salesforce Data Loader using the Update operation with the API user 'Integration_User_FSUpdate'.
3. After import completes, query Provider_Change_Notification__c for PCN_Type__c = 'Rate Expiration Warning' related to the updated Fee_Schedule__c records.

**Expected Behavior:** A Provider_Change_Notification__c record with PCN_Type__c = 'Rate Expiration Warning' should be created for each updated Fee_Schedule__c record where the new Expiration_Date__c is within 60 days, consistent with behavior when records are updated via the UI.

**Actual Behavior:** Zero Provider_Change_Notification__c records are created. No errors are reported in the Data Loader log. The Fee_Schedule__c Expiration_Date__c fields are updated correctly.

**Root Cause (Preliminary):** The Record-Triggered Flow ('Fee_Schedule_Expiration_PCN_Flow') responsible for PCN creation has its trigger context set to run only on 'Created' events, not 'Updated' events. Additionally, the Integration_User_FSUpdate API profile has 'Bypass Flows' custom permission enabled, which suppresses all Flow execution during batch imports.

**Impact:** MEDIUM-HIGH — bulk contract rate renegotiation cycles performed by the Provider Network Operations team via Data Loader are silently bypassing PCN creation, leaving contract managers uninformed of imminent expirations loaded in batch.

## Acceptance Criteria

As per Description. Post-fix verification: re-run Data Loader import with 10 test Fee_Schedule__c records updating Expiration_Date__c to within 60 days and confirm Provider_Change_Notification__c records are created for each. Confirm Integration_User_FSUpdate no longer has 'Bypass Flows' custom permission. Confirm the Record-Triggered Flow trigger fires on both Created and Updated contexts. Regression test UI-based updates to ensure no duplicate PCNs are created.

## Solution

Update the Record-Triggered Flow 'Fee_Schedule_Expiration_PCN_Flow' trigger conditions to include the 'Updated' event with an entry condition checking ISCHANGED(Expiration_Date__c). Remove the 'Bypass Flows' custom permission from the Integration_User_FSUpdate API profile — if batch performance is a concern, migrate the PCN creation logic to an Apex Trigger (after insert, after update) with a feature flag (Custom Metadata: Flow_vs_Apex_PCN__mdt) allowing runtime toggle between Flow and Apex execution paths. Add a Process Exception Log entry for any PCN creation failures during batch context. Update the integration runbook to prohibit 'Bypass Flows' on API integration profiles without Change Advisory Board approval.
