## Description

**Bug Summary:** Claim_Line__c.Allowed_Amount__c is being populated with the rate from an expired Fee_Schedule__c version instead of the currently active version when multiple Fee_Schedule__c records exist for the same CPT_Code__c and Provider_Network_Agreement__c.

**Steps to Reproduce:**
1. Create two Fee_Schedule__c records for the same CPT_Code__c (e.g., '99213') and same PNA:
   - Record A: Status__c = 'Expired', Termination_Date__c = 2024-12-31, Allowed_Amount__c = $85.00
   - Record B: Status__c = 'Active', Effective_Date__c = 2025-01-01, Allowed_Amount__c = $92.50
2. Create a Claim__c with Service_Date__c = 2025-02-15 linked to the same PNA.
3. Add a Claim_Line__c with CPT_Code__c = '99213'.
4. Trigger adjudication by setting Claim__c.Status__c = 'Ready for Adjudication'.
5. Observe Claim_Line__c.Allowed_Amount__c after adjudication.

**Expected Result:** Claim_Line__c.Allowed_Amount__c = $92.50 (from the Active Fee_Schedule__c record with Effective_Date__c <= Service_Date__c)

**Actual Result:** Claim_Line__c.Allowed_Amount__c = $85.00 (from the Expired Fee_Schedule__c record — appears the SOQL query is not filtering by Status__c = 'Active' and is returning the first record by CreatedDate ASC)

**Impact:** Overpayments or underpayments on adjudicated claims. Identified in UAT by Claims Operations Manager. Affects all claims processed since INS-0917 deployment on 2025-03-01. Estimated 47 Claim_Line__c records impacted in Staging.

**Environment:** Full Sandbox (CS47). Reproducible 100% of the time when duplicate CPT/PNA fee schedule versions exist.

## Acceptance Criteria

As per Description.

Additional regression criteria:
Given the fix is deployed
When adjudication runs on a Claim_Line__c with Service_Date__c = 2025-02-15 and two Fee_Schedule__c records exist for the same CPT_Code__c and PNA (one Expired, one Active)
Then Claim_Line__c.Allowed_Amount__c must equal the Allowed_Amount__c from the Fee_Schedule__c record where Status__c = 'Active' AND Effective_Date__c <= Service_Date__c AND (Termination_Date__c = null OR Termination_Date__c >= Service_Date__c)

Given the 47 impacted Claim_Line__c records in production
When the data remediation script runs
Then all affected records must have Allowed_Amount__c corrected and a Claim_Line__c.Correction_Note__c field stamped with 'Corrected per INS-0950 remediation - [timestamp]'

## Solution

**Root Cause:** The Apex method 'FeeScheduleResolver.getApplicableRate()' in class FeeScheduleService.cls executes a SOQL query: 'SELECT Id, Allowed_Amount__c FROM Fee_Schedule__c WHERE CPT_Code__c = :cptCode AND Related_PNA__c = :pnaId ORDER BY CreatedDate ASC LIMIT 1' — missing Status__c and date range filters.

**Fix:**
1. Update SOQL in FeeScheduleService.cls to: SELECT Id, Allowed_Amount__c FROM Fee_Schedule__c WHERE CPT_Code__c = :cptCode AND Related_PNA__c = :pnaId AND Status__c = 'Active' AND Effective_Date__c <= :serviceDate AND (Termination_Date__c = null OR Termination_Date__c >= :serviceDate) ORDER BY Effective_Date__c DESC LIMIT 1
2. Add a unit test in FeeScheduleServiceTest.cls covering the multi-version scenario with 100% assertion coverage.
3. Execute anonymous Apex data remediation script in production during off-hours maintenance window to re-adjudicate impacted Claim_Line__c records.
4. Add a duplicate-prevention Validation Rule on Fee_Schedule__c to prevent overlapping active date ranges for the same CPT/PNA combination.
