# INS-0965: Implement Group Service Agreement (GSA) rate tier selection on Fee Schedule for employer group LOBs

## Description

As a Provider Network Specialist, I need the Fee_Schedule__c engine to support tiered rate selection driven by the Group_Service_Agreement__c (GSA) associated with a Policy__c, so that employer group members receive the negotiated rate tier (e.g., 'Tier 1 Preferred', 'Tier 2 Standard') rather than the default network rate.

Background: Acme Insurance recently onboarded three large employer groups (GSA records GSA-00041, GSA-00042, GSA-00043) each with bespoke rate tiers negotiated under their GSAs. Without this feature, all claims from these groups fall through to the default fee schedule, resulting in incorrect Claim_Line__c.Allowed_Amount__c values and downstream EOB discrepancies.

## Acceptance Criteria

Given a Claim__c where Policy__c.Group_Service_Agreement__c is populated and GSA.Rate_Tier__c = 'Tier 1 Preferred',
When adjudication resolves the applicable Fee_Schedule__c,
Then the system first queries Fee_Schedule__c WHERE GSA_Tier__c = 'Tier 1 Preferred' AND LOB__c = Policy__c.LOB__c AND Status__c = 'Active' before falling back to the default PNA-linked schedule.

Given the GSA-linked fee schedule lookup returns zero results,
When adjudication falls back to the default schedule,
Then Claim_Line__c.Rate_Resolution_Method__c = 'Default PNA Rate' and a warning flag Claim_Line__c.GSA_Tier_Fallback__c = true is set for operations review.

Given a Fee_Schedule__c record with GSA_Tier__c populated,
When a Provider Network Specialist views the record,
Then the record detail page displays a 'GSA Tier Assignments' related list showing all Group_Service_Agreement__c records referencing this schedule via GSA.Preferred_Fee_Schedule__c.

Given a new GSA is created with Rate_Tier__c populated,
When the GSA record is saved,
Then a Record-Triggered Flow validates that a Fee_Schedule__c record with matching GSA_Tier__c and LOB__c exists; if not, a validation warning (not a hard error) is surfaced to the user and a Task is assigned to the Provider Network Specialist team queue.

Given Claim_Line__c.GSA_Tier_Fallback__c = true,
When the Claims Operations Manager runs the 'GSA Tier Fallback Report' (custom Report Type: Claim Lines with GSA Fallback),
Then all affected records are visible with columns: Claim__c.Name, Policy__c.Group_Service_Agreement__c, Claim_Line__c.CPT_Code__c, Claim_Line__c.Allowed_Amount__c, Claim_Line__c.Rate_Resolution_Method__c.

## Solution

1. Add custom fields to Fee_Schedule__c: GSA_Tier__c (Picklist: Tier 1 Preferred, Tier 2 Standard, Tier 3 Basic, Default) and GSA_Linked__c (Checkbox).
2. Add custom fields to Claim_Line__c: Rate_Resolution_Method__c (Picklist), GSA_Tier_Fallback__c (Checkbox).
3. Update FeeScheduleLookupService.cls to implement a two-pass lookup strategy: Pass 1 — GSA tier match; Pass 2 — PNA default fallback. Encapsulate in a new inner class FeeScheduleResolutionResult for clean result passing.
4. Record-Triggered Flow on Group_Service_Agreement__c (after insert/update) triggers a Flow-called Apex action to validate tier-schedule alignment.
5. Custom Report Type 'Claim Lines with GSA Fallback' created linking Claim_Line__c → Claim__c → Policy__c → Group_Service_Agreement__c.
6. Update OmniStudio DataRaptor Extract for the adjudication summary FlexCard to include Rate_Resolution_Method__c in the returned payload.
