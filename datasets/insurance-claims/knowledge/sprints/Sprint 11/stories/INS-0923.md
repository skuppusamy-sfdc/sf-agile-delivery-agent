## Description

As a Claims Operations Manager, I need the system to automatically validate that the Fee_Schedule__c record applied to each Claim_Line__c falls within the Fee_Schedule__c.Effective_Start_Date__c and Fee_Schedule__c.Effective_End_Date__c window relative to the Claim__c.Date_of_Service__c, so that adjudicated payments always reflect the contractually correct allowed amount and audit findings related to incorrect fee application are eliminated.

## Acceptance Criteria

Given a Claim_Line__c is submitted for adjudication,
When the system looks up the applicable Fee_Schedule__c via the Provider_Network_Agreement__c,
Then the Fee_Schedule__c.Effective_Start_Date__c must be <= Claim__c.Date_of_Service__c AND Fee_Schedule__c.Effective_End_Date__c must be >= Claim__c.Date_of_Service__c.

Given no active Fee_Schedule__c exists for the Date_of_Service__c,
When adjudication attempts to calculate the allowed amount,
Then Claim_Line__c.Adjudication_Status__c must be set to 'Pended - Fee Schedule Gap' and a Task is auto-created and assigned to the Provider Network Specialist queue.

Given multiple Fee_Schedule__c records overlap for the same CPT code and PNA,
When the engine selects a rate,
Then the record with the most recent Fee_Schedule__c.Effective_Start_Date__c is applied and Fee_Schedule__c.Priority__c tiebreaker logic is honored.

Given a Fee_Schedule__c.Effective_End_Date__c is null,
When evaluating date range validity,
Then the schedule is treated as open-ended and valid indefinitely beyond the start date.

-Given a manual override field Fee_Schedule__c.Override_Approved__c = true bypasses all date validation- (deprecated per compliance review INS-0901)

Given adjudication completes successfully,
When Claim_Line__c.Allowed_Amount__c is populated,
Then Claim_Line__c.Fee_Schedule_Applied__c (lookup) must reference the exact Fee_Schedule__c record used and be stamped with the adjudication timestamp in Claim_Line__c.Fee_Schedule_Applied_Date__c.

## Solution

Implement an Apex invocable method 'FeeScheduleValidationService.validateAndApply()' called from a Record-Triggered Flow on Claim_Line__c (before save, on status transition to 'Pending Adjudication'). The service queries Fee_Schedule__c with a SOQL date range filter against Claim__c.Date_of_Service__c, applies CPT-code matching via Fee_Schedule_Line__c child records, and resolves conflicts using Priority__c ordering. On gap detection, a Platform Event 'FeeScheduleGap__e' is published to trigger async Task creation via a separate Apex handler. Fee_Schedule_Applied__c lookup field added to Claim_Line__c via custom field metadata. Unit tests cover >=90% coverage including null Effective_End_Date__c and multi-record overlap scenarios.
