## Description

**Bug Report**

**Summary:** When a Claim_Line__c is submitted with a Service_Date__c that falls in a prior fee schedule period, the FeeScheduleEngine incorrectly applies the currently active Fee_Schedule__c rate instead of the historically correct rate, resulting in overpayments or underpayments.

**Steps to Reproduce:**
1. Ensure two Fee_Schedule__c records exist for CPT code 99213: Record A (Effective 01/01/2024–06/30/2024, Rate = $85.00, Status = 'Expired') and Record B (Effective 07/01/2024–12/31/2024, Rate = $92.00, Status = 'Active').
2. Create a new Claim__c with Date_of_Service__c = 03/15/2024 (within Record A's range).
3. Add a Claim_Line__c with CPT_Code__c = '99213'.
4. Trigger adjudication by setting Claim__c.Status__c = 'Ready for Adjudication'.
5. Observe Claim_Line__c.Allowed_Amount__c after adjudication completes.

**Expected Result:** Claim_Line__c.Allowed_Amount__c = $85.00 and Fee_Schedule_Applied__c references Fee_Schedule__c Record A.

**Actual Result:** Claim_Line__c.Allowed_Amount__c = $92.00 and Fee_Schedule_Applied__c references Fee_Schedule__c Record B (the currently active record).

**Environment:** Acme Insurance Full Sandbox (ACME-FULL)
**Regression:** Introduced in deployment of INS-0853 on Sprint 10.

## Acceptance Criteria

As per Description.

Additional verification: After fix, run regression test suite 'FeeSchedule_Historical_Rate_Tests' (15 test cases) and confirm all pass with 0 failures.

## Solution

Root cause: The SOQL query in FeeScheduleResolutionService.cls uses a WHERE clause filtering only on Status__c = 'Active', ignoring date range overlap with Service_Date__c. Fix requires updating the query to filter on Effective_Start_Date__c <= :serviceDate AND Effective_End_Date__c >= :serviceDate regardless of Status__c, with a fallback to 'Active' only when no historical match is found. Update associated Apex test class FeeScheduleResolutionServiceTest to add backdated scenarios. Deploy via change set to ACME-FULL for QA validation.
