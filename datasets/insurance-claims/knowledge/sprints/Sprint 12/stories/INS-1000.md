## Description

As a Claims Operations Manager, I need the system to automatically reconcile inbound Electronic Remittance Advice (ERA) 835 transaction files against open Payment__c records in Salesforce so that payment posting is automated, manual matching errors are eliminated, and unmatched ERAs are flagged for exception handling within the same business day. The solution must handle both full-match and partial-match scenarios across Claim_Line__c adjudicated amounts.

## Acceptance Criteria

Given an ERA 835 file is received via the DataIntegration middleware and parsed into ERA_Transaction__c staging records,
When the auto-reconciliation batch job runs,
Then each ERA_Transaction__c is matched to a Payment__c record using the matching key: ERA_Transaction__c.Check_Number__c = Payment__c.Check_Number__c AND ERA_Transaction__c.TIN__c = Payment__c.Provider_TIN__c.

Given a full match is found (all Claim_Line__c amounts reconcile within $0.01 tolerance),
When the batch processes the record,
Then Payment__c.Status__c is updated to 'Posted', Payment__c.ERA_Reconciled_Date__c = TODAY(), and each linked Claim_Line__c.Payment_Status__c = 'Paid'.

Given a partial match is found (some Claim_Line__c amounts differ from ERA line-level adjustments),
When the batch processes the record,
Then Payment__c.Status__c = 'Partial Match', a Payment_Exception__c child record is created with Exception_Type__c = 'ERA Partial Match', and a Task is assigned to the Claims Operations Queue for manual review.

Given an ERA_Transaction__c has no matching Payment__c record after 3 matching attempts,
When the batch completes,
Then ERA_Transaction__c.Reconciliation_Status__c = 'Unmatched', a Payment_Exception__c is created with Exception_Type__c = 'ERA Unmatched', Priority__c = 'High', and a Platform Event (ERA_Exception__e) is published to notify the claims ops dashboard.

Given the batch job completes each run,
When all records are processed,
Then a Reconciliation_Summary__c record is created capturing: Total_ERAs_Processed__c, Matched_Count__c, Partial_Match_Count__c, Unmatched_Count__c, and Run_Timestamp__c.

Given Payment__c.Status__c is updated to 'Posted',
When the record is saved,
Then a Record-Triggered Flow generates an EOB (Explanation_of_Benefits__c) record linked to the parent Claim__c with EOB.Status__c = 'Issued' and EOB.Issue_Date__c = TODAY().

## Solution

Implement an Apex Batch class 'ERA_ReconciliationBatch' (Batchable, Schedulable) executing nightly and on-demand. The batch queries ERA_Transaction__c records where Reconciliation_Status__c = 'Pending', performs SOQL joins against Payment__c using Check_Number__c and Provider_TIN__c composite key. Full/partial match logic implemented in an Apex service class 'ERAMatchingService' with configurable tolerance via Custom Metadata (ERA_Config__mdt.Amount_Tolerance__c). Payment_Exception__c records are bulk-inserted. Platform Event ERA_Exception__e is published via EventBus.publish() for real-time claims ops dashboard updates (LWC subscribing via empApi). EOB generation handled by a separate Record-Triggered Flow on Payment__c to avoid governor limit conflicts with the batch. All DML wrapped in Database.SaveResult[] with error logging to Integration_Error_Log__c.
