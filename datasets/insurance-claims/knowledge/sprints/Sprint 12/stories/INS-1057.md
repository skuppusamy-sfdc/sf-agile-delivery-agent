## Description

**Bug Report – Commission Dashboard Unhandled Exception for High-Volume Agents**

**Environment:** Acme Insurance – Full Sandbox (Sprint 12 deploy, build 12.4)
**Related Story:** INS-1050 (Agent Portal Commission Dashboard)

**Steps to Reproduce:**
1. Log in as a broker user whose associated Contact has more than 500 linked Commission_Transaction__c records (test user: agent.highvolume@acmetest.com).
2. Navigate to the Commission Dashboard tab in the Agent Portal.
3. Observe the FlexCard loading behavior.

**Expected Result:** The Commission Dashboard FlexCard should load within 3 seconds and display aggregated KPI totals (Earned, Pending, YTD Paid) regardless of the total number of Commission_Transaction__c records for the agent.

**Actual Result:** The FlexCard throws an unhandled 'System.LimitException: Too many query rows: 50001' error in the DataRaptor Turbo Extract action ('DR_GetAgentCommissions') when the agent has more than 500 Commission_Transaction__c records. The dashboard renders blank with a generic 'An error occurred. Please try again.' toast message. Agents with fewer than 500 records are unaffected.

**Root Cause Hypothesis:** The DataRaptor Turbo Extract on Commission_Transaction__c is fetching all individual transaction rows and summing them in memory rather than using an aggregate SOQL query (SUM/GROUP BY). This causes the SOQL row governor limit to be breached for high-volume agents.

**Impact:** Affects approximately 8 senior brokers in the UAT environment who have been active for 3+ years. Will affect a larger population in production at go-live.

**Workaround:** None available. Affected agents cannot view their commission dashboard.

## Acceptance Criteria

As per Description. Post-fix validation: Commission Dashboard FlexCard must load successfully and display correct KPI totals for the test user agent.highvolume@acmetest.com (500+ records) within 3 seconds. No LimitException errors should appear in debug logs. KPI totals must match values produced by a verified manual SOQL aggregate query run against the same dataset.

## Solution

Replace the DataRaptor Turbo Extract ('DR_GetAgentCommissions') row-level fetch with an Integration Procedure that calls an Apex @InvocableMethod ('GetAgentCommissionAggregates') executing aggregate SOQL: SELECT Status__c, SUM(Amount__c) total FROM Commission_Transaction__c WHERE Agent__c = :agentId GROUP BY Status__c. This returns a maximum of one row per status value regardless of total record count, eliminating the governor limit risk. The Integration Procedure maps the aggregate results to the FlexCard KPI state properties (EarnedTotal, PendingTotal, YTDPaidTotal). Update the FlexCard ('AgentCommissionDashboard') to consume the Integration Procedure response instead of the DataRaptor. Add an Apex test class asserting correct aggregate behavior at 600+ mock records. Performance target: < 3s load time validated via Chrome DevTools in UAT.
