## Description

**Bug Report: EOB Generation Failure — Multi-Payer COB Scenarios**

**Environment:** Production Sandbox (Full Copy) — Sprint 12 regression

**Steps to Reproduce:**
1. Navigate to an existing Claim__c record where Claim__c.COB_Indicator__c = TRUE and two or more Payment__c records exist with Payment__c.Payer_Sequence__c values of 'Primary' and 'Secondary'.
2. Click the 'Generate EOB' button on the Claim__c record page.
3. Observe the Explanation_of_Benefits__c generation process.

**Expected Behavior:**
An Explanation_of_Benefits__c record is created with EOB.Primary_Payer_Amount__c, EOB.Secondary_Payer_Amount__c, EOB.Patient_Responsibility__c correctly calculated, and EOB.Status__c = 'Generated'.

**Actual Behavior:**
The OmniScript throws an unhandled null pointer error: 'List has no rows for assignment to SObject' at the DataRaptor Load step 'DR_EOB_PaymentAggregation'. EOB record is not created. Claim__c.EOB_Status__c remains 'Pending'. Error logged in Salesforce debug log: 'DRLoadException: Secondary payer Payment__c not found for COB aggregation'.

**Root Cause (Preliminary):** DataRaptor 'DR_EOB_PaymentAggregation' assumes exactly one Payment__c per Payer_Sequence__c value. When a secondary payer has made a partial payment split across two Payment__c records (e.g., two ERA batches for the same secondary payer), the SOQL in the DataRaptor returns multiple rows and the SObject assignment fails.

**Frequency:** Reproducible on 100% of COB claims with split secondary payments. Affects approximately 340 claims in current backlog.

**Attachments:** Debug log excerpt, sample Claim__c Id: a0Q3x000004TESTCOB01

## Acceptance Criteria

As per Description.

Additionally:
Given a Claim__c with COB_Indicator__c = TRUE and multiple Payment__c records for the same Payer_Sequence__c,
When 'Generate EOB' is triggered,
Then the DataRaptor aggregates all Payment__c.Amount__c values for each Payer_Sequence__c group (SUM) and populates EOB.Primary_Payer_Amount__c and EOB.Secondary_Payer_Amount__c correctly without error.

Given the EOB is generated successfully,
When the record is saved,
Then Explanation_of_Benefits__c.Status__c = 'Generated' and Claim__c.EOB_Status__c is updated to 'Sent' via the existing Record-Triggered Flow.

## Solution

Fix DataRaptor Extract 'DR_EOB_PaymentAggregation': replace the single-row SObject assignment with an aggregate SOQL using SUM(Amount__c) grouped by Payer_Sequence__c. Update the DataRaptor output JSON structure to use a list node for payer amounts and map to EOB fields using DataRaptor Transform with a FOR-EACH loop summing per sequence. Update the OmniScript step 'DR_EOB_PaymentAggregation' input/output mappings accordingly. Add a pre-step Decision element in the OmniScript to validate that at least one Payment__c with Status__c = 'Paid' or 'Partial' exists before invoking the DataRaptor, displaying a user-friendly error message if not. Regression test against 340 backlog claims in sandbox before promoting to production.
