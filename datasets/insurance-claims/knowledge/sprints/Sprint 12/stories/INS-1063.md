## Description

As a Claims Operations Manager, I need the system to automatically generate an EDI 835 Electronic Remittance Advice (ERA) file and transmit it to the designated TPA when a Claim__c batch is adjudicated so that downstream payment posting and EOB generation are triggered without manual file handling.

The ERA must include all adjudicated Claim_Line__c records from the batch, reference the associated Explanation_of_Benefits__c, and be transmitted via SFTP to the TPA's configured endpoint stored in the TPA's Account record.

## Acceptance Criteria

Given a Claim__c batch where all member Claim__c.Status__c = 'Adjudicated' and Claim__c.Payment_Batch__c is populated,
When the batch status transitions to 'Ready for ERA',
Then a Queueable Apex job (ERAGenerationJob.cls) is enqueued to compile all Claim_Line__c records where Claim_Line__c.Claim__c IN batch.

Given the Queueable Apex job runs,
When Claim_Line__c records are retrieved,
Then the job maps each line to EDI 835 segments: CLP (Claim Payment), SVC (Service Payment), PLB (Provider Level Adjustment) using Claim_Line__c.CPT_Code__c, Claim_Line__c.Billed_Amount__c, Claim_Line__c.Allowed_Amount__c, Claim_Line__c.Adjustment_Reason_Code__c, and Claim_Line__c.Paid_Amount__c.

Given the ERA file is constructed,
When the file is ready for transmission,
Then the job retrieves the TPA SFTP endpoint from Account.TPA_SFTP_Host__c, Account.TPA_SFTP_Port__c, and Account.TPA_SFTP_Path__c and transmits the file via an Apex HTTP callout to a middleware SFTP relay (named credential: TPA_SFTP_Relay).

Given the SFTP transmission succeeds (HTTP 200 response),
When the acknowledgment is received,
Then Payment_Batch__c.ERA_Status__c is updated to 'Transmitted', Payment_Batch__c.ERA_Transmitted_DateTime__c is stamped, and a related Explanation_of_Benefits__c record is created with EOB__c.Status__c = 'Issued'.

Given the SFTP transmission fails (non-200 response or timeout),
When the error is caught,
Then Payment_Batch__c.ERA_Status__c = 'Transmission Failed', an error log record (Integration_Error_Log__c) is inserted with the HTTP response code and body, and a Case is auto-created assigned to the 'Claims_Payment_Queue' for manual retry.

Given the ERA file is transmitted,
When the Compliance Officer reviews the audit trail,
Then a ContentVersion (file) record is linked to the Payment_Batch__c record storing the raw EDI 835 content for regulatory retention (minimum 7 years per CMS guidelines).

## Solution

Implement a Record-Triggered Flow on Payment_Batch__c (After Update, ERA_Status__c Changed to 'Ready for ERA') to enqueue ERAGenerationJob.cls (Queueable Apex). The Apex class queries Claim_Line__c records, constructs the EDI 835 string via a dedicated EDI835Builder utility class, and calls an Apex HTTP callout using the TPA_SFTP_Relay named credential pointing to the internal SFTP relay microservice. Use ContentVersion insert to archive the file. Integration_Error_Log__c custom object captures failures. Schedule a retry Scheduled Apex (ERARetryScheduler.cls) to re-attempt failed transmissions every 30 minutes up to 3 retries.
