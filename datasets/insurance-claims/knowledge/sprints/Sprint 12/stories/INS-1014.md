## Description

As a Provider Relations Specialist, I need a unified Provider 360 FlexCard on the Provider__c (Person Account) record page that consolidates NPI data from NPPES, TIN verification status, and CAQH attestation details so that I can assess provider data completeness without switching between external portals or spreadsheets.

This story covers read-only display of integrated data. Write-back and update workflows are covered in INS-1015.

## Acceptance Criteria

Given a Provider__c record is open in Salesforce,
When the Provider 360 FlexCard loads,
Then the following fields are displayed: Provider__c.NPI__c, Provider__c.TIN__c, Provider__c.CAQH_Provider_ID__c, NPPES_Last_Verified__c, CAQH_Attestation_Date__c, and CAQH_Status__c.

Given Provider__c.CAQH_Status__c = 'Expired',
When the FlexCard renders,
Then the CAQH status badge displays in red with label 'Attestation Expired' and a 'Re-Attest' action button is visible.

Given a NPPES API callout is made on FlexCard load,
When the response returns HTTP 200,
Then Provider__c.NPPES_Last_Verified__c is updated to today's date and the returned enumeration type (MD, DO, NP, etc.) populates Provider__c.Provider_Type__c if currently blank.

Given the NPPES API returns HTTP 404 or times out,
When the FlexCard handles the error,
Then a warning banner 'NPPES verification unavailable — displaying cached data' is shown and no field values are overwritten.

## Solution

OmniStudio FlexCard 'Provider_360_Card' with two DataRaptor Extract actions: one for Salesforce-stored fields and one invoking Apex REST action 'NPPESVerificationAction' via Integration Procedure. NPPES callout uses Named Credential 'NPPES_API'. CAQH status color logic handled by FlexCard conditional styling states. Integration Procedure caches NPPES response in a custom object 'NPPES_Cache__c' with a 24-hour TTL checked before making live callout. Deployed to Provider__c Lightning Record Page via App Builder.
