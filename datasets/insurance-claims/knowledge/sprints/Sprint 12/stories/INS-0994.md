## Description

As a Credentialing Specialist, I need the system to automatically pull updated provider data from CAQH ProView into the corresponding Credentialing_Case__c record when a provider completes their annual re-attestation so that our credentialing team does not have to manually re-enter licensure, board certification, and malpractice insurance data — reducing re-credentialing cycle time from 15 days to under 5 days. The sync should update the associated Provider (Person Account) record and flag discrepancies for human review.

## Acceptance Criteria

Given a provider completes re-attestation in CAQH ProView,
When the CAQH webhook payload is received by the Salesforce REST endpoint (/services/apexrest/caqh/attestation),
Then a Credentialing_Case__c record is identified by matching Provider__c.CAQH_Provider_ID__c and Credentialing_Case__c.Status__c is set to 'CAQH Data Received'.

Given the CAQH payload is parsed,
When the following fields are present in the response: license_number, license_state, license_expiration, board_certification, malpractice_carrier, malpractice_expiration,
Then the corresponding Credentialing_Case__c fields are updated: License_Number__c, License_State__c, License_Expiration_Date__c, Board_Certification__c, Malpractice_Carrier__c, Malpractice_Expiration_Date__c.

Given the CAQH data is synced,
When any synced field value differs from the existing value on the Provider (Person Account) record by more than an allowable threshold (e.g., name differs, license state changes),
Then a Credentialing_Discrepancy__c child record is created with fields: Field_Name__c, Previous_Value__c, New_Value__c, Reviewed__c = false, and the Credentialing_Case__c.Requires_Manual_Review__c = true.

Given Credentialing_Case__c.Requires_Manual_Review__c = false after sync,
When all data passes automated validation rules (license not expired, malpractice not lapsed),
Then Credentialing_Case__c.Status__c auto-advances to 'Pending Committee Review' and is assigned to the Credentialing Committee queue.

Given the CAQH API returns a 401 or 503 error,
When the callout fails,
Then the failure is logged in Integration_Error_Log__c with fields: Source__c = 'CAQH', Error_Code__c, Timestamp__c, and a retry is scheduled via Schedulable Apex within 30 minutes (max 3 retries).

Given a Credentialing_Case__c has no matching CAQH_Provider_ID__c,
When the webhook payload arrives,
Then the payload is quarantined in a CAQH_Unmatched_Payload__c staging object and an alert email is sent to the Credentialing Specialist queue.

Given the integration is active in production,
When volume exceeds 200 re-attestations per day,
Then processing uses a Queueable Apex chain to stay within governor limits with no data loss.

## Solution

Expose a global Apex REST resource class (CAQHAttestationWebhook) at /services/apexrest/caqh/attestation accepting POST. Payload deserialized via a typed Apex wrapper class (CAQHPayload). Matching logic queries Provider__c by CAQH_Provider_ID__c. Field mapping handled in a custom metadata type (CAQH_Field_Mapping__mdt) for maintainability. Discrepancy detection compares incoming vs existing values in a dedicated Apex service class (CAQHDiscrepancyDetector). Queueable Apex chain (CAQHSyncQueueable) processes batches of 50 records. Retry scheduler (CAQHRetryScheduler) uses a Schedulable Apex class with exponential backoff. Integration errors logged via a centralized IntegrationErrorLogger utility class. All logic covered by mock HttpCalloutMock test classes with minimum 85% coverage.
