## Description

**Bug Report — UAT Cycle 3, Sprint 15**

**Summary:** When CAQH returns a verification webhook response for a provider, the Credentialing_Case__c.Status__c field is not updating from 'Pending CAQH' to 'CAQH Verified'. The webhook is confirmed to be received by the Salesforce Connected App endpoint (visible in API logs), but no field update occurs on the Credentialing_Case__c record.

**Steps to Reproduce:**
1. Navigate to an existing Credentialing_Case__c record with Status__c = 'Pending CAQH' and a valid CAQH_Provider_ID__c populated.
2. Trigger a CAQH verification request via the 'Send to CAQH' quick action on the record.
3. In the CAQH sandbox environment, approve the provider profile to simulate a positive verification response.
4. Wait for the webhook callback (observed within 2–5 minutes in UAT environment).
5. Refresh the Credentialing_Case__c record in Salesforce.

**Expected Result:** Credentialing_Case__c.Status__c = 'CAQH Verified', Credentialing_Case__c.CAQH_Verification_Date__c = today's date, and Credentialing_Case__c.CAQH_Response_Payload__c populated with the JSON response.

**Actual Result:** Credentialing_Case__c.Status__c remains 'Pending CAQH'. No field updates observed. No error in debug logs at the time of webhook receipt.

**Additional Context:** Issue appears to have been introduced in the Sprint 14 deployment (INS-1318 — CAQH integration refactor). The Named Credential endpoint was updated but the Apex REST resource class 'CAQHWebhookHandler' may not have been redeployed. Confirmed reproducible in UAT org; not yet tested in production. Affects 14 open Credentialing_Case__c records currently stuck in 'Pending CAQH' status.

## Acceptance Criteria

As per Description. Verify Credentialing_Case__c.Status__c, CAQH_Verification_Date__c, and CAQH_Response_Payload__c update correctly upon positive webhook receipt. Verify negative/rejection webhook sets Status__c = 'CAQH Rejected' and populates CAQH_Rejection_Reason__c. Confirm no data loss for the 14 existing stuck records after fix deployment.

## Solution

Root cause investigation: Confirm whether CAQHWebhookHandler Apex REST class is active and correctly mapped to the updated Named Credential endpoint post-Sprint 14 refactor. Check if the @RestResource URL mapping was inadvertently changed. Fix: Redeploy CAQHWebhookHandler with corrected endpoint mapping; add explicit try/catch with Platform Event-based error alerting. Backfill script (Anonymous Apex or Data Loader) to re-trigger status evaluation for the 14 stuck Credentialing_Case__c records by re-querying CAQH via batch job. Add integration test in CI pipeline to validate webhook round-trip in future deployments.
