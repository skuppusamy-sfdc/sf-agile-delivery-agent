## Description

As a Broker Relations Coordinator, I need licensed brokers to be able to submit a First Notice of Loss (FNOL) on behalf of a policyholder directly through the Agent/Broker Portal using a guided OmniScript, so that claims intake is accelerated and brokers can serve their clients without requiring the policyholder to call into the claims center directly.

The OmniScript must validate broker licensure, confirm the broker's active association to the policyholder's Policy__c via the Group_Service_Agreement__c, and create a Claim__c record in 'FNOL Received' status upon submission.

## Acceptance Criteria

Given a broker authenticated in the Agent Portal with an active Broker_License__c record where License_Status__c = 'Active' and License_Expiry__c > today,
When the broker initiates the FNOL OmniScript,
Then the script must pre-populate the policyholder lookup from the broker's associated Account relationships and display only Policy__c records where Policy__c.Status__c = 'Active'.

Given the broker selects a Policy__c and enters loss details,
When the broker completes the OmniScript and clicks 'Submit FNOL',
Then a Claim__c record must be created with Claim__c.Status__c = 'FNOL Received', Claim__c.LOB__c populated from Policy__c.Line_of_Business__c, Claim__c.FNOL_Submitted_By__c = the broker's Contact ID, and Claim__c.Submission_Channel__c = 'Broker Portal'.

Given the FNOL is submitted,
When Claim__c is created,
Then an assignment rule must route the new claim to the appropriate Claims Operations queue based on Claim__c.LOB__c and Claim__c.Loss_State__c using Assignment Coordinator routing logic.

Given a broker whose Broker_License__c is expired or inactive,
When they attempt to launch the FNOL OmniScript,
Then the script must block entry at step 1 with message: 'Your broker license is not currently active. Please contact your Broker Relations Coordinator.' and log a Broker_Access_Attempt__c record.

Given the FNOL submission is successful,
When the OmniScript completes,
Then the broker must receive an on-screen confirmation displaying the new Claim__c.Claim_Number__c and an automated email confirmation sent to both the broker and the policyholder's email on file (Account.PersonEmail).

-Step 3 of OmniScript must integrate with DocuSign to capture broker e-signature on FNOL form — deferred to Phase 3 pending legal sign-off-

## Solution

Build a multi-step OmniScript (BrokerFNOLSubmission) in OmniStudio with the following steps: (1) Broker Validation — DataRaptor Lookup on Broker_License__c; (2) Policyholder & Policy Selection — DataRaptor Extract filtered by broker-account relationships; (3) Loss Details Entry — custom LWC step for date, loss type, description; (4) Review & Submit — DataRaptor Transform + Integration Procedure to create Claim__c via Apex REST. An Integration Procedure (CreateFNOLClaim_IP) handles Claim__c creation, assignment rule trigger, and email alert via Salesforce Email Alert action. The OmniScript is embedded in the Agent Portal Experience Cloud site and secured with a Permission Set Group (Broker_Portal_User) that restricts object access to relevant records only. Broker_Access_Attempt__c logging handled by a Record-Triggered Flow on failed validation path.
