## Description

As a Claims Operations Manager, I need the system to automatically generate and stage an Electronic Remittance Advice (ERA) file in the 835 transaction format whenever a Payment__c record is finalized for a TPA-managed claim, so that downstream TPA partners receive timely, accurate remittance data without manual intervention and we reduce payment reconciliation errors.

Background: During Phase 2 UAT, the claims operations team identified that ERA files were only being generated via a nightly batch job, causing a 24-hour lag for TPA partners who require near-real-time remittance data for auto-posting. This story extends INS-1344 (Payment__c finalization workflow) to add ERA generation as a downstream action.

TPA partners consume the 835 ERA file via the existing SFTP integration endpoint. The file must include loop 2100/2110 detail segments populated from Claim_Line__c and Explanation_of_Benefits__c records associated with the finalized Payment__c.

## Acceptance Criteria

Given a Payment__c record where Payment__c.Status__c is updated to 'Finalized' and Payment__c.TPA_Managed__c = TRUE,
When the Record-Triggered Flow fires on the Payment__c after-save event,
Then a Platform Event (ERA_Generation_Event__e) is published containing Payment__c.Id, Payment__c.Claim__c, and Payment__c.TPA_Partner_Code__c within 5 seconds of the status change.

Given the ERA_Generation_Event__e is received by the async Apex subscriber (ERAGenerationEventHandler),
When the handler queries all active Claim_Line__c records where Claim_Line__c.Claim__c = Payment__c.Claim__c and Claim_Line__c.Status__c = 'Adjudicated',
Then an ERA_File__c staging record is created with ERA_File__c.Transaction_Set__c = '835', ERA_File__c.Format_Version__c = '005010X221A1', ERA_File__c.Payment__c populated, and ERA_File__c.Generation_Status__c = 'Pending'.

Given the ERA_File__c staging record is created with Generation_Status__c = 'Pending',
When the ERAFileBuilderService Apex class constructs the 835 payload,
Then loop 2100 (Claim Payment Information) segments must be populated from Claim__c.Claim_Number__c, Claim__c.Total_Billed_Amount__c, Payment__c.Payment_Amount__c, and the associated Explanation_of_Benefits__c.Adjustment_Reason_Code__c values.

Given loop 2100 is populated,
When loop 2110 (Service Payment Information) segments are built,
Then each Claim_Line__c contributes one SVC segment using Claim_Line__c.CPT_Code__c, Claim_Line__c.Billed_Amount__c, Claim_Line__c.Allowed_Amount__c, and Claim_Line__c.Denial_Reason_Code__c (if populated); ICD-10 diagnosis pointers must be sourced from Claim_Line__c.Primary_Diagnosis_Code__c.

Given the 835 payload is fully constructed,
When the DataIntegration outbound callout to the SFTP staging service is invoked,
Then ERA_File__c.Generation_Status__c transitions to 'Transmitted', ERA_File__c.Transmitted_Datetime__c is stamped, and a Chatter post is added to the parent Payment__c record confirming transmission with the ERA file reference number.

Given the SFTP callout returns a non-200 HTTP response or throws a callout exception,
When the retry logic in ERAGenerationEventHandler evaluates the failure,
Then ERA_File__c.Generation_Status__c is set to 'Failed', ERA_File__c.Failure_Reason__c is populated with the error message, and a High-priority Case is auto-created and assigned to the 'Claims-ERA Errors' queue with the Payment__c and ERA_File__c record links in the case description.

-Given Payment__c.TPA_Managed__c = FALSE, the ERA generation flow should still trigger but write to an internal archive only- (deprecated per INS-1344 revised scope; non-TPA payments are excluded entirely from ERA generation).

Given a TPA partner has a custom ERA_Delimiter__c value configured on the TPA_Partner__c record,
When the 835 file is serialized,
Then the file uses the delimiter specified in TPA_Partner__c.ERA_Delimiter__c rather than the default pipe character, and the ISA segment reflects the correct TPA_Partner__c.Interchange_Sender_ID__c value.

## Solution

1. Record-Triggered Flow on Payment__c (after-save, optimized for 'Finalized' status entry criteria with TPA_Managed__c = TRUE filter) publishes ERA_Generation_Event__e Platform Event to decouple ERA generation from the synchronous transaction.
2. Apex Platform Event subscriber class ERAGenerationEventHandler (with chunked governor-limit-safe SOQL) queries Claim_Line__c and EOB records, then delegates to ERAFileBuilderService for 835 serialization.
3. ERAFileBuilderService is a standalone Apex service class (no UI dependency) implementing the X12 835 005010X221A1 spec; uses Custom Metadata Type (ERA_Format_Config__mdt) for segment delimiters and version strings to avoid hardcoding.
4. TPA_Partner__c lookup on Payment__c drives partner-specific delimiter and sender ID configuration sourced from Custom Metadata.
5. ERA_File__c is a new custom object with a Master-Detail to Payment__c; Sharing Rules ensure 'Claims-ERA Errors' queue members have read access.
6. SFTP outbound callout uses Named Credential (NC_TPA_SFTP_Staging) with per-org certificate-based auth; retry up to 3 times with exponential backoff via Queueable chaining.
7. FlexCard on Payment__c record page surfaces ERA_File__c child records (status, transmitted datetime, file reference) for Claims Operations visibility without navigating away.
8. Unit tests target ≥ 85% coverage on ERAGenerationEventHandler and ERAFileBuilderService; test factory generates Claim__c → Claim_Line__c → Payment__c hierarchy with TPA and non-TPA variants.
