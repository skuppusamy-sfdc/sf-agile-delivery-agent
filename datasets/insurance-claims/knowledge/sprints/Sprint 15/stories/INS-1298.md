## Description

As a Compliance Officer, I need an automated workflow triggered when a Network_Termination_Case__c is approved so that CMS notification letters are generated via Conga CLM within the regulatory timeframe, the provider's portal access is revoked, all active Provider_Network_Agreement__c records are end-dated, and a Provider_Change_Notification__c (PCN) is broadcast to downstream systems — ensuring Acme Insurance meets CMS termination reporting obligations and reduces manual compliance risk.

This story covers involuntary terminations (cause-based). Voluntary terminations are handled in INS-1287.

## Acceptance Criteria

Given a Network_Termination_Case__c where NTC.Termination_Type__c = 'Involuntary' and NTC.Status__c transitions to 'Approved',
When the approval is recorded,
Then the system timestamps NTC.Approval_DateTime__c and calculates NTC.CMS_Notification_Due_Date__c = NTC.Approval_DateTime__c + 15 calendar days.

Given NTC.Status__c = 'Approved' and NTC.CMS_Notification_Due_Date__c is set,
When the Conga CLM document generation is triggered,
Then a CMS termination notification letter is generated using template 'CMS_Involuntary_Termination_v3', merged with Provider__c.NPI__c, Provider__c.TIN__c, NTC.Termination_Reason__c, and NTC.Effective_Termination_Date__c, and attached to the NTC record as a Salesforce File (ContentDocument).

Given the CMS letter is successfully generated,
When the document is attached,
Then NTC.CMS_Letter_Status__c = 'Generated — Pending Send' and a Task is assigned to the owning Compliance Officer with Subject = 'Review and Send CMS Termination Letter' and Due Date = NTC.CMS_Notification_Due_Date__c - 3 days.

Given NTC.Status__c = 'Approved',
When the workflow processes provider access revocation,
Then all Community User records linked to the provider (User.ContactId = Provider__c.PersonContactId) have User.IsActive set to false and a Chatter post is added to the Provider__c record: 'Provider portal access revoked per Network Termination Case [NTC.Name] effective [NTC.Effective_Termination_Date__c].'

Given NTC.Status__c = 'Approved',
When PNA end-dating runs,
Then all Provider_Network_Agreement__c records where PNA.Provider__c = NTC.Provider__c AND PNA.Status__c = 'Active' are updated: PNA.Contract_End_Date__c = NTC.Effective_Termination_Date__c and PNA.Status__c = 'Terminated — Involuntary'.

Given all PNA records are end-dated,
When PCN broadcast is triggered,
Then a Provider_Change_Notification__c record is created with PCN.Change_Type__c = 'Network Termination', PCN.Provider__c = NTC.Provider__c, PCN.Effective_Date__c = NTC.Effective_Termination_Date__c, and a Platform Event (Provider_Network_Change__e) is published for downstream MCO and TPA system subscribers.

-Given the provider has open Claim__c records in 'FNOL' or 'Pending Review' status, auto-deny all open claims upon termination approval.- (Removed — legal review determined auto-denial violates state prompt payment regulations; claims must be individually reviewed)

## Solution

Implement a Record-Triggered Flow (NTC_Approval_Termination_Orchestrator) on Network_Termination_Case__c, entry condition: Status__c changes to 'Approved' AND Termination_Type__c = 'Involuntary'. Flow uses scheduled paths for time-based actions. Conga CLM integration via REST API callout in an Apex action (CongaTerminationLetterService.generateLetter) called from the flow. User deactivation handled via Apex (ProviderPortalAccessService.revokeAccess) due to mixed DML restrictions — invoked as a queueable to avoid governor limits. PNA bulk end-dating via Apex invocable (PNATerminationService.endDateActivePNAs) with bulkification for providers with large PNA portfolios. Platform Event publish (Provider_Network_Change__e) via flow Publish Message element. All steps include fault paths writing to a custom exception log (Termination_Workflow_Exception__c) for ops visibility.
