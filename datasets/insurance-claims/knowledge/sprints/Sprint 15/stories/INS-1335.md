## Description

As a Provider Network Specialist, I need a bulk CPT code rate update tool within the Fee Schedule Engine so that I can upload the annual CMS rate table refresh (typically 8,000–12,000 CPT codes) via CSV, preview changes before committing, and maintain a full audit trail of rate changes on Fee_Schedule__c records — ensuring compliance with CMS reimbursement mandates and eliminating the current manual one-by-one update process that takes 3+ weeks per refresh cycle.

## Acceptance Criteria

Given a Provider Network Specialist navigates to the 'Fee Schedule Management' tab,
When they select 'Bulk CPT Rate Update' and upload a CSV file conforming to the CMS rate table format (columns: CPT_Code, Modifier, Geographic_Area, New_Rate, Effective_Date),
Then the system validates the CSV for required columns, data types, and that all CPT codes exist in the CPT_Code__c master object before proceeding to preview.

Given a valid CSV is uploaded,
When the preview step renders,
Then a data table displays each impacted Fee_Schedule__c record with current Fee_Schedule__c.Rate__c, proposed new rate, CPT code, modifier, and a row-level change indicator (Increase / Decrease / No Change) — with a summary count of total rows, rows with rate increases, rows with rate decreases, and rows unchanged.

Given the specialist reviews and approves the preview,
When they click 'Commit Update',
Then all Fee_Schedule__c records are updated in bulk (Apex batch, scope 200) and for each updated record a Fee_Schedule_Rate_History__c child record is created capturing: FSH.Previous_Rate__c, FSH.New_Rate__c, FSH.Change_Date__c = TODAY(), FSH.Changed_By__c = running user, FSH.Source__c = 'CMS Annual Refresh'.

Given a CPT code in the upload file does not match any existing CPT_Code__c master record,
When validation runs,
Then those rows are flagged in the preview as 'Unmatched CPT — Will Be Skipped' and are excluded from the commit batch; the specialist can download an error report CSV of skipped rows.

Given the bulk commit batch completes,
When all records are processed,
Then the specialist receives an in-app notification and email summarizing: total records updated, total skipped, batch job Id, and a link to the Fee Schedule Audit Report in the ComplianceReporting component.

-Given the update affects more than 5,000 records, a second approval from the Claims Operations Manager is required before commit.- (Deprecated — approval gate removed per compliance team sign-off on 11/14; audit trail alone is sufficient)

Given a Compliance Officer audits the rate change,
When they query Fee_Schedule_Rate_History__c filtered by FSH.Source__c = 'CMS Annual Refresh' and FSH.Change_Date__c = current year,
Then all changed records are retrievable with full before/after rate values and the running user's name.

## Solution

Build an LWC 'bulkCptRateUpload' with a multi-step wizard: Step 1 — file upload using lightning-file-upload component, CSV parsed client-side via PapaParse JS library; Step 2 — preview table calling Apex @AuraEnabled method 'FeeScheduleBulkUpdateController.previewRateChanges()' which performs SOQL validation against CPT_Code__c and Fee_Schedule__c objects and returns a wrapper list; Step 3 — commit triggers Apex @AuraEnabled(cacheable=false) 'commitRateChanges()' which enqueues 'FeeScheduleRateBatch' (implements Database.Batchable, Database.Stateful) for bulk DML. Fee_Schedule_Rate_History__c records created within the batch execute method. Error report CSV generated via a separate Apex method using Blob/ContentVersion. In-app notification via Custom Notification API. LWC deployed as a standalone tab in the Provider Network app. Sharing Rules ensure only users with 'Fee_Schedule_Manager' Permission Set can access the commit action.
