## Description

**Bug Report — UAT Cycle 3 — COB Module**

On the Policyholder 360 page, the Coordination of Benefits (COB) sequence indicator is displaying the wrong primary/secondary policy order when a member holds more than one active Policy__c record across different LOBs. This causes claims adjudicators to incorrectly identify the primary payer, leading to erroneous payment calculations during UAT test scenarios.

**Steps to Reproduce:**
1. Identify or create a Contact (member) with exactly two active Policy__c records: one with Policy__c.LOB__c = 'Medical' (Policy__c.COB_Order__c = 1) and one with Policy__c.LOB__c = 'Dental' (Policy__c.COB_Order__c = 2).
2. Navigate to the Policyholder 360 LWC page for that member.
3. Observe the 'Coverage Summary' FlexCard — the COB sequence panel shows the Dental policy as Primary (COB Order 1) and the Medical policy as Secondary.
4. Verify Policy__c.COB_Order__c field values directly on each record — they are correct (Medical = 1, Dental = 2).

**Expected Behavior:** The COB sequence panel should display policies in ascending COB_Order__c order, with the lowest value labeled 'Primary'.

**Actual Behavior:** The DataRaptor Extract driving the COB panel appears to be sorting by Policy__c.CreatedDate DESC instead of Policy__c.COB_Order__c ASC, reversing the display order.

**Severity:** High — directly impacts claims payment accuracy in UAT scenarios.
**Reproducible:** 100% when member has 2+ active policies with different LOBs.

## Solution

Root cause confirmed: DataRaptor Extract 'DR_Extract_MemberCOBPolicies' has an incorrect Order By configuration — currently sorting on 'CreatedDate' descending. Fix: update the DR Extract's Filter/Sort tab to sort by 'COB_Order__c' ascending. Additionally, add a secondary sort on 'Policy__c.Effective_Date__c' descending as a tiebreaker for edge cases where COB_Order__c values are equal. Regression test in UAT with members having 1, 2, and 3 active policies. Update the FlexCard 'PolicyholderCoverageSummary' state condition labels to dynamically render 'Primary', 'Secondary', 'Tertiary' based on array index position after the sort fix is applied.
