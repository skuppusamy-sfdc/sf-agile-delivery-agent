## Description

**Bug Report — UAT Sprint 15**

During UAT regression testing of COB workflows, the Explanation_of_Benefits__c record is not reflecting the correct patient responsibility amount when a secondary payer adjudication is applied to a Claim_Line__c.

**Steps to Reproduce:**
1. Create a Claim__c record with LOB = 'Medical' and COB_Indicator__c = true.
2. Add two Claim_Line__c records with CPT codes 99213 and 71046.
3. Submit primary payer adjudication — set Claim_Line__c.Adjudication_Status__c = 'Primary Paid'.
4. Trigger secondary payer COB calculation via the 'Apply COB' button on the Claim__c record.
5. Navigate to the related Explanation_of_Benefits__c record.

**Expected Result:** EOB.Patient_Responsibility_Amount__c should equal the sum of all Claim_Line__c.Patient_Responsibility__c values after COB adjustment. EOB.COB_Adjustment_Amount__c should be populated with the secondary payer's contribution.

**Actual Result:** EOB.Patient_Responsibility_Amount__c retains the pre-COB value. EOB.COB_Adjustment_Amount__c remains NULL. The Claim_Line__c records show correct adjusted values but the rollup to EOB is not firing.

**Environment:** Full Sandbox (ACME-FS2)
**Frequency:** 100% reproducible
**Severity:** High — impacts all COB claims in payment run

## Solution

Root cause identified: the Record-Triggered Flow 'EOB_Rollup_After_COB_Adjustment' has an entry condition filtering on Claim_Line__c.Adjudication_Status__c = 'Adjudicated' but COB secondary path sets status to 'COB Adjusted' — a new enum value added in Sprint 13 (INS-1241) that was not accounted for in the flow entry criteria. Fix: update the flow entry condition to include OR Claim_Line__c.Adjudication_Status__c = 'COB Adjusted'. Additionally, add a scheduled Apex batch job (EOBRollupRecalcBatch) to retroactively correct any EOB records created since Sprint 13 deployment. Deploy fix via Change Set to ACME-FS2 for UAT re-test before production push.
