## Description

**Bug Report — UAT Cycle 3 Finding**

**Steps to Reproduce:**
1. Create a Claim__c with two Claim_Line__c records; set Claim_Line__c.Billed_Amount__c = 500.00 on each.
2. Attach a Fee_Schedule__c with a contractual adjustment code CO-45 reducing each line to 350.00.
3. Trigger ERA auto-posting via the 'Post ERA' button on the Claim__c record.
4. Observe Payment__c record created by the posting process.

**Expected Result:**
Payment__c.Net_Paid_Amount__c = 700.00 (sum of adjusted allowed amounts across both Claim_Line__c records).

**Actual Result:**
Payment__c.Net_Paid_Amount__c = 0.00. The contractual adjustment is being applied but the net calculation is not accumulating the adjusted amounts before writing to Payment__c.

**Environment:** Full Sandbox (ACME-FS2)
**Frequency:** Reproducible 100% when CO-45 adjustment reason code is present.
**Regression:** Introduced in Sprint 14 deployment (INS-1318 ERA parsing refactor).

## Acceptance Criteria

As per Description.

Additionally: After fix, Payment__c.Net_Paid_Amount__c must equal the sum of all Claim_Line__c.Allowed_Amount__c values on the parent Claim__c, and Payment__c.Adjustment_Total__c must equal the sum of all Claim_Line__c.Contractual_Adjustment__c values, such that Billed_Total - Adjustment_Total = Net_Paid_Amount.

## Solution

Root cause identified in ERAAdjudicationService.cls: the contractual adjustment loop was calling lineItem.Allowed_Amount__c = 0 as a reset step before applying the CO-45 factor, but the accumulator variable netPaidAccumulator was being read prior to the assignment completing due to a missing DML flush. Fix: restructure the loop to compute Allowed_Amount__c per line first, accumulate into a local Decimal variable, then perform a single bulk update on Claim_Line__c records before writing the total to Payment__c.Net_Paid_Amount__c. Add regression test ERAAdjudicationServiceTest.testCO45ContractualAdjustment() covering multi-line scenario.
