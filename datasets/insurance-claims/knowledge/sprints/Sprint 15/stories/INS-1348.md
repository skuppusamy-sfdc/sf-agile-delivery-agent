## Description

**Bug Report — UAT Sprint 15**

**Summary:** Duplicate Provider_Change_Notification__c (PCN) records are being created when the QGenda integration sync fires at the same time a Provider Relations Specialist manually saves a change to the Provider (Person Account) record. This results in two PCN records with identical External_Reference_Id__c values and causes downstream duplicate processing alerts.

**Environment:** Acme Insurance Full Sandbox (FS1)

**Steps to Reproduce:**
1. Open any active Provider (Person Account) record.
2. Begin editing Provider__c.Primary_Specialty__c.
3. Simultaneously (within the same ~5-second window), trigger the QGenda scheduled integration job (can be manually triggered via the 'QGenda_Sync' Named Credential test endpoint in FS1).
4. Save the Provider record.
5. Navigate to the Related List for Provider_Change_Notification__c on the Provider record.

**Expected Result:** Only one PCN record is created reflecting the specialty change, with PCN__c.Source__c indicating the originating system (either 'QGenda' or 'Manual').

**Actual Result:** Two PCN records are created with identical PCN__c.External_Reference_Id__c and PCN__c.Change_Type__c = 'Specialty Update' values. The duplicate triggers a validation rule error on downstream Network_Termination_Case__c creation.

**Frequency:** Intermittent — observed in 4 of 10 UAT test cycles. Higher frequency during scheduled QGenda sync windows (top of each hour).

**Root Cause Hypothesis:** The Apex Trigger on Provider (Account) and the Platform Event subscriber for QGenda sync both independently create PCN records without checking for an existing in-flight record with the same External_Reference_Id__c.

## Acceptance Criteria

As per Description. Fix must implement an idempotency check using PCN__c.External_Reference_Id__c as a unique key. Verified in FS1 with concurrent manual save and QGenda sync simulation. Zero duplicate PCN records after 20 repeated test cycles.

## Solution

Add a UNIQUE constraint (duplicate rule + matching rule) on PCN__c.External_Reference_Id__c at the object level. Refactor the Apex Trigger helper class 'PCNTriggerHandler' to perform an upsert using External_Reference_Id__c as the external ID field instead of insert, preventing duplicate creation. Update the Platform Event subscriber Flow for QGenda sync to similarly use an upsert DataRaptor or Apex upsert on PCN__c. Add a try-catch in the trigger handler to log DuplicateException errors to Error_Log__c without re-throwing, ensuring the parent Provider save is not blocked.
