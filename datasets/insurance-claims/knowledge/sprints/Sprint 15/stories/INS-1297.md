## Description

**Bug Report — UAT Sprint 15**

During UAT testing of the Group/Employer onboarding OmniScript ('Group_Onboarding_OmniScript_v2'), the final DataRaptor Transform step fails to create the Group_Service_Agreement__c record when the user selects 'Yes' for the 'Administered by TPA' toggle on Step 3 of the script.

**Steps to Reproduce:**
1. Navigate to Accounts > New Group Employer and launch 'Group_Onboarding_OmniScript_v2'.
2. Complete Step 1 (Employer Demographics) and Step 2 (Plan Selection — select any LOB).
3. On Step 3 (Administration), toggle 'Administered by TPA?' to 'Yes'.
4. Populate the TPA_Name__c lookup field with any existing TPA account.
5. Complete Step 4 (Review & Submit) and click 'Submit'.
6. Observe the confirmation screen and navigate to the employer Account record.

**Expected Result:** A Group_Service_Agreement__c record is created with GSA.TPA_Administered__c = true, GSA.TPA_Account__c populated with the selected TPA account ID, and GSA.Status__c = 'Pending Countersignature'.

**Actual Result:** The OmniScript shows a generic 'Submission Error' toast on the confirmation screen. No GSA record is created. The DataRaptor log (DR_Create_GSA_From_Onboarding) shows a field mapping error: 'Field TPA_Account__c is not writable via DataRaptor upsert — relationship field requires Account ID format.'

**Root Cause Hypothesis:** The DataRaptor is passing the TPA Account Name string rather than the Account ID (18-character Salesforce ID) to the GSA.TPA_Account__c lookup field.

**Environment:** Full Sandbox (ACME-FS2)
**Frequency:** 100% reproducible when TPA flag = Yes; non-TPA path works correctly.
**Severity:** High — blocks onboarding of all TPA-administered groups

as per Description

## Solution

Fix the DataRaptor Transform (DR_Create_GSA_From_Onboarding): update the TPA_Account__c field mapping to reference the hidden OmniScript element 'TPA_Account_Id' (which stores the 18-character Account ID from the lookup component) rather than 'TPA_Account_Name' (display label). Additionally, add a pre-submit validation step in the OmniScript using a DataRaptor Extract to confirm the TPA Account ID resolves to an existing Account with Account.TPA_Certified__c = true before allowing submission. Update OmniScript version to v2.1 and regression test both TPA and non-TPA paths in ACME-FS2.
