## Description

**Bug Report — UAT Phase 2**

**Summary:** When a provider's NPI record is updated in the NPPES integration feed (daily batch sync), the expected Provider_Change_Notification__c (PCN) record is not being auto-generated on the corresponding Provider__c (Person Account) record. This breaks the compliance audit trail required for network management.

**Steps to Reproduce:**
1. Confirm a Provider__c record exists with Provider__c.NPI__c = '1234567890' in UAT sandbox.
2. Simulate an NPPES inbound update by running the 'NPPES_Daily_Sync_Batch' Apex batch job in UAT (execute via Developer Console: `Database.executeBatch(new NPPESDailySyncBatch(), 200);`).
3. Confirm the batch updates Provider__c.NPI_Last_Verified_Date__c and Provider__c.NPI_Status__c on the target record.
4. Navigate to the Provider__c record and check the Provider_Change_Notification__c related list.

**Expected Result:** A new Provider_Change_Notification__c record is created with PCN.Change_Type__c = 'NPI Update', PCN.Previous_Value__c = old NPI value, PCN.New_Value__c = new NPI value, PCN.Source__c = 'NPPES', and PCN.Status__c = 'Open'.

**Actual Result:** No Provider_Change_Notification__c record is created. The Provider__c fields are updated correctly but the PCN creation step is silently skipped.

**Root Cause Hypothesis:** The Record-Triggered Flow ('Provider_PCN_Generation_Flow') that creates PCN records appears to have its entry condition filtering on `ISCHANGED(Provider__c.TIN__c)` only — the NPI change condition (`ISCHANGED(Provider__c.NPI__c)`) was likely removed during a recent flow version deployment (suspected during INS-1350 deployment).

**Environment:** UAT Sandbox (CS112)
**Frequency:** 100% reproducible
**Impact:** Compliance — PCN records are required within 5 business days of any provider data change per network contract terms.

## Acceptance Criteria

As per Description. PCN records must be generated within the same transaction as the Provider__c field update for NPI changes. PCN.Change_Type__c must = 'NPI Update' and PCN.Source__c must = 'NPPES'. Regression test must confirm TIN-change PCN generation (existing behavior) is not broken by the fix.

## Solution

Investigate and restore the 'Provider_PCN_Generation_Flow' Record-Triggered Flow: (1) Open the active flow version and inspect the entry conditions — confirm ISCHANGED(NPI__c) condition was dropped. (2) Add back the NPI change condition using an OR logic group alongside the existing TIN change condition. (3) Ensure the Decision element downstream correctly branches on Change_Type__c value ('NPI Update' vs 'TIN Update') to populate PCN fields appropriately. (4) Add PCN.Previous_Value__c capture using a before-save flow variable storing the prior NPI value. (5) Activate the corrected flow version and deactivate the broken version. (6) Write Apex unit test 'NPPESNPIChangeTest' asserting PCN record creation on NPI field update. (7) Perform regression test on TIN change scenario in UAT.
