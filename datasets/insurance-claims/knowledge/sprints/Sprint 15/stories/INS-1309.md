## Description

As a Claims Operations Manager, I need the system to automatically validate that an active Fee_Schedule__c record exists for the CPT code and provider NPI combination on each Claim_Line__c before repricing is triggered, so that Acme Insurance does not inadvertently pay claims at incorrect rates or without a contractual basis, reducing financial leakage and compliance risk.

Currently, the repricing engine attempts to apply a fee schedule regardless of effective date coverage, sometimes pulling expired schedules and producing incorrect Allowed_Amount__c values on Claim_Line__c records.

## Acceptance Criteria

Given a Claim_Line__c record is submitted for repricing with a populated CPT_Code__c and associated Provider__c NPI,
When the FeeScheduleEngine validation runs,
Then the system queries Fee_Schedule__c WHERE Fee_Schedule__c.CPT_Code__c = Claim_Line__c.CPT_Code__c AND Fee_Schedule__c.Provider_NPI__c = Claim_Line__c.Provider_NPI__c AND Fee_Schedule__c.Effective_Date__c <= Claim_Line__c.Service_Date__c AND Fee_Schedule__c.Expiration_Date__c >= Claim_Line__c.Service_Date__c AND Fee_Schedule__c.Status__c = 'Active'.

Given a matching active Fee_Schedule__c is found,
When repricing completes,
Then Claim_Line__c.Allowed_Amount__c is populated with the value from Fee_Schedule__c.Allowed_Amount__c and Claim_Line__c.Fee_Schedule_Applied__c is set to the Fee_Schedule__c record ID.

Given NO matching active Fee_Schedule__c exists for the CPT/NPI/service date combination,
When the validation runs,
Then Claim_Line__c.Status__c is set to 'Pended - No Fee Schedule' and a Task is auto-created assigned to the Provider Network Specialist queue with Subject = 'Fee Schedule Gap: CPT [code] for NPI [npi]'.

Given Claim_Line__c.Status__c is set to 'Pended - No Fee Schedule',
When the parent Claim__c record is evaluated,
Then Claim__c.Pend_Reason__c is updated to include 'Fee Schedule Gap' and Claim__c.Status__c transitions to 'Pended' if not already.

-Given the fee schedule lookup falls back to a default UCR (Usual, Customary and Reasonable) table when no exact match exists- (deprecated — UCR fallback removed per compliance review CR-2024-88)

Given a batch of 500+ Claim_Line__c records are processed simultaneously during nightly adjudication,
When the fee schedule validation runs in bulk,
Then all records are processed within Apex governor limits using bulkified SOQL with a maximum of 1 query per batch execution context.

## Solution

Implement an Apex Trigger (ClaimLineRepricingTrigger) on Claim_Line__c BEFORE UPDATE, firing when Repricing_Requested__c changes to TRUE. The trigger calls a bulkified Apex service class (FeeScheduleValidationService) that performs a single SOQL query across all trigger.new records grouped by CPT_Code__c and Provider_NPI__c. Results are mapped in a nested Map<String, Map<String, Fee_Schedule__c>> keyed by NPI then CPT. Records with no match are flagged and a separate DML operation updates Status__c and creates Tasks via a List<Task> insert. A Platform Event (FeeScheduleGapEvent__e) is fired for real-time monitoring on the Claims Ops dashboard. Unit test coverage target: 95%+. Custom metadata type (Fee_Schedule_Config__mdt) used to configure lookback tolerance window (default 0 days) without code changes.
