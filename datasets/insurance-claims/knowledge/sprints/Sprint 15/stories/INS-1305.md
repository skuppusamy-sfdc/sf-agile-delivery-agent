## Description

**Bug Report – UAT Sprint 15**

**Summary:** During Claim__c adjudication, the Fee Schedule Engine Apex service (FeeScheduleLookupService) incorrectly returns a $0.00 Allowed_Amount__c for Claim_Line__c records where CPT_Code__c includes a modifier suffix (e.g., '99213-25', '27447-RT', '93000-26'). This results in Payment__c records being generated with Total_Paid_Amount__c = $0.00 and EOB records showing incorrect denial reason 'Fee Schedule Not Found' even though a valid Fee_Schedule__c record exists for the base CPT code.

**Steps to Reproduce:**
1. Create a Claim__c record with Status__c = 'Under Review' linked to an active PNA.
2. Add a Claim_Line__c with CPT_Code__c = '99213-25' (modifier -25 appended).
3. Confirm Fee_Schedule__c record exists for CPT_Code__c = '99213' (base code, no modifier) with Allowed_Amount__c = $150.00.
4. Trigger adjudication via the 'Adjudicate Claim' Flow action.
5. Observe Claim_Line__c.Allowed_Amount__c after adjudication.
6. Expected: Claim_Line__c.Allowed_Amount__c = $150.00 (matched on base CPT code '99213').
7. Actual: Claim_Line__c.Allowed_Amount__c = $0.00; Claim_Line__c.Adjudication_Status__c = 'Denied'; EOB.Denial_Reason__c = 'Fee Schedule Not Found'.

**Root Cause Hypothesis:** FeeScheduleLookupService.getRate() performs an exact-match SOQL query on Fee_Schedule__c.CPT_Code__c without stripping the modifier suffix before lookup. The query WHERE CPT_Code__c = '99213-25' returns 0 rows.

**Impact:** 1,847 Claim_Line__c records in UAT incorrectly denied. Estimated financial exposure in PROD: ~$2.3M in underpayments if not resolved before Phase 2 go-live.

**Environment:** UAT Sandbox (CS92); reproducible on all modifier formats: -25, -59, -RT, -LT, -26, -TC.

## Acceptance Criteria

As per Description.

Additionally: Given the fix is deployed, when FeeScheduleLookupService.getRate() is called with a CPT code containing any standard modifier suffix (hyphen-separated), then the service must first attempt an exact match, and if no record is found, must retry using the base CPT code (substring before '-') before returning $0.00.

## Solution

Update FeeScheduleLookupService.getRate() Apex method to implement a two-pass lookup: Pass 1 = exact match on CPT_Code__c; Pass 2 = substring match using String.substringBefore('-') if Pass 1 returns empty. Add Fee_Schedule__c.Modifier_Applicable__c (Checkbox) field to indicate records where modifier logic should apply a rate adjustment multiplier (configurable via Custom Metadata Type 'CPT_Modifier_Rules__mdt'). Write Apex unit tests covering: exact match, modifier -25, modifier -59, bilateral modifier -50, and professional component -26. Re-run adjudication for the 1,847 affected UAT Claim_Line__c records via a one-time Batch Apex re-adjudication job 'BatchReadjudicateAffectedLines'.
