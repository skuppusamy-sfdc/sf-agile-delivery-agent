## Description

As a Credentialing Specialist, I need Credentialing_Case__c records to be automatically routed to the correct credentialing queue based on the associated Provider__c specialty (Provider__c.Primary_Specialty__c) and the state of licensure (Provider__c.License_State__c), so that manual triage is eliminated and cases reach the right specialist within SLA. This story also incorporates the CAQH integration status check (Provider__c.CAQH_Status__c) as a routing factor.

## Acceptance Criteria

Given a new Credentialing_Case__c record is created with a linked Provider__c,
When Provider__c.Primary_Specialty__c = 'Behavioral Health' AND Provider__c.License_State__c IN ('CA', 'NY', 'TX'),
Then Credentialing_Case__c.OwnerId is set to Queue 'Credentialing_BH_West_East' and Credentialing_Case__c.Routing_Reason__c = 'Specialty: Behavioral Health | State: High-Volume'.

Given a new Credentialing_Case__c is created,
When Provider__c.Primary_Specialty__c = 'Cardiology' AND Provider__c.CAQH_Status__c = 'Not Enrolled',
Then Credentialing_Case__c.OwnerId is set to Queue 'Credentialing_CAQH_Remediation' and a Task is auto-created with Subject 'Initiate CAQH Enrollment' assigned to the queue owner.

Given a Credentialing_Case__c is created for a specialty not covered by any routing rule,
When no matching rule is found,
Then Credentialing_Case__c.OwnerId is set to Queue 'Credentialing_General' and Credentialing_Case__c.Routing_Reason__c = 'Default Routing — No Specialty Match'.

Given the routing assignment completes,
When Credentialing_Case__c.OwnerId is updated,
Then an email notification is sent to the assigned queue using the 'Credentialing_Case_Assignment' email template and Credentialing_Case__c.Assignment_Timestamp__c is stamped with the current datetime.

Given a Credentialing_Case__c is updated (not created) and the Provider__c specialty changes,
When the specialty change triggers a re-evaluation,
Then the routing logic re-runs and the case is reassigned if the new specialty maps to a different queue, with the previous queue logged in Credentialing_Case__c.Previous_Queue__c.

Given a bulk creation of 200+ Credentialing_Case__c records via Data Loader,
When the assignment logic fires,
Then all records are correctly routed within governor limits with no SOQL or DML exceptions.

-Given integration with QGenda is used to validate provider schedule availability before assignment (deprecated — QGenda integration scoped to INS-1388)-

## Solution

Implement a Record-Triggered Flow ('Credentialing_Case_Auto_Assignment_Flow') on Credentialing_Case__c, firing after-save on create and update (with an entry condition checking if Primary_Specialty__c or License_State__c changed). Flow uses Decision elements to evaluate specialty/state/CAQH combinations against a Custom Metadata Type ('Credentialing_Routing_Rule__mdt') storing queue mappings — this avoids hardcoded logic. Queue IDs resolved via a Get Records element on Group object. Email alerts via Flow Send Email action using org-wide address. For bulk safety, routing rule evaluation logic is extracted into an invocable Apex method ('CredentialingCaseRouter.assignQueue') called from the Flow, ensuring bulkification. Previous_Queue__c updated via a before-save entry on update path.
