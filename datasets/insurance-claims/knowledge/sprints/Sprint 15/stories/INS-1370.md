## Description

As a Credentialing Specialist, I need the system to automatically identify Provider__c records whose Credentialing_Case__c is approaching expiration and generate a Provider_Change_Notification__c (PCN) alert with a task assigned to the responsible Credentialing Specialist, so that re-credentialing workflows are initiated proactively and providers do not lapse into an unverified network status.

This requirement emerged from a Phase 2 UAT finding where three test providers had simulated credentialing lapses that were not caught by any automated process, posing a compliance risk flagged by the Compliance Officer persona.

## Acceptance Criteria

Given a Scheduled Flow runs nightly at 02:00 AM,
When it identifies Credentialing_Case__c records where Credentialing_Case__c.Expiration_Date__c is within 90 days of TODAY() and Credentialing_Case__c.Status__c = 'Active' and no open PCN of type 'Credentialing Renewal' exists for the linked Provider__c,
Then a Provider_Change_Notification__c record shall be created with PCN__c.Type__c = 'Credentialing Renewal', PCN__c.Status__c = 'Open', and PCN__c.Due_Date__c = Credentialing_Case__c.Expiration_Date__c - 60 days.

Given a PCN is created by the scheduled process,
When the PCN record is inserted,
Then a Task shall be auto-created via Flow with Task.Subject = 'Re-credentialing Required: [Provider Name]', Task.OwnerId = Credentialing_Case__c.Owner.Id, and Task.ActivityDate = PCN__c.Due_Date__c.

Given the Scheduled Flow completes its batch,
When more than 0 PCN records were created,
Then a summary email shall be sent to the Credentialing Team distribution list (credentialing-alerts@acmeinsurance.example.com) listing provider names, NPI numbers, and expiration dates.

Given a Credentialing_Case__c.Expiration_Date__c has already passed (lapsed),
When the Scheduled Flow runs,
Then the linked Provider__c.Network_Status__c shall be set to 'Suspended — Credentialing Lapsed' and a high-priority PCN with PCN__c.Priority__c = 'Critical' shall be created.

Given the Scheduled Flow is configured,
When it is deployed to production,
Then it shall be schedulable via Setup > Scheduled Jobs with a configurable run time and shall not consume more than 50% of the org's daily scheduled Apex limits based on estimated provider volume of 12,000 records.

## Solution

Implement a Scheduled Flow (SF_CredentialingExpirationMonitor) using a Get Records element querying Credentialing_Case__c with the 90-day window filter. Loop element processes each record: a Decision element checks for existing open PCN using a Get Records sub-query on Provider_Change_Notification__c. Create Records elements generate the PCN and associated Task. A separate Decision branch handles lapsed credentials and updates Provider__c.Network_Status__c via Update Records. Post-loop, a custom Apex Action (CredentialingAlertEmailService.cls) sends the summary email using a Salesforce Email Template and OrgWideEmailAddress. Flow bulkification ensured by using collection variables and single DML operations per loop iteration. Deployment includes a Custom Metadata Type (Credentialing_Alert_Config__mdt) for configurable threshold days (default: 90) and email recipient list, avoiding hardcoded values.
