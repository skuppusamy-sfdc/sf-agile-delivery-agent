## Description

As a Claims Operations Manager, I need the system to automatically determine Coordination of Benefits (COB) priority order when a Claim__c is submitted against a Policy__c that has a secondary payer relationship recorded, so that payment responsibilities are correctly sequenced between carriers and overpayment risk is minimized.

This story extends INS-1337 (Primary Payer Integration) by adding secondary and tertiary payer adjudication logic. UAT testing in Sprint 14 revealed that claims with COB flags were being fully adjudicated against the primary payer without triggering secondary payer workflows, resulting in incorrect Payment__c records.

## Acceptance Criteria

Given a Claim__c record is submitted with Claim__c.COB_Flag__c = true and a related Secondary_Payer__c lookup populated,
When the claim reaches status Claim__c.Status__c = 'Pending Adjudication',
Then the system shall create a COB_Coordination__c child record linked to the Claim__c with COB_Coordination__c.Priority_Order__c values of 'Primary', 'Secondary' derived from Policy__c.COB_Rule__c.

Given the primary payer adjudication is complete and Payment__c.Payer_Type__c = 'Primary' is recorded,
When Claim__c.Status__c transitions to 'Awaiting COB',
Then a Platform Event COB_Adjudication_Event__e shall fire and the secondary payer workflow shall be triggered automatically within 2 minutes.

Given the secondary payer Payment__c record is created,
When the sum of Payment__c.Allowed_Amount__c across all payers equals or exceeds Claim__c.Total_Billed_Amount__c,
Then Claim__c.COB_Overpayment_Flag__c shall be set to true and a Case shall be auto-created with Subject = 'COB Overpayment Review Required'.

Given a Claim__c with COB_Flag__c = true,
When the LOB on Policy__c.Line_of_Business__c = 'Medicare Advantage',
Then CMS coordination rules shall apply and the system shall enforce Medicare-as-secondary logic per CMS MSP guidelines.

-Given COB determination fails, the claim shall be routed to a manual queue with priority 'Critical'-
(Superseded: claims now route to SIU queue only if fraud indicators are present per compliance review.)

Given COB processing completes successfully,
When all Payment__c records are finalized,
Then the Explanation_of_Benefits__c (EOB) record shall reflect all payer contributions with EOB__c.COB_Summary__c populated and EOB__c.Patient_Responsibility__c correctly calculated.

## Solution

Implement a Record-Triggered Flow on Claim__c that fires on update when COB_Flag__c = true and Status__c changes to 'Pending Adjudication'. Flow invokes an Apex Action (CobPriorityDeterminationService.cls) that queries Policy__c and related Secondary_Payer__c records to build the priority chain. COB_Coordination__c records are inserted via the Apex service. A Platform Event (COB_Adjudication_Event__e) is published upon primary Payment__c creation. A separate Platform Event-Triggered Flow subscribes to this event and initiates the secondary payer sub-flow. Overpayment detection logic runs in an Apex Trigger (AfterInsert/AfterUpdate) on Payment__c. EOB population handled via existing PaymentEngine DataRaptor Transform (DR_EOB_COBSummary). Permission Set Group 'Claims_COB_Processor' grants access to COB_Coordination__c object.
