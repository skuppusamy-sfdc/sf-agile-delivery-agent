## Description

Bug introduced in Sprint 18 deployment. After ERA inbound integration posts remittance data, Claim_Line__c.Status__c remains stuck at 'Pending Adjudication' instead of transitioning to 'Adjudicated'. The Fee_Schedule__c repricing calculation completes successfully (confirmed via debug logs), but the downstream status update on Claim_Line__c is not firing.

**Steps to Reproduce:**
1. Create a Claim__c record with LOB = 'Medical' and at least two Claim_Line__c child records with CPT codes 99213 and 99214.
2. Associate a valid Fee_Schedule__c record via the parent Provider_Network_Agreement__c (PNA).
3. Trigger the ERA inbound Platform Event (ERA_Remittance__e) via the integration sandbox endpoint.
4. Observe Claim_Line__c.Status__c after the Platform Event trigger fires.

**Expected:** Claim_Line__c.Status__c = 'Adjudicated' and Claim_Line__c.Allowed_Amount__c populated with repriced value from Fee_Schedule__c.

**Actual:** Claim_Line__c.Status__c remains 'Pending Adjudication'. Claim_Line__c.Allowed_Amount__c is correctly populated, confirming repricing ran. Status update is silently failing.

## Solution

Root cause identified: the Record-Triggered Flow 'Update_ClaimLine_Status_On_ERA' has a faulty entry condition checking Claim_Line__c.Allowed_Amount__c != null BEFORE the ERA Platform Event subscriber Apex class (ERARemittanceEventHandler) commits the allowed amount. Race condition between async Platform Event handler and synchronous Flow execution. Fix: convert status update logic from Record-Triggered Flow into the ERARemittanceEventHandler Apex class directly, updating Claim_Line__c.Status__c = 'Adjudicated' in the same DML transaction as Claim_Line__c.Allowed_Amount__c. Add a try/catch with custom exception logging to ClaimProcessingException__c object. Deploy updated Apex class and deactivate the conflicting Record-Triggered Flow. Regression test with ERA test harness covering single-line and multi-line claims.
