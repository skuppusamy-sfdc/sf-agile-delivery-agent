## Description

**Bug Report — Discovered during Sprint 19 regression testing of Fee Schedule Engine**

When a Claim_Line__c record has more than one CPT modifier populated (e.g., Modifier_1__c = '25' and Modifier_2__c = 'GT'), the Fee Schedule Engine applies only the reimbursement rule for Modifier_1__c and ignores Modifier_2__c entirely, resulting in an incorrect Allowed_Amount__c calculation.

**Steps to Reproduce:**
1. Create a Claim_Line__c with: CPT_Code__c = '99213', Modifier_1__c = '25', Modifier_2__c = 'GT', Billed_Amount__c = 150.00.
2. Ensure a Fee_Schedule__c record exists with modifier rules for both '25' (separate E&M modifier, 100% reimbursement) and 'GT' (telehealth modifier, applies 80% of allowed amount per GSA).
3. Trigger adjudication by setting Claim__c.Status__c = 'Ready for Adjudication'.
4. Observe Claim_Line__c.Allowed_Amount__c after adjudication completes.

**Expected Behavior:**
Allowed_Amount__c should reflect the combined modifier logic: base fee schedule rate for 99213 × modifier '25' rule × modifier 'GT' rule = e.g., $120.00 × 1.00 × 0.80 = $96.00.

**Actual Behavior:**
Allowed_Amount__c = $120.00 — only the '25' modifier is applied; 'GT' telehealth reduction is ignored.

**Impact:**
Affects all telehealth claims with compound modifiers across Commercial and Medicare Advantage LOBs. Estimated overpayment exposure: ~$34,000 across claims processed in Sprint 18 production deployment window.

**Environment:** Production (mirrored in Full Sandbox). Regression introduced in INS-1591 Fee Schedule Engine refactor.

## Solution

1. Identify the defect in FeeScheduleCalculatorService.cls — the modifier lookup currently breaks after finding the first matching modifier rule in Fee_Schedule__c modifier child records (Fee_Schedule_Modifier_Rule__c) due to a premature return statement in the modifier iteration loop.
2. Refactor the modifier resolution method to collect ALL matching Fee_Schedule_Modifier_Rule__c records for a given CPT code and apply each modifier's Adjustment_Factor__c multiplicatively in priority order (Modifier_Priority__c field on the rule record).
3. Add a unit test (FeeScheduleCalculatorTest.cls) specifically covering dual-modifier scenarios: 25+GT, 59+95, and 26+TC combinations against mock Fee_Schedule__c data.
4. Run a data fix script (AnonymousApex) in the Full Sandbox to recalculate Allowed_Amount__c for all Claim_Line__c records where Modifier_2__c != null AND Claim__r.Status__c = 'Adjudicated' AND CreatedDate >= [Sprint 18 deploy date] — validate results before running in Production.
5. Coordinate with Claims Operations Manager to identify and reprocess affected Payment__c records via the existing Payment Reversal & Reissuance flow.
