## Description

**Bug Report — Regression from INS-1601 Fee Schedule batch repricing changes**

**Steps to Reproduce:**
1. Navigate to an existing Claim__c record in 'Pending Adjudication' status with 3 or more Claim_Line__c child records.
2. Trigger the nightly batch repricing job (ClaimRepricingBatch) manually via Anonymous Apex or wait for scheduled run.
3. Confirm all Claim_Line__c records have Adjudication_Status__c = 'Priced' after batch completes.
4. Observe the parent Claim__c.Status__c field.

**Expected Result:** Parent Claim__c.Status__c should roll up to 'Adjudicated' when all child Claim_Line__c records reach Adjudication_Status__c = 'Priced'.

**Actual Result:** Claim__c.Status__c remains stuck at 'Pending Adjudication'. The roll-up logic in the Record-Triggered Flow (RTF_ClaimLineStatusRollup) is not firing because the batch job updates Claim_Line__c records via Database.update() in a context that bypasses the after-update flow trigger entry criteria.

**Impact:** Claims Operations Managers cannot identify adjudicated claims for payment release. Payment__c disbursement queue is blocked. Approximately 340 claims affected in UAT environment.

## Solution

Root cause: ClaimRepricingBatch was using a static flag (TriggerBypass__c = true) set on a Custom Metadata record to suppress recursive triggers during batch, inadvertently suppressing RTF_ClaimLineStatusRollup. Fix: Introduce a scoped bypass pattern using a Trigger Handler static Boolean (ClaimLineTriggerHandler.bypassRollup) that is only set for the specific recursive scenario, not globally. Re-enable RTF_ClaimLineStatusRollup for the after-update context when called from batch. Add a compensating batch step (ClaimStatusReconciliationBatch) to retroactively correct Claim__c.Status__c for the 340 impacted records. Unit tests updated to assert Claim__c.Status__c post-batch.
