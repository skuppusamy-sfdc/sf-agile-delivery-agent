## Description

As a Credentialing Specialist, I need Salesforce to automatically create a Credentialing_Case__c record when CAQH posts an attestation-complete webhook event so that manual data entry is eliminated and newly attested providers enter the credentialing queue within minutes rather than the current 24–48 hour lag caused by manual intake.

CAQH sends a JSON webhook payload to our Named Credential endpoint containing: provider NPI, CAQH profile ID, attestation date, and attestation type (Initial | Re-attestation | Update). The integration must match the incoming NPI to an existing Provider__c record or create a new one if no match is found, then create the Credentialing_Case__c and route it to the appropriate Omni-Channel queue based on attestation type and LOB.

## Acceptance Criteria

Given the CAQH webhook fires with attestation_type = 'Initial',
When the inbound REST endpoint processes the payload,
Then a new Credentialing_Case__c is created with: Credentialing_Case__c.Record_Type = 'Initial_Credentialing', Credentialing_Case__c.CAQH_Profile_ID__c = payload.caqhProfileId, Credentialing_Case__c.Attestation_Date__c = payload.attestationDate, Credentialing_Case__c.Status__c = 'Pending Review', and Credentialing_Case__c.Provider__c = matched Provider__c.Id.

Given the incoming NPI does not match any existing Provider__c.NPI__c,
When the endpoint processes the payload,
Then a new Provider__c (Person Account) record is created with Provider__c.NPI__c = payload.npi, Provider__c.CAQH_Profile_ID__c = payload.caqhProfileId, Provider__c.Credentialing_Status__c = 'In Process', and the new Credentialing_Case__c is linked to this new Provider__c.

Given a Credentialing_Case__c is created with Record_Type = 'Initial_Credentialing',
When the record is saved,
Then it is pushed to the Omni-Channel queue 'Initial_Credentialing_Queue' and Credentialing_Case__c.OwnerId = Initial_Credentialing_Queue.Id.

Given the webhook fires with attestation_type = 'Re-attestation',
When the Credentialing_Case__c is created,
Then Record_Type = 'Re_Credentialing' and the case is routed to 'Recredentialing_Queue'.

Given the CAQH webhook payload is malformed or missing required fields (npi, caqhProfileId),
When the endpoint attempts to process it,
Then no records are created, a Platform Event (Integration_Error__e) is published with Integration_Error__e.Source__c = 'CAQH_Webhook', Integration_Error__e.Payload__c = raw JSON, and HTTP 400 is returned to CAQH.

Given a duplicate Credentialing_Case__c already exists (same Provider__c and Status__c IN ('Pending Review','In Review')),
When a second webhook fires for the same NPI,
Then no new case is created and HTTP 200 is returned with response body indicating 'duplicate_suppressed'.

-Given attestation_type = 'Update', route to 'Initial_Credentialing_Queue'- (superseded — 'Update' type now routes to 'Provider_Data_Update_Queue' per compliance directive CR-2024-047)

## Solution

Implement an Apex REST resource class (CAQH_AttestationWebhookHandler) exposed via a Connected App with IP whitelisting for CAQH's published CIDR ranges. Use Named Credential 'CAQH_Inbound_NC' for endpoint registration. NPI matching via SOQL on Provider__c.NPI__c with FOR UPDATE to prevent duplicate inserts under concurrent webhook delivery. Record creation wrapped in a Savepoint/rollback pattern. Omni-Channel routing via PendingServiceRouting object insert with ServiceChannel = 'Credentialing_Cases_SC'. Platform Event published on error via EventBus.publish(). Custom Permission Set 'CAQH_Integration_PS' grants access to the REST endpoint profile. Unit tests to cover: happy path Initial, Re-attestation, malformed payload, duplicate suppression.
