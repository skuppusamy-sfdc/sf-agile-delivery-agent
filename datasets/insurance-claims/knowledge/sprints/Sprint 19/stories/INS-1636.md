## Description

As a Provider Network Specialist, I need a structured termination workflow on the Network_Termination_Case__c object that enforces the 90-day advance notice requirement, generates a PCN for the terminating provider, and produces a downstream claims impact report showing all open Claim__c records associated with the provider so that terminations are handled in compliance with CMS and state regulatory requirements and claims teams can proactively manage in-flight claims.

Extends INS-1612 (Network Termination Case object setup). This story adds the workflow automation and impact assessment capability.

## Acceptance Criteria

Given a Network_Termination_Case__c is created with Termination_Reason__c populated and linked Provider__c:
When Termination_Case_Status__c = 'Initiated':
Then the system calculates Advance_Notice_Deadline__c = TODAY() + 90 and sets Proposed_Termination_Date__c = TODAY() + 90 (adjustable by the specialist, minimum TODAY() + 90).

Given Proposed_Termination_Date__c is manually set to less than TODAY() + 90:
When the record is saved:
Then a validation error fires: 'Proposed Termination Date must be at least 90 days from today to comply with CMS advance notice requirements.'

Given Termination_Case_Status__c transitions to 'Notice Sent':
When the record is saved:
Then a Provider_Change_Notification__c is auto-created with PCN_Type__c = 'Network Termination Notice', Related_Provider__c = terminating provider, Effective_Date__c = Proposed_Termination_Date__c, and PCN_Status__c = 'Distributed'.

Given a Network_Termination_Case__c exists with a linked Provider__c:
When the 'Run Claims Impact Assessment' quick action is triggered:
Then the system queries all Claim__c records WHERE Claim__c.Billing_Provider__c = Provider__c.Id AND Claim__c.Status__c IN ('Submitted', 'In Adjudication', 'Pending Information') and displays results in a modal table showing: Claim_Number__c, Status__c, Total_Billed_Amount__c, Date_of_Service__c.

Given the Claims Impact Assessment returns 1 or more open claims:
When results are displayed:
Then a summary field Network_Termination_Case__c.Open_Claims_Count__c is updated with the count and Network_Termination_Case__c.Total_Open_Claims_Billed__c is updated with the SUM of Claim__c.Total_Billed_Amount__c.

Given the Proposed_Termination_Date__c is reached:
When a scheduled batch runs nightly:
Then Provider_Network_Agreement__c.Contract_Status__c for the terminating provider is updated to 'Terminated' and Provider__c (Person Account) Network_Status__c = 'Out of Network'.

## Solution

1. Add a Validation Rule `VR_Termination_MinNoticeDate` on Network_Termination_Case__c: `Proposed_Termination_Date__c < (TODAY() + 90)` with error message per compliance requirement.
2. Build a Record-Triggered Flow `RTF_Termination_Advance_Notice` on Network_Termination_Case__c BEFORE SAVE on Create to auto-populate Advance_Notice_Deadline__c and Proposed_Termination_Date__c using a Formula element.
3. Add a second RTF path AFTER SAVE on Status__c change to 'Notice Sent' to create Provider_Change_Notification__c via Create Records element, reusing the PCN creation pattern established in INS-1632.
4. Build an LWC `claimsImpactAssessmentModal` invoked via a Quick Action on Network_Termination_Case__c. The LWC calls an Apex controller `TerminationClaimsImpactService.getOpenClaims(providerId)` and renders results in a lightning-datatable. On confirmation, invokes a second Apex method to write summary totals back to the case.
5. Create a Scheduled Apex class `ProviderTerminationActivationBatch` running nightly that queries Network_Termination_Case__c WHERE Proposed_Termination_Date__c = TODAY() AND Termination_Case_Status__c = 'Notice Sent' and updates PNA.Contract_Status__c and Provider__c.Network_Status__c.
6. Add a Sharing Rule ensuring Network_Termination_Case__c records are visible to the Provider_Network_Ops public group.
