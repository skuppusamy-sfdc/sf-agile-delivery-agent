## Description

**Bug Report — Performance Regression**

Bulk Claim_Line__c adjudication jobs (batch size > 200 records) are timing out during the Fee Schedule lookup phase. The Apex batch class 'FeeScheduleAdjudicationBatch' is executing SOQL queries inside a loop against Fee_Schedule__c and related Fee_Schedule_Line__c records, resulting in CPU time limit exceptions and governor limit violations in production-scale data volumes.

**Steps to Reproduce:**
1. Execute 'FeeScheduleAdjudicationBatch' with a scope of 250+ Claim_Line__c records containing diverse CPT codes (>80 unique CPT codes in the batch).
2. Monitor Apex debug logs — observe repeated SOQL queries: SELECT Id, Allowed_Amount__c FROM Fee_Schedule_Line__c WHERE CPT_Code__c = :cptCode AND Fee_Schedule__c = :fscId inside execute() loop.
3. After ~180 records processed, System.LimitException: Too many SOQL queries: 101 is thrown.
4. Batch job aborts; affected Claim_Line__c records remain in Status__c = 'Pending Adjudication'.

**Expected:** Batch completes successfully for 250+ records within governor limits.
**Actual:** LimitException thrown; partial batch failure; no retry mechanism.

**Impact:** Claims Operations team reports ~340 Claim_Line__c records stuck daily since Sprint 18 deployment.

## Solution

1. Refactor 'FeeScheduleAdjudicationBatch.execute()' to collect all unique CPT codes and Fee_Schedule__c Ids from the batch scope BEFORE the processing loop.
2. Execute a single bulk SOQL query: SELECT Id, CPT_Code__c, Allowed_Amount__c, Fee_Schedule__c FROM Fee_Schedule_Line__c WHERE CPT_Code__c IN :cptCodeSet AND Fee_Schedule__c IN :fscIdSet and load results into a Map<String, Fee_Schedule_Line__c> keyed by 'CPT|FeeScheduleId'.
3. Replace all in-loop SOQL with Map lookups.
4. Add a Custom Metadata Type 'Batch_Config__mdt' record for FeeScheduleAdjudicationBatch with Batch_Size__c = 150 to reduce scope and stay within CPU limits.
5. Implement a retry Platform Event 'Claim_Line_Adjudication_Retry__e' published for any Claim_Line__c that fails, consumed by a separate lightweight retry flow.
6. Add SOQL and CPU time assertions in the Apex test class using Test.startTest()/stopTest() to catch regressions.
