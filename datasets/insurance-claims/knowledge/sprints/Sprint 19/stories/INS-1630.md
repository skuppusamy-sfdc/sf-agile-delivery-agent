## Description

**Bug Report — Regression identified in Sprint 19 regression cycle**

When the nightly batch job bulk-closes Claim__c records with Status__c = 'Adjudicated', the associated Claim_Line__c child records are NOT updating their Line_Status__c field to 'Closed'. This was working correctly in Sprint 17 prior to the Fee Schedule Engine refactor (INS-1588).

**Steps to Reproduce:**
1. Create a Claim__c record with Status__c = 'In Adjudication' and at least 3 child Claim_Line__c records with Line_Status__c = 'Pending'.
2. Run the nightly batch class `ClaimBulkClosureBatch` via Anonymous Apex or schedule trigger.
3. Verify Claim__c.Status__c updates to 'Adjudicated' — this succeeds.
4. Navigate to the child Claim_Line__c records and observe Line_Status__c — remains 'Pending' instead of updating to 'Closed'.

**Expected Behavior:** All child Claim_Line__c records with Line_Status__c IN ('Pending', 'In Review') should update to Line_Status__c = 'Closed' when parent Claim__c.Status__c transitions to 'Adjudicated' via batch.

**Actual Behavior:** Claim_Line__c.Line_Status__c remains 'Pending'. No error is thrown; the batch completes with 0 failed records, masking the issue.

**Root Cause Hypothesis:** The Record-Triggered Flow `Update_Claim_Lines_On_Closure` has a entry condition checking `Claim__c.LastModifiedById = UserInfo.getUserId()`, which evaluates to false in batch context since the running user is the integration service account. The flow silently skips execution.

## Solution

1. Modify the Record-Triggered Flow `Update_Claim_Lines_On_Closure` to remove the `LastModifiedById` user-context entry condition — replace with a field-change condition: `ISCHANGED(Status__c) AND Status__c = 'Adjudicated'`.
2. Add a bulkification-safe Apex fallback in `ClaimBulkClosureBatch.finish()` that queries all Claim_Line__c records WHERE Claim__c IN :processedClaimIds AND Line_Status__c IN ('Pending','In Review') and performs a bulk DML update.
3. Add a Platform Event `Claim_Closure_Event__e` fired post-batch to trigger downstream EOB generation — ensuring the Payment Engine picks up the closed lines.
4. Deploy updated Flow version and batch class to sandbox; run regression test with 200+ claim records to validate governor limit compliance.
