## Description

As a Provider Network Specialist, I need Salesforce to automatically generate a Provider_Change_Notification__c (PCN) record whenever a provider's NPI__c or TIN__c field is modified on the Provider__c (Person Account) record, so that downstream systems, credentialing teams, and contracted payers are notified of the data change in compliance with CMS provider data accuracy requirements and internal SLA obligations.

This automation must capture both the old and new values for audit purposes and route the PCN to the appropriate review queue based on the change type.

## Acceptance Criteria

Given a Provider__c record where NPI__c is updated by any user,
When the record is saved,
Then a Provider_Change_Notification__c record is automatically created with: PCN__c.Provider__c = Provider__c.Id, PCN__c.Change_Type__c = 'NPI Update', PCN__c.Previous_Value__c = old NPI value, PCN__c.New_Value__c = new NPI value, PCN__c.Status__c = 'Pending Review', PCN__c.Created_Date__c = system timestamp.

Given a Provider__c record where TIN__c is updated,
When the record is saved,
Then a PCN is created with PCN__c.Change_Type__c = 'TIN Update' and the old/new TIN values masked to show only last 4 digits in PCN__c.Previous_Value__c and PCN__c.New_Value__c (e.g., '***-**-1234') for PII compliance.

Given both NPI__c and TIN__c are changed in the same save operation,
When the record is saved,
Then TWO separate PCN records are created — one for each change type — not a single combined record.

Given a PCN is created with Change_Type__c = 'NPI Update',
When the PCN is saved,
Then it is automatically assigned to Queue: 'Provider_NPI_Review_Queue' via Assignment Rule.

Given a PCN is created with Change_Type__c = 'TIN Update',
When the PCN is saved,
Then it is assigned to Queue: 'Provider_TIN_Compliance_Queue' and PCN__c.Compliance_Flag__c is set to TRUE.

Given the PCN creation fails due to a system error,
When the failure occurs,
Then the originating Provider__c field update is NOT rolled back, but an error log entry is written to the Custom Metadata-backed Error_Log__c object and the Provider Network Specialist receives an in-app notification.

-Given the provider is flagged as TPA-managed, suppress PCN creation and route to the TPA integration queue instead.- (Requirement removed — TPA routing handled by separate integration middleware, not Salesforce-native)

## Solution

1. Implement an Apex Trigger (ProviderChangeTrigger.cls) on Account (Person Account / Provider__c) firing on Before Update to capture old NPI__c and TIN__c values into a map, and After Update to invoke a handler class (ProviderChangePCNHandler.cls).
2. PCNHandler iterates changed records, instantiates PCN sObjects for each changed field, applies TIN masking via a utility method (TINMaskingUtil.maskTIN()), and performs a bulkified DML insert.
3. Use a Record-Triggered Flow as an alternative for Assignment Rule routing — After Insert on Provider_Change_Notification__c, evaluate Change_Type__c and set OwnerId to the appropriate Queue Id retrieved via a Get Records element.
4. Configure a Custom Metadata Type (PCN_Routing_Config__mdt) to store Queue API names per Change_Type__c value, allowing routing changes without code deployments.
5. Wrap PCN insert in a try/catch; on exception, use a Platform Event (PCNCreationError__e) to trigger an async error logging flow that writes to Error_Log__c and sends a Custom Notification to the triggering user.
