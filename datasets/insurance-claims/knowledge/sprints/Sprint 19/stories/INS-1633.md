## Description

**Bug Report — Regression introduced in Sprint 18 Conga CLM template upgrade**

The Explanation_of_Benefits__c (EOB) PDF generation via Conga CLM is silently failing for any Claim__c that has more than 50 associated Claim_Line__c records. The Conga merge job returns a 200 HTTP status but the generated document is either a blank PDF or contains only the header section with no line-item detail.

**Steps to Reproduce:**
1. Identify or create a Claim__c record with exactly 51 or more Claim_Line__c child records, all with Line_Status__c = 'Adjudicated'.
2. Trigger EOB generation by setting Explanation_of_Benefits__c.EOB_Status__c = 'Ready for Generation' and saving.
3. The Conga merge is invoked via the `EOBGenerationQueueable` Apex class.
4. Observe: Conga API returns HTTP 200. The PDF is attached to the EOB record but contains only the header/footer — no Claim_Line__c data rows.
5. Reproduce with 49 lines — PDF generates correctly.

**Expected:** Full EOB PDF including all Claim_Line__c rows: CPT code (Claim_Line__c.CPT_Code__c), billed amount (Claim_Line__c.Billed_Amount__c), allowed amount (Claim_Line__c.Allowed_Amount__c), and patient responsibility (Claim_Line__c.Patient_Responsibility__c).

**Actual:** PDF body is empty beyond page 1 header. No error is surfaced to the Claims Operations Manager or logged in Integration_Error_Log__c.

**Root Cause Hypothesis:** The Conga CLM template `EOB_Master_Template_v3` uses a SOQL query in the template data source limited to 50 rows (hard-coded LIMIT 50 added during Sprint 18 template revision). This needs to be removed and replaced with server-side pagination or chunked merge.

## Acceptance Criteria

as per Description

## Solution

1. Update the Conga CLM template `EOB_Master_Template_v3` data source SOQL query: remove hard-coded `LIMIT 50` from the Claim_Line__c subquery. Replace with dynamic query using Conga's `&QueryId` parameter referencing a named query `QID_ClaimLines_For_EOB` with no row limit.
2. Update `EOBGenerationQueueable` to pass the full Claim__c Id as a merge parameter and ensure the Conga API callout uses the updated QueryId.
3. Implement chunked rendering: if Claim_Line__c count > 100, split into multiple Conga merge calls and use Conga's document assembly feature to concatenate PDFs before attaching to Explanation_of_Benefits__c.
4. Add explicit error handling in `EOBGenerationQueueable` — if the Conga response body ContentLength < 5000 bytes for a claim with >10 lines, log to Integration_Error_Log__c with Error_Source__c = 'Conga EOB Generation' and set EOB_Status__c = 'Generation Failed'.
5. Write a test class with @testSetup creating 75 Claim_Line__c records and mock the Conga HTTP callout to validate error handling paths.
