## Description

As an Underwriting Analyst, I need Salesforce to validate that a new Group_Service_Agreement__c (GSA) effective date is not earlier than the earliest Policy__c inception date for the associated employer group account, so that we prevent retroactive GSA coverage gaps that create compliance exposure and billing discrepancies during group onboarding.

This validation should surface a clear, actionable error message on the GSA record creation screen rather than silently failing or allowing the save.

## Acceptance Criteria

Given an Underwriting Analyst creates a new Group_Service_Agreement__c record linked to an Account (Employer Group),
When GSA__c.Effective_Date__c is set to a date EARLIER than the earliest Policy__c.Inception_Date__c among all active policies under that Account,
Then the save is blocked with the error message: 'GSA Effective Date cannot precede the earliest active Policy inception date for this group ([Policy Inception Date]). Please review and correct.'

Given GSA__c.Effective_Date__c equals or is later than the earliest Policy__c.Inception_Date__c,
When the record is saved,
Then the GSA is created successfully with Status__c = 'Pending Activation' and no validation error is shown.

Given the employer group Account has NO active Policy__c records (Policy__c.Status__c = 'Active'),
When the GSA is created,
Then the date validation is skipped and a soft warning toast message is displayed: 'No active policies found for this group — GSA created without policy date alignment check.'

Given the validation fires,
When the error is triggered,
Then the error is also logged to Error_Log__c with Error_Type__c = 'GSA Date Validation', Record_Id__c = the GSA Id (if available), and Triggered_By__c = running user.

-Given the GSA is created via the Conga CLM integration, bypass Salesforce-side date validation and rely on Conga contract rules.- (Removed — Conga CLM integration does not currently pass effective date into the GSA record on creation; Salesforce validation must cover all entry paths)

## Solution

1. Implement a Record-Triggered Flow 'GSA_EffectiveDate_Validation_Flow' on Group_Service_Agreement__c, firing Before Save on Create.
2. Flow uses a Get Records element to query MIN(Policy__c.Inception_Date__c) WHERE Account__c = GSA__c.Account__c AND Status__c = 'Active'.
3. If no policies found, flow sets a flow variable $noActivePolicies = TRUE and proceeds to a custom notification action (no blocking error).
4. If policies exist and GSA__c.Effective_Date__c < MIN(Inception_Date__c), flow calls a custom error via the 'Add Error' element on the Effective_Date__c field with the parameterized message.
5. Error logging to Error_Log__c is handled by a separate After Save flow path using a Fault connector to capture any unexpected errors during the Get Records query, writing to Error_Log__c via a Create Records element.
