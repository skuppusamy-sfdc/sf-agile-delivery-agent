## Description

**Bug Report — Regression identified in Sprint 19 regression cycle**

During high-volume EOB generation batch jobs (>2,500 records), Claim_Line__c.Adjudication_Status__c fails to update from 'Pending' to 'Adjudicated' even when the parent Explanation_of_Benefits__c record is successfully created and linked.

**Steps to Reproduce:**
1. Queue a batch EOB generation job with at least 2,500 Claim__c records across mixed LOBs (Commercial, Medicare Advantage).
2. Confirm batch completes with 0 reported failures in the Apex Job log.
3. Query Claim_Line__c records associated with the processed Claims: SELECT Id, Adjudication_Status__c, EOB__c FROM Claim_Line__c WHERE Claim__r.Status__c = 'Adjudicated'
4. Observe that approximately 12–18% of Claim_Line__c records still show Adjudication_Status__c = 'Pending' despite EOB__c being populated.
5. Manually re-trigger the status update flow on an affected record — status updates correctly.

**Expected Behavior:**
All Claim_Line__c.Adjudication_Status__c values should transition to 'Adjudicated' within the same batch transaction that generates the parent EOB record.

**Actual Behavior:**
Roughly 12–18% of Claim_Line__c records remain in 'Pending' status post-batch, causing downstream ERA file generation to exclude those lines and resulting in partial provider payments.

**Impact:**
Payment__c records generated from affected claims are understated. Provider Relations Specialists are receiving escalations from Provider accounts regarding short payments on ERA remittance files.

**Root Cause Hypothesis:**
Suspected governor limit collision on DML rows within the batch execute() method when chunk size hits 200 records and the Record-Triggered Flow on Claim_Line__c fires synchronously, consuming additional DML statements beyond the available limit.

as per Description

## Solution

1. Refactor the EOB batch class (EOBGenerationBatch.cls) to separate the Claim_Line__c status update DML into a dedicated chained Queueable (ClaimLineStatusUpdateQueueable.cls) invoked via System.enqueueJob() at the end of each batch execute() chunk, bypassing the synchronous DML row ceiling.
2. Deactivate the Record-Triggered Flow 'Update Claim Line on EOB Creation' that fires After Insert on Explanation_of_Benefits__c — replace its logic with the new Queueable to eliminate redundant DML.
3. Add a Platform Event (ClaimLineStatusEvent__e) as a fallback audit mechanism: if the Queueable fails, publish the event and a separate subscriber flow flags the Claim_Line__c with a Processing_Error__c = true field for manual review.
4. Update batch chunk size from 200 to 150 as a short-term governor limit buffer while the Queueable refactor is validated in the full regression suite.
5. Add unit tests covering >2,500 record scenarios using Test.startTest()/stopTest() with mock EOB data across Commercial and Medicare Advantage LOBs — target 95% code coverage.
