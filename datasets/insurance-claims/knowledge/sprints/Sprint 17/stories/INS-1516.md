## Description

**Bug Report**

**Summary:** During claims adjudication, the Fee Schedule lookup logic is matching Fee_Schedule__c records using Claim__c.Submission_Date__c instead of Claim__c.Date_of_Service__c as the effective date anchor, causing incorrect allowed amount calculations on Claim_Line__c records when a fee schedule was updated between the date of service and the submission date.

**Steps to Reproduce:**
1. Create a Provider__c with two Fee_Schedule__c records for CPT code 99213:
   - Fee Schedule A: Effective_Date__c = 01/01/2024, Expiration_Date__c = 06/30/2024, Allowed_Amount__c = 85.00
   - Fee Schedule B: Effective_Date__c = 07/01/2024, Expiration_Date__c = 12/31/2024, Allowed_Amount__c = 92.00
2. Create a Claim__c with Date_of_Service__c = 06/15/2024 and Submission_Date__c = 07/10/2024.
3. Add a Claim_Line__c with CPT_Code__c = '99213' and run adjudication.
4. Observe Claim_Line__c.Allowed_Amount__c.

**Expected Behavior:** Claim_Line__c.Allowed_Amount__c = 85.00 (Fee Schedule A, based on Date_of_Service__c = 06/15/2024).

**Actual Behavior:** Claim_Line__c.Allowed_Amount__c = 92.00 (Fee Schedule B, incorrectly using Submission_Date__c = 07/10/2024 as the lookup date).

**Impact:** Payment leakage risk. Affects all claims where submission date crosses a fee schedule boundary. Estimated 340 Claim_Line__c records in UAT environment may be mispriced.

**Environment:** Full Sandbox (Acme-FS1), UAT Sandbox (Acme-UAT2)
**Regression:** Introduced in Sprint 14 during FeeScheduleEngine refactor (INS-1389).

## Acceptance Criteria

As per Description. Fee schedule effective date lookup must use Claim__c.Date_of_Service__c in all adjudication contexts. Regression test must cover: DOS before fee schedule change, DOS after fee schedule change, DOS on exact boundary date (first day of new schedule).

## Solution

Locate the fee schedule lookup logic in Apex class 'FeeScheduleLookupService.getApplicableFeeSchedule()'. Replace the current date anchor variable (currently populated from Claim__c.Submission_Date__c) with Claim__c.Date_of_Service__c. SOQL filter should read: WHERE Fee_Schedule__c.Provider__c = :providerId AND Fee_Schedule__c.CPT_Code__c = :cptCode AND Fee_Schedule__c.Effective_Date__c <= :dateOfService AND Fee_Schedule__c.Expiration_Date__c >= :dateOfService AND Fee_Schedule__c.Status__c = 'Active'. Write Apex unit tests covering boundary conditions. After fix, run a data remediation script in UAT to re-adjudicate the ~340 impacted Claim_Line__c records and validate corrected allowed amounts before promoting to production.
