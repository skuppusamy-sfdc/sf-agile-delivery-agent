# INS-1535: Mass Reassign Provider Network Agreements When Facility Terminates from Network

## Description

As a Claims Operations Manager, I need a guided mass-reassignment tool that, when a Facility__c (Business Account) is terminated from the network, automatically identifies all affiliated Provider__c records with active PNAs tied to that facility and presents a bulk reassignment or termination workflow so that claims are not routed to out-of-network providers post-termination. Extends INS-1503.

## Acceptance Criteria

Given a Network_Termination_Case__c (NTC) record is created with NTC.Termination_Type__c = 'Facility' and NTC.Status__c = 'Approved',
When the record is saved,
Then a Record-Triggered Flow must query all Provider_Network_Agreement__c records where PNA.Facility__c = NTC.Facility__c AND PNA.Status__c = 'Active' and create a child NTC_Affected_PNA__c junction record for each.

Given NTC_Affected_PNA__c records exist on the NTC,
When the Claims Operations Manager opens the NTC record,
Then an OmniScript 'FacilityTermination_MassReassign_OS' must be accessible via a 'Manage Affected Providers' quick action, listing all affected providers with checkboxes.

Given the user selects one or more providers in the OmniScript and chooses action 'Terminate PNA',
When the user confirms,
Then each selected PNA.Status__c must be updated to 'Terminated', PNA.Termination_Date__c set to NTC.Effective_Termination_Date__c, and a Provider_Change_Notification__c created with PCN.Change_Type__c = 'Network Termination'.

Given the user selects one or more providers and chooses action 'Reassign to Network',
When the user selects a target network and confirms,
Then new Provider_Network_Agreement__c records must be cloned from the originals with PNA.Network__c updated to the selected network and PNA.Status__c = 'Pending Approval'.

Given any in-flight Claim__c records with Claim__c.Provider__c matching a terminated provider and Claim__c.Status__c NOT IN ('Paid','Denied','Closed'),
When PNA termination is confirmed,
Then those claims must be flagged with Claim__c.Requires_Provider_Review__c = TRUE and a Task created for the Claims Operations Manager queue.

Given the mass reassignment OmniScript completes,
When all actions are processed,
Then a compliance report record (Compliance_Report__c) must be generated with a PDF attachment (via Conga CLM) summarizing all affected PNAs, actions taken, and the acting user, for audit purposes.

Given more than 500 affiliated providers are affected,
When the OmniScript is launched,
Then processing must be delegated to an Apex Queueable chain rather than synchronous DML to avoid governor limit violations.

## Solution

Record-Triggered Flow on Network_Termination_Case__c fires on Status__c change to 'Approved' where Termination_Type__c = 'Facility'. Flow calls Apex Invocable Method 'IdentifyAffectedPNAs' which creates NTC_Affected_PNA__c junction records. OmniScript 'FacilityTermination_MassReassign_OS' uses DataRaptor Load for bulk DML on PNA records. For >500 records, OmniScript calls an Apex REST endpoint that enqueues an Apex Queueable chain (FacilityTermination_QueueableChain). Conga CLM integration triggered via Apex HTTP callout on OmniScript completion step. Sharing Rules updated to ensure Claims Operations Manager profile has access to all affected Claim__c records via Criteria-Based Sharing on Claim__c.Requires_Provider_Review__c = TRUE.
