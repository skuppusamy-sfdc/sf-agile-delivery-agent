## Description

As a Provider Network Specialist, I need a bulk update tool to correct or refresh NPI and TIN fields on Provider__c (Person Account) records so that downstream claim adjudication and ERA matching remain accurate without requiring manual record-by-record edits.

Background: Following the Q3 NPPES reconciliation cycle, approximately 1,200 provider records have stale or mismatched NPI values. The current workaround of using native Salesforce list views for inline editing is unreliable at this volume and does not enforce validation rules or trigger the required Provider_Change_Notification__c (PCN) workflow.

## Acceptance Criteria

Given a Provider Network Specialist uploads a CSV with columns [Provider_Id__c, New_NPI__c, New_TIN__c, Change_Reason__c] via the Bulk Provider Update OmniScript,
When the file is submitted,
Then the system validates each NPI against the NPPES integration endpoint and each TIN against the CMS format regex (\d{2}-\d{7}) before committing any record.

Given a row in the CSV fails NPI validation,
When the batch job processes that row,
Then Provider__c.NPI__c is NOT updated, the row is flagged with Error_Code__c = 'NPPES_MISMATCH', and a summary error report is surfaced in the OmniScript completion screen.

Given all rows in a batch pass validation,
When Provider__c.NPI__c or Provider__c.TIN__c is updated,
Then a Provider_Change_Notification__c record is auto-created with PCN.Change_Type__c = 'NPI/TIN Update', PCN.Status__c = 'Pending Review', and PCN.Source__c = 'Bulk Upload'.

Given a PCN record is created from bulk update,
When PCN.Status__c transitions to 'Approved',
Then all active Provider_Network_Agreement__c records linked to that provider are stamped with PNA.Last_Data_Refresh__c = TODAY() and PNA.Refresh_Trigger__c = 'PCN Bulk'.

Given the bulk job completes (success or partial failure),
When the Provider Network Specialist views the job history tab,
Then they see a record in Bulk_Update_Log__c with fields: Total_Rows__c, Success_Count__c, Failure_Count__c, Processed_By__c, and a downloadable error CSV attachment.

-Given the tool is accessible to all internal users, When any user navigates to the bulk update page, Then the page loads without restriction.- (Superseded: access now restricted via Permission Set Group 'Provider_Data_Ops')

Given a user not assigned the 'Provider_Data_Ops' Permission Set Group,
When they attempt to navigate to the Bulk Provider Update page,
Then they receive a 'Insufficient Privileges' message and are redirected to the Provider Network home page.

## Solution

1. Build an OmniScript (Type: ProviderBulkUpdate, Sub-Type: NPI_TIN) with a File Upload step feeding a DataRaptor Extract to parse CSV rows into a collection.
2. Invoke an Apex-invocable method 'NPPESValidationService.validateBatch()' per row using a Step loop; results stored in OmniScript state.
3. On submission, trigger an Apex Queueable batch class 'ProviderBulkUpdateQueueable' that performs DML on Provider__c (Person Account) records in chunks of 200.
4. A Record-Triggered Flow on Provider__c (after update, NPI__c or TIN__c changed) creates Provider_Change_Notification__c records.
5. A separate Record-Triggered Flow on Provider_Change_Notification__c (after update, Status__c = 'Approved') bulk-updates related PNA records via a Loop + Update Records element.
6. Bulk_Update_Log__c records written by the Queueable's finish() method. Error CSV generated via ContentVersion and linked via ContentDocumentLink.
7. Permission Set Group 'Provider_Data_Ops' gates access to the OmniScript LWC wrapper component via a Custom Permission check in the component's connectedCallback.
