## Description

As a Provider Network Specialist, I need a nightly scheduled Apex batch job that queries the NPPES NPI Registry API for all Provider__c records with an active NPI__c value and updates key demographic fields (name, address, taxonomy codes, license status) in Salesforce so that provider records remain current without requiring manual data entry and support accurate claims adjudication and credentialing workflows.

## Acceptance Criteria

Given the nightly NPPES sync batch job (NPPESSyncBatch) is scheduled to run at 02:00 AM system time,
When the batch executes,
Then it queries all Provider__c records where Provider__c.NPI__c != null AND Provider__c.Network_Status__c != 'Terminated' AND Provider__c.NPPES_Sync_Enabled__c = true.

Given a Provider__c record is included in the batch scope,
When the NPPES NPI Registry REST API returns a valid 200 response for the provider's NPI,
Then the following fields are updated on Provider__c: Provider__c.NPPES_Last_Name__c, Provider__c.NPPES_First_Name__c, Provider__c.NPPES_Primary_Taxonomy__c, Provider__c.NPPES_License_State__c, Provider__c.NPPES_Address_Line1__c, Provider__c.NPPES_City__c, Provider__c.NPPES_State__c, Provider__c.NPPES_Zip__c, Provider__c.Last_NPPES_Sync_Date__c = today.

Given the NPPES API returns a 200 response but the returned provider name differs from Provider__c.LastName or Provider__c.FirstName by more than a fuzzy match threshold of 80%,
When the batch processes that record,
Then the NPPES values are written to the NPPES_* staging fields only (not to the core Person Account name fields), and Provider__c.NPPES_Name_Mismatch_Flag__c = true, triggering a Task assigned to the Provider Relations Specialist queue for manual review.

Given the NPPES API returns a 404 or the NPI is deactivated (enumeration_type = 'NPI-1' with deactivation_date populated),
When the batch processes that record,
Then Provider__c.NPI_Validation_Status__c = 'Deactivated — Requires Review' and a Provider_Change_Notification__c is auto-created with PCN__c.Notification_Type__c = 'NPI Deactivation Alert'.

Given the NPPES API is unavailable (HTTP 500 or timeout),
When the batch encounters a callout failure,
Then the affected Provider__c records are logged to NPPES_Sync_Error__c object with Error_Type__c = 'API Unavailable' and the batch continues processing remaining records without failing the entire job.

Given the batch job completes each nightly run,
When processing is finished,
Then a Platform Event NPPES_Sync_Complete__e is published with fields: Records_Processed__c, Records_Updated__c, Records_Failed__c, Records_Flagged_For_Review__c, and a summary email is sent to the Provider_Data_Ops distribution list via an Email Alert triggered by the Platform Event subscriber Flow.

## Solution

Implement NPPESSyncBatch as Database.Batchable + Database.AllowsCallouts Apex class with batch size of 10 (to respect NPPES API rate limits and Salesforce per-transaction callout limits). Use Named Credential 'NPPES_NPI_Registry' for secure endpoint management. Implement exponential backoff retry logic for transient failures within a wrapper HttpCalloutService. Schedule via System.scheduleBatch in a Schedulable class (NPPESSyncScheduler) configured for 02:00 AM daily. Publish NPPES_Sync_Complete__e Platform Event from a post-batch finish() method. Subscriber Flow triggers email alert via Email Alert action. Custom Metadata Type NPPES_Sync_Config__mdt stores fuzzy match threshold, enabled flag defaults, and API timeout values for configurability without code changes.
