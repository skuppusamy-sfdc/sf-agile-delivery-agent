## Description

As a Claims Operations Manager, I need the system to automatically place a processing hold on all open Claim_Line__c records associated with a provider when a Network_Termination_Case__c is initiated for that provider so that claims are not incorrectly adjudicated at in-network rates after the termination effective date, thereby reducing erroneous payment exposure.

A 'hold' means Claim_Line__c.Processing_Status__c = 'On Hold — Provider Termination' and no payment runs should include these lines until the hold is resolved or the termination is reversed.

## Acceptance Criteria

Given a Network_Termination_Case__c record is created with NTC.Status__c = 'Initiated' and NTC.Provider__c populated,
When the record is saved,
Then a Record-Triggered Flow queries all Claim__c records where Claim__c.Billing_Provider__c = NTC.Provider__c AND Claim__c.Status__c IN ('Submitted', 'In Adjudication', 'Pending Information').

Given the query returns one or more matching Claim__c records,
When the Flow processes each claim,
Then all child Claim_Line__c records for those claims where Claim_Line__c.Status__c != 'Paid' and Claim_Line__c.Status__c != 'Denied' are updated: Claim_Line__c.Processing_Status__c = 'On Hold — Provider Termination', Claim_Line__c.Hold_Reason__c = 'Network_Termination_Case__c: ' + NTC.Id, Claim_Line__c.Hold_Date__c = TODAY().

Given Claim_Line__c records are placed on hold,
When the nightly Payment Engine batch job 'PaymentRunBatchable' evaluates claim lines for payment,
Then any Claim_Line__c where Processing_Status__c = 'On Hold — Provider Termination' is excluded from the payment run and logged in Payment_Exclusion_Log__c with Exclusion_Reason__c = 'Provider Termination Hold'.

Given a Network_Termination_Case__c.Status__c is updated to 'Withdrawn' or 'Rejected',
When the record is saved,
Then all Claim_Line__c records previously held under that NTC.Id (matched via Hold_Reason__c) have their Processing_Status__c restored to their prior value stored in Claim_Line__c.Pre_Hold_Status__c, and Hold_Date__c is cleared.

Given the total number of affected Claim_Line__c records exceeds 500,
When the Record-Triggered Flow would exceed governor limits,
Then the Flow invokes an Apex-invocable method 'ClaimLineHoldService.applyHoldAsync()' to process via a Queueable job, and a notification is sent to the Claims Operations Manager queue via Custom Notification.

Given the hold is applied,
When a Claims Operations Manager views the Network_Termination_Case__c record,
Then a related list 'Affected Claim Lines' displays all Claim_Line__c records in hold status, with a count badge showing the total number of impacted lines.

## Solution

1. Record-Triggered Flow on Network_Termination_Case__c (after insert, Status__c = 'Initiated'): Get Records on Claim__c (filter by Billing_Provider__c and Status__c), loop through child Claim_Line__c records, update Processing_Status__c and store pre-hold value in Pre_Hold_Status__c.
2. Decision element in Flow: if collection size > 500, invoke Apex action 'ClaimLineHoldService.applyHoldAsync()' (Queueable); otherwise update records inline via Update Records elements.
3. ClaimLineHoldService Apex class handles both synchronous and async paths; async path chains Queueable jobs for collections >2000 lines.
4. PaymentRunBatchable modified to add SOQL filter: AND Processing_Status__c != 'On Hold — Provider Termination' in Claim_Line__c query scope.
5. Second Record-Triggered Flow on Network_Termination_Case__c (after update, Status__c changed to 'Withdrawn' OR 'Rejected'): Get Records on Claim_Line__c filtered by Hold_Reason__c LIKE 'Network_Termination_Case__c: ' + recordId, restore Pre_Hold_Status__c values.
6. Custom Notification Type 'ClaimHoldAlert' configured for Claims Operations Manager profile; fired via Send Custom Notification Flow action when async path is taken.
7. 'Affected Claim Lines' related list added to NTC Lightning Record Page via App Builder using standard related list component filtered by Hold_Reason__c.
