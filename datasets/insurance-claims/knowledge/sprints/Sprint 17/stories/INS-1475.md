## Description

**Bug Report**

**Summary:** The Bulk Provider Update Tool (delivered in INS-1469) fails silently — with no error log entry — when the import CSV contains special characters (e.g., accented characters: é, ñ, ü, or ampersands: &) in the Provider__c.LastName or Provider__c.FirstName fields. The records are skipped without any Bulk_Import_Error_Log__c entry being created, leaving the Provider Network Specialist unaware of which records failed.

**Steps to Reproduce:**
1. Prepare a CSV file with at least one row where the Provider Last Name contains a special character, e.g., 'García' or 'O'Brien'.
2. Navigate to the Bulk Provider Update Tool LWC on the Provider Network Admin App.
3. Upload the CSV and submit the import job.
4. Wait for the async Queueable job to complete and review the email summary.
5. Observe: The email summary shows the record as 'Successfully Updated' OR the count totals do not add up correctly.
6. Navigate to Bulk_Import_Error_Log__c and confirm no error log was created for the affected row.
7. Navigate to the Provider__c record and confirm NPI__c was NOT updated (record silently skipped).

**Expected Behavior:** Rows with special characters in name fields should either: (a) process successfully if the character is valid UTF-8, or (b) create a Bulk_Import_Error_Log__c entry with Error_Reason__c = 'Invalid character in name field' if the character causes a DML exception.

**Actual Behavior:** The record is silently skipped. No error log is created. The email summary count is inaccurate.

**Environment:** QA-FULL Sandbox and Production (confirmed by Provider Network Specialist after go-live of INS-1469)
**Root Cause Hypothesis:** CSV parsing in the LWC client-side JavaScript is not handling UTF-8 BOM or encoded characters correctly before passing data to the Apex controller. The Queueable Apex catch block may also be swallowing the exception without logging.
**Severity:** Medium — data quality risk; silent failures undermine trust in bulk tool.

## Acceptance Criteria

As per Description.

Given the fix is deployed,
When a CSV row contains UTF-8 special characters (é, ñ, ü, &, apostrophe) in name fields,
Then the record must be processed correctly if the resulting Provider__c field value does not exceed 255 characters and contains no SOQL injection risk characters.

Given a CSV row causes any exception in the Queueable Apex execute method,
When the exception is caught,
Then a Bulk_Import_Error_Log__c record must always be created with Error_Reason__c populated, regardless of exception type.

Given the job completes,
When the email summary is generated,
Then Successfully_Updated_Count + Failed_Count must always equal Total_Submitted_Count with no silent skips.

## Solution

Fix 1 — LWC CSV Parser: Update the JavaScript CSV parsing utility in BulkProviderUpdateTool.js to explicitly handle UTF-8 BOM stripping (check for \uFEFF at start of file) and use decodeURIComponent / proper TextDecoder API with 'utf-8' encoding. Sanitize apostrophes and ampersands before passing to Apex. Fix 2 — Apex Queueable: Audit BulkProviderUpdateQueueable.cls catch blocks to ensure ALL exception types (DmlException, StringException, System.Exception) result in a Bulk_Import_Error_Log__c insert. Wrap error log insert in a separate try-catch to prevent log creation itself from failing silently. Add counter reconciliation assertion before sending completion email. Deploy to QA-FULL and run regression test suite including UTF-8 test CSV fixture.
