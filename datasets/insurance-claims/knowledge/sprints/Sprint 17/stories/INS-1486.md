## Description

**Bug Report**

**Summary:** When a Provider_Change_Notification__c (PCN) record is manually created from the Provider__c record page using the 'New PCN' quick action, the PCN.Provider_Network_Agreement__c lookup field is not auto-populated even when the provider has exactly one active PNA record. Users must manually search for and attach the PNA, causing data entry errors and orphaned PCN records that break downstream compliance reporting.

**Steps to Reproduce:**
1. Navigate to any Provider__c (Person Account) record that has exactly one Provider_Network_Agreement__c with PNA.Status__c = 'Active'.
2. Click the 'New PCN' quick action button on the Provider 360 Lightning Record Page.
3. The PCN quick action form opens (OmniScript: ProviderChangePCN).
4. Observe the 'Network Agreement' lookup field — it is blank.
5. Save the PCN record without manually populating the lookup.
6. The saved PCN record has PCN.Provider_Network_Agreement__c = NULL.

**Expected Behavior:** When a provider has exactly one active PNA, the PCN.Provider_Network_Agreement__c field should be auto-populated with that PNA's Id when the quick action form opens.

**Actual Behavior:** PCN.Provider_Network_Agreement__c remains NULL regardless of whether an active PNA exists.

**Environment:** Production and Full Sandbox. Regression introduced in Sprint 16 deployment (2024-11-14).

**Impact:** ~340 PCN records created since Sprint 16 go-live have NULL PNA lookups. Compliance reporting query on ComplianceReporting component returns incorrect counts for 'PCNs Without Agreement' metric.

**Workaround:** Provider Relations Specialists manually search and attach PNA on each PCN — estimated 3 minutes overhead per record.

## Acceptance Criteria

As per Description.

Additional verification:
Given the fix is deployed,
When a new PCN is created via the 'New PCN' quick action for a provider with exactly one active PNA,
Then PCN.Provider_Network_Agreement__c is auto-populated with the active PNA Id upon form load.

Given a provider has zero active PNA records,
When the PCN quick action form opens,
Then the lookup field remains blank and no error is thrown.

Given a provider has more than one active PNA record,
When the PCN quick action form opens,
Then the lookup field remains blank and a helper text message 'Multiple active agreements found — please select manually' is displayed below the field.

## Solution

1. Root cause: The OmniScript 'ProviderChangePCN' Step 1 DataRaptor Extract 'DR_GetActivePNAForProvider' has an incorrect filter — it queries PNA.Status__c = 'Active' but the merge field binding to pass the Provider Id is referencing '%recordId%' which resolves to NULL when launched from a quick action context (should be '%ParentId%' or passed via OmniScript input JSON).
2. Fix: Update the OmniScript Launch URL on the quick action to pass {!Account.Id} as the 'providerId' input parameter. Update DR_GetActivePNAForProvider to use the 'providerId' input key.
3. Add OmniScript conditional logic: if DR result returns exactly 1 record, auto-set the PNA lookup field value; if 0 or >1, display appropriate helper text.
4. Data remediation: Run a one-time Apex script to back-fill PCN.Provider_Network_Agreement__c for the ~340 impacted records where a single active PNA can be unambiguously matched by PCN.Provider__c.
5. Add unit test coverage to OmniScript integration test suite and Apex test class 'PCNAutoLinkTest' covering all three PNA count scenarios.
