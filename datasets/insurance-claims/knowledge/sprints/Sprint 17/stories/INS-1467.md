## Description

**Bug Report — Duplicate PCN records generated during concurrent Provider__c API updates**

**Environment:** Full Sandbox (Sprint 17)

**Related Story:** Introduced after INS-1463 deployment to sandbox.

**Steps to Reproduce:**
1. Using a data integration tool (e.g., MuleSoft or Postman), fire two simultaneous PATCH API requests to update the same Provider__c record's Provider_Specialty__c field within the same second.
2. Both requests carry the same field value change (e.g., Provider_Specialty__c: 'Internal Medicine' → 'Cardiology').
3. Observe Provider_Change_Notification__c records created for that Provider__c.Id.

**Expected Result:** Exactly one Provider_Change_Notification__c record is created for the specialty change event, with PCN.Change_Type__c = 'Specialty Change'.

**Actual Result:** Two (or more) duplicate PCN records are created with identical PCN.Change_Type__c, PCN.Provider__c, PCN.Old_Value__c, and PCN.New_Value__c values. The duplicate check logic in the Record-Triggered Flow (INS-1463) queries for existing open PCNs but both concurrent transactions read the database before either has committed, bypassing the duplicate guard.

**Impact:** Medium-High — duplicate PCNs inflate the Provider Relations queue, cause double-handling by specialists, and can result in duplicate CMS directory sync events.

**Frequency:** Reproducible under concurrent load; observed in integration test runs with MuleSoft batch sync jobs.

## Acceptance Criteria

As per Description.

Post-fix validation: Run a concurrent load test (10 simultaneous PATCH requests on the same Provider__c record) and confirm exactly 1 PCN record is created per unique change type per Provider__c per calendar day.

## Solution

Replace the flow-based duplicate check with an Apex Trigger-based upsert using an External ID / unique index approach. Add a Unique Constraint (External ID field: PCN_Dedup_Key__c, Text 255, Unique, External ID) to Provider_Change_Notification__c, populated as a concatenation of Provider__c.Id + Change_Type__c + Date.today().format(). The Apex trigger (ProviderChangePCNTrigger, After Update on Provider__c) uses Database.upsert with the PCN_Dedup_Key__c external ID field, which leverages Salesforce's built-in upsert idempotency to prevent duplicates under concurrent transactions. Remove the duplicate-check 'Get Records' element from the INS-1463 flow and delegate PCN creation entirely to the Apex layer. Add a trigger handler test class asserting idempotency under simulated concurrent execution.
