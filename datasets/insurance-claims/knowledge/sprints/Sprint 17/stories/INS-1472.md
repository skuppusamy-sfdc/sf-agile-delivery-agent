## Description

**Bug Report**

**Summary:** Provider__c.Credentialing_Status__c field is not being updated to 'Credentialed' when the associated Credentialing_Case__c record transitions to Case_Stage__c = 'Approved'. This causes Provider 360 views and downstream claim adjudication eligibility checks to incorrectly flag credentialed providers as 'Pending Credentialing', resulting in false claim holds.

**Steps to Reproduce:**
1. Navigate to an existing Credentialing_Case__c record where Case_Stage__c = 'In Review' and Provider__c lookup is populated.
2. Update Case_Stage__c from 'In Review' to 'Approved' and save the record.
3. Navigate to the linked Provider__c (Person Account) record.
4. Observe Provider__c.Credentialing_Status__c — it remains 'Pending Credentialing' instead of updating to 'Credentialed'.
5. Reproduce consistently across sandbox org (QA-FULL) on both UI save and API update.

**Expected Behavior:** Provider__c.Credentialing_Status__c should be set to 'Credentialed' and Provider__c.Credentialing_Approval_Date__c should be stamped with TODAY() when Credentialing_Case__c.Case_Stage__c transitions to 'Approved'.

**Actual Behavior:** Provider__c.Credentialing_Status__c remains unchanged at 'Pending Credentialing'. Provider__c.Credentialing_Approval_Date__c is null.

**Environment:** QA-FULL Sandbox
**Regression:** Introduced in Sprint 16 deployment (INS-1441 Record-Triggered Flow modification)
**Severity:** High — blocking Claims adjudication for 3 newly credentialed providers in UAT.

## Acceptance Criteria

As per Description.

Given the fix is deployed,
When Credentialing_Case__c.Case_Stage__c is updated to 'Approved',
Then Provider__c.Credentialing_Status__c must equal 'Credentialed' and Provider__c.Credentialing_Approval_Date__c must equal the date of the stage change.

Given the fix is deployed,
When Credentialing_Case__c.Case_Stage__c is updated to 'Denied',
Then Provider__c.Credentialing_Status__c must equal 'Credentialing Denied' and no approval date should be stamped.

Given the fix is deployed,
When a Credentialing_Case__c is updated via API (e.g., CAQH integration),
Then the Provider__c update must fire correctly (not only on UI save).

## Solution

Root cause identified: The Record-Triggered Flow 'Update_Provider_Credentialing_Status_Flow' introduced in INS-1441 has an entry condition filter incorrectly set to ONLY fire when Case_Stage__c changes FROM 'Initial Review' (not 'In Review'), missing the correct prior stage value after a stage picklist rename in Sprint 16. Fix: Update the flow entry condition to use 'is changed' = true on Case_Stage__c field (remove prior-value filter) and add a Decision element to branch on current Case_Stage__c value. Update the 'Update Provider Record' element to set Credentialing_Status__c and Credentialing_Approval_Date__c correctly per branch. Deploy fix to QA-FULL for regression testing. Add unit test coverage in CredentialingCaseTriggerTest.cls for API-triggered scenario.
