## Description

As a Provider Network Specialist, I need a bulk update interface for Provider__c (Person Account) NPI and TIN fields so that I can correct mass data discrepancies identified during the quarterly NPPES reconciliation cycle without opening individual records. This is particularly critical ahead of the Q3 fee schedule refresh where incorrect NPIs will cause Claim_Line__c adjudication failures.

## Acceptance Criteria

Given a Provider Network Specialist with the 'Provider_Data_Admin' Permission Set Group navigates to the Provider Data Maintenance tab,
When they upload a CSV containing Provider__c records with updated NPI__c and TIN__c values (max 500 rows per batch),
Then the system validates each NPI against the NPPES integration endpoint before committing the update.

Given the NPPES validation returns a mismatch for one or more rows,
When the bulk job completes,
Then the invalid rows are surfaced in a 'Validation Exceptions' sub-list with Provider__c.Name, Provider__c.NPI__c (current), submitted NPI, and failure reason; valid rows are committed.

Given a successful bulk update is committed,
When Provider__c.NPI__c or Provider__c.TIN__c is changed,
Then a Provider_Change_Notification__c (PCN) record is auto-created with PCN__c.Change_Type__c = 'NPI/TIN Update', PCN__c.Status__c = 'Pending Review', and PCN__c.Effective_Date__c = today.

Given the bulk job finishes (success or partial),
When the job status is resolved,
Then an in-app notification and email are sent to the initiating user with a summary: total submitted, total updated, total failed.

-Given a TPA-linked provider, the TIN update should auto-notify the TPA via outbound message- (superseded by PCN workflow above)

Given a Provider__c record has an active Provider_Network_Agreement__c (PNA) with PNA__c.Status__c = 'Active',
When its TIN__c is changed via bulk update,
Then PNA__c.TIN_Verification_Status__c is reset to 'Pending Re-verification' and a task is created for the assigned Provider Relations Specialist.

## Solution

Implement an OmniScript ('BulkProviderNPITINUpdate') with a CSV upload step using a DataRaptor Extract to parse and stage rows into a custom Bulk_Update_Job__c object. A queueable Apex class (ProviderNPITINBulkUpdateQueueable) processes batches of 50, calling a named credential-backed NPPES REST callout for NPI validation. A Record-Triggered Flow on Provider__c fires post-save to create PCN records and reset PNA TIN verification status. Platform Events (Provider_Data_Change__e) broadcast job completion to trigger in-app notifications via a subscriber Flow.
