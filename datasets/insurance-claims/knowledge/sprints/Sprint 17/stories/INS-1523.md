## Description

**Bug Report — GSA Effective Date Validation Bypassed on API Insert**

**Environment:** Acme Insurance Full Sandbox

**Steps to Reproduce:**
1. Using Postman or a data loader tool, perform a bulk API INSERT of Group_Service_Agreement__c records via the Salesforce REST API.
2. Include records where GSA.Effective_Date__c is set to a date in the past (e.g., more than 365 days prior to TODAY()).
3. Also include records where GSA.Effective_Date__c > GSA.Expiration_Date__c.
4. Submit the API call.
5. Observe the inserted records in Salesforce.

**Expected Behavior:**
The Apex Trigger ('GSAValidationTrigger') should fire on before insert for API-originated DML and reject records where:
- GSA.Effective_Date__c is more than 365 days in the past (unless GSA.Backdating_Approved__c = TRUE)
- GSA.Effective_Date__c > GSA.Expiration_Date__c

**Actual Behavior:**
Both invalid record types are inserted successfully without error. The validation logic in GSAValidationTrigger is wrapped in an `if (!System.isBatch())` guard condition that inadvertently also excludes standard API calls from the TPA integration middleware, allowing invalid GSA records to persist.

**Impact:** Medium-High — Invalid GSA records with backdated or inverted date ranges can result in incorrect Group/Employer billing calculations and erroneous GSA.Status__c = 'Active' assignments.

**Root Cause:** Incorrect use of `System.isBatch()` guard in GSAValidationTrigger.cls — should use a more targeted bypass mechanism (e.g., a TriggerBypass__c Custom Setting) rather than suppressing all non-batch validations.

## Acceptance Criteria

As per Description.

Post-fix verification: Re-run the API bulk insert test with invalid records and confirm all records with backdated Effective_Date__c (>365 days, Backdating_Approved__c = FALSE) and inverted date ranges are rejected with a descriptive addError() message. Confirm legitimate TPA integration inserts with valid dates succeed without error.

## Solution

Remove the `if (!System.isBatch())` guard from GSAValidationTrigger.cls. Implement a targeted bypass mechanism using a Trigger_Bypass__c Hierarchy Custom Setting with a field 'Bypass_GSA_Validation__c' (Boolean). The TPA integration connected app's running user profile will have this set to TRUE only for legitimate migration scenarios, approved by Compliance Officer. All other contexts (API, UI, batch) will execute the full validation. Add a new validation branch for Effective_Date__c > Expiration_Date__c using addError() on the GSA.Expiration_Date__c field. Update unit tests (GSAValidationTriggerTest) to cover API context simulation using Test.isRunningTest() patterns and assert error messages for all invalid scenarios. Minimum 85% code coverage required.
