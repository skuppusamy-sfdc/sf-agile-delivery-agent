## Description

As a Provider Network Specialist, I need a mass expiration tool that allows me to bulk-expire Fee_Schedule__c records filtered by Network_Name__c and LOB__c so that I can efficiently manage annual fee schedule rollover cycles without manually updating hundreds of individual records, reducing the risk of stale CPT code pricing being applied to new Claim_Line__c adjudications.

## Acceptance Criteria

Given a Provider Network Specialist accesses the 'Fee Schedule Mass Expiration' custom Lightning App Page,
When they select filter criteria (Network_Name__c, LOB__c, and optional CPT_Code_Range__c),
Then the tool returns all Fee_Schedule__c records where Fee_Schedule__c.Status__c = 'Active' AND Fee_Schedule__c.Effective_Date__c < TODAY() matching the selected filters.

Given the filtered list is displayed,
When the user reviews and selects records for expiration,
Then they must enter a Fee_Schedule__c.Expiration_Date__c value (must be >= TODAY()) before the 'Expire Selected' button is enabled.

Given the user clicks 'Expire Selected',
When the batch operation executes,
Then each selected Fee_Schedule__c record is updated: Status__c = 'Expired', Expiration_Date__c = entered date, Last_Modified_Reason__c = 'Mass Expiration — Network Rollover', and the running user is stamped on Fee_Schedule__c.Expired_By__c.

Given a Fee_Schedule__c record has active Claim_Line__c records referencing it (Claim_Line__c.Fee_Schedule__c = Fee_Schedule__c.Id AND Claim_Line__c.Claim__r.Status__c NOT IN ('Closed','Paid')),
When the mass expiration attempts to expire that record,
Then the record is excluded from the batch, flagged in an error log with reason 'Active Claim Lines Exist', and the user is shown a warning count before final confirmation.

Given the batch completes,
When the Compliance Officer reviews the audit log,
Then a Fee_Schedule_Audit__c record is created per expired schedule capturing: Audit.Record_Id__c, Audit.Action__c = 'Mass Expiration', Audit.Batch_Run_Id__c, Audit.Expired_By__c, Audit.Expiration_Date__c.

Given the mass expiration job processes more than 500 records,
When the operation is triggered,
Then it is routed to an Apex Batchable class (FeeScheduleMassExpirationBatch) with a batch size of 50 to respect DML governor limits and the user receives an email notification upon job completion.

## Solution

Create a custom Lightning Web Component ('feeScheduleMassExpiration') embedded in a dedicated Lightning App Page, accessible via 'Fee_Schedule_Admin' Permission Set Group. The LWC calls an Apex controller (FeeScheduleMassExpirationController) that performs the initial SOQL query with dynamic WHERE clause construction based on user-selected filters. For batches >500 records, the controller invokes FeeScheduleMassExpirationBatch (Database.Batchable) with scope size 50. Implement a pre-expiration validation query to detect active Claim_Line__c references. Create Fee_Schedule_Audit__c custom object with appropriate fields. Use Custom Metadata (Fee_Schedule_Expiration_Config__mdt) to configure batch size and notification email addresses. Add a confirmation modal in the LWC using lightning-modal before final submission.
