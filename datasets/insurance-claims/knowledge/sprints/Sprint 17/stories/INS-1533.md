## Description

As a Provider Network Specialist, I need the system to ingest the weekly NPPES delta feed and bulk-update Provider__c (Person Account) NPI and TIN fields so that provider demographic data remains accurate for claims routing and adjudication without requiring manual record-by-record edits.

## Acceptance Criteria

Given the NPPES weekly delta file has been dropped in the designated S3 bucket,
When the scheduled DataRaptor Extract job runs at 02:00 EST every Sunday,
Then a batch Apex job (Provider_NPPES_BulkUpdate_Batch) must process all delta records and update Provider__c.NPI__c and Provider__c.TIN__c accordingly.

Given a Provider__c record exists with a matching NPI__c value in the feed,
When the batch processes that row,
Then Provider__c.Last_NPPES_Sync_Date__c must be stamped with the current timestamp and Provider__c.Data_Source__c must be set to 'NPPES'.

Given a delta record contains an NPI that does not match any existing Provider__c,
When the batch encounters that row,
Then a Provider_Change_Notification__c (PCN) record must be created with PCN.Change_Type__c = 'New NPI – No Match' and PCN.Status__c = 'Pending Review' and assigned to the Provider Relations queue.

Given a Provider__c record has an active Provider_Network_Agreement__c (PNA) with PNA.Status__c = 'Active',
When the NPI or TIN is changed by the batch,
Then a Platform Event (Provider_Data_Changed__e) must be published with the old and new NPI/TIN values so downstream Fee Schedule and claims routing listeners can react.

Given the batch job completes (success or partial failure),
When the run finishes,
Then a summary email must be sent to the Provider Network Specialist distribution list including: total records processed, total updated, total errors, and a CSV attachment of failed rows.

-Given a TIN change is detected, automatically trigger a re-credentialing workflow on the associated Credentialing_Case__c.- (Superseded by INS-1521 — re-credentialing trigger handled in CredentialingService component)

Given the batch processes more than 10,000 records in a single run,
When Salesforce governor limits are approached,
Then the batch must use a chunk size of 200 and implement exponential back-off retry logic for any DML failures.

## Solution

Implement a scheduled Apex Batch class (Provider_NPPES_BulkUpdate_Batch) invoked by a Scheduled Flow at 02:00 EST Sundays. Use a DataRaptor Extract to pull the NPPES delta CSV from an External Data Source (AWS S3 via Named Credential). Map fields using a custom metadata type (NPPES_Field_Mapping__mdt) for maintainability. On NPI/TIN change, publish Provider_Data_Changed__e Platform Event consumed by a Record-Triggered Flow on Fee_Schedule__c and a separate Apex trigger on Claim__c for in-flight claims re-routing. PCN records created via Apex. Error rows written to a custom object Bulk_Job_Error_Log__c. Permission Set Group 'Provider_Data_Admin_PSG' required for manual re-run button on Provider Network Dashboard.
