## Description

As a Provider Network Specialist, I need the system to automatically generate a Provider_Change_Notification__c (PCN) record whenever a Provider__c or Facility__c record undergoes a change to key demographic fields (Specialty, Primary Address, Phone, or Board Certification Status) so that our network directory remains compliant with CMS directory accuracy requirements and internal SLA obligations under active PNA contracts. Extends INS-1431.

## Acceptance Criteria

Given a Provider__c record's Provider_Specialty__c field is updated,
When the record is saved,
Then a Provider_Change_Notification__c is automatically created with: PCN.Change_Type__c = 'Specialty Change', PCN.Provider__c = Provider__c.Id, PCN.Old_Value__c = prior specialty, PCN.New_Value__c = updated specialty, PCN.Status__c = 'Open', PCN.Created_Date__c = today.

Given a Facility__c record's BillingStreet or BillingCity is updated,
When the record is saved,
Then a PCN is created with PCN.Change_Type__c = 'Address Change' and PCN.Facility__c = Facility__c.Id.

Given a PCN is created,
When PCN.Status__c = 'Open',
Then the system auto-assigns the PCN to the Provider Relations Specialist queue (Queue Name: 'Provider_Relations_Queue') via assignment rules.

Given the PCN is created for a Provider__c with an active Credentialing_Case__c (where Credentialing_Case__c.Status__c != 'Closed'),
When the PCN record is saved,
Then the PCN is linked to the open Credentialing_Case__c via PCN.Credentialing_Case__c lookup and a chatter post is added to the Credentialing_Case__c: 'Provider demographic change detected — PCN [PCN Number] created. Please review.'.

Given a bulk data import updates more than 10 Provider__c records in a single transaction,
When PCNs are generated,
Then PCN creation is batched and does not cause governor limit violations (verified by test class asserting < 100 SOQL queries per batch).

## Solution

Implement a Record-Triggered Flow (Before Save for old value capture, After Save for PCN creation) on Provider__c and Facility__c objects. The flow uses a 'Get Records' element to check for existing open PCNs of the same change type to prevent duplicates. PCN record creation is handled via a Subflow to allow reuse across both object triggers. For the Credentialing_Case__c linkage, an Apex invocable method is called from the flow to handle the chatter post (ConnectAPI) and avoid flow DML limits. A Platform Event (ProviderChangeDetected__e) is also published for potential downstream CMS directory sync. Bulk safety is ensured by routing high-volume transactions through an After Save flow with 'Run Asynchronously' option enabled.
