## Description

**Bug Report – INS-1482**

**Summary:** During Claim_Line__c adjudication, when a rendering provider has more than one active Provider_Network_Agreement__c for the same LOB and the same CPT code appears in multiple associated Fee_Schedule__c records, the fee schedule resolution logic returns the first Fee_Schedule__c record ordered by CreatedDate ASC instead of the most recently effective record (Fee_Schedule__c.Effective_Date__c DESC). This results in providers being reimbursed at outdated (lower) contracted rates, causing provider payment disputes.

**Environment:** Production, QA Sandbox SF17-QA

**Steps to Reproduce:**
1. Identify a Provider__c with two active PNA records for LOB = 'Medical', both containing a Fee_Schedule__c entry for CPT code 99213.
   - PNA-A Fee Schedule: Allowed_Amount__c = $85.00, Effective_Date__c = 2023-01-01
   - PNA-B Fee Schedule: Allowed_Amount__c = $97.50, Effective_Date__c = 2024-06-01
2. Submit a Claim_Line__c with: Rendering_Provider__c = that provider, CPT_Code__c = '99213', Date_of_Service__c = 2025-01-15, LOB__c = 'Medical'.
3. Observe Claim_Line__c.Allowed_Amount__c after adjudication.

**Expected Behavior:** Claim_Line__c.Allowed_Amount__c = $97.50 (most recently effective Fee_Schedule__c for the given CPT and LOB).

**Actual Behavior:** Claim_Line__c.Allowed_Amount__c = $85.00 (oldest Fee_Schedule__c by CreatedDate).

**Root Cause:** SOQL query in Apex class 'FeeScheduleResolverService.cls' is missing ORDER BY Fee_Schedule__c.Effective_Date__c DESC LIMIT 1; it currently uses default CreatedDate ordering.

**Impact:** Financial — incorrect underpayment to providers; estimated $12,400 in disputed payments identified in last billing cycle.

## Acceptance Criteria

As per Description.

Additional regression tests:
- Single active PNA with single CPT entry: allowed amount unchanged.
- Provider with two PNAs where only one covers the billed CPT: correct PNA's amount used.
- Provider with a PNA where Fee_Schedule__c.Effective_Date__c > Date_of_Service__c: that fee schedule must NOT be selected (future-dated schedules excluded).
- Confirm Claim_Line__c.Fee_Schedule__c (lookup field) is stamped with the resolved Fee_Schedule__c.Id post-fix.

## Solution

1. Update SOQL in 'FeeScheduleResolverService.cls' method 'resolveAllowedAmount()': add ORDER BY Effective_Date__c DESC LIMIT 1 and add WHERE Effective_Date__c <= :dateOfService to the existing filter.
2. Add unit tests covering multi-PNA scenarios, future-dated exclusion, and single-PNA baseline.
3. Run data fix script in Production to identify and log all Claim_Line__c records adjudicated in the last 90 days where Fee_Schedule__c.Effective_Date__c was not the most recent — flag for Payment__c adjustment review by Claims Operations Manager.
4. Deploy via change set to SF17-QA for regression testing, then Production hotfix deployment.
5. Add a custom validation on Fee_Schedule__c to warn when a new record is created with an Effective_Date__c overlapping an existing active schedule for the same PNA and CPT code.
