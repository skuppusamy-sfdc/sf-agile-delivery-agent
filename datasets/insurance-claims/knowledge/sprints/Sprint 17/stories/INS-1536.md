## Description

**Bug Report – INS-1536**

**Summary:** The existing validation rule 'TIN_Format_Check' on Provider__c enforces a 9-digit numeric format (XX-XXXXXXX) when records are saved via the UI. However, when records are inserted or updated via the NPPES bulk import Apex batch (introduced in INS-1533), the validation rule is bypassed and malformed TIN values (e.g., 'N/A', '000-00-0000', blank strings) are written to Provider__c.TIN__c, causing downstream EOB generation failures and ERA matching errors.

**Steps to Reproduce:**
1. Prepare a CSV with at least one row where TIN = 'N/A' and a valid NPI.
2. Drop CSV into the designated S3 bucket used by the NPPES integration.
3. Allow the Provider_NPPES_BulkUpdate_Batch to execute (or trigger manually via Developer Console).
4. Query Provider__c records updated by the batch: SELECT Id, TIN__c, NPI__c FROM Account WHERE RecordType.Name = 'Provider' AND LastModifiedDate = TODAY.
5. Observe that records with TIN = 'N/A' or blank are persisted without error.
6. Attempt to generate an EOB for a Claim__c linked to one of those providers — EOB generation fails with error 'TIN_MISSING_OR_INVALID'.

**Expected:** Batch DML should respect the TIN_Format_Check validation rule and move invalid rows to Bulk_Job_Error_Log__c without updating the Provider__c record.

**Actual:** Validation rule is silently bypassed; invalid TIN values overwrite valid existing TIN data in Provider__c.TIN__c.

## Acceptance Criteria

As per Description.

Given the fix is deployed,
When the batch processes a row with an invalid TIN format,
Then the row must be written to Bulk_Job_Error_Log__c with Error_Reason__c = 'Invalid TIN Format' and the existing Provider__c.TIN__c value must remain unchanged.

Given the fix is deployed,
When the batch processes a row with a valid 9-digit TIN,
Then Provider__c.TIN__c must be updated as expected with no regression.

## Solution

Root cause: the Apex batch uses Database.update() with allOrNothing=false and does not pre-validate field formats before DML, and the validation rule has a bypass condition checking a custom permission 'Bypass_Validation_Rules' that was incorrectly assigned to the integration user profile. Fix involves: (1) Remove 'Bypass_Validation_Rules' custom permission from the NPPES Integration User profile. (2) Add an Apex utility method 'TINValidator.isValid(String tin)' using regex pattern '^\d{2}-\d{7}$' called in the batch's execute() method before constructing the update list. (3) Invalid rows are added to a separate error list and bulk-inserted into Bulk_Job_Error_Log__c. Unit tests to cover valid TIN, invalid TIN, and blank TIN scenarios with minimum 85% code coverage.
