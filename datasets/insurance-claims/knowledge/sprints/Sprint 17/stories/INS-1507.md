## Description

**Bug Report — Provider TIN Validation Bypass**

**Environment:** Acme Insurance Production Sandbox (Sprint 17 refresh)
**Reported By:** Amanda Foster (Provider Network Specialist)
**Date Observed:** 2024-11-04

**Steps to Reproduce:**
1. Navigate to any existing Provider__c (Person Account) record.
2. Click Edit on the TIN__c field.
3. Enter a value containing letters and special characters, e.g., '12-345AB78' or '##-1234567'.
4. Click Save.
5. Observe that the record saves successfully without any validation error.

**Expected Behavior:**
The existing Validation Rule 'VR_Provider_TIN_Format' on Provider__c should reject any TIN__c value that does not match the 9-digit numeric pattern (regex: ^[0-9]{9}$) and display the error message: 'TIN must be exactly 9 numeric digits with no dashes or special characters.'

**Actual Behavior:**
The record saves without triggering the validation rule. The TIN__c field stores the invalid value '12-345AB78' and the validation rule fires ONLY when the record is edited again subsequently — not on the initial offending save.

**Root Cause Hypothesis:**
The Validation Rule 'VR_Provider_TIN_Format' contains a condition `ISCHANGED(TIN__c)` that appears to be evaluating incorrectly when the field was previously NULL. When TIN__c transitions from NULL to a non-null value, ISCHANGED() may not be triggering the regex check as expected in the current formula.

**Impact:**
Invalid TIN data is flowing downstream into Fee_Schedule__c lookups and ERA (Electronic Remittance Advice) file generation, causing 835 transaction rejections from payer clearinghouse.

**Affected Records:** 47 Provider__c records identified with malformed TIN__c values in the Sprint 17 sandbox data refresh.

## Acceptance Criteria

As per Description. 

Additionally: After fix deployment, all 47 affected Provider__c records with malformed TIN__c values must be identified via a SOQL report and corrected or flagged with Data_Quality_Flag__c = 'TIN_Invalid' for Provider Network Specialist review.

## Solution

Review and correct Validation Rule 'VR_Provider_TIN_Format' on Provider__c. Replace ISCHANGED(TIN__c) conditional wrapper with a BLANKVALUE-safe formula: NOT(REGEX(TIN__c, '[0-9]{9}')) ensuring the rule fires on both insert and update when TIN__c is non-null. Additionally add a complementary LWC input mask on the Provider Edit form to enforce numeric-only input client-side. Run a post-deployment Data Loader job to identify and bulk-flag affected records using Bulk_Update_Error_Log__c pattern from INS-1504. Deploy validation rule change through a Change Set with unit tests covering null-to-invalid, null-to-valid, valid-to-invalid, and valid-to-valid transitions.
