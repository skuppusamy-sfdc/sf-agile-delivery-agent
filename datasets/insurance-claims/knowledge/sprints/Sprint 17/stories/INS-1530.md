## Description

As a Provider Network Specialist, I need a mass inactivation tool for Fee_Schedule__c records associated with CPT codes that have been retired by CMS so that our fee schedules remain compliant with the annual CMS CPT code updates and adjudication does not apply reimbursement rates for non-billable procedure codes. The annual CMS update typically affects 200–400 Fee_Schedule__c records across the provider network.

## Acceptance Criteria

Given a Provider Network Specialist accesses the 'Fee Schedule Mass Update' utility page,
When they input a list of retired CPT codes (paste or CSV upload, max 500 codes),
Then the system performs a preview query showing all Fee_Schedule__c records where Fee_Schedule__c.CPT_Code__c IN (submitted list) AND Fee_Schedule__c.Status__c = 'Active', displaying count by LOB and provider network tier.

Given the preview is confirmed by the specialist,
When the mass inactivation job is submitted,
Then all matching Fee_Schedule__c records are updated: Fee_Schedule__c.Status__c = 'Inactive', Fee_Schedule__c.Inactivation_Reason__c = 'CMS CPT Retirement', Fee_Schedule__c.Inactivated_Date__c = TODAY(), Fee_Schedule__c.Inactivated_By__c = running user.

Given the mass inactivation is complete,
When any active Claim_Line__c exists with Claim_Line__c.CPT_Code__c matching an inactivated code AND Claim_Line__c.Status__c IN ('Pending', 'In Adjudication'),
Then those Claim_Line__c records are flagged with Claim_Line__c.Fee_Schedule_Exception__c = TRUE and routed to the Claims Operations Manager review queue.

Given the job completes,
When results are available,
Then a downloadable audit report is generated listing: Fee_Schedule__c.Id, Fee_Schedule__c.CPT_Code__c, Provider__c.Name, Fee_Schedule__c.Allowed_Amount__c (pre-inactivation), and timestamp.

Given a CPT code submitted for inactivation is not found in any active Fee_Schedule__c,
When the preview renders,
Then the code appears in an 'Unmatched Codes' section with a note 'No active fee schedule records found.'

## Solution

Develop an LWC ('FeeScheduleMassInactivation') embedded in a custom Lightning App Page accessible via the Provider Network app. The component uses wire adapters for the preview query and an Apex controller (FeeScheduleMassInactivationController) for the DML operation executed via Batchable Apex (FeeScheduleInactivationBatch) in chunks of 200. A post-batch step queries impacted Claim_Line__c records and updates flags. The audit report is generated using a Conga Composer template triggered via a custom button on the batch job result record.
