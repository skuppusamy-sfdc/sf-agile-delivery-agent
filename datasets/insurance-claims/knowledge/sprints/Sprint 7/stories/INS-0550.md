## Description

**Bug Report**

**Summary:** When a Claim_Line__c record contains a CPT modifier code (e.g., CPT_Modifier__c = '25', '59', 'GT'), the Payment__c.Amount__c is calculated as $0.00 instead of the correct reimbursement amount derived from the Fee_Schedule__c.

**Environment:** Acme Insurance - Full Sandbox (AcmeFS1)

**Steps to Reproduce:**
1. Navigate to an existing Claim__c record in 'In Adjudication' status.
2. Open a related Claim_Line__c record and set Claim_Line__c.CPT_Code__c = '99213' and Claim_Line__c.CPT_Modifier__c = '25'.
3. Set Claim_Line__c.Line_Status__c = 'Approved' and save.
4. Trigger adjudication completion by setting Claim__c.Status__c = 'Adjudicated'.
5. Observe the created Payment__c record — Payment__c.Amount__c = $0.00.

**Expected Behavior:** Payment__c.Amount__c should equal the Fee_Schedule_Line__c.Reimbursement_Amount__c matched on CPT_Code__c '99213' with modifier '25', or fall back to the base CPT rate if no modifier-specific rate exists.

**Actual Behavior:** Payment__c.Amount__c = $0.00 for all Claim_Line__c records where CPT_Modifier__c is non-null.

**Root Cause (Preliminary):** The FeeScheduleLookupService.cls SOQL query filters on Fee_Schedule_Line__c.CPT_Code__c only and does not account for CPT_Modifier__c, returning no results when a modifier is present, causing the amount to default to 0.

**Impact:** High — affects approximately 34% of adjudicated claims in the current sprint test cycle. Blocks INS-0545 UAT sign-off.

## Acceptance Criteria

As per Description

## Solution

Fix FeeScheduleLookupService.cls to implement a two-pass lookup strategy: (1) first query Fee_Schedule_Line__c WHERE CPT_Code__c = :cptCode AND CPT_Modifier__c = :modifier; (2) if no result, fall back to query WHERE CPT_Code__c = :cptCode AND CPT_Modifier__c = null or CPT_Modifier__c = ''. Add a unit test (FeeScheduleLookupServiceTest.cls) covering modifier-present, modifier-absent, and modifier-fallback scenarios with minimum 90% code coverage. Deploy fix to AcmeFS1 for re-test by QA.
