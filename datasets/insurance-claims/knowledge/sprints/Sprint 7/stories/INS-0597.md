# INS-0597: Bug: Payment__c records created with NULL Payee_TIN__c when Provider is a Facility (Business Account)

## Description

**Bug Summary:** Payment__c records are being created with a NULL value in Payment__c.Payee_TIN__c when the claim's rendering provider is a Facility__c (Business Account) rather than a Provider__c (Person Account). This causes ERA file generation (INS-0594) to fail and prevents check printing for facility-based claims.

**Steps to Reproduce:**
1. Navigate to an existing Claim__c record where Claim__c.Provider_Type__c = 'Facility'.
2. Ensure the associated Facility__c record has Facility__c.TIN__c populated (e.g., '12-3456789').
3. Trigger the payment finalization flow (set Claim__c.Status__c = 'Adjudicated').
4. Navigate to the newly created Payment__c record.
5. Observe that Payment__c.Payee_TIN__c = NULL despite the Facility__c.TIN__c being populated.

**Expected Behavior:** Payment__c.Payee_TIN__c should be populated from Facility__c.TIN__c when Claim__c.Provider_Type__c = 'Facility', or from Provider__c.TIN__c when Provider_Type__c = 'Individual'.

**Actual Behavior:** Payment__c.Payee_TIN__c = NULL for all facility-based claims. The PaymentCreationFlow only resolves TIN from the Provider__c (Person Account) lookup and does not handle the Facility__c (Business Account) path.

**Impact:** High — affects approximately 34% of claims volume (all facility claims). ERA batch (INS-0594) throws a NullPointerException on TIN grouping logic. Estimated 210 affected Payment__c records in current sprint cycle.

**Environment:** Full Sandbox (Acme-FS1)
**Regression:** Introduced in Sprint 6 deployment (INS-0581 PaymentCreationFlow update)

## Acceptance Criteria

As per Description.

Additional verification:
Given a Claim__c with Claim__c.Provider_Type__c = 'Facility' and a linked Facility__c with Facility__c.TIN__c = '12-3456789',
When the payment finalization flow executes,
Then Payment__c.Payee_TIN__c = '12-3456789' and Payment__c.Payee_Account__c = Facility__c.Id.

Given a Claim__c with Claim__c.Provider_Type__c = 'Individual' and a linked Provider__c with Provider__c.TIN__c = '98-7654321',
When the payment finalization flow executes,
Then Payment__c.Payee_TIN__c = '98-7654321' and Payment__c.Payee_Account__c = Provider__c.Id (regression check — existing behavior must be preserved).

Given the 210 existing NULL-TIN Payment__c records,
When the data remediation script executes,
Then all affected records are backfilled with the correct TIN from the linked Facility__c or Provider__c.

## Solution

Fix the PaymentCreationFlow (Record-Triggered Flow on Claim__c) by adding a Decision element after the existing 'Get Provider TIN' Get Records element. The Decision evaluates Claim__c.Provider_Type__c: if 'Facility', a new Get Records element retrieves Facility__c.TIN__c via Claim__c.Facility__c lookup; if 'Individual', the existing Provider__c TIN path is used. Both paths feed into a single Assignment element that sets Payment__c.Payee_TIN__c before the Create Records element. Additionally, an Apex Data Loader script (or anonymous Apex in sandbox) will backfill the 210 affected Payment__c records by querying Claim__c.Provider_Type__c and joining to the appropriate TIN source. Code review required before deployment to ensure no regression on INS-0594 ERA batch TIN grouping logic.
