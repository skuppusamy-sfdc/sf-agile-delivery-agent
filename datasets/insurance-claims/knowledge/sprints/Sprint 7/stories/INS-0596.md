## Description

As a Claims Operations Manager, I need a guided OmniScript to void an erroneously issued Payment__c and reissue a corrected payment, so that disbursement errors are resolved in a controlled, auditable manner with appropriate approvals and without creating duplicate financial records.

The OmniScript must capture the void reason, validate the payment is eligible for voiding, route for supervisor approval if the payment exceeds a configurable threshold, update the original Payment__c, create a new corrected Payment__c, and log a Payment_Audit__c record for compliance tracking.

## Acceptance Criteria

Given a Claims Operations Manager initiates the Void & Reissue OmniScript for a Payment__c record,
When the OmniScript Step 1 loads,
Then the current payment details are displayed (Payment__c.Reference_Number__c, Payment__c.Payment_Amount__c, Payment__c.Payment_Date__c, Payment__c.Payee_Name__c) in read-only fields.

Given the user selects a Void Reason from the picklist (values: 'Incorrect Payee', 'Incorrect Amount', 'Duplicate Payment', 'Provider Request', 'Other'),
When 'Other' is selected,
Then a mandatory free-text field 'Void_Reason_Detail__c' appears requiring minimum 20 characters.

Given the Payment__c.Payment_Amount__c exceeds the threshold defined in Custom Metadata (Payment_Void_Config__mdt.Approval_Threshold__c),
When the user submits the void request,
Then an Approval Process is triggered (Payment_Void_Approval) and Payment__c.Void_Approval_Status__c = 'Pending Supervisor Approval'; the OmniScript displays a confirmation step with expected SLA.

Given the void is approved (or amount is below threshold),
When the OmniScript finalizes,
Then Payment__c.Status__c = 'Voided', Payment__c.Void_Date__c = TODAY(), Payment__c.Void_Reason__c is populated, a new Payment__c record is created with Payment__c.Status__c = 'Pending Issuance' and Payment__c.Reissued_From__c = original Payment__c.Id, and a Payment_Audit__c record is created with all before/after values.

Given a Payment__c with Payment__c.Status__c != 'Issued',
When the OmniScript attempts to load,
Then an error message is displayed: 'This payment is not eligible for voiding. Current status: [Status]' and the OmniScript cannot proceed past Step 1.

## Solution

Build a multi-step OmniStudio OmniScript (VoidReissuePayment) with DataRaptor Extract for pre-population of payment details, a Conditional View element for the 'Other' reason free-text, and an Integration Procedure (IP_VoidPaymentOrchestrator) that orchestrates: (1) eligibility validation via DataRaptor, (2) threshold check against Payment_Void_Config__mdt, (3) Approval Process submission via Apex Action if threshold exceeded, (4) DML operations to void original and create new Payment__c via a bulkified Apex invocable method (PaymentVoidService.voidAndReissue()), and (5) Payment_Audit__c creation. The Approval Process (Payment_Void_Approval) uses a 2-step hierarchy with auto-escalation after 4 business hours using a Time-Based Workflow. Custom Metadata (Payment_Void_Config__mdt) stores the approval threshold, making it configurable per LOB without code changes.
