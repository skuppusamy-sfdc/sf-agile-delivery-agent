## Description

**Bug Report**

**Summary:** When a Claim__c contains a mix of Claim_Line__c records with Line_Status__c = 'Approved' and Line_Status__c = 'Denied', the Payment__c.Payment_Amount__c is incorrectly summing ALL Claim_Line__c.Allowed_Amount__c values regardless of line status, resulting in overpayment.

**Steps to Reproduce:**
1. Create a Claim__c record with Status__c = 'In Adjudication'
2. Create 3 Claim_Line__c records under the Claim: Line 1 (Line_Status__c = 'Approved', Allowed_Amount__c = 150.00), Line 2 (Line_Status__c = 'Approved', Allowed_Amount__c = 200.00), Line 3 (Line_Status__c = 'Denied', Allowed_Amount__c = 75.00)
3. Update Claim__c.Status__c to 'Adjudicated'
4. Observe the auto-created Payment__c record

**Expected Result:** Payment__c.Payment_Amount__c = 350.00 (sum of Approved lines only)

**Actual Result:** Payment__c.Payment_Amount__c = 425.00 (sum of all lines including Denied)

**Environment:** Full Sandbox (Acme-FS1)
**Regression:** Introduced in Sprint 6 deployment (INS-0571 changes to payment creation flow)

## Acceptance Criteria

As per Description. Payment__c.Payment_Amount__c must equal the sum of Claim_Line__c.Allowed_Amount__c WHERE Line_Status__c IN ('Approved', 'Partially Approved') only. Denied lines must be excluded from the sum.

## Solution

Root cause identified in the Record-Triggered Flow (Claim_Adjudication_Payment_Flow) — the Get Records element retrieving Claim_Line__c records is missing the Line_Status__c filter condition. Fix: add filter Line_Status__c IN ('Approved', 'Partially Approved') to the Get Records element. Update the Flow formula variable used for payment amount summation. Deploy fix to sandbox, execute regression test covering all line status combinations, then promote to production via change set. Add dedicated test method in ClaimPaymentTriggerTest covering mixed-status line scenarios with assertion on Payment_Amount__c.
