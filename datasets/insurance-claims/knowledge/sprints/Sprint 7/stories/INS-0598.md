## Description

As a Claims Operations Manager, I need the claims adjudication engine to automatically apply Coordination of Benefits (COB) logic when a policyholder has dual insurance coverage, so that Acme Insurance pays only its share of the claim after the primary insurer's payment is accounted for, reducing overpayment risk and ensuring compliance with NAIC COB model regulations.

COB data is captured on the Policy__c object and must be evaluated during Claim_Line__c adjudication to calculate the correct Plan_Paid__c amount. Both 'Standard COB' and 'Non-Duplication' COB methods must be supported, driven by Policy__c.COB_Method__c.

## Acceptance Criteria

Given a Claim__c where the linked Policy__c.COB_Applicable__c = true and Policy__c.Primary_Insurer__c is populated,
When the adjudication engine processes each Claim_Line__c,
Then Claim_Line__c.COB_Applied__c = true and Claim_Line__c.Primary_Insurer_Paid__c is populated from the COB data entered on the claim.

Given Policy__c.COB_Method__c = 'Standard COB',
When the Claim_Line__c adjudication calculates the Plan_Paid__c,
Then Plan_Paid__c = MAX(0, Claim_Line__c.Allowed_Amount__c - Claim_Line__c.Primary_Insurer_Paid__c - Claim_Line__c.Member_Deductible_Applied__c - Claim_Line__c.Member_Copay__c).

Given Policy__c.COB_Method__c = 'Non-Duplication',
When the Claim_Line__c adjudication calculates Plan_Paid__c,
Then Plan_Paid__c = MAX(0, Acme_Standalone_Benefit__c - Claim_Line__c.Primary_Insurer_Paid__c), where Acme_Standalone_Benefit__c is the amount Acme would have paid as primary insurer.

Given COB is applied and Plan_Paid__c is calculated,
When the parent Claim__c is updated,
Then Claim__c.COB_Savings__c = SUM(Claim_Line__c.Primary_Insurer_Paid__c) across all lines and Claim__c.COB_Applied__c = true.

-Given COB data is missing (Policy__c.Primary_Insurer__c is null) but COB_Applicable__c = true, When adjudication runs, Then the claim is auto-suspended to the COB Research queue.- (Deprecated: auto-suspend logic moved to a separate SIU referral story INS-0603.)

Given COB is applied and the calculated Plan_Paid__c = 0 for all Claim_Line__c records,
When the Claim__c is finalized,
Then Claim__c.Status__c = 'Adjudicated - Zero Pay' and no Payment__c record is created; an EOB is still generated per INS-0593 with zero payment amounts.

Given the adjudication results are finalized,
When the EOB (INS-0593) is generated,
Then EOB__c.COB_Savings__c is populated and a COB explanation section is included in the PDF output.

## Solution

Extend the existing ClaimsAdjudicationEngine Apex class (introduced in Sprint 5) with a new COBCalculationService Apex class that implements both Standard COB and Non-Duplication calculation methods, driven by Policy__c.COB_Method__c. The service is invoked from the Claim_Line__c adjudication loop within the batch processing context (ClaimsAdjudicationBatch). COB method logic is encapsulated using a Strategy design pattern with two concrete Apex classes (StandardCOBStrategy, NonDuplicationCOBStrategy) implementing a COBStrategy interface, enabling easy addition of future COB methods. Custom Metadata type (COB_Config__mdt) stores method-specific parameters per LOB. A Record-Triggered Flow on Claim__c rolls up COB_Savings__c from child Claim_Line__c records after adjudication completes. Integration with INS-0593 EOB generation is handled by passing COB flag through the EOBGenerationService invocable method parameters.
