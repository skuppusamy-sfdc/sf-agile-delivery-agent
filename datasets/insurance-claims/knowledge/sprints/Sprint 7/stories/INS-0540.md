## Description

**Bug Report — INS-0540**

**Environment:** Acme Insurance Full Sandbox (SF-FULL-01)
**Discovered by:** Rachel Torres, Claims Operations Manager, during UAT regression on INS-0537

**Steps to Reproduce:**
1. Create a Claim__c record with Status__c = 'Under Review' linked to an active Provider__c with a valid PNA.
2. Add 11 or more Claim_Line__c records to the claim, each with Adjudication_Status__c = 'Approved' and Approved_Amount__c = $100.00 (expected total: $1,100.00).
3. Update Claim__c.Status__c to 'Adjudicated'.
4. Observe the auto-created Payment__c record.

**Expected Result:** Payment__c.Payment_Amount__c = $1,100.00

**Actual Result:** Payment__c.Payment_Amount__c = $2,200.00 (exactly doubled)

**Additional Notes:** Bug does not reproduce with 10 or fewer Claim_Line__c records. Appears to be a SOQL query or aggregation issue in PaymentRecordFactory.cls — possibly a duplicate query execution or a loop running twice when the list exceeds the default SOQL row limit threshold set in the current implementation. Checked debug logs: AggregateResult query for SUM(Approved_Amount__c) is being called twice in the transaction when record count > 10.

## Acceptance Criteria

As per Description. Fix verified when Payment__c.Payment_Amount__c equals the exact SUM of all Claim_Line__c.Approved_Amount__c records regardless of line count (tested with 1, 10, 11, 50, and 200 lines).

## Solution

Root cause identified in PaymentRecordFactory.cls: the aggregation SOQL (AggregateResult) is invoked once in the flow-invocable method and a second time redundantly within a for-loop iterating Claim_Line__c records. Fix: remove the redundant in-loop SOQL call and rely solely on the single AggregateResult query result. Add a guard condition to prevent double-invocation. Update unit tests (PaymentRecordFactoryTest.cls) to include test scenarios with 11+ Claim_Line__c records. Deploy fix to sandbox and rerun UAT regression suite before promoting to production.
