## Description

**Bug Report — EOB Records Not Updated When Parent Claim Is Voided**

**Steps to Reproduce:**
1. Create a Claim__c record and advance to Status__c = 'Adjudicated'
2. Confirm Explanation_of_Benefits__c record is created with Delivery_Status__c = 'Sent' (as per INS-0514 behavior)
3. Navigate to the Claim__c record and change Status__c = 'Voided' (e.g., due to duplicate claim identification or SIU referral)
4. Observe the linked Explanation_of_Benefits__c record

**Expected Behavior:**
When Claim__c.Status__c is set to 'Voided', all linked Explanation_of_Benefits__c records should have their EOB_Status__c updated to 'Voided' and Delivery_Status__c updated to 'Void Notice Pending' so that a corrected void notice can be dispatched to the member.

**Actual Behavior:**
Explanation_of_Benefits__c records retain EOB_Status__c = 'Generated' and Delivery_Status__c = 'Sent'. No void notice is triggered. This causes members to hold an active EOB for a voided claim, creating compliance exposure and member confusion.

**Additionally:** Any Payment__c records linked to the voided Claim__c remain in Payment_Status__c = 'Disbursed' without triggering a reversal workflow, which may cause financial reconciliation failures in Workday.

**Environment:** Acme Insurance Full Sandbox (SF-SANDBOX-02) and Production (confirmed by Claims Ops team on 2024-11-14)
**Frequency:** Every claim void action
**Severity:** Medium-High — compliance and financial reconciliation risk

## Acceptance Criteria

As per Description.

Given Claim__c.Status__c is changed to 'Voided'
When the record is saved
Then all child Explanation_of_Benefits__c records linked via Claim__c lookup have EOB_Status__c = 'Voided' and Delivery_Status__c = 'Void Notice Pending'

Given Explanation_of_Benefits__c.Delivery_Status__c = 'Void Notice Pending'
When the void notice process runs
Then a DocuSign void notice envelope is dispatched to the member and Delivery_Status__c = 'Void Notice Sent'

Given Payment__c records exist with Payment_Status__c = 'Disbursed' linked to the voided Claim__c
When the claim void action completes
Then Payment__c.Payment_Status__c = 'Reversal Pending' and a Payment_Reversal_Case__c is auto-created and assigned to the Claims Operations queue

## Solution

Add an Apex Trigger (ClaimVoidTrigger) on Claim__c firing after update when Status__c changes to 'Voided'. The trigger handler (ClaimVoidHandler) queries all related Explanation_of_Benefits__c and Payment__c records in bulk. EOB records are bulk-updated via DML with EOB_Status__c = 'Voided' and Delivery_Status__c = 'Void Notice Pending'. A Platform Event (Claim_Voided__e) is published with the Claim Id; a subscribing Flow handles DocuSign void envelope dispatch using the existing DocuSign Salesforce app Send action. Payment reversal cases are created via a separate Apex invocable called from the same trigger handler. Add a Process Exception handler to log any DML failures to an Error_Log__c record to prevent silent failures. Ensure trigger is bulkified to handle batch void scenarios from data migration or SIU mass-void operations.
