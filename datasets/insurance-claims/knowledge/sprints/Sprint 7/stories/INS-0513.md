## Description

**Bug Report**

**Summary:** When an Explanation_of_Benefits__c (EOB) PDF is generated via the Conga CLM merge for a Claim__c that has more than 20 associated Claim_Line__c records, the Conga merge job completes with a success callback but the resulting PDF is either blank in the line items section or truncated at row 20. No error is thrown, no Integration_Error_Log__c record is written, and EOB__c.Status__c is incorrectly set to 'Generated' despite the incomplete document. This was discovered during UAT testing with a hospital facility claim (Facility__c) that had 34 Claim_Line__c records.

**Steps to Reproduce:**
1. Create a Claim__c record with Status__c = 'Adjudicated' and 21 or more associated Claim_Line__c records (each with CPT_Code__c, Billed_Amount__c, Allowed_Amount__c populated).
2. Create a linked Payment__c and set Payment__c.Status__c = 'Approved'.
3. Observe the Record-Triggered Flow fire and the EOBGenerationService Queueable execute.
4. Download the generated PDF attached to the resulting EOB__c record.
5. Observe line items in the PDF.

**Expected Result:** All Claim_Line__c records (21+) appear as line items in the EOB PDF.

**Actual Result:** Only the first 20 Claim_Line__c records appear; remaining lines are silently dropped. EOB__c.Status__c = 'Generated' despite incomplete content.

**Environment:** Acme Insurance UAT Sandbox (AIS-UAT)
**First Observed:** UAT Cycle 2, 2024-11-15, Tester: Kevin Nguyen
**Severity:** High — regulatory non-compliance risk; EOBs sent to members with incomplete claim line detail
**Related Story:** INS-0507 (EOB generation implementation)

## Acceptance Criteria

As per Description.

Given a Claim__c with 21 or more Claim_Line__c child records,
When the Conga CLM merge is triggered for EOB PDF generation,
Then all Claim_Line__c records must appear in the generated PDF and EOB__c.Status__c = 'Generated' only when the PDF line item count matches the total Claim_Line__c count.

Given the PDF line item count does not match the expected Claim_Line__c count post-merge,
When the EOBGenerationService callback processes the Conga response,
Then EOB__c.Status__c must be set to 'Generation Failed', an Integration_Error_Log__c record must be created with Error_Source__c = 'CongaEOBMerge' and a descriptive Error_Message__c, and a Case must be created for the Claims Operations queue.

## Solution

Root cause: the Conga CLM template 'EOB_Standard_Template_v3' has a static repeating table capped at 20 rows due to a template design constraint. Fix requires two changes: (1) Update the Conga template to use a dynamic repeating section (Conga '{{TableStart:ClaimLines}}...{{TableEnd:ClaimLines}}' merge syntax) removing the static row cap — coordinate with Conga admin; (2) Update EOBGenerationService Queueable to pass all Claim_Line__c records in the DataSource query without a LIMIT clause (current query has 'LIMIT 20' — remove this). Add a post-merge validation step in the Queueable's callback handler that compares the ContentVersion page count heuristic or a returned line count against the expected Claim_Line__c count (queried fresh), and sets EOB__c.Status__c = 'Generation Failed' with Integration_Error_Log__c creation if mismatch detected. Update EOBGenerationServiceTest with a 25-line test scenario asserting full line item inclusion.
