## Description

As a Claims Operations Manager, I need Payment__c records to be automatically created and staged when a Claim__c reaches 'Adjudicated' status so that disbursement workflows are triggered without manual intervention, reducing processing lag and human error in high-volume LOB queues.

## Acceptance Criteria

Given a Claim__c record where Status__c transitions to 'Adjudicated'
When the Record-Triggered Flow fires on the Claim__c update
Then a Payment__c record is created with Payment__c.Status__c = 'Pending', Payment__c.Claim__c populated with the parent Claim__c.Id, Payment__c.Payment_Amount__c = Claim__c.Approved_Amount__c, and Payment__c.Payment_Method__c defaulting to 'EFT'

Given a Payment__c record is created in 'Pending' status
When Payment__c.Payment_Method__c = 'EFT'
Then Payment__c.Expected_Settlement_Date__c must be set to TODAY() + 3 business days

Given a Payment__c record is created in 'Pending' status
When Payment__c.Payment_Method__c = 'Check'
Then Payment__c.Expected_Settlement_Date__c must be set to TODAY() + 7 business days

Given a Claim__c with Claim__c.COB_Flag__c = TRUE
When the adjudication completes
Then Payment__c creation must be held with Payment__c.Status__c = 'COB Review Hold' and a Task is created assigned to the COB queue

Given a Claim__c where Claim__c.SIU_Flag__c = TRUE
When adjudication completes
Then Payment__c.Status__c must be set to 'SIU Hold' and no disbursement Platform Event is published until the hold is lifted

-Given Payment__c.Payment_Amount__c > $50,000 a second approval step is required via manual queue- (superseded by INS-0531 threshold logic)

Given a Payment__c record is successfully created
When all validations pass
Then a ClaimPaymentInitiated__e Platform Event is published containing Claim__c.Id, Payment__c.Id, and Payment__c.Payment_Amount__c

## Solution

Implement a Record-Triggered Flow on Claim__c (after-save, on update) that evaluates Status__c transition to 'Adjudicated'. Flow creates Payment__c via a Create Records element. COB and SIU flag checks are handled via Decision elements branching to separate assignment paths. A separate Apex invocable method publishes the ClaimPaymentInitiated__e Platform Event post-creation to decouple disbursement processing. Business day calculation for Expected_Settlement_Date__c handled via a custom Apex utility class (BusinessDayCalculator.cls). Permission Set Group 'Claims_Payment_Ops' grants FLS on Payment__c fields to relevant roles.
