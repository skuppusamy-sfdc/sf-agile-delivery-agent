## Description

As a Claims Operations Manager, I need each Claim_Line__c to be validated against the active Provider_Network_Agreement__c (PNA) and associated Fee_Schedule__c for the rendering provider at the time of service so that Claim_Line__c.Allowed_Amount__c is accurately calculated based on contracted rates rather than billed charges. This prevents overpayment and ensures compliance with provider contract terms.

## Acceptance Criteria

Given a Claim_Line__c record is submitted for adjudication
When the adjudication engine processes the line
Then the system queries for an active Provider_Network_Agreement__c where PNA__c.Provider__c = Claim__c.Rendering_Provider__c AND PNA__c.Effective_Date__c <= Claim__c.Date_of_Service__c AND PNA__c.Termination_Date__c >= Claim__c.Date_of_Service__c AND PNA__c.Status__c = 'Active'

Given an active PNA is found
When the system looks up the Fee_Schedule__c
Then it queries Fee_Schedule__c where Fee_Schedule__c.PNA__c = PNA__c.Id AND Fee_Schedule__c.CPT_Code__c = Claim_Line__c.CPT_Code__c AND Fee_Schedule__c.Effective_Date__c <= Claim__c.Date_of_Service__c

Given a matching Fee_Schedule__c record is found
When the allowed amount is calculated
Then Claim_Line__c.Allowed_Amount__c = Fee_Schedule__c.Contracted_Rate__c and Claim_Line__c.Pricing_Method__c = 'Contracted Rate' and Claim_Line__c.PNA__c = the matched PNA__c.Id

Given no active PNA is found for the rendering provider at date of service
When the adjudication engine evaluates
Then Claim_Line__c.Allowed_Amount__c is calculated using the default UCR (Usual, Customary, and Reasonable) schedule and Claim_Line__c.Pricing_Method__c = 'UCR' and Claim_Line__c.Out_of_Network_Flag__c = TRUE

Given a PNA is found but no matching Fee_Schedule__c CPT code entry exists
When the lookup returns no results
Then Claim_Line__c.Pricing_Method__c = 'UCR Fallback', Claim_Line__c.Missing_Fee_Schedule_Flag__c = TRUE, and a FeeScheduleGap__c record is created for provider network team review

Given a GSA (Group_Service_Agreement__c) is also active for the same provider and LOB
When both a PNA and GSA apply
Then PNA contracted rate takes precedence over GSA rate per adjudication hierarchy rules (PNA > GSA > UCR)

-Claim_Line__c.Allowed_Amount__c must not exceed Claim_Line__c.Billed_Amount__c under any pricing method- (logic moved to separate balance billing validation story INS-0572)

## Solution

Apex service class (FeeScheduleValidationService.cls) invoked from the Claim adjudication OmniScript step. Service performs SOQL chain: PNA lookup → Fee_Schedule__c lookup by CPT code and date of service. Results cached using Platform Cache (Partition: ClaimsAdjudication) keyed by Provider__c.NPI__c + CPT code + DOS to reduce SOQL calls in bulk scenarios. UCR fallback logic references a separate UCR_Schedule__c custom object. FeeScheduleGap__c records created via a separate Queueable to avoid governor limit issues in bulk adjudication. Adjudication hierarchy (PNA > GSA > UCR) enforced via ordered IF-ELSE logic in the service class. Unit tests in FeeScheduleValidationServiceTest.cls cover all pricing paths with 95%+ coverage.
