# INS-0544: Implement COB (Coordination of Benefits) adjudication logic to calculate primary vs secondary payer obligations on Claim__c

## Description

As a Claims Operations Manager, I need the system to automatically apply Coordination of Benefits (COB) logic when a Claim__c is flagged as having multiple payers so that Acme Insurance's liability as primary or secondary payer is correctly calculated, preventing overpayment and ensuring compliance with CMS COB guidelines. This story extends INS-0537 and INS-0539 by adding a pre-payment COB calculation step before Payment__c creation.

## Acceptance Criteria

Given a Claim__c where COB_Applicable__c = true and COB_Payer_Order__c = 'Primary'
When adjudication runs
Then Claim__c.COB_Primary_Payer_Amount__c = SUM(Claim_Line__c.Approved_Amount__c) and Claim__c.COB_Secondary_Payer_Amount__c = 0, and COB_Calculation_Status__c = 'Calculated'

Given a Claim__c where COB_Payer_Order__c = 'Secondary'
When adjudication runs
Then Claim__c.COB_Secondary_Payer_Amount__c = MAX(0, SUM(Claim_Line__c.Billed_Amount__c) - Claim__c.COB_Primary_Paid_Amount__c - SUM(Claim_Line__c.Patient_Responsibility_Amount__c)) and Claim__c.COB_Adjusted_Amount__c is set accordingly

Given COB_Payer_Order__c = 'Secondary' and COB_Primary_Paid_Amount__c >= SUM(Claim_Line__c.Billed_Amount__c)
When COB calculation runs
Then COB_Secondary_Payer_Amount__c = 0, Claim__c.COB_Zero_Pay_Flag__c = true, and an informational note is added to Claim__c.Adjudication_Notes__c = 'COB: Primary payer satisfied full billed amount. No secondary liability.'

Given a Claim__c where COB_Applicable__c = true but COB_Primary_Paid_Amount__c is null or blank
When adjudication is attempted
Then Claim__c.Status__c is set to 'Pending_COB_Info', a Task is created assigned to the COB Coordination queue with Subject = 'COB Primary Payment Info Required — ' + Claim__c.Name, and adjudication is halted

Given COB_Calculation_Status__c = 'Calculated'
When the Payment__c creation flow from INS-0537 fires
Then Payment__c.Payment_Amount__c uses Claim__c.COB_Adjusted_Amount__c instead of the raw SUM of Claim_Line__c.Approved_Amount__c

-Given MCO-administered claims, apply MCO-specific COB rate tables from GSA- (deprecated: MCO COB rate table integration deferred to Phase 2 per Underwriting Analyst review)

Given a completed COB calculation
When Claim__c.COB_Calculation_Status__c = 'Calculated'
Then a COB_Audit_Record__c is created capturing COB_Payer_Order__c, COB_Primary_Paid_Amount__c, COB_Secondary_Payer_Amount__c, COB_Adjusted_Amount__c, and Calculated_By_User__c = the running user's Id for compliance reporting purposes

## Solution

Implement COBCalculationService.cls (Apex) invoked as an invocable action within the Claim adjudication Record-Triggered Flow, executing before the Payment__c creation step. COB logic follows CMS standard COB calculation rules encoded as a Custom Metadata Type (COB_Rule__mdt) keyed by Payer_Order__c and LOB__c to allow rule updates without code deployment. COB_Audit_Record__c creation handled within the same Apex transaction to ensure atomicity. The INS-0537 Record-Triggered Flow is updated to check COB_Applicable__c and route through COBCalculationService before PaymentRecordFactory. A COB Coordination queue is configured in Salesforce with round-robin assignment to Claims Operations team members. Compliance reports on COB_Audit_Record__c built using CRM Analytics (Tableau CRM) dashboard 'COB Compliance Overview'.
